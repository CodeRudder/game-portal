## 二、关键经验教训

### 1. 大型TSX文件修复策略
**问题**: PixiGame.tsx 6077行，commit 4dba620将主组件JSX错误放入子组件CampaignPanel，导致22个编译错误。

**尝试过的失败方案**:
- 子任务修复(递归限制5/6/8/10/15/20轮均失败)
- ACP agent(claude_code)无法访问本地文件系统
- 直接sed删除代码行导致JSX结构级联错误(759个)
- str_replace修复单个ternary后暴露更多错误

**最终成功方案**:
1. 用python脚本提取代码块(1079-1308行, 230行)
2. 从原位置删除代码块
3. 插入到主组件return末尾(footer之前)
4. 修复移动版bonds ternary缺少`: null`

**教训**: 
- 6077行TSX文件太大，子agent无法在上下文窗口内处理
- python脚本做原子操作比子任务更可靠
- JSX结构修复需要精确计算div嵌套深度
- 应尽早拆分大文件为多个子组件

### 2. 编译缓存陷阱
**问题**: 某次编译显示只有1个错误(ternary)，但实际有22个变量未定义错误。

**根因**: TypeScript增量编译缓存(tsconfig.tsbuildinfo)掩盖了部分错误。

**教训**: 修复后务必执行 `rm -rf tsconfig.tsbuildinfo && npx tsc --noEmit` 清除缓存验证。

### 3. 子任务递归限制
**问题**: 所有子任务(无论max_turns设为多少)都在处理大型TSX文件时触发递归限制。

**根因**: 大文件读取+分析+修改的操作链太长，每次read_file都会消耗上下文。

**教训**:
- 对于大型文件修改，优先在主会话用python脚本执行
- 子任务适合做代码搜索、分析等只读操作
- 将大任务拆分为多个小原子操作(每个只做一件事)
