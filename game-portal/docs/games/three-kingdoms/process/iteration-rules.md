# 三国霸业 — 版本迭代调度规则

> 本文档为 v1.0~v20.0 全部版本迭代的执行纲领，所有开发必须严格遵守。

---

## 一、核心原则

### 1.1 堆积木式迭代，按DDD架构扩展

> 详细架构设计见 `../architecture/ddd-architecture.md`

每个版本在DDD架构上追加新业务域，或在已有域内追加功能：

- Engine主类（`engine/ThreeKingdomsEngine.ts`）始终只有一个，每版本追加编排逻辑
- 每个业务域目录（`engine/{domain}/`）包含 System + Config + Types 三个文件
- UI主组件（`ThreeKingdomsGame.tsx`）始终只有一个，每版本追加面板编排
- 每个业务域UI目录（`panels/{domain}/`）包含该域的面板组件

**禁止**：每个版本创建独立引擎文件（如 `EngineV1.ts`、`EngineV2.ts`）
**正确**：每版本在唯一文件上追加积木，git commit 封版

### 1.2 严格按版本顺序，禁止跳版

```
v1.0 → v2.0 → v3.0 → ... → v20.0
```

每个版本必须封版（评测>9.8 + 测试100%通过 + commit）后，才能开始下一个版本。

## 一点五、DDD架构设计规范

> 领域驱动设计（Domain-Driven Design），按职责分层，每层独立可测试。

### 核心分层（从上到下依赖方向单向）

- **UI层（表现层）**：ThreeKingdomsGame.tsx 主组件编排 + panels/ 面板子组件
- **引擎层（应用服务层）**：ThreeKingdomsEngine.ts 编排各子系统
- **子系统层（领域层）**：ResourceSystem.ts / BuildingSystem.ts 等，每个子系统 = 一个领域聚合根
- **配置层（基础设施层）**：config/*.ts 纯数据零逻辑 + types.ts 共享类型定义

### 文件行数限制
- **所有代码文件 ≤ 500行**，硬性限制
- 核心文件（引擎/组件）严格遵守单一职责原则
- 超过500行必须拆分

### 单一职责原则
- 引擎文件只负责逻辑编排，不包含数值定义和具体业务逻辑
- 每个子系统文件 = 一个领域聚合根（如 ResourceSystem、BuildingSystem）
- 数值/常量拆分为独立的配置文件（如 `building-config.ts`、`resource-config.ts`）
- 类型定义拆分为 `types.ts`
- UI主组件只负责编排面板，每个面板一个独立文件

### 目录结构规范（按业务域纵向划分）

> 详细目录结构见 DDD 架构设计文档：`../architecture/ddd-architecture.md`

核心原则：**按业务域纵向划分子目录**，而非按文件类型横向划分。每个业务域目录内包含该域的全部文件（子系统逻辑+配置+类型），便于独立开发和扩展。

### DDD设计规则
1. **子系统自治**：每个子系统管理自己的状态，对外暴露清晰的API
2. **配置与逻辑分离**：config/ 只放纯数据（数值表），engine/ 只放逻辑
3. **类型集中管理**：所有跨子系统共享的类型定义在 types.ts
4. **引擎只编排**：ThreeKingdomsEngine 不包含具体业务逻辑，只负责子系统间的编排和协调
5. **UI只展示**：UI组件不包含游戏逻辑，所有逻辑通过引擎API调用
6. **依赖方向单向**：UI → Engine → SubSystem → Config，禁止反向依赖

### 架构审查检查清单（阶段3.5）
- [ ] 所有文件 ≤ 500行
- [ ] 引擎文件不含硬编码数值
- [ ] 每个文件职责单一，可一句话描述
- [ ] 类型定义与逻辑分离
- [ ] 配置数据与业务逻辑分离
- [ ] 无循环依赖
- [ ] 子系统API清晰，无内部状态泄漏
- [ ] UI组件不包含游戏逻辑

## 二、单版本开发调度流程

每个版本的开发分为 **4个阶段**，存在严格的依赖关系：

```
┌──────────────────────────────────────────────────────┐
│ 阶段1: 引擎层开发（串行，不可并行）                     │
│   读取plan+PRD → 在各职责文件中追加引擎逻辑             │
│   ↓ 必须完成                                          │
├──────────────────────────────────────────────────────┤
│ 阶段2: 逻辑层测试 + UI层开发（可并行）                  │
│   ┌─ 子任务A: 引擎单元测试                             │
│   └─ 子任务B: UI组件开发（依赖引擎API）                 │
│   ↓ 两者都必须完成                                     │
├──────────────────────────────────────────────────────┤
│ 阶段3: 编译验证 + 全量测试（串行）                      │
│   tsc编译 + vitest全量运行                             │
│   ↓ 必须通过                                          │
├──────────────────────────────────────────────────────┤
│ 阶段3.5: 架构审查 + 优化（串行，阶段3通过后）           │
│   文件行数检查(≤500行) + 单一职责检查 + 拆分优化        │
│   ↓ 审查通过才能继续                                   │
├──────────────────────────────────────────────────────┤
│ 阶段4: 评测师评审（串行，依赖阶段3.5通过）              │
│   10维度评分 → 是否>9.8                               │
│   ↓ 通过 → commit → 进入下一版本                       │
│   ↓ 不通过 → 改进 → 回到阶段2重新评测                   │
└──────────────────────────────────────────────────────┘
```

### 2.1 阶段依赖关系图

```
阶段1(引擎) ──→ 阶段2A(引擎测试) ──→ 阶段3(编译+全量测试) ──→ 阶段3.5(架构审查) ──→ 阶段4(评测)
                阶段2B(UI开发)   ──↗
```

- **阶段1 → 阶段2**：强依赖。引擎API确定后，测试和UI才能并行开发。
- **阶段2A ∥ 阶段2B**：可并行。引擎测试和UI开发互不阻塞。
- **阶段2 → 阶段3**：强依赖。代码和测试都完成后才能编译验证。
- **阶段3 → 阶段3.5**：强依赖。编译+测试通过后才能进行架构审查。
- **阶段3.5 → 阶段4**：强依赖。架构审查通过后才能评测。

### 2.2 子任务并发规则

| 阶段 | 子任务 | 可并行 | 说明 |
|------|--------|--------|------|
| 阶段1 | 引擎开发 | ❌ 单任务 | 必须先完成，确定API |
| 阶段2A | 引擎单元测试 | ✅ 与2B并行 | 基于阶段1的引擎API |
| 阶段2B | UI组件开发 | ✅ 与2A并行 | 基于阶段1的引擎API |
| 阶段3 | 编译+全量测试 | ❌ 单任务 | 等待2A+2B都完成 |
| 阶段3.5 | 架构审查+优化 | ❌ 单任务 | 等待阶段3通过 |
| 阶段4 | 评测师评审 | ❌ 单任务 | 等待阶段3.5通过 |

## 三、子任务调度约束

### 3.1 硬性并发限制

- **每个response最多3个task调用**（系统硬限制）
- 超过3个子任务必须分批执行
- 优先调度阻塞其他任务的关键路径任务

### 3.2 子任务类型分配

| 任务类型 | 推荐subagent | 说明 |
|---------|-------------|------|
| 引擎开发 | developer | 核心逻辑实现 |
| UI开发 | developer | 组件+样式实现 |
| 引擎测试 | tester | 单元测试+覆盖率 |
| UI测试 | tester | 交互测试+覆盖率 |
| 编译验证 | bash | tsc + vitest |
| 评测评审 | game-reviewer | 10维度评分 |

### 3.3 子任务拆分与超时控制

**每个子任务必须在10分钟内完成（超时硬限制15分钟）。**

拆分规则：
- 引擎开发按功能模块拆分：一个子系统 = 一个子任务
- UI开发按面板拆分：一个面板/组件 = 一个子任务
- 测试按模块拆分：一个模块的测试 = 一个子任务
- 评测评审 = 一个子任务

判断标准：
- ✅ 合格子任务：单一职责，代码量 ≤ 500行，10分钟可完成
- ❌ 过大子任务：多职责混合，代码量 > 500行，容易超时
- ❌ 过小子任务：合并到相关子任务中一起执行，禁止在主会话执行编码、调研等操作。主会话唯一职责是任务编排。

**子任务prompt必须明确**：
1. 任务计划文档路径（PLAN文件链接，其中包含PRD/UI章节链接）
2. 简述需要完成的功能点（禁止转述开发内容，只给文档路径+功能点编号/名称）
3. 上下文信息（如果是失败重试，说明已完成的进度和失败原因）
4. 验收要求（测试通过条件、编译零错误、覆盖率目标等）

### 3.4 子任务失败恢复（全自动，无需用户干预）

```
子任务失败
    ↓
分析失败原因：
    ├─ 超时(>15min) → 拆分为更小的子任务，继续执行
    ├─ 递归限制 → 换用str_replace等原子操作方式，继续执行
    ├─ 编译错误 → 修复后继续
    └─ 其他错误 → 调整策略后继续
    ↓
创建新子任务继续（prompt包含失败原因+已完成进度+调整策略）
    ↓
自动循环直到完成，无重试次数上限
```

**核心原则：全自动完成，用户不监控、不干预执行过程。**
- 禁止因失败而暂停等待用户决策
- 禁止向用户报告失败后等待指令
- 主会话负责分析失败原因并调整策略，自动创建新子任务继续
- 过大的失败任务必须进一步拆分，而非原样重试

## 四、版本封版标准（缺一不可）

### 4.1 代码标准
- [ ] `npx tsc --noEmit` 零错误
- [ ] 引擎层代码严格按PRD数值表实现
- [ ] UI层代码严格按UI布局文档实现
- [ ] 无废弃/注释掉的代码块

### 4.2 测试标准
- [ ] 引擎逻辑层分支覆盖率 ≥ 100%
- [ ] 引擎逻辑层测试通过率 = 100%
- [ ] UI交互分支覆盖率 ≥ 100%
- [ ] UI交互测试通过率 = 100%

### 4.3 评测标准
- [ ] 10维度评分每项 ≥ 9.8分
- [ ] 总分 ≥ 9.8分
- [ ] 评测师明确给出"通过"结论

### 4.4 提交标准
- [ ] git commit 消息简述完成事项，格式：`feat(three-kingdoms): v{版本号} {具体完成内容描述}`
- [ ] 禁止使用"封版"等无意义词汇，必须描述实际完成的功能
- [ ] 推送到 GitHub main 分支

## 五、评测-改进循环

```
评测不通过（任一维度<9.8）
    ↓
读取评测报告中的改进建议
    ↓
按优先级修复（P0→P1→P2）
    ↓
重新运行测试确认通过
    ↓
重新提交评测（同一版本，R2/R3/R4...）
    ↓
直到评分>9.8 → 封版
```

**无最大循环次数限制**，直到通过为止。

## 六、开发任务描述规范

每个开发子任务的prompt**必须包含以下4项**：

```
### 任务计划文档
PLAN: `docs/games/three-kingdoms/plans/v{X.0}-{名称}.md`

### 功能点
- 简述本子任务要完成的具体功能（引用功能点编号或名称）
- 明确涉及的文件路径和模块范围

### 上下文信息
- 当前已完成的进度（哪些文件已存在、哪些功能已实现）
- 前置依赖的状态（如：资源系统已完成，建筑系统待追加）
- 如果是失败重试：说明失败原因和已完成的部分

### 验收标准
- 编译零错误（`npx tsc --noEmit`）
- 测试通过条件（覆盖率目标、用例数等）
- 代码量限制（≤ 500行）
- 其他具体要求
```

**注意事项**：
- PLAN文件中已包含PRD/UI章节链接，不需要在任务中重复列出PRD/UI路径
- 禁止在任务描述中转述开发内容，开发者应自行读取PLAN和PRD文档
- 禁止在任务描述中粘贴大段代码模板，开发者应根据PRD自行设计实现

## 七、文件操作规则

### 7.1 大型文件修改策略

- `ThreeKingdomsPixiGame.tsx` 可能增长到数千行
- 使用 `str_replace` 做精确原子修改，每次只改一处
- 禁止在子任务中读取整个6000+行文件（会触发递归限制）
- 子任务中用 `grep` 定位行号，再用 `read_file` 按行号范围读取

### 7.2 旧代码处理

- v1.0 开发前，旧代码移到 `bak/` 目录
- 旧文件：`ThreeKingdomsEngine.ts` → `bak/ThreeKingdomsEngine.old.ts`
- 旧文件：`ThreeKingdomsPixiGame.tsx` → `bak/ThreeKingdomsPixiGame.old.tsx`
- 新文件从空文件开始，逐步堆叠
