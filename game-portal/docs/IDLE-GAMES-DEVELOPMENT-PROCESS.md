> **版本**: v3.0  
> **更新日期**: 2026-04-15  
> **适用范围**: Game Portal 放置游戏专区 100 款游戏开发

## 一、开发流水线总览

每款游戏按 **五阶段流水线** 开发，每个阶段有明确的质量门禁（Quality Gate）：

```mermaid
graph LR
    A[📋 策划设计] -->|玩法文档| B[🔍 评审把关]
    B -->|通过/打回| A
    B -->|评审通过| C[🎨 美术设计]
    C -->|视觉规范| D[💻 开发实现]
    D -->|引擎+测试| E[🧪 测试验收]
    E -->|不通过| D
    E -->|通过| F[✅ 交付完成]
```

### 1.1 五阶段流水线

| 阶段 | Agent | 职责 | 产出物 | 质量门禁 |
|------|-------|------|--------|----------|
| 📋 **策划设计** | `game-designer` | 游戏玩法文档（GDD）、数值设计、子系统设计、科技树 | `docs/gdd/GDD-{game-id}.md` | 多子系统+多发展路线+科技树+原创性 |
| 🔍 **评审把关** | `architect` | 评审玩法是否达标，不同游戏需有原创性 | 评审意见（通过/打回修改） | 评审通过才能进入下一阶段 |
| 🎨 **美术设计** | `game-artist` | 视觉风格、配色、UI/UX、动画参数、人物/建筑/地图设定 | `docs/art/ART-{game-id}.md` | 风格与题材匹配 |
| 💻 **开发实现** | `game-developer` | 引擎实现、Canvas渲染、状态管理 | `XxxEngine.ts` + `constants.ts` + `index.ts` | TypeScript 零编译错误 |
| 🧪 **测试验收** | `tester` | 覆盖所有场景+资源/对象+事件的测试 | `XxxEngine.test.ts` | 120+ 用例全通过 |

### 1.2 流水线执行时序

```
game-designer  ──→  architect    ──→  game-artist  ──→  game-developer  ──→  tester
  (策划设计)         (评审把关)       (美术设计)       (开发实现)         (测试验收)
      │                │                │                │                  │
      ▼                ▼                ▼                ▼                  ▼
  GDD文档           评审意见         美术文档          游戏引擎            测试用例
  docs/gdd/         通过/打回        docs/art/         XxxEngine.ts       120+用例
  GDD-{id}.md                       ART-{id}.md       constants.ts       全部通过
```

## 二、阶段一：策划设计（game-designer）

### 2.1 策划设计要求

策划设计是整个流水线的**源头**，决定了游戏的质量上限。每款游戏的策划文档必须包含以下内容：

#### 2.1.1 游戏概念文档（GDD）

| 章节 | 内容要求 | 最低标准 |
|------|----------|----------|
| **游戏概述** | 名称、题材、核心概念、目标玩家 | 200字以上 |
| **核心玩法** | 主要玩法机制、操作方式、游戏循环 | 500字以上 |
| **资源体系** | 资源类型、获取方式、消耗途径、平衡关系 | 至少3种资源 |
| **建筑/升级系统** | 建筑列表、费用曲线、效果、前置依赖 | 至少8种建筑 |
| **声望/转生系统** | 转生条件、永久加成、保留内容 | 完整的转生循环 |
| **成就系统** | 成就列表、解锁条件、奖励 | 至少15个成就 |
| **事件系统** | 随机/定期/触发事件列表 | 至少10种事件 |
| **数值平衡表** | 前期/中期/后期/大后期数值曲线 | 四阶段平衡 |

#### 2.1.2 多子系统要求（核心）

**每款游戏必须包含至少 3 个独立子系统**，且子系统之间有交互关系：

| 子系统类型 | 描述 | 示例 |
|------------|------|------|
| 🏗️ **建筑/生产系统** | 建造建筑、产出资源 | 农田产粮食、兵营产士兵 |
| 👥 **雇佣/角色系统** | 招募角色、分配工作、培养升级 | 雇佣农民种田、招募将军打仗 |
| 🗺️ **地图/探索系统** | 地图浏览、区域解锁、探索发现 | 开拓新领土、发现隐藏资源 |
| ⚔️ **战斗/防御系统** | 敌人波次、防御部署、战斗策略 | 抵御外敌入侵、讨伐山贼 |
| 📚 **科技/研究系统** | 科技树、研究路线、技术解锁 | 冶铁技术→铁器时代 |
| 🎭 **事件/故事系统** | 剧情推进、随机事件、NPC交互 | 历史事件触发、商人来访 |
| 🏆 **成就/收集系统** | 里程碑成就、收集图鉴、排行榜 | 百人村成就、英雄图鉴 |
| 🔄 **声望/转生系统** | 转生重置、永久加成、新内容解锁 | 涅槃重生获得永久加成 |

**子系统交互示例**：
```
建筑系统 ←→ 雇佣系统：建筑需要雇员才能运转
雇佣系统 ←→ 战斗系统：将军统帅军队，士兵提供战力
科技系统 ←→ 建筑系统：解锁科技才能建造高级建筑
地图系统 ←→ 战斗系统：不同区域有不同敌人
事件系统 ←→ 雇佣系统：特殊事件可招募稀有英雄
```

#### 2.1.3 多发展路线要求（核心）

**每款游戏必须提供至少 2 条可选择的差异化发展路线**：

| 发展路线类型 | 描述 | 示例 |
|--------------|------|------|
| **经济路线 vs 军事路线** | 专注发展经济或军事力量 | 农商强国 vs 军事扩张 |
| **科技路线 vs 传统路线** | 科技驱动或传统积累 | 工业革命 vs 农耕文明 |
| **和平路线 vs 征服路线** | 外交合作或武力征服 | 联盟贸易 vs 征战四方 |
| **单点突破 vs 全面发展** | 专注一个方向或均衡发展 | 军事强国 vs 均衡帝国 |
| **快速扩张 vs 稳健经营** | 快速占领或稳步积累 | 闪电战 vs 龟缩发展 |

**路线差异必须体现在**：
- 不同的建筑解锁顺序
- 不同的资源优先级
- 不同的英雄/角色搭配
- 不同的战斗策略
- 不同的声望加成方向

#### 2.1.4 玩法维度要求

每款游戏应覆盖以下玩法维度中的 **至少 3 个**：

| 维度 | 描述 | 设计要点 |
|------|------|----------|
| 📜 **历史进程** | 时代/朝代/纪元演进 | 每个时代解锁新内容，有明确的历史叙事 |
| ☕ **休闲放置** | 离线收益、自动生产 | 玩家不在线也能获得进度，回归时有奖励 |
| ⚔️ **战斗对抗** | 敌人入侵、防御战斗 | 不同发展阶段有不同敌人，需要策略应对 |
| 🔍 **探索发现** | 地图探索、隐藏内容 | 未知区域、随机发现、隐藏宝藏 |
| 🌱 **养成培养** | 角色/英雄培养、属性提升 | 有名字的角色，可升级的天赋和技能 |

#### 2.1.5 原创性要求

**不同游戏必须有本质区别**，禁止"换皮"设计：

| 检查项 | 标准 | 不合格示例 |
|--------|------|------------|
| **核心机制** | 每款游戏有不同的核心玩法 | 所有游戏都是"点击→买建筑→等收益" |
| **资源体系** | 资源类型和用途有主题差异 | 所有游戏都是"资源A/B/C"换名字 |
| **发展路径** | 有独特的发展路线选择 | 所有游戏发展路径完全相同 |
| **子系统组合** | 子系统类型和交互方式不同 | 所有游戏子系统完全一样 |
| **视觉风格** | 每款游戏有独特的视觉呈现 | 所有游戏使用相同的 UI 布局和配色 |

## 三、阶段二：评审把关（architect）

### 3.1 评审标准

策划文档完成后，由架构师（architect）进行评审。评审维度：

| 评审维度 | 权重 | 评分标准 |
|----------|------|----------|
| **原创性** | 30% | 与已开发游戏的差异化程度 |
| **系统深度** | 25% | 子系统数量、交互复杂度、发展路线数量 |
| **玩法维度** | 20% | 覆盖历史/休闲/战斗/探索/养成中的几个 |
| **数值合理性** | 15% | 增长曲线、平衡性、节奏感 |
| **可实现性** | 10% | 技术可行性、开发工作量 |

### 3.2 评审结果

| 结果 | 处理方式 |
|------|----------|
| ✅ **通过** | 进入美术设计阶段 |
| ⚠️ **有条件通过** | 修改指定问题后直接进入下一阶段 |
| ❌ **打回修改** | 退回策划阶段，说明修改意见，重新设计 |

### 3.3 评审检查清单

```
□ 核心玩法是否有别于已开发的同系列游戏？
□ 是否包含至少 3 个独立子系统？
□ 是否提供至少 2 条差异化发展路线？
□ 是否覆盖至少 3 个玩法维度（历史/休闲/战斗/探索/养成）？
□ 资源体系是否与游戏主题紧密关联？
□ 数值增长曲线是否合理（前期不无聊、中期有挑战、后期有目标）？
□ 声望/转生系统是否有意义（不是简单重置）？
□ 成就系统是否与游戏主题相关？
□ 事件系统是否有故事性和策略性？
□ 技术实现是否可行？
```

## 四、阶段三：美术设计（game-artist）

### 4.1 美术设计要求

| 设计项 | 要求 | 产出 |
|--------|------|------|
| **视觉风格** | 与游戏题材匹配（8种风格选一） | 风格定义文档 |
| **配色方案** | 主色/辅色/强调色/背景色 HEX值 | COLORS 常量 |
| **UI布局** | PC横向 + 移动端自适应布局 | 布局规范 |
| **动画参数** | 帧率/粒子/动画时长 | 动画规范 |
| **分辨率** | 逻辑分辨率 1920×1080，支持高清适配 | 分辨率规范 |

### 4.2 分辨率与画质规范

| 画质等级 | 逻辑分辨率 | devicePixelRatio | 粒子数量 | 阴影 | 适用设备 |
|----------|------------|------------------|----------|------|----------|
| 低 | 1280×720 | 1x | 0 | 无 | 低端手机 |
| 中 | 1920×1080 | 1.5x | 少量 | 简单 | 中端设备 |
| 高 | 1920×1080 | 2x | 中等 | 标准 | 高端设备/PC |
| 极致 | 2560×1440 | 2x+ | 丰富 | 精细 | 4K显示器 |

**坚决不使用 480×640 等低分辨率。** Canvas 逻辑坐标统一为 1920×1080（PC横屏）或 1080×1920（移动竖屏），通过 devicePixelRatio 适配高清屏。

## 五、阶段四：开发实现（game-developer）

### 5.1 开发规范

| 规范项 | 要求 |
|--------|------|
| **引擎继承** | 继承 IdleGameEngine 基类 |
| **TypeScript** | 零编译错误 |
| **Canvas 渲染** | 支持 devicePixelRatio 高清适配 |
| **存档系统** | 自动存档 + 手动存档 + 导入导出 |
| **离线收益** | 离线期间资源自动积累 |
| **交互操作** | 鼠标 + 键盘 + 触控三端支持 |
| **响应式** | PC横屏 + 移动竖屏/横屏自适应 |

### 5.2 引擎文件结构

```
src/games/{game-id}/
├── constants.ts          — 常量定义（资源/建筑/事件/成就/配色）
├── XxxEngine.ts          — 游戏引擎（继承 IdleGameEngine）
├── index.ts              — 模块导出
└── __tests__/
    └── XxxEngine.test.ts — 测试文件（120+ 用例）
```

### 5.3 引擎必须实现的方法

```typescript
class XxxEngine extends IdleGameEngine {
  // 构造函数：Object.defineProperty 处理 statistics
  constructor() { ... }
  
  // 生命周期
  protected onInit(): void { super.onInit(); initializeResources + initializeUpgrades; }
  protected onUpdate(deltaTime: number): void { ... }
```

### 9.1 game-designer（游戏策划）

### 9.4 game-developer（游戏主程序）

## 十、常见问题与经验教训

### 10.1 引擎开发常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `prestige()` 方法名冲突 | 基类有 `prestige` 字段 | 用 `doPrestige()` |
| `statistics` getter 被覆盖 | 基类实例字段遮蔽 | `Object.defineProperty` 在构造函数中 |
| Canvas 渲染报错 | mock 缺少方法 | mock 必须包含 ellipse 等 |
| `start()` 抛 "Canvas not initialized" | 未传入 canvas | `engine.init(createCanvas())` |
| 测试 API 不匹配 | 测试未读引擎源码 | **先读引擎再写测试** |

### 10.2 质量常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 游戏同质化 | 缺少策划评审 | 增加评审环节 |
| 数值堆砌 | 缺少多子系统 | 要求至少3个子系统 |
| 单一发展路线 | 缺少路线设计 | 要求至少2条差异化路线 |
| 测试不充分 | 只测基本功能 | 要求覆盖所有场景/资源/事件 |
