# 放置游戏战斗子系统 — 审查分析报告

> **文档版本**: v1.0 | **日期**: 2026-04-15
> **审查范围**: `src/engines/idle/modules/BattleSystem.ts` (297行) + 测试 (746行, 56用例)
> **审查人**: 架构师 Agent

---

## 一、现有系统概览

### 1.1 模块定位

BattleSystem 是 P2 特殊层模块，专为战争/策略/神话类放置游戏设计。当前提供**波次制战斗**功能，包含敌人实例化、伤害计算、Buff 管理、概率掉落、事件通知和存档支持。

### 1.2 代码规模

| 指标 | 数值 |
|------|------|
| 源码行数 | 297 行 |
| 类型定义 | 7 个（interface） |
| 公开方法 | 23 个 |
| 测试用例 | 56 个（746 行） |
| 依赖 | 零外部依赖 |

### 1.3 核心类型

```typescript
EnemyDef      // 敌人静态定义（hp/attack/defense/drops/abilities/isBoss）
BattleDef     // 波次定义（enemies[]/rewards/timeLimit/nextWave/tags）
BattleBuff    // Buff/Debuff（stat/value/remainingMs）
BattleEnemy   // 运行时实例（currentHp/maxHp/isAlive/buffs[]）
BattleStats   // 累计统计（totalDamageDealt/Taken/wavesCleared/bossesDefeated）
BattleState   // 持久化状态
BattleEvent   // 7种事件类型
```

### 1.4 核心流程

```
startWave → [attack/enemyAttack/update循环] → checkWin → settleWave
```

### 1.5 伤害公式

| 方向 | 公式 | 说明 |
|------|------|------|
| 玩家→敌人 | `max(1, damage)` | 直接传入，最低1 |
| 敌人→玩家 | `max(1, attack - defense/2)` | 防御减半，保底1 |

## 二、审查发现

### 2.1 ✅ 优点

| # | 优点 | 说明 |
|---|------|------|
| 1 | 零依赖设计 | 纯 TypeScript，无框架耦合，可独立测试 |
| 2 | 泛型扩展 | `BattleSystem<Def>` 允许游戏自定义波次定义 |
| 3 | 事件驱动 | 7 种事件覆盖完整战斗生命周期 |
| 4 | 完整存档 | saveState/loadState 支持序列化/反序列化 |
| 5 | 概率掉落 | 独立掷骰机制，支持稀有物品 |
| 6 | 状态隔离 | getCurrentState() 返回深拷贝 |
| 7 | 测试充分 | 56 个用例覆盖所有公开方法 |

### 2.2 ⚠️ 不足与风险

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| 1 | **战斗模式单一** | 🔴 高 | 仅支持波次制，无法满足多种游戏类型需求 |
| 2 | **无地图/位置系统** | 🔴 高 | 无法支持自由移动、阵型、地形等 |
| 3 | **伤害公式过于简单** | 🟡 中 | 缺少暴击、闪避、元素克制、连击等机制 |
| 4 | **Buff 仅限敌人侧** | 🟡 中 | 玩家侧 Buff/技能效果需在引擎层自行实现 |
| 5 | **无 AoE/范围攻击** | 🟡 中 | 不支持群体伤害、溅射、穿透 |
| 6 | **无仇恨/目标选择** | 🟡 中 | 无法实现前排/后排、嘲讽、集火等策略 |
| 7 | **无镜头系统** | 🟡 中 | 不支持镜头移动、缩放、特写 |
| 8 | **无战斗回放/日志** | 🟢 低 | 缺少战斗过程记录和回放功能 |
| 9 | **无波次自动衔接** | 🟢 低 | 需手动调用 startWave(nextWave) |
| 10 | **无难度动态调整** | 🟢 低 | 敌人强度固定，无法根据玩家实力调整 |

### 2.3 架构瓶颈

当前 BattleSystem 本质上是一个**回合制波次战斗管理器**，其架构假设：

1. **敌人是静态列表** — 一波敌人全部出现，逐个击杀
2. **战斗是线性的** — 开始→攻击→结算，无中间状态
3. **无空间概念** — 敌人无位置，无移动，无阵型
4. **无时间概念** — 无回合/帧/实时区分
5. **无策略预设** — 玩家无法在战斗前设置战术

这些假设严重限制了战斗系统的适用范围，无法支撑用户要求的多种战斗模式。

## 三、用户需求分析

### 3.1 要求支持的战斗模式

| 模式 | 参考游戏 | 核心特征 | 放置游戏适配 |
|------|---------|---------|-------------|
| **回合制** | 最终幻想、博德之门 | 回合序列、行动顺序、技能选择 | 预设策略，自动执行回合 |
| **半回合制** | 仙剑奇侠传、轩辕剑 | ATB时间条、实时但可暂停 | ATB自动填充，到达即执行预设技能 |
| **地图自由战斗** | 魔兽争霸3、红色警戒 | 实时移动、多单位、资源点 | 自动移动攻击，玩家预设编队和优先级 |
| **攻城战** | 三国志系列 | 城墙/城门/箭塔、攻城器械、士气 | 预设攻城方案，自动执行攻防 |
| **战棋类** | 英雄无敌系列 | 六角/方格地图、移动范围、相遇战斗 | 预设行军路线，自动移动遇敌战斗 |
| **即时对战** | 格斗游戏、MOBA | 实时操作、连招、闪避 | 预设技能释放顺序和时机 |
| **塔防战斗** | 植物大战僵尸、Kingdom Rush | 防御塔放置、敌人路径、波次 | 自动放置塔，玩家预设塔布局 |
| **海战/空战** | 大航海时代、皇牌空战 | 舰船/飞机编队、特殊地形 | 预设编队阵型和攻击策略 |

### 3.2 通用需求

- **镜头移动**：平移、缩放、跟随特定单位
- **武将技能特写**：技能释放时镜头拉近、特效展示
- **策略预设**：玩家在战斗前设置阵型、技能优先级、目标选择策略
- **自动执行**：所有战斗自动进行，体现放置游戏特色
- **效果还原**：战斗效果与原始游戏/主题保持一致或更优秀

## 四、现有系统与需求的差距

```
                    现有能力                    需求能力
                    ────────                    ────────
战斗模式:          波次制仅1种        →        6-8种战斗模式
空间系统:          无                 →        地图/网格/自由移动
单位系统:          仅敌人实例          →        双方多单位+属性
伤害系统:          简单减法公式        →        暴击/闪避/克制/连击
技能系统:          无                 →        主动/被动/终极技能
Buff系统:          仅敌人侧+倒计时    →        双方+多层+叠加规则
镜头系统:          无                 →        平移/缩放/特写/跟随
策略系统:          无                 →        预设阵型/技能顺序/优先级
AI系统:            无                 →        自动执行预设策略
视觉系统:          无                 →        特效/动画/粒子/特写
回放系统:          无                 →        战斗日志+回放
```

**结论**：现有 BattleSystem 需要进行**根本性重构**，从单一波次管理器升级为**多模式战斗引擎框架**。

---

## 五、重构建议概要

### 5.1 架构方向

采用**策略模式 + 组合模式**，将战斗系统拆分为：

```
BattleEngine (战斗引擎框架)
├── BattleMode (战斗模式策略)
│   ├── TurnBasedMode      回合制
│   ├── SemiTurnBasedMode  半回合制(ATB)
│   ├── FreeRoamMode       地图自由战斗
│   ├── SiegeMode          攻城战
│   ├── TacticalMode       战棋类
│   └── TowerDefenseMode   塔防
├── BattleMap (战斗地图)
│   ├── GridMap            方格地图
│   ├── HexMap             六角地图
│   └── FreeMap            自由地图
├── BattleCamera (镜头系统)
│   ├── 平移/缩放/跟随
│   └── 特写镜头
├── BattleUnit (战斗单位)
│   ├── 属性系统
│   ├── 技能系统
│   └── AI行为树
├── BattleEffect (战斗特效)
│   ├── 粒子效果
│   ├── 技能动画
│   └── 飘字/震屏
└── BattleReplay (战斗回放)
    ├── 战斗日志
    └── 回放引擎
```

### 5.2 放置游戏适配原则

1. **策略预设**：玩家在战斗前设置阵型、技能顺序、目标优先级
2. **自动执行**：战斗开始后全自动，无需玩家操作
3. **可加速**：支持 1x/2x/4x/8x 速度
4. **可跳过**：快速结算，直接显示结果
5. **可回看**：保存战斗日志，支持回放

### 5.3 优先级建议

| 阶段 | 内容 | 预计代码量 | 适用游戏 |
|------|------|-----------|---------|
| Phase 1 | 回合制 + 半回合制 + 技能系统 | ~1500行 | 最终幻想、博德之门、仙剑 |
| Phase 2 | 战棋地图 + 六角网格 + 移动AI | ~1200行 | 英雄无敌 |
| Phase 3 | 攻城战 + 城墙/器械/士气 | ~1000行 | 三国志 |
| Phase 4 | 自由战斗 + 实时移动 + 编队 | ~1500行 | 魔兽争霸3、红色警戒 |
| Phase 5 | 镜头系统 + 特写 + 特效 | ~800行 | 所有游戏 |
| Phase 6 | 塔防模式 + 海战/空战 | ~1000行 | 植物大战僵尸、大航海 |

---

## 六、结论

当前 BattleSystem 作为 P2 子系统完成了基础的波次战斗功能，代码质量良好、测试充分。但其架构**无法扩展**到用户要求的多种战斗模式。

建议保持现有 BattleSystem 作为**向后兼容**的波次战斗模块，同时新建 **BattleEngine** 战斗引擎框架，支持多种战斗模式。两者可以共存，游戏根据需要选择使用。

详细的改进方案和技术设计请参见配套文档 `BATTLE-SYSTEM-IMPROVEMENT.md`。
