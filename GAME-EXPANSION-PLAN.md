## 一、总览

本计划将 20 款候选小游戏按 **复杂度 + 类型** 分为 4 个批次，每批次 5 款游戏，逐批迭代交付。每批次预计 1–2 周开发周期。

```
v1.0 ████████░░░░░░░░░░░░ 已完成 (3 款)
v2.0 ████████████░░░░░░░░ 第一批 (5 款) — 入门级 + 益智
v3.0 ████████████████░░░░ 第二批 (5 款) — 反应 + 策略
v4.0 ████████████████████░ 第三批 (5 款) — 街机 + 棋类
v5.0 █████████████████████ 第四批 (5 款) — 进阶 + 对战
```

**最终目标**: 23 款游戏（3 已有 + 20 新增），覆盖益智、反应、策略、街机、模拟、对战六大类型。

## 三、分批开发计划

### Batch 1 — v2.0（入门 + 益智）

**目标**: 快速扩充游戏库，以低复杂度、高辨识度游戏为主，验证扩展架构的可行性。

| 序号 | 游戏 | 复杂度 | 预计工期 | 核心实现要点 |
|------|------|--------|----------|-------------|
| B1-1 | **Flappy Bird** | ⭐ | 1 天 | 重力模拟、管道生成、碰撞检测、点击跳跃 |
| B1-2 | **2048** | ⭐⭐ | 1.5 天 | 4×4 Grid、滑动合并逻辑、动画过渡、分数计算 |
| B1-3 | **记忆翻牌** | ⭐ | 1 天 | 卡片翻转动画、配对逻辑、计时器、难度分级 |
| B1-4 | **井字棋** | ⭐ | 1 天 | 3×3 棋盘、Minimax AI、胜负判定 |
| B1-5 | **生命游戏** | ⭐ | 1 天 | 元胞自动机规则、Canvas 网格渲染、播放/暂停/步进 |

**交付标准**:
- 每款游戏独立 Engine 类，继承 `GameEngine` 抽象基类
- 完整的 Vitest 测试套件（每款 ≥ 15 用例）
- Game Portal 首页 GameCard 展示 + 路由接入
- 构建零错误、全量测试通过

**架构任务**（首次扩展前置）:

## 四、技术架构规划

### 4.1 引擎架构（GameEngine 基类扩展）

```
GameEngine (Abstract Base)
├── Core Lifecycle
│   ├── init()          — 初始化游戏状态
│   ├── start()         — 启动游戏循环
│   ├── pause()         — 暂停
│   ├── resume()        — 恢复
│   ├── reset()         — 重置
│   └── destroy()       — 销毁释放资源
├── Game Loop
│   ├── update(dt)      — 逻辑帧更新
│   └── render(ctx)     — Canvas 渲染
├── Input Handling
│   ├── handleKeyDown()
│   ├── handleKeyUp()
│   └── handleClick()
├── State Management
│   ├── getState()      — 获取序列化状态
│   └── loadState()     — 恢复状态（存档/读档）
└── Events (EventEmitter)
    ├── stateChange     — 状态变更
    ├── scoreChange     — 分数变更
    ├── levelChange     — 等级变更
    └── gameover        — 游戏结束
```

### 4.2 游戏注册机制（GameRegistry）

```typescript
// 游戏注册 Schema
interface GameRegistration {
  id: string;                    // 唯一标识，如 'minesweeper'
  name: string;                  // 显示名称
  nameEn: string;                // 英文名
  icon: string;                  // 图标（emoji 或 SVG）
  category: GameCategory;        // 分类
  difficulty: 1 | 2 | 3 | 4;    // 复杂度
  controls: ControlScheme;       // 操作方式
  engine: new () => GameEngine;  // Engine 构造函数
  thumbnail?: string;            // 缩略图
  description: string;           // 简介
  tags: string[];                // 标签
}

type GameCategory = 'puzzle' | 'arcade' | 'strategy' | 'reaction' | 'simulation' | 'versus';

// 注册示例
GameRegistry.register({
  id: 'flappy-bird',
  name: 'Flappy Bird',
  nameEn: 'Flappy Bird',
  icon: '🐦',
  category: 'arcade',
  difficulty: 1,
  controls: { type: 'click', description: '点击/空格跳跃' },
  engine: FlappyBirdEngine,
  description: '点击控制小鸟穿越管道间隙',
  tags: ['反应', '街机', '休闲']
});
```

## 五、开发流程规范

### 5.1 单款游戏开发 SOP

```
1. Engine 开发
   ├── 创建引擎目录 src/engines/<game-id>/
   ├── 实现 <Name>Engine extends GameEngine
   ├── 完成 7 个抽象方法 + 游戏特有逻辑
   └── 导出至 src/engines/index.ts

2. 测试编写
   ├── 创建 src/__tests__/engines/<game-id>.test.ts
   ├── 基础测试（初始化/状态/重置）≥ 5 用例
   ├── 核心逻辑测试 ≥ 10 用例
   └── 边界/异常测试 ≥ 5 用例

3. 注册接入
   ├── GameRegistry.register({...})
   ├── 首页自动展示（无需改路由）
   └── GamePage 动态加载 Engine

4. 集成验证
   ├── 全量 npm test 通过
   ├── npm run build 零错误
   └── Docker 构建验证
```

## 七、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| Engine 基类抽象不足 | 新游戏需大量 hack | Batch 1 前先重构基类，建立 GameRegistry |
| Canvas 性能瓶颈（弹球台物理） | 帧率下降 | 使用 requestAnimationFrame + 空间分区优化碰撞 |
| AI 算法复杂度（棋类） | 开发周期延长 | 先实现简单 AI，后续迭代升级 |
| 测试用例膨胀 | 维护成本高 | 按引擎目录隔离，CI 按批次运行 |
| 移动端适配 | 操作体验差 | 统一 GameControls 组件，自动检测触屏 |
