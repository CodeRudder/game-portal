## LL-R28-001: 三模块9K行架构的集成测试策略——5文件140用例的精炼覆盖

**分类**: 测试策略

**场景**: v6.0天下大势涵盖calendar/event/npc三大模块共9,093行代码。集成测试仅用5个文件140个用例即实现100%通过，覆盖: era-trend(23)、npc-affinity(32)、event-system(30)、territory-map(31)、cross-validation(24)。

**根因**: 测试设计采用"维度正交"策略——每个测试文件聚焦一个独立维度(时代趋势/NPC好感/事件系统/领地地图/交叉验证)，避免了用例间的重复覆盖。cross-validation文件专门验证模块间交互，确保不遗漏跨系统边界。

**教训**: 集成测试文件数不随源码行数线性增长。9K行代码5个集成测试文件即可充分覆盖，关键在于: 1) 每个文件对应一个正交维度；2) 交叉验证文件覆盖模块间交互；3) 单文件用例数控制在23-32之间，保持可读性。

**推广**: 集成测试文件数估算公式: `ceil(核心模块数 × 0.7) + 1`。v6.0有3个核心模块 → 3×0.7+1 ≈ 3个模块测试 + 1-2个交叉测试 = 4-5个文件。实际5个文件，符合预期。

## LL-R28-002: 4个400+行文件的模块分布与控制策略

**分类**: 架构设计

**场景**: v6.0有4个文件超过400行: OfflineEventSystem(451)、NPCMapPlacer(449)、CalendarSystem(423)、EventChainSystem(403)。分布在event(2个)和npc/calendar(各1个)模块中。

**根因**: 天下大势系统天然需要处理复杂的状态转换——日历推进触发事件链、NPC在地图上移动产生交互。OfflineEventSystem 451行包含离线期间事件积累与回放的完整逻辑；NPCMapPlacer 449行处理NPC在领地地图上的智能放置算法。两者逻辑高度内聚，拆分会破坏算法完整性。

**教训**: 跨模块的400+行文件分布均匀(每模块1-2个)是健康信号，说明没有单一模块过度膨胀。最大文件451行，远低于500行硬上限。应继续监控但无需立即拆分。

**推广**: 400+行文件健康度指标: 1) 最大文件 < 500行；2) 400+行文件数 ≤ 模块数；3) 分布均匀(无单一模块占3个以上)。v6.0三项均满足。

## LL-R28-003: 编译时间从31.58s到20.55s的回归改善

**分类**: 构建性能

**场景**: Phase 1首次编译31.58s，Phase 4回归编译降至20.55s，改善35%。集成测试也从首次运行(未记录时间)到回归3.82s完成140用例。

**根因**: Vite增量编译缓存生效。首次编译需要完整解析所有TypeScript文件和依赖关系，后续编译利用缓存仅处理变更部分。20.55s的构建时间在20+源文件规模下属于正常范围。

**教训**: 构建时间应记录首次和回归两个值。首次编译反映冷启动性能，回归编译反映增量构建效率。两者差距过大(>50%)说明缓存命中率高，是正常的；差距过小(<10%)说明缓存失效，需排查。

**推广**: 构建性能基线: 1) 冷启动: 30-35s可接受；2) 增量构建: 18-22s正常；3) 测试运行: 140用例 < 5s为优秀。超出基线时优先检查: tree-shaking配置、动态导入、chunk拆分。

## ER-R28-001: 配置值偏差作为中等问题而非P0的处理策略

**分类**: 缺陷管理

**场景**: Phase 1-3发现2个中等问题(配置值偏差)和3个轻微问题。配置值偏差指某些系统参数(如好感度阈值、事件触发概率)与设计文档存在微小偏差，但不影响核心游戏逻辑正确性。

**根因**: 配置值偏差通常源于实现时的默认值选择与最终设计文档不同步。这类问题不影响代码逻辑正确性，仅需调整配置常量即可修复。将其定为"中等"而非P0，避免了阻塞封版流程。

**教训**: 缺陷分级应区分"逻辑错误"和"配置偏差"。逻辑错误(P0/P1)需要代码修改和回归测试；配置偏差(中等)仅需调整常量值，风险极低。两者混为一谈会导致封版流程不必要的延迟。

**推广**: 缺陷分级补充规则: 1) P0: 逻辑错误导致功能不可用；2) P1: 逻辑错误影响非核心功能；3) 中等: 配置值偏差，功能正确但参数需微调；4) 轻微: 代码风格、注释等非功能性问题。

## ER-R28-002: 2个中等+3个轻微问题的零P0封版

**分类**: 质量管理

**场景**: R28 v6.0以0个P0/P1问题封版，仅遗留2个中等+3个轻微问题。这是连续多轮保持零P0封版，表明v6.0天下大势引擎的质量稳定性持续提升。

**根因**: 得益于前序轮次(R25-R27)建立的质量基础设施: 严格的TypeScript类型检查、零as any策略、集成测试全覆盖。这些前置措施使得新模块从设计阶段就避免了P0级别的逻辑错误。

**教训**: 零P0封版不是偶然，而是质量基础设施的必然结果。每轮建立的质量措施(类型安全、测试覆盖、架构审查)都在为后续轮次降低缺陷密度。v6.0的5个遗留问题全部为非逻辑性问题，验证了这一策略。

**推广**: 零P0封版路径: 1) strict TypeScript + 零as any → 消除类型错误；2) 集成测试全覆盖 → 暴露逻辑缺陷；3) 架构审查(400行/模块数) → 控制复杂度；4) 回归验证 → 确保修复不引入新问题。四步闭环确保封版质量。
