## LL-R24-001: 拆分后必须立即编译验证，避免错误累积到封版阶段

**分类**: 质量保证

**场景**: R24 Phase 4完成了18个400+行文件的拆分，Phase 1-2报告"编译0错误"。但Phase 5实际编译验证时发现17个TS错误，分布在4个文件中（QuestSystem.helpers.ts、QuestSystem.ts、EventTriggerSystem.ts、FusionTechSystem.links.ts、responsive.types.ts）。

**根因**: 拆分过程中函数重命名（如`updateProgressByType`→`updateProgressByTypeLogic`）但调用处未同步更新；重复import未清理；类型签名变更后回调参数丢失类型注解。Phase 1-2的冒烟检查可能只运行了部分测试而非完整`vite build`。

**教训**: 每完成一批文件拆分后，必须立即执行`pnpm run build`验证编译通过，而非依赖Phase 2的冒烟报告。拆分操作的风险点: 1) 函数重命名后调用处遗漏；2) 移动文件后import路径错误；3) 类型导出重复定义。每个风险点都应在拆分后立即验证。

**推广**: 拆分SOP增加"拆后即检"步骤: 拆分→编译→修复→再编译→确认0错误→下一批。禁止累积超过5个文件的拆分后不编译。

## LL-R24-002: 函数重命名应采用"导出别名"过渡策略，避免全局搜索替换遗漏

**分类**: 重构方法

**场景**: EventTriggerSystem.helpers.ts将`checkAndTriggerEvents`重命名为`checkAndTriggerEventsLogic`，QuestSystem.helpers.ts将`updateProgressByType`重命名为`updateProgressByTypeLogic`。调用处QuestSystem.ts和EventTriggerSystem.ts仍使用旧名，导致TS2724错误。

**根因**: 重命名时只改了定义处，未同步更新所有import和调用处。TypeScript的`Did you mean 'xxxLogic'?`提示虽然友好，但在非IDE环境中容易被忽略。

**教训**: 函数重命名应采用两步过渡: 1) 先在原位置添加导出别名 `export { newFunc as oldFunc }`；2) 逐步更新调用处后再移除别名。或者使用IDE的Rename Symbol功能确保全局替换。手动重命名必须配合`grep -rn oldName`确认无遗漏。

**推广**: 所有函数重命名必须在helpers文件底部添加别名导出: `export { newFuncName as oldFuncName }`，待所有调用方更新后再移除。这作为拆分checklist的必选项。

## LL-R24-003: 重复import是文件拆分的高频缺陷，应自动化检测

**分类**: 自动化

**场景**: FusionTechSystem.links.ts中`PathGroupCheckResult`被import了两次（第12行和第132行），QuestSystem.helpers.ts中`QuestDef`被import了两次（第11行和第240行）。TypeScript对同一文件的重复import不会报错，但当`isolatedModules`启用时，重复的type import可能导致`TS2300: Duplicate identifier`。

**根因**: 拆分时将代码段从一个文件移到另一个文件，移动的代码段自带import语句，与目标文件已有的import产生重复。人工审查时容易忽略，因为重复import不影响大多数编译配置。

**教训**: 文件合并/移动后应运行`tslint no-duplicate-imports`或等效检查。也可以在拆分SOP中增加: 移动代码后，检查目标文件的所有import是否与已有import重复，合并重复项。

**推广**: 拆分后自动执行: `grep "^import" target.ts | sort | uniq -d` 检测重复import。CI中可配置`no-duplicate-imports`规则。

## LL-R24-004: `isolatedModules`模式下re-export类型必须用`export type`

**分类**: TypeScript

**场景**: responsive.types.ts中`export { PowerSaveState } from './responsive-navigation.types'`在`isolatedModules: true`配置下报错TS1205，因为PowerSaveState是一个类型，需要使用`export type`语法。

**根因**: 文件拆分时将值导出和类型导出混在同一个`export {}`语句中。当`isolatedModules`启用时（Vite默认启用），TypeScript要求明确区分值导出和类型导出。

**教训**: 拆分文件时，re-export必须区分: 1) 值导出用`export { ValueA, ValueB }`；2) 类型导出用`export type { TypeA, TypeB }`。不可混用。如果本地也需要使用该类型，还需要额外的`import type`语句。

**推广**: 所有re-export审查必须检查: 是否包含类型？是则用`export type`。本地是否也使用该类型？是则添加`import type`。这应纳入拆分checklist。

## LL-R24-005: 深度评测需要"实现验证"而非"文档验证"，避免误判

**分类**: 评测方法

**场景**: R22和R23的深度评测连续两轮将UP武将和每日免费招募标记为P1未实现。但R24实际检查引擎代码发现，HeroRecruitSystem已完整实现了UP池配置、限定池banner、每日免费次数重置等功能。前两轮评测仅基于文件名和注释搜索，未深入阅读实现代码。

**根因**: 评测方法过度依赖关键词搜索（如`grep "UP武将"`, `grep "每日免费"`），而实际代码中使用的是英文命名（`featuredGeneral`, `dailyFreeCount`）。搜索策略与代码命名风格不匹配导致误判。

**教训**: 深度评测的功能点检查应采用"PRD→接口→实现"三层验证: 1) 从PRD提取功能描述；2) 查找对应的接口/方法签名；3) 验证方法体有实质逻辑（非空函数/TODO）。仅靠关键词搜索不足以判定功能是否实现。

**推广**: 功能覆盖度评测模板增加"实现验证"列: 功能点→对应方法→方法行数→是否有实质逻辑。方法行数<5或包含`/* 预留 */`标记为未实现，否则标记为已实现。

## 进化规则建议

### ER-R24-001: 拆分后强制编译验证
**规则**: 每完成一批文件拆分（≥3个文件），必须执行`pnpm run build`并确认0错误后才可继续。违反此规则的拆分结果不计入Phase 4成果。

### ER-R24-002: 函数重命名过渡期别名
**规则**: helpers文件中函数重命名时，必须在原文件底部添加别名导出`export { newFunc as oldFunc }`，保留至所有调用方更新完毕。禁止直接删除旧名。

### ER-R24-003: 封版前全量编译+测试
**规则**: Phase 6封版判定前必须执行完整的`pnpm run build`（非tsc --noEmit）和全量测试（非抽样），确保编译和运行时均无错误。
