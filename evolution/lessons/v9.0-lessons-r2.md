# v9.0 离线收益 — 经验教训 Round 2

> 日期: 2025-07-16 | 轮次: Round 2

## LL-1: 聚合根+纯函数双轨架构控制复杂度

**分类**: 架构设计

**观察**: `OfflineRewardSystem`(374行有状态聚合根) + `OfflineRewardEngine`(306行纯函数) + `OfflineEstimateSystem`(184行无状态) 三者协作，聚合根负责状态管理和流程编排，纯函数模块负责核心计算。这使得：
- 纯函数模块可独立测试，无需 mock
- 聚合根只做编排，不重复计算逻辑
- `OfflineEstimateSystem.reset()` 为空操作但保留接口一致性

**启示**: 有状态聚合根 + 无状态纯函数的双轨模式是 DDD 战术设计的有效实践，应在后续版本（如邮件系统 v9.0 模块C）中延续。

## LL-2: 衰减计算逻辑去重是下一轮优先改进项

**分类**: 代码整洁

**观察**: 6档衰减分段计算逻辑在4个文件中重复实现（RewardSystem/RewardEngine/EstimateSystem/PanelHelper），虽各自参数略有差异，但核心循环结构相同。此外 `3600`（秒/小时）魔数散布18处，资源键遍历方式不一致（硬编码 vs Object.keys）。

**启示**: 应在 R3 中提取 `calculateTierBreakdown()` 共享函数 + `SECONDS_PER_HOUR` 常量 + `RESOURCE_KEYS` 常量到 `offline-utils.ts`，预计可消除 ~80 行重复代码。

---

## LL-3: 快照系统 Storage 抽象保障可测试性

**分类**: 测试策略

**观察**: `OfflineSnapshotSystem` 通过构造函数注入 `Storage?` 参数，生产环境传入 `localStorage`，测试环境传入内存 Map 或省略。这使得快照的创建/验证/过期/持久化 全链路可测，无需 jsdom 或 mock localStorage。

**启示**: 外部依赖（Storage/Date/Random）通过构造函数或参数注入是实现确定性测试的关键模式。后续 MailSystem 的邮件存储也应采用相同策略。
