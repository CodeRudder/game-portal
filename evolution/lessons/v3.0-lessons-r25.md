## LL-R25-001: 集成测试应与Play文档流程严格对应

**分类**: 测试方法

**场景**: R25编写15个集成测试文件时，初期测试用例按模块直觉组织，导致部分流程点的测试覆盖出现遗漏或重复。后续对照v3-play.md逐条核对才发现§4.7操作评分、§6.4碎片掉落等流程点需要显式skip标记。

**根因**: 缺乏"Play文档→测试用例"的严格映射机制。测试编写者倾向于按代码结构而非业务流程组织测试，导致流程覆盖度难以精确追踪。

**教训**: 每个测试文件的describe块应直接对应Play文档的章节编号（如§1.1~§1.4），每个it用例应标注对应的流程点编号。编写前先建立映射表，编写后逐条核对。

**推广**: 集成测试SOP增加"映射核对"步骤: 1) 从Play文档提取流程点清单；2) 为每个流程点分配test文件和it用例；3) 编写完成后逐条核对覆盖状态。

## LL-R25-002: 功能缺失类P1不应阻塞封版

**分类**: 封版策略

**场景**: R25深度评测发现12个P1问题，全部为功能缺失（VIP系统、手动战斗模式、操作评分等），而非代码质量bug。若按传统标准（P1=阻塞），v3.0将无法封版。但代码质量本身达到封版标准（0 as any、0 TODO、559/563测试通过）。

**根因**: 传统封版标准未区分"功能缺失"和"代码缺陷"。功能缺失属于版本规划问题，应在PRD层面决定是否纳入当前版本；代码缺陷才是封版阻塞项。

**教训**: P1问题应分为两类: 1) **P1-功能缺失**: PRD已规划但未实现，不阻塞封版，记录为后续版本待办；2) **P1-代码缺陷**: 已实现但有bug，阻塞封版。封版条件中的"P1=0"应仅指P1-代码缺陷。

**推广**: 封版判定模板增加P1分类列: 功能缺失/代码缺陷。仅P1-代码缺陷计数影响封版判定。P1-功能缺失记录为下版本backlog。

## LL-R25-003: Vitest的testPathPattern匹配规则不同于Jest

**分类**: 工具链

**场景**: R25运行集成测试时，使用`vitest run --testPathPattern`筛选特定测试文件，发现匹配规则与Jest不同。Vitest使用的是文件路径的正则匹配而非glob模式，导致部分文件未被正确匹配，需要调整pattern写法。

**根因**: 从Jest迁移到Vitest时，团队习惯性使用Jest的glob匹配语法（如`**/*.test.ts`），但Vitest的`--testPathPattern`使用的是正则表达式（如`campaign/01-campaign-map`）。两者语法差异导致测试筛选失败。

**教训**: Vitest环境下应使用`vitest run <file-pattern>`直接指定文件路径，或使用正则表达式作为`--testPathPattern`参数。避免使用Jest风格的glob语法。分批运行测试时，每批文件应逐一确认匹配结果。

**推广**: 测试运行脚本统一使用`vitest run src/path/to/file.test.ts`直接指定文件，避免pattern匹配歧义。CI脚本中使用`vitest run`不带filter运行全量测试。

## LL-R25-004: 集成测试编写应分批并行（每批3个文件）

**分类**: 工作效率

**场景**: R25需要编写15个集成测试文件，共563个用例。初期尝试逐个文件串行编写，效率低下且容易因单文件过长导致认知疲劳。改为每批3个文件并行编写后，效率显著提升，且每批可独立运行验证。

**根因**: 单文件串行编写时，每个文件需要完整加载上下文（模块依赖、mock配置、测试数据），切换成本高。分批并行允许复用上下文，且每批完成后可立即验证，快速发现共性问题。

**教训**: 大规模测试编写应按优先级分批: 1) P0核心流程（3个文件）→ 验证；2) P0辅助流程（3个文件）→ 验证；3) P1扩展功能（3个文件）→ 验证；4) P1边界场景（3个文件）→ 验证；5) P2异常处理（3个文件）→ 验证。每批完成后立即运行确认通过。

**推广**: 测试编写SOP增加分批策略: 每批≤3个文件，按P0→P1→P2优先级排列，每批完成后必须`vitest run`确认通过再进入下一批。

## LL-R25-005: 测试检查清单提升可追溯性

**分类**: 质量保证

**场景**: R25创建了v3-test-checklist.md，将69个流程点逐一映射到测试文件和用例。在Phase 4回归验证时，检查清单使得"哪些流程有测试、哪些被skip、原因是什么"一目了然，大幅提升了验收效率。

**根因**: 前期版本缺少测试检查清单，验收时需要逐一打开测试文件查找对应用例，耗时且容易遗漏。检查清单将分散信息集中展示，成为验收的核心参考文档。

**教训**: 每个版本的集成测试完成后，必须维护一份测试检查清单，包含: 流程点编号、对应测试文件、用例数量、通过/skip状态、skip原因。清单应与Play文档章节一一对应。

**推广**: 测试检查清单模板纳入版本交付物清单，与Play文档、集成测试文件并列。清单格式: `| 流程点 | 测试文件 | 用例数 | 状态 | 备注 |`。

## 进化规则建议

### ER-R25-001: 每个版本迭代必须包含集成测试编写步骤
**规则**: 版本迭代流程中，Phase 3必须包含"编写集成测试"子步骤。集成测试文件数≥Play文档章节数，用例数覆盖所有已实现流程点。未编写集成测试的版本不得进入封版判定。

### ER-R25-002: 每个版本迭代必须维护测试检查清单
**规则**: 集成测试完成后，必须创建`v{N}-test-checklist.md`，将所有Play流程点映射到测试用例。清单作为封版验收的必要输入文档，缺失清单不得封版。

### ER-R25-003: 集成测试按1-3个流程拆分文件
**规则**: 单个集成测试文件覆盖的Play流程点不超过3个（特殊情况下可扩展至5个）。超过5个流程点的测试文件必须拆分。每个文件用例数建议控制在30-60个之间。
