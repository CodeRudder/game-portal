## LL-R26-001: 源码行数控制从40个超标降至0个

**分类**: 代码质量

**场景**: R25封版时v3.0有40个400+行文件。R26 v4.0聚焦map模块，8个源文件最大仅398行，0个超标。对比R25全项目40个超标文件，map模块作为新开发模块展现了严格的行数控制。

**根因**: v4.0采用"小文件+单一职责"策略，每个System不超过400行。SiegeSystem(347行)和SiegeEnhancer(390行)的拆分是关键——将攻城逻辑和增强逻辑分离，避免了单一巨型攻城文件。

**教训**: 新模块开发时应先定义行数上限（建议400行），超过时立即拆分。拆分粒度以"单一职责"为标准，而非人为截断。SiegeSystem/SiegeEnhancer的拆分模式可作为参考。

**推广**: 新模块开发SOP增加步骤: 1) 设计阶段确定每个文件职责和预估行数；2) 编码时实时监控行数；3) 超过350行时评估是否需要拆分；4) 超过400行时强制拆分。

## LL-R26-002: Skip测试作为功能缺失的精确追踪手段

**分类**: 测试策略

**场景**: R26集成测试有52个skip用例，分布在cross-system-linkage(22)、cross-validation(10)、territory-garrison-filter-landmarks(8)等文件中。这些skip精确标记了MapEventSystem、MoraleSystem、PrestigeSystem等未实现功能的测试占位。

**根因**: R25建立了"skip标记功能缺失"的模式，R26延续此策略。每个skip用例对应一个具体的未实现功能点，使得功能缺口一目了然，且后续实现时可直接取消skip验证。

**教训**: Skip用例应包含三个要素: 1) 清晰的describe说明缺失什么功能；2) 完整的测试逻辑（实现后可直接运行）；3) skip原因注释。这样skip不仅是占位符，更是功能规格说明。

**推广**: Skip用例模板: `it.skip('功能名: 具体行为', () => { /* 完整测试逻辑 */ }); // TODO-R27: 实现XXX后取消skip`。每个skip对应一个明确的后续迭代任务。

## LL-R26-003: 编译错误可能因依赖更新自愈

**分类**: 工具链

**场景**: Phase 1-3报告4个TS导出错误（LeaderboardSystem相关，非v4.0范围），但Phase 4回归时编译完全通过，0错误。无任何人为修复操作。

**根因**: 可能是增量编译缓存问题，或中间步骤的git操作触发了依赖重新解析。TypeScript的增量编译有时会产生过时的错误缓存，clean build后可能自愈。

**教训**: 编译错误应区分"确定性错误"和"缓存性错误"。遇到非当前模块的编译错误时，先执行`pnpm run build`全量编译确认，再决定是否修复。避免为缓存性错误浪费修复时间。

**推广**: 编译错误处理流程: 1) 全量`pnpm run build`确认；2) 若仍存在，检查是否属于当前版本范围；3) 非当前范围的错误记录但不修复；4) 当前范围的错误立即修复。

## LL-R26-004: 集成测试文件数与模块复杂度应匹配

**分类**: 测试设计

**场景**: R26为map模块编写13个集成测试文件（8旧+5新），504个用例。新增5个文件覆盖: cross-system-linkage(30用例)、siege-execution-territory-capture(26用例)、map-event-system(29用例)、mobile-responsive(18用例)、map-rendering-siege-conditions(28用例)。

**根因**: v4.0新增的攻城功能涉及多系统交互（SiegeSystem↔TerritorySystem↔GarrisonSystem），需要专门的集成测试验证跨系统行为。单靠单元测试无法覆盖"攻城→领地变更→驻防调整"的完整链路。

**教训**: 集成测试文件规划应基于"系统交互链路"而非单一系统。每个跨系统链路至少一个测试文件。v4.0的5条核心链路: 1) 攻城执行→领地捕获；2) 跨系统串联；3) 地图事件触发；4) 移动端响应；5) 地图渲染攻城条件。

**推广**: 集成测试规划步骤: 1) 列出所有系统间交互链路；2) 每条链路分配一个测试文件；3) 每个文件30-60个用例覆盖正常/边界/异常场景。

## LL-R26-005: 已知bug应标记但不阻塞封版

**分类**: 封版策略

**场景**: WorldMapSystem.getRegions()测试期望4个区域但实际返回3个，这是从R25继承的已知P2 bug。该bug不影响v4.0攻城核心功能，且WorldMapSystem属于基础模块，修复需要评估影响范围。

**根因**: getRegions()的返回值在v3.0和v4.0之间存在语义差异。测试期望4个区域（魏蜀吴+中原），但实现只返回3个（魏蜀吴）。可能是因为中原区域的数据结构与其他区域不同。

**教训**: 已知bug应分三级管理: 1) **P0-阻塞**: 影响核心功能，必须修复才能封版；2) **P1-跟踪**: 影响次要功能，记录为下版本修复；3) **P2-容忍**: 影响边缘场景，长期跟踪。getRegions()属于P2-容忍级别。

**推广**: 封版报告增加"已知bug清单"章节，每个bug记录: 级别、影响范围、复现路径、计划修复版本。P0必须清零，P1不超过3个，P2不限。

## 进化规则建议

### ER-R26-001: 新模块源码行数上限400行
**规则**: 新开发模块的每个源文件不得超过400行。超过时必须拆分为职责更单一的子模块。拆分时应以功能边界为切割点，禁止为满足行数限制而人为截断逻辑。

### ER-R26-002: Skip用例必须包含完整测试逻辑
**规则**: `it.skip`标记的测试用例必须包含完整的测试逻辑代码（arrange-act-assert），而非仅占位。实现对应功能后，取消skip即可直接运行验证，无需重写测试。

### ER-R26-003: 封版报告必须包含已知bug清单
**规则**: 封版报告增加"已知bug清单"章节，每个bug记录级别(P0/P1/P2)、影响范围、复现路径、计划修复版本。P0 bug必须清零方可封版，P1 bug不超过3个，P2 bug不限但需逐一说明。
