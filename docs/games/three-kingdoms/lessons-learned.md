### 2.2 哪些问题类型最容易被遗漏

| 遗漏类型 | 描述 | 出现频率 | 原因 |
|---------|------|---------|------|
| **跨面板交互问题** | 面板间数据联动（升级建筑→资源栏刷新→武将升级可用） | 高 | 从未测试面板间数据一致性 |
| **边界条件崩溃** | 资源为 0、武将满级、建筑满级、全解锁等极端情况 | 高 | 审计只看代码不看运行时 |
| **存档/读档完整性** | save/load 后 UI 状态是否正确恢复 | 高 | 从未验证存档数据完整性 |
| **性能瓶颈** | 39 个子系统注册后 100ms tick 下的渲染性能 | 中 | 无性能测试 |
| **可访问性** | aria-label、对比度、键盘导航、WCAG 合规 | 中 | 从未进行专项审计 |
| **移动端真实体验** | 17/25 面板无移动端适配 | 高 | CSS 审计发现但修复滞后 |
| **视觉回归** | 修复引入新的视觉问题 | 中 | 无截图对比机制 |
| **加载状态** | 全局面板缺少加载状态（覆盖率仅 3%） | 高 | 审计维度独立但未优先修复 |

### 2.3 评测频率和深度的最佳平衡

**推荐三级评测体系**:

```
Level 1 — 自动化检查（每次提交，<5 分钟）
├── TypeScript 编译检查（engine: any 报错）
├── ESLint 规则（禁止 inline styles、强制 CSS 文件）
├── 单元测试覆盖率检查
├── import 依赖检查（禁止从 ui/components/ 导入）
└── 禁止模式检查（alert/prompt/Math.random/console.log）

Level 2 — 迭代审计（每轮迭代，2-3 小时）
├── 代码审计：引擎注册完整性 + 面板 Props 规范性
├── 功能测试：按 UI-CHECKLIST 逐项检查
├── 视觉检查：关键面板截图对比（3 个断点）
└── 响应式检查：375px / 768px / 1280px

Level 3 — 深度验证（每 5 轮迭代或里程碑，1-2 天）
├── E2E 测试：完整用户流程模拟
├── 性能测试：100ms tick 下的渲染帧率
├── 存档测试：save/load 数据完整性
├── 边界测试：极端数值、满级、空数据
└── 可访问性审计：WCAG 2.1 AA 合规
```

**经验法则**: 每轮迭代至少执行 Level 2，每 5 轮或到达里程碑时执行 Level 3。Level 1 应自动化为 CI 流水线。

## 三、修复效率优化

### 3.1 修复模式效率排名

| 模式 | 适用场景 | 效率 | 案例 |
|------|---------|------|------|
| **单面板完整对接** | 引擎注册 + 面板创建 + 主入口挂载 + 测试 | ⭐⭐⭐⭐⭐ | EquipmentTab 从无到有 |
| **按问题级别批量修复** | P0 全部清零 → P1 批量修复 | ⭐⭐⭐⭐ | R11 修复 3 个 P0（alert/prompt/Math.random） |
| **按维度专项审计** | 单一维度深入检查 | ⭐⭐⭐ | R18 错误处理审计 |
| **按层级拆分** | 引擎→面板→集成 | ⭐⭐⭐ | R10 引擎对接 |
| **全量审计** | 所有维度全覆盖 | ⭐⭐ | R25 中期验收（发现 12 个问题但未修复） |

### 3.2 子任务拆分的最佳粒度

| 粒度 | 适用场景 | 优缺点 |
|------|---------|--------|
| **单面板完整对接（推荐）** | 引擎注册 + 面板创建 + 主入口挂载 + 测试 | ✅ 一个 PR 完成端到端；❌ 单次工作量较大（2-4h） |
| **按层级拆分** | 大规模新功能开发 | ✅ 每步可独立验证；❌ 中间状态不可用 |
| **按问题级别拆分** | Bug 修复迭代 | ✅ 优先级清晰；❌ 同一面板可能跨多轮 |
| **按维度拆分** | 全面审计 | ✅ 每轮聚焦；❌ 可能遗漏跨维度问题 |

**最佳实践**: 以"单面板完整对接"为基本单位，配合"按问题级别"的优先级排序。

### 3.3 超时与效率瓶颈

| 瓶颈 | 描述 | 解决方案 |
|------|------|---------|
| **问题发现滞后** | R1 发现 P0 问题，到 R10 才完全验证引擎对接 | 每轮迭代执行引擎 API 追踪 |
| **修复引入新问题** | R6 发现 R1-R5 修复引入了 Tab 图标重复、FeaturePanel 重复渲染 | 每轮修复后执行回归验证 |
| **缺乏自动化验证** | 无 CI 检查、无 E2E 测试、无视觉回归测试 | 建立 Level 1 自动化流水线 |
| **面板类型不安全** | 大量 `engine: any` 导致编译期无法发现问题 | 强制 TypeScript 严格模式 |
| **内联样式迁移成本高** | 25 个面板中 22 个使用内联样式，迁移到 CSS 工作量巨大 | 新面板强制使用 CSS 文件，旧面板渐进迁移 |
| **视觉一致性问题积压** | R16 发现 32 个视觉一致性问题，修复率仅 31% | 建立设计系统（共享组件 + CSS token） |

### 3.4 高效修复的"黄金法则"

1. **P0 不过夜**: P0 问题在发现当轮必须修复（历史执行率 75%，需提升至 100%）
2. **修复即验证**: 每个修复后立即运行测试 + TypeScript 编译
3. **单面板闭环**: 一个面板的引擎注册→面板创建→挂载→测试在同一轮完成
4. **禁止半成品**: 不允许"面板已创建但引擎未注册"的中间状态合入主分支
5. **回归必跑**: 每轮修复后执行全量测试（192 文件 / 6351 用例）

## 四、代码质量模式

### 4.1 常见代码问题模式（按出现频率排序）

| 排名 | 问题模式 | 出现次数 | 占比 | 代表性问题 |
|------|---------|---------|------|-----------|
| 1 | **引擎未集成/未注册** | 13 | 15% | core/ 有类型但 engine/ 无实现 |
| 2 | **内联样式 vs CSS 文件** | 22 面板 | 25% | 22/25 面板使用内联样式 |
| 3 | **类型不安全（engine as any）** | 10+ 面板 | 12% | Equipment/Arena/Expedition 等 |
| 4 | **响应式缺失** | 17 面板 | 19% | 17/25 面板无移动端适配 |
| 5 | **硬编码颜色值** | 100+ 处 | 11% | 未使用 CSS 变量 |
| 6 | **z-index/图层冲突** | 15+ 处 | 7% | 硬编码 z-index 与 token 体系脱节 |
| 7 | **重复组件/代码冗余** | 5 处 | 3% | EquipmentPanel vs EquipmentTab |
| 8 | **Props 接口不统一** | 4 种模式 | 2% | engine:any vs ThreeKingdomsEngine |
| 9 | **错误处理缺失** | 13 面板 | 4% | 无 try-catch 包裹引擎调用 |
| 10 | **加载状态缺失** | 33 面板 | 2% | 覆盖率仅 3% |

### 4.2 根因分析

#### 根因 #1: 两套 UI 体系并行（影响最广）

- `ui/components/` 使用 GameContext + inline styles
- `panels/` 使用 props + CSS files
- **后果**: 迁移成本高、组件重复、样式不统一、21+ 孤立文件成为死代码

#### 根因 #2: 引擎与 UI 开发脱节

- core/ 类型定义 → engine/ 实现 → UI 面板 三层开发未同步
- **后果**: 大量引擎子系统有类型但无实现或无注册，形成"空壳面板"

#### 根因 #3: 缺乏统一的设计系统

- 无共享 Button/Card/ProgressBar/SubTab 组件
- CSS token 已定义但面板未使用
- **后果**: 25 种面板 25 种样式，视觉一致性极差

#### 根因 #4: 缺乏面板接入规范

- 无强制检查清单
- Props 接口无统一标准
- **后果**: 每个面板的接入方式、类型安全、错误处理参差不齐

### 4.3 最佳实践总结

#### 面板接入检查清单（5 分钟/面板）

```
□ 引擎 getter 是否存在
□ Props 类型为 ThreeKingdomsEngine（非 any）
□ snapshotVersion 在 useMemo 依赖中
□ 有 try-catch 包裹引擎调用
□ 有空数据占位 UI
□ 有 CSS 文件（非纯 inline styles）
□ 有移动端 @media 适配
□ 有列表滚动限制（maxHeight + overflow）
□ 按钮触摸热区 ≥ 44px
□ 关闭按钮有 hover 反馈
```

#### 引擎注册检查清单（3 分钟/子系统）

```
□ core/ 类型已定义
□ engine/ 实现已完成
□ ThreeKingdomsEngine 私有字段已声明
□ register() 已注册到 SubsystemRegistry
□ getter 方法已暴露
□ getSnapshot() 已加入快照
□ buildSaveCtx() 已加入存档
□ init() 中已初始化
□ reset() 中已重置
```

#### 设计系统规范

| 用途 | 规范值 | CSS Token |
|------|--------|-----------|
| 面板容器内边距 | 12px | `var(--tk-gap-md)` |
| 面板标题字号 | 16px | `--tk-panel-title-size` |
| 主操作按钮圆角 | 6px | `var(--tk-radius-md)` |
| 主操作按钮背景 | `rgba(200,168,76,0.2)` | — |
| 主操作按钮边框 | `1px solid var(--tk-gold)` | — |
| 弹窗背景 | `rgba(26,35,50,0.97)` | — |
| 弹窗圆角 | 10px | — |
| 品质色（传说） | `#C9A84C` | `--tk-quality-legendary` |
| 品质色（史诗） | `#D4553A` | `--tk-quality-epic` |
| 品质色（稀有） | `#9B6DBF` | `--tk-quality-rare` |

### 4.4 防止回归的策略

1. **TypeScript 严格模式**: 禁止 `engine: any`，编译期拦截类型不安全
2. **ESLint 规则**: 禁止 `alert()`/`prompt()`/`Math.random()`，禁止纯内联样式
3. **共享组件优先**: 新面板必须使用 `Panel`/`Modal`/`Button` 共享组件
4. **CSS Token 强制引用**: 新样式必须使用 CSS 变量，禁止硬编码颜色
5. **测试覆盖门槛**: 新面板必须包含基本测试（渲染 + 空数据 + 错误处理）
6. **PR 检查清单**: 合入主分支前必须通过面板接入检查清单

## 七、关键经验教训 Top 10

### 1. 🏗️ 架构一致性 > 功能数量

两套 UI 体系并行（ui/components/ vs panels/）是最大的技术债。**教训**: 项目启动时就应确定唯一的 UI 架构模式，不允许并行体系存在。

### 2. 🔗 引擎与 UI 必须同步开发

"面板有 UI 但引擎无数据"是最常见的问题模式（13 个面板）。**教训**: 引擎注册→面板创建→主入口挂载→测试验证必须在同一轮迭代完成。

### 3. 📐 设计系统必须先行

25 个面板 25 种样式，修复成本远高于预防成本。**教训**: 在第一个面板开发前就建立共享组件库（Button/Card/Panel/Modal）和 CSS Token 体系。

### 4. 🧪 自动化验证不可省略

18 份报告、258 个问题全部通过人工审计发现，效率极低。**教训**: Level 1 自动化检查（TypeScript + ESLint + 测试）应在项目早期建立。

### 5. 📱 移动端适配必须同步

68% 的面板无移动端适配，后期补齐成本巨大。**教训**: 每个面板开发时就包含移动端适配，不允许"先 PC 后移动"的延期策略。

### 6. 🎯 评测深度比广度更重要

R25 中期验收覆盖了所有维度但未修复任何问题。**教训**: 每轮迭代聚焦 1-2 个维度深入修复，优于全面扫描但不修复。

### 7. 🚫 禁止模式必须零容忍

alert/prompt/Math.random 在多个面板中出现。**教训**: 建立禁止模式清单，通过 ESLint 和 Code Review 强制执行。

### 8. 📊 评分卡驱动改进

面板对接评分卡清晰展示了每个面板的短板。**教训**: 量化评分比定性描述更能驱动改进。

### 9. 🔄 修复后必须回归验证

R6 发现 R1-R5 修复引入了新问题（Tab 图标重复、FeaturePanel 重复渲染）。**教训**: 每轮修复后必须执行全量测试 + TypeScript 编译。

### 10. ⏰ P0 问题不过夜

历史 P0 修复率为 75%，部分 P0 跨越多轮未修复。**教训**: P0 问题必须在发现当轮修复，不允许降级或延期。
