# 三国霸业 — 开发流程规范

> 本文档定义每个版本的标准开发流程、评测维度和封版判定标准。
> 调度规则见 [iteration-rules.md](./iteration-rules.md)

---

## 一、每个版本的标准流程（4阶段串行流水线）

```
阶段1: 引擎层开发（串行）
   ├── 读取plan+PRD，在各职责文件中追加引擎逻辑
   └── 必须完成后才能进入阶段2

阶段2: 逻辑层测试 + UI层开发（可并行）
   ├── 子任务A: 引擎单元测试
   └── 子任务B: UI组件开发（依赖引擎API）
   └── 两者都必须完成后才能进入阶段3

阶段3: 编译验证 + 全量测试（串行）
   ├── npx tsc --noEmit 零错误
   ├── npx vitest run 全量通过
   └── 必须通过后才能进入阶段3.5

阶段3.5: 架构审查 + 优化（串行）
   ├── 文件行数检查(代码≤500行/测试≤1000行)
   ├── 单一职责检查
   └── 拆分优化

阶段4: 评测师评审（串行）
   ├── 10维度评分
   ├── 评分>9.8 → commit → 进入下一版本
   └── 评分≤9.8 → 改进 → 回到阶段2重新评测
```

---

## 二、开发实现子流程（阶段1+阶段2内部）

```
1. 引擎开发（子任务：developer）
   ├── 读取PLAN文档（含PRD/UI链接）
   ├── 按DDD架构在对应业务域目录追加代码
   │   ├── engine/{domain}/{Domain}System.ts  — 领域逻辑
   │   ├── engine/{domain}/{domain}-config.ts — 数值配置
   │   ├── engine/{domain}/{domain}.types.ts  — 类型定义
   │   └── engine/ThreeKingdomsEngine.ts       — 编排
   └── 确保编译通过

2. 引擎测试（子任务：tester，可与UI开发并行）
   ├── 编写领域逻辑单元测试
   ├── 确保分支覆盖率 100%
   └── 确保所有测试通过

3. UI开发（子任务：developer，可与引擎测试并行）
   ├── 严格按UI布局文档实现
   │   ├── panels/{domain}/  — 面板子组件
   │   └── ThreeKingdomsGame.tsx — 编排
   └── 确保编译通过

4. UI测试（子任务：tester）
   ├── 编写UI交互测试
   └── 确保分支覆盖率 100%
```

## 三、评测维度（每项0-10分）

| # | 维度 | 评判标准 |
|---|------|---------|
| 1 | **功能完整性** | PLAN中规划的功能是否100%实现 |
| 2 | **PRD符合度** | 游戏逻辑是否100%符合PRD数值表 |
| 3 | **UI设计符合度** | UI是否100%符合UI布局文档 |
| 4 | **代码质量** | 代码结构/命名/可维护性/代码≤500行/测试≤1000行 |
| 5 | **测试覆盖** | 分支覆盖率/通过率是否100% |
| 6 | **可玩性** | 游戏是否有真实可玩性（不是空壳） |
| 7 | **交互体验** | 点击/弹窗/Tab切换是否流畅自然 |
| 8 | **动画规范** | 动画时长/缓动是否符合设计规范 |
| 9 | **响应式** | PC端和手机端是否正确适配 |
| 10 | **整体完成度** | 版本整体交付质量 |

---

## 四、封版判定

- **通过条件**：10维度每项 ≥ 9.8分，总分 ≥ 9.8分
- **不通过**：改进后重新评测，无循环次数上限
- **封版commit**：简述完成事项，不写"封版"

---

## 五、评测-改进循环

```
评测不通过（任一维度<9.8）
    ↓
读取评测报告中的改进建议
    ↓
按优先级修复（P0→P1→P2）
    ↓
重新运行测试确认通过
    ↓
重新提交评测（同一版本，R2/R3/R4...）
    ↓
直到评分>9.8 → 封版
```
