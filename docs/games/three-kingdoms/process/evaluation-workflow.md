## 一、核心原则

### 1.1 四大铁律

| # | 铁律 | 说明 | 违反后果 |
|---|------|------|---------|
| 1 | **小范围** | 每轮只评1-3个关联版本 | 范围过大→修复率暴跌（历史33%） |
| 2 | **真实验证** | 每轮必须启动dev-server，亲眼看到页面运行 | 不启动→运行时错误零发现（历史最大盲点） |
| 3 | **持续进化** | 每轮记录经验教训，启发式改进评测方法 | 不复盘→同类问题反复出现 |
| 4 | **眼见为实** | 以真实页面/数据/控制台输出为准，禁止纯代码推理判定 | 推理≠真实，历史258个问题中大量是推理发现但无法确认真实影响的 |

### 1.2 "眼见为实"原则详解

**这是60轮迭代最大的教训**：之前所有评测都是"看代码推理页面应该怎样"，从未真正打开浏览器验证。

```
❌ 错误模式：推理式评测
   "代码里有 engine.getChapters() 调用 → 战役功能应该可用"
   "BuildingPanel.tsx 渲染了升级按钮 → 建筑升级应该能点"
   "CSS里有@media规则 → 移动端应该适配了"

✅ 正确模式：实证式评测
   启动dev-server → 打开浏览器 → 点击建筑Tab → 看到面板 → 点击升级 → 看到Toast → 确认可用
   打开F12 → Console无Error → 确认无运行时异常
   切换375px → 看到布局正确 → 点击按钮 → 确认触摸热区够大
```

**具体要求**：

| 场景 | ❌ 禁止 | ✅ 必须 |
|------|--------|--------|
| 判断功能是否可用 | 读代码推理"应该可用" | 启动dev-server，真实点击验证 |
| 判断页面是否报错 | grep搜索"catch"推理错误已处理 | 打开F12 Console，看真实输出 |
| 判断数据是否对接 | 检查import推理"已引用引擎" | 在页面上看数据是否真实显示 |
| 判断样式是否正确 | 读CSS推理"规则已写" | 截图对比，肉眼确认渲染效果 |
| 判断修复是否生效 | 读diff推理"代码已改" | 重新启动dev-server验证修复效果 |

**唯一例外**：纯逻辑bug（如计算公式错误）可通过代码审查+单元测试验证，无需启动dev-server。

### 1.3 迭代单元

```
一个迭代单元 = 1-3个关联版本的功能

示例：
  迭代单元A = v1.0 基业初立（建筑+资源，2个子系统）
  迭代单元B = v2.0 招贤纳士（武将招募+升级+碎片，3个子系统）
  迭代单元C = v3.0+v4.0 攻城略地（战役+战斗，关联版本合并）
  迭代单元D = v5.0 百家争鸣（科技树，1个子系统）
```

**合并规则**：功能强关联的版本可合并为一个迭代单元（如v3+v4都是战斗系统），但不超过3个版本。

## 二、单轮评测标准流程

### 2.1 总览

```
Phase 1: 静态检查 ──── 15min
Phase 2: 动态验证 ──── 20min  ← 核心改进：必须启动dev-server
Phase 3: 问题修复 ──── 30-60min
Phase 4: 复盘记录 ──── 10min
                        ─────────
                        总计: 75-105min/轮
```

### 2.2 Phase 1: 静态检查（15min）

**目标**：在代码层面确认基础质量。

| # | 检查项 | 命令 | 通过标准 |
|---|--------|------|---------|
| 1 | TypeScript编译 | `npx tsc --noEmit` | 0错误 |
| 2 | 全量测试 | `npx vitest run src/games/three-kingdoms` | 100%通过 |
| 3 | 文件行数 | 检查活跃文件 | 所有文件≤500行 |
| 4 | 禁止模式扫描 | grep检查 | 无alert/prompt/Math.random伪造 |

**禁止模式扫描脚本**：
```bash
# P0禁止模式
grep -rn "alert(" src/components/idle/ --include="*.tsx" && echo "❌ 发现alert"
grep -rn "prompt(" src/components/idle/ --include="*.tsx" && echo "❌ 发现prompt"
grep -rn "Math.random()" src/components/idle/ --include="*.tsx" | grep -v "test\|mock" && echo "❌ 发现Math.random伪造"
grep -rn "engine: any" src/components/idle/ --include="*.tsx" && echo "❌ 发现engine:any"
```

### 2.3 Phase 2: 动态验证（20min）⭐ 核心改进

**目标**：在真实浏览器环境中验证游戏可用性。

#### 2.3.1 启动开发服务器

```bash

npm run dev
# 等待编译完成，确认无编译错误
```

#### 2.3.2 浏览器验证检查点

| # | 检查项 | 验证方法 | 通过标准 |
|---|--------|---------|---------|
| D-1 | 页面加载 | 打开游戏URL | 无白屏、无JS错误 |
| D-2 | 控制台错误 | F12→Console | 0个Error（Warning≤5可接受） |

##### 工具E: AI自创工具

**AI有权根据需要创建全新的评测工具**，不限于上述列表。例如：

- 数据一致性校验器（面板数据vs引擎数据对比）

- 无障碍扫描器（WCAG合规性自动检查）

#### 2.3.4 工具选择策略

| 场景 | 推荐工具 | 理由 |
|------|---------|------|
| 快速验证组件是否渲染 | UITreeExtractor Console脚本 | 零配置，即时反馈 |
| 完整用户流程验证 | Playwright E2E | 可靠、可重复、可CI |
| 探索性测试（发现未知问题） | Agent Browser | AI自主探索，覆盖盲区 |
| 修复后回归验证 | 截图对比 + UITreeDiffer | 快速确认无回归 |
| 特定问题深度分析 | AI自创工具 | 针对性强 |

### 2.4 Phase 3: 问题修复（30-60min）

**修复原则**：

1. **发现即修复**：Phase 2发现的问题在本轮Phase 3立即修复
2. **修一个验一个**：每个修复后立即运行测试+编译确认
3. **修复后重验证**：Phase 3结束后重新执行Phase 2确认修复有效
4. **P0不过夜**：P0问题必须在当轮修复

**修复流程**：
```
发现问题 → 定位根因 → 编写修复 → 运行测试 → 启动dev-server验证 → 确认修复
```

### 5.3 适应阶段：启发式改进评测方法

**启发式规则**：基于感知阶段的真实数据，自动触发对应的改进动作。

| 感知信号 | 启发式规则 | 改进动作 |
|---------|-----------|---------|
| Console有Error但代码审查未发现 | **规则H-1: 运行时错误盲区** | 下轮增加Console监控检查点，优先使用dev-server验证 |
| 修复后回归测试仍发现同类问题 | **规则H-2: 修复不彻底** | 下轮对同类问题做全量扫描而非单点修复 |
| 某工具连续2轮未发现任何问题 | **规则H-3: 工具低效** | 降级该工具为抽样使用，或替换为更有效的工具 |
| 某类问题在多轮中反复出现 | **规则H-4: 根因未消除** | 升级为架构级问题，在经验教训库中标记根因 |
| 新发现的问题无现有工具可检测 | **规则H-5: 工具空白** | 创建新工具或扩展现有工具 |
| 页面截图与预期不一致 | **规则H-6: 视觉回归** | 建立截图基线，下轮增加截图对比检查点 |
| 评测耗时持续增长 | **规则H-7: 效率下降** | 分析耗时瓶颈，自动化高频检查项 |

**AI有权自主触发以上规则之外的启发式改进**。当AI发现新的模式或规律时，应：
1. 在经验教训库中记录新发现的模式
2. 自主决定改进方案（新增检查点/创建工具/调整流程）
3. 在下一轮中验证改进效果
4. 如果有效，将新规则正式加入启发式规则库

## 六、评测迭代循环：多轮收敛过程

> 一个迭代单元（1-3版本）可能需要多轮评测才能封版。以下是多轮迭代的收敛模型。

### 6.1 迭代收敛模型

```
迭代单元: v1.0 基业初立

R1 ──▶ 发现5个问题 → 修复5个 → 重验证 → 仍剩2个问题
                                                    │
R2 ──▶ 发现2个遗留+1个回归 → 修复3个 → 重验证 → 仍剩1个问题
                                                          │
R3 ──▶ 发现1个遗留 → 修复1个 → 重验证 → 0个问题 → ✅ 封版
```

**收敛判定**：连续1轮无新问题发现 + 所有检查点通过 = 封版。

### 6.2 每轮迭代的完整闭环

```
┌─────────────────────────────────────────────────────────────┐
│                    单轮迭代完整闭环                           │
│                                                             │
│  ┌─────────────────────────────────────────────┐           │
│  │ Step 1: 查阅知识库                            │           │
│  │ - 读取历史经验教训                             │           │
│  │ - 读取常见问题模式库                           │           │
│  │ - 读取上一轮的改进措施                         │           │
│  │ - 确定本轮重点关注的检查项                      │           │
│  └──────────────────────┬──────────────────────┘           │
│                         ▼                                   │
│  ┌─────────────────────────────────────────────┐           │
│  │ Step 2: 执行评测（Phase 1-3）                  │           │
│  │ - 静态检查（编译+测试+扫描）                    │           │
│  │ - 动态验证（dev-server+浏览器+工具）            │  ← 眼见为实 │
│  │ - 问题修复（发现即修复+修完即验证）              │           │
│  │ - 所有结论必须有真实截图/数据/日志作为证据       │           │
│  └──────────────────────┬──────────────────────┘           │
│                         ▼                                   │
│  ┌─────────────────────────────────────────────┐           │
│  │ Step 3: 复盘进化（Phase 4）                    │           │
│  │ - 记录本轮真实数据（非推理结论）                 │           │
│  │ - 对比上轮指标，量化改进效果                     │           │
│  │ - 触发启发式规则，生成改进措施                    │           │
│  │ - 更新知识库（新问题模式/新教训/新工具经验）      │           │
│  └──────────────────────┬──────────────────────┘           │
│                         ▼                                   │
│  ┌─────────────────────────────────────────────┐           │
│  │ Step 4: 封版判定                               │           │
│  │ - 所有检查点通过？                              │           │
│  │ - 无P0/P1遗留？                                │           │
│  │ - dev-server验证无问题？                        │           │
│  │ - 是 → 封版 → 进入下一个迭代单元                │           │
│  │ - 否 → 回到Step 1，开始下一轮(R+1)              │           │
│  └─────────────────────────────────────────────┘           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 多版本迭代的流水线

当多个迭代单元排队时，形成流水线：

```
迭代单元1(v1) ──R1──▶──R2──▶──R3──✅ 封版
                                    │
迭代单元2(v2) ──────────────────────R1──▶──R2──✅ 封版
                                                        │
迭代单元3(v3+v4) ───────────────────────────────────────R1──▶──R2──▶──R3──✅ 封版
```

**关键约束**：
- 同一时间只处理一个迭代单元
- 前一个封版后才开始下一个
- 每个迭代单元的评测报告独立存储
- 知识库跨迭代单元共享（v2的教训可应用到v3的评测）

### 6.4 迭代轮次上限

| 场景 | 上限 | 超限处理 |
|------|------|---------|
| 单个迭代单元 | 5轮 | 暂停，分析根因，考虑是否需要架构级重构 |
| 连续3个迭代单元都超过3轮 | — | 暂停，重新评估评测方法有效性 |
| 同类问题在3个迭代单元中重复出现 | — | 升级为架构级问题，一次性解决 |

### 6.5 "眼见为实"在迭代循环中的体现

每一轮迭代的每一个结论，都必须有**真实证据**支撑：

| 结论类型 | 必须的证据 | ❌ 禁止的推理 |
|---------|-----------|-------------|
| "功能可用" | 截图：功能正常使用的画面 | "代码里有调用，应该可用" |
| "修复生效" | 截图：修复前vs修复后的对比 | "diff显示代码已改，应该修好了" |
| "无运行时错误" | 截图：F12 Console无Error | "代码里有try-catch，应该不会报错" |
| "数据已对接" | 截图：面板显示真实数据 | "import了引擎模块，应该有数据" |
| "样式正确" | 截图：页面渲染效果 | "CSS规则已写，应该生效" |
| "移动端适配" | 截图：375px下的真实渲染 | "@media规则已写，应该适配了" |
| "性能达标" | 数据：FPS/加载耗时的真实测量值 | "代码没有循环嵌套，应该不卡" |

**总结：没有截图/数据/日志的评测结论 = 无效结论。**

### C.1 UITreeExtractor扩展规范

5. 测试文件: ui-extractor/__tests__/{Name}Adapter.test.ts

```

5. 截图基线: e2e/screenshots/{version}/
```
