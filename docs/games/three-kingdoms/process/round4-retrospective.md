## 四、失败分类分析

### 4.1 Canvas渲染器 (9个文件, 127个用例)

**根因**: jsdom环境缺少Canvas API mock，导致所有Canvas/WebGL相关渲染测试失败。

**影响范围**: Scene渲染器、地图绘制、特效系统等。

**R5建议**: 引入 `jest-canvas-mock` 或在setup中补全Canvas mock。

### 4.2 UI组件 (6个文件, 24个用例)

**根因**: modal渲染和交互测试依赖DOM结构，R3拆分后部分选择器失效。

**影响范围**: BuildingUpgradeModal、Dialog组件等。

**R5建议**: 使用 `data-testid` 替代CSS选择器，提高测试稳定性。

### 4.3 引擎逻辑 (2个文件, 2个用例)

**根因**: 浮点精度比较和模拟器状态断言问题。

**影响范围**: 战斗计算、资源产出等。

**R5建议**: 使用 `toBeCloseTo()` 替代严格相等比较。

### 4.4 空测试 (5个文件)

**根因**: p2拆分时部分describe块内测试用例未正确复制。

**R5建议**: 清理空测试文件或补充实际测试用例。

### 4.5 UI回归 (1个文件, 1个用例)

**根因**: R4修复过程中引入的轻微回归。

**R5建议**: 增加回归测试覆盖。

## 五、经验教训

### LL-R4-001: 测试文件拆分必须确保每个文件自包含

R3拆分28个超标测试文件时，生成的p2文件缺少独立import、helper函数和完整describe块，导致p2文件无法独立运行。本轮逐一修复28个p2文件的自包含问题，耗时13个提交。

**教训**: 拆分操作应作为原子事务——每个p2文件生成时必须包含完整的import、helper和describe，确保 `npx jest <file>` 可独立通过。

**规则建议**: 新增进化规则「测试拆分原子性」——拆分后的每个文件必须通过 `npx jest <file>` 独立验证。

### LL-R4-002: 花括号自动修复只是临时方案

R3使用自动修复处理了54个文件的花括号不匹配问题，但R4发现仍有53处花括号问题。自动修复只能处理表面语法，无法保证代码结构完整性。

**教训**: 花括号修复应配合AST级别的结构验证，而非简单的文本匹配。后续应使用TypeScript Compiler API进行结构验证。

### LL-R4-003: 子任务超时频发，简单操作应优先在主会话执行

R4修复过程中多次出现子任务超时（1200s限制），但实际工作往往已完成。对于简单的文件修复操作（补全import、复制helper），应优先在主会话bash中直接执行，避免子任务框架的通信开销。

**教训**: 简单批量操作（<10个文件）用主会话bash；复杂重构（>10个文件或跨模块）才使用子任务。
