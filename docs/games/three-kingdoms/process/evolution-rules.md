# 游戏迭代进化规则

> **范围**: v1-v20 所有版本，每轮评测改进一个迭代单元（1-3关联版本）。
> **编译要求**: `pnpm run build` 每次变更后必须通过。
> **执行方式**: 主会话编排，subagent 执行每个步骤。
> **定位**: 本文档是进化迭代的**执行手册**，定义"怎么进化"；评测细节见 [evaluation-workflow.md](./evaluation-workflow.md)，开发流程见 [development-workflow.md](./development-workflow.md)。

---

## 一、核心原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **眼见为实** | 所有结论必须有截图/数据/日志作证据，禁止纯推理判定 |
| 2 | **小范围迭代** | 每轮只处理1-3个关联版本，避免修复率暴跌 |
| 3 | **发现即修复** | 问题在本轮立即修复，不遗留到下一轮 |
| 4 | **每轮必复盘** | 复盘产出经验教训+进化规则，这是进化的核心引擎 |
| 5 | **UI测试不可跳过** | 每个Play流程步骤必须启动dev-server真实验证，禁止仅凭代码推理判定 |
| 6 | **修复必须回归** | 修复代码后必须重新测试验证，不通过则继续修复，直到回归通过 |

---

## 二、评测线索：Plan + Play 双驱动

> **Plan（版本计划）定义"要测什么"**——功能清单、覆盖度保证。
> **Play（用户流程）定义"怎么测"**——执行顺序、验证深度、跨系统集成。
> 两者交叉验证：Play走完后用Plan检查覆盖度，确保无遗漏。

### 2.1 为什么 Play 驱动而非功能点驱动

| 对比 | 功能点驱动（旧） | Play流程驱动（新） |
|------|----------------|-------------------|
| 视角 | 开发者视角，孤立测试每个功能 | 用户视角，按真实游戏流程走 |
| 跨系统bug | 功能点间不关联，跨系统问题发现晚 | 流程自然串联多系统，早发现集成问题 |
| 文档读取 | 同一PRD被多个功能点反复引用 | Play步骤直接关联PRD/UI，一次读取 |
| 效率 | 每个功能点独立测试，重复操作多 | 一个Play流程覆盖多个功能点 |

### 2.2 Play 文档的 PRD/UI 关联格式

每个Play步骤直接标注对应的PRD/UI引用，评测时无需反复查找：

```markdown
## 步骤3: 武将招募
> PRD: ui-design/prd/HER-heroes-prd.md §3.2 招募流程
> UI:  ui-design/ui-layout/HER-heroes.md §2 招募面板

操作: 点击"招贤馆"Tab → 选择武将 → 点击招募
验证:
  - [ ] 招募按钮资源不足时置灰 (PRD §3.2.1)
  - [ ] 招募成功后武将进入已拥有列表 (UI §2.3)
  - [ ] 碎片数量正确扣除 (PRD §3.2.3 数值表)
  - [ ] 移动端375px下招募面板布局正常
```

## 三、单轮进化流程（6阶段）

> 每个迭代单元（1-3关联版本）执行一轮完整流程。封版不通过则回到 Phase 2。

```
Phase 1: 准备 ──────────────────────── 10min
    │  读取Plan/Play/PRD/UI/进化规则
    ▼
Phase 2: 冒烟测试（Play第1遍）────────── 15min
    │  快速走通Play流程，验证基本可用性
    │  P0问题立即修复→回归验证（循环直到通过）
    ▼
Phase 3: 深度评测（Play第2遍）────────── 30-40min
    │  对照PRD/UI逐条验证每个Play步骤
    │  发现问题记录到UI评测报告
    ▼
Phase 4: 修复 + 回归验证（循环）──────── 20-40min
    │  修复Phase 3发现的问题
    │  每个修复后必须UI回归测试
    │  回归不通过→继续修复→再次回归（循环直到全部通过）
    ▼
Phase 5: 架构审查 + 修复回归 ────────── 20min
    │  全局架构合规检查
    │  发现问题→修复→回归验证（循环直到通过）
    ▼
Phase 6: 封版判定 + 复盘进化 ────────── 15min
    ├─ 通过 → 复盘→进化→commit→下一版本
    └─ 不通过 → 回到 Phase 2
```

### Phase 1: 准备（10min）

| 步骤 | 动作 | 产出 |
|------|------|------|
| 1.1 | 读取版本计划 `plans/v{X}.md` 功能清单 | 本轮要评测的功能点列表 |
| 1.2 | 读取/更新 Play 流程 `play/{version}-play.md`，标注PRD/UI关联 | 带引用的评测路线图 |
| 1.3 | 读取评测规范 `checklist/ui-checklist.md` + `architecture/ddd-architecture.md` | 检查标准 |
| 1.4 | 读取进化规范 `evolution/` 目录下的规则和进度 | 历史经验+当前进度 |
| 1.5 | 读取评测工具索引 `tools/` | 可用工具清单 |
| 1.6 | 创建/更新进度文件 `evolution/progress/evolution-progress-{round}.md` | 追踪本轮处理 |

补充Play流程时确保覆盖以下维度：

| 检查维度 | 说明 |
|---------|------|
| 主流程跑通 | 每个子系统的核心业务链是否完整（如：装备获取→穿戴→锻造→分解） |
| 关联系统验证 | 跨系统数据流是否正确（如：武将升星→属性增加→界面刷新→战斗生效） |
| 条件与限制 | 每个功能点的前置条件、数值限制是否覆盖 |
| 边界与异常 | 空数据/满级/资源不足等边界场景是否覆盖 |

### Phase 2: 冒烟测试（Play第1遍，15min）

**目标**: 快速走通Play流程，验证基本可用性。发现P0问题立即修复。

**必须启动dev-server，禁止跳过UI测试步骤。**

```
启动 dev-server（npm run dev）
    ↓
按 Play 流程逐步执行，每步只验证"能不能用"
    ↓
记录P0问题（白屏/崩溃/功能完全不可用）
    ↓
P0问题 → 立即修复 → UI回归验证（循环直到通过）
    ↓
冒烟通过 → 进入 Phase 3
```

| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| 页面加载 | 打开游戏URL | 无白屏、无JS Error |
| 资源栏 | 查看顶部 | 4种资源+数值可见 |
| Tab切换 | 点击各Tab | 面板正常显示 |
| 核心操作 | 按Play流程点击 | 有反馈（数值变化/Toast/动画） |

### Phase 3: 深度评测（Play第2遍，30-40min）

**目标**: 对照PRD/UI逐条验证每个Play步骤，发现所有P0/P1/P2问题。

**必须启动dev-server，在真实浏览器中逐条验证，禁止跳过。**

```
按 Play 流程逐步走:
    每到一个步骤 →
        ① 读取该步骤关联的 PRD 章节 → 验证业务规则/数值是否正确
        ② 读取该步骤关联的 UI 规范 → 验证布局/样式/交互是否符合
        ③ 截图保存证据
        ④ 记录问题到 UI 评测报告
    ↓
所有 Play 步骤走完后 → 统计问题清单
    ↓
进入 Phase 4 修复循环
```

| 验证维度 | 对照文档 | 典型检查项 |
|---------|---------|-----------|
| 业务规则 | PRD数值表、流程描述 | 资源消耗数值、产出公式、触发条件 |
| UI布局 | UI Layout规范 | 元素位置、尺寸、间距、对齐 |
| 交互行为 | PRD交互+UI交互 | 弹窗关闭、按钮响应、Tab切换、ESC |
| 数据对接 | PRD数据结构 | 面板显示非空、非undefined、非占位文字 |
| 边界场景 | PRD限制条件 | 空列表、满级、资源不足、网络异常 |
| 响应式 | UI SPEC-responsive | 375px/768px布局、触摸热区≥44px |

问题记录到 `ui-reviews/{prd-code}-review-{round}.md`，格式：

```markdown
| # | Play步骤 | 问题描述 | P级 | PRD/UI引用 | 截图 |
|---|---------|---------|-----|-----------|------|
| 1 | 步骤3:武将招募 | 招募按钮资源不足时未置灰 | P1 | PRD §3.2.1 | [截图] |
```

### Phase 4: 修复 + 回归验证循环（20-40min）

**铁律: 修复代码后必须UI回归测试，不通过则继续修复，直到回归通过。禁止未验证的修复。**

```
┌─────────────────────────────────────────────────────┐
│  修复-回归循环（对每个问题重复）                        │
│                                                     │
│  取一个待修复问题                                     │
│      ↓                                              │
│  修复代码                                            │
│      ↓                                              │
│  pnpm run build 编译验证                             │
│      ├─ 编译失败 → 继续修复 → 重新编译               │
│      └─ 编译通过 ↓                                   │
│  UI回归测试（重新执行该问题关联的Play步骤）             │
│      ├─ 回归失败 → 分析原因 → 继续修复 → 重新回归     │
│      ├─ 回归通过，但引入新问题 → 记录新问题 → 继续修复  │
│      └─ 回归通过，无新问题 → ✅ 该问题关闭            │
│      ↓                                              │
│  下一个问题（回到循环起点）                             │
│                                                     │
│  全部问题关闭 → 退出循环 → 进入Phase 5               │
└─────────────────────────────────────────────────────┘
```

**回归验证要求**：
- 必须重新执行该问题所在的Play步骤（dev-server真实验证）
- 必须检查修复是否引入新问题（运行前后Play步骤对比）
- P0/P1问题修复后，额外执行一次冒烟测试（Phase 2的验证项）
- 每个修复的回归结果记录到评测报告

### Phase 5: 架构审查 + 修复回归（20min）

**目标**: 全局架构合规检查，不依赖Play流程，直接审查代码结构。

| 审查维度 | 标准 | 检查方法 |
|---------|------|---------|
| 文件行数 | ≤500行（预警400行） | `find src/ -name "*.ts" -o -name "*.tsx" \| xargs wc -l \| sort -rn \| head -20` |
| 单一职责 | 每个文件职责单一 | 代码审查 |
| DDD分层 | 依赖方向单向（UI→Engine→SubSystem→Config） | import检查 |
| ISubsystem合规 | 接口实现率100% | `grep -rn "implements ISubsystem" engine/` |
| 门面违规 | 无跨层直接引用 | `grep -rn "from.*engine/(resource\|building\|...)" src/components/` |
| 类型安全 | as any零容忍 | `grep -rn "as any" src/engine/` |
| 死代码 | 无废弃文件/bak目录 | `find . -name "bak" -type d` + `find . -name "exports-v*" -type f` |
| 禁止模式 | 无alert/prompt/Math.random伪造/engine:any | grep扫描 |
| data-testid | 关键组件根元素有data-testid | `grep -rn "data-testid" src/components/` |

审查结果写入 `tech-reviews/{prd-code}-review-{round}.md`。

**发现架构问题后，同样执行修复-回归循环**：
1. 修复架构问题 → `pnpm run build` → UI回归测试（Phase 2冒烟项）→ 不通过则继续修复
2. 全部通过 → 进入 Phase 6

### Phase 6: 封版判定 + 复盘进化（15min）

#### 6.1 Plan 覆盖度检查

用Plan功能清单反向验证Play覆盖度：

```
Plan中的每个功能点 → 是否被至少一个Play步骤覆盖？
    ├─ 有覆盖 → ✅
    └─ 无覆盖 → 追加Play步骤 → 补测 → 修复（如需要）
```

#### 6.2 封版判定

| 判定条件 | 要求 |
|---------|------|
| 静态检查 | TypeScript编译0错误 + 全量测试通过 |
| 冒烟测试 | Phase 2所有验证项通过 |
| 深度评测 | Phase 3发现的所有问题已修复并回归通过 |
| 架构审查 | Phase 5无P0/P1遗留 |
| Plan覆盖度 | 所有功能点被Play覆盖且验证通过 |

- **通过** → 6.3 复盘进化 → commit → 进入下一个迭代单元
- **不通过** → 回到 Phase 2，开启新一轮（R+1）

#### 6.3 复盘进化

| 步骤 | 动作 | 产出 |
|------|------|------|
| 复盘 | 记录经验教训 | `lessons/{version}-lessons.md` |
| 进化 | 分析本轮问题模式，修订进化方案 | `evolution/evolution-{round}.md` |
| 实施 | 落地进化方案（开发/测试新规则或工具） | 代码变更+验证 |

进化方案必须覆盖 [五、进化方向体系](#五进化方向体系) 中的至少一个方向。

## 五、进化方向体系

> 进化方向是评测迭代中**持续改进的维度**。每轮复盘时，分析问题属于哪个方向，制定对应的改进措施。
> 进化规则累积在 `evolution/INDEX.md`（当前 EVO-001~EVO-072）。

### 方向1: 评测工具进化

**目标**: 提升评测效率、覆盖率和准确性。

| 进化维度 | 具体内容 |
|---------|---------|
| 工具能力扩展 | UITreeExtractor 新增适配器（ConsoleErrorCollector / AccessibilityChecker / StyleAnalyzer / DataConsistencyChecker） |
| 测试脚本复用 | 封装常用操作为 game-actions 库，跨版本复用 |
| 自动化提升 | Playwright E2E 覆盖核心流程，截图基线对比防回归 |
| 工具选择策略 | 快速验证用Console脚本、流程验证用Playwright、探索测试用Agent Browser、回归验证用截图+UITreeDiffer |

**相关EVO规则**: EVO-004, EVO-007, EVO-011, EVO-013, EVO-014, EVO-019, EVO-020, EVO-028, EVO-038, EVO-041, EVO-044, EVO-050

### 方向2: 架构审查进化

**目标**: 确保代码结构持续符合DDD规范，防止架构腐化。

| 进化维度 | 具体内容 |
|---------|---------|
| 代码组织检查 | 文件行数≤500行（预警400行）、单一职责、目录按业务域纵向划分 |
| DDD合规检查 | 依赖方向单向（UI→Engine→SubSystem→Config）、子系统API清晰 |
| 门面违规检测 | grep扫描UI层是否绕过门面直接引用引擎子系统 |
| 类型安全 | as any零容忍（EVO-054）、ISubsystem同步实现率100% |
| 死代码治理 | @deprecated标记→下一轮删除、bak/目录全局扫描 |

**相关EVO规则**: EVO-001, EVO-005, EVO-008, EVO-009, EVO-010, EVO-015, EVO-017, EVO-046~048, EVO-053~055

### 方向3: 游戏基础设施进化

**目标**: 强化引擎核心能力，支撑更复杂的游戏功能和更可靠的评测。

| 进化维度 | 具体内容 |
|---------|---------|
| 子系统注册标准化 | 新增系统必须通过6项接入清单（create/register/init/reset/getter/export） |
| 引擎API健壮性 | getState()返回类型安全的默认值、try-catch保护构造函数 |
| 事件系统统一 | 消除双系统重叠、统一触发链路（参考Round 6事件重构） |
| 持久化安全 | localStorage读写包裹try/catch、损坏存档自动恢复 |
| 白屏防护 | ErrorBoundary包裹主组件、引擎崩溃时显示降级UI |

**相关EVO规则**: EVO-006, EVO-024~027, EVO-030, EVO-036, EVO-037, EVO-040, EVO-042, EVO-065~072

### 方向4: PRD/UI/PLAN/PLAY流程完善

**目标**: 确保设计文档和实际实现之间的一致性，减少因文档模糊导致的问题。

| 进化维度 | 具体内容 |
|---------|---------|
| Play文档完善 | 每个版本评测前补充Play流程，覆盖主流程+关联系统+条件限制+边界场景 |
| Gap分析 | Plan功能↔PRD需求↔Play覆盖的交叉验证，识别遗漏和矛盾 |
| PRD一致性 | 评测中发现PRD矛盾时记录并修正（如v20.0发现9处PRD矛盾） |
| 模块命名对齐 | 版本功能模块命名与实际目录一致，以实际目录为准 |
| 文档质量 | PRD数值表完整、UI布局文档精确、交互规范代码化（EVO-045） |

**相关EVO规则**: EVO-031, EVO-042, EVO-044, EVO-045, EVO-051, EVO-055

### 方向5: 启发式进化

**目标**: 通过"感知→适应→验证"闭环，让评测方法像生物一样持续进化。

| 进化维度 | 具体内容 |
|---------|---------|
| 感知 | 每轮记录真实运行数据（非推理结论）：截图、Console输出、组件树统计 |
| 适应 | 基于感知数据触发启发式规则（H-1~H-7），自动改进评测方法 |
| 验证 | 量化对比前后轮次指标（问题发现数、修复率、耗时、回归率） |
| 知识积累 | 常见问题模式库(CL-XXX)、评测目标库(RT-XXX)、技能库(SKILL-XXX)、教训库(LL-XXX) |
| 规则进化 | 启发式规则命中后验证有效性，有效则正式加入规则库 |

**启发式规则**:

| 规则 | 触发信号 | 改进动作 |
|------|---------|---------|
| H-1 | Console有Error但代码审查未发现 | 增加Console监控检查点 |
| H-2 | 修复后回归测试仍发现同类问题 | 对同类问题做全量扫描 |
| H-3 | 某工具连续2轮未发现问题 | 降级或替换该工具 |
| H-4 | 某类问题多轮反复出现 | 升级为架构级问题一次性解决 |
| H-5 | 新问题无现有工具可检测 | 创建新工具或扩展现有工具 |
| H-6 | 截图与预期不一致 | 建立截图基线，增加对比检查 |
| H-7 | 评测耗时持续增长 | 分析瓶颈，自动化高频检查项 |

**进化度量**:

| 指标 | 目标趋势 |
|------|---------|
| 问题修复率 | 持续上升→稳定>90% |
| 回归率 | 持续下降→<5% |
| 评测效率（问题数/耗时） | 持续上升 |
| 工具覆盖率 | 持续上升 |
| 知识库利用率 | →100% |

### 方向6: 数值平衡与经济系统验证

**目标**: 确保游戏数值设计的合理性，防止经济系统失衡。

| 进化维度 | 具体内容 |
|---------|---------|
| 5维数值审查 | 资源产出/战力成长/难度曲线/经济闭环/倍率叠加，每轮至少检查1维 |
| 资源链完整性 | 验证每种资源的获取→消耗→产出路径是否闭环（如：无装备获取路径=断链） |
| 数值表对齐 | 引擎实现数值与PRD数值表100%一致，偏差自动检测 |
| 长线平衡 | 后期版本（v14+）验证转生/声望/传承系统的数值是否合理 |

### 方向7: 性能优化与监控

**目标**: 保持游戏流畅运行，防止性能退化。

| 进化维度 | 具体内容 |
|---------|---------|
| 监控先行 | 先写PerformanceMonitor建立基线，再针对性优化（EVO-043） |
| 关键指标 | FPS ≥30、页面加载 <3s、引擎tick <100ms |
| 优化工具箱 | ObjectPool（对象复用）、DirtyRectManager（脏矩形）、组件树惰性渲染 |
| 构建产物检查 | 每次进化验证 pnpm build 成功且入口chunk非空（EVO-071） |
| 内存泄漏 | 长时间运行后内存不持续增长（EventBus解绑、定时器清理） |

### 方向8: 回归防护体系

**目标**: 防止修复引入新问题，保障每轮进化不倒退。

| 进化维度 | 具体内容 |
|---------|---------|
| 修复后重验证 | 每个修复后立即运行测试+编译+dev-server验证 |
| 快照对比 | UITreeDiffer组件树快照、Playwright截图基线对比 |
| 冒烟测试 | 每轮首先执行引擎API冒烟测试（EVO-041），确认基础可用 |
| 全链路验证 | 收官版本必须做端到端全链路（资源→建筑→武将→战斗→转生→声望） |
| CI防护 | 冲突标记检测（EVO-039）、禁止模式扫描 |

### 方向9: 可访问性与兼容性

**目标**: 确保游戏在不同设备和用户场景下的可用性。

| 进化维度 | 具体内容 |
|---------|---------|
| 响应式适配 | 覆盖375px-768px断点，移动端触摸热区≥44px |
| 键盘可访问 | Tab/Enter/ESC键盘导航，不依赖hover |
| 色彩无障碍 | 色盲友好的配色方案（参考v20.0色彩无障碍检查） |
| ARIA合规 | 关键UI元素有aria-label，弹窗有role="dialog" |
| 浏览器兼容 | 主流浏览器（Chrome/Firefox/Safari）核心功能一致 |

## 六、多轮迭代循环

### 6.1 迭代收敛模型

```
迭代单元: v1.0 基业初立

R1 ──▶ 发现5个问题 → 修复→回归 → 仍剩2个
R2 ──▶ 发现2遗留+1回归 → 修复→回归 → 仍剩1个
R3 ──▶ 发现1遗留 → 修复→回归 → 0个 → ✅ 封版
```

**封版判定**: 连续1轮无新问题 + 所有检查点通过 + UI回归验证通过 = 封版。

### 6.2 多版本流水线

```
迭代单元1(v1) ──R1──▶──R2──▶──R3──✅
                                    │
迭代单元2(v2) ──────────────────────R1──▶──R2──✅
                                                        │
迭代单元3(v3+v4) ───────────────────────────────────────R1──▶──R2──✅
```

- 同一时间只处理一个迭代单元
- 知识库跨迭代单元共享（v2的教训应用到v3）

### 6.3 迭代轮次上限

| 场景 | 上限 | 超限处理 |
|------|------|---------|
| 单个迭代单元 | 5轮 | 暂停，分析根因，考虑架构级重构 |
| 连续3个单元都超3轮 | — | 暂停，重新评估评测方法有效性 |
| 同类问题3个单元重复出现 | — | 升级为架构级问题，一次性解决 |

### 6.4 全局进化循环

所有版本处理完毕后：
1. **全局复盘** — 汇总所有经验教训，识别跨版本共性问题
2. **进化方法升级** — 基于全局视角修订进化规则、检查清单、工具体系
3. **重启新一轮** — 从v1重新开始，应用升级后的进化方法

## 七、白屏零容忍规则

> 白屏是最严重的用户体验问题。以下规则为 P0/P1 级别，每次进化迭代必检。

| 规则 | 级别 | 内容 |
|------|------|------|
| **EVO-065** 白屏零容忍 | P0 | `<div id="root">` 无内容即阻塞；所有UI组件必须有try-catch防御；引擎getState()返回类型安全默认值；ErrorBoundary包裹主组件；禁止未防御的Object.values(null/undefined) |
| **EVO-066** getState()规范 | P0 | 所有子系统getState()必须返回完整默认状态，返回类型禁止unknown，默认状态能安全传入所有方法 |
| **EVO-067** UI防御性编程 | P0 | 引擎获取数据做空值守卫；Object.values/keys前确保参数非null/undefined；子组件渲染失败不导致父组件白屏（错误边界隔离） |
| **EVO-068** ErrorBoundary强制 | P0 | 所有游戏主组件必须被ErrorBoundary包裹，引擎崩溃时显示错误信息页而非白屏 |
| **EVO-069** 引擎构造安全 | P0 | ThreeKingdomsEngine构造必须在try/catch内，构造失败进入安全降级UI |
| **EVO-070** 存档容错 | P1 | localStorage读取try/catch包裹，损坏数据自动清除并提示用户 |
| **EVO-071** 构建产物检查 | P1 | 每次进化验证 pnpm build 成功 + 游戏入口JS chunk > 0 bytes |
| **EVO-072** 基本功能可用性 | P1 | 每轮验证页面加载、引擎初始化、资源栏显示、Tab切换4项基本功能 |
