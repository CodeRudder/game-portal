# 游戏迭代进化

>范围：v1-v20所有的游戏版本.
>粒度：每次评测改进一个版本.
>编译要求：pnpm run build 
>任务执行方式： 主会话负责编排任务，使用task(subagent)执行每个步骤.

## 迭代进化流程：
### 1、读取Next游戏版本计划(plans/vx-xxx.md)的功能清单. 
1. evolution : three-kingdoms/process/evolution-rules.md
2. 评测规范: three-kingdoms/checklist/ui-checklist.md, three-kingdoms/architecture/ddd-architecture.md.
3. 进化规范: three-kingdoms/evolution/

### 2、按功能清单的每个功能点循环处理：
#### 2.1、读取功能点关联的PRD/UI章节内容. 
追加本轮的处理进度 (three-kingdoms/evolution/progress/evolution-progress-{round}.md) 

#### 2.2 认真推理，补充完善玩游戏流程（three-kingdoms/play/{version}-play.md）
2.2.1 每个子系统的主要业务流程是否跑通，如装备系统的主流程：获取装备 -> 穿戴/脱下装备 -> 锻造升级装备 -> 分解回收装备。
2.2.2 梳理本轮测试的关联系统，验证关联流程是否正确：武将升星->增加四维属性->武将主界面是否刷新属性->战斗是否有效。
2.2.3 梳理每个功能点的条件及限制，并进行验证。
2.2.4 查规则补充其它常见的流程，让Agent更真实地玩游戏评测。

#### 2.3、UI测试：使用常见UI测试工具及自研测试工具(UITreeExtractor)等检查功能点实际实现情况, 玩游戏流程。
编译通过并根据需要重启dev-server，一般热加载不需要重启。
##### 2.3.1 将常用功能操作编写为脚本，支持重复使用，如游戏初始化-“打开游戏页面url关闭相关弹窗并显示主界面”。
##### 2.3.2 复用测试的浏览器实例（先考虑非无头模式），保留状态/现场，支持多个任务连续测试（恢复测试或者改进测试）。
##### 2.3.3 将发现的问题记录到文档（three-kingdoms/ui-reviews/{prd-code}-review-{round}.md ）。

#### 2.4、修复问题，重新跳转到2.3进行UI测试步骤。

#### 2.5、技术审查：检查代码是否已经实现功能点且符合架构设计规范
单一职责+DDD设计+4层：引擎层/逻辑层/UI层/渲染层+每个代码文件不超过500行）.
生成审查结果文档（three-kingdoms/tech-reviews/{prd-code}-review-{round}.md ）。

#### 2.6、修复问题：修复技术审查发现的问题,编译通过.

#### 2.7、修复当前迭代版本的问题后，复盘评测过程，记录经验教训( three-kingdoms/lessons/xxx.md, 二级索引  )

#### 2.8、补充修订进化方法( three-kingdoms/evolution/xxx.md, 二级索引  )
包括不限于修订评测规则、UI检查清单、引入或者改进测试工具、优化测试流程、重构游戏引擎/基础设施支持更好评测、完善PRD/UI等。需要带启发式思维进行思考，在多轮迭代中持续进化。需要每个迭代进化的日志。

#### 2.9、实现修订的进化方案，根据情况进行安排任务，包括开发、测试进化方案规则/工具是否达到预期。

#### 2.10、当前功能点修复问发现的问题后，更新处理进度(evolution-progress-{round}.md)

#### 2.11 跳转到第2点继续处理下一个功能点。

### 3、当前版本评测及修复完成后
#### 3.1 进行版本评测复盘，重复2.7-2.8步骤。
#### 3.2 继续跳转到第1点进行下一个版本的处理。

### 4、所有迭代版本处理完全后
#### 4.1 进行每轮评测复盘，重复2.7-2.8步骤。
#### 4.2 重新跳转到第1点，处理第一个版本，开启新一轮进化。

## 白屏零容忍规则

### EVO-065: 白屏零容忍规则
- **触发条件**: 任何页面加载后 `<div id="root">` 无内容
- **检查方法**: 每次提交前必须通过白屏E2E测试
- **强制要求**: 
  1. 所有UI组件必须有 try-catch 防御
  2. 引擎 getState() 方法必须返回类型安全的默认值
  3. React 错误边界必须包裹游戏主组件
  4. 不允许 `Object.values(null/undefined)` 未防御调用

### EVO-066: 引擎 getState() 规范
- 所有子系统的 getState() 必须返回该子系统的完整默认状态
- 返回类型必须是具体类型，禁止 `unknown`
- 默认状态必须能安全传入该子系统的所有方法

### EVO-067: UI组件防御性编程规范
- 所有从引擎获取的数据必须做空值守卫
- Object.values/Object.keys 调用前必须确保参数非null/undefined
- 子组件渲染失败不应导致父组件白屏（错误边界隔离）

---

### EVO-068: 白屏防护 — ErrorBoundary 强制要求
- **级别**: P0 (阻塞)
- **内容**: 所有游戏主组件必须被 ErrorBoundary 包裹，确保任何渲染/初始化异常不会导致白屏
- **触发**: 每次进化迭代的技术审查必须检查 ErrorBoundary 覆盖率
- **验收**: 访问游戏路由时，即使引擎崩溃也应显示错误信息页而非白屏

### EVO-069: 引擎构造安全 — try/catch 保护
- **级别**: P0 (阻塞)
- **内容**: ThreeKingdomsEngine 构造必须在 try/catch 内执行，不允许在 React 渲染阶段裸构造
- **触发**: 代码审查时检查引擎实例化代码
- **验收**: 引擎构造失败时组件进入安全的降级UI状态

### EVO-070: 存档损坏容错 — 自动恢复
- **级别**: P1 (重要)
- **内容**: localStorage 读取必须包裹 try/catch，损坏数据自动清除并提供用户提示
- **触发**: 每次引擎 load() 操作
- **验收**: 损坏存档不会导致崩溃，用户可选择清除存档重试

### EVO-071: 构建产物白屏检查
- **级别**: P1 (重要)
- **内容**: 每次进化迭代必须执行 pnpm build 验证构建成功，且构建产物中游戏入口chunk存在且非空
- **触发**: 每次提交前
- **验收**: pnpm build 成功 + 游戏入口JS chunk > 0 bytes

### EVO-072: 基本功能可用性验证
- **级别**: P1 (重要)
- **内容**: 每次进化迭代必须验证游戏基本功能可用：页面加载、引擎初始化、资源栏显示、Tab切换
- **触发**: 每次封版测评
- **验收**: 4项基本功能全部正常，任一失败则阻塞封版
