## 经验教训

### LL-v1-001: 重复代码是最常见的质量问题
**现象**：resource-calculator.ts 和 OfflineEarningsCalculator.ts 有3个完全相同的函数。
**根因**：代码提取后原文件未清理。
**教训**：每次代码提取/迁移后，必须检查原文件是否已清理。建立"提取即删除"规则。

### LL-v1-002: 文件行数超限通常因为职责混合
**现象**：BuildingSystem.ts 612行，包含推荐逻辑。
**根因**：推荐方法与核心状态管理职责不同。
**教训**：当文件接近500行时，主动检查是否有可提取的职责。推荐/策略类逻辑适合独立为纯函数模块。

### LL-v1-003: 内联弹窗违反单一职责
**现象**：BuildingPanel.tsx内联了收支详情弹窗。
**根因**：快速开发时直接在面板中写弹窗，未考虑职责分离。
**教训**：弹窗应始终独立为组件，使用SharedPanel统一容器。建立"弹窗独立"规则。

### LL-v1-004: 废弃文件是技术债的温床
**现象**：旧版ResourceBar和BuildingPanel文件仍存在。
**根因**：重构时创建了新文件但未删除旧文件。
**教训**：每次重构完成后，立即清理废弃文件。grep验证无引用后删除。

### LL-v1-005: 可复用测试基础设施是高效评测的关键
**现象**：创建game-actions.cjs后，测试脚本从~200行降至~50行。
**根因**：提取通用操作（进入游戏、切换Tab、截图等）为可复用函数。
**教训**：在第一个版本的评测中就建立测试基础设施，后续版本直接复用。

### LL-v1-006: 技术审查应先于UI测试
**现象**：先做技术审查发现代码问题，修复后再做UI测试，避免在问题代码上测UI。
**根因**：代码质量问题可能影响UI表现。
**教训**：评测流程应为：技术审查→修复代码→UI测试→修复UI。这个顺序是正确的。

### LL-v1-007: DDD四层分离模式验证有效
**现象**：资源系统和建筑系统的聚合根/计算器/配置/类型四层分离效果良好。
**根因**：DDD设计规范清晰，执行到位。
**教训**：DDD四层分离是三国霸业项目的有效架构模式，后续版本应继续遵循。
