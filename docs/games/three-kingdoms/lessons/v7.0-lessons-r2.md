# v7.0 草木皆兵 — Round 2 复盘 (装备域)

## LL-XXX-1: 聚合根+委托模式是控制文件行数的最佳实践

**分类**: 架构设计

**观察**: EquipmentSystem 作为聚合根仅 412 行, 通过委托 EquipmentBagManager(211行) 和 EquipmentDecomposer(129行) 管理背包和分解。EquipmentForgeSystem(311行) 委托 ForgePityManager(138行) 管理保底。每个文件职责单一且行数可控。

**教训**: 当聚合根超过 300 行时, 应主动识别可委托的子职责(如背包管理、保底计数), 拆分为独立 Manager 类。委托通过构造函数注入回调(emitEvent)保持解耦, 不引入循环依赖。

**适用场景**: 任何域的聚合根行数增长时。

## LL-XXX-2: RNG 注入模式决定测试确定性

**分类**: 测试策略

**观察**: EquipmentForgeSystem 内部直接调用 `Math.random()`, 导致测试需要 mock 全局对象。而 EquipmentEnhanceSystem 使用 `rngState` 内部状态 + 可选 `rng` 参数注入, 测试时可传入确定性函数。

**教训**: 涉及随机的系统应在核心方法上提供 `rng?: () => number` 可选参数。生产环境不传则用默认 `Math.random()`, 测试环境传入确定性序列。这比 mock `Math.random` 更优雅, 也比内部 LCG 更灵活。

**适用场景**: 任何包含随机逻辑(掉落、锻造、强化成功率)的系统。

## LL-XXX-3: 构建缓存可产生虚假失败, clean build 是最终裁判

**分类**: 工程实践

**观察**: `pnpm run build` 首次因 dist 缓存报 `Could not resolve "./RebirthSystemV16"` 错误, 但源码中无此导入。`rm -rf dist && build` 后成功通过(20.91s)。这是 Vite/Rollup 增量构建缓存了已删除文件的引用。

**教训**: 当构建报出看似不存在的文件引用错误时, 先执行 clean build 再判断。CI 环境应始终 clean build。本地开发如遇诡异构建错误, 删除 dist/ 和 node_modules/.vite 是第一步排查手段。

**适用场景**: 任何遇到"文件找不到但代码中没有引用"的构建错误时。
