# v9.0 离线收益 — 经验教训

> **日期**: 2026-04-22
> **版本**: v9.0 离线收益
> **审查轮次**: R1

---

## LL-v9-001: 编译错误修复 — 重复导出是常见问题

- **背景**: 新增 exports-v9.ts 文件后，与已有 exports-v11/exports-v12 中的 LeaderboardSystem 产生重复导出冲突（TS2308）
- **问题**: 版本合并时不同 exports-v{N}.ts 文件可能导出同名符号
- **教训**: 新增 exports 文件时必须立即 `pnpm run build` 验证无重复导出
- **规则**: EVO-031 — 新增exports文件时验证无重复导出
- **影响**: 后续所有新增版本导出文件

## LL-v9-002: 引擎集成模式 — 6步清单逐项验证

- **背景**: 离线收益系统（OfflineRewardSystem/OfflineSnapshotSystem/OfflineEstimateSystem）代码完整，但完全没有集成到主引擎中，形同虚设
- **问题**: 三个子系统未在构造函数创建、未注册、未初始化、未重置、无getter、无门面导出
- **教训**: EVO-030 的6步检查清单有效，但执行时容易遗漏（如 MailTemplateSystem 被遗漏），需要更系统化地逐项验证
- **规则**: EVO-032 — 引擎集成6步清单必须逐项验证，每步验证编译
- **影响**: 此后所有新增子系统接入引擎的流程

## LL-v9-003: 类型统一原则 — 同一域的类型定义应集中

- **背景**: MailSystem.ts 和 mail.types.ts 各自定义了同名但结构不同的类型（MailCategory/MailStatus/MailAttachment/MailData/MailSaveData），导致 mail/index.ts 不得不使用 V2 别名导出
- **问题**: `MailCategory` 在两文件中值不同（`'battle'` vs `'combat'`），`MailAttachment` 结构完全不同，`MailData` 字段名不同（`content` vs `body`）
- **教训**: 同一域的类型定义应集中在 `{domain}.types.ts` 文件，其他文件只导入不定义。违反标志：出现 `TypeV2`、`TypeAlias` 等别名导出
- **规则**: EVO-033 — 类型定义单一数据源原则
- **影响**: 所有新增域的类型文件组织方式

## LL-v9-004: 门面拆分策略 — exports-v{N}.ts 模式成熟

- **背景**: engine/index.ts 达到 616 行，超过 500 行限制
- **问题**: 新增 offline/ 和 mail/ 域的导出后门面文件膨胀
- **教训**: 按版本拆分到 exports-v{N}.ts 已是成熟模式（v6/v9/v10/v11/v13/v17 均采用），应在新增域时主动拆分而非等到超限
- **影响**: 后续版本新增域时，同步创建 exports 文件

## LL-v9-005: UI组件缺失处理 — 引擎完整但UI缺失标记为"警告"

- **背景**: v9.0 的5个专用UI面板组件（OfflineRewardModal/OfflineSummary/OfflineEstimate/MailPanel/MailDetail）未实现，但引擎层数据组装完整
- **问题**: 离线收益弹窗需要存档数据才能触发，UI测试无法模拟完整离线场景
- **教训**: 引擎层完整但UI组件缺失的功能点，在UI测试中应标记为"警告"而非"失败"，避免误导。可在后续迭代中补全
- **影响**: UI测试报告的准确性和测试策略

## LL-v9-006: 子任务超时应对 — 复杂修复任务应拆分

- **背景**: 离线系统集成涉及多个文件修改（ThreeKingdomsEngine/engine-getters/exports-v9/mail类型统一），单任务容易超时
- **教训**: 复杂修复任务应拆分为 ≤10分钟的小任务（如：1.创建deps文件 2.集成构造函数 3.集成注册 4.集成初始化 5.添加getter 6.更新exports）
- **影响**: 任务执行效率

## LL-v9-007: data-testid补充应作为技术审查固定检查项

- **背景**: 技术审查发现多个UI组件缺少 data-testid 属性，导致UI测试定位困难
- **教训**: data-testid 补充应纳入技术审查的固定检查项，而非临时发现临时补
- **规则**: 强化 EVO-014（data-testid强制规则）
- **影响**: 后续所有版本的技术审查流程
