## LL-167: 科技域七子系统拆分是DDD最佳实践标杆

**背景**: v5.0 科技域拆分为7个ISubsystem + 2个工具类 + 2个类型文件 + 1个配置文件 = 13文件。

**经验**: 
- 聚合根 TechTreeSystem(420行) 只管节点状态和互斥，不碰研究流程
- 流程控制 TechResearchSystem(361行) 专注队列和加速
- 效果分发 TechEffectSystem(413行) + TechEffectApplier(425行) 分离查询与应用
- 联动管理 TechLinkSystem(489行) 独立注册建筑/武将/资源关联
- 融合科技 FusionTechSystem(487行) 独立管理跨路线组合
- 离线研究 TechOfflineSystem(457行) 独立处理衰减计算

**量化**: 平均文件行数 340行，最大487行，零文件超500行限制。7/7 ISubsystem 合规。

**复用建议**: 后续版本新增复杂系统时，参照此模式：聚合根→流程控制→效果计算→辅助功能。

## LL-168: 互斥分支的 chosenMutexNodes 设计简洁有效

**背景**: 科技树三条路线各有2个互斥层(Tier1和Tier3)，共6组互斥。

**经验**:
- 互斥组ID格式 `{path}_t{tier}` 简洁可读（如 `mil_t1`, `eco_t3`）
- chosenMutexNodes 使用 `Record<string, string>` 映射，查询O(1)
- refreshAllAvailability() 在构造和状态变更时统一刷新，避免不一致
- 互斥锁定不可逆设计正确——避免玩家反复切换破坏游戏平衡

**复用建议**: 未来版本如需互斥机制（如阵营选择、技能分支），直接复用此模式。

## LL-169: TechEffectApplier 作为纯工具类不实现ISubsystem是正确选择

**背景**: TechEffectApplier(425行) 和 TechDetailProvider(437行) 未实现 ISubsystem。

**经验**:
- 纯计算层不需要生命周期管理（init/update/reset），强行实现ISubsystem是过度设计
- TechEffectApplier 通过 getBattleBonuses() / getResourceBonuses() / getCultureBonuses() 提供纯函数式接口
- TechDetailProvider 组装详情展示数据，无状态依赖
- 两者作为 export class 而非 ISubsystem，在 index.ts 中正常导出，不影响门面完整性

**复用建议**: DDD中不是所有 export class 都必须是 ISubsystem。纯计算/查询工具类保持轻量，只在需要生命周期管理时才实现接口。
