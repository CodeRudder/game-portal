## 按类别
### 架构设计
- DDD四层分离有效降低耦合，聚合根/计算器/配置/类型的模式值得推广
- 文件行数限制500行是合理的，超过时拆分推荐逻辑是有效策略
- Mixin模式可有效解决引擎主类getter膨胀问题（LL-R1-005）
- 按功能边界拆分比按行数机械切割更有效（LL-R1-001）

### 代码质量
- DRY原则违反是最常见问题（重复函数、重复组件、重复常量）
- 废弃文件未清理是普遍问题，应在每次重构后立即清理
- DDD门面违规可通过grep快速检测（LL-R1-002）
- 同一域的类型定义应集中在 `{domain}.types.ts`，避免V2别名导出（LL-v9-003）
- 新增exports文件后必须验证无重复导出（LL-v9-001）

### 测试方法
- 可复用测试操作库(game-actions)大幅提升测试效率
- Playwright headless模式适合CI，非headless适合调试
- UI测试应结合截图分析，不能仅依赖DOM文字搜索（LL-R1-003）
- 新增UI元素必须添加data-testid属性（LL-R1-003）
- 离线收益弹窗需要存档数据才能触发，测试需模拟数据场景（LL-v9-005）
- 引擎完整但UI缺失的功能点应标记为"警告"而非"失败"（LL-v9-004）

### 工具进化
- 技术审查应先于UI测试，先修代码问题再测UI
- 编译验证(pnpm run build)是必要的，不能只依赖tsc --noEmit
- 进化迭代前必须先编译检查，确保基线可编译（LL-v9-001）
- 子系统集成6步清单必须逐项验证，每步验证编译（LL-v9-002）

### 任务管理
- 子任务代码变更量控制在500行以内，避免超时（LL-R1-004）
- 大任务应拆分为多个小任务，每个预估时间≤10分钟
