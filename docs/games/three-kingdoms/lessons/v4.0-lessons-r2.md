## 经验教训

### LL-v4-r2-001: 类型文件膨胀预警
- **发现**: battle.types.ts 达到476行，接近500行上限
- **影响**: 如后续版本继续添加战斗相关类型，将突破行数限制
- **建议**: 当类型文件超过400行时，考虑按功能域拆分（如 battle-core.types.ts / battle-v4.types.ts / battle-effect.types.ts）
- **适用范围**: 所有引擎模块的类型文件

### LL-v4-r2-002: 测试文件门面引用宽容度
- **发现**: 6处测试文件直接引用 `engine/battle/battle.types` 而非通过门面index.ts
- **分析**: 测试文件仅import type和enum，不涉及运行时依赖，风险可控
- **建议**: 测试文件允许直接引用类型定义文件，但生产代码必须走门面导出
- **规则修订**: DDD门面规则适用于生产代码，测试文件的type-only引用可豁免

### LL-v4-r2-003: 战斗子系统拆分粒度标杆
- **发现**: battle模块16个文件平均281行，最大476行，拆分粒度优秀
- **亮点**:
  - DamageCalculator 独立为391行，便于单独测试和复用
  - BattleSpeedController (306行) 与 UltimateSkillSystem (439行) 各自封装完整
  - DamageNumberSystem (362行) 包含完整的弹道/合并/配置逻辑
  - BattleEffectManager (329行) + BattleEffectApplier (336行) 管理与应用分离
- **建议**: 此拆分粒度可作为后续模块的参考标杆
- **度量**: 文件数16个，平均281行，标准差约120行，分布均匀
