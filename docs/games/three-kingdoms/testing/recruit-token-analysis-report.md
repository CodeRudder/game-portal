## 一、执行摘要

求贤令机制的**技术实现完全正确**，所有集成测试通过（70+ tests）。问题在于**数值设计**：被动产出速率过低（0.001/秒），导致新玩家需等待 16.7 分钟才能进行一次普通招募，体验不佳。

**核心问题**：
- ❌ 新玩家无初始求贤令，无法快速体验招募功能
- ❌ 被动产出速率过低，等待时间过长
- ❌ 缺少主动获取途径（关卡奖励、商店购买等）
- ❌ 资源产出速率显示为 0.0/s，玩家误以为无产出

## 五、改进方案

### 5.1 方案概述

| 改进项 | 当前值 | 改进后 | 提升幅度 |
|--------|-------|--------|---------|
| 新手礼包 | 0 | 10 个 | +10 |
| 被动产出速率 | 0.001/秒 | 0.003/秒 | 3倍 |
| 关卡奖励 | 无 | 10 个/大关卡 | 新增 |
| 产出速率显示 | 0.0/s | 0.003/s（1~3位小数） | 优化显示 |

### 5.2 详细改进方案

#### 改进1: 新手礼包 +10 个求贤令

**目的**: 让新玩家立即体验招募功能

**实现位置**:
- `src/games/three-kingdoms/engine/resource/resource-config.ts`
- 修改 `INITIAL_RESOURCES.recruitToken: 0` → `10`

**影响**:
- ✅ 新玩家可立即进行 10 次普通招募
- ✅ 快速体验核心玩法
- ✅ 提升新手留存率

#### 改进2: 被动产出速率 0.001/秒 → 0.003/秒（3倍）

**目的**: 缩短等待时间，提升体验流畅度

**实现位置**:
- `src/games/three-kingdoms/engine/resource/resource-config.ts`
- 修改 `INITIAL_PRODUCTION_RATES.recruitToken: 0.001` → `0.003`

**产出计算**（改进后）:
- 每秒：0.003 个
- 每小时：10.8 个
- 每天：259.2 个
- 7天：1814.4 个

**影响**:
- ✅ 等待时间从 16.7 分钟缩短到 5.6 分钟
- ✅ 每天可进行 259 次普通招募（vs 86 次）
- ✅ 玩家体验更流畅

#### 改进3: 关卡奖励 +10 个/大关卡

**目的**: 提供正反馈，鼓励玩家推进关卡

**实现位置**:
- `src/games/three-kingdoms/engine/campaign/campaign-config.ts`
- 在关卡奖励配置中添加 `recruitToken: 10`

**触发条件**:
- 每通过一个大关卡（如第1章、第2章等）
- 奖励 10 个求贤令

**影响**:
- ✅ 推进关卡有明确奖励
- ✅ 提升玩家推图动力
- ✅ 增加求贤令获取途径

#### 改进4: 资源产出速率显示优化

**目的**: 避免显示 0.0/s，让玩家看到真实产出

**实现位置**:
- `src/components/idle/panels/resource/ResourceBar.tsx`
- 优化产出速率显示逻辑

**显示规则**:
```typescript
// 根据数值大小动态调整小数位数
if (rate >= 1) {
  return rate.toFixed(1);      // 显示 1 位小数：10.5/s
} else if (rate >= 0.01) {
  return rate.toFixed(2);      // 显示 2 位小数：0.05/s
} else if (rate > 0) {
  return rate.toFixed(3);      // 显示 3 位小数：0.003/s
} else {
  return '0';                  // 真正的 0
}
```

**影响**:
- ✅ recruitToken 显示为 0.003/s（vs 0.0/s）
- ✅ 玩家清楚看到资源在增长
- ✅ 提升信息透明度
