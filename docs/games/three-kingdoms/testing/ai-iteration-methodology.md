# AI迭代开发方法论 — 基于三国霸业游戏三轮迭代的实践总结

> 从三国霸业游戏引擎的三轮迭代实践中提炼的可复用方法论
> 版本: 1.0 | 2026-04-25

---

## 一、方法论概述

### 1.1 核心理念

**"测评驱动的渐进式迭代"** — 通过结构化的评测-改进循环，将AI生成代码的质量从初始的C级逐步提升至9+/10分。

### 1.2 适用场景

- AI Agent驱动的复杂软件开发项目
- 需要多轮迭代才能达到质量标准的产品
- 有明确PRD/设计文档的项目
- 需要同时保证功能完整性和代码质量的项目

### 1.3 三轮迭代演进

```
R1: 设计稿驱动 → 视觉迭代(R1~R16) → 功能迭代(R1~R7) → 可玩版本
R2: 进化循环(R1~R42) → 每轮7步评测-改进 → 全版本封版 → 79+进化规则
R3: 集成测试驱动 → 6步标准化流程 → 双角色评审 → 量化封版门槛
```

**演进方向**: 模糊 → 结构化 → 量化 → 自动化

## 二、迭代模式详解

### 2.1 模式一：测评-改进循环（R1/R2）

```
┌─────────────────────────────────────────┐
│  评测 → 发现问题 → 修复 → 重测 → 封版   │
│  ↑                                       │
│  └───── 未达标准则继续循环 ───────────────┘
│
│  特点: 简单直接，但缺少标准化
│  适用: 早期探索阶段、视觉优化
│  风险: 评测标准模糊，容易"感觉良好"就封版
```

**R1实践**: 视觉迭代16轮(R1~R16)，从C级提升至7.6/10
**R2实践**: 功能迭代42轮(R1~R42)，79+进化规则，连续7轮零P0

**关键经验**:
- 评测标准必须提前定义（R1早期没有标准，导致反复）
- 进化规则（EVO-001~079）是宝贵的知识积累，避免同类问题重复出现
- "连续N轮零P0"是有效的封版信号

### 2.2 模式二：6步标准化流程（R3）

```
① 修正集成测试脚本 → 执行测试发现问题
② 修复测试脚本 + 修复代码问题
③ 重测确认全通过
④ 游戏评测师评审 + 修复
⑤ 架构评审 + 修复
⑥ 封版提交

特点: 双角色评审（功能+架构），每步有明确输入输出
适用: 质量要求高的版本封版
优势: 发现问题的系统性更强
```

**R3实践**: 20版本，19封版，12个P0修复，平均评测8.4/架构8.7

**关键经验**:
- 评测师和架构师分离，避免"功能好就忽略代码质量"
- 集成测试方法文档是标准化的基础
- 批量测试+逐版本评审效率最高

## 三、核心方法论要素

### 3.1 五大核心原则

#### 原则一：单一权威源（Single Source of Truth）

**问题**: R3发现50%的P0问题是"PRD数值与代码不一致"

**方案**:
```
PRD文档 → 提取数值常量 → 共享配置文件 ← 代码引用
                                    ↑
                              测试断言引用
```

**实践规则**:
- 所有数值（概率/倍率/阈值/公式）必须从配置文件读取
- PRD文档中的数值表必须与配置文件一一对应
- 测试断言引用配置文件而非硬编码数值
- 修改配置时自动触发全量回归测试

#### 原则二：版本Scope控制

**问题**: v13.0联盟争霸scope过大（联盟+活动+活跃度+代币），导致5个核心引擎类缺失

**方案**:
```
版本规划检查清单:
□ 新增子系统 ≤ 3个
□ 新增源文件 ≤ 15个
│
├─ 超出 → 拆分为两个版本
└─ 未超出 → 可进入开发
```

**实践规则**:
- 每版本新增ISubsystem实现不超过3个
- Play文档章节不超过10个
- 集成测试文件新增不超过10个
- 超出时必须拆分版本或标记为"多版本交付"

#### 原则三：Skip测试硬上限

**问题**: 6+个版本存在skip积压，v4有52项skip

**方案**:
```
Skip测试管理规则:
- 每版本skip上限: ≤ 5个
- 超出5个: 必须修复或移除测试
- Skip原因必须标注: "未实现API" / "后续版本" / "已知限制"
- 封版前: 所有skip必须有对应的后续版本计划
```

#### 原则四：双角色评审

**问题**: 单一评审容易遗漏（功能好但代码差，或代码好但功能缺）

**方案**:
```
评测师评审（功能维度）:
- Play流程完整性: 每个玩法流程是否有引擎实现
- PRD一致性: 数值/公式/阈值是否对齐
- 合理性: 游戏设计逻辑是否自洽

架构师评审（代码维度）:
- ISubsystem接口: 是否完整实现
- 类型安全: as any数量
- 模块边界: 文件行数/耦合度
- 代码质量: Jest残留/@deprecated/TODO
```

#### 原则五：量化封版门槛

**问题**: 封版标准模糊，"有条件封版"vs"不可封版"边界不清

**方案**:
```
封版门槛（必须全部满足）:
□ 评测评分 ≥ 8.0/10
□ 架构评分 ≥ 8.0/10
□ P0问题 = 0
□ Skip测试 ≤ 5个
□ 集成测试通过率 = 100%（非skip）
□ 编译错误 = 0
□ as any = 0（新增）
□ 超500行文件 = 0（新增）

条件封版（需标注原因和后续计划）:
- 评测7.0~7.9 且有明确改进路径
- Skip 6~15个 且均为后续版本功能
```

### 3.2 任务编排模式

#### 模式A：主会话编排+子任务执行

```
主会话职责:
- 任务规划和拆分
- 子任务创建和监控
- 结果汇总和决策
- ❌ 禁止在主会话执行编码/调研

子任务类型:
- developer: 代码修改（有200次递归限制）
- bash: 命令执行（无递归限制）
- game-reviewer: 游戏评审
```

**关键经验**:
- developer子任务容易递归超限(200次)，简单操作用bash子任务
- 复杂修复拆分为更小的步骤，每个步骤一个子任务
- 子任务prompt必须自包含，不能依赖主会话上下文

#### 模式B：批量+逐个混合

```
批量阶段（高效）:
- 批量运行集成测试 → 一次性确认v4~v20全部通过
- 批量收集质量指标 → 一次性扫描as any/超500行等

逐个阶段（深度）:
- 逐版本评测师评审 → 每版本独立评审报告
- 逐版本架构评审 → 每版本独立架构分析
```

**效率对比**: 纯逐个(20×6步=120步) vs 混合(批量1次+逐个评审20次=21步)

## 四、质量保障体系

### 4.1 三层测试金字塔

```
         ╱╲
        ╱  ╲        冒烟测试（15~26验证点）
       ╱────╲       构建后自动运行，验证白屏/路由/核心渲染
      ╱      ╲
     ╱ 集成测试 ╲    189文件/7,418用例
    ╱────────────╲   验证完整游戏流程（Play文档→引擎→测试）
   ╱              ╲
  ╱   单元测试      ╲  覆盖每个System的核心逻辑
 ╱──────────────────╲  ISubsystem接口/序列化/边界条件
╱                    ╲
```

### 4.2 质量门禁检查清单

```bash
# 每次提交前必须通过的质量门禁
echo "=== as any ===" && grep -rn "as any" src/ --include="*.ts" | grep -v __tests__ | wc -l  # 必须=0
echo "=== 超500行 ===" && find src/ -name "*.ts" ! -path "*__tests__*" -exec wc -l {} + | awk '$1>500' | wc -l  # 必须=0
echo "=== Jest残留 ===" && grep -rn "jest\." src/ --include="*.ts" | grep -v __tests__ | wc -l  # 必须=0
echo "=== ISubsystem ===" && grep -rn "implements ISubsystem" src/ --include="*.ts" | grep -v __tests__ | wc -l  # 必须>0
echo "=== 编译 ===" && pnpm run build  # 必须0错误
echo "=== 测试 ===" && pnpm run test  # 必须0失败
```

### 4.3 进化规则积累

R2积累了79+条进化规则（EVO-001~079+），这些是不可替代的项目知识：

```
规则分类:
- 架构规则: ISubsystem接口规范、模块边界、依赖注入
- 测试规则: 测试命名、断言规范、DOM清理、pool配置
- 代码规则: 零as any、文件行数限制、序列化版本管理
- 流程规则: 封版门槛、skip管理、跨版本回归
```

**关键洞察**: 进化规则是AI Agent自我改进的知识基础，每轮迭代发现的新问题都应提炼为规则。

## 五、常见陷阱与对策

### 陷阱1：测试通过≠功能完整

**案例**: v13.0有372个测试全部通过，但5个核心引擎类不存在。测试基于模拟/桩代码运行。

**对策**:
- 集成测试必须验证"真实引擎类存在"（import检查）
- 测试覆盖率统计排除mock/stub
- 评审时区分"测试通过"和"功能实现"

### 陷阱2：跨版本变更的蝴蝶效应

**案例**: v2.0将普通池hardPityThreshold从100改为Infinity，导致v10.0的unification测试失败。

**对策**:
- 修改核心配置时执行全局搜索（grep所有引用该配置的文件）
- 建立配置-测试映射表
- 修改后运行全量回归而非仅模块测试

### 陷阱3：文档与代码的慢性漂移

**案例**: v1.0~v4.0共6处PRD数值与代码不一致，随时间累积。

**对策**:
- 数值常量集中管理（单一权威源）
- PRD文档标注"以代码为准"或"以文档为准"
- 每次封版时执行文档-代码一致性检查

### 陷阱4：Scope蔓延

**案例**: v13.0 Play文档包含联盟+活动+活跃度+代币+PvP赛季，远超单版本交付能力。

**对策**:
- 版本规划时检查scope（新增子系统≤3个）
- Play文档与版本号严格对应
- 超出scope的功能标记为"未来版本"

### 陷阱5：子任务递归超限

**案例**: developer子任务经常在200次递归后超限，特别是需要读取多个文件+修改代码的复杂任务。

**对策**:
- 复杂任务拆分为更小的原子步骤
- 简单命令执行用bash子任务
- 子任务prompt必须自包含（不依赖主会话上下文）

## 七、总结

### 三轮迭代的本质进化

```
R1: "能不能做出来" → 从0到1
R2: "能不能做好"   → 从1到N（42轮循环）
R3: "能不能做对"   → 从N到✓（量化验证）
```

### 方法论核心公式

```
高质量交付 = 标准化流程 × 双角色评审 × 量化门槛 × 进化规则积累
```

### 适用性声明

本方法论从三国霸业游戏引擎的三轮迭代实践中提炼，适用于：
- AI Agent驱动的复杂软件开发
- 需要多轮迭代的质量敏感项目
- 有明确PRD/设计文档的项目
- 代码+测试+文档三位一体交付的项目

**不适用于**: 简单脚本、一次性工具、无需迭代的确定性任务。
