## 五、经验教训总结

### 5.1 Round 3 核心经验 (LL-R3)

| 编号 | 经验教训 |
|------|----------|
| LL-R3-001 | 版本后缀文件(V15/V16)不一定冗余，需分析职责差异后再决定是否合并 |
| LL-R3-002 | 跨模块类重叠可通过重导出(re-export)保持向后兼容，避免破坏性变更 |
| LL-R3-003 | 文件拆分优先提取辅助函数/数据类型到独立文件，保持主文件职责聚焦 |

### 5.2 跨版本通用经验

#### 架构设计
- DDD四层分离有效降低耦合，聚合根/计算器/配置/类型的模式值得推广
- 文件行数限制500行合理，超过时按功能边界拆分比机械切割更有效
- Mixin模式可有效解决引擎主类getter膨胀问题
- 按功能边界拆分比按行数机械切割更有效

#### 代码质量
- DRY原则违反是最常见问题（重复函数、重复组件、重复常量）
- 废弃文件未清理是普遍问题，应在每次重构后立即清理
- 同一域的类型定义应集中在 `{domain}.types.ts`，避免V2别名导出
- 新增exports文件后必须验证无重复导出

#### 测试方法
- Play文档先行→技术审查→UI测试的流程效率最高
- 可复用测试操作库(game-actions)大幅提升测试效率
- UI测试应结合截图分析，不能仅依赖DOM文字搜索
- 引擎完整但UI缺失的功能点应标记为"警告"而非"失败"
- 条件性UI测试需采用状态注入模式跳过前置条件

#### 任务管理
- 子任务代码变更量控制在500行以内，避免超时
- 大任务应拆分为多个小任务，每个预估时间≤10分钟
- 子系统集成6步清单必须逐项验证，每步验证编译

### 7.3 进化方法论总结

三轮迭代验证了以下方法论的有效性：

```
进化迭代流程（已制度化）:
1. Play文档编写 → 定义功能验证场景
2. 技术审查 → 发现P0/P1/P2问题
3. 问题修复 → 先修代码再测UI
4. UI测试 → E2E自动化+截图分析
5. 经验教训 → 提炼LL编号记录
6. 进化规则 → 修订EVO编号方法论
7. 进度更新 → 更新progress文档
```

关键成功因素:
- **技术审查先行**: 先修代码问题再测UI，避免重复返工
- **编译验证必做**: `pnpm run build` 是必要步骤，不能只依赖 `tsc --noEmit`
- **子系统集成6步清单**: 构造→注册→初始化→重置→getter→导出，缺一不可
- **按DDD域导出**: 禁止按版本号导出(exports-vN反模式)
- **Play文档驱动测试**: 先写玩法流程再设计测试用例
