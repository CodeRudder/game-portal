## 三、经验教训

### LL-R21-001: P0根因——类型系统扩展遗漏
**现象**: 书院(academy)的resourceType为'mandate'而非'techPoint'，ResourceType联合类型缺少'techPoint'。
**根因**: 在前序迭代中添加新资源类型时，只更新了部分文件（building-config.ts和shared/types.ts），但Phase 3深度评测基于旧快照发现了"问题"。实际上P0在Phase 4确认时已经修复。
**教训**: 类型系统扩展时应建立检查清单：①类型定义 ②配置引用 ③测试覆盖 ④文档同步。避免"以为有问题但实际已修"的误报。

### LL-R21-002: PRD/Play/代码三方数值不一致是系统性问题
**现象**: Phase 3发现5处数值不一致（初始粮草/铜钱、离线效率系数、升级时间等）。
**根因**: 三方文档独立维护，缺乏"单一数据源"(Single Source of Truth)机制。PRD定义了规格，Play文档写了测试期望值，代码实现了实际值，三者未强制同步。
**教训**: 应建立"数值权威源"规则——以代码常量为权威源，PRD和Play文档引用代码常量。当发现不一致时，以代码为准修正文档，除非代码明显错误。

### LL-R21-003: DDD分层违规的典型模式——常量反向引用
**现象**: core/mail/mail.types.ts从engine/mail/mail.types.ts导入常量，违反core不依赖engine的原则。
**根因**: 常量最初写在engine层，core层需要使用时直接反向引用，而非将常量上移到core层。
**教训**: 常量定义应遵循"最低公共层"原则——如果core和engine都需要使用的常量，应定义在core层，engine从core导入。建立自动化检查防止此类违规。

### LL-R21-004: Phase 4测试失败数与Phase 2不一致需关注
**现象**: Phase 2全量测试6099通过，Phase 4为6061/6081（20个失败）。
**根因**: 20个失败测试为既有问题（HeroRecruitSystem 19个 + BattleEffectManager 1个），非P0修复引入。
**教训**: 应建立"测试基线"机制——每轮记录测试通过/失败基线数，区分"新引入失败"和"既有失败"。避免混淆修复效果评估。

### LL-R21-005: 大文件预警是技术债的早期信号
**现象**: 63个非测试文件处于400-500行预警区间，其中4个逼近500行上限。
**根因**: 随着功能迭代，文件自然膨胀。虽然有500行红线，但缺乏主动拆分机制。
**教训**: 应建立"400行预警→自动拆分计划"规则。当文件超过400行时，在下一轮迭代中优先安排拆分，而非等到触碰500行红线。

### LL-R21-006: 功能缺失与架构违规应分类管理
**现象**: P1问题包含三类：数值不一致(4个)、功能缺失(2个)、架构违规(1个)、代码质量(1个)。
**根因**: 不同类型的P1问题修复策略和优先级不同，但当前统一标记为P1。
**教训**: P1问题应细化分类标签（P1-NUM数值/P1-FUNC功能/P1-ARCH架构/P1-QUALITY质量），便于按类别批量处理和优先级排序。

## 四、进化规则修订

### EVO-063: 数值权威源规则（来自R21复盘）
当PRD/Play/代码三方数值不一致时，按以下优先级确定权威源：
1. **代码常量**为第一权威源（可运行验证）
2. **PRD规格**为第二权威源（设计意图）
3. **Play文档**为第三权威源（测试期望）
修复方向：以代码为准→修正PRD/Play，除非代码值明显违反设计意图。
每轮Phase 3发现数值不一致时，必须明确"以谁为准"并记录。

### EVO-064: 类型系统扩展检查清单（来自R21复盘）
添加新ResourceType/枚举值时，必须完成以下检查清单：
- [ ] 类型定义文件（shared/types.ts）
- [ ] 配置引用文件（*-config.ts）
- [ ] 生产者配置（building-config.ts等）
- [ ] 消费者逻辑（ResourceSystem.ts等）
- [ ] 测试覆盖（新增类型的测试用例）
- [ ] 文档同步（PRD/Play/README）
Phase 2冒烟测试应包含"类型完整性"检查项。

### EVO-065: 常量最低公共层原则（来自R21复盘）
常量定义应遵循DDD分层规则：
1. core和engine都使用的常量 → 定义在core层
2. 仅engine使用的常量 → 定义在engine层
3. 禁止core层从engine层导入任何内容
建立自动化脚本检测core→engine的反向引用。

### EVO-066: P1问题分类标签规则（来自R21复盘）
P1问题必须附加分类标签：
- **P1-NUM**: 数值不一致（PRD/Play/代码数值偏差）
- **P1-FUNC**: 功能缺失（PRD要求但代码未实现）
- **P1-ARCH**: 架构违规（DDD分层/设计模式违反）
- **P1-QUALITY**: 代码质量（导入缺失/类型不安全等）
每轮Phase 6复盘时按类别统计P1分布，识别系统性薄弱环节。

### EVO-067: 测试基线管理规则（来自R21复盘）
每轮进化记录测试基线：
- Phase 2冒烟测试通过数（基线）
- Phase 4修复后通过数（对比）
- 新引入失败数 = Phase 4失败数 - Phase 2已知失败数
- 新引入失败 > 0 时必须回退修复
避免将"既有失败"误判为"修复引入"。

### EVO-068: 400行预警拆分规则（来自R21复盘）
当非测试文件超过400行时：
1. 在当轮Phase 5架构审查中标记为"预警文件"
2. 在下一轮Phase 1准备中纳入拆分计划
3. 拆分优先级：490+行 > 470-489行 > 400-469行
4. 拆分原则：按职责拆分，不按函数数量拆分
目标：每轮至少拆分2个预警文件。

## 九、v2.0 经验教训

### LL-R21-v2-001: Play文档静默修正是数值偏差的根源
**现象**: 招募概率表PRD与Code存在系统性差异，Play文档主动调整了PRD概率但未同步更新PRD原文。
**根因**: Play文档在测试过程中"修正"了PRD数值（如高级池Uncommon从20%调至40%），但仅标注"v2.0修正"而未更新PRD。导致Phase 3发现PRD↔Code偏差时无法追溯原因。
**教训**: Play文档对PRD的任何修正必须同步更新PRD原文或明确标注修正原因。引入EVO-069规则。

### LL-R21-v2-002: "预留注释"≠"已实现"——功能完成度需量化
**现象**: P0-3每日免费招募在代码中仅有一行注释占位，Phase 1误判为已实现。
**根因**: 缺乏功能完成度的量化标准，注释占位符被等同于功能实现。
**教训**: 建立功能完成度三级标准（0%/50%/80%/100%），Phase 1必须按标准标注。引入EVO-070规则。

### LL-R21-v2-003: P0修复也需遵守文件行数管控
**现象**: Phase 4在HeroRecruitSystem.ts(471行)上追加UP武将+免费招募逻辑，虽未超过500行红线，但文件职责已复合化。
**根因**: P0修复以"快速止血"为导向，忽视了代码组织质量。
**教训**: P0修复新增代码>50行时，应评估是否提取独立模块。建议在EVO-068中补充"修复阶段拆分"条款。

## 十、v2.0 进化规则修订

### EVO-069: Play文档静默修正禁止规则（来自R21-v2复盘）
Play文档对PRD数值的任何调整必须满足以下条件之一：
1. 同步更新PRD原文（推荐）
2. 在Play文档中明确标注"PRD修正：原值X→新值Y，原因Z"，并在Phase 1中确认
禁止Play文档"静默修正"PRD数值而不留记录。
