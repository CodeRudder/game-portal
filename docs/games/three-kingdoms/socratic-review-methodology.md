## 一、方法论总览

### 1.1 设计目标

本方法论解决三个核心问题：

| 问题 | 传统测试的盲区 | 本方法论的解法 |
|------|---------------|---------------|
| 流程断裂 | 测试只验证单个函数，不验证完整玩家路径 | **深度流程追问**：从玩家第一步操作追问到最后一步，穷举所有分支 |
| 数据链路黑洞 | 只验证数据产生，不追踪数据消耗→效果→关联影响 | **数据链路追问**：追踪每条数据的完整生命周期 |
| 集成测试遗漏 | 有测试但不知道哪些路径没覆盖 | **覆盖矩阵**：将追问出的每一步映射到测试用例 |

### 1.2 三层架构

```
┌─────────────────────────────────────────────────────────┐
│                   评测执行流程                            │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │ 第一层：深度流程追问（穷举法）                        │  │
│  │ 目的：发现所有流程断裂点                              │  │
│  │ 输出：追问记录 + 数据链路图 + 问题清单               │  │
│  └──────────────────────┬─────────────────────────────┘  │
│                         │                                 │
│                         ▼                                 │
│  ┌────────────────────────────────────────────────────┐  │
│  │ 第二层：10维评测（打分法）                            │  │
│  │ 目的：量化评估功能整体体验                            │  │
│  │ 输入：第一层发现的问题                                │  │
│  │ 输出：10D评分表 + 封版判定                           │  │
│  └──────────────────────┬─────────────────────────────┘  │
│                         │                                 │
│                         ▼                                 │
│  ┌────────────────────────────────────────────────────┐  │
│  │ 第三层：集成测试覆盖检查                              │  │
│  │ 目的：确保每个步骤有测试覆盖                          │  │
│  │ 输入：第一层的追问步骤 + 现有测试文件                  │  │
│  │ 输出：覆盖矩阵 + 缺失测试清单                        │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  最终输出：一份完整的评测报告（见第五章）                  │
└─────────────────────────────────────────────────────────┘
```

### 1.3 与现有流程的关系

本方法论嵌入到[开发流程规范](process/development-workflow.md)的**阶段4：评测师评审**中：


## 二、第一层：深度流程追问（穷举法）

### 2.1 核心思想

**苏格拉底式追问**：对每个功能，从玩家第一次接触开始，逐步追问"然后呢？"，直到流程结束。每个分支也要追问。每一步至少追问3个子问题。主流程至少追问10步。

### 2.2 追问的五大维度

每个功能模块的追问必须覆盖以下五个维度：

```
┌─────────────────────────────────────────────────────────┐
│                    追问五大维度                           │
├─────────────┬───────────────────────────────────────────┤
│ ① 主流程    │ 玩家从进入到完成功能的每一步操作路径         │
│ ② 分支路径  │ 每个决策点的所有可能分支（成功/失败/取消）   │
│ ③ 异常场景  │ 断网/崩溃/数据异常/并发操作/边界值          │
│ ④ 数据链路  │ 数据的产生→存储→消耗→效果→关联影响          │
│ ⑤ 跨系统影响│ 该操作触发的其他系统变化                    │
└─────────────┴───────────────────────────────────────────┘
```

### 2.3 追问模板

对每个功能模块，按以下结构组织追问：

```markdown
=== [模块名] 深度流程追问 ===

评测模块：[模块代码] [模块名]
评测版本：v[版本号]
评测日期：[日期]

## 一、主流程追问（至少10步）

Q1: [玩家如何接触到这个功能？]
  → [预期行为描述]
  → 追问1a: [子问题：UI层面的可达性]
  → 追问1b: [子问题：前置条件/解锁条件]
  → 追问1c: [子问题：数据来源]

Q2: [进入功能后看到什么？]
  → [预期行为描述]
  → 追问2a: [子问题：数据从哪来？引擎哪个子系统？]
  → 追问2b: [子问题：数据为空/异常时显示什么？]
  → 追问2c: [子问题：UI元素是否完整？]

Q3: [玩家执行核心操作后发生什么？]
  → [预期行为描述]
  → 追问3a: [子问题：前置条件检查]
  → 追问3b: [子问题：数值计算公式]
  → 追问3c: [子问题：操作反馈]
  → 追问3d: [子问题：并发/防抖]

Q4~Q10: [继续追问后续步骤...]

## 二、分支路径追问

B1: [条件不满足时] → [预期行为]
  → 追问B1a: [提示信息是否明确？]
  → 追问B1b: [是否有引导玩家满足条件？]

B2: [取消/中断时] → [预期行为]
  → 追问B2a: [已消耗的资源是否返还？]
  → 追问B2b: [状态是否回滚？]

B3: [达到上限时] → [预期行为]
  → 追问B3a: [UI如何提示？]
  → 追问B3b: [是否有扩展上限的途径？]

## 三、异常场景追问

E1: [操作过程中关闭游戏] → [状态如何？]
E2: [操作过程中切换Tab] → [状态保持？]
E3: [存档数据损坏] → [如何修复？]
E4: [并发操作] → [是否有竞态条件？]
E5: [网络断开]（如适用）→ [数据是否丢失？]

## 四、数据链路追问（至少3条链路）

D1: [核心资源链路]
  产生: [资源从哪来？]
  → 存储: [存在哪？数据结构？]
  → 消耗: [消耗的条件和公式？]
  → 效果: [消耗后的即时效果？]
  → 关联: [影响了哪些其他系统？]

D2: [状态变更链路]
  [同上结构]

D3: [事件触发链路]
  [同上结构]

## 五、跨系统影响追问

X1: [该操作是否触发任务进度更新？]
X2: [该操作是否触发成就检查？]
X3: [该操作是否触发引导步骤推进？]
X4: [该操作是否解锁新功能/新内容？]
X5: [该操作是否影响其他系统的数值？]
```

### 2.4 追问质量标准

| 指标 | 最低要求 | 优秀标准 |
|------|---------|---------|
| 主流程追问步数 | ≥ 10步 | ≥ 15步 |
| 每步子问题数 | ≥ 3个 | ≥ 5个 |
| 分支路径数 | ≥ 3条 | ≥ 5条 |
| 异常场景数 | ≥ 5个 | ≥ 8个 |
| 数据链路数 | ≥ 3条 | ≥ 5条 |
| 跨系统影响检查 | ≥ 5项 | ≥ 8项 |
| 总追问数 | ≥ 50个 | ≥ 100个 |

### 2.5 追问时的代码溯源规则

每个追问必须追溯到具体代码位置，格式如下：

```
追问3b: 资源扣除的精确值是什么？
  → 代码位置: engine/building/building-config.ts → getUpgradeCost(type, level)
  → 公式: baseCost × (1.5 ^ level)
  → 验证方式: 查看building-config.ts中BUILDING_COSTS表
```

**代码溯源路径**（按项目DDD架构）：

| 追问类型 | 溯源路径 |
|---------|---------|
| UI可达性 | `ui/panels/{domain}/` → React组件层级 |
| 数据来源 | `engine/{domain}/{Domain}System.ts` → 子系统API |
| 数值公式 | `engine/{domain}/{domain}-config.ts` → 配置表 |
| 状态变更 | `engine/ThreeKingdomsEngine.ts` → 编排逻辑 |
| 事件触发 | `core/EventBus.ts` → 事件订阅/发布 |
| 持久化 | `core/SaveManager.ts` → 存档结构 |
| 前置条件 | `engine/{domain}/{Domain}System.ts` → canXxx() 方法 |

## 三、第二层：10维评测（打分法）

### 3.1 评测维度定义

每个维度1-10分，评分必须有代码依据。

| # | 维度 | 定义 | 评判标准 | 代码溯源重点 |
|---|------|------|---------|-------------|
| D1 | **可发现性** | 玩家能否容易找到功能入口 | 入口是否明显、Tab/按钮是否可见、是否需要解锁 | `ThreeKingdomsGame.tsx` Tab定义、`panels/` 入口组件 |
| D2 | **可理解性** | 玩家能否理解功能含义 | 文案是否清晰、图标是否直观、数值含义是否明确 | UI文案常量、Tooltip内容、数值显示格式 |
| D3 | **可操作性** | 操作是否顺畅无阻碍 | 点击响应、流程步骤数、是否有不必要的确认 | 事件处理函数、操作流程链 |
| D4 | **反馈性** | 操作后是否有明确反馈 | Toast提示、动画效果、数值变化可见性 | Toast/Modal调用、事件emit、UI刷新机制 |
| D5 | **完整性** | 功能流程是否闭环 | 从进入到完成是否有断裂、是否有死胡同 | 状态机完整性、流程终点检查 |
| D6 | **数据合理性** | 数值是否平衡合理 | 消耗/产出比例、上限设置、公式曲线 | `{domain}-config.ts` 数值表、公式函数 |
| D7 | **前置条件** | 解锁/使用条件是否合理 | 条件是否过严/过松、条件提示是否明确 | `canXxx()` 方法、解锁配置 |
| D8 | **错误处理** | 异常情况是否友好处理 | 错误提示、数据保护、状态恢复 | try-catch、边界检查、错误事件 |
| D9 | **连贯性** | 与其他功能的衔接是否自然 | 数据流转、功能联动、切换不丢失状态 | 跨子系统调用、事件总线、状态同步 |
| D10 | **重复可玩性** | 是否有持续使用的动力 | 长期收益、成长曲线、内容消耗速率 | 数值曲线、解锁节奏、内容量 |

### 3.2 评分标准

| 分数区间 | 等级 | 含义 | 封版判定 |
|:-------:|:----:|------|---------|
| 9-10 | 🟢 优秀 | 体验完善，无需修改 | 可封版 |
| 7-8 | 🟡 良好 | 基本可用，有小问题 | 需修复P1项后封版 |
| 4-6 | 🟠 一般 | 有明显问题 | 需重点修复，不可封版 |
| 1-3 | 🔴 严重 | 功能断裂或不可用 | 必须重做，不可封版 |

```markdown

|------|:----:|---------|-----------|
| D1 可发现性 | ?/10 | [引用具体代码位置] | [问题描述或"无"] |
| D2 可理解性 | ?/10 | [引用具体代码位置] | [问题描述或"无"] |
| D3 可操作性 | ?/10 | [引用具体代码位置] | [问题描述或"无"] |

| D6 数据合理性 | ?/10 | [引用具体代码位置] | [问题描述或"无"] |
```

### 3.4 评分与第一层追问的映射

每个维度的评分**必须**基于第一层追问的发现：

| 维度 | 对应的追问类型 | 评分依据 |
|------|-------------|---------|
| D1 可发现性 | 主流程Q1（入口可达性） | 追问1a/1b的答案 |
| D2 可理解性 | 主流程Q2（界面展示） | 追问2a/2c的答案 |
| D3 可操作性 | 主流程Q3-Q6（操作步骤） | 操作步骤数、是否有卡点 |
| D4 反馈性 | 主流程每步追问中的反馈相关子问题 | Toast/动画/数值变化 |
| D5 完整性 | 全部主流程+分支路径 | 是否有断裂/死胡同 |
| D6 数据合理性 | 数据链路追问D1-D3 | 公式/比例/曲线是否合理 |
| D7 前置条件 | 分支路径B1（条件不满足） | 条件是否过严/过松 |
| D8 错误处理 | 异常场景E1-E5 | 异常是否友好处理 |
| D9 连贯性 | 跨系统影响X1-X5 | 联动是否完整 |
| D10 重复可玩性 | 数据链路+跨系统 | 长期动力是否足够 |

## 四、第三层：集成测试覆盖检查

### 4.1 目的

基于第一层追问发现的每个步骤，检查是否有对应的测试覆盖。**目标：追问出的每一步都有至少一层测试保护。**

### 4.2 测试层级映射

根据项目[测试子系统设计](architecture/test-system-design.md)，测试分为三层：

| 层级 | 目录模式 | 覆盖范围 | 工具 |
|------|---------|---------|------|
| 引擎层测试（单元） | `engine/{domain}/__tests__/{Domain}System.test.ts` | 子系统内部逻辑 | Vitest |
| 集成测试 | `engine/__tests__/integration/v{n}-{domain}-flow.integration.test.ts` | 子系统间交互 | Vitest |
| E2E测试 | `e2e/*.spec.ts` 或 `e2e/*.cjs` | 完整用户路径 | Playwright |
| 验收测试（ACC） | `engine/{domain}/__tests__/ACC-{nn}.{domain}-engine.test.ts` | PRD符合度 | Vitest |

### 4.3 覆盖矩阵模板

```markdown
## 集成测试覆盖矩阵

| 追问步骤 | 操作描述 | 引擎层测试 | 集成测试 | E2E测试 | ACC验收 | 数据验证 | 覆盖状态 |
|---------|---------|-----------|---------|---------|---------|---------|---------|
| Q1 | 进入功能Tab | — | v{n}-nav-flow | tab-smoke | ACC-XX-01 | — | ✅ 完整 |
| Q2 | 显示数据列表 | {Domain}System.test | v{n}-{domain}-flow | — | ACC-XX-02 | ✅ | ⚠️ 缺E2E |
| Q3 | 执行核心操作 | {Domain}System.test | v{n}-{domain}-flow | — | ACC-XX-10 | ✅ | ⚠️ 缺E2E |
| Q4 | 检查前置条件 | {Domain}System.test | — | — | — | — | ❌ 仅单元 |
| ... | ... | ... | ... | ... | ... | ... | ... |

### 覆盖统计
- 总步骤数: ?
- 完整覆盖(✅): ? (?%)
- 部分覆盖(⚠️): ? (?%)
- 未覆盖(❌): ? (?%)

### 缺失测试清单
| 优先级 | 缺失测试 | 对应追问步骤 | 建议测试类型 |
|--------|---------|-------------|-------------|
| P0 | [缺失的测试描述] | Q{n} | [集成/E2E/ACC] |
```

### 4.4 覆盖状态定义

| 状态 | 图标 | 含义 | 标准 |
|------|:----:|------|------|
| 完整覆盖 | ✅ | 所有层级都有测试 | ≥ 2层测试覆盖 |
| 部分覆盖 | ⚠️ | 有测试但不全 | 仅1层测试覆盖 |
| 未覆盖 | ❌ | 无任何测试 | 0层测试覆盖 |
| 不适用 | — | 该步骤不需要该层测试 | 如UI展示不需要引擎层测试 |

### 4.5 测试文件命名规范

| 测试类型 | 命名规范 | 示例 |
|---------|---------|------|
| 引擎单元 | `{Domain}System.test.ts` | `BuildingSystem.test.ts` |
| 引擎特性 | `{Domain}System.features.test.ts` | `BuildingSystem.features.test.ts` |
| 集成流程 | `v{n}-{domain}-flow.integration.test.ts` | `v1-bld-flow.integration.test.ts` |
| 跨系统集成 | `v{n}-cross-system-flow.integration.test.ts` | `v4-cross-system-flow.integration.test.ts` |
| ACC验收 | `ACC-{nn}.{domain}-engine.test.ts` | `ACC-02.building-engine.test.ts` |
| E2E冒烟 | `{domain}-smoke.spec.ts` | `building-smoke.spec.ts` |

### 6.2 第一层：深度流程追问

#### 一、主流程追问

**Q1: 玩家如何进入建筑Tab？**
  → 点击底部Tab栏的"建筑"按钮
  → 追问1a: Tab栏是否始终可见？是否可能被遮挡？
    - 代码溯源: `ThreeKingdomsGame.tsx` → Tab栏固定在底部，`position: fixed; bottom: 0`
    - 结论: Tab栏始终可见，不被遮挡 ✅
  → 追问1b: 建筑Tab是否需要解锁？初始状态是否可点击？
    - 代码溯源: `ThreeKingdomsGame.tsx` → TABS配置，建筑Tab无解锁条件
    - 结论: 初始即可点击 ✅
  → 追问1c: 点击Tab后触发什么引擎操作？
    - 代码溯源: `ThreeKingdomsEngine.ts` → `getBuildings()` 返回建筑列表
    - 结论: 调用引擎获取建筑数据 ✅

**Q2: 进入建筑Tab后看到什么？**
  → 显示8座建筑卡片（城堡/农田/市场/兵营/铁匠铺/学堂/医馆/城墙）
  → 追问2a: 建筑列表数据从哪来？
    - 代码溯源: `BuildingSystem.ts` → `getAllBuildings()` → 从内部state读取
    - 初始数据: `building-config.ts` → INITIAL_BUILDING_STATES
    - 结论: 引擎初始化时创建8座建筑 ✅
  → 追问2b: 如果BuildingSystem未初始化，显示什么？
    - 代码溯源: `ThreeKingdomsEngine.ts` → `init()` 按序注册子系统，BuildingSystem在init中创建
    - 结论: 不可能出现未初始化状态（init是同步的） ✅
  → 追问2c: 建筑图标、名称、等级是否正确显示？
    - 代码溯源: `BuildingCard.tsx` → 从 `building-config.ts` 的 BUILDING_INFO 读取图标和名称
    - 结论: 图标和名称由配置驱动 ✅
  → 追问2d: PC端和手机端布局是否不同？
    - 代码溯源: `BuildingPanel.tsx` → PC端4列网格，手机端列表布局
    - 结论: 双端适配 ✅

**Q3: 玩家点击某个建筑卡片后发生什么？**
  → 弹出建筑详情/升级面板
  → 追问3a: 面板显示哪些信息？
    - 代码溯源: `BuildingDetailPanel.tsx` → 显示：等级、当前产出、下一级产出、升级消耗、升级时间
    - 结论: 信息完整 ✅
  → 追问3b: 这些数据从哪来？引擎实时计算还是缓存？
    - 代码溯源: `BuildingSystem.ts` → `getBuildingDetail(type)` 实时从state计算
    - 下一级数据: `getUpgradePreview(type)` → `building-config.ts` 的 `getUpgradeCost()` 和 `getProductionAtLevel()`
    - 结论: 实时计算，数据准确 ✅
  → 追问3c: 数据为空或异常时面板显示什么？
    - 代码溯源: `BuildingDetailPanel.tsx` → 有空状态检查，显示"数据加载中..."
    - 结论: 有兜底显示 ⚠️ 但"数据加载中"文案不准确（建筑数据不会异步加载）

**Q4: 玩家点击"升级"按钮后发生什么？**
  → 检查资源是否足够 → 扣除资源 → 开始升级计时
  → 追问4a: 资源不足时按钮是否禁用？是否有提示？
    - 代码溯源: `BuildingDetailPanel.tsx` → 按钮有 `disabled={resourcesInsufficient}` 属性
    - 提示: 鼠标悬停显示"资源不足"Tooltip
    - 结论: 按钮禁用+提示 ✅
  → 追问4b: 资源扣除的精确值是什么？公式是什么？
    - 代码溯源: `building-config.ts` → `getUpgradeCost(type, level)`
    - 公式: `baseCost[type] × (1.5 ^ level)` (各资源分别计算)
    - 结论: 公式明确，可通过配置调整 ✅
  → 追问4c: 扣除后资源栏是否立即刷新？
    - 代码溯源: `ThreeKingdomsEngine.ts` → `upgradeBuilding()` → `resourceSys.consume()` → emit `resource:changed`
    - UI: `ResourcePanel.tsx` 订阅 `resource:changed` 事件
    - 结论: 立即刷新 ✅
  → 追问4d: 如果扣除失败（并发操作导致资源不足）怎么办？
    - 代码溯源: `ResourceSystem.ts` → `consume()` 有原子性检查，失败返回false
    - `ThreeKingdomsEngine.ts` → 检查返回值，失败时emit `upgrade:failed` 事件
    - 结论: 有并发保护 ✅

**Q5: 升级过程中玩家看到什么？**
  → 升级进度条/倒计时
  → 追问5a: 等待时间多长？是固定值还是随等级变化？
    - 代码溯源: `building-config.ts` → `getUpgradeTime(type, level)` = `baseTime × (1.3 ^ level)`
    - 结论: 随等级指数增长 ✅
  → 追问5b: 等待期间玩家能否操作其他功能？
    - 代码溯源: 升级计时在引擎tick中推进，不阻塞UI
    - 结论: 可以操作其他功能 ✅
  → 追问5c: 等待期间能否加速？加速需要什么条件？
    - 代码溯源: `BuildingSystem.ts` → `instantUpgrade()` 消耗天命值
    - 天命消耗公式: `remainingTime(秒) × 0.1`
    - 结论: 有加速机制 ✅

**Q6: 升级完成后发生什么？**
  → 建筑等级+1，产出增加，emit `building:upgraded` 事件
  → 追问6a: 产出增加的精确值是多少？公式是什么？
    - 代码溯源: `building-config.ts` → `getProductionAtLevel(type, level)` = `baseProduction × (1 + 0.1 × level)`
    - 结论: 公式明确 ✅
  → 追问6b: 产出增加后资源栏是否立即刷新？
    - 代码溯源: `ThreeKingdomsEngine.ts` → `syncBuildingToResource()` → 更新产出速率 → emit `resource:rate-changed`
    - UI: `ResourcePanel.tsx` 订阅 `resource:rate-changed`
    - 结论: 立即刷新 ✅
  → 追问6c: 是否有升级完成的动画/提示？
    - 代码溯源: `BuildingCard.tsx` → 升级完成播放闪光动画 + Toast提示"升级完成"
    - 结论: 有反馈 ✅

**Q7: 升级完成后影响了哪些其他系统？**
  → 追问7a: 是否触发任务进度更新？（如"升级建筑1次"任务）
    - 代码溯源: `ThreeKingdomsEngine.ts` → emit `building:upgraded` → QuestSystem（v9.0前未实现）暂不触发
    - 结论: v1.0无任务系统，不影响 ⚠️ v9.0需补充验证
  → 追问7b: 是否触发成就检查？
    - 同上，v1.0无成就系统
  → 追问7c: 是否触发引导步骤推进？
    - 代码溯源: v1.0无引导系统（v18.0实现）
  → 追问7d: 是否解锁新建筑或新功能？
    - 代码溯源: `building-config.ts` → 主城升级可能解锁新功能
    - 结论: 主城升级触发解锁检查 ✅
  → 追问7e: 主城升级是否影响子建筑上限？
    - 代码溯源: `building-config.ts` → `getMaxLevel(buildingType, castleLevel)`
    - 结论: 主城等级限制其他建筑上限 ✅

**Q8: 升级操作是否被正确保存？**
  → 追问8a: 自动保存还是手动保存？
    - 代码溯源: `SaveManager.ts` → 每30秒自动保存 + 关键操作后立即保存
    - 升级操作: `upgradeBuilding()` 后调用 `markDirty()` → 下次tick保存
    - 结论: 自动保存 ✅
  → 追问8b: 保存的数据结构是什么？
    - 代码溯源: `SaveManager.ts` → 序列化 `GameState.buildings` → `{ [type]: { level, upgradeStartTime, ... } }`
    - 结论: 结构明确 ✅
  → 追问8c: 重新加载后升级状态是否恢复？
    - 代码溯源: `SaveManager.ts` → `load()` → 反序列化 → 恢复建筑状态 + 计算离线期间已完成的升级
    - 结论: 完整恢复 ✅

**Q9: 异常场景**
  → 追问9a: 升级过程中关闭游戏，重新打开后状态如何？
    - 代码溯源: `OfflineEarningsCalculator.ts` → 计算离线时间 → 如果升级完成则自动完成
    - 结论: 状态正确恢复 ✅
  → 追问9b: 升级过程中切换Tab，回来后状态如何？

    - 结论: 状态保持 ✅
  → 追问9c: 网络断开时升级操作是否丢失？

    - 结论: 不受影响 ✅
  → 追问9d: 存档数据损坏时建筑数据如何修复？
    - 代码溯源: `SaveManager.ts` → `load()` 有try-catch + checksum校验 → 失败时使用默认数据
    - 结论: 有兜底机制 ✅

**Q10: 重复操作场景**
  → 追问10a: 玩家能否同时升级多个建筑？

  → 追问10c: 快速连续点击升级按钮是否有防抖？

#### 三、数据链路追问

**D1: 金币消耗链路**
```
产生: ResourceSystem初始值300 + 建筑产出(市场baseProduction×等级系数)
  → 存储: GameState.resources.gold (无上限)
  → 消耗: BuildingSystem.upgradeBuilding() → ResourceSystem.consume('gold', cost)
  → 效果: gold减少 → 建筑等级+1 → 产出增加
  → 关联: gold减少可能影响商店购买力、科技研究（v4.0）
```

**D2: 粮草消耗链路**
```
产生: ResourceSystem初始值500 + 农田产出
  → 存储: GameState.resources.grain (上限2000，由enforceCaps截断)
  → 消耗: 升级部分建筑消耗粮草
  → 效果: grain减少 → 建筑等级+1
  → 关联: 粮草也用于招兵(v3.0)、出征(v3.0)，需注意资源竞争
```

**D3: 建筑状态变更链路**
```
产生: BuildingSystem.init() → 8座建筑初始状态(level=1)
  → 存储: GameState.buildings[type] → { level, upgradeStartTime, upgradeTargetLevel }
  → 变更: upgradeBuilding() → level+1, 产出公式重算
  → 效果: syncBuildingToResource() → 更新ResourceSystem的产出速率和上限
  → 关联: 主城等级 → 其他建筑上限; 建筑产出 → 资源总量 → 解锁检查
```

**P1-01: 缺少升级路线推荐**

- 涉及文件: `BuildingPanel.tsx`, `BuildingRecommender.ts`

**P1-02: 缺少E2E测试**
- 问题: 建筑升级流程无E2E测试覆盖
- 修复方案: 新增 `e2e/building-upgrade-smoke.spec.ts`
- 验证方法: 运行 `npx playwright test building-upgrade-smoke`
- 涉及文件: 新增文件

**P2-01: "数据加载中"文案不准确**
- 修复方案: 改为"暂无数据"或直接隐藏空状态面板
- 涉及文件: `BuildingDetailPanel.tsx`
