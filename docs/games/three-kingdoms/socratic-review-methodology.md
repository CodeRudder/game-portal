## 一、方法论概述

### 1.1 什么是"苏格拉底式游戏评测"

苏格拉底式提问（Socratic Questioning）源自古希腊哲学家苏格拉底的教学法，核心思想是：

> **不直接给出答案，而是通过一系列精心设计的问题，引导被问者自己发现矛盾、揭示假设、达到更深层的理解。**

将这一方法应用于游戏评测，我们不是简单地"打分"或"找Bug"，而是：

1. **以玩家视角设问** — 模拟不同类型玩家在不同场景下的真实行为
2. **层层追问** — 从表面现象追问到底层原因
3. **暴露假设** — 检验设计者隐含的假设是否成立
4. **验证闭环** — 追踪每个操作从开始到结果的完整路径

### 1.2 方法论的五个原则

| # | 原则 | 含义 | 示例 |
|---|------|------|------|
| P1 | **玩家本位** | 所有提问以"玩家能否……"开头，而非"代码是否……" | "玩家能否在3秒内理解这个按钮的作用？" |
| P2 | **追问到底** | 发现问题不停止，追问"为什么"至少3层 | 找不到→为什么找不到→入口是否可见→可见但不像按钮→视觉层级不够 |
| P3 | **场景驱动** | 在具体游戏场景中提问，不脱离上下文 | "当玩家粮草为0时，点击升级建筑会怎样？" |
| P4 | **数据验证** | 用真实数据验证每个假设 | "初始粮草500够升几级农田？实际试一下" |
| P5 | **闭环思维** | 每个功能必须有完整的 开始→过程→结果 闭环 | "升级建筑后，玩家能否看到资源变化？产出是否增加？" |

### 1.3 评测适用角色

| 角色 | 使用方式 |
|------|---------|
| **QA工程师** | 按模板执行评测，输出评测报告 |
| **游戏策划** | 自查设计文档，验证设计假设 |
| **开发工程师** | 实现后自测，对照提问检查遗漏 |
| **制作人** | 审阅评测报告，决策是否封版 |

## 二、评测维度体系（10D）

### 2.1 十大维度总览

```
┌─────────────────────────────────────────────────────────────┐
│                    苏格拉底式评测 10D 模型                     │
├──────────┬──────────┬──────────┬──────────┬──────────────────┤
│  D1发现  │  D2理解  │  D3操作  │  D4反馈  │    D5闭环        │
│ 可发现性  │ 可理解性  │ 可操作性  │ 反馈性   │   完整性         │
│ 能找到?   │ 能看懂?   │ 能做到?   │ 有回应?  │  有始有终?       │
├──────────┼──────────┼──────────┼──────────┼──────────────────┤
│  D6数据  │  D7前置  │  D8异常  │  D9衔接  │    D10复玩       │
│ 数据合理  │ 前置条件  │ 错误处理  │ 连贯性   │   重复可玩性     │
│ 数值对?   │ 条件满足?  │ 有提示?   │ 顺畅?    │  愿意再来?       │
└──────────┴──────────┴──────────┴──────────┴──────────────────┘
```

### 2.2 各维度详细定义

---

#### D1 可发现性（Discoverability）

**核心问题**：玩家能否找到这个功能？

**定义**：功能入口在界面上的可见程度和可达程度。包括入口位置、视觉突出度、导航路径长度。

**评测重点**：
- 入口是否在玩家视线自然落点内
- 是否需要多次点击才能到达
- 图标/文字是否传达正确含义
- 新手引导是否指向该入口

**关联模块示例**：
- 天下Tab → Tab栏是否醒目？地图缩放后能否回到操作区？
- 更多菜单 → 网格列表中的功能是否需要翻页？图标是否直觉？
- 招贤馆 → 入口在哪里？是否在武将Tab内有引导？

---

#### D2 可理解性（Comprehensibility）

**核心问题**：玩家能否理解这个功能的用途和规则？

**定义**：功能界面传达信息的清晰程度。包括文案、图标含义、规则说明、数据展示方式。

**评测重点**：
- 功能名称是否自解释
- 关键信息是否需要tooltip或帮助才能理解
- 数值含义是否清晰（如"攻击力+15%"vs"攻击力+15"）
- 规则是否需要外部文档才能理解

**关联模块示例**：
- 科技研究 → 科技加成效果是否直观展示？
- 武将升星 → 升星材料需求和成功率是否清晰？
- 建筑升级 → 升级后产出变化是否预览可见？

---

#### D3 可操作性（Operability）

**核心问题**：玩家能否成功完成期望的操作？

**定义**：从玩家意图到操作完成的顺畅程度。包括交互方式、操作步骤、触控目标大小、响应速度。

**评测重点**：
- 操作步骤是否最少化
- 触控/点击目标是否足够大（移动端≥44px）
- 是否支持撤销/取消操作
- 操作过程是否有loading/等待提示

**关联模块示例**：
- 编队系统 → 拖拽武将到阵容位是否流畅？
- 地图攻城 → 选城→派兵→出征的操作链是否连贯？
- 商店购买 → 购买流程是否需要确认？是否支持批量？

---

#### D4 反馈性（Feedback）

**核心问题**：操作后是否有明确、即时的反馈？

**定义**：系统对玩家操作的响应质量和时效。包括视觉反馈、音效反馈、数值变化动画、状态变化提示。

**评测重点**：
- 点击按钮是否有即时视觉响应
- 资源变化是否有数字飘字/动画
- 操作成功/失败是否有明确区分
- 长时间操作（升级/研究）是否有进度展示

**关联模块示例**：
- 建筑升级 → 升级开始是否有确认动画？升级完成是否有通知？
- 武将升级 → 经验值变化是否有动画？升级是否有特效？
- 攻城战斗 → 战斗过程是否有可视化？结果是否清晰展示？

---

#### D5 完整性（Completeness / 闭环）

**核心问题**：功能是否有完整的"开始→过程→结果"闭环？

**定义**：功能流程的完整程度。每个功能都应该有明确的入口、操作过程、结果展示和后续引导。

**评测重点**：
- 流程是否有死胡同（操作后无路可走）
- 结果是否有明确展示（成功/失败/获得什么）
- 完成后是否有下一步引导
- 是否存在无法退出的状态

**闭环检查模板**：
```
入口触发 → [前置条件检查] → 操作界面 → [操作过程] → 结果展示 → [后续引导]
    ↑                                                              │
    └──────────── 返回/继续 ←──────────────────────────────────────┘
```

**关联模块示例**：
- 招贤馆 → 招募→获得武将→武将去哪了→如何编队→回到主界面，是否连贯？
- 远征 → 开始远征→战斗→结算→奖励领取→回到远征列表，是否完整？
- 任务 → 查看任务→完成条件→领取奖励→下一个任务，是否闭环？

---

#### D6 数据合理性（Data Rationality）

**核心问题**：初始数据是否合理？数值是否平衡？

**定义**：游戏数值设计的合理程度。包括初始值、成长曲线、消耗产出比、数值展示精度。

**评测重点**：
- 初始资源是否足够完成新手引导
- 消耗/产出比是否在合理区间
- 数值成长是否有明确规律（线性/指数/阶梯）
- 展示精度是否合适（整数vs小数）

**三国霸业基准数据**（源自 `resource-config.ts`）：

| 资源 | 初始值 | 上限 | 产出速率 |
|------|--------|------|----------|
| grain（粮草） | 500 | 2,000 | 基础+建筑加成 |
| gold（金币） | 300 | 无上限 | 基础+建筑加成 |
| troops（兵力） | 50 | 500 | 兵营产出 |
| mandate（天命） | 0 | 无上限 | 特殊途径 |
| recruitToken（求贤令） | 10 | 无上限 | 0.002/秒被动 |
| copper（铜钱） | 1,000 | 无上限 | 货币系统 |

**关联模块示例**：
- 建筑升级费用 → Lv1→Lv2消耗是否合理？与初始资源的关系？
- 武将升星 → 材料需求是否与获取速率匹配？
- 科技研究 → 研究时间是否过长？科技点获取是否足够？

---

#### D7 前置条件（Prerequisites）

**核心问题**：功能的前置条件是否满足？是否需要解锁？

**定义**：功能解锁逻辑的清晰程度和合理性。包括解锁条件、条件提示、解锁路径引导。

**评测重点**：
- 解锁条件是否在功能入口处可见
- 条件不满足时是否有明确提示（而非灰色不可点且无说明）
- 解锁路径是否可达（是否存在死锁）
- 前置条件的难度是否合理

**三国霸业解锁依赖链**：
```
主城等级 → 限制所有建筑上限
    ├── 主城Lv4 → 解锁特殊前置建筑
    ├── 主城Lv9 → 解锁高级建筑
    └── 建筑等级 → 解锁对应功能
         ├── 兵营等级 → 兵力上限
         ├── 太学等级 → 科技研究
         └── 市场等级 → 贸易路线
```

**关联模块示例**：
- 科技研究 → 太学是否已建造？等级是否足够？
- 编队系统 → 是否有武将可编？武将数量是否足够？
- 竞技场 → 是否达到解锁等级？是否有足够兵力？

---

#### D8 错误处理（Error Handling）

**核心问题**：异常情况是否有友好提示？

**定义**：系统对异常和边界情况的处理质量。包括错误提示、边界保护、数据校验、恢复机制。

**评测重点**：
- 资源不足时的提示是否明确（缺什么、差多少）
- 网络断开/数据异常时的保护
- 重复操作的防抖/防重复提交
- 异常状态是否能恢复（不卡死）

**错误场景清单**：
| 场景 | 期望行为 |
|------|---------|
| 资源不足 | 显示"需要X，当前Y，还差Z" |
| 建筑队列满 | 提示"队列已满，请等待或使用加速" |
| 武将已在编队中 | 提示"该武将已在第N编队" |
| 网络中断 | 本地缓存，恢复后同步 |
| 存档损坏 | 自动备份恢复，或友好提示 |
| 操作过快 | 防抖处理，不丢失操作 |

**关联模块示例**：
- 建筑升级 → 资源不足提示是否精确到"差多少"？
- 招贤馆 → 求贤令不足时能否继续？提示什么？
- 出征 → 兵力不足时是阻止还是允许？提示是否清晰？

---

#### D9 连贯性（Coherence）

**核心问题**：与其他功能的衔接是否顺畅？

**定义**：功能之间的流转和衔接质量。包括功能跳转、数据传递、状态同步、上下文保持。

**评测重点**：
- 完成一个功能后能否自然跳转到相关功能
- 跨功能的数据是否实时同步
- 功能之间的因果关系是否清晰
- 是否存在功能孤岛（与其他功能无关联）

**三国霸业核心功能流转图**：
```
资源产出 → 建筑升级 → 解锁新建筑/功能
    ↓                        ↓
  攒资源 ←── 科技研究 ←── 太学升级
    ↓                        ↓
  招募武将 → 武将升级 → 编队 → 出征
    ↓                        ↓
  获得战利品 → 装备强化 → 战力提升
    ↓
  攻城略地 → 领土扩张 → 资源加成
```

**关联模块示例**：
- 招贤馆→编队 → 获得新武将后能否直接编队？
- 建筑升级→资源栏 → 升级后资源栏是否实时刷新？
- 战斗结束→邮件 → 战报是否自动发送到邮件？

---

#### D10 重复可玩性（Replayability）

**核心问题**：玩家是否愿意重复使用这个功能？

**定义**：功能的长期吸引力。包括随机性、成长感、奖励变化、策略深度、社交驱动。

**评测重点**：
- 每次使用是否有不同体验（随机性/变化）
- 是否有明确的成长反馈（数值提升/新解锁）
- 是否有策略选择空间（不同路径）
- 是否有社交/竞争驱动（排行榜/联盟）
- 是否有"再来一次"的驱动力

**关联模块示例**：
- 招贤馆 → 抽卡随机性是否足够？保底机制是否合理？
- 竞技场 → 对手匹配是否有趣？奖励是否值得重复挑战？
- 远征 → 每次远征是否有变化？难度是否递增？
- 建筑升级 → 升级选择是否有策略性？是否只是无脑点？

## 三、标准提问模板库

### 3.1 通用提问模板（适用于所有功能）

以下每个维度提供 **核心提问** + **追问链** + **场景变体** 三层结构。

---

#### D1 可发现性 — 提问模板

**核心提问**：
> "一个新玩家第一次进入游戏，能否在 **30秒内** 找到 [功能] 入口？"

**追问链**：
1. "如果玩家当前不在 [功能] 所在的页面上，需要几步操作才能到达？"
2. "入口的图标/文字是否能让玩家在 **不看教程** 的情况下猜到功能用途？"
3. "如果屏幕较小（手机竖屏），入口是否仍然可见且可点击？"
4. "是否有新手引导指向这个入口？如果没有，玩家是否可能忽略它？"

**场景变体**：

| 场景 | 提问 |
|------|------|
| 首次进入 | "玩家刚完成新手引导后，能否独立找到这个功能？" |
| 中途返回 | "玩家离开游戏3天后回来，还能记得这个功能在哪里吗？" |
| 深层页面 | "玩家正在查看武将详情页，能否从这里跳转到 [功能]？" |
| 更多菜单 | "如果功能在'更多'菜单中，菜单入口是否足够醒目？" |

---

#### D2 可理解性 — 提问模板

**核心提问**：
> "玩家看到 [功能] 界面后，能否在 **10秒内** 理解这个功能是做什么的？"

**追问链**：
1. "界面上的文字是否使用了玩家能理解的术语？（避免'天命''求贤令'等专有名词无解释）"
2. "关键数值是否有单位？（如'500'是粮草还是金币？）"
3. "功能规则是否需要查看帮助才能理解？如果是，帮助入口是否明显？"
4. "界面上的信息层级是否清晰？玩家第一眼看到的是最重要的信息吗？"

**场景变体**：

| 场景 | 提问 |
|------|------|
| 复杂规则 | "科技加成的计算公式，玩家能否通过界面推断出来？" |
| 数值展示 | "升级费用'粮草1000/金币500'，是否清晰标注哪个是哪个？" |
| 状态说明 | "建筑'升级中'状态，玩家能否理解还需要多久完成？" |
| 概率事件 | "招贤馆'SSR概率2%'，玩家是否理解这个概率的含义？" |

---

#### D3 可操作性 — 提问模板

**核心提问**：
> "玩家想要完成 [操作] 时，能否在 **3步以内** 完成？"

**追问链**：
1. "每一步操作的目标区域是否足够大？（移动端≥44×44px）"
2. "操作过程中是否有确认步骤防止误操作？"
3. "是否支持撤销/取消？如果支持，撤销路径是否明显？"
4. "操作响应时间是否在可接受范围内？（点击→响应<200ms）"

**场景变体**：

| 场景 | 提问 |
|------|------|
| 批量操作 | "玩家想升级多个建筑，是否需要逐个操作？能否批量？" |
| 拖拽操作 | "编队拖拽武将时，拖拽区域是否明确？放下后是否有吸附效果？" |
| 长流程 | "出征流程（选城→选将→派兵→确认）是否可以保存中间状态？" |
| 移动端 | "在手机上单手操作时，所有按钮是否都能触及？" |

---

#### D4 反馈性 — 提问模板

**核心提问**：
> "玩家完成 [操作] 后，能否 **立即** 看到操作结果？"

**追问链**：
1. "操作成功和失败是否有 **视觉区分**？（颜色/图标/动画不同）"
2. "资源变化是否有 **数字动画**？（如粮草+100飘字）"

## 四、评分标准

### 4.1 十分制评分体系

| 分数 | 等级 | 定义 | 典型表现 |
|:----:|:----:|------|---------|
| **1-2** | 🔴 **严重断裂** | 功能完全不可用，核心流程无法走通 | 入口不存在、点击崩溃、数据全错、死循环 |
| **3-4** | 🟠 **明显问题** | 功能部分可用，有明显的体验断裂 | 入口难找、操作复杂、无反馈、数据不合理 |
| **5-6** | 🟡 **基本可用** | 功能可以完成，但体验有瑕疵 | 需要探索才能理解、反馈不够明显、数值微调 |
| **7-8** | 🟢 **体验良好** | 功能完整可用，有小优化空间 | 流程顺畅、反馈清晰、偶有不够直觉的地方 |
| **9-10** | 🔵 **体验优秀** | 功能体验流畅，无明显问题 | 入口直觉、操作丝滑、反馈即时、数值平衡 |

### 4.2 各维度评分细则

#### D1 可发现性 评分标准

| 分数 | 标准 |
|:----:|------|
| 1-2 | 功能入口不存在，或被其他UI完全遮挡 |
| 3-4 | 入口存在但极难发现（需要3次以上点击，或藏在深层菜单中无提示） |
| 5-6 | 入口可见但不够直觉（图标含义不清，或位置不符合视线习惯） |
| 7-8 | 入口位置合理，大部分玩家能在1分钟内找到 |
| 9-10 | 入口在视线自然落点，30秒内可发现，且有引导指向 |

#### D2 可理解性 评分标准

| 分数 | 标准 |
|:----:|------|
| 1-2 | 界面信息完全无法理解，专有名词无解释 |
| 3-4 | 需要外部帮助/文档才能理解基本功能 |
| 5-6 | 基本功能可理解，但细节规则需要探索 |
| 7-8 | 界面信息清晰，10秒内可理解功能用途 |
| 9-10 | 一目了然，文案精准，数值含义清晰，无需任何解释 |

#### D3 可操作性 评分标准

| 分数 | 标准 |
|:----:|------|
| 1-2 | 无法完成操作（按钮不可点击、交互失效、流程中断） |
| 3-4 | 操作可完成但步骤繁琐（>5步）或目标区域过小 |
| 5-6 | 操作基本顺畅，偶有不便（缺少批量操作、确认步骤过多） |
| 7-8 | 操作流畅，3步内完成，支持撤销/取消 |
| 9-10 | 操作直觉丝滑，最少步骤，支持批量/快捷操作 |

#### D4 反馈性 评分标准

| 分数 | 标准 |
|:----:|------|
| 1-2 | 操作后无任何反馈，不知道是否成功 |
| 3-4 | 有反馈但不明确（仅文字变化，无视觉/动画区分） |
| 5-6 | 基本反馈存在，但延迟或不够醒目 |
| 7-8 | 即时反馈，成功/失败有明确区分，有数值变化动画 |
| 9-10 | 多层次反馈（视觉+动画+音效+推送），信息完整且美观 |

#### D5 完整性 评分标准

| 分数 | 标准 |
|:----:|------|
| 1-2 | 流程有死胡同，操作后无处可去 |
| 3-4 | 流程基本完整但缺少结果展示或后续引导 |
| 5-6 | 闭环存在但某环节体验较弱 |
| 7-8 | 完整闭环，入口→操作→结果→引导全链路畅通 |
| 9-10 | 闭环完美，每步都有清晰的状态和引导，无死角 |

#### D6 数据合理性 评分标准

| 分数 | 标准 |
|:----:|------|
| 1-2 | 数据严重错误（负数、溢出、公式错误） |
| 3-4 | 数据存在明显不平衡（消耗远大于产出） |
| 5-6 | 数据基本合理，个别数值需要微调 |
| 7-8 | 数据平衡，成长曲线平滑，消耗产出比合理 |
| 9-10 | 数据精调完美，每级成长感明确，无断层无冗余 |

#### D7 前置条件 评分标准

| 分数 | 标准 |
|:----:|------|

| 9-10 | 条件透明，进度可视化，解锁路径有引导，无死锁 |

#### D8 错误处理 评分标准

| 分数 | 标准 |
|:----:|------|

| 9-10 | 全面的异常处理，边界情况完善，有恢复机制 |

#### D9 连贯性 评分标准

| 分数 | 标准 |
|:----:|------|
| 1-2 | 功能完全孤立，与其他功能无任何关联 |
| 3-4 | 有关联但跳转困难，数据不同步 |
| 5-6 | 基本连贯，偶有跳转不便或延迟同步 |
| 7-8 | 功能间跳转顺畅，数据实时同步，因果关系清晰 |
| 9-10 | 功能间无缝衔接，智能推荐下一步操作，上下文完美保持 |

#### D10 重复可玩性 评分标准

| 分数 | 标准 |
|:----:|------|
| 1-2 | 无任何重复使用动机 |
| 3-4 | 有被动触发（任务要求），但无主动使用欲望 |
| 5-6 | 有一定吸引力，但重复使用后体验趋同 |
| 7-8 | 有随机性/策略性，每次体验有变化 |
| 9-10 | 强驱动力，随机性+策略性+成长感+社交驱动兼备 |

### 4.3 综合评分计算

```
功能总分 = Σ(维度得分 × 维度权重) / Σ(维度权重)

维度权重分配：
  D1 可发现性   × 1.0  （所有功能必评）
  D2 可理解性   × 1.0  （所有功能必评）
  D3 可操作性   × 1.5  （操作密集型功能加权）
  D4 反馈性     × 1.0  （所有功能必评）
  D5 完整性     × 1.5  （流程型功能加权）
  D6 数据合理性  × 1.0  （数值型功能必评）
  D7 前置条件   × 0.8  （解锁型功能加权）
  D8 错误处理   × 1.0  （所有功能必评）
  D9 连贯性     × 1.2  （系统型功能加权）
  D10重复可玩性  × 1.0  （核心循环功能加权）
```

### 4.4 封版标准

| 条件 | 要求 |
|------|------|
| 单维度最低分 | ≥ 9.0 |
| 功能总分 | ≥ 9.0 |
| 严重问题数（1-4分） | = 0 |
| 迭代轮数 | ≥ 5轮 |
| 连续达标轮数 | ≥ 2轮 |

## 附录A：苏格拉底追问技法

### A.1 六种经典追问类型

| 类型 | 目的 | 示例 |
|------|------|------|
| **澄清性追问** | 确认观察到的现象 | "你说的'找不到'是指完全看不到，还是看到了但不知道它是入口？" |
| **假设性追问** | 检验隐含假设 | "我们假设玩家知道'求贤令'是什么，但如果他们不知道呢？" |
| **证据性追问** | 要求具体证据 | "你说反馈不够明显，具体指什么？能给个截图吗？" |
| **视角性追问** | 切换不同视角 | "如果这是一个从没玩过策略游戏的玩家，他还能理解吗？" |
| **因果性追问** | 追溯因果关系 | "为什么这里没有提示？是因为设计遗漏，还是技术限制？" |
| **预测性追问** | 预测未来影响 | "如果玩家连续使用这个功能100次，会发生什么？" |

### A.2 追问深度模型（5层）

```
Layer 1: 现象层 — "发生了什么？"
    "玩家点击升级按钮后，没有反应。"
        │
Layer 2: 原因层 — "为什么发生？"
    "因为资源不足，按钮被禁用了。"
        │
Layer 3: 感知层 — "玩家知道为什么吗？"
    "不知道，按钮只是变灰了，没有提示原因。"
        │
Layer 4: 设计层 — "设计意图是什么？"
    "设计者可能认为灰色就代表不可用。"
        │
Layer 5: 解决层 — "怎样做得更好？"
    "灰色按钮应该hover/点击时显示tooltip：'需要粮草1000，当前500'"
```

### A.3 常用追问话术

```
发现问题时：
  "这只是一个偶然现象，还是每次都会发生？"
  "如果玩家不知道[背景知识]，他还能理解这个提示吗？"
  "这个问题在手机上会更严重还是更轻微？"

评分犹豫时：
  "给6分和给7分的区别是什么？需要什么证据才能确定？"
  "如果有10个真实玩家测试，你觉得几个会遇到这个问题？"

修复验证时：
  "修复后是否引入了新问题？"
  "修复方案是否是最优解？有没有更简单的方案？"
  "这个修复是否会影响其他功能？"
```

### C.2 快速状态检查脚本

在浏览器Console中执行：

```javascript
// 查看当前资源状态
const state = window.__gameEngine?.getState();
console.table({
  grain: state?.resources?.grain,
  gold: state?.resources?.gold,
  troops: state?.resources?.troops,
  mandate: state?.resources?.mandate,
});

// 查看建筑等级
console.table(state?.buildings);

// 查看武将列表
console.table(state?.heroes?.map(h => ({
  name: h.name, level: h.level, star: h.star, rarity: h.rarity
})));
```
