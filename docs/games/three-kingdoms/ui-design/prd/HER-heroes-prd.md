### 战力计算 {#her-1-3}

> ⚠️ **以引擎实现为准**：公式和系数来自 `hero-config.ts` 的 `POWER_WEIGHTS` 和 `LEVEL_COEFFICIENT_PER_LEVEL`，装备系数来自 `HeroSystem.calculatePower()`。

```
单将战力 = (ATK × 2.0 + INT × 2.0 + CMD × 1.5 + POL × 1.0) × 等级系数 × 品质系数 × 星级系数 × 装备系数
```

| 系数 | 公式 | 引擎来源 | 实现状态 |
|------|------|---------|:-------:|
| 等级系数 | 1 + 等级 × 0.05 | `hero-config.ts` → `LEVEL_COEFFICIENT_PER_LEVEL = 0.05` | ✅ |
| 品质系数 | 查表：COMMON=1.0, FINE=1.15, RARE=1.3, EPIC=1.5, LEGENDARY=1.8 | `hero-config.ts` → `QUALITY_MULTIPLIERS` | ✅ |
| 星级系数 | 查表：1星=1.0, 2星=1.15, 3星=1.35, 4星=1.6, 5星=2.0, 6星=2.5 | `star-up-config.ts` → `STAR_MULTIPLIERS` | ✅ |
| 装备系数 | 1 + totalEquipmentPower / 1000 | `HeroSystem.calculatePower()` 第三参数 | ✅ |

**装备系数集成说明**：

> ✅ **已集成到引擎**：`HeroSystem.calculatePower(general, star, totalEquipmentPower)` 已包含装备系数计算。
> - 公式：`equipmentCoeff = 1 + totalEquipmentPower / 1000`
> - `totalEquipmentPower` 由调用方（UI层）通过 `EquipmentSystem.getHeroEquipments()` 聚合后传入
> - 默认值 `totalEquipmentPower = 0`，即无装备时装备系数 = 1.0（不影响战力）
> - 引擎代码位置：`src/games/three-kingdoms/engine/hero/HeroSystem.ts` → `calculatePower()` 方法
> - 注意：品质系数（`QUALITY_MULTIPLIERS`）在 v1.3 确认已集成到战力计算中，原PRD公式遗漏此系数，现已补正

> ⚠️ **v1.1 修正**：星级系数由原来的线性公式 `1 + (星级-1) × 0.10` 修正为查表法，与引擎 `STAR_MULTIPLIERS` 对齐。
>
> ⚠️ **v1.3 冻结修正**：补正品质系数到公式中（引擎已实现但PRD公式遗漏），补充装备系数的引擎集成状态说明。

**等级上限动态规则**（引擎实际逻辑）：
- `HeroStarSystem.getLevelCap(generalId)` 根据突破阶段返回当前等级上限：50 / 60 / 70 / 80 / 100
- `HeroLevelSystem` 通过 `setLevelDeps({ getLevelCap })` 注入的回调动态获取等级上限（✅ v1.5 已修复）
- 回调注入链路：`engine-hero-deps.ts` 第97行 → `systems.heroStar.getLevelCap(generalId)` → `HeroLevelSystem.getMaxLevel(generalId)`
- **联动机制**：突破成功时 `breakthroughStages[generalId] +1`，`getLevelCap()` 自动返回新的等级上限，升级系统无需额外处理
- `HERO_MAX_LEVEL = 50` 仅作为未注入回调时的 fallback 值，以及 `LEVEL_EXP_TABLE` 的覆盖范围标记

#### 等级上限联动机制（v1.4 新增）

> ✅ **已修复**：连续3轮P0问题（R1→R2→R3），v1.4 彻底解决。

**联动原理**：每次突破成功后，武将的等级上限自动提升 +10，升级系统实时感知新上限。

**数据流链路**：
```
突破成功 → breakthroughStages[heroId] += 1
         → HeroStarSystem.getLevelCap(heroId) 返回新上限
         → HeroLevelSystem.getMaxLevel(heroId) 通过回调获取新上限
         → 升级操作的上限检查自动更新
```

**回调注入点**（`engine-hero-deps.ts`）：
```typescript
// HeroLevelSystem 注入
systems.heroLevel.setLevelDeps({
  getLevelCap: (generalId) => systems.heroStar.getLevelCap(generalId),
});
// HeroSystem 注入
systems.hero.setLevelCapGetter(
  (generalId) => systems.heroStar.getLevelCap(generalId)
);
```

**等级上限变化时间线**：

| 游戏阶段 | 突破阶段 | 等级上限 | 可升级范围 | 关键节点 |
|:-------:|:-------:|:-------:|:---------:|---------|
| 初始 | 0（未突破） | 50 | 1~50 | 新手期，快速升级 |
| 一阶突破后（Lv30） | 1 | 60 | 51~60 | 中期，需要突破石5；觉醒技能解锁 |
| 二阶突破后（Lv45） | 2 | 70 | 61~70 | 中后期，需要突破石10；EPIC觉醒 |
| 三阶突破后（Lv60） | 3 | 80 | 71~80 | 后期，需要突破石20；LEGENDARY觉醒 |
| 四阶突破后（Lv75） | 4（满突破） | 100 | 81~100 | 终局，需要突破石40；终极觉醒 |

**验证要点**：
1. 突破前：武将等级达到当前上限时无法继续升级（`level >= getLevelCap()` 返回 true）
2. 突破后：`getLevelCap()` 立即返回新上限，无需刷新或重载
3. 满突破：`getLevelCap()` 返回 `FINAL_LEVEL_CAP = 100`，达到最终上限
4. Fallback：未注入回调时回退到 `HERO_MAX_LEVEL = 50`（仅影响无引擎集成的测试场景）

| 品质 | 枚举 | 普通招贤概率 | 高级招贤概率 | 收集节奏 | 说明 |
|------|------|:----------:|:----------:|:-------:|------|
| 普通 Uncommon | COMMON | 60% | 20% | 1~2天 | 每日免费+普通招贤即可获得 |
| 精良 Rare | FINE | 30% | 40% | 1~3天 | 普通招贤常见，高级招贤主力 |
| 稀有 Epic | RARE | 8% | 25% | 3~7天 | 普通招贤较难，高级招贤概率可观 |
| 史诗 Legendary | EPIC | 2% | 13% | 7~14天 | 需要高级招贤，十连保底保障 |
| 传说 Mythic | LEGENDARY | 0% | 2% | 30天+ | 仅高级招贤，100抽硬保底 |

**收集节奏验证**（基于4h在线，日均~241招贤令）：
- **普通武将(COMMON)**：每日免费1次 + 普通招贤~48次 × 60% = ~30次命中 → 1天内集齐
- **精良武将(FINE)**：普通招贤~48次 × 30% = ~14次命中 → 1~2天集齐
- **稀有武将(RARE)**：普通招贤~48次 × 8% = ~4次命中 + 高级招贤~2次 × 25% = ~0.5次 → 3~5天集齐
- **史诗武将(EPIC)**：普通招贤~48次 × 2% = ~1次 + 高级招贤~2次 × 13% = ~0.3次 → 7~14天集齐
- **传说武将(LEGENDARY)**：仅高级招贤~2次 × 2% = ~0.04次/日 → 100抽保底约50天

**设计目标**：
- 中度活跃玩家（4h在线）每日约 2~3 次高级招贤或 40~55 次普通招贤
- 新手礼包提供100招贤令，可立即体验1次高级招贤 + 10次普通招贤
- 普通招募消耗5招贤令有效控制了抽卡频率，14名武将完整收集约需2~4周
- 传说武将100抽保底约50天，作为长期追求目标

3. **优先级引导**：一键强化按"升级 > 技能 > 商店 > 碎片"优先级自动分配
4. **铜钱存储上限**：建议 500,000 铜钱，防止无限囤积
5. **溢出转化**：满仓后铜钱自动转化为经验书（小），比率 1000:1
