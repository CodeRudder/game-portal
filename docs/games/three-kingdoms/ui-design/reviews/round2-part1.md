#### 🟢 优点

1. **回归核心闭环设计堪称教科书级**：「回来→看到增长→知道做什么→行动→离开」五步闭环，每个环节都有对应的UI组件支撑（离线收益弹窗→回归欢迎面板→军师建议→一键操作→离线收益持续累积）。这是放置游戏留存设计的核心骨架，v3.0给出了完整方案。

2. **回归欢迎面板信息密度把控精准**：PC端480×520px面板分5区（领土变动/建筑升级/武将动态/未处理事件/离线总收益），每区有跳转链接，信息层次清晰。手机端Bottom Sheet采用横滑卡片+分区折叠，适配小屏。

3. **通知优先级规则系统化**：8级优先级+排队机制+免打扰模式+Toast堆叠策略，解决了Round 1指出的"信息过载风险"问题。这是多数放置游戏设计文档中缺失的部分。

4. **军师建议算法量化**：优先级评分公式（基础权重×紧迫系数×收益系数）+8种推荐类型权重定义，使推荐系统可落地实现。

#### 🟡 问题

1. **回归欢迎流程5步弹窗序列过长**：离线收益弹窗→版本公告→限时活动→回归面板→军师建议，5个弹窗依次弹出。即使部分可跳过，首次回归体验仍显冗长。建议合并为2步：①离线收益弹窗（不可跳）→②回归综合面板（整合公告/活动/军师建议，可跳过）。

2. **手机端快捷图标条"动态增减"增加认知负担**：图标根据状态动态出现/消失，玩家可能找不到固定入口。建议保留3个固定图标（任务/邮件/活动）+最多2个动态图标（远征/签到），固定图标始终可见。

3. **「更多」菜单中核心功能入口层级过深**：装备/远征/商店放在「更多」下拉中，但这些是放置游戏高频操作。建议将"内政"概念引入PC端，增加一级Tab或在导航栏增加可自定义的快捷位。

4. **离线收益效率系数在不同文档中不一致**：主界面文档未定义具体效率系数（引用离线收益弹窗），资源系统文档定义0~24h衰减，建筑系统文档定义0~72h衰减。需要统一。

#### 💡 建议

1. **回归流程简化为2步**：离线收益弹窗（不可跳）+ 回归综合面板（整合其余所有信息，可跳过）
2. **PC端导航栏增加可自定义快捷位**：允许玩家将1-2个高频功能钉在Tab栏（如远征/装备）
3. **统一离线效率系数定义**：建议在资源系统文档中定义全局标准，其他模块引用

#### 🟢 优点

1. **状态机模型是本次迭代最佳技术方案**：5种状态（空闲/滚动/Tooltip/恢复过渡/拖拽）+7条转换规则+关键时间参数表+完整伪代码。这不是"设计师的愿望"，而是"工程师可以直接实现的方案"。特别是200ms Tooltip延迟+200ms恢复延迟的双延迟设计，精准解决了快速划过和频繁切换两个边界问题。

2. **收益可视化三套方案层次分明**：产出气泡（日常浏览）→城池增强标识（重点地标）→热力图模式（战略决策），覆盖了从"扫一眼"到"深度分析"的完整需求场景。

3. **筛选系统组合逻辑严谨**：阵营AND等级（纵向过滤）× 地形OR收益（横向选择），逻辑清晰且实用。"快捷筛选按钮"（我的领土/高产领地/可征服/热力图）是高频操作的好入口。

4. **手机端手势冲突处理表**：5种手势（点击/长按/拖拽/捏合/双指点击）的判定条件和优先级关系清晰，防冲突规则（拖拽优先于长按、10px容差）实用。

#### 🟡 问题

1. **产出气泡性能预算缺失**：文档提到"气泡使用离屏Canvas预渲染"和"视口裁剪"，但未定义具体性能指标。15块领土×每块2行气泡+连接线，在缩放动画期间重绘频率如何控制？建议增加帧率目标（如地图交互≥30fps）和最大同时渲染气泡数。

2. **热力图模式缺少交互深化**：热力图仅改变颜色深浅，但玩家看到高收益领土后如何快速行动？建议热力图模式下点击领土直接弹出"征服/查看详情"，并在热力图图例旁增加"收益排名Top5"快捷列表。

3. **手机端长按显示所有产出气泡存在可用性问题**：长按空白处500ms→所有己方领土同时显示气泡，在领土较多（10+）时屏幕信息量爆炸。建议改为：长按空白→仅显示当前视口内的领土气泡，或改为点击单个领土时显示该领土气泡。

4. **离线变化回顾序列动画时间过长**：每个变化领土间隔300ms，若离线期间有8块领土变化，需2.4秒逐一高亮。建议超过5个变化时改为"前5个逐一高亮+其余同时淡入"策略。

#### 💡 建议

1. **增加地图性能预算章节**：定义帧率目标、最大同时渲染气泡数、缩放动画期间渲染策略
2. **热力图增加交互层**：点击热力图领土→直接行动入口；增加"收益Top5"快捷列表
3. **手机端产出气泡改为"点击触发单领土"**：避免长按全显的信息过载
4. **离线变化回顾动画分级**：≤5个逐一播放，>5个前5个逐一+其余同时

#### 🟢 优点

1. **建筑等级数据表是本次迭代最大亮点**：8座建筑×25-30级，每级包含产出、升级费用（粮草/铜钱/兵力）、升级时间、前置条件。高级别采用"每级×1.8"等系数公式简化，兼顾精确性和可读性。这是从"无法开发"到"可以直接配表"的质变。

2. **队列管理系统设计精巧**：槽位随主城等级解锁（1→2→3→4），平衡了早期引导和后期效率。自动升级开关+排队等待+拖拽排序，覆盖了放置游戏"批量安排→离线→回归收获"的核心体验。

3. **科技/武将联动加成设计深度足够**：6项科技加成+4类武将派遣，每座建筑最多受3重加成影响。联动信息区展示综合产出计算公式（8.2 × 1.15 × 1.28 × 1.18 = 14.2/s），让玩家清晰理解"为什么产出是这个数"。

4. **手机端适配从"最差"变为"标杆"**：双行卡片布局（355×88px）+左滑快捷菜单+Bottom Sheet详情+一键收取悬浮按钮+队列紧凑条，每个组件都有尺寸规格和触控安全校验（≥44×44px）。

#### 🟡 问题

1. **城池俯瞰布局缺乏空间定位规则**：PC端8座建筑"自由布局"在城池场景区，但未定义具体坐标或相对位置规则。建筑间是否会重叠？不同屏幕尺寸下如何自适应？建议定义网格布局规则或锚点定位方案。

2. **建筑外观4阶段递进缺乏视觉参考**：Lv1-5茅屋→Lv6-12瓦房→Lv13-20楼阁→Lv21-30宫殿的文字描述清晰，但缺乏视觉参考图或更详细的美术规范（颜色、材质、轮廓线粗细等）。建议补充关键等级（Lv1/Lv6/Lv13/Lv21）的视觉参考描述。

3. **队列取消返还80%资源的理由未说明**：取消升级返还80%资源是常见设计，但文档未解释为什么是80%而非100%。建议增加设计理由说明（防止频繁取消滥用队列？资源回收税？）。

4. **费用计算器"一键升至目标"存在资源安全风险**：一键消耗大量资源升至目标等级，如果玩家误操作（如选错目标等级），损失巨大。建议增加二次确认弹窗（显示总费用和资源剩余预览）。

## 三、跨模块问题

### 🔴 跨模块问题1：离线效率系数定义冲突（P0级）

**严重程度**：🔴 P0  
**影响范围**：资源系统 ↔ 建筑系统 ↔ 主界面（回归欢迎面板）

| 文档 | 衰减阶梯 | 封顶时间 |
|------|---------|---------|
| 资源系统（08） | 0-2h→100%, 2-8h→80%, 8-16h→60%, 16-24h→40%, >24h→20% | 24h硬封顶 |
| 建筑系统（06） | 0-2h→100%, 2-8h→80%, 8-24h→60%, 24-72h→40%, >72h→20% | 无硬封顶 |

两套衰减曲线**阶梯数不同、衰减速率不同、封顶时间不同**。这会导致：
- 玩家离线8小时：资源系统收益按60%算，建筑系统收益也按60%算（巧合一致）
- 玩家离线16小时：资源系统收益按40%算，建筑系统收益仍按60%算（**不一致**）
- 玩家离线48小时：资源系统已封顶无收益，建筑系统仍按40%产出（**严重不一致**）

**建议**：创建独立的"离线收益系统设计"文档（Round 1 P0-14建议），统一定义全局离线效率系数，所有模块引用同一套规则。

### 🟡 跨模块问题2：手机端资源进度条方案不一致

**严重程度**：🟡 P1  
**影响范围**：主界面布局 ↔ 资源系统

- 主界面文档v3.0：手机端采用"图标底部环形进度"（直径28px，≥2mm）
- 资源系统文档v2.2：手机端仍为"微型进度条(30×3px)"

两份文档描述同一组件（手机端资源栏进度条）但方案不同。开发团队会困惑于以哪个为准。

**建议**：资源系统文档同步更新为环形进度方案。

### 🟡 跨模块问题3：回归欢迎面板与离线收益弹窗信息重叠

**严重程度**：🟡 P1  
**影响范围**：主界面布局 ↔ 资源系统

- 主界面回归欢迎面板包含"离线总收益"区域（🌾+12,400 💰+8,200...）
- 资源系统离线收益弹窗也展示相同数据（🌾+12,450 💰+8,320...）

两处数据来源相同但展示格式不同（欢迎面板4行简略 vs 弹窗详细含容量条），可能导致数值微差时玩家困惑。

**建议**：回归欢迎面板的"离线总收益"区域改为"查看详细离线收益→"链接，点击跳转到离线收益弹窗（已领取则展示历史记录）。

### 🟢 跨模块问题4：建筑联动跳转路径需验证闭环

**严重程度**：🟢 P2  
**影响范围**：建筑系统 ↔ 科技树 ↔ 武将系统

建筑系统定义了6条跳转路径（建筑→科技/武将/资源/队列/军师/离线日志），但反向路径（科技树→建筑、武将详情→派遣建筑）是否已定义需在科技树和武将系统文档中验证。

**建议**：在后续Round中验证跳转路径的双向闭环。
