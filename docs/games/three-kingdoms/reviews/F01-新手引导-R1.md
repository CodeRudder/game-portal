## 二、10D 详细分析

### D1 可发现性 — 8/10

**Q1："一个全新玩家第一次打开游戏，他看到的第一个界面是什么？引导从哪里开始？他能否在5秒内理解'接下来该做什么'？"**

**代码证据：**

1. **首次启动检测链路完整**：
   - `FirstLaunchDetector.detectFirstLaunch()` 检测 `TutorialStateMachine.isFirstLaunch()`（`state.currentPhase === 'not_started'`）
   - 首次启动流程4阶段：语言检测→画质检测→权限申请→触发引导
   - 文件：`engine/guide/FirstLaunchDetector.ts:72-100`

2. **双欢迎弹窗设计**：
   - `WelcomeModal`（`components/idle/three-kingdoms/WelcomeModal.tsx`）：游戏级欢迎，展示4大功能（建筑/武将/科技/关卡），单按钮"开始游戏"
   - `GuideWelcomeModal`（`components/idle/panels/hero/GuideWelcomeModal.tsx`）：引导级欢迎，展示6步预览+步骤数量，双按钮"开始引导"/"跳过"
   - 两层弹窗存在叠加显示风险

3. **5秒可理解性**：
   - ✅ GuideWelcomeModal 展示了步骤预览网格（6个图标+文字：招募武将/查看属性/强化升级/排兵布阵/资源管理/科技研究）
   - ✅ "让我们用 **6** 个简单步骤" 明确告知时长预期
   - ⚠️ WelcomeModal 的"开始游戏"按钮点击后，GuideWelcomeModal 可能紧接着弹出，用户需再次点击"开始引导"，存在两次点击摩擦

**评分依据**：引导入口清晰，首次启动检测逻辑完善。扣分点在于双层欢迎弹窗增加了认知负担，理想情况下应合并为一个统一入口。

### D3 可操作性 — 7/10

**Q3："玩家能否顺利完成引导的每一步操作？是否存在需要'先解锁某功能'才能继续的情况？引导步骤之间的衔接是否顺畅？"**

**代码证据：**

1. **前置条件链式检查**（`TutorialStepManager.ts:135-145`）：
   ```typescript
   // getNextStep 会跳过前置未完成的步骤
   for (const step of CORE_STEP_DEFINITIONS) {
     if (!this._stateMachine.isStepCompleted(step.stepId)) {
       if (step.prerequisite && !this._stateMachine.isStepCompleted(step.prerequisite)) {
         continue; // 跳过前置未完成的步骤
       }
       return step;
     }
   }
   ```
   - step2 依赖 step1，step3 依赖 step2，形成严格线性链
   - `startStep` 方法会拒绝前置未完成的步骤（`TutorialStepManager.ts:155-170`）

2. **⚠️ Overlay↔引擎步骤映射不一致**（`guide-utils.ts:195-220`）：
   - `OVERLAY_TO_ENGINE_STEP` 映射：
     - `recruit → step1_castle_overview`（招募→主城概览？语义不匹配）
     - `detail → step2_build_farm`（查看属性→建造农田？完全不匹配）
     - `enhance → step3_recruit_hero`（强化升级→招募武将？不匹配）
   - `ENGINE_TO_OVERLAY_STEP` 反向映射中，step1/step2/step3 全部映射到 `recruit`，这意味着引擎的3个步骤在UI层只对应1个步骤
   - **这是一个严重的映射问题**：引擎认为有6步（主城概览→建造农田→招募武将→首次出征→查看资源→科技研究），但UI层展示的是完全不同的6步（招募→查看属性→强化升级→排兵布阵→资源管理→科技研究）

3. **步骤衔接**：
   - ✅ 有 `handleNext/handlePrev` 支持前进/后退
   - ✅ 每步切换时有 `setTimeout(0)` 延迟启动下一步，避免状态冲突
   - ⚠️ 引擎步骤完成时 `completeCurrentStep()` 会触发奖励，但映射不一致可能导致错误的奖励被发放

**评分依据**：前置条件检查逻辑正确，但Overlay与引擎的步骤语义映射存在根本性不一致，这是最大的操作风险点。玩家在UI上看到的"招募武将"步骤，引擎记录的却是"主城概览"步骤的完成。

### D7 前置条件 — 9/10

**Q7："引导的每一步是否有前置条件检查？如果玩家跳过引导或中途退出，重新进入时能否正确恢复引导状态？"**

**代码证据：**

1. **前置条件检查**（`TutorialStepManager.ts:155-170`）：
   ```typescript
   startStep(stepId): { success, reason } {
     if (definition.prerequisite && !this._stateMachine.isStepCompleted(definition.prerequisite)) {
       return { success: false, reason: `前置步骤 ${definition.prerequisite} 未完成` };
     }
   }
   ```

2. **进度持久化**（`TutorialStorage.ts`）：
   - 保存：`save()` → `localStorage.setItem('three-kingdoms-tutorial-save', JSON.stringify(data))`
   - 恢复：`restore()` → `load()` → `_stateMachine.loadSaveData(data)`
   - 自动保存：`autoSave()` 在步骤完成/状态转换后调用
   - 文件：`engine/guide/TutorialStorage.ts:50-100`

3. **断点续引**（`GuideOverlay.tsx:100-120`）：
   - 初始化时优先从引擎 `StepManager.getNextStep()` 获取未完成步骤
   - 回退到 `StateMachine.getCompletedStepCount()` 推算步骤索引
   - 最终回退到 `localStorage` 的 `loadProgress()`
   - 文件：`components/idle/panels/hero/GuideOverlay.tsx:100-130`

4. **冲突解决**（`TutorialStateMachine.ts:220-260`）：
   - 策略：取 `completedSteps` 并集 + `completedEvents` 并集（最大进度）
   - 阶段取进度更高的那个
   - 文件：`engine/guide/TutorialStateMachine.ts:220-260`

5. **回归玩家处理**（`GuideOverlay.tsx:90-98`）：
   ```typescript
   // Bug-3: 回归玩家检测 — 已有存档数据时直接跳过引导
   if (phase === 'free_play' || phase === 'mini_tutorial') return -1;
   const completedCount = tutorialSM.getCompletedStepCount();
   if (completedCount > 0 && phase !== 'core_guiding') return -1;
   ```

**评分依据**：前置条件检查、进度持久化、断点续引、冲突解决、回归玩家处理全部到位。测试覆盖了ACC-11-30（引导中刷新页面恢复进度）和ACC-11-36（存档冲突解决）。仅扣1分因为 `TutorialStorage` 的 `restore()` 没有版本迁移逻辑（当前 `TUTORIAL_SAVE_VERSION = 1`，未来版本升级时可能需要）。

### D8 错误处理 — 7/10

**Q8："如果引导过程中出现异常，系统如何处理？是否有重试机制？"**

**代码证据：**

1. **存储层错误处理**（`TutorialStorage.ts`）：
   - `save()` 有 try-catch，返回 `{ success: false, reason: '保存失败: ...' }`
   - `load()` 有 try-catch + 数据格式验证（`validateSaveData`）
   - `loadProgress()` 静默失败（`catch { /* ignore */ }`），返回0（从头开始）
   - 文件：`engine/guide/TutorialStorage.ts:55-95`

2. **步骤执行错误处理**（`TutorialStepManager.ts`）：
   - `startStep()` 返回 `{ success: false, reason }` 而非抛异常
   - 前置条件不满足、步骤已完成、步骤不存在都有明确的错误原因
   - 文件：`engine/guide/TutorialStepManager.ts:155-175`

3. **遮罩定位容错**（`guide-utils.ts:240-280`）：
   - `getTargetElementRect()` 有 try-catch，元素不存在返回 null
   - 视口边界安全检查（`safeTop/safeLeft`）
   - `scrollElementIntoView()` 有回退（`scrollIntoView(false)`）
   - MutationObserver + 定时重试（500ms间隔）处理延迟渲染的元素
   - 文件：`guide-utils.ts:240-280`

4. **⚠️ 缺少的错误处理**：
   - **无步骤超时机制**：`QUICK_COMPLETE_THRESHOLD_MS = 60000` 只标记"可快速完成"，但没有自动超时跳过
   - **无网络断开处理**：引导系统完全基于本地状态，没有网络重连后的状态同步
   - **无步骤执行重试**：`completeCurrentStep()` 失败后没有重试逻辑
   - **GuideOverlay 的 `grantStepRewards`** 静默失败（`if (!engine || typeof engine.grantTutorialRewards !== 'function') return`）

**评分依据**：存储层和UI层的基础错误处理到位，但缺少关键场景的重试和恢复机制。特别是步骤执行卡住时没有超时自动跳过，可能导致玩家被永久卡在某个步骤。

### D9 连贯性 — 8/10

**Q9："引导完成后，玩家是否被自然地引导到日常游戏循环？还是突然失去方向？"**

**代码证据：**

## 四、架构评价

### 4.1 引擎层架构（优秀）

引导系统的引擎层设计遵循了良好的软件工程原则：

- **单一职责**：`TutorialStateMachine`（状态管理）、`TutorialStepManager`（步骤执行）、`TutorialStepExecutor`（加速/跳过/重玩）、`TutorialMaskSystem`（遮罩高亮）、`StoryEventPlayer`（剧情播放）、`FirstLaunchDetector`（首次启动）、`TutorialStorage`（持久化）各司其职
- **依赖注入**：通过 `init(deps)` 和 `setStateMachine(sm)` 实现松耦合
- **事件驱动**：通过 `eventBus.emit` 发射事件，UI层可监听响应
- **可测试性**：所有子系统都有对应的单元测试和集成测试

### 4.2 UI层架构（良好，有改进空间）

- **GuideOverlay** 功能完整，支持前进/后退/跳过/完成
- **手机端适配** 考虑周全：底部弹出、安全区域、触摸目标增大、虚拟键盘适配
- **主要问题**：UI层与引擎层存在"两套数据源"问题，没有完全对接引擎的步骤定义

### 4.3 测试覆盖（优秀）

- 引擎层：ACC-11 验收测试覆盖了20+个验收标准
- 集成测试：7个集成测试文件覆盖了全流程、跳过重玩、剧情事件、状态机、遮罩保护等场景
- UI测试：ACC-11 UI测试覆盖了引导步骤渲染、跳过、完成、奖励展示等场景

## 五、封版判定

| 判定项 | 结果 |
|--------|------|
| 综合评分 | **8.1/10（A-）** |
| 是否达到9分封版线 | ❌ 未达到 |
| 是否需要R2修复 | ✅ **需要R2修复** |
| 阻塞问题 | P0-1（步骤映射不一致）、P0-2（子步骤文案未消费） |
| R2修复优先级 | P0 > P1-2（自由探索过渡UI） > P1-4（遮罩系统统一） |

### R2修复建议

1. **统一步骤映射**（P0-1）：重新设计 `OVERLAY_TO_ENGINE_STEP` 映射，确保UI步骤与引擎步骤语义一致。建议方案：
   - 方案A：UI层直接消费引擎的 `CORE_STEP_DEFINITIONS`，不再维护独立的 `DEFAULT_STEPS`
   - 方案B：修正映射关系，使UI步骤ID与引擎步骤ID一一对应

2. **消费引擎子步骤**（P0-2）：GuideOverlay 应读取引擎的 `TutorialSubStep` 数据来渲染高亮目标和文案，而非使用硬编码的 `targetSelector`

3. **添加自由探索过渡UI**（P1-2）：在引导完成后、奖励弹窗之前，展示 `getFreeExploreData()` 的推荐行动和已解锁功能

4. **统一遮罩系统**（P1-4）：GuideOverlay 应委托给 `TutorialMaskSystem` 管理遮罩和高亮，避免两套遮罩系统并存
