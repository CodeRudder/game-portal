## 二、流程追问记录

### 一、主流程追问

**Q1: 玩家如何获取装备？**
→ 关卡掉落、炼制、商店购买、活动奖励、装备箱
→ 追问1a: 初始状态下玩家有装备吗？
  - 代码溯源: `EquipmentSystem` → 初始背包为空，需通过`generateEquipment()`生成
  - 测试辅助: `createEquipSim()` → 需手动调用`generateEquipment()`
  - 结论: ⚠️ 初始无装备，新手可能困惑如何获取
→ 追问1b: 装备生成的随机性如何？
  - 代码溯源: `EquipmentSystem.generateEquipment(slot, rarity)` → 指定部位和品质生成
  - 副属性: `seed`随机种子控制副属性抽取
  - 结论: 生成机制完整 ✅
→ 追问1c: 装备品质分布如何？
  - 代码溯源: `equipment.types.ts` → 五级品质：白(凡品)/绿(良品)/蓝(上品)/紫(精品)/金(传说)
  - `RARITY_ORDER` → white:1, green:2, blue:3, purple:4, gold:5
  - 结论: 五级品质体系完整 ✅

**Q2: 装备面板显示什么？**
→ 背包容量 + 装备卡片列表 + 装备详情弹窗

  - `equipment.types.ts` → `BagOperationResult` 定义操作结果
  - 结论: 有容量管理 ✅
→ 追问2b: 空背包时显示什么？
  - 代码溯源: `FLOW-09-01` → 空背包时面板正常渲染，无装备卡片

**Q3: 玩家如何穿戴装备？**
→ 选择装备 → 选择武将 → 穿戴到对应部位
→ 追问3a: 一个武将有几个装备部位？

→ 追问3c: 装备已穿戴给其他武将时能否再穿戴？
  - 代码溯源: `FLOW-09-06~08` → 需先卸下再穿戴给新武将

### 四、数据链路追问

**D1: 装备实例生命周期**
```
产生: generateEquipment(slot, rarity) → 创建EquipmentInstance
→ 存储: EquipmentSystem背包 equipments[]
→ 穿戴: equipItem(heroId, uid) → isEquipped=true, equippedHeroId=heroId
→ 卸下: unequipItem(heroId, slot) → isEquipped=false, equippedHeroId=null
→ 强化: enhance(uid) → enhanceLevel++, 属性重算
→ 分解: decompose(uid) → 从背包移除 → 产出copper+enhanceStone
```

**D2: 装备属性→战力链路**
```
装备主属性: MainStat { type, baseValue, value }
→ 装备副属性: SubStat[] { type, baseValue, value }
→ 特殊词条: SpecialEffect | null { type, value, description }
→ 武将战力: power = basePower + Σ(装备属性加权)
→ 关联: 战力影响出征、竞技场匹配
```

**D3: 品质→属性倍率链路**
```
品质: white(1) → green(2) → blue(3) → purple(4) → gold(5)
→ 品质颜色: RARITY_COLORS / RARITY_GRADIENTS
→ 属性倍率: 品质越高，baseValue越高，副属性条数越多
→ 强化上限: 可能与品质关联（需验证）
```

## 七、架构亮点

1. **五级品质体系**：白/绿/蓝/紫/金，颜色+渐变+中文名三位一体，视觉区分度高
2. **四部位设计**：武器/防具/饰品/坐骑，每个部位有独立主属性池和副属性池
3. **副属性随机性**：12种副属性按部位分池，增加装备多样性和追求动力
4. **特殊词条系统**：12种特殊词条为装备提供独特效果，增加build多样性
5. **同部位替换机制**：穿戴新装备自动替换旧装备，操作简洁
6. **批量分解保护**：已穿戴装备自动跳过，防止误操作
