## 维度1: PRD/UI一致性

### 追问1: PRD定义了哪几种任务类型？代码实现了几种？

**PRD定义**（`QST-quests-prd.md §QST-1`）：
- 5种任务类型：主线任务、支线任务、日常任务、周常任务、成就任务

**代码实现**（`quest.types.ts QuestCategory`）：
```typescript
export type QuestCategory = 'main' | 'side' | 'daily' | 'weekly';
```

**⚠️ 差异发现**：

| 任务类型 | PRD定义 | 代码类型 | 状态 |
|---------|:------:|:-------:|:----:|
| 主线任务 | ✅ | `main` | ✅ |
| 支线任务 | ✅ | `side` | ✅ |
| 日常任务 | ✅ | `daily` | ✅ |
| 周常任务 | ✅ | `weekly` | ✅ |
| 成就任务 | ✅ | ❌ 缺失 | 🔴 |

**结论**：代码缺少 `achievement`（成就任务）类型。PRD §QST-4 定义了完整的成就系统（60-100个成就，5大分类，铜/银/金三级），但代码中 `QuestCategory` 不包含 `achievement`。

### 追问2: UI设计稿定义了什么布局？代码实现是否一致？

**代码实现**（`QuestPanel.tsx`）：
- 使用 `SharedPanel` 统一弹窗容器，宽度520px
- 顶部活跃度条（进度条+里程碑按钮）
- Tab切换：日常📅/主线📖/支线📜
- 任务卡片列表（进度条+领取按钮）
- 一键领取按钮

**⚠️ 差异发现**：

| 对比项 | PRD期望 | 代码实现 | 一致？ |
|--------|---------|---------|:------:|
| Tab数量 | 5种任务类型（含成就/周常） | 3种（日常/主线/支线） | ❌ 缺少周常和成就Tab |
| 活跃度系统 | §QST-3 定义4个阈值(40/60/80/100) | 5个里程碑(20/40/60/80/100) | ⚠️ 阈值不完全匹配 |
| 任务跳转按钮 | §QST-2 定义跳转映射 | 无跳转按钮 | ❌ |
| 任务描述文字 | 应显示description | 只显示questDefId | ❌ |
| 奖励预览 | 应显示奖励内容 | 无奖励预览 | ❌ |
| 任务追踪面板 | §QST-2 主界面常驻显示 | 无追踪面板 | ❌ |

### 追问3: 任务奖励配置是否与PRD一致？

**PRD定义**（§QST-2 奖励规则）：

| 任务类型 | PRD典型奖励 |
|---------|-----------|
| 主线任务 | 铜钱×2000~8000 + 经验×1000~5000 + 蓝色装备箱 |
| 支线任务 | 铜钱×5000~15000 + 元宝×30~100 + 紫色装备箱 |
| 日常任务 | 铜钱×1000~3000 + 强化石×1~2 |

**代码实现**（`quest-config.ts`）：

| 任务 | 代码奖励 | PRD范围 | 一致？ |
|------|---------|---------|:------:|
| quest-main-001（初入乱世） | gold: 200, exp: 50 | 铜钱×2000~8000 | ⚠️ 远低于PRD下限 |
| quest-main-002（招兵买马） | gold: 300, exp: 80 | 铜钱×2000~8000 | ⚠️ 远低于PRD下限 |
| quest-main-003（初试锋芒） | gold: 500, exp: 120 | 铜钱×2000~8000 | ⚠️ 远低于PRD下限 |
| daily-001（勤劳建设） | gold: 50, activityPoints: 10 | 铜钱×1000~3000 | ⚠️ 远低于PRD下限 |

**结论**：❌ 任务奖励数值远低于PRD定义。主线任务奖励只有PRD下限的1/10~1/4。日常任务奖励只有PRD下限的1/20~1/60。可能是有意调低（测试阶段），但需要在正式上线前校准。

### 追问4: 日常任务刷新机制是否与PRD一致？

**PRD定义**（§QST-3）：
- 每日05:00重置
- 从固定池20个中随机抽取6个
- 保证类型多样性（至少1个战斗类、1个养成类、1个自动完成类、最多2个同类、D01必定出现）

**代码实现**（`QuestSystem.helpers.ts refreshDailyQuestsLogic`）：
- 使用日期字符串（`new Date().toISOString().slice(0, 10)`）判断是否已刷新
- 从 `DAILY_QUEST_TEMPLATES`（20个模板）中 `Math.random() - 0.5` 随机抽取6个
- `DEFAULT_DAILY_POOL_CONFIG: { poolSize: 20, dailyPickCount: 6, refreshHour: 5 }`

**⚠️ 差异发现**：

| 规则 | PRD定义 | 代码实现 | 一致？ |
|------|---------|---------|:------:|
| 刷新时间 | 每日05:00 | 每日0:00（ISO日期切片，非05:00） | ❌ |
| 抽取数量 | 20选6 | 20选6 | ✅ |
| 保证多样性 | 至少1战斗/1养成/1自动 | 无保证逻辑 | ❌ |
| D01必定出现 | 是 | 无保证 | ❌ |
| 随机算法 | — | `sort(() => Math.random() - 0.5)` | ⚠️ 非均匀分布 |

**结论**：❌ 刷新时间不一致（05:00 vs 0:00），缺少多样性保证规则，随机算法有偏差。

## 维度2: 游戏玩法流程完整性

### 追问1: 玩家如何打开任务面板？

**追踪代码链路**：
- 任务面板通过"更多→任务"路径打开
- `QuestPanel` 使用 `SharedPanel` 弹窗容器
- 接收 `engine`、`visible`、`onClose` props

**✅ 结论**：打开路径正确，弹窗式交互符合放置游戏习惯。

### 追问2: 任务列表是否显示进度？

**代码验证**（`QuestPanel.tsx`）：
```tsx
{o.currentCount}/{o.targetCount}
```
进度条使用百分比宽度：`width: ${Math.min(100, (o.currentCount / o.targetCount) * 100)}%`

**✅ 结论**：进度条和数字都正确显示，如"升级建筑 1/3"。

### 追问3: 任务完成后是否有明显提示？

**代码验证**：
- 任务完成后显示 ✅ 图标：`{done && <span style={{ color: '#7EC850' }}>✅</span>}`
- 领取按钮出现：`{done && !claimed && <button>领取奖励</button>}`
- Toast消息：`flash('🎉 奖励已领取！')`

**⚠️ 缺失**：
- ❌ 无Badge红点提示（主界面任务入口无未完成数量提示）
- ❌ 无任务完成弹窗/推送通知
- ❌ 无声音/震动反馈

**结论**：⚠️ 面板内有完成标记，但缺少外部通知机制。

### 追问4: 领取奖励后资源是否立即到账？

**代码追踪**：
1. `QuestPanel.handleClaim()` → `qs.claimReward(id)`
2. `QuestSystem.claimReward()` → `claimRewardLogic()`
3. `claimRewardLogic()` → 调用 `rewardCallback?.(def.rewards)`
4. `rewardCallback` 由外部注入，负责实际资源发放

**⚠️ 问题**：`rewardCallback` 是否被正确注入？`QuestPanel` 通过 `engine?.getQuestSystem?.() ?? engine?.quest` 获取 QuestSystem，但未看到 rewardCallback 的注入点。

**结论**：⚠️ 奖励发放依赖外部回调注入，需要确认引擎初始化时是否正确设置了 `setRewardCallback()`。

### 追问5: 每日任务刷新时间是什么时候？

**代码验证**：
- `refreshDailyQuestsLogic()` 使用 `new Date().toISOString().slice(0, 10)` 判断日期
- `DEFAULT_DAILY_POOL_CONFIG.refreshHour = 5`（配置值05:00）

**⚠️ Bug**：配置了 `refreshHour: 5`，但代码实际使用日期字符串比较（0:00刷新），未使用 `refreshHour` 配置。这意味着：
- 实际刷新时间：每日0:00（UTC）
- PRD要求：每日05:00（本地时间）
- 时区差异：UTC 0:00 = 北京时间 08:00

**结论**：❌ 刷新时间与PRD不一致。

## 维度3: 资源经济平衡

### 追问1: 任务奖励的资源数量是否合理？

**日常任务奖励分析**（20个模板）：

| 指标 | 最小值 | 最大值 | 平均值 |
|------|:------:|:------:|:------:|
| 铜钱奖励 | 30 | 200 | ~85 |
| 活跃度 | 5 | 25 | ~13 |
| 目标数量 | 1 | 10 | ~3.5 |

**每日最大收益估算**（完成全部6个日常）：
- 铜钱：~510（6×85）
- 活跃度：~78（6×13）

**对比PRD §QST-3 活跃度宝箱**：

| 阈值 | PRD奖励 | 代码里程碑 |
|:----:|---------|-----------|
| 40 | 铜钱×5000+强化石×2 | 20: gold×100 |
| 60 | 元宝×50+招募令×1 | 40: gold×200+grain×100 |
| 80 | 元宝×100+紫色装备箱×1 | 60: gold×300+grain×200+exp×50 |
| 100 | 元宝×200+金色装备碎片×3 | 80: gold×500+grain×300+exp×100 |
| — | — | 100: gold×1000+grain×500+exp×200 |

**结论**：❌ 活跃度里程碑奖励远低于PRD定义。代码只有铜钱/粮草/经验，缺少元宝、招募令、装备箱等高级奖励。

### 追问2: 任务奖励是否足以支撑日常消耗？

**日常铜钱收入**：~510（6个日常）+ ~2100（5个里程碑）= ~2610铜钱/天

**日常铜钱消耗**：
- 建筑升级（农田Lv5→6）：900铜钱/次
- 武将招募：5000铜钱/次（高级）
- 科技研究：200~1000铜钱/次

**结论**：⚠️ 日常任务铜钱收入约2610，勉强支撑2-3次建筑升级。如果奖励数值按PRD校准（日常铜钱×1000~3000），经济会宽裕很多。

### 追问3: 任务完成条件是否与日常玩法匹配？

**日常任务目标分析**：

| 目标类型 | 数量 | 每日可行性 |
|---------|:----:|:---------:|
| build_upgrade（升级建筑） | 4个模板 | ✅ 合理 |
| battle_clear（通关关卡） | 4个模板 | ✅ 合理 |
| recruit_hero（招募武将） | 2个模板 | ⚠️ 招贤令有限 |
| collect_resource（收集资源） | 2个模板 | ✅ 自动完成 |
| npc_interact/npc_gift | 3个模板 | ⚠️ NPC系统依赖 |
| tech_research（研究科技） | 3个模板 | ⚠️ 科技点有限 |
| event_complete（完成事件） | 2个模板 | ⚠️ 事件触发随机 |
| daily_login（每日登录） | 1个模板 | ✅ 自动完成 |

**⚠️ 问题**：如果抽到3个tech_research任务，但玩家科技点只够研究1项，则无法完成。PRD §QST-3 的多样性保证规则（"最多2个同类"）就是为了解决这个问题，但代码未实现。

### 追问4: 活跃度奖励的里程碑设置是否合理？

**代码实现**（5个里程碑）：20/40/60/80/100

**PRD定义**（4个里程碑）：40/60/80/100

**差异**：
- 代码多了20分里程碑（PRD没有）
- PRD的40分奖励（铜钱×5000+强化石×2）远高于代码的40分奖励（gold×200+grain×100）

**结论**：⚠️ 里程碑阈值和奖励内容与PRD不一致。

### 追问5: 任务系统是否与其他系统有资源冲突？

**分析**：
- 日常任务奖励铜钱/粮草 → 与建筑产出、贸易系统资源重叠
- 活跃度宝箱 → 独立奖励通道，不冲突
- 任务进度追踪 → 通过EventBus监听游戏事件，与其他系统事件驱动

**✅ 结论**：资源通道设计合理，不存在直接冲突。但奖励数值需要整体平衡。

## 维度4: 初始数据合理性

### 追问1: 引擎init()后是否有默认任务数据？

**代码验证**（`QuestSystem.init()`）：
```typescript
init(deps: ISystemDeps): void {
  this.deps = deps;
  this.loadPredefinedQuests();
}
```

`loadPredefinedQuests()` 加载 `PREDEFINED_QUESTS`：
- `quest-main-001`（初入乱世）
- `quest-main-002`（招兵买马）
- `quest-main-003`（初试锋芒）
- `quest-side-tech`（科技兴邦）
- `quest-side-npc`（广结善缘）

**⚠️ 问题**：
- 预定义任务只注册了定义（`questDefs`），但未自动接受（`acceptQuest`）
- 日常任务未自动刷新（需要手动调用 `refreshDailyQuests()`）
- 新玩家进入后，`getActiveQuests()` 返回空数组

**结论**：⚠️ 引擎初始化后只有任务定义，没有活跃任务实例。需要外部调用 `acceptQuest()` 和 `refreshDailyQuests()` 来激活任务。

### 追问2: 默认任务是否合理？（新玩家能否完成？）

**主线任务1（初入乱世）**：
- 目标：升级任意建筑1次
- 新玩家状态：主城Lv1、农田Lv1，初始资源500粮/300铜/50兵
- 可行性：✅ 资源足够升级农田Lv1→2

**主线任务2（招兵买马）**：
- 目标：招募武将1次
- 前置：完成主线1
- 可行性：⚠️ 需要招贤令（初始10个），消耗5个/次 → ✅ 可完成

**主线任务3（初试锋芒）**：
- 目标：通关关卡3次
- 前置：完成主线2
- 可行性：⚠️ 需要战斗系统就绪

**结论**：✅ 前2个主线任务对新玩家合理。第3个任务依赖战斗系统。

### 追问3: 如果没有默认数据，面板是否显示空状态？

**代码验证**（`QuestPanel.tsx`）：
```tsx
{quests.length === 0 && <div style={s.empty}>暂无{TABS.find(t => t.id === tab)?.label ?? ''}任务</div>}
```

**✅ 结论**：空状态有友好提示，如"暂无日常任务"。

### 追问4: 任务刷新后数据是否正确重置？

**代码验证**（`refreshDailyQuestsLogic`）：
1. 检查日期是否变更（`dailyRefreshDate !== today`）
2. 清除旧日常任务（`instance.status = 'expired'`，从 `activeQuests` 删除）
3. 随机抽取6个新任务
4. 注册并接受新任务
5. 更新 `dailyRefreshDate`

**⚠️ 问题**：
- 已完成但未领取奖励的日常任务会被直接删除（`activeQuests.delete(id)`），不检查 `rewardClaimed` 状态
- PRD §QST-3 规定"未领取奖励保留至下次重置前"

**结论**：❌ 刷新时未领取的奖励会丢失。

### 追问5: 历史任务进度是否保留？

**代码验证**：
- 已完成任务ID记录在 `completedQuestIds: Set<QuestId>` 中
- 序列化/反序列化正确保存和恢复 `completedQuestIds`
- `acceptQuest()` 检查 `completedQuestIds.has(questId)` 防止重复接受

**✅ 结论**：历史任务完成状态正确保留。

## 维度6: 常见问题检查

### 1. 任务进度是否实时更新？

**代码追踪**：
- `QuestTrackerSystem` 通过 EventBus 监听游戏事件
- `handleGameEvent()` → `questSystem.updateProgressByType()`
- `updateProgressByTypeLogic()` 遍历所有活跃任务，匹配目标类型，更新进度

**✅ 结论**：进度更新是事件驱动的，只要EventBus正确触发，进度会实时更新。

### 2. 领取奖励后任务状态是否正确变化？

**代码验证**（`claimRewardLogic`）：
1. 检查 `status === 'completed'` → 不是则返回null
2. 检查 `rewardClaimed === false` → 已领取则返回null
3. 设置 `rewardClaimed = true`
4. 日常任务增加活跃度
5. 调用 `rewardCallback`
6. 从 `activeQuests` 中删除

**✅ 结论**：领取后状态正确变化（`rewardClaimed = true`，任务从活跃列表移除）。

### 3. 重复领取是否有防护？

**代码验证**：
```typescript
if (instance.rewardClaimed) return null;  // 第一层防护
// ...领取后...
ctx.activeQuests.delete(instanceId);      // 第二层防护：从Map中删除
```

**UI层防护**（`QuestPanel.tsx`）：
```typescript
if (quest.rewardClaimed) {
  flash('📋 奖励已领取，请勿重复操作');
  return;
}
```

**✅ 结论**：双重防护（引擎+UI），重复领取安全。

### 4. 任务面板打开时是否显示loading？

**代码验证**：`QuestPanel` 使用 `useMemo` 计算任务列表，无loading状态。

**⚠️ 问题**：如果引擎初始化较慢，面板可能显示空状态（"暂无任务"），而非loading提示。

**结论**：⚠️ 缺少loading状态，可能在引擎未就绪时显示空状态。

### 5. 空任务列表是否有友好提示？

**代码验证**：
```tsx
{quests.length === 0 && <div style={s.empty}>暂无{TABS.find(t => t.id === tab)?.label ?? ''}任务</div>}
```

**✅ 结论**：有空状态提示，如"暂无日常任务"、"暂无主线任务"。
