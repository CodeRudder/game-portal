### 修复质量评估

**P1-1（招募→编队引导）**：
- 采用自定义事件（`tk:navigate-to-formation`）实现跨组件通信，解耦合理
- 事件流：RecruitModal 点击"前往编队" → dispatch 自定义事件 → HeroTab useEffect 监听 → setSubTab('formation') + setShowRecruitModal(false)
- 关闭结果面板 → 关闭招募弹窗 → 切换到编队Tab，流程连贯
- 按钮位于招募结果网格下方，位置合理，用户获得新武将后自然引导到编队

**P1-2（概率0%标注）**：
- 使用 `row.rate === 0 ? '无法获得' : ...` 三元表达式，语义清晰
- CSS类 `tk-recruit-rate-pct--zero` 提供差异化样式（红色/灰色）
- 概率条宽度为0%，双重视觉传达
- 概率表可折叠（`showRates` 状态），不影响主界面简洁性

**P1-3（战斗失败重试）**：
- `onRetry` 回调设计灵活，由上层组件决定重试逻辑
- 失败弹窗包含：失败原因 + 战斗统计 + 提升建议（基于战力比计算） + 重新挑战按钮
- 提升建议算法智能：战力比<0.5建议升级建筑/招募、<0.8建议调整编队、全军覆没建议提升防御、快速失败建议增加生存能力

## 二、苏格拉底式10D评分（R3）

### D1 功能完整性 — 9.5/10

> **问**：武将招募系统是否覆盖了从招募到使用的完整链路？  
> **答**：完整链路已建立。招募→查看结果→前往编队→排兵布阵→战斗→失败重试，全链路通畅。招募类型（普通/高级）、单抽/十连、免费招募、保底机制、招募历史、概率透明度、武将对比等功能齐全。  
> **扣分**：精良/普通品质武将各仅2名（R2 P1-5未修复），池子深度有限（-0.5）。

### D2 PRUD符合度 — 9.0/10

> **问**：招募流程是否符合渐进式揭示原则？  
> **答**：三重入口（Tab按钮+招募按钮+空状态引导）可发现性好。概率表可折叠，不干扰主流程。保底进度可视化（十连保底+硬保底双条），信息透明。"前往编队"按钮在获得武将后出现，时机精准。  
> **扣分**：概率表默认折叠，新手可能不知道展开查看（-1.0）。

### D3 UI设计符合度 — 9.0/10

> **问**：招募UI是否符合三国主题设计规范？  
> **答**：招募类型卡片（📜普通招贤/🏆高级招贤）图标+标签+描述三层信息。品质揭示动画5级递进（淡入→蓝色闪光→紫色脉冲→红色爆发→金色光芒）。概率条使用品质对应颜色。保底进度条使用差异化填充色。  
> **扣分**：招募结果卡片间距在十连时可能偏密（-1.0）。

### D4 代码质量 — 9.0/10

> **问**：代码架构是否清晰、可维护？  
> **答**：RecruitModal 独立组件，职责单一。防抖机制完善（`isRecruitingRef` 同步锁 + `isRecruiting` 状态双重保护）。资源名称映射（`resourceNameMap`）集中管理。品质揭示动画通过 CSS class 映射（`QUALITY_REVEAL_ANIM`）实现，易扩展。  
> **扣分**：`handleRecruit` 和 `handleFreeRecruit` 逻辑高度相似，可抽取公共方法（-1.0）。

### D5 测试覆盖 — 9.0/10

> **问**：测试是否覆盖了核心场景和边界情况？  
> **答**：测试文件丰富：`RecruitModal.test.tsx` + `RecruitResultModal.test.tsx` + `RecruitPanel.test.tsx`（UI层）+ `HeroRecruitSystem.test.ts` + `HeroRecruitSystem.edge.test.ts` + `HeroRecruitExecutor.edge.test.ts`（引擎层）。引擎层测试覆盖保底机制、免费招募、资源不足等边界。  
> **扣分**：缺少"前往编队"按钮的集成测试（仅验证事件派发，未验证HeroTab接收事件后的Tab切换）（-1.0）。

### D6 可玩性 — 9.0/10

> **问**：招募体验是否有足够的策略深度和惊喜感？  
> **答**：普通/高级双池设计提供选择策略。保底机制（十连保底+硬保底）确保长期投入有回报。品质揭示动画增加惊喜感。每日免费招募提供日常登录动力。招募历史支持回顾。  
> **扣分**：缺少招募动画（如卡牌翻转、光效爆发），当前仅依赖CSS动画（-1.0）。

### D7 交互体验 — 9.5/10

> **问**：操作反馈是否及时、自然？  
> **答**：招募按钮防抖（同步锁+状态锁）防止重复点击。招募中显示"招募中..."状态。资源不足时 Toast 提示具体差额。ESC键关闭弹窗。点击遮罩关闭。招募结果逐个揭示（`delay={idx * 80}`），节奏感好。"前往编队"按钮出现时机精准（获得结果后）。  
> **扣分**：十连招募结果揭示等待时间较长（9×80ms=720ms），可考虑加速（-0.5）。

### D8 动画规范 — 9.0/10

> **问**：动画是否流畅、符合设计规范？  
> **答**：品质揭示动画5级（COMMON→淡入、FINE→蓝色闪光、RARE→紫色脉冲、EPIC→红色爆发、LEGENDARY→金色光芒），层级递进感强。光效层（`tk-recruit-result-glow`）增加视觉冲击。延迟递进（`animationDelay: idx * 80`）创造节奏感。  
> **扣分**：招募类型切换时无过渡动画（-1.0）。

### D9 响应式 — 9.0/10

> **问**：移动端适配是否完善？  
> **答**：HeroTab 武将列表支持PC端4列/手机端2列自适应。分页机制（`PAGINATION_THRESHOLD=100`）避免大量武将时性能问题。筛选/排序控件使用 select 下拉，移动端友好。  
> **扣分**：招募弹窗在极小屏幕（<360px）上可能溢出（-1.0）。

### D10 整体完成度 — 9.5/10

> **问**：系统整体是否达到发布标准？  
> **答**：R2三个P1问题全部修复。招募→编队引导链路补齐是最大改善，将原本断裂的"招募后不知所措"体验转变为"招募→编队→战斗"完整闭环。概率透明度和战斗重试进一步增强了用户体验。无P0问题阻塞。  
> **扣分**：武将池深度有限（精良/普通各仅2名）（-0.5）。

## 四、修复亮点总结

### 1. 招募→编队引导链路（P1-1）⭐⭐
- **方案**：自定义事件 `tk:navigate-to-formation` 跨组件通信
- **效果**：解决了R2最大的体验断裂——招募武将后用户不知道下一步做什么
- **设计**：按钮位于招募结果网格下方，文案"⚔️ 前往编队"明确传达意图
- **流程**：关闭结果→关闭弹窗→切换编队Tab，无感知跳转

### 2. 概率0%标注"无法获得"（P1-2）⭐
- **方案**：`row.rate === 0 ? '无法获得' : `${(row.rate * 100).toFixed(0)}%``
- **效果**：普通池传说概率从显示"0%"改为"无法获得"，语义更清晰
- **视觉**：CSS类 `tk-recruit-rate-pct--zero` 提供红色差异化样式

### 3. 战斗失败"重新挑战"按钮（P1-3）⭐
- **方案**：BattleResultModal 失败分支新增 `onRetry` 回调按钮
- **效果**：用户无需退出→重新进入关卡，直接重试
- **附加价值**：失败弹窗包含智能提升建议（基于战力比计算）
