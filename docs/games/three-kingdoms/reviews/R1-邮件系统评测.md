## 维度1: PRD/UI一致性

### 追问1: PRD定义了哪几种邮件类型？代码实现了几种？

**PRD定义**（`MAL-mail-prd.md §MAL-1`）：
- 4种邮件类型：系统邮件、战斗邮件、社交邮件、奖励邮件

**代码实现**（`shared/mail-types.ts MailCategory`）：
```typescript
export type MailCategory =
  | 'system'       // 系统通知
  | 'reward'       // 奖励邮件
  | 'battle'       // 战报（兼容旧数据）
  | 'combat'       // 战报（新名称）
  | 'trade'        // 贸易通知
  | 'social'       // 社交消息
  | 'alliance';    // 联盟邮件
```

**⚠️ 差异发现**：

| 邮件类型 | PRD定义 | 代码类型 | 状态 |
|---------|:------:|:-------:|:----:|
| 系统邮件 | ✅ | `system` | ✅ |
| 战斗邮件 | ✅ | `battle`/`combat` | ⚠️ 两个类型名 |
| 社交邮件 | ✅ | `social` | ✅ |
| 奖励邮件 | ✅ | `reward` | ✅ |
| — | ❌ | `trade` | ⚠️ PRD未定义 |
| — | ❌ | `alliance` | ⚠️ PRD未定义 |

**结论**：⚠️ 代码实现了7种邮件类型，比PRD多3种（`trade`、`alliance`、`combat`）。其中 `battle` 和 `combat` 功能重叠，可能是版本迁移遗留。

## 维度3: 资源经济平衡

### 追问1: 欢迎邮件的附件奖励是否合理？

**代码检查**：引擎初始化时没有自动发送欢迎邮件。

**MailTemplateSystem 内置模板**：
- `offline_reward`：离线收益报告
- `offline_trade_complete`：贸易完成通知
- `level_up_reward`：升级奖励
- `building_complete`：建筑升级完成
- `combat_report`：战报
- `vip_benefit`：VIP特权提醒

**⚠️ 问题**：无"欢迎邮件"模板。PRD §MAL-3 定义了七日签到奖励，但签到系统由独立的 SignInSystem 管理，不在 MailSystem 初始化范围内。

**结论**：⚠️ 引擎初始化后无欢迎邮件，需要外部系统（如引导系统）触发。

### 追问2: 系统邮件的奖励频率是否合理？

**PRD定义**（§MAL-2）：
- 系统邮件：维护/更新时发送，30天保留
- 战斗邮件：战斗结束时发送，7天保留
- 奖励邮件：签到/活动/成就时发送，30天保留

**代码实现**（`MailConstants.ts`）：
```typescript
DEFAULT_RETAIN_SECONDS = 7 * 24 * 3600;    // 默认7天
SYSTEM_RETAIN_SECONDS = 30 * 24 * 3600;    // 系统30天
REWARD_RETAIN_SECONDS = 14 * 24 * 3600;    // 奖励14天
```

**⚠️ 差异**：
- PRD定义奖励邮件保留30天，代码只保留14天
- 战斗邮件PRD定义7天，代码使用默认7天 ✅
- 社交邮件PRD定义14天，代码使用默认7天 ❌

**结论**：⚠️ 部分邮件类型的保留时长与PRD不一致。

### 追问3: 附件资源是否立即到账？

**代码追踪**：
1. `claimAttachments()` 返回 `Record<string, number>` 资源汇总
2. UI层收到后显示 Toast 提示
3. 但**未调用引擎的资源系统添加资源**

**⚠️ 严重问题**：`MailPanel.handleClaimAttachments` 只显示Toast，没有将领取的资源实际添加到玩家账户。引擎层 `claimAttachments` 只修改邮件状态，不操作资源系统。

**结论**：❌ 附件领取后资源**未实际到账**，需要接入资源系统。

### 追问4: 过期邮件的处理是否合理？

**引擎层**（`MailSystem.processExpired`）：
- 扫描所有邮件，过期时间 <= 当前时间的标记为 `expired`
- 返回过期数量

**缺失功能**：
- ❌ 无自动清理（PRD要求每日05:00自动清理）
- ❌ 无过期前3天/1天提醒
- ❌ 无过期当天闪烁效果
- ❌ 无过期补偿邮件（PRD要求铜钱/经验补发50%）

**结论**：❌ 过期处理只有基础的状态标记，缺少PRD要求的提醒机制和补偿逻辑。

### 追问5: 邮件容量限制是否合理？

**代码定义**（`shared/mail-types.ts`）：
```typescript
export const MAILBOX_CAPACITY = 100;
```

**⚠️ 问题**：容量常量已定义（100封），但引擎和UI层均未使用此限制：
- `sendMail()` 不检查容量
- `MailPanel` 不显示容量条
- 无暂存队列逻辑

**结论**：❌ 容量限制仅定义了常量，未实际执行。

## 维度6: 常见问题检查

### 1. 重复领取附件是否有防护？

**引擎层**（`MailSystem.claimAttachments`）：
```typescript
for (const attachment of mail.attachments) {
  if (!attachment.claimed) {    // 第一层防护
    attachment.claimed = true;
    claimed[...] = ...;
  }
}
if (Object.keys(claimed).length > 0) {
  mail.status = 'read_claimed'; // 第二层防护：状态变更
}
```

**重复领取测试**（FLOW-18-14）：
```typescript
mailSys.claimAttachments(mail.id);
const secondClaim = mailSys.claimAttachments(mail.id);
assertStrict(Object.keys(secondClaim).length === 0, ...);
```

**✅ 结论**：双重防护（附件级 claimed 标记 + 邮件级状态变更），重复领取安全。

### 2. 邮件删除后是否可恢复？

**引擎层**（`MailSystem.deleteMail`）：
```typescript
deleteMail(mailId: string): boolean {
  // ...
  this.mails.delete(mailId);  // Map.delete，不可恢复
  this.persist();
  return true;
}
```

**⚠️ 问题**：删除是硬删除，无法恢复。PRD未定义恢复机制，但建议添加确认弹窗。

**结论**：⚠️ 邮件删除不可恢复，UI层应添加确认机制。

### 3. 大量邮件时性能是否正常？

**代码分析**：
- `getMails()` 遍历所有邮件进行过滤和排序：O(n log n)
- `claimAllAttachments()` 遍历所有邮件：O(n)
- 无分页加载优化（虽然定义了 `MAILS_PER_PAGE = 20`，但 `getMails` 的分页只是 slice）

**⚠️ 问题**：100封邮件时性能应该没问题，但每次操作都遍历全量数据。如果邮件量增大（如暂存队列20封+100封=120封），可能有性能问题。

**结论**：⚠️ 当前规模性能可接受，但缺乏大规模优化。

### 4. 邮件时间显示是否正确？

**代码验证**（`MailPanel.tsx`）：
```tsx
const formatTime = (ts: number) => {
  const d = new Date(ts);
  return `${d.getMonth() + 1}/${d.getDate()} ${d.getHours()}:${String(d.getMinutes()).padStart(2, '0')}`;
};
```

**✅ 结论**：时间格式正确（如 "1/15 8:30"），使用本地时间。

### 5. 附件领取后邮件状态是否正确变化？

**代码验证**（`MailSystem.claimAttachments`）：
```typescript
if (Object.keys(claimed).length > 0) {
  mail.status = 'read_claimed';
  mail.isRead = true;
}
```

**状态流转**：
- `unread` → 领取后 → `read_claimed` ✅（跳过 read_unclaimed，因为直接领取）
- `read_unclaimed` → 领取后 → `read_claimed` ✅

**⚠️ 问题**：从 `unread` 直接领取时，状态从 `unread` 变为 `read_claimed`，跳过了 `read_unclaimed`。这在逻辑上是正确的（因为标记了 isRead=true），但与PRD定义的4态流转不完全一致。

**结论**：✅ 附件领取后状态正确变化，流转逻辑合理。
