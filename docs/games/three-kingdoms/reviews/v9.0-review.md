## 五、问题清单

### 🔴 P0 阻塞问题

| # | 问题 | 文件 | 描述 |
|---|------|------|------|
| 1 | **衰减系数分段区间与PRD不一致** | `engine/offline/offline-config.ts` | PRD定义: `0~2h:100% \| 2~8h:80% \| 8~24h:60% \| 24~48h:40% \| 48~72h:25% \| >72h:15%`。代码实现: `0~2h:100% \| 2~4h:80% \| 4~8h:60% \| 8~24h:40% \| 24~48h:25% \| 48~72h:15%`。代码将PRD的5档拆为6档，每档区间缩短，导致中长期离线（>4h）收益显著低于PRD预期。例如离线8h，PRD应为`2h×100%+6h×80%=6.8h等效`，代码为`2h×100%+2h×80%+4h×60%=6.0h等效`，偏差约12%。 |

### 🟡 P1 严重问题

| # | 问题 | 文件 | 描述 |
|---|------|------|------|
| 2 | **邮件状态枚举不一致** | `engine/mail/MailSystem.ts` vs `engine/mail/mail.types.ts` | MailSystem内部定义 `MailStatus = 'unread' \| 'read_unclaimed' \| 'read_claimed' \| 'expired'`（4态），而mail.types.ts定义 `MailStatus = 'unread' \| 'read' \| 'claimed' \| 'expired'`（4态但名称不同）。两套类型并存可能导致类型不兼容。 |
| 3 | **测试文件超过500行限制** | `OfflineRewardSystem.test.ts`(628行)、`MailSystem.test.ts`(549行) | 违反项目≤500行规范，需拆分为多个测试文件。 |

### 🟢 P2 一般问题

| # | 问题 | 文件 | 描述 |
|---|------|------|------|
| 4 | **旧版OfflineRewardCalculator与v9.0并存** | `core/save/OfflineRewardCalculator.ts` | 旧版使用5档衰减（与PRD一致），新版使用6档（与PRD不一致），两套计算器并存可能造成混淆。建议明确废弃旧版或统一衰减表。 |
| 5 | **MailSystem缺少MailPersistence测试** | `engine/mail/` | MailPersistence.ts没有独立测试文件，其序列化/反序列化/Storage操作仅通过MailSystem间接测试。 |
| 6 | **回归综合面板缺少独立组件** | PLAN要求Step2回归综合面板 | OfflineSnapshotSystem提供了队列完成检测能力，但缺少独立的回归综合面板组件来汇总建筑/科技/远征/事件摘要。 |

### 🔵 P3 轻微问题

## 六、修复建议

### P0 #1 修复方案（二选一）

**方案A（推荐）**: 修改 `offline-config.ts` 衰减表，对齐PRD：

```typescript
export const DECAY_TIERS: readonly DecayTier[] = [
  { id: 'tier1', startHours: 0,  endHours: 2,  efficiency: 1.0,  label: '完整' },
  { id: 'tier2', startHours: 2,  endHours: 8,  efficiency: 0.80, label: '高效' },
  { id: 'tier3', startHours: 8,  endHours: 24, efficiency: 0.60, label: '中效' },
  { id: 'tier4', startHours: 24, endHours: 48, efficiency: 0.40, label: '低效' },
  { id: 'tier5', startHours: 48, endHours: 72, efficiency: 0.25, label: '衰退' },
  // >72h: 0% 封顶，不再产出
] as const;
```

**方案B**: 更新PRD文档以匹配代码的6档设计（需产品确认）。

> 注意：无论选哪个方案，测试用例中的分段断言也需同步更新。
