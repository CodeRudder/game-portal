## 一、第一层：深度流程追问

### Step 1: 玩家第一次打开游戏 — 引擎创建与初始化

**代码位置**: `ThreeKingdomsGame.tsx:77-85`

```typescript
if (!engineRef.current && !engineError) {
  try {
    engineRef.current = new ThreeKingdomsEngine();
  } catch (e) {
    console.error('[ThreeKingdoms] 引擎创建失败:', e);
    setEngineError(e instanceof Error ? e : new Error(String(e)));
  }
}
```

**追问 1a: 引擎初始化失败时显示什么？**

→ 显示错误页面（`ThreeKingdomsGame.tsx:447-470`），包含：
- 错误图标 ⚔️
- "引擎初始化失败" 标题
- 错误信息文本
- "🔄 清除数据并重试" 按钮（调用 `localStorage.clear(); window.location.reload()`）

**追问 1b: 引擎创建后，引导子系统何时初始化？**

→ 引擎创建时在构造函数中注册所有子系统。引导子系统通过 `init(deps)` 注入事件总线等依赖。但关键连接（如 `stepManager.setStateMachine(stateMachine)`）需要在引擎初始化阶段完成。

**追问 1c: 加载状态是否有 loading 指示？**

→ **无专门的 loading 状态**。引擎创建是同步的（`new ThreeKingdomsEngine()`），不显示加载指示。引擎创建失败时直接显示错误页面。

**追问 1d: 引擎快照获取失败时的降级策略？**

→ `ThreeKingdomsGame.tsx:311-387` 中 `snapshot` useMemo 有完整的降级逻辑：引擎为 null 时返回默认快照，`getSnapshot()` 抛异常时也返回默认快照。**设计合理**。

**追问 2a: WelcomeModal 的显示条件是什么？**

→ 检查 `localStorage.getItem('tk-has-visited')`：
- **不存在** → 显示欢迎弹窗，同时写入 `'true'`
- **已存在** → 不显示

**追问 2b: WelcomeModal 的内容是什么？**

→ `WelcomeModal.tsx:26-74`：硬编码的三国主题欢迎界面：
- 标题："⚔️ 欢迎来到三国霸业！"
- 描述文本（硬编码，非配置化）
- 4个功能预览卡片（🏰建筑 / 🦸武将 / 📜科技 / ⚔️关卡）
- 提示文字："💡 点击底部 Tab 栏切换功能，右上角菜单查看更多玩法"
- 单个按钮："开始游戏"

**追问 2c: WelcomeModal 关闭后触发什么？**

→ `handleWelcomeClose`（`ThreeKingdomsGame.tsx:424-434`）：
1. `setShowWelcome(false)` — 关闭弹窗
2. 尝试获取引擎的 `tutorialSM`
3. 如果引导阶段不是 `free_play` 或 `mini_tutorial`，自动切换到 `hero` Tab

**追问 2d: WelcomeModal 和 GuideWelcomeModal 是什么关系？**

→ **两套独立的欢迎弹窗**：
- `WelcomeModal`：在 `ThreeKingdomsGame` 中渲染，标记键 `tk-has-visited`，只有一个"开始游戏"按钮
- `GuideWelcomeModal`：在 `GuideOverlay` 中渲染，标记键 `tk-tutorial-welcome-dismissed`，有"开始引导"和"跳过"两个按钮

**风险**: 玩家首次打开游戏时，**可能先看到 WelcomeModal，关闭后进入 hero Tab，然后又看到 GuideWelcomeModal**。两个弹窗串联显示，体验冗余。

**追问 3a: GuideWelcomeModal 的显示条件？**

2. `tk-tutorial-progress` 未标记为 `completed` → 可能显示
3. 引擎状态机阶段不是 `free_play`/`mini_tutorial` → 显示

**追问 3b: 点击"开始引导"后触发什么？**

→ `handleWelcomeStart`（`GuideOverlay.tsx:225-231`）：
1. `setShowWelcome(false)` — 关闭弹窗
2. `localStorage.setItem('tk-tutorial-welcome-dismissed', 'true')` — 标记已关闭
3. 引导遮罩变为可见，显示第一步

**追问 3c: 点击"跳过，稍后再说"后触发什么？**

→ `handleWelcomeSkip`（`GuideOverlay.tsx:233-242`）：
1. `setShowWelcome(false)` — 关闭弹窗
2. `localStorage.setItem('tk-tutorial-welcome-dismissed', 'true')` — 标记已关闭
3. `saveProgress(0, true)` — 标记引导已完成
4. `setVisible(false)` — 隐藏引导遮罩

**追问 3d: 步骤预览展示的内容是否与实际引导步骤一致？**

### Step 4: 引导第一步渲染 — 遮罩与高亮

**代码位置**: `GuideOverlay.tsx:148-211`（高亮定位 useEffect）

**追问 4a: 高亮目标元素如何定位？**

→ `getTargetElementRect()`（`guide-utils.ts:197-230`）：
1. `document.querySelector(selector)` 查找元素
2. 检查元素是否在视口内（`getBoundingClientRect()` 判断）
3. 不在视口内则 `scrollIntoView({ block: 'center' })`
4. 重新获取 `getBoundingClientRect()`，加上 6px padding
5. 视口边界安全裁剪

**追问 4b: 目标元素不存在时如何处理？**

→ `getTargetElementRect()` 返回 `null`。`highlightRect` 为 null 时，遮罩层仍渲染但**无高亮镂空区域**（`GuideOverlay.tsx:269-277` 的条件渲染 `{highlightRect && (...)}`）。引导仍可继续操作，但**视觉上无聚焦效果**。

**追问 4c: MutationObserver 和重试机制？**

→ `GuideOverlay.tsx:180-186`：
- `MutationObserver` 监听 DOM 变化，目标元素出现时更新位置
- `setInterval(500ms)` 定时重试获取位置
- `setTimeout(350ms)` 在 scrollIntoView 动画完成后重取精确位置
- `resize` / `scroll` / `visualViewport.resize` 事件监听

**设计评价**: 高亮定位的健壮性很好，考虑了延迟渲染、滚动、窗口大小变化、虚拟键盘弹出等移动端场景。

**追问 4d: 引擎子步骤文案如何消费？**

→ `GuideOverlay.tsx:160-164`：
```typescript
const engineDescription = useMemo(() => {
  if (!step || !tutorialStepMgr) return null;
  return getEngineStepDescription(tutorialStepMgr, step.id);
}, [step?.id, tutorialStepMgr]);
```

优先使用引擎 `TutorialStepManager.getCurrentSubStep()` 的子步骤文案，回退到 `DEFAULT_STEPS` 的静态 description。**但实际只取第一个子步骤文案**，不会随子步骤推进更新。

### Step 5: 步骤推进 — 点击"下一步"

**代码位置**: `GuideOverlay.tsx:168-224`（handleNext）

**追问 5a: 点击"下一步"时引擎层发生了什么？**

→ 完整调用链：
1. `onGuideAction({ type: step.id, stepIndex, stepId: step.id })` — 通知外部执行游戏操作
2. `tutorialStepMgr.startStep(engineStepId)` — 确保引擎步骤已启动（幂等）
3. `tutorialStepMgr.completeCurrentStep()` — 完成当前步骤（触发奖励发放事件）
4. `grantStepRewards(step.id)` — 调用 `engine.grantTutorialRewards()` 发放奖励
5. 启动下一步骤（`setTimeout(0)` 延迟启动，避免同一渲染周期重复调用）
6. `tutorialSM.completeStep(engineStepId)` — 通知状态机步骤完成
7. `saveProgress(next, false)` — 保存进度到 localStorage

**追问 5b: 奖励发放的完整链路？**

→ `grantStepRewards()`（`GuideOverlay.tsx:67-76`）：
1. 将 overlay 步骤ID 映射为引擎步骤ID（`OVERLAY_TO_ENGINE_STEP`）
2. 从 `tutorialStepMgr.getStepDefinition(engineStepId)` 获取步骤定义
3. 调用 `engine.grantTutorialRewards(stepDef.rewards)` 将奖励写入玩家资源

**问题**: `completeCurrentStep()` 内部也会通过事件总线发射 `tutorial:rewardGranted` 事件（`TutorialStepManager.ts:181`），但这是事件通知，**实际资源发放依赖 UI 层的 `grantStepRewards()` 调用**。如果 `engine.grantTutorialRewards` 方法不存在或失败，奖励只被事件通知但未实际发放。

**追问 5c: 步骤映射的语义正确性？**

→ `OVERLAY_TO_ENGINE_STEP`（`guide-utils.ts:131-138`）映射关系：

| Overlay ID | 引擎 StepId | 语义匹配度 |
|-----------|-------------|-----------|
| `recruit` | `step3_recruit_hero` | ✅ 匹配 |
| `detail` | `step1_castle_overview` | ❌ **不匹配** — "查看详情"映射到"主城概览" |
| `enhance` | `step2_build_farm` | ❌ **不匹配** — "强化升级"映射到"建造农田" |
| `formation` | `step4_first_battle` | ❌ **不匹配** — "排兵布阵"映射到"首次出征" |
| `resources` | `step5_check_resources` | ✅ 匹配 |
| `tech` | `step6_tech_research` | ✅ 匹配 |

**P1 问题**: 6个映射中有3个语义不匹配。这意味着：
- 引擎层奖励配置与 UI 层展示的奖励描述可能不一致
- 不可跳过检测（`UNSKIPPABLE_STEPS`）可能作用于错误的步骤
- 步骤前置条件检查可能基于错误的步骤ID

**追问 5d: 最后一步完成时触发什么？**

→ `handleNext` 中 `isLastStep` 分支（`GuideOverlay.tsx:172-210`）：
1. 触发当前步骤的引擎动作
2. 完成当前步骤并发放奖励
3. **补发所有步骤的奖励**（遍历 `steps[0..currentStep]`，对每个有引擎映射的步骤调用 `grantStepRewards`）——这是 Bug-2 的修复
4. 推进状态机：`core_guiding` → `free_explore` → `free_play`（连续两次 transition）
5. 显示奖励确认弹窗（汇总所有步骤奖励文本）
6. 调用 `onComplete()` 外部回调

### Step 6: 引导遮罩系统 — TutorialMaskSystem

**代码位置**: `engine/guide/TutorialMaskSystem.ts`

**追问 6a: 遮罩渲染数据如何生成？**

→ `getRenderData()` 返回 `TutorialOverlayRenderData`：
- `mask`: 可见性、透明度、高亮区域、padding、圆角、手指动画、是否阻止非目标点击
- `bubble`: 可见性、文本、位置、箭头目标、最大宽度、计算后位置

**追问 6b: 气泡自动定位逻辑？**

→ `computeAutoPosition()`（`TutorialMaskSystem.ts:192-204`）：
1. 估算气泡高度约 80px
2. 计算高亮元素上方和下方的可用空间
3. 优先放在下方（`spaceBelow >= 80` → `'bottom'`）
4. 空间不足则放上方（`spaceAbove >= 80` → `'top'`）
5. 都不够则放右侧（→ `'right'`）

**追问 6c: 遮罩系统是否被 UI 层实际使用？**

→ **未使用**。`GuideOverlay` 组件自行实现了遮罩和高亮逻辑（CSS + DOM 操作），没有调用 `TutorialMaskSystem` 的 API。`TutorialMaskSystem` 是引擎层的纯数据计算模块，UI 层选择了独立实现。

**P2 问题**: 引擎层遮罩系统与 UI 层遮罩实现重复，增加了维护成本。

### Step 7: 引导状态机 — 5状态转换

**代码位置**: `engine/guide/TutorialStateMachine.ts` + `TutorialTransitions.ts`

**追问 7a: 5个状态的转换规则？**

→ `VALID_TRANSITIONS`（`TutorialTransitions.ts:16-22`）：

| 当前状态 | 允许的转换事件 | 目标状态 |
|---------|--------------|---------|
| `not_started` | `first_enter` | `core_guiding` |
| `core_guiding` | `step6_complete` / `skip_to_explore` | `free_explore` |

| `mini_tutorial` | `mini_done` | `free_play` |

**追问 7b: 状态转换失败时如何处理？**

→ `transition()` 返回 `{ success: false, reason: '状态 xxx 不允许通过事件 xxx 转换' }`。**不抛异常，不记录日志**。调用方需要检查返回值。

**追问 7c: UI 层是否正确处理了转换失败？**

→ `GuideOverlay.tsx` 中多处调用 `tutorialSM.transition()` 但**未检查返回值**：

- `handleSkip` 中 `tutorialSM.transition('skip_to_explore')` 和 `tutorialSM.transition('explore_done')`（L221-222）

**P1 问题**: 如果状态机当前不在预期状态，转换会静默失败，但 UI 层仍继续执行后续逻辑（如标记完成、保存进度），导致状态不一致。

### Step 8: 引导进度存储 — localStorage

**代码位置**: `engine/guide/TutorialStorage.ts` + `guide-utils.ts`

**追问 8a: 存储了哪些数据？**

→ **两套独立的存储**：

| 存储键 | 数据格式 | 写入方 |
|--------|---------|--------|
| `three-kingdoms-tutorial-save` | `TutorialSaveData`（完整状态机快照） | `TutorialStorage` |
| `tk-tutorial-progress` | `{ step: number, completed: boolean }` | `guide-utils.saveProgress()` |
| `tk-has-visited` | `'true'` / null | `ThreeKingdomsGame` |
| `tk-tutorial-welcome-dismissed` | `'true'` / null | `GuideOverlay` |
| `three-kingdoms-first-launch` | timestamp / null | `TutorialStorage` |

**P1 问题**: 5个 localStorage 键由不同系统写入，缺乏统一管理。`three-kingdoms-tutorial-save`（引擎存储）和 `tk-tutorial-progress`（UI 层存储）可能不一致。

**追问 8b: 断点续引如何工作？**

→ `TutorialStorage.restore()`（`TutorialStorage.ts:84-92`）：
1. 从 localStorage 读取 JSON
2. 验证数据格式（`validateSaveData`）
3. 调用 `stateMachine.loadSaveData(data)` 恢复状态

**但 UI 层的恢复逻辑不同**：`GuideOverlay` 的 `currentStep` 初始化（`GuideOverlay.tsx:87-107`）：
1. 检查引擎状态机阶段 → 已完成则返回 -1
2. 检查 `tutorialStepMgr.getNextStep()` → 获取下一个未完成步骤
3. 通过 `ENGINE_TO_OVERLAY_STEP` 反向映射到 overlay 步骤索引
4. 回退到 `tutorialSM.getCompletedStepCount()` 推算
5. 最终回退到 `loadProgress()` 读取 localStorage

### Step 15: 异常场景与边界情况

**追问 15a: 引擎不可用时的回退机制？**

→ `GuideOverlay` 支持 `engine=null`，此时回退到 localStorage 模式：
- `getTutorialSM(null)` → 返回 null
- `getTutorialStepMgr(null)` → 返回 null
- `loadProgress()` 从 `tk-tutorial-progress` 读取步骤索引
- `saveProgress()` 写入步骤索引

**追问 15b: 快速连续点击"下一步"是否安全？**

→ ACC-11-33 测试验证：快速点击3次后到达第4步（最后一步），**不会出现状态错乱**。因为 `setCurrentStep(next)` 是 React state 更新，在同一渲染周期内多次调用会批处理。

**追问 15c: 空步骤列表如何处理？**

→ `shouldRender = currentStep >= 0 && steps.length > 0 && visible`。空步骤时 `shouldRender=false`，不渲染任何内容。ACC-11-39 测试覆盖。

**追问 15d: 两套引导系统同时运行会冲突吗？**

→ **会**。`TutorialSystem`（4步简化版）和 `GuideSystems`（6+6步完整版）独立运行：
- 各自有独立的完成状态追踪
- 各自有独立的 localStorage 键
- `TutorialSystem` 的步骤完成不会通知 `TutorialStateMachine`
- `GuideOverlay` 只对接 `GuideSystems`，不感知 `TutorialSystem`

**P0 问题**: 两套系统可能同时向玩家展示引导提示，造成混乱。

```

  ├── three-kingdoms-tutorial-save  → TutorialSaveData (引擎层，完整状态机快照)
```
