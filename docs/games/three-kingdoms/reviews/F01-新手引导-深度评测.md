# F01 新手引导 — 深度流程评测

> **评测日期**: 2025-07-09
> **评测范围**: 新手引导全流程穷举（v18.0 引导系统 + TutorialSystem 双系统）
> **评测方法**: 逐步追问穷举完整流程链路，每步追踪数据来源→API调用→数据变化→UI响应→关联影响→异常处理
> **代码基准**: `src/games/three-kingdoms/engine/guide/` + `engine/tutorial/` + `core/guide/`

---

## 〇、系统架构概览

新手引导由**两套并行系统**组成：

| 系统 | 代码位置 | 步骤数 | 定位 |
|------|---------|--------|------|
| **v18.0 引导系统** (GuideSystems) | `engine/guide/` | 6核心+6扩展+8剧情 | 完整引导框架 |
| **TutorialSystem** (简化版) | `engine/tutorial/` | 4步（礼包→招募→查看→编队） | 武将引导子集 |

**关键发现**: 两套系统**独立运行、互不感知**。TutorialSystem 注册为 `tutorialGuide`，GuideSystems 注册为 `tutorialStateMachine` 等7个子系统。引擎存档(`engine-save.ts`)中**未序列化任何引导数据**。

## 一、流程穷举（逐步追问）

### Step 1: 玩家第一次打开游戏 — 首次启动检测

- **代码位置**: `engine/guide/FirstLaunchDetector.ts:88-112`
- **数据来源**: `TutorialStateMachine.getState().currentPhase === 'not_started'` 判定为首次启动
- **API调用**: `FirstLaunchDetector.detectFirstLaunch()` → 读取状态机阶段
- **数据变化**: 无状态变更，仅查询
- **UI响应**: 触发 `tutorial:firstLaunchDetected` 事件，携带 `{ isFirstLaunch, detectedLanguage, recommendedQuality, permissionStatus }`
- **关联影响**: 无
- **异常处理**: 无显式异常处理。若 `_stateMachine` 未注入（`setStateMachine` 未调用），将抛出运行时错误
- **集成测试覆盖**: ✅ `tutorial-full-flow.integration.test.ts` §7.1.1 "首次启动检测应识别未开始状态为首次"

#### 追问：首次启动流程具体做什么？

`executeFirstLaunchFlow()` (`FirstLaunchDetector.ts:119-155`) 按序执行4步：

| 阶段 | 操作 | 代码行 |
|------|------|--------|
| 1. 语言检测 | 调用 `languageDetector()` 回调（默认返回 `'zh-CN'`） | L125 |
| 2. 画质检测 | 调用 `hardwareInfoProvider()` 获取 CPU/内存，按阈值推荐 low/medium/high | L128 |
| 3. 权限申请 | 调用外部 `permissionRequester` 回调（storage + network） | L131-140 |
| 4. 触发引导 | `this.triggerTutorial()` → `stateMachine.transition('first_enter')` | L144 |

- **集成测试覆盖**: ✅ `tutorial-full-flow.integration.test.ts` §7.1.1 "首次启动流程应依次完成4个阶段并触发引导"

#### 追问：画质检测阈值是什么？

`first-launch-types.ts` 定义：
- **low**: CPU ≥ 2核, 内存 ≥ 2GB
- **medium**: CPU ≥ 4核, 内存 ≥ 4GB
- **high**: CPU ≥ 8核, 内存 ≥ 8GB

#### 追问：`languageDetector` 和 `hardwareInfoProvider` 谁注入？

由 UI 层通过 `setLanguageDetector()` / `setHardwareInfoProvider()` 注入。若未注入，使用默认值（`zh-CN` / 4核4GB）。**风险**: 引擎层无法自行获取浏览器 `navigator.language` 或 `navigator.hardwareConcurrency`，完全依赖 UI 层注入。

### Step 2: 首次进入 → 状态机转换 `not_started → core_guiding`

- **代码位置**: `engine/guide/TutorialStateMachine.ts:82-110`
- **数据来源**: `FirstLaunchDetector.triggerTutorial()` 触发
- **API调用**: `stateMachine.transition('first_enter')`
- **数据变化**:
  - `currentPhase`: `'not_started'` → `'core_guiding'`
  - `tutorialStartTime`: `null` → `Date.now()`
  - `protectionStartTime`: `null` → `Date.now()`（开启30分钟新手保护）
  - `transitionLogs`: 追加一条 `{ from: 'not_started', to: 'core_guiding', event: 'first_enter', timestamp }`
- **UI响应**: 发射 `tutorial:phaseChanged` 事件
- **关联影响**: 新手保护机制激活（`isNewbieProtectionActive()` 返回 `true`）
- **异常处理**: 若当前阶段不是 `not_started`，转换失败返回 `{ success: false, reason }`
- **集成测试覆盖**: ✅ `tutorial-state-machine.integration.test.ts` #4 "first_enter → core_guiding 并记录时间戳"

#### 追问：新手保护具体效果？

`guide.types.ts` 定义 `DEFAULT_NEWBIE_PROTECTION`:
- **时长**: 30分钟 (`30 * 60 * 1000 ms`)
- **资源消耗折扣**: 0.5（消耗减半）
- **战斗难度系数**: 0.7（敌人属性×0.7）
- **仅正面事件**: `true`（不触发负面随机事件）

这些参数通过 `FirstLaunchDetector.applyResourceDiscount()` 和 `applyBattleDifficulty()` 提供给外部系统调用。**但无强制机制**——依赖其他系统主动查询。

### Step 3: E1 剧情播放 — 桃园结义

- **代码位置**: `engine/guide/StoryEventPlayer.ts:68-93`
- **数据来源**: 剧情定义 `STORY_EVENT_DEFINITIONS[0]` (`core/guide/guide-config.ts:105-121`)
- **触发条件**: `triggerCondition: { type: 'first_enter' }` — 首次进入时触发
- **API调用**: `storyPlayer.startEvent('e1_peach_garden')`
- **数据变化**:
  - `playState`: `'idle'` → `'playing'`
  - `currentEventId`: `null` → `'e1_peach_garden'`
  - `typewriter`: `{ lineIndex: 0, charIndex: 0, lineComplete: false, allComplete: false }`
- **UI响应**: 发射 `tutorial:storyTriggered` 事件
- **剧情内容**: 6行对话（旁白→刘备→关羽→张飞→旁白→旁白）
- **预计时长**: 45秒
- **完成奖励**: 铜钱 × 500
- **异常处理**: 若已有剧情在播放（`playState === 'playing'`），返回 `{ success: false, reason: '已有剧情正在播放' }`
- **集成测试覆盖**: ✅ `tutorial-full-flow.integration.test.ts` §7.1.2

#### 追问：剧情播放交互规则？

`StoryEventPlayer.tap()` (`StoryEventPlayer.ts:101-133`):
1. **当前行未完成** → 立即显示完整行（`action: 'reveal_line'`）
2. **当前行已完成** → 推进到下一行（`action: 'next_line'`）
3. **全部对话完成** → 返回 `action: 'complete'`

打字机速度: 30ms/字 (`TYPEWRITER_SPEED_MS`)，加速时 10ms/字。
自动播放: 5秒无操作自动推进 (`AUTO_PLAY_DELAY_MS = 5000`)。

#### 追问：剧情跳过机制？

`StoryEventPlayer.ts:143-175`:
1. `requestSkip()` → 设置 `skipConfirmPending = true`，返回 `{ requireConfirm: true }`
2. `confirmSkip()` → 执行跳过，返回 `{ transitionEffect: 'ink_wash', rewards: [...] }`
3. `cancelSkip()` → 取消确认，继续播放
4. **未请求直接确认** → 返回 `{ success: false }`

**关键**: 跳过剧情**不影响奖励发放**（`confirmSkip` 仍返回 rewards）。

- **集成测试覆盖**: ✅ `tutorial-skip-replay-sync.integration.test.ts` §7.2.3

### Step 4: 核心引导步骤 1 — 主城概览

- **代码位置**: `core/guide/guide-config.ts:30-56` (定义) + `engine/guide/TutorialStepManager.ts:112-130` (执行)
- **数据来源**: `CORE_STEP_DEFINITIONS[0]`
- **API调用**: `stepManager.startStep('step1_castle_overview')` → `stepManager.advanceSubStep()` × 5
- **触发条件**: 无前置步骤（`prerequisite` 未定义）
- **数据变化**:
  - `activeStepId`: `null` → `'step1_castle_overview'`
  - `currentSubStepIndex`: 0 → 1 → 2 → 3 → 4 → 完成
  - `stepStartTime`: `null` → `Date.now()`
- **子步骤详情**:

| 子步骤 | 文本 | 目标元素 | 不可跳过 | 完成方式 |
|--------|------|---------|---------|---------|
| 1-1 | 欢迎来到三国乱世！这是你的主城。 | `#main-castle` | ✅ 是 | click |
| 1-2 | 上方是资源栏，显示你的铜钱、粮草等资源。 | `#resource-bar` | 否 | click |
| 1-3 | 底部是导航Tab，可以切换不同功能界面。 | `#nav-tab` | 否 | click |
| 1-4 | 左侧是建筑区域，可以建造和升级各种建筑。 | `#building-area` | 否 | click |
| 1-5 | 准备好了吗？让我们开始建造吧！ | `#building-area` | 否 | click |

- **完成奖励**: 铜钱 × 200
- **UI响应**: 每个子步骤激活遮罩+气泡，高亮对应元素
- **关联影响**: 完成后解锁步骤2（建造农田）的前置条件
- **异常处理**: 若前置步骤未完成，`startStep` 返回 `{ success: false, reason: '前置步骤 xxx 未完成' }`
- **集成测试覆盖**: ✅ `tutorial-full-flow.integration.test.ts` §7.1.3 "步骤1~6应按顺序执行且前置条件依次满足"

#### 追问：遮罩系统如何配合？

`TutorialMaskSystem.setupForSubStep()` (`TutorialMaskSystem.ts:106-124`):
1. 激活遮罩（`activate()`）
2. 设置高亮目标（`setHighlightTarget(selector, boundsProvider)`）
3. 显示气泡（`showBubble({ text, position: 'auto', ... })`）

遮罩默认配置 (`guide.types.ts`):
- **透明度**: 0.8（80%黑色遮罩）
- **内边距**: 8px
- **圆角**: 8px
- **引导手指动画**: 显示
- **非目标区域点击**: 阻止（`blockNonTargetClicks: true`）

#### 追问：元素不存在怎么办？

`setHighlightTarget()` 返回 `{ success: false, reason: '无法获取元素 xxx 的位置' }`。但**不会阻止引导继续**——`setupForSubStep` 返回失败结果后，调用方需要处理。**风险**: 若 UI 层未检查返回值，遮罩将处于激活但无高亮目标的异常状态。

### Step 5: 核心引导步骤 2 — 建造农田

- **代码位置**: `core/guide/guide-config.ts:57-82`
- **前置条件**: `prerequisite: 'step1_castle_overview'`
- **子步骤**:

| 子步骤 | 文本 | 目标元素 | 不可跳过 | 完成方式 |
|--------|------|---------|---------|---------|
| 2-1 | 点击空地可以建造新建筑。 | `#empty-plot` | 否 | click |
| 2-2 | 选择农田建筑，它是资源产出的基础。 | `#building-farm` | 否 | click |
| 2-3 | 确认建造！农田将持续产出粮草。 | `#confirm-build` | ✅ 是 | action |
| 2-4 | 建造需要一定时间，可以点击加速。 | `#speed-up` | 否 | click |
| 2-5 | 很好！农田已建造完成。 | `#building-area` | 否 | auto |

- **完成奖励**: 粮草 × 500
- **关联影响**: 
  - 建筑系统：引导玩家建造农田（但引导系统本身**不直接调用**建筑系统的建造API）
  - 解锁步骤3（招募武将）
- **集成测试覆盖**: ✅ `tutorial-state-machine.integration.test.ts` #16-17

#### 追问：引导步骤是否真正执行建造操作？

**关键发现**: 引导系统只管理步骤的推进和完成状态，**不直接调用建筑/招募/战斗等游戏系统的API**。引导步骤的 `completionType`（click/action/auto）仅作为 UI 层的提示，告诉 UI 层玩家需要执行什么操作。**实际的建造/招募/战斗操作由 UI 层触发**，引导系统只是监听结果。

这意味着：引导系统与游戏系统之间存在**松耦合**——引导不知道玩家是否真的建造了农田，只知道步骤被标记为完成。

### Step N+2: 异常场景穷举

#### 场景A: 中途退出游戏，重新进入

- **恢复逻辑**: `TutorialStorage.restore()` (`engine/guide/TutorialStorage.ts:87-95`)
  - 从 `localStorage` 读取 `three-kingdoms-tutorial-save`
  - 调用 `stateMachine.loadSaveData(data)` 恢复状态
- **恢复粒度**: 恢复到**步骤级别**（`completedSteps` + `currentStepId` + `currentSubStepIndex`）
- **问题**: `stepStartTime` 未保存，恢复后无法正确计算「快速完成」阈值（60秒）
- **集成测试覆盖**: ✅ `tutorial-full-flow.integration.test.ts` §7.1.6 "反序列化应恢复完整状态"

#### 场景B: 引导过程中资源不够

- **引导系统无感知**: 引导步骤不检查资源是否充足
- **处理方式**: 完全依赖 UI 层处理（如显示"资源不足"提示）
- **风险**: 若 UI 层未处理，玩家可能卡在引导步骤中无法前进

#### 场景C: 引导步骤的某个UI元素不存在

- **遮罩系统**: `setHighlightTarget()` 返回 `{ success: false, reason: '无法获取元素 xxx 的位置' }`
- **引导系统**: 不检查遮罩设置是否成功
- **风险**: 遮罩激活但无高亮区域，玩家看到全黑遮罩无法操作
- **测试覆盖**: ❌ 无专门测试

#### 场景D: localStorage 不可用

- **TutorialStorage.save()**: try-catch 捕获，返回 `{ success: false, reason: '保存失败: ...' }`
- **后果**: 引导进度无法持久化，每次刷新重新开始
- **测试覆盖**: ❌ 无 localStorage 不可用的测试

#### 场景E: 存档数据格式损坏

- **TutorialStorage.validateSaveData()**: 检查 `version`(number), `completedSteps`(array), `completedEvents`(array), `currentPhase`(string)
- **处理**: 返回 `{ success: false, reason: '存档数据格式无效' }`，不恢复
- **测试覆盖**: ❌ 无损坏数据测试

#### 场景F: 两套引导系统同时运行

- **现状**: TutorialSystem 和 GuideSystems 独立运行
- **风险**: 玩家可能同时看到两套引导提示
- **测试覆盖**: ❌ 无双系统冲突测试

## 六、评测总结

### 优点

1. **架构设计精良**: 引导系统采用状态机模式，5阶段转换规则清晰，合法转换表 (`VALID_TRANSITIONS`) 限制了非法状态跳转
2. **子系统职责分明**: 7个子系统（状态机/步骤管理/步骤执行/剧情播放/遮罩/存储/首次检测）各司其职，通过依赖注入解耦
3. **测试覆盖充分**: 引导子系统自身有完善的单元测试和集成测试（约100+测试用例），覆盖正常流程和大部分边界条件
4. **冲突解决策略合理**: 跨设备同步取 `completedSteps` 并集 + 更高阶段，确保进度不丢失
5. **新手保护设计周到**: 30分钟保护期、资源消耗减半、战斗难度降低、仅正面事件

### 不足

1. **引导与游戏系统脱钩**: 引导系统是"纯状态管理器"，不与建筑/招募/战斗等系统直接交互，引导步骤的完成不等于游戏操作的实际执行
2. **奖励发放断裂**: 事件发射了但无人监听，引导奖励可能无法到账
3. **存档断裂**: 引导进度独立于引擎主存档，清除浏览器数据即丢失
4. **双系统冗余**: GuideSystems 和 TutorialSystem 功能重叠，增加维护成本和出错概率
5. **功能锁定缺失**: 引导系统不控制功能可用性，玩家可以跳过引导直接使用所有功能

| **总评** | **6.0/10** | 架构优秀但集成不完整，需修复关键断裂点 |
