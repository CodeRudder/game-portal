## 维度1: PRD/UI一致性

### 追问1: PRD定义了哪些地图元素（城市/关卡/资源点）？代码实现了几种？

**PRD定义**（`02-map-system.md §一`）：
- 15块领土（三大区域各4城 + 3中立城市）
- 6种地形：平原、山地、水域、城池、森林、关隘
- 3个特殊地标：洛阳（全资源+50%）、长安（科技点+30%）、建业（铜钱+30%）

**代码实现**（`map-config.ts DEFAULT_LANDMARKS`）：

| 类型 | PRD定义 | 代码实现 | 一致？ |
|------|---------|---------|:------:|
| 魏国城市 | 4座（邺城/许昌/濮阳/北海） | 4座 ✅ | ✅ |
| 蜀国城市 | 4座（成都/汉中/永安/南中） | 4座 ✅ | ✅ |
| 吴国城市 | 4座（建业/会稽/柴桑/庐江） | 4座 ✅ | ✅ |
| 中立城市 | 3座（洛阳/长安/襄阳） | 3座 ✅ | ✅ |
| 关隘 | PRD未明确数量 | 4座（虎牢关/潼关/剑阁/阳平关） | ⚠️ |
| 资源点 | PRD未定义 | 5个（许田/金矿场/稻田/兵营/天命台） | ⚠️ |

**⚠️ 差异发现**：

| 差异 | 影响 | 优先级 |
|------|------|:------:|

### 追问2: 攻城战斗流程是否完整？（选择目标→派兵→战斗→结果）

**追踪完整代码链路**：

```
① WorldMapTab 点击领土 → handleSelectTerritory(id) → TerritoryInfoPanel 显示
② TerritoryInfoPanel 攻城按钮 → onSiege(id) → handleSiege(id)
③ handleSiege → 
   外部模式: onSiegeTerritory(id) → SceneRouter
   内部模式: setSiegeTarget(target) → SiegeConfirmModal 弹出
④ SiegeConfirmModal 确认 → onConfirm → handleSiegeConfirm
⑤ handleSiegeConfirm → engine.getMapSystem().executeSiege(id)
⑥ SiegeSystem.executeSiege() → checkSiegeConditions → simulateBattle → resolveSiege
⑦ resolveSiege → victory? captureTerritory() + emit('siege:victory') : emit('siege:defeat')
⑧ SceneRouter → Toast.success/warning (仅Toast，无结果弹窗)
```

**流程完整性评估**：

| 环节 | 实现状态 | 代码路径 |
|------|:--------:|---------|

| **结果展示** | ❌ | 仅Toast，无详细战报 |
| 领土归属变更 | ✅ | TerritorySystem.captureTerritory() |
| 产出更新 | ✅ | getPlayerProductionSummary() 自动重算 |

**⚠️ 关键缺失**：
1. **派兵选择**（P1）：玩家无法策略性分配兵力，降低了攻城的策略深度

## 维度3: 资源经济平衡

### 追问1: 占领领土的收益是否值得攻城消耗？

**成本收益分析**（以许昌Lv4为例）：

| 项目 | 数值 |
|------|------|
| **攻城消耗** | |
| 兵力消耗 | ceil(100 × 4000/100) = 4000 |
| 粮草消耗 | 500 |
| **占领收益（每秒）** | |
| 粮草 | 5 × 2.0 = 10.0/s |
| 铜钱 | 5 × 2.0 = 10.0/s |
| 兵力 | 3 × 2.0 = 6.0/s |
| 天命 | 1 × 2.0 = 2.0/s |
| **回本时间** | |
| 粮草回本 | 500 / 10.0 = 50秒 |
| 兵力回本 | 4000 / 6.0 ≈ 667秒（~11分钟） |

**✅ 结论**：粮草回本极快（50秒），兵力回本约11分钟。占领领土的收益远大于消耗，攻城是高回报投资。

### 追问3: 防守方AI是否有合理的驻军？

**代码验证**：

- `SiegeSystem` 中无独立的AI驻军系统
- 城防值由 `territory-config.ts calculateBaseDefenseValue()` 生成：`1000 × level`
- `GarrisonSystem` 管理武将驻防，但仅限玩家操作
- `SiegeEnhancer.calculateDefenderPower()` 考虑了驻防加成

**⚠️ 问题**：中立/敌方领土的防御完全依赖静态城防值（`1000 × level`），无动态驻军系统。这意味着：
- Lv1中立领土城防仅1000，任何兵力都能轻松攻占
- Lv5领土城防5000，需要至少5000兵力才有机会
- 但没有"驻军恢复"或"驻军增长"机制

**评估**：对于放置游戏，静态城防值是合理的设计简化。但长期来看，缺少AI驻军增长机制会让游戏后期过于简单。

| 城市 | 等级 | 城防值 | 兵力消耗 | 胜率（5000兵力） |
|------|:----:|:------:|:-------:|:---------------:|

| 潼关 | 3 | 3000 | 3000 | ~33% |
| 许昌 | 4 | 4000 | 4000 | ~12.5% |
| 邺城 | 4 | 4000 | 4000 | ~12.5% |
| 襄阳 | 4 | 4000 | 4000 | ~12.5% |

**胜率计算**（PRD §7.6 线性比率公式）：
- 胜率 = min(95%, max(5%, (攻方有效兵力/防方战力) × 50%))
- 5000兵力攻虎牢关：有效兵力 = 5000 - 3000 = 2000，胜率 = (2000/3000) × 50% = 33%
- 5000兵力攻许昌：有效兵力 = 5000 - 4000 = 1000，胜率 = (1000/4000) × 50% = 12.5%

**⚠️ 问题**：新玩家积累到5000兵力后，对最近的许昌/邺城胜率仅12.5%。需要更多兵力才能有合理胜率（50%胜率需要约8000兵力攻许昌）。

**评估**：对放置游戏来说，低胜率是可以接受的 — 鼓励玩家积累更多兵力或选择更简单的目标（关隘）。但需要在UI上显示胜率预估，帮助玩家做出决策。

## 维度6: 常见问题检查

### 1. 攻城后兵力是否正确扣除？

**代码验证**（`SiegeSystem.resolveSiege()`）：

```typescript
// 胜利时：消耗cost.troops（通过autoGarrison事件通知上层扣减）
// 失败时：损失30%出征兵力
const defeatTroopLoss = Math.floor(cost.troops * 0.3);
result.defeatTroopLoss = defeatTroopLoss;
```

**⚠️ 问题**：胜利时的兵力扣减依赖上层消费 `siege:victory` 事件。如果上层（SceneRouter）未正确处理事件，兵力可能不会被扣除。

**代码验证**（`SceneRouter.tsx`）：
```typescript
// SceneRouter仅显示Toast，未扣减资源
onSiegeTerritory={(territoryId: string) => {
  const result = siegeSys.executeSiege(territoryId, 'player', troops, grain);
  if (result.victory) {
    Toast.success(`攻城成功！占领了 ${result.targetName}`);
  }
}}
```

**⚠️ 关键发现**：`SiegeSystem.executeSiege()` 内部调用 `resolveSiege()` 时，仅发出事件和变更归属，**并未直接扣减资源**。资源扣减依赖上层消费事件或另行处理。

**影响**：如果上层未正确扣减资源，可能出现"攻城了但资源没扣"的bug。

**建议**：SiegeSystem应在 `resolveSiege()` 中直接调用 ResourceSystem 扣减资源，而非依赖事件通知。

### 2. 连续攻城是否有限制？（冷却时间）

**代码验证**：

| 限制类型 | 实现 | 代码依据 |
|---------|------|---------|
| 每日次数 | ✅ 3次/天 | `DAILY_SIEGE_LIMIT = 3` |
| 占领冷却 | ✅ 24h | `CAPTURE_COOLDOWN_MS = 24h` |
| 跨天重置 | ❌ 需外部调用 | `resetDailySiegeCount()` |

**⚠️ 问题**：每日攻城次数重置依赖外部调用 `resetDailySiegeCount()`，无自动跨天重置机制。如果上层忘记调用，玩家永远无法继续攻城。

### 3. 失去所有领土后是否有保底机制？

**代码检查**：
- `TerritorySystem` 无"失去领土"的主动逻辑（无AI反攻）
- `SiegeSystem` 仅处理玩家发起的攻城
- 玩家领土永远不会被AI攻占

**✅ 结论**：当前不存在"失去所有领土"的场景。玩家领土是永久的。这是放置游戏的合理设计 — 不惩罚玩家。

### 4. 地图滚动/缩放是否正常？

**代码验证**（`WorldMapSystem.ts`）：

| 功能 | 实现 | 代码依据 |
|------|------|---------|
| 视口偏移 | ✅ | `setViewportOffset()`, `panViewport()` |
| 缩放控制 | ✅ | `setZoom()`, `zoomViewport()` |
| 缩放范围 | ✅ 50%~200% | `VIEWPORT_CONFIG.minZoom/maxZoom` |
| 重置视口 | ✅ | `resetViewport()` |

**⚠️ 问题**：`WorldMapSystem` 提供了视口控制API，但 `WorldMapTab.tsx` UI组件使用CSS Grid布局，未使用瓦片渲染和视口控制。地图滚动/缩放功能在UI层未实现。

### 5. 多个城市同时被攻击如何处理？

**代码验证**：

**✅ 结论**：当前设计为串行攻城，一次一个目标。对于放置游戏是合理的设计。

## 封版判定

### 当前评分：8.3/10（B+）

### 距离封版（9.0+）需要修复的问题

| 优先级 | 需修复问题 | 预期评分提升 |
|--------|-----------|-------------|

| **P1-3** | 胜率预估UI | +0.1 |
| — | 补充经济流ACC测试 | +0.1 |

**预估修复后评分**：9.3/10（A-）

### 封版判定

**❌ 未达到9.0分封版标准**

**核心差距**：
1. **P0-1 攻城结果弹窗**：当前仅Toast反馈，玩家无法了解战斗细节，是攻城体验的最大短板
2. **P0-3 资源扣减不直接**：依赖事件通知而非直接扣减，存在资源不一致风险
3. **P1-1 兵力部署**：缺少策略性兵力分配，降低了攻城的策略深度

**建议封版路径**：
1. **R2目标**：修复P0-1（结果弹窗）+ P0-2（奖励展示）+ P0-3（资源扣减）→ 达到8.8+
2. **R3目标**：修复P1-1（兵力滑块）+ P1-2（自动重置）+ P1-3（胜率UI）→ 达到9.2+
3. **R4目标**：补充ACC测试 + P2级体验优化 → 达到9.5+

### 架构亮点

1. **引擎层分层清晰**：SiegeSystem（攻城逻辑）→ TerritorySystem（领土管理）→ SiegeEnhancer（增强功能）→ GarrisonSystem（驻防），职责分明
2. **配置与逻辑分离**：`map-config.ts`（静态配置）vs `WorldMapSystem.ts`（动态逻辑），便于调整
3. **相邻关系双向对称**：`makeSymmetric()` 确保邻接表双向一致，避免单向连接bug
4. **双模式攻城设计**：WorldMapTab 支持外部回调和内部集成两种攻城模式，灵活度高
5. **胜率公式透明**：PRD §7.6 线性比率公式，攻防相等50%，直观易懂
6. **测试覆盖充分**：76个ACC/FLOW测试，涵盖正常流程+边界+异常
