### 3. 架构设计 — 9.9/10

**DDD域划分**:
```
core/（基础设施层）
  ├── events/EventBus.ts     — 事件总线（发布/订阅/通配符）
  ├── engine/SubsystemRegistry.ts — 子系统注册/查找
  ├── save/SaveManager.ts    — 存档管理器
  ├── config/ConfigRegistry.ts — 配置注册表
  └── types/                 — 基础接口定义

engine/（引擎层）
  ├── ThreeKingdomsEngine.ts — 编排层（500行，不超限）
  ├── resource/              — 资源域聚合根
  ├── building/              — 建筑域聚合根
  └── calendar/              — 日历域聚合根

ui/（展示层）
  ├── ThreeKingdomsGame.tsx  — 主容器
  ├── panels/                — 面板组件
  └── common/                — 通用组件库
```

**架构亮点**:
1. **ISubsystem统一接口**: `init(deps)` / `update(dt)` / `getState()` / `reset()` 四方法契约，新子系统只需实现接口
2. **子系统零跨域引用**: Building/Resource/Calendar完全独立，仅通过EventBus通信
3. **依赖注入**: `ISystemDeps`接口注入EventBus/Config/Registry，子系统不自行创建依赖
4. **不可变数据**: `cloneResources()` / `{...spread}` 确保无外部可变引用泄漏
5. **变化检测优化**: JSON浅比较避免无变化时频繁emit（`ThreeKingdomsEngine.ts:L290-305`）

**扣分说明**: 架构审查报告指出1处UI→Subsystem类型违规（`useGameEngine.ts:20`引用了`resource.types`内部类型），已记录但未修复（-0.1分）。

### 8. 存档可靠性 — 9.9/10

**存档架构**:
```
ThreeKingdomsEngine.save()
  → GameSaveData → IGameState → StateSerializer.serialize()
    → { v, checksum, data } → localStorage

ThreeKingdomsEngine.load()
  → localStorage → StateSerializer.deserialize()
    → IGameState → GameSaveData → applyLoadedState()
      → computeOfflineAndFinalize()
```

**可靠性验证**:
- ✅ **序列化往返一致性**: 测试验证 serialize→deserialize 后资源数值完全相等
- ✅ **损坏存档处理**: 非法JSON返回null，不抛异常
- ✅ **版本兼容**: 版本不匹配时console.warn但仍尝试加载
- ✅ **旧格式兼容**: `tryLoadLegacyFormat()` 支持无checksum包装的旧存档
- ✅ **离线收益**: 5段衰减（100%→80%→60%→40%→25%），72h封顶，15%封底效率
- ✅ **自动保存**: 30秒间隔，SaveManager委托，成功后发出game:saved事件
- ✅ **离线期间升级完成**: `BuildingSystem.deserialize()` 自动处理离线期间完成的建筑升级

**扣分说明**: 
- 无存档压缩机制，大型存档可能占用较多localStorage空间（-0.1分，对v1.0规模影响极小）

## 亮点（Top 5）

### 1. 🏗️ ISubsystem统一接口设计
所有子系统实现统一的`ISubsystem`接口（`init/update/getState/reset`），通过`ISystemDeps`依赖注入，`SubsystemRegistry`管理注册/查找。扩展新子系统只需实现接口，零修改现有代码。这是教科书级的开闭原则实践。

### 2. 🔗 子系统零跨域引用
Building/Resource/Calendar三个域完全独立，互不import，仅通过EventBus事件通信。这符合DDD聚合根原则，使得各域可以独立开发、测试和替换。

### 3. 🧮 纯函数计算模块分离
`resource-calculator.ts`(272行)和`OfflineEarningsCalculator.ts`(166行)为无状态纯函数模块，可独立测试，不持有任何状态。Config文件全部纯数据零逻辑。这种"数据-计算-状态"三层分离非常清晰。

### 4. 💾 存档系统健壮性
SaveManager + StateSerializer双层架构，支持checksum校验、版本兼容、旧格式迁移。离线收益5段衰减+72h封顶的设计既鼓励回归又防止刷资源。建筑系统deserialize时自动处理离线期间完成的升级。

### 5. 🎨 CSS变量设计体系
水墨江山·铜纹霸业风格通过CSS变量体系统一管理（主色调/强调色/文字色/字体/间距/圆角/动画/阴影），组件间视觉高度一致。容量进度条3色警告（绿→橙→红）直观有效。

**现状**: `CalendarSystem`引擎完整实现年号/季节/天气，但主界面仅静态显示"建安元年 春 ☀️晴"，未随游戏时间实时更新。  
**影响**: 玩家感知不到时间流逝，日历系统形同虚设。  
**建议**: 在`ThreeKingdomsGame.tsx`中订阅`CALENDAR_DAY_CHANGED`事件，将日历数据注入快照，ResourceBar或独立日历组件实时显示。  
**预估工时**: 2小时。

### [P1] 2. Engine构造函数 `as any` 类型断言
**现状**: `ThreeKingdomsEngine.ts:L48-49`使用`this.resource as any`绕过类型检查。  
**影响**: 类型安全性降低，重构时可能遗漏错误。  
**建议**: 让`SubsystemRegistry.register`使用泛型或调整接口继承关系，消除断言。  
**预估工时**: 1小时。

### [P2] 3. 资源产出粒子效果未实现
**现状**: PLAN #12（P2优先级）资源产出时飞出粒子动画未实现。  
**影响**: 视觉反馈层次不够丰富，产出感不够"爽"。  
**建议**: v1.1版本使用CSS animation或轻量粒子库实现，参考UI设计文档"产出粒子效果"章节。  
**预估工时**: 4小时。

### [P2] 4. 建筑升级路线推荐未实现
**现状**: PLAN #19（P2优先级）新手/发展/中后期路线推荐UI未实现。  
**影响**: 新手可能不知道优先升级哪些建筑。  
**建议**: v1.1版本在BuildingPanel中增加"推荐"标签，根据当前主城等级显示推荐升级路线。  
**预估工时**: 3小时。

### [P2] 5. UI层类型引用违规
**现状**: `useGameEngine.ts:20`直接引用`engine/resource/resource.types`中的`OfflineEarnings`类型。  
**影响**: 违反分层原则，UI层不应依赖子系统内部类型。  
**建议**: 将`OfflineEarnings`类型提升到`shared/types.ts`。  
**预估工时**: 30分钟。
