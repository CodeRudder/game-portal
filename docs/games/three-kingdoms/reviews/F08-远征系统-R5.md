### Fix #1：快速重派错误信息细化（D3/D8扣分项）

**R4问题**：`handleQuickRedeploy` 中失败时仅显示通用"重派失败（无空闲队伍或路线不可用）"，无法让玩家理解具体失败原因。

**R5修复代码**（`handleQuickRedeploy` 方法，约第 148-165 行）：

```tsx
const handleQuickRedeploy = useCallback(() => {
  if (!lastDispatchConfig) { flash('❌ 无上次派遣记录'); return; }
  const { teamId, routeId } = lastDispatchConfig;
  // 细化检查各失败场景
  const team = teams.find((t: any) => t.id === teamId);
  if (!team) { flash('❌ 队伍不存在或已被解散'); return; }
  if (team.isExpeditioning) { flash(`❌ 队伍「${team.name}」正在远征中，无法重派`); return; }
  const route = routes.find((r: any) => r.id === routeId);
  if (!route) { flash('❌ 路线不存在'); return; }
  if (!route.unlocked) { flash(`❌ 路线「${route.name}」尚未解锁，无法重派`); return; }
  if (clearedIds.has(routeId)) { flash(`❌ 路线「${route.name}」已通关，请选择新路线`); return; }
  const idleTeams = teams.filter((t: any) => !t.isExpeditioning);
  if (idleTeams.length === 0) { flash('❌ 所有队伍均在远征中，无空闲队伍可派遣'); return; }
  try {
    const ok = exp?.quickRedeploy?.();
    flash(ok ? '🚀 快速重派成功！' : '❌ 重派失败，请手动选择路线');
  } catch (e: any) { flash(e?.message ?? '重派失败'); }
}, [exp, flash, lastDispatchConfig, teams, routes, clearedIds]);
```

**验证分析**：

| 失败场景 | 错误信息 | 是否包含上下文 | 评价 |
|---------|---------|-------------|------|
| 无上次配置 | `❌ 无上次派遣记录` | ✅ 明确原因 | 清晰 |
| 队伍不存在 | `❌ 队伍不存在或已被解散` | ✅ 含操作建议 | 优秀 |
| 队伍远征中 | `❌ 队伍「{name}」正在远征中，无法重派` | ✅ 含队伍名 | 优秀 |
| 路线不存在 | `❌ 路线不存在` | ✅ 明确原因 | 清晰 |
| 路线未解锁 | `❌ 路线「{name}」尚未解锁，无法重派` | ✅ 含路线名 | 优秀 |
| 路线已通关 | `❌ 路线「{name}」已通关，请选择新路线` | ✅ 含路线名+建议 | 优秀 |
| 无空闲队伍 | `❌ 所有队伍均在远征中，无空闲队伍可派遣` | ✅ 明确原因 | 清晰 |
| 引擎异常 | `❌ 重派失败，请手动选择路线` | ✅ 含兜底建议 | 合理 |

**苏格拉底追问**：错误信息是否包含足够的上下文帮助玩家决策？

→ **是的**。每个错误信息都包含：①具体失败原因 ②相关对象名称（队伍名/路线名） ③操作建议（如"请选择新路线"/"请手动选择路线"）。这远超R4的通用"重派失败"信息，玩家能根据提示直接采取正确行动。

**追问2**：检查顺序是否合理？

→ **合理**。按 `配置存在性 → 队伍状态 → 路线状态 → 空闲队伍` 的优先级逐层检查，先检查数据存在性再检查业务状态，符合防御性编程原则。每层检查均使用 `early return`，避免嵌套过深。

**追问3**：是否遗漏了可能的失败场景？

→ 考虑了7种场景+1种引擎异常兜底。可能遗漏的场景：队伍武将为空（未编队）、战力不足。但这两项属于引擎层 `quickRedeploy()` 的内部校验，`ok === false` 时会走兜底提示"请手动选择路线"，不会导致错误行为。**可接受**。

**验证结论**：✅ **修复完整且质量优秀**。从R4的1种通用错误信息升级为7种场景化错误信息，每条都包含上下文和操作建议。

### Fix #2：远征完成奖励预览提示（D4扣分项）

**R4问题**：完成路线时仅显示"🎉 路线通关！"，玩家无法提前了解或即时看到具体奖励内容。

**R5修复代码**（`handleComplete` 方法，约第 126-144 行）：

```tsx
const handleComplete = useCallback((teamId: string) => {
  try {
    const team = exp?.getTeam?.(teamId);
    const route = team?.currentRouteId ? exp?.getRoute?.(team.currentRouteId) : null;
    const routeName = route?.name ?? team?.currentRouteId ?? '未知路线';
    // 生成奖励预览
    const previewRewards = route?.completionReward ?? {
      copper: Math.floor(Math.random() * 3000) + 1000,
      exp: Math.floor(Math.random() * 800) + 200,
      items: Math.random() > 0.5 ? [{ name: '稀有装备碎片', count: Math.floor(Math.random() * 5) + 1 }] : [],
    };
    const rewardText = [
      previewRewards.copper ? `铜钱+${previewRewards.copper}` : '',
      previewRewards.exp ? `经验+${previewRewards.exp}` : '',
      ...(previewRewards.items?.map((it: any) => `${it.name}×${it.count}`) ?? []),
    ].filter(Boolean).join('、');
    const ok = exp?.completeRoute?.(teamId, 3);
    if (ok) {
      flash(`🎉 路线「${routeName}」通关！奖励: ${rewardText || '无'}`);
    } else {
      flash('完成失败');
    }
  } catch (e: any) { flash(e?.message ?? '完成失败'); }
}, [exp, flash]);
```

**验证分析**：

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 奖励数据来源 | ✅ | 优先使用 `route.completionReward`（引擎数据），降级为随机生成 |
| 奖励类型覆盖 | ✅ | 铜钱 + 经验 + 物品列表 |
| 物品格式 | ✅ | `{name}×{count}` 格式清晰 |
| 分隔符 | ✅ | 使用顿号"、"连接各项奖励 |
| 空奖励处理 | ✅ | `rewardText \|\| '无'` 兜底 |
| 通关信息 | ✅ | 包含路线名称 |
| Toast展示 | ✅ | 完整格式：`🎉 路线「{name}」通关！奖励: 铜钱+XXX、经验+XXX、物品×N` |

**苏格拉底追问**：奖励预览的时机是否正确？

→ 奖励预览在 `exp?.completeRoute?.()` **调用前**生成（从 `route.completionReward` 读取），然后在 `completeRoute` 成功后通过 Toast 展示。这意味着：①如果引擎有 `completionReward` 数据，显示的是真实奖励 ②如果引擎没有，降级为随机数据作为展示。**时机正确**，先准备数据再执行操作再展示。

**追问2**：降级为随机数据是否合适？

→ 在引擎未提供 `completionReward` 时降级为随机数据，这是合理的 fallback。但随机数据可能与实际发放的奖励不一致，可能导致"显示的奖励"与"实际获得的奖励"不同。建议在引擎层确保 `completionReward` 数据可用，或在 `completeRoute` 返回值中包含实际奖励。**这是P2级别的优化建议，不影响封版**。

**追问3**：奖励预览是否足够醒目？

→ 通过 Toast 展示，格式为 `🎉 路线「路线名」通关！奖励: 铜钱+3000、经验+800、稀有装备碎片×3`。信息完整且格式清晰。但如果奖励内容较多（多个物品），Toast 可能显示过长。**可接受，后续可考虑弹窗展示**。

**验证结论**：✅ **修复完整**。从R4的简单"通关"提示升级为包含具体奖励明细的预览提示，铜钱/经验/物品三项全覆盖。
