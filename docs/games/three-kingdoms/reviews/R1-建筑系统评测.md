## 维度1: PRD/UI一致性

### 追问1: PRD定义了哪些建筑类型？代码实现了几种？是否一致？

**PRD定义**（`BLD-buildings-prd.md §BLD-1`）：
- 8座建筑：主城、农田、市集、兵营、铁匠铺、书院、医馆、城墙

**代码实现**（`building-config.ts` BUILDING_DEFS）：
- 8座建筑：castle、farmland、market、barracks、smithy、academy、clinic、wall

**✅ 结论**：完全一致，8种建筑类型PRD与代码一一对应。

### 追问2: UI设计稿定义了什么布局？代码实现的布局是否一致？

**UI设计稿**（`BLD-buildings.md §BLD-1`）：
- PC端：6×5网格布局，每格180×160px，间距16px
- 8座建筑按固定网格位置放置
- 升级队列右上角悬浮，280×80px

**代码实现**（`BuildingPanel.tsx`）：
- PC端：使用绝对定位百分比布局（`BUILDING_MAP_POSITIONS`），非网格
- 8座建筑按百分比位置放置（top/left）
- 升级队列右上角悬浮 ✅
- 手机端：纵向列表布局 ✅

**⚠️ 差异发现**：
| 对比项 | UI设计稿 | 代码实现 | 影响 |
|--------|---------|---------|------|
| 布局方式 | 6×5网格 | 绝对定位百分比 | **P2** — 视觉效果不同但功能一致 |
| 卡片尺寸 | 固定180×160px | 响应式百分比 | **P2** — 适配性更好 |
| 空地块 | 有空地+建造按钮 | 无空地块 | **P1** — PRD §BLD-1-2 定义了空地块，代码未实现 |

## 维度2: 游戏玩法流程完整性

### 追问1: 玩家进入游戏后，建筑Tab默认显示什么内容？

**追踪代码链路**：
1. `BuildingPanel.tsx` 渲染 → `BUILDING_TYPES.map(type => ...)` 遍历8座建筑
2. 初始状态（`BuildingStateHelpers.createAllStates()`）：
   - 主城(castle): Lv1, idle ✅
   - 农田(farmland): Lv1, idle ✅
   - 市集(market): Lv0, locked 🔒
   - 兵营(barracks): Lv0, locked 🔒
   - 铁匠铺(smithy): Lv0, locked 🔒
   - 书院(academy): Lv0, locked 🔒
   - 医馆(clinic): Lv0, locked 🔒
   - 城墙(wall): Lv0, locked 🔒

**✅ 结论**：新玩家看到主城和农田已解锁，其余6座建筑锁定，符合PRD设计。

### 追问2: 每个建筑的升级条件是什么？是否合理？

**代码实现**（`BuildingSystem.checkUpgrade()`）：

| 条件类型 | 代码逻辑 | PRD定义 | 一致？ |
|---------|---------|---------|--------|
| 建筑未锁定 | `status === 'locked'` → 拒绝 | ✅ | ✅ |
| 未在升级中 | `status === 'upgrading'` → 拒绝 | ✅ | ✅ |
| 等级上限 | `level >= maxLevel` → 拒绝 | ✅ | ✅ |
| 子建筑≤主城+1 | `level > castle.level` → 拒绝 | PRD: ≤主城等级 | ⚠️ 见下 |
| 主城特殊前置 | Lv4→5需其他建筑Lv4, Lv9→10需其他建筑Lv9 | ✅ | ✅ |
| 队列未满 | `isQueueFull()` → 拒绝 | ✅ | ✅ |
| 资源充足 | 逐项检查grain/gold/troops | ✅ | ✅ |

**⚠️ 差异**：PRD定义"每座建筑等级 ≤ 主城等级"，代码实现为"允许子建筑领先主城1级"（`level > castle.level` 才拒绝）。这是一个有意的P0-1修复，改善新手体验，**合理但需要文档同步**。

### 追问3: 升级等待时间如何计算？是否有加速机制？

**代码实现**：
- 升级时间来自 `LevelData.upgradeCost.timeSeconds`，为静态配置值
- `startUpgrade()` 中 `endTime = now + cost.timeSeconds * 1000`
- `tick()` 每帧检查 `now >= endTime` 自动完成

**⚠️ 加速机制缺失**：PRD §BLD-4 定义了"铜钱加速 / 天命加速 / 元宝秒完成"，代码完全未实现。目前只有：
- `forceCompleteUpgrades()` — 仅测试用
- `cancelUpgrade()` — 取消返还80%

### 追问4: 升级完成后，产出增加是否立即可见？

**追踪代码链路**：
1. `tick()` → `state.level += 1` → 返回completed列表
2. `BuildingPanel` 通过 `snapshotVersion` prop 重渲染
3. `getProductionText()` 调用 `engine.building.getProduction(type)` 查表
4. 查表逻辑：`levelTable[lv - 1].production`，等级变化后立即返回新值

**✅ 结论**：升级完成后产出立即更新，无需额外操作。

### 追问5: 建筑等级上限是多少？满级后如何处理？

**代码实现**（`BuildingUpgradeModal.tsx`）：
- 检测 `info.level >= maxLevel` → 显示"🏆 已达最高等级"
- 确认按钮显示"已满级"并禁用
- 各建筑上限：主城30、农田/市集/兵营25、铁匠铺/书院/医馆/城墙20

**✅ 结论**：满级处理正确，UI清晰提示。

## 维度3: 资源经济平衡

### 追问1: 初始资源能否支撑第一次建筑升级？

**计算验证**：

| 资源 | 初始值 | 农田Lv1→2消耗 | 主城Lv1→2消耗 |
|------|--------|-------------|-------------|
| 粮草 | 500 | 100 ✅ | 200 ✅ |
| 铜钱 | 300 | 50 ✅ | 150 ✅ |
| 兵力 | 50 | 0 ✅ | 0 ✅ |

**✅ 结论**：初始资源500粮/300铜/50兵，足以升级农田到Lv2或主城到Lv2，甚至两者都升（消耗300粮+200铜，初始500+300=800，余粮200铜100）。

### 追问2: 建筑产出是否足以支撑后续升级消耗？

**经济模型分析 — 前5级**：

| 操作 | 累计消耗（粮/铜） | 产出速率变化 | 回本时间 |
|------|------------------|-------------|---------|
| 农田Lv1→2 | 100粮+50铜 | 0.8→1.0 粮/s (+0.2) | 500s粮 / — 铜 |
| 主城Lv1→2 | 200粮+150铜 | 全资源+2% | ~10min |
| 农田Lv2→3 | 250粮+120铜 | 1.0→1.5 粮/s (+0.5) | 500s粮 / 240s铜 |
| 主城Lv2→3 | 500粮+400铜+50兵 | 全资源+4% | ~15min |

**✅ 结论**：早期经济循环健康，产出增长足以覆盖升级成本。主城+2%全资源加成是关键倍增器。

### 追问3: 资源上限是否会限制产出？溢出如何处理？

**代码验证**（`resource-config.ts`）：
- 粮草上限：初始2000，随等级提升
- 铜钱上限：`null`（无上限）
- 兵力上限：初始500

**粮草溢出风险**：农田Lv1产出0.8/s，2000上限约2500秒（~42分钟）填满。新玩家如果42分钟不操作，粮草产出浪费。

**⚠️ 问题**：新玩家初始粮草上限2000，被动产出0.8/s，约42分钟满仓。对于放置游戏，这个时间偏短。建议初期上限提高到5000或增加溢出保护提示。

### 追问4: 不同资源类型之间的兑换/转化是否平衡？

**代码检查**：建筑系统本身不涉及资源兑换。资源转化通过贸易系统（TRD）处理，不在本次评测范围内。

### 追问5: 长期来看，经济系统是否可持续？

**数值曲线分析**：
- 费用增长：×1.8（Lv5-10）→ ×1.6（Lv10-15）→ ×1.5（Lv15-20）→ ×1.4（Lv20+）
- 产出增长：线性到准线性（0.8→45 for farmland）
- 主城加成：+2%/级，Lv30时+58%

**✅ 结论**：费用指数增长 vs 产出线性增长，长期需要主城加成（乘数效应）和科技加成来维持经济循环。设计合理，符合放置游戏的"加速-瓶颈-突破"节奏。

## 维度4: 初始数据合理性

### 追问1: 新玩家初始有哪些建筑？等级分别是多少？

**代码验证**（`BuildingStateHelpers.createInitialState()` + `BUILDING_UNLOCK_LEVELS`）：

| 建筑 | 解锁等级 | 初始状态 | 初始等级 |
|------|---------|---------|---------|
| 主城 | 0 | idle | 1 |
| 农田 | 0 | idle | 1 |
| 市集 | 2 | locked | 0 |
| 兵营 | 2 | locked | 0 |
| 铁匠铺 | 3 | locked | 0 |
| 书院 | 3 | locked | 0 |
| 医馆 | 4 | locked | 0 |
| 城墙 | 5 | locked | 0 |

**✅ 结论**：2座初始建筑（主城+农田），6座待解锁，符合PRD §BLD-1。

### 追问2: 初始资源是否足够升级第一个建筑？

**见维度3追问1** — ✅ 足够，且可以同时升级2座建筑。

### 追问3: 如果不升级，被动产出多久能让玩家进行第一次操作？

**计算**：
- 初始粮草：500，粮草上限：2000
- 农田Lv1产出：0.8粮/s
- 农田Lv1→2需要：100粮+50铜
- 铜钱初始产出：0（市集未解锁）
- 铜钱初始持有：300

**关键问题**：如果玩家不操作，铜钱不会增长（市集未解锁），初始300铜钱用完后无法继续升级（农田升级也需要铜钱）。

**计算**：
- 初始300铜钱可支持：农田Lv1→2(50铜) + 主城Lv1→2(150铜) + 农田Lv2→3(120铜) = 320铜 → 不够！
- 可支持：农田Lv1→2(50铜) + 主城Lv1→2(150铜) = 200铜 → 余100铜
- 第3次升级（农田Lv2→3需120铜）→ 铜钱不够

**⚠️ 问题**：铜钱是早期瓶颈资源。初始300铜钱只能支持2次升级。解锁市集（主城Lv2后）后才有铜钱产出。设计意图是引导玩家先升主城解锁市集，这是合理的。

### 追问4: 初始数据是否让玩家有"立即有事做"的感觉？

**✅ 结论**：
- 进入游戏即可看到2座建筑（主城Lv1、农田Lv1）
- 资源足够立即升级 → 有事可做
- 升级主城到Lv2可解锁2座新建筑 → 有目标感
- PRD §BLD-5 推荐路线清晰：主城→Lv2→解锁市集/兵营

### 追问5: 与同类放置游戏相比，初始进度是否合理？

**对比分析**：
| 对比项 | 三国霸业 | 放置奇兵 | 天天三国 |
|--------|---------|---------|---------|
| 初始建筑 | 2座 | 3-4座 | 2-3座 |
| 首次升级时间 | 即时（资源够） | 1-2分钟 | 即时 |
| 首个解锁节点 | 主城Lv2（~10秒） | 5分钟 | 3分钟 |
| 初始资源量 | 中等 | 丰富 | 中等 |

**✅ 结论**：初始进度合理，略偏快节奏，符合现代放置游戏"快速上手"趋势。

## 维度6: 常见问题检查

### 1. 建筑升级后资源是否立即扣除？

**✅ 正确**。`executeBuildingUpgrade()` 中先 `consumeBatch()` 扣费，再 `startUpgrade()` 开始升级。顺序正确，不会出现"升级了但资源没扣"。

### 2. 同时升级多个建筑是否有限制？

**✅ 正确**。队列槽位机制：
- 主城Lv1-5：1个槽位
- 主城Lv6-10：2个槽位
- 主城Lv11-20：3个槽位
- 主城Lv21-30：4个槽位

`checkUpgrade()` 中检查 `isQueueFull()`，队列满时拒绝新升级。

### 3. 升级中途关闭游戏，重新进入后状态是否正确恢复？

**✅ 正确**。`deserialize()` 中：
1. 遍历所有建筑，检查 `status === 'upgrading'` 且 `upgradeEndTime` 已过 → 自动完成升级
2. 未完成的升级 → 重新加入队列
3. 调用 `checkAndUnlockBuildings()` 检查新解锁

**测试覆盖**：`FLOW-02-38`（序列化/反序列化）、`FLOW-02-39`（离线期间完成升级）。

### 4. 建筑产出是否受离线时间影响？

**✅ 正确**。离线收益由 `OfflineRewardSystem` 处理，建筑产出通过 `calculateTotalProduction()` 参与。离线效率分段衰减（100%→80%→60%→40%→20%），最长计算72小时。

### 5. 快速点击升级按钮是否会导致重复升级？

**✅ 有保护**。`BuildingUpgradeModal.tsx` 中：
- `const [upgrading, setUpgrading] = useState(false)` — 防重复点击状态
- `handleConfirm()` 中先 `setUpgrading(true)` 再调用 `onConfirm`
- `isConfirmDisabled` 包含 `upgrading` 条件

**测试覆盖**：`ACC-02-33`（连续快速点击只触发一次）。

### ✅ P2-5 已修复：城墙产出类型不一致

**问题描述**：`BUILDING_DEFS.wall.production.resourceType` 为 `recruitToken`（招贤令），但PRD §BLD-1 明确定义城墙产出为"城防值/防御加成"。

**根因分析**：
- PRD §BLD-1 建筑总览表：城墙核心产出 = "城防值/防御加成"
- PRD §BLD-2 城墙等级数据：城防值（固定值）+ 防御加成（百分比）
- 代码中城墙的 `production` 字段被错误地设置为 `resourceType: 'recruitToken'`
- `recruitToken`（招贤令）由独立的 `RecruitTokenEconomySystem` 管理，与城墙无关

**修复方案**：移除城墙 `BUILDING_DEFS.wall` 的 `production` 字段。城墙的产出通过以下方式正确表达：
- `levelTable[lv-1].production`：防御加成百分比（如 +3%、+5%）
- `levelTable[lv-1].specialValue`：城防值（如 300、500）
- `BuildingSystem.getWallDefense()`：获取城防值
- `BuildingSystem.getWallDefenseBonus()`：获取防御加成

**验证结果**：
- ✅ TypeScript编译通过
- ✅ `calculateTotalProduction()` 会跳过无 `production` 字段的建筑（`if (!def.production) continue`）
- ✅ 88个建筑ACC/FLOW测试全部通过
- ✅ 城墙不再错误地产出招贤令

### ⏸️ P1-5 可行性评估：一键收取功能

**PRD定义**（§BLD-5 一键收取）：
> - 右下角悬浮按钮，一键收取所有建筑累积产出
> - 收取动画：资源数字从各建筑飞向顶部资源栏
> - 自动收取：VIP3+ 解锁离线自动收取

**可行性评估**：

| 评估维度 | 结论 |
|---------|------|
| **架构冲突** | ⚠️ 当前建筑产出是**被动累加**模式——ResourceSystem的tick()每帧自动将建筑产出加到玩家资源中，不存在"累积待收取"概念 |
| **PRD理解差异** | PRD的"一键收取"可能指：(A) 收取离线累积的挂机收益，或 (B) 每座建筑有独立的"产出池"需要手动领取 |
| **当前实现** | 产出已自动累加到资源栏，无需手动收取。离线收益通过 OfflineRewardSystem 弹窗一次性领取 |

**判定**：🔴 **需要PRD澄清+独立版本实现**

**理由**：
1. 当前架构是"自动累加"模式，与PRD的"手动收取"模式存在根本冲突
2. 如果要实现手动收取，需要重构资源产出模型（每座建筑增加"产出池"概念）
3. 如果PRD实际意图是"离线收益一键领取"，则当前 OfflineRewardSystem 已覆盖此场景
4. 建议与产品经理确认需求后决定实现方案
