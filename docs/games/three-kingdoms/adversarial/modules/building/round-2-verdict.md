## 四、Arbiter 独立发现（~10%补充）

### AD-201: FIX-404 实现与文档不一致

- **发现**: R1 修复报告描述 FIX-404 为"两阶段执行：预验证→统一执行"，但实际源码为"单阶段执行+资源递减"。
- **影响**: 低。两种设计在正常路径下行为一致，差异仅在 startUpgrade 抛错时的回滚行为。
- **建议**: 更新修复报告以反映实际实现。

### AD-202: getUpgradeCost 中 `!` 非空断言

- **发现**: `BuildingSystem.ts:170` — `const cost = this.getUpgradeCost(type)!`
- **R1 AD-001 已指出**: 此 `!` 断言在 NaN level 场景下可能导致 null dereference。
- **R2 评估**: CH-209 详细分析了触发路径（deserialize NaN level → checkUpgrade 通过 → startUpgrade → null dereference）。实际风险为 P1（需要特定触发条件）。
- **建议**: 改为 `const cost = this.getUpgradeCost(type); if (!cost) throw new Error('...')`。

### AD-203: BuildingRecommender 中 priority NaN 排序

- **发现**: `BuildingRecommender.ts:130` — `priority = 50 + Math.round(prodGain * 10)`
- **分析**: 当 `prodGain` 为 NaN 时，`priority = NaN`，`sort((a,b) => b.priority - a.priority)` 中 NaN 比较导致排序不稳定。
- **实际风险**: 低。`prodGain = nextProd - currentProd`，两个值都来自 `?.production ?? 0`，隐式防护了 NaN。
- **建议**: 防御性编程，在 priority 计算后增加 `if (!Number.isFinite(priority)) priority = 0`。
