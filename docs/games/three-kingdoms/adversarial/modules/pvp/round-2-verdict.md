## 逐项裁决

### P0 项裁决

| 挑战ID | 标题 | Challenger评级 | Arbiter裁决 | 理由 |
|--------|------|---------------|-------------|------|
| P0-R2-01 | addDefenseLog签名不一致 | P0 | **降级 P1** | 两个方法服务于不同场景（stateful vs stateless），是设计差异而非bug。调用方通常只使用其中一个。风险较低。 |
| P0-R2-02 | serialize season参数类型不匹配 | P0 | **确认 P0** | TypeScript的import()类型在运行时无验证，传入不完整对象会导致存档损坏。需添加运行时验证或使用完整类型。 |
| P0-R2-03 | buyItem内部状态修改不一致 | P0 | **确认 P0** | 状态一致性漏洞确实存在。调用方忽略返回state时竞技币不扣但购买计数增加。需改为先返回新state再修改内部状态，或使用事务模式。 |

## 修复方案建议

### Fix-R2-01: serialize 添加运行时类型验证

```typescript
serialize(playerState?, season?, highestRankId?): ArenaSaveData {
  // 验证 season 参数完整性
  const defaultSeason = { seasonId: '', startTime: 0, endTime: 0, currentDay: 1, isSettled: false };
  const safeSeason = season && typeof season === 'object'
    ? { ...defaultSeason, ...season }
    : defaultSeason;
  // ...
}
```

### Fix-R2-02: buyItem 改为原子操作

```typescript
buyItem(playerState, itemId, count) {
  // ...所有验证...
  // 先构造新state，再修改内部状态
  const newState = { ...playerState, arenaCoins: playerState.arenaCoins - totalCost };
  // 最后才修改内部状态（确保无异常路径）
  this.items[itemIdx] = { ...item, purchased: item.purchased + count };
  return { state: newState, item: { ...this.items[itemIdx] } };
}
```

或者更好的方案：**buyItem 不修改内部状态，改为返回新的 items 数组**，由调用方负责更新。
