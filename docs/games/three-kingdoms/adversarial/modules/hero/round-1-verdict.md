## 裁决

**判定: CONTINUE**

**理由:**

1. **评分7.0远低于封版线9.0**，差距2.0分，需要至少2轮迭代才有可能收敛。

2. **系统性缺陷未解决**: Challenger发现的5个NaN绕过`<= 0`检查的问题（A1/A7/A8/A12/CH-009）是系统性缺陷，不是孤立bug。这说明代码中可能还有更多同类问题未被发现。Builder虽然标记了部分uncovered节点，但未识别出"NaN绕过`<= 0`"这个系统性模式。

3. **架构级缺陷待决策**: 三套羁绊系统并存（CH-001）和战力公式羁绊系数永远为1.0（CH-002）是架构级问题，需要设计决策而非简单修复。这类问题不应在测试树中标记为missing，而应升级为架构评审项。

4. **配置交叉验证缺失**: 6名PRD v1.3新增武将的碎片获取路径断裂（CH-NEW-005/006）说明Builder的配置审计仅检查了"配置项存在"，未做"配置项之间的可达性验证"。这是方法论层面的改进方向。

5. **F-Cross覆盖率38%远低于75%门槛**: 跨系统交互是最大盲区。Part B的跨系统覆盖率仅29%，Part C为43%。Builder正确识别了缺失节点但未深入分析。

6. **Phase 1历史参考**: Hero模块在Phase 1经历了R1(7.7)→R2(9.0)→R3(8.2)→R4(9.0)的波动收敛。当前R1评分7.0与Phase 1的R1(7.7)接近，预计R2会有显著提升。

## Builder表现: 6.5/10

### 亮点
1. **节点枚举规模惊人**: 745个节点覆盖了29个源文件、约258个公开API，枚举工作量巨大
2. **源码验证严格**: 每个子系统都标注了文件名、行数范围和验证方式，Part A的Source Verification表完整
3. **P0识别准确率高**: Part B的P0标注经Challenger验证无虚报，Part A的P0遗漏主要是"未充分描述严重性"而非"完全遗漏"
4. **方法论自觉**: 在汇总中提出了6条Rule Evolution Suggestions，说明Builder有自我反思能力
5. **F-Normal覆盖扎实**: 正常路径的测试节点覆盖率高，复现场景描述详细

### 扣分项
1. **系统性模式识别不足** (-1.5): NaN绕过`<= 0`是JavaScript特有的系统性风险，Builder标记了5个uncovered节点但未识别出共同模式。应建立"NaN绕过数值比较"的扫描规则。
2. **配置交叉验证缺失** (-1.0): 6名新增武将无碎片获取路径是最严重的配置遗漏，Builder的配置审计仅检查了"配置项存在性"而非"配置可达性"。
3. **算法正确性审查不足** (-0.5): FormationRecommendSystem的3个算法缺陷（方案重复、羁绊分数硬编码、武将数≤6时方案相同）说明Builder对算法逻辑的审查停留在"分支覆盖"而非"正确性验证"。
4. **covered标注水分** (-0.5): Part A虚报率7.7%，Part C虚报率21%。主要模式是"正常路径covered但null/NaN路径未覆盖"，应拆分为独立节点。
5. **跨系统分析深度不够** (-0.5): F-Cross节点大量标missing但未给出具体风险分析，仅标注"需验证"。

### 改进建议
1. **新增NaN系统性扫描规则**: 对所有`if (x <= 0)` / `if (x > 0)` / `if (x < threshold)`模式，额外枚举`x=NaN`和`x=Infinity`场景
2. **新增配置可达性验证规则**: 对每个配置项（如SHOP_FRAGMENT_EXCHANGE），验证其引用的实体（如武将ID）在关联配置（如STAGE_FRAGMENT_DROPS）中有对应条目
3. **新增算法正确性审查维度**: 对包含排序、分组、去重等算法逻辑的API，额外枚举"算法输出正确性"节点（不仅仅是"不崩溃"）
4. **拆分covered标注**: 将"正常路径covered"和"全路径covered"区分开，对P0节点要求全路径覆盖
5. **深入分析F-Cross missing节点**: 对每个标missing的跨系统节点，至少给出调用链分析和风险等级

## Arbiter独立发现

基于对Builder树和Challenger挑战的交叉审查，Arbiter独立发现以下问题：

### 1. 序列化版本迁移策略缺失 — P1
- **发现**: 所有子系统的`deserialize`都检查`data.version !== SAVE_VERSION`，但未定义版本不匹配时的迁移策略。当前行为是"返回空状态"，这意味着版本升级后所有玩家数据丢失。
- **影响**: 游戏版本更新时，如果SAVE_VERSION递增，所有老玩家存档将被清空。
- **建议**: 定义版本迁移链（v1→v2→v3），每个版本定义迁移函数。

### 2. update(dt) 空实现的一致性问题 — P2
- **发现**: HeroSystem、HeroFormation、HeroRecruitUpManager的`update(dt)`是空实现，但HeroRecruitSystem和TokenEconomySystem的`update(dt)`有实际逻辑（日重置检查、被动产出）。这种不一致可能导致集成时遗漏调用。
- **建议**: 空实现的update应添加注释说明为何为空，或统一由引擎调度。

### 3. GENERAL_DEF_MAP 运行时一致性 — P1
- **发现**: 多个子系统依赖`GENERAL_DEF_MAP`（武将定义映射），但没有任何节点验证"GENERAL_DEF_MAP中的武将ID集合"与"HERO_FACTION_MAP中的武将ID集合"是否一致。如果GENERAL_DEF_MAP新增了武将但HERO_FACTION_MAP未同步更新，阵营筛选会遗漏该武将。
- **建议**: 添加配置一致性校验节点。

### 4. TokenEconomy被动产出精度问题 — P2
- **发现**: `PASSIVE_RATE_PER_SECOND = 0.002`，每帧累加`0.002 * dt`（dt≈0.016），即每帧约0.000032。JavaScript浮点精度可能导致长时间运行后累积误差。建议使用整数计数器（如每500帧产出1令）。
- **影响**: 运行24小时后，理论产出172.8令，实际可能因浮点误差偏差0.1~0.5令。

### 5. 保底计数器序列化时机 — P1
- **发现**: HeroRecruitSystem的保底计数器（pity）在每次`executeSinglePull`中被就地修改（CH-006），但序列化时机由外部控制。如果在两次序列化之间发生崩溃（如浏览器关闭），pity状态丢失，玩家保底进度回退。
- **影响**: 玩家体验——接近保底时崩溃会重置计数器，引发投诉。
- **建议**: 关键状态变更后应触发增量序列化，或使用localStorage做实时持久化。

### 6. 编队推荐系统stageType枚举扩展性 — P2
- **发现**: `FormationRecommendSystem.analyzeStage`使用switch-case处理stageType，default分支按normal计算。如果Campaign系统新增stageType（如'raid'、'pvp'），推荐系统会静默使用不合适的参数。
- **建议**: 使用策略模式或配置映射替代switch-case，提高扩展性。
