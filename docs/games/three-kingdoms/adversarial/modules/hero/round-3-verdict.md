## 四、裁决

**判定: CONTINUE**

**理由:**

1. **评分8.6距封版线9.0差0.4**，进步趋势明确（7.0→8.3→8.6），但未达封版线。基于收敛速度（R1→R2 +1.3, R2→R3 +0.3），R4需再提升0.4，需要P0大幅修复且无新P0发现。

2. **R3仅1个新P0，但影响面极广**：R3-B001（引擎保存流程缺失6个子系统）是R1/R2三轮都遗漏的重大问题。这意味着HeroStarSystem、SkillUpgradeSystem、HeroDispatchSystem、HeroBadgeSystem、AwakeningSystem、TokenEconomy的状态在游戏重启后全部丢失。这不是一个普通的null guard问题，而是**数据持久化架构缺陷**，影响6个子系统的全部状态。修复需要：SaveContext接口扩展、GameSaveData接口扩展、buildSaveData/applySaveData流程补全、2个子系统新增deserialize方法。修复复杂度中高。

3. **FIX穿透质量完美，方法论成熟**：4/4 FIX全部穿透验证通过，0遗漏。这证明R2新增的BR-019（修复穿透检查）和CR-016（FIX穿透验证）规则有效。R1的系统性问题（NaN绕过、null崩溃）在R2被修复，R2的穿透遗漏在R3被完全消除。方法论在三轮迭代中已趋于成熟。

4. **7个R2遗留P0形成修复积压**：setFormation null guard、addToFormation武将验证、getState深拷贝、推荐算法去重、羁绊分数硬编码、SkillUpgrade序列化、配置冲突——这些R2就标记的P0在R3仍未修复。其中配置冲突（5项）三轮未修复，需要策划参与决策。建议R4将遗留P0分为"代码可修"和"需策划决策"两类，前者Builder直接修复。

5. **可玩性从6.3恢复到7.1**：FIX-201修复了羁绊系统失效（+趣味性）、FIX-204修复了碎片溢出经济漏洞（+经济平衡）、NaN/null防护减少了崩溃（+玩家体验）。但6名武将碎片路径仍断裂、推荐系统与羁绊断连、SkillUpgrade升级历史丢失，这三个可玩性阻断项阻止评分进一步提升。

6. **F-Cross覆盖率是最大结构性短板**：~60%距离75%门槛差15%。R3的跨系统验证主要集中在FIX穿透验证（纵向），缺少横向的跨系统交互验证（如"招募→编队→羁绊→战力计算"完整链路）。这是R4需要突破的方向。

### Arbiter 独立发现（~10%补充）

基于对R3 Builder树和R3 Challenger挑战的交叉审查，Arbiter独立发现以下问题：

#### 1. calculateFormationPower 缺少NaN最终防护 — P1（确认Challenger R3-A001）
- **发现**: Challenger标注R3-A001（calculateFormationPower缺少NaN最终防护），Builder树中Part A也有对应节点。Arbiter确认：FIX-203为calculatePower添加了NaN防护（L191），但calculateFormationPower内部虽然调用了calculatePower（已防护），其自身的羁绊倍率和装备战力乘法仍可能产生NaN（如bondMultiplier=NaN且未防护时）。这不是穿透遗漏（FIX-201已修复setter调用），而是calculateFormationPower自身的防护缺失。
- **建议**: 在calculateFormationPower末尾添加与calculatePower相同的NaN最终防护：`if (!Number.isFinite(result) || result < 0) return 0`。
- **优先级**: P1（calculatePower已防护，此为第二层防护）

#### 2. addExp Infinity无限循环风险 — P1（确认Challenger R3-A002）
- **发现**: Challenger标注R3-A002（addExp缺少NaN/Infinity/负数参数校验）。Arbiter补充分析：如果exp=Infinity传入addExp，内部`currentExp + exp = Infinity`，后续while循环`while (currentExp >= expToNextLevel)`中`Infinity >= anyFiniteNumber = true`，导致无限循环。这不是理论风险——任何未校验的上游调用（如活动奖励、成就系统）都可能传入Infinity。
- **建议**: addExp入口添加`if (!Number.isFinite(exp) || exp < 0 || exp > MAX_EXP_PER_CALL) return`。MAX_EXP_PER_CALL建议设为1e9。
- **优先级**: P1（需要上游传入Infinity才触发，但触发后果严重）

#### 3. 引擎保存流程缺失的版本迁移影响 — P2
- **发现**: R3-B001发现6个子系统不在保存流程中。Arbiter补充：如果R4修复了保存流程（添加了6个子系统的序列化），则旧存档（不包含这些字段）加载时需要迁移逻辑。applySaveData中需要`if (!saveData.heroStarSystem) saveData.heroStarSystem = HeroStarSystem.defaultState()`。这不是P0（旧存档本身就没有这些数据，缺失是预期行为），但如果不处理会导致加载失败。
- **建议**: applySaveData中为每个新增子系统添加默认值回退。
- **优先级**: P2（R4修复R3-B001时一并处理）

#### 4. BondSystem三套并存的具体风险量化 — 补充分析
- **发现**: Builder标注"三套羁绊系统并存未统一"（架构级P0），但未量化具体风险。Arbiter分析：当前HeroFormation使用BondSystem（搭档羁绊），FactionBondSystem（阵营羁绊）由谁调用未定义。如果编队系统只调用BondSystem不调用FactionBondSystem，则阵营羁绊完全失效。如果两者都调用但结果未合并，则羁绊加成重复计算。
- **建议**: R4应明确编队系统的羁绊调用链：先BondSystem（搭档）→ 再FactionBondSystem（阵营）→ 合并结果。这是设计决策，需要策划参与。
- **优先级**: P0（影响羁绊计算正确性）

#### 5. TokenEconomy序列化与引擎保存流程的关系 — 补充分析
- **发现**: R3-B001包含TokenEconomy不在保存流程中。Arbiter补充：TokenEconomy管理招募令牌的自动恢复（tick），如果重启后状态丢失，玩家令牌数重置为初始值。但更严重的是：pityCounter（保底计数器）也在HeroRecruitSystem中，如果HeroRecruitSystem在保存流程中但TokenEconomy不在，则保底计数器保留但令牌数重置——玩家可能利用这一点"重置令牌但保留保底"进行刷保底。
- **建议**: TokenEconomy和HeroRecruitSystem的保存/加载必须原子性——要么同时保存要么同时不保存。
- **优先级**: P1（经济漏洞风险）

#### 6. setLevelAndExp边界条件 — P1（确认Challenger R3-A003）
- **发现**: Challenger标注R3-A003（setLevelAndExp缺少level/exp范围校验）。Arbiter补充：如果level=0传入，内部`expToNextLevel(0)`可能返回0或负数，导致后续addExp除零。如果level超过当前最大等级（如100），则经验曲线可能溢出。

## 七、收敛判断

### 关键问题回答

**Q: R3仅1个新P0，是否接近封版？**

**A: 是的，非常接近，但有一个"重量级"新P0。**

R3仅1个新P0（R3-B001）是强收敛信号。对比R1的41个和R2的18个，新P0的下降趋势是指数级的：

```
新P0趋势: 41 → 18 → 1
下降率:    -56%   -94%
预测R4:    0~1个（如果R3-B001修复后无新维度发现）
```

但R3-B001不是普通P0——它是**数据持久化架构缺陷**，影响6个子系统的全部状态。修复复杂度中高，且修复后需要完整的集成测试（保存→重启→加载→状态一致性验证）。这个P0的存在使得R3不能封版。

**Q: R4能否达到9.0封版线？**

**A: 有55%概率可以达到，但需要满足3个条件。**

#### 条件1: 修复R3-B001 + 7个R2遗留P0中的"代码可修"部分（概率70%）

| 修复项 | 复杂度 | 预计耗时 |
|--------|--------|---------|
| R3-B001 引擎保存流程补全 | 中高 | 核心修复 |
| R3-B002 setFormation null guard | 低（1行） | 1分钟 |
| R3-B003 addToFormation武将验证 | 低（2行） | 1分钟 |
| R3-B004 getState深拷贝 | 低（1行） | 1分钟 |
| R3-B007 SkillUpgrade序列化 | 中 | 需新增方法 |
| R3-B005/B006 推荐算法修复 | 中 | 算法修改 |
| R3-B008 配置冲突（5项） | 低~中 | 需策划决策 |

"代码可修"部分（R3-B002/B003/B004/B007）修复概率高。配置冲突（R3-B008）需要策划参与，可能遗留到R5。

#### 条件2: Challenger R4新P0≤2（概率65%）

R3仅1个新P0，说明Challenger的发现空间已大幅压缩。但R4修复R3-B001后，Challenger会验证保存/加载流程的完整性，可能发现序列化/反序列化的边界问题。预计R4新P0为0~2个。

#### 条件3: F-Cross覆盖率提升至75%（概率40%）

这是最大的不确定性。当前~60%，需要+15%。R4需要新增跨系统交互验证节点（如"招募→编队→羁绊→战力计算"完整链路），这需要Builder投入更多精力在横向分析上。
