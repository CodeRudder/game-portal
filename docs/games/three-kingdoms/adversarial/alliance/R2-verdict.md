## 二、P0缺陷终审

### P0-01: createAllianceSimple扣费回滚 → 降级为P2 ✅

**Challenger R2重新分析**: 
- `createAlliance()`返回新对象, **不修改**`this._alliance/this._playerState`
- 赋值`this._alliance = result.alliance`在扣费成功之后
- 扣费失败时, 内部状态未被修改

**Arbiter验证**: ✅ 确认降级合理。代码逻辑是:
```
1. createAlliance → 返回新对象(result), 不改this
2. spendCallback → 失败则return, 不走到步骤3
3. this._alliance = result.alliance → 只有成功才执行
```
**不是数据一致性Bug**, 但代码可读性差(容易误判), 建议重构为先扣费后创建。

**最终等级**: **P2** (代码质量问题, 非Bug)

### P0-02: Boss伤害不持久化 → 维持P0 ⚠️

**Arbiter分析**: 
- `AllianceData`接口确实无boss/damageRecords字段
- `getCurrentBoss()`每次重建, damageRecords={}
- 存档加载后Boss满血

**影响评估**: 
- Boss每日刷新, 伤害数据生命周期仅当天
- 如果每日重置正确触发, 实际影响为: **当天内如果存档/读档, Boss进度丢失**
- 对于单机/弱联网游戏, 这是严重体验问题
- 对于强服务器游戏, Boss数据可由服务端维护

**最终等级**: **P0** 但附条件:
- **如果是服务端存储Boss数据**: 降级为P1(服务端可独立管理)
- **如果是客户端本地存储**: 维持P0(必须修复)

**封版条件**: 需明确Boss数据存储架构, 至少提供设计方案。

### P0-03: challengeBoss NaN防护 → 维持P0 ✅

**Arbiter验证**: 
```javascript
Math.max(0, Math.min(NaN, 100000))
= Math.max(0, NaN)  // Math.min(NaN, x) = NaN
= NaN               // Math.max(0, NaN) = NaN (不是0!)
```

**修复方案**: 1行代码
```typescript
const actualDamage = Number.isFinite(damage) ? Math.max(0, Math.min(damage, boss.currentHp)) : 0;
```

**最终等级**: **P0** — 必须修复
