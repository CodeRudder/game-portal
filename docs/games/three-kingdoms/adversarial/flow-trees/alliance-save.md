# Part 1: 联盟社交流程 (ALLIANCE-001 ~ ALLIANCE-020)

---

## 阶段一：联盟创建与加入 (ALLIANCE-001 ~ ALLIANCE-005)

---

### ALLIANCE-001: 创建联盟
- 前置: 无
- 操作: 玩家等级 ≥ 10 且未加入任何联盟，消耗元宝 × 500，输入联盟名称（2~8字符，不含敏感词）和联盟宣言，点击"创建联盟"
- 预期: 联盟创建成功，玩家自动成为盟主（`role=leader`）；联盟初始等级=1，成员上限=20，联盟资金=0；联盟列表中可搜索到新联盟；扣除元宝 × 500
- 优先级: P0
- 关联系统: 联盟核心
- 状态: covered

### ALLIANCE-002: 搜索与申请加入联盟
- 前置: 无
- 操作: 玩家等级 ≥ 10 且未加入联盟，在联盟列表中搜索目标联盟名称，点击"申请加入"
- 预期: 申请提交成功，目标联盟的"申请列表"中新增一条待审批记录；若联盟设置为"自由加入"则直接加入成功，玩家角色变为成员（`role=member`）；若联盟已满员则提示"联盟人数已满"
- 优先级: P0
- 关联系统: 联盟核心
- 状态: covered

### ALLIANCE-003: 审批加入申请
- 前置: ALLIANCE-001
- 操作: 盟主/副盟主打开联盟管理面板 → 申请列表，对一条待审批申请执行"通过"操作
- 预期: 申请人即时加入联盟，联盟成员数 +1；申请人收到系统邮件通知"已通过XX联盟的审批"；若执行"拒绝"操作，申请人收到"申请被拒绝"通知
- 优先级: P1
- 关联系统: 联盟核心 / 邮件系统
- 状态: covered

### ALLIANCE-004: 联盟等级与成员上限
- 前置: ALLIANCE-001
- 操作: 联盟资金达到升级阈值（如 Lv1→Lv2 需 10000 联盟资金），盟主点击"升级联盟"
- 预期: 联盟等级 +1（Lv2），成员上限提升至 25 人；联盟等级影响：Boss 解锁等级、商店可兑换物品、战争匹配权重；升级消耗的联盟资金从联盟账户扣除
- 优先级: P1
- 关联系统: 联盟核心
- 状态: covered

### ALLIANCE-005: 联盟成员权限管理
- 前置: ALLIANCE-001
- 操作: 盟主将一名成员提升为"副盟主"（`role=vice_leader`），再将其降级为普通成员
- 预期: 副盟主拥有审批申请、发起Boss、编辑宣言权限；降级后权限回收，仅保留基础成员权限；每次操作后联盟频道广播角色变更消息
- 优先级: P1
- 关联系统: 联盟核心 / 聊天系统
- 状态: partial

## 阶段二：联盟日常互动 (ALLIANCE-006 ~ ALLIANCE-010)

---

### ALLIANCE-006: 联盟每日签到
- 前置: ALLIANCE-002（已加入联盟）
- 操作: 每日首次进入联盟界面，点击"签到"按钮
- 预期: 签到成功，个人贡献点 +20，联盟活跃度 +10；连续签到7天额外奖励联盟宝箱 ×1；当日重复签到提示"今日已签到"；联盟签到面板显示已签到人数和总人数
- 优先级: P1
- 关联系统: 联盟社交
- 状态: covered

### ALLIANCE-007: 联盟捐赠（资源→贡献点）
- 前置: ALLIANCE-002
- 操作: 成员选择捐赠类型（铜钱捐赠 / 元宝捐赠 / 粮草捐赠），执行铜钱捐赠：消耗铜钱 × 10000
- 预期: 个人贡献点 +100，联盟资金 +500（捐赠转换率 5:1）；每日捐赠次数上限 3 次，超出提示"今日捐赠次数已用尽"；资源不足时提示"铜钱不足"
- 优先级: P0
- 关联系统: 联盟资源 / 资源系统
- 状态: covered

### ALLIANCE-008: 联盟捐赠（元宝档位）
- 前置: ALLIANCE-002
- 操作: 成员执行元宝捐赠：消耗元宝 × 200
- 预期: 个人贡献点 +300，联盟资金 +1000；元宝捐赠每日限 1 次；元宝与铜钱捐赠次数独立计算
- 优先级: P1
- 关联系统: 联盟资源 / 资源系统
- 状态: covered

### ALLIANCE-009: 联盟聊天与公告
- 前置: ALLIANCE-002
- 操作: 成员在联盟频道发送文字消息；盟主/副盟主编辑联盟公告（最多200字）
- 预期: 聊天消息实时同步给所有在线成员，离线成员上线后可查看最近 50 条历史消息；公告置顶显示在联盟信息页顶部；消息包含敏感词时被过滤为 `***`
- 优先级: P2
- 关联系统: 联盟社交 / 聊天系统
- 状态: covered

### ALLIANCE-010: 联盟成员列表与排序
- 前置: ALLIANCE-002
- 操作: 打开联盟成员列表，分别按"贡献排行""战力排行""在线状态"排序
- 预期: 贡献排行按累计贡献点降序；战力排行按玩家总战力降序；在线状态排序：在线 > 今日离线 > 3天未登录；每个成员卡片显示：头像、名称、角色、等级、战力、贡献点
- 优先级: P2
- 关联系统: 联盟社交
- 状态: partial

## 阶段三：联盟Boss与商店 (ALLIANCE-011 ~ ALLIANCE-015)

---

### ALLIANCE-011: 联盟Boss挑战发起
- 前置: ALLIANCE-004（联盟等级 ≥ 2）
- 操作: 盟主/副盟主选择Boss难度（普通/精英/噩梦），消耗联盟资金 × 2000 发起Boss挑战
- 预期: Boss出现在联盟领地场景，所有成员可见；联盟频道广播"XX发起了Boss挑战"；Boss存在时间 24 小时，超时未击杀则消失且不返还资金；联盟等级决定可挑战的Boss难度上限
- 优先级: P0
- 关联系统: 联盟战斗
- 状态: covered

### ALLIANCE-012: 联盟Boss战斗与伤害累计
- 前置: ALLIANCE-011
- 操作: 成员编队进入Boss战斗，执行一轮攻击（限时 90 秒），记录对Boss造成的伤害
- 预期: 每位成员每日可挑战 Boss 2 次；伤害累计到Boss总血量，Boss血量归零时判定击杀；战斗结束后显示：本次伤害、累计伤害、伤害排行；战斗中武将阵亡不影响正常阵容
- 优先级: P0
- 关联系统: 联盟战斗 / 战斗系统
- 状态: covered

### ALLIANCE-013: 联盟Boss奖励分配
- 前置: ALLIANCE-012（Boss被击杀）
- 操作: Boss被击杀后，系统按伤害贡献比例分配奖励
- 预期: 伤害第1名获得：高级装备箱 ×2 + 武将碎片 ×20 + 铜钱 ×50000；参与奖（至少攻击1次）：铜钱 ×10000 + 联盟宝箱 ×1；最后一击（补刀）额外奖励：元宝 ×100；奖励通过邮件发放，有效期 7 天
- 优先级: P0
- 关联系统: 联盟战斗 / 奖励系统 / 邮件系统
- 状态: covered

### ALLIANCE-014: 联盟商店兑换（贡献点→物品）
- 前置: ALLIANCE-007（已获得贡献点）
- 操作: 成员打开联盟商店，选择物品（如：武将碎片×5，标价贡献点 200），点击"兑换"
- 预期: 扣除贡献点 200，获得武将碎片 ×5；商店物品按联盟等级解锁（Lv2 解锁中级物品，Lv4 解锁高级物品）；每周限购物品计数正确，达到上限后显示"本周已售罄"
- 优先级: P0
- 关联系统: 联盟商店 / 联盟资源
- 状态: covered

### ALLIANCE-015: 联盟商店刷新机制
- 前置: ALLIANCE-014
- 操作: 每周一 00:00 系统自动刷新商店限购次数；玩家消耗元宝 × 50 手动刷新商品列表
- 预期: 自动刷新后所有限购物品计数归零，商品列表随机更新；手动刷新后商品列表重新随机生成，限购计数不重置；每日手动刷新上限 3 次
- 优先级: P2
- 关联系统: 联盟商店
- 状态: missing

## 阶段四：联盟战争与退出解散 (ALLIANCE-016 ~ ALLIANCE-020)

---

### ALLIANCE-016: 联盟战争匹配
- 前置: ALLIANCE-004（联盟等级 ≥ 3）
- 操作: 盟主在联盟战争界面点击"报名参战"，系统在每周三 20:00 进行匹配
- 预期: 匹配算法按联盟总战力 ±15% 范围内匹配对手；匹配成功后双方联盟进入备战状态（24小时准备期）；匹配失败（无合适对手）退还报名费用；备战期间新成员不可加入
- 优先级: P0
- 关联系统: 联盟战争 / 匹配系统
- 状态: covered

### ALLIANCE-017: 联盟战争攻防战斗
- 前置: ALLIANCE-016
- 操作: 战争开始后，成员选择敌方目标（建筑/据点），派遣队伍发起进攻
- 预期: 每位成员每日可发起 3 次进攻、设置 1 次防守；进攻胜利占领据点，失败则防守方获得积分；战争持续 2 小时，时间结束后按占领据点数判定胜负；胜方联盟获得：联盟资金 ×5000 + 全体成员元宝 ×200
- 优先级: P0
- 关联系统: 联盟战争 / 战斗系统
- 状态: partial

### ALLIANCE-018: 联盟战争奖励结算
- 前置: ALLIANCE-017
- 操作: 战争结束后系统自动结算，按胜负和个人贡献发放奖励
- 预期: 胜方盟主额外获得"霸主宝箱"×1；个人贡献按攻防积分排名，前3名获得额外元宝奖励；败方获得安慰奖（联盟资金 ×1000）；奖励通过邮件发放，30天内未领取则过期
- 优先级: P1
- 关联系统: 联盟战争 / 奖励系统 / 邮件系统
- 状态: covered

### ALLIANCE-019: 主动退出联盟
- 前置: ALLIANCE-002
- 操作: 普通成员在联盟界面点击"退出联盟"，确认退出操作
- 预期: 成员立即退出联盟，联盟成员数 -1；退出后贡献点清零，已兑换的物品不回收；退出后 24 小时内不可加入新联盟（冷却期）；盟主不可直接退出，需先转让盟主身份
- 优先级: P1
- 关联系统: 联盟核心
- 状态: covered

### ALLIANCE-020: 联盟解散
- 前置: ALLIANCE-001
- 操作: 盟主在联盟管理界面点击"解散联盟"，需二次确认（输入联盟名称确认）
- 预期: 联盟解散后所有成员被移除，收到系统邮件"XX联盟已解散"；联盟资金、等级、商店数据全部清除；解散后联盟名称释放，24小时后可被重新使用；解散操作不可撤销
- 优先级: P1
- 关联系统: 联盟核心 / 邮件系统
- 状态: covered

# Part 2: 存档恢复流程 (SAVE-001 ~ SAVE-020)

## 阶段五：存档创建与保存 (SAVE-001 ~ SAVE-006)

---

### SAVE-001: 自动保存触发
- 前置: 无
- 操作: 玩家进行任意操作（战斗结束、建筑升级、资源采集等），系统检测到距上次保存 ≥ `AUTO_SAVE_INTERVAL = 60` 秒
- 预期: 后台自动触发存档写入，不阻塞玩家操作；自动保存包含完整游戏状态快照（资源、武将、建筑、科技、联盟、任务进度）；保存成功后更新 `lastSaveTime` 时间戳；保存失败时本地缓存待重试队列
- 优先级: P0
- 关联系统: 存档核心
- 状态: covered

### SAVE-002: 手动保存
- 前置: 无
- 操作: 玩家在设置界面点击"手动保存"按钮
- 预期: 立即触发完整存档写入，显示保存进度条；保存成功提示"存档成功"；若距上次保存 < 10秒，提示"保存过于频繁，请稍后再试"；手动保存不受自动保存间隔限制
- 优先级: P1
- 关联系统: 存档核心
- 状态: covered

### SAVE-003: 保存数据完整性校验
- 前置: SAVE-001
- 操作: 系统在每次保存完成后，对存档数据执行 CRC32 校验和验证
- 预期: 生成校验码 `checksum = CRC32(serialized_data)`，与存档数据一同写入；校验失败时自动重试保存（最多3次）；3次均失败则标记存档为 `corrupted = true`，触发告警并尝试回滚到上一个有效存档
- 优先级: P0
- 关联系统: 存档核心
- 状态: covered

### SAVE-004: 武将子系统序列化
- 前置: SAVE-001
- 操作: 系统序列化武将子系统数据：武将列表（ID、等级、星级、技能等级、装备槽位）、碎片背包、编队配置
- 预期: 序列化数据结构：`{ heroes: [{id, level, star, skills: [], equipSlots: {}}], fragments: {heroId: count}, formations: [slotConfig] }`；反序列化后武将属性值与保存前一致（误差 ≤ 0.01）；空编队位序列化为 `null` 而非省略
- 优先级: P0
- 关联系统: 序列化引擎 / 武将系统
- 状态: covered

### SAVE-005: 战斗与战役子系统序列化
- 前置: SAVE-001
- 操作: 系统序列化战斗状态和战役进度：当前关卡进度、章节解锁状态、战斗回放数据、每日挑战次数
- 预期: 序列化数据结构：`{ campaign: {chapterId, stageId, stars}, dailyChallenges: {id: remainingCount}, battleReplay: [] }`；战役三星评价数据正确保存；每日挑战次数在跨日后正确重置
- 优先级: P1
- 关联系统: 序列化引擎 / 战斗系统 / 战役系统
- 状态: covered

### SAVE-006: 建筑/科技/联盟子系统序列化
- 前置: SAVE-001
- 操作: 系统序列化建筑队列、科技研究队列、联盟相关数据
- 预期: 建筑数据：`{ buildings: [{id, level, upgradeEndTime}], queues: {build: [], research: []} }`；科技数据：`{ techs: [{id, level, researching}], techPoints: number }`；联盟数据：`{ allianceId, role, contribution, joinTime }`；升级中的建筑/科技保存剩余时间和结束时间戳
- 优先级: P0
- 关联系统: 序列化引擎 / 建筑系统 / 科技系统 / 联盟系统
- 状态: covered

## 阶段六：存档加载与版本兼容 (SAVE-007 ~ SAVE-012)

---

### SAVE-007: 正常加载存档
- 前置: SAVE-001
- 操作: 玩家启动游戏，系统检测到本地存在有效存档（`corrupted = false`），执行加载
- 预期: 加载存档数据并反序列化，恢复所有子系统状态；资源数值与上次保存一致；武将列表、编队配置正确恢复；建筑升级队列中的剩余时间 = `upgradeEndTime - now`，若已到期则自动完成升级
- 优先级: P0
- 关联系统: 存档核心
- 状态: covered

### SAVE-008: 存档版本兼容（向前兼容）
- 前置: SAVE-007
- 操作: 使用 v1.2 版本客户端加载 v1.0 版本生成的旧存档（缺少新增字段）
- 预期: 系统执行 `migrateSaveData(oldVersion, newVersion)` 迁移函数；新增字段填充默认值（如 `newFeature.unlocked = false`）；旧字段正确映射到新结构；迁移完成后存档版本号更新为 v1.2；迁移日志记录变更内容
- 优先级: P0
- 关联系统: 存档核心 / 版本迁移
- 状态: covered

### SAVE-009: 存档版本降级处理
- 前置: SAVE-007
- 操作: 使用 v1.0 版本客户端加载 v1.2 版本生成的存档（包含新版本字段）
- 预期: 系统忽略无法识别的新字段，不抛出异常；核心字段（资源、武将、建筑）正确加载；提示用户"检测到新版本存档，部分功能需要更新客户端才能使用"
- 优先级: P1
- 关联系统: 存档核心 / 版本迁移
- 状态: partial

### SAVE-010: 损坏存档检测与恢复
- 前置: SAVE-003
- 操作: 系统加载存档时 CRC32 校验失败（`stored_checksum ≠ computed_checksum`）
- 预期: 标记当前存档为损坏，尝试加载上一个有效备份存档（`backup_slot = previous`）；若无有效备份，提示玩家"存档损坏，将使用云端存档恢复"；若云端存档也损坏，创建全新初始存档并补偿元宝 × 500
- 优先级: P0
- 关联系统: 存档核心 / 备份系统
- 状态: covered

### SAVE-011: 多存档槽位管理
- 前置: SAVE-002
- 操作: 玩家在存档管理界面查看存档列表（最多3个槽位），选择槽位2执行"保存到该槽位"
- 预期: 槽位2保存当前游戏状态，不影响槽位1数据；每个槽位显示：保存时间、玩家等级、主线进度；切换槽位加载后，游戏状态完全切换到对应存档
- 优先级: P2
- 关联系统: 存档核心
- 状态: missing

### SAVE-012: 存档数据压缩与加密
- 前置: SAVE-001
- 操作: 系统对序列化后的存档数据执行压缩（LZ4）和加密（AES-256）
- 预期: 压缩后存档体积 ≤ 原始数据的 40%；加密后数据不可通过文本编辑器直接读取/篡改；解密解压后数据与原始数据完全一致（byte-level 对比）；加解密耗时 ≤ 500ms（1MB存档）
- 优先级: P1
- 关联系统: 存档核心 / 安全系统
- 状态: covered

## 阶段七：跨设备同步与状态验证 (SAVE-013 ~ SAVE-020)

---

### SAVE-013: 云端存档上传
- 前置: SAVE-001
- 操作: 玩家在WiFi环境下触发云端同步，系统将本地最新存档上传至服务器
- 预期: 上传成功后服务器返回 `syncToken`，本地记录 `cloudSyncTime`；上传数据包含：存档数据 + CRC32校验码 + 版本号 + 设备标识；移动网络环境下默认不自动上传，需玩家手动触发
- 优先级: P0
- 关联系统: 跨设备同步 / 网络模块
- 状态: covered

### SAVE-014: 跨设备存档下载与冲突解决
- 前置: SAVE-013
- 操作: 玩家在B设备登录同一账号，系统检测到云端存档比本地更新（`cloudSyncTime > localSyncTime`）
- 预期: 弹出提示"检测到云端有更新的存档（XX时间），是否覆盖本地数据？"；玩家选择"覆盖"：下载云端存档并替换本地数据；玩家选择"保留本地"：上传本地存档到云端覆盖；冲突解决后两端数据一致
- 优先级: P0
- 关联系统: 跨设备同步 / 冲突解决
- 状态: covered

### SAVE-015: 恢复后资源一致性验证
- 前置: SAVE-007
- 操作: 存档加载完成后，系统自动校验资源子系统数据完整性
- 预期: 验证项：铜钱、元宝、粮草、兵力数值 ≥ 0 且 ≤ 存储上限；资源产出速率与建筑等级匹配；离线期间资源增长量 = `Σ(产出速率 × 离线时长 × 衰减系数)`，误差 ≤ 1；验证失败时回滚到备份存档
- 优先级: P0
- 关联系统: 恢复验证 / 资源系统
- 状态: covered

### SAVE-016: 恢复后武将状态一致性验证
- 前置: SAVE-007
- 操作: 存档加载后校验武将子系统：武将数量、属性、编队配置
- 预期: 武将总数 = `heroList.length`，无重复ID；每个武将的 `level ≤ maxLevel(star)`，技能等级 ≤ 武将等级；编队中的武将ID均存在于武将列表中；装备槽位引用的装备存在于背包中；验证失败的具体武将标记为异常并记录日志
- 优先级: P0
- 关联系统: 恢复验证 / 武将系统
- 状态: covered

### SAVE-017: 恢复后建筑与科技状态验证
- 前置: SAVE-007
- 操作: 存档加载后校验建筑和科技子系统
- 预期: 建筑等级 ≤ 建筑上限等级（受主城等级约束）；升级队列中的建筑 `upgradeEndTime` 有效（`> saveTime`），已过期的自动完成；科技研究点数 ≥ 0，研究中的科技进度有效；建筑功能解锁状态与建筑等级一致（如：兵营 Lv3 解锁盾兵）
- 优先级: P1
- 关联系统: 恢复验证 / 建筑系统 / 科技系统
- 状态: covered

### SAVE-018: 恢复后联盟状态一致性验证
- 前置: SAVE-007
- 操作: 存档加载后校验联盟相关数据与服务器状态一致性
- 预期: 本地 `allianceId` 与服务器查询结果一致；若服务器显示已不在联盟（被踢出/联盟解散），本地清除联盟数据并提示"您已不在联盟中"；贡献点、联盟等级等数据以服务器为准进行同步；联盟签到状态与服务器时间校准
- 优先级: P1
- 关联系统: 恢复验证 / 联盟系统
- 状态: covered

### SAVE-019: 断线重连存档保护
- 前置: SAVE-001
- 操作: 玩家在战斗过程中网络断开，30秒后网络恢复
- 预期: 断线时本地立即保存战斗快照（战斗状态 + 剩余时间）；重连后服务器校验快照合法性（战斗ID、时间戳、伤害数据）；校验通过则从断点继续战斗；校验失败（检测到篡改）则判定战斗失败，不发放奖励
- 优先级: P1
- 关联系统: 存档核心 / 网络模块 / 反作弊
- 状态: partial

### SAVE-020: 存档回滚与异常恢复全流程
- 前置: SAVE-010
- 操作: 模拟存档严重损坏场景：本地存档损坏 + 云端存档版本不兼容 + 备份存档过期（7天前）
- 预期: 系统按优先级尝试恢复：最新云端存档 → 云端备份存档 → 本地备份存档 → 新建初始存档；每步尝试均有日志记录；最终恢复后执行全量一致性验证（SAVE-015~018）；向玩家发送详细恢复报告邮件，说明恢复来源和数据状态
- 优先级: P2
- 关联系统: 存档核心 / 恢复验证 / 邮件系统
- 状态: missing
