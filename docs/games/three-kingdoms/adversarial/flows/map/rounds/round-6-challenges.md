## P0 质疑

### CH-MAP-R6-01 遗迹探索Lv10行概率数据表与公式矛盾 + R6"优化声明"不准确 [P0]

**质疑流程**: MAP-F09-02-R6 二、遗迹探索三档判定修正
**追问**: "这个数值对吗？" — R6的优化后科技加成效果表的数据是否与公式一致？

**问题**:

R6提供了"优化后科技加成效果"表(Lv10行)，声称解决了R5的问题。但经逐项数学验证，发现**两个错误**:

**错误1: Lv10行部分成功/成功概率互换**

R6表(Lv10行)声称:

| 攻城术等级 | 失败概率 | 部分成功概率 | 成功概率 |
|-----------|---------|------------|---------|
| Lv10 (+20%) | 10% | **60%** | **30%** |

按R6公式`partial_threshold = max(fail+5%, 50% - tech*0.5)`计算:
- fail_threshold = max(5%, 30%-20%) = **10%**
- partial_threshold = max(15%, 50%-10%) = **40%**
- 失败概率 = roll < 10% = **10%** (正确)
- 部分成功概率 = 10% <= roll < 40% = **30%** (表中写60%，错误!)
- 成功概率 = roll >= 40% = **60%** (表中写30%，错误!)

**Lv10行的部分成功概率(60%)和成功概率(30%)被互换了。** 正确值应为部分成功30%、成功60%。

**错误2: R6声称"优化前后在Lv10结果相同"的说法不准确**

R6文档第162行说: "为进一步激励科技升级，将成功阈值调整公式优化如下"，暗示优化前公式有问题。但R6随后的对比表(Lv10行)显示优化前后结果"相同"(fail=10%, partial=30%, success=60%)。

实际验证:
- **优化前**(R5原始公式)`partial_threshold = min(95%, 50% + tech)`:
  - Lv10: partial_threshold = min(95%, 70%) = **70%** → success = **30%** (success随tech增加而**下降**)
- **优化后**(R6公式)`partial_threshold = max(fail+5%, 50% - tech*0.5)`:
  - Lv10: partial_threshold = max(15%, 40%) = **40%** → success = **60%** (success随tech增加而**上升**)

两个公式在Lv10的结果**完全不同**(30% vs 60%)。R6的优化公式方向是正确的(使success随tech增加)，但文档中"优化前后相同"的对比表是错误的——实际上是优化前30%、优化后60%。

**影响**:

1. **开发实现风险**: 开发者如果按照R6表(Lv10: partial=60%, success=30%)实现，将得到与公式完全相反的结果。这是**数据表与公式矛盾**的硬伤。
2. **设计验证误导**: R6声称"优化前公式在Lv10产生success=30%"是正确的(原始公式确实如此)，但声称"优化后也产生success=30%"是错误的(优化后应为60%)。这导致评审者误以为优化没有效果。
3. **R5质疑落实存疑**: CH-MAP-R5-05的核心诉求是"三档判定逻辑混乱"。R6的优化公式方向正确，但数据表错误使得"已修复"的判断不可信——需要确认开发者究竟按公式还是按表实现。

**建议**:

1. 修正Lv10行数据: fail=10%, partial=**30%**, success=**60%**
2. 在对比表中明确区分: 优化前(success=30%, 方向错误) vs 优化后(success=60%, 方向正确)
3. 添加开发者注释: "以公式为准，表为示意"

### CH-MAP-R6-02 山贼战斗胜利损耗公式存在溢出BUG——胜利可损失超过出征兵力 [P0]

**质疑流程**: MAP-F09-02-R6 一、山贼战斗公式修正，胜利损耗公式
**追问**: "如果失败呢？" — 如果玩家以少量兵力侥幸胜利，损耗会如何？

**问题**:

R6定义的胜利损耗公式:
```
预计兵力损失(胜利时) = 出征兵力 × max(5%, 20% × (山贼战力 / 玩家总兵力))
```

该公式在"低兵力 vs 高战力山贼"场景下存在**溢出BUG**:

| 场景 | 山贼战力 | 出征兵力 | 损耗比率 | 损耗值 | 问题 |
|------|---------|---------|---------|-------|------|
| Lv70带1000兵打山贼王 | 6300 | 1000 | max(5%, 20%x6.3)=**126%** | **1260** | **损耗超过出征兵力!** |
| Lv70带500兵打山贼王 | 6300 | 500 | max(5%, 20%x12.6)=**252%** | **1260** | **损耗是出征兵力的2.5倍!** |
| Lv50带1000兵打精锐山贼 | 3500 | 1000 | max(5%, 20%x3.5)=**70%** | 700 | 虽未溢出但损耗率极高 |

数学验证(Lv70带1000兵):
```
损耗比率 = max(5%, 20% × (6300/1000))
         = max(5%, 20% × 6.3)
         = max(5%, 126%)
         = 126%

损耗值 = 1000 × 126% = 1260

但玩家只带了1000兵! 胜利后损耗1260兵意味着:
- 要么玩家兵力变为负数(不可能)
- 要么损耗被截断为1000(全军覆没)
- 无论哪种处理，"胜利=全军覆没"对玩家体验是灾难性的
```

**核心问题**: `山贼战力 / 玩家总兵力`这个比值在兵力悬殊时可以远超5.0(即损耗比率超过100%)，但公式没有上限截断。

**影响**:

1. **极端场景下"胜利"比"失败"更惨**: 失败损耗固定为出征兵力x20%(即200兵)，但胜利损耗可达126%+。这意味着在某些场景下，玩家"故意输"比"侥幸赢"损失更少——这是严重的数值设计缺陷。
2. **战斗预估面板误导**: 预估面板显示"预计胜利损耗"时，如果显示1260(超过出征兵力)，玩家会困惑; 如果截断为1000(全军覆没)，玩家也不会选择战斗。
3. **R6极端值验证遗漏**: R6的极端值验证表(第86-90行)测试了Lv70带10000兵(损耗12.6%，正常)和Lv70带1000兵(胜率7.9%)，但**没有验证胜利损耗在低兵力场景下的溢出**。

**建议**:

修正公式，增加上限截断:
```
预计兵力损失(胜利时) = 出征兵力 × min(100%, max(5%, 20% × (山贼战力 / 玩家总兵力)))
```

或更精确地限制为"胜利损耗不超过失败损耗":
```
预计兵力损失(胜利时) = min(出征兵力, 出征兵力 × max(5%, 20% × (山贼战力 / 玩家总兵力)))
```

验证修正后效果:
- Lv70带1000兵: min(1000, 1000×126%) = min(1000, 1260) = **1000** (全军覆没，至少合理)
- Lv70带5000兵: min(5000, 5000×25.2%) = min(5000, 1260) = **1260** (25.2%，正常)

### CH-MAP-R6-03 内应暴露冷却状态流程未覆盖"暴露+无信件"同时存在的UI展示优先级 [P1]

**质疑流程**: MAP-F06-07-R6 三态卡片判定(状态B/状态C)
**追问**: "然后呢？" — 当两种不可用原因同时存在时，UI展示哪个？

**问题**:

R6定义了内应卡片的三态:
```
状态A: insiderLetter >= 1 && !insiderExposed → 可点击
状态B: insiderExposed = true → 置灰+冷却倒计时
状态C: insiderLetter = 0 && !insiderExposed → 置灰+"需要内应信x1"
```

R6在状态C的描述中提到: "若同时 city.insiderExposed = true: 优先显示状态B(暴露冷却优先于信件不足)"。但存在以下问题:

1. **状态B的条件定义不完整**: 状态B仅要求`insiderExposed = true`，未检查`insiderLetter`数量。当`insiderExposed = true && insiderLetter = 0`时:
   - 按状态B定义: 显示"内应已暴露"+冷却倒计时
   - 按状态C定义: 显示"需要内应信x1"
   - R6说"优先显示B"，但这个优先级规则只在状态C的文字描述中出现，未在状态B的条件定义中体现

2. **冷却结束后状态切换的瞬时问题**: 当冷却刚好结束(`remaining = 0`)时，`insiderExposed`被自动设为`false`。此时如果`insiderLetter = 0`:
   - 冷却刚结束 → 自动跳到状态C(需要内应信)
   - 但玩家可能正在看冷却倒计时，突然变成"需要内应信"——视觉跳变

3. **城池详情面板的冷却展示未在交互流程中定义**: R6的"暴露冷却debuff的UI展示位置汇总"(第277行)列出了5处展示位置，其中"城池详情面板"被列为第2处。但MAP-F06-07-R6的完整交互流程(Step 1-11)中**没有提到城池详情面板的冷却展示步骤**。这意味着:
   - 开发者知道"应该在城池详情面板显示冷却"，但不知道"在什么时机触发更新"
   - 是打开面板时实时计算？还是需要监听`insiderExposed`变化事件？

**影响**:

1. **状态切换边界模糊**: "暴露冷却优先于信件不足"的优先级规则在代码实现中需要明确的条件判断顺序，当前定义分散在不同段落，容易遗漏。
2. **冷却结束瞬间的UI跳变**: 玩家正在观察冷却倒计时，冷却结束瞬间卡片从"暴露冷却(置灰)"跳变到"需要内应信(置灰)"——虽然都是置灰，但文字和图标完全不同，可能造成困惑。
3. **城池详情面板的冷却展示缺乏流程定义**: 5处UI展示位置中，只有策略卡片、冷却弹窗、攻城结果面板在交互流程中有明确步骤。城池详情面板和攻城确认弹窗的策略说明区缺少更新触发时机的定义。

**建议**:

1. 将"暴露冷却优先于信件不足"的规则提升为独立的判定规则(而非状态C的附注):
   ```
   判定优先级:
     1. insiderExposed = true → 状态B(无论信件数量)
     2. insiderLetter >= 1 → 状态A
     3. 其他 → 状态C
   ```
2. 在冷却结束的`getCooldownDisplay`函数中，增加状态切换的过渡处理(如1秒延迟或toast提示"冷却已结束")
3. 在交互流程中补充城池详情面板和攻城确认弹窗的冷却展示触发时机

### CH-MAP-R6-05 一键收取离线溢出处理——玩家离线期间资源已超上限时的行为未定义 [P1]

**质疑流程**: MAP-F01-04-R6 二、一键收取全部，资源上限溢出处理
**追问**: "如果失败呢？" — 如果收取时资源已超上限会怎样？

**问题**:

R6定义了溢出处理:
```
if player.resources[type] + collectible[type] > storageCap[type]:
  actualCollect = storageCap[type] - player.resources[type]
  overflow = collectible[type] - actualCollect
  player.resources[type] = storageCap[type]
```

这个逻辑假设`player.resources[type] <= storageCap[type]`(当前资源不超过上限)。但以下场景可能导致**当前资源已超上限**:

1. **离线期间其他系统注入资源**: 如活动奖励、邮件领取、其他领土的自动产出(如果有)等，可能使当前资源超过仓库上限
2. **仓库降级**: 如果仓库等级被降级(如失去对应领土)，storageCap降低，当前资源可能超过新上限
3. **多资源类型部分溢出**: 铜钱到上限但粮草未到，一键收取时铜钱溢出丢弃、粮草正常收取——确认弹窗是否分别提示？

当`player.resources[type] > storageCap[type]`时:
```
actualCollect = storageCap[type] - player.resources[type]  // 结果为负数!
overflow = collectible[type] - actualCollect  // 溢出量计算错误
```

**影响**:

1. **负数收取量**: 如果当前资源已超上限，`actualCollect`为负数，逻辑上意味着"不仅要丢弃所有收取量，还要倒扣资源"——这显然不是预期行为。
2. **确认弹窗信息误导**: 弹窗显示"合计: 铜钱x15,400"，但实际收取可能为0(因为已超上限)，玩家会觉得被骗。
3. **溢出提示缺失**: 如果玩家不知道资源已超上限，一键收取后只看到"已收取0个领土的产出"会困惑。

**建议**:

在溢出处理前增加"已超上限"的检查:
```
if player.resources[type] >= storageCap[type]:
  // 已超上限，不收取该类型资源
  skipCollect(type)
 提示: "{资源名}已达上限，无法收取。建议消费后重试"
else if player.resources[type] + collectible[type] > storageCap[type]:
  // 正常溢出处理(同R6)
```

同时在确认弹窗中分别标注各资源的收取状态(正常/部分收取/已满跳过)。

### CH-MAP-R6-06 5处冷却UI展示位置的更新频率与触发机制不一致 [P1]

   - 是页面打开时启动1分钟定时器？还是全局定时器？
   - 如果玩家在攻城面板停留5分钟，倒计时是否每分钟自动刷新？
   - 如果玩家切换到其他Tab再切回来，倒计时是否立即刷新？

2. **冷却弹窗(每秒)与策略卡片(每分钟)的不一致**: 玩家在策略卡片上看到"23:45"(每分钟更新)，点击后弹窗显示"23:44:32"(每秒更新)。两个数字不一致(分钟级精度 vs 秒级精度)，玩家可能困惑"到底还剩多久"。

3. **冷却结束时的全局刷新**: 当冷却从"进行中"变为"结束"时(`remaining`从正数变为0)，5处UI都需要从"冷却中"状态切换为"正常"状态。这个状态变化是否需要全局事件通知？还是每个UI各自检查？

**影响**:

1. **开发实现不明确**: "每分钟"和"每秒"的更新频率需要不同的定时器策略，但触发机制未定义，开发者需要自行决定实现方式。
2. **数据一致性风险**: 5处UI各自独立计算`remaining`值，如果计算时机不同步，可能出现"策略卡片显示冷却中，但城池详情显示已结束"的不一致。
3. **性能隐患**: 如果每分钟全局轮询所有城池的冷却状态，在城池数量较多时(15块领土)可能产生不必要的性能开销。

**建议**:

1. 统一冷却状态的更新机制: 定义一个全局的冷却状态管理器(cooldownManager)，所有UI从该管理器读取状态
2. 明确更新策略:
   - 策略卡片/城池详情: 打开时计算 + 每分钟自动刷新 + 冷却结束时事件通知
   - 冷却弹窗: 每秒定时器(仅弹窗打开期间)
   - 攻城结果面板: 一次性写入
3. 定义冷却状态变化的事件通知机制: `cooldownStateChanged`事件，触发所有相关UI刷新

### R6整体评价

R6在"数值公式修正"方面取得了实质性进展——山贼战斗公式从"兵力被约掉"修正为"独立基础战力"，遗迹探索从"三档混乱"修正为"三档随机判定"，方向都是正确的。但在**数据验证的严谨性**上仍有不足:

1. **数据表与公式的矛盾**: 遗迹探索Lv10行的概率数据(partial=60%, success=30%)与公式计算结果(partial=30%, success=60%)完全相反——这是R6文档中最具破坏力的错误，直接影响开发实现。
2. **胜利损耗公式的边界溢出**: 山贼战斗的胜利损耗公式在低兵力场景下可产生超过100%的损耗率——这是R6新增的公式(CH-MAP-R5-03的修正)，但极端值验证未覆盖此场景。
3. **新增系统的数据完整性**: 情报值系统作为R6新增数据，关键参数(上限、获取速率、兑换规则)均未定义，处于"半成品"状态。

**R5质疑解决率**: 6项R5质疑中，4项完全解决(R5-02/R5-03/R5-04/R5-06)，2项部分解决(R5-01/R5-05)。整体解决质量较高，但P0级的数据表错误需要立即修正。

**建议R7重点**: 数据表与公式一致性验证 + 胜利损耗溢出修正 + 情报值系统闭环 + 声望衰减完整流程。
