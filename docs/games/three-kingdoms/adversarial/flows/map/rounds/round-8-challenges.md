## P0 质疑

### CH-MAP-R8-01 k=1.75推导依赖未声明的游戏设计哲学假设，[1.0,2.0]区间边界缺乏实证支撑 [P0]

**质疑流程**: MAP-F09-02-R8 胜利损耗上限数值推导与修正
**追问**: "这个数值对吗？" — k=1.75的选择是否真正有推导依据，还是在合理区间内的主观选择？

**问题**:

R8的推导链条在数学上是完整的: `35% = 20% x 1.75`。但推导的关键环节——k值范围[1.0, 2.0]的确定——依赖三个未被声明、未被验证的游戏设计哲学假设:

**假设1: "低兵力突袭应当有风险惩罚"**

R8第69行写道: "低兵力突袭(以少打多)应当有风险惩罚——防止无脑以少打多"。这是一个设计哲学判断，不是数学推导的公理。以下反论同样合理:

- 反论A: 低兵力突袭本身就是"高风险"(胜率低)，不需要额外的损耗惩罚。胜率7.9%(Lv70带1000兵)已经是足够的风险信号。
- 反论B: 低兵力突袭是玩家的策略选择(如试探敌情、消耗战术)，惩罚这种选择会降低策略多样性。
- 反论C: 如果"以少打多"不划算，玩家永远选择"以多打少"(带满兵)，战斗决策空间被压缩为"带兵量=max(安全量)"。

R8未回应这些反论，直接将"低兵力突袭应有惩罚"作为公理使用。

**假设2: "胜利/失败比不超过2.0x"的边界**

R8第100行: "k > 2.0: 低兵力突袭惩罚过重，胜利不如失败(设计矛盾)"。但2.0x作为边界是武断的:

| k值 | 胜利代价(极端) | 失败代价 | 比值 | 玩家感知 |
|-----|-------------|---------|------|---------|
| 1.5x | 30% | 20% | 1.5x | "赢了多付50%，还行" |
| 1.75x(R8) | 35% | 20% | 1.75x | "赢了多付75%，有点亏" |
| 2.0x | 40% | 20% | 2.0x | "赢了多付100%，亏了" |
| 2.5x(R7) | 50% | 20% | 2.5x | "赢了反而亏大了" |

2.0x作为"合理上限"缺乏依据。为何不是1.8x或2.2x？R8未提供任何玩家体验测试数据或心理学研究来支持这个边界。

**假设3: "1.75x位于[1.5, 2.0]区间中值偏上"的选择理由**

R8第102行: "理由: 1.75x位于[1.5, 2.0]区间中值偏上"。这不是推导，这是在主观区间内选择了一个"看起来合理"的位置。如果区间是[1.5, 2.0]，中值是1.75，那为何不选1.6(偏保守)或1.9(偏激进)？

R8的"设计意图验证清单"(第164-170行)5项全PASS——但验证的是"R8的数据是否符合R8自己的设计意图"，而非"R8的设计意图是否合理"。这是循环论证。

**影响**:

1. **推导链条的根基不稳**: k=1.75的推导表面上完整，但核心假设(低兵力应有惩罚、2.0x为边界)未被论证。如果这些假设被推翻，整个35%上限的合理性需要重新评估
2. **开发者实现困惑**: 如果产品经理问"为什么是1.75而不是1.6或1.9"，R8文档无法给出非循环论证的回答
3. **与其他系统的隐含约束**: 35%上限意味着Lv70带3000兵以下打山贼王时，胜利损耗恒定为35%。这可能影响玩家的"带兵量决策"——玩家可能选择不带3000以下兵力(避免触发截断)，反而降低了策略空间

**建议**:

1. 将k=1.75的选择理由从"区间中值偏上"改为基于玩家体验测试的结论，或明确标注为"初始设计值，需玩家测试验证"
2. 在文档中声明: "k值范围[1.0, 2.0]基于设计哲学判断而非实证数据，R9应基于玩家测试数据调整"
3. 提供k=1.5、k=1.75、k=2.0三种方案的A/B测试建议，而非仅选择一种

## P1 质疑

### CH-MAP-R8-02 声望衰减豁免条件1的"昨日活跃"引入新的边界漏洞——登录但不处理事件的玩家可永久免疫衰减 [P1]

**质疑流程**: MAP-F12-01-R8a 声望衰减时序边界修正
**追问**: "然后呢？" — R8修正了00:00时序问题，但"昨日活跃"是否引入了新的漏洞？

**问题**:

R8将豁免条件1从"当日活跃(today)"改为"昨日活跃(yesterday)"，解决了00:00时序边界问题。但引入了一个新的设计漏洞:

**漏洞场景**:
```
玩家A: 每日登录一次(打开游戏→看一眼→关闭)，不做任何事件
  - lastActiveDate = yesterday → 满足条件1 → 豁免
  - 结果: 声望永不衰减(即使完全不参与阵营系统)

玩家B: 从不登录
  - lastActiveDate < yesterday → 不满足条件1
  - 结果: 声望每日-1(正确)
```

**核心问题**: R8的"昨日活跃"将"登录"等同于"活跃"，但R7的原设计意图是"参与游戏的玩家不被衰减"。一个只登录不参与任何事件的玩家，在R8规则下声望永不衰减——这与衰减机制的设计意图(惩罚不活跃玩家)不一致。

**R7设计意图vs R8实际效果**:

| 玩家行为 | R7设计意图 | R7实际效果 | R8实际效果 |
|---------|-----------|-----------|-----------|
| 每日登录+处理事件 | 豁免 | 豁免(条件1或2) | 豁免(条件1或2) |
| 每日登录+不做事件 | 应衰减(不参与) | 豁免(条件1:today) | **豁免(条件1:yesterday)** |
| 不登录 | 衰减 | 衰减 | 衰减 |
| 只做快速处理 | R6声明豁免 | 衰减(矛盾) | 豁免(条件2) |

注意: R7的条件1(today)本身就有这个问题——每日登录的玩家自动豁免。R8的yesterday修正并没有使问题变严重，但也没有解决这个根本设计问题。

**影响**:

1. **衰减机制失效**: 对于"每日登录但不参与阵营"的玩家，衰减机制完全不生效。这些玩家的声望只增不减(如果有任何被动声望来源)
2. **设计意图偏离**: 声望衰减的目的是"惩罚不参与阵营系统的玩家"，但R8的豁免条件使"仅登录"就能免疫衰减
3. **与R8声望曲线模拟的隐含假设矛盾**: R8的模拟中"轻度活跃"玩家"每日+3"——但这个+3来自"快速处理3次/日"。如果一个玩家登录但不做快速处理，声望为+0且不衰减，声望曲线完全平坦

**建议**:

1. 考虑将条件1改为"昨日有声望相关操作"(而非仅"登录")，确保衰减机制惩罚的确实是"不参与阵营系统"的玩家
2. 或在条件2中增加"当日有任意游戏操作(非仅登录)"，将"纯登录"与"有操作登录"区分
3. 如果"登录即豁免"是设计意图(鼓励每日登录)，应在文档中明确声明这一设计决策

### CH-MAP-R8-03 R8声望曲线模拟缺少"登录但不处理事件"玩家类型，且6类玩家的数值依赖多个未验证假设 [P1]

**质疑流程**: MAP-F12-01-R8b 声望获取速率与平衡性验证
**追问**: "这个数值有意义吗？" — 6类玩家模拟是否覆盖所有场景？模拟数值的置信度如何？

**问题**:

R8定义了6类玩家(全勤/活跃/轻度活跃/周末/休闲/流失)，模拟了30天和60天的声望曲线。但存在以下问题:

**问题1: 缺少关键玩家类型**

R8未覆盖以下玩家类型:

| 缺失类型 | 行为特征 | 预期声望变化 | 为何需要覆盖 |
|---------|---------|-----------|------------|
| 登录但不处理事件 | 每日登录，不做任何事件 | +0(不衰减，见CH-MAP-R8-02) | 验证"登录即豁免"的影响 |
| 间歇性重度玩家 | 某周每日活跃，下两周不登录 | 波动(先快涨后衰减) | 验证声望波动的极端情况 |
| 新玩家(声望0起步) | 刚开始游戏 | 从0开始增长 | 验证0起步的积累速度 |
| 回流玩家(高声望) | 声望80，60天不活跃 | 80→0 | 验证高声望回流的惩罚是否过重 |

**问题2: 模拟数值依赖未验证的频率假设**

R8的"典型活跃"玩家每日获取+8.5的计算:
```
手动处理: +3 x 1次 = +3
快速处理: +1 x 3次 = +3
阵营任务: +5 x 0.5次 = +2.5
合计: +8.5
```

这些频率假设(手动1次/日、快速3次/日、任务0.5次/日)缺乏数据来源。如果实际频率不同:

| 假设偏差 | 对+8.5/天的影响 | 对30天声望的影响 |
|---------|---------------|----------------|
| 手动处理实际为0.5次/日 | +8.5→+7.0 | 210→210(满100截断) |
| 快速处理实际为1次/日 | +8.5→+6.5 | 210→195 |
| 阵营任务实际为0.2次/日 | +8.5→+7.0 | 210→210(满100截断) |

虽然对活跃玩家影响不大(7天即满100)，但对"周末玩家"影响显著:
```
周末玩家: 假设每周3天登录，每日+8.5 → 每周+25.5，减去4天衰减 → 每周净+21.5
如果实际每日+5 → 每周+15，减去4天衰减 → 每周净+11(差距近一倍)
```

**问题3: "周末玩家约+2/天"的推算过程不透明**

R8第440行声称周末玩家"约+2/天"，但未给出完整计算。按R8的数据:
```
每周3天登录，每日+8.5 → 每周获取+25.5
每周4天不登录 → 每周衰减约4(取决于豁免条件判定)
每周净变化: +25.5 - 4 = +21.5
每日平均: +21.5/7 ≈ +3.1/天
```

R8声称"+2/天"与计算结果"+3.1/天"不一致。除非衰减计算不同(如每日登录但不处理事件时也被衰减)，否则这个数字需要修正或解释。

**影响**:

1. **模拟可信度不足**: 缺少关键玩家类型和频率假设来源，模拟结果的参考价值有限
2. **周末玩家声望推算可能有误**: "+2/天"与按R8数据推算的"+3.1/天"不一致
3. **设计决策依据不充分**: 基于不完整模拟得出的"平衡性结论"可能误导后续设计

**建议**:

1. 补充"登录但不处理事件"玩家类型的模拟，验证CH-MAP-R8-02的影响
2. 标注频率假设的来源(如"基于R6玩家行为数据"或"初始假设值，需实测验证")
3. 修正周末玩家的推算过程，或解释与"+3.1/天"计算的差异

### CH-MAP-R8-04 冷却管理器60s扫描的并发竞态风险和内存管理边界未被讨论 [P1]

**质疑流程**: MAP-F06-07-R8 冷却管理器主动检查机制
**追问**: "如果失败呢？" — 60s扫描与实时操作并发时是否安全？

**问题**:

R8的冷却管理器增加了60s定时扫描机制，但未讨论以下实现风险:

**风险1: getCooldown()与_scanAllCooldowns()并发竞态**

```
时间线(竞态场景):
  T+0s    → 玩家打开天下Tab，UI调用getCooldown(cityA)
  T+0.001 → getCooldown()检测到remaining<=0，准备清理
  T+0.002 → 60s定时器触发_scanAllCooldowns()，也检测到cityA已过期
  T+0.003 → getCooldown()执行: state.status="ended"，emit事件，cooldowns.delete(cityA)
  T+0.004 → _scanAllCooldowns()执行: state.status="ended"(已设置)，emit事件(重复!)，cooldowns.delete(cityA)(已删除)
```

结果: `cooldownStateChanged`事件被触发两次，可能导致UI重复更新或弹窗重复显示。

**风险2: 扫描回调持有cooldowns引用的内存管理**

R8的`setInterval`回调通过闭包持有`this`(cooldownManager)的引用。如果cooldownManager被销毁(如玩家退出游戏Tab、页面卸载)但未调用`stopScan()`，定时器会继续运行:
```
后果:
  1. 定时器回调继续执行，遍历已失效的cooldowns Map
  2. 如果cooldowns Map已被GC回收，可能导致TypeError
  3. 如果cooldowns Map未被GC(被定时器引用)，内存泄漏
```

R8的`stopScan()`策略是"cooldowns为空时停止"，但未处理"cooldownManager被销毁"的场景。

**风险3: 60s扫描间隔的状态不一致窗口**

R8声称"状态不一致窗口从数小时缩短至最长60s"。但对于以下场景:
```
  10:00:00 → 冷却结束
  10:00:30 → 玩家打开天下Tab，调用getCooldown() → 状态正确
  vs
  10:00:00 → 冷却结束
  10:00:59 → 玩家打开天下Tab，UI立即显示"冷却中"(因为_scanAllCooldowns还未执行)
  10:01:00 → _scanAllCooldowns触发，状态更正 → UI突然跳变
```

60s窗口内玩家可能看到"冷却中→已结束"的UI跳变，体验不佳。

**影响**:

1. **事件重复触发**: 并发竞态可能导致同一冷却事件被通知两次
2. **内存管理风险**: 未处理cooldownManager销毁场景，可能导致内存泄漏或TypeError
3. **UI跳变**: 60s窗口内玩家可能看到不一致的UI状态

**建议**:

1. 在`_scanAllCooldowns()`中增加去重保护: 检查`state.status === "active"`后再操作，或使用`cooldowns.has(cityId)`前置检查
2. 增加`destroy()`方法: 在cooldownManager销毁时调用`stopScan()`，清理定时器引用
3. 考虑将扫描间隔缩短至10-15s(冷却通常为24h，10s扫描的性能开销可忽略)，减少UI跳变窗口

### CH-MAP-R8-05 R8声称"R6选择dailyLimit=3"的依据来自R6质疑文档而非R6流程文档，跨轮溯源链存在断裂 [P1]

R8验证: R8接受了R7的定义 → R8声称"与R6差异说明" → 但R6本身未定义3

正确的跨轮一致性应该是:
- R6: 未定义dailyLimit(仅"+1/事件，上限未定")
- R7: 定义dailyLimit=5(新定义，非"修改R6的3")
- R8: 应标注"R6未定义dailyLimit，R7新增此参数=5"，而非"R6=3→R8=5"

**影响**:

1. **跨轮溯源链不准确**: R8的"R6差异说明"建立在错误的R6数据上，开发者查阅R6流程文档时找不到dailyLimit=3的定义
2. **文档可信度受损**: 如果开发者发现R6并没有定义3次，会质疑R8其他跨轮引用的准确性

## P2 质疑

### CH-MAP-R8-06 战斗重试令牌"原兵力重试"在胜利损耗35%规则下的等效收益过高 [P2]

**质疑流程**: MAP-F12-02-R8b 战斗重试令牌精确定义
**追问**: "这个数值有意义吗？" — 重试令牌的实际收益是否合理？

**问题**:

R8精确定义了重试令牌的5要素，消除了歧义。但结合R8的胜利损耗35%规则，重试令牌的等效收益在特定场景下过高:

**场景分析**:

```
场景: Lv70带1000兵打山贼王(战力6300)
  胜率: max(5%, (1000/6300)*50%) = 7.9% → 截断为5%

  路径A: 战斗失败(95%概率)
    损耗: 1000 x 20% = 200兵
    净结果: -200兵

  路径B: 使用令牌重试
    第一次失败: 损耗200兵(已发生)
    重试(令牌消耗): 不额外征兵
    重试失败(95%概率): 损耗200兵(第二次)
    净结果: -400兵 + 令牌x1

    重试胜利(5%概率): 损耗35% = 350兵(胜利损耗)
    净结果: -200(第一次失败) - 350(重试胜利损耗) = -550兵 + 奖励
```

在极端低胜率场景(5%)下，重试令牌的期望收益:
```
E(令牌价值) = 5% * (奖励价值 - 550兵) + 95% * (-400兵 - 令牌)
```

如果奖励价值不足以覆盖550兵的损耗，令牌在低胜率场景下是**负期望值**。但R8文档声称令牌"节省了一次失败损耗"(第691行)，这仅在重试胜利时成立。

**更根本的问题**: 令牌"不额外征兵"意味着玩家可以无限次"试探"——虽然每次山贼事件限1枚，但如果情报值充足(上限100，可兑换10个令牌)，玩家可以在10个不同的山贼事件中各使用1枚。对于中等等级玩家(胜率30-50%)，10次重试中大约有3-5次成功，大幅降低了战斗风险。

**影响**:

1. **低胜率场景令牌无价值**: 5%胜率下重试几乎必然再次失败，令牌等效为"浪费一个道具+多损失200兵"
2. **中等胜率场景令牌过强**: 50%胜率下重试成功率50%，10枚令牌可额外获得约5次胜利奖励
3. **R8文档的极端值验证(第689-695行)缺少期望值分析**: 仅列出了"重试成功"和"重试失败"两个结果，未计算期望收益

**建议**:

1. 在极端值验证表中增加期望值分析列，展示不同胜率下令牌的期望收益
2. 考虑为令牌增加"保底机制": 如"重试时胜率+10%"，提升低胜率场景的令牌价值
3. 或限制令牌使用场景: "仅在胜率>20%时可使用重试令牌"，避免在无意义场景浪费

### CH-MAP-R8-07 dailyLimit=5的推导链中"离线事件上限5个"的假设未被验证 [P2]

**质疑流程**: MAP-F12-02-R8 情报值跨轮一致性修正
**追问**: "这个数值对吗？" — 离线事件上限确实是5个吗？

**问题**:

R8的推导链:
```
1. 情报值来源: 山贼快速处理"绕行"分支
2. 山贼事件触发机制: 离线期间累计，登录时批量展示
3. 离线事件上限: 系统设计为5个(防止长期离线后事件堆积)
4. 因此: 每日山贼快速处理上限 = 离线事件上限 = 5次
```

第3步"离线事件上限为5个"是一个系统设计假设，但在R8之前的流程文档中未被明确定义。R8将此作为已知事实使用，但:

- R6流程文档中未定义离线事件上限的具体数值
- R7流程文档中也未明确定义"离线事件上限=5"
- R8第210行写道"离线事件上限: 系统设计为5个"，但未给出此定义的来源

如果离线事件上限不是5(例如是3或8)，则dailyLimit=5的推导基础就不成立。

**影响**:

1. **推导链的前置条件未验证**: dailyLimit=5的合理性依赖"离线事件上限=5"这一假设
2. **如果离线事件上限为3**: 则dailyLimit应为3，与R6质疑建议一致
3. **如果离线事件上限为8**: 则dailyLimit=5限制了玩家的正常收益

**建议**:

1. 在R8文档中明确标注"离线事件上限=5"的来源(如"MAP-F01-04定义"或"系统配置参数")
2. 如果此参数尚未在流程中定义，应作为R8的数据变更项补充定义
3. 验证dailyLimit与离线事件上限的绑定关系是否合理——如果玩家某天登录多次(如早中晚各一次)，每次处理3个事件，每日实际可处理9个山贼事件。此时dailyLimit=5会限制玩家的正常收益

### CH-MAP-R8-08 声望衰减伪代码中条件1与条件2的判定存在逻辑重叠——昨日活跃+当日声望事件同时满足时的优先级未说明 [P2]

**质疑流程**: MAP-F12-01-R8a 声望衰减时序边界修正
**追问**: "这个边界对吗？" — 多个豁免条件同时满足时的行为是否明确？

**问题**:

R8第349-375行的伪代码:
```javascript
// 条件1: 昨日活跃
if player.lastActiveDate >= yesterday:
  player.decayExempt[faction] = true
  continue  // 跳过后续条件检查

// 条件2: 当日声望事件
if player.factionEventToday:
  player.decayExempt[faction] = true
  continue
```

伪代码使用`continue`实现了"短路求值"——条件1满足时不再检查条件2。但文档未说明:

1. **`decayExempt`字段的含义**: 是per-faction的还是全局的？伪代码中`decayExempt[faction]`暗示per-faction，但`player.lastActiveDate`和`player.factionEventToday`是全局的。如果玩家对"魏"有声望事件但对"蜀"没有，条件2应该如何判定？

2. **条件1与条件2的逻辑重叠**: 昨日活跃的玩家通常也会有当日声望事件(如果今日也登录)。两个条件的重叠区域很大，但伪代码的短路求值意味着条件2几乎不会被触达(条件1优先)。

3. **`factionEventToday`的重置时机**: 伪代码中未定义`factionEventToday`何时重置为false。如果是每日00:00重置，那么00:00之后到定时任务执行之间的声望事件会被忽略。

**影响**:

1. **per-faction vs 全局的歧义**: 开发者可能实现为全局判定(任何一个阵营有事件就全部豁免)或per-faction判定(每个阵营独立判定)
2. **条件2的可触达性低**: 由于条件1(昨日活跃)几乎覆盖了所有"有登录"的玩家，条件2(当日声望事件)只有在"今日未登录但有声望事件"时才被触达——这是一个极小概率的边界场景

**建议**:

1. 明确`decayExempt[faction]`的per-faction语义，说明`lastActiveDate`和`factionEventToday`是否需要per-faction判定
2. 如果条件2的目的是"只做快速处理的玩家不被衰减"(CH-MAP-R7-04)，应明确条件2的触发前提为"今日未登录"(否则条件1已覆盖)
3. 定义`factionEventToday`的重置时机(建议: 每日00:00重置，与衰减定时任务同步)

## 总结

| 级别 | 数量 | 核心主题 |
|------|------|---------|
| P0 | 1 | k=1.75推导依赖未声明的设计哲学假设 |
| P1 | 4 | "昨日活跃"引入登录免疫漏洞、声望模拟缺少关键玩家类型、冷却扫描并发竞态、dailyLimit跨轮溯源断裂 |
| P2 | 3 | 重试令牌期望值未分析、离线事件上限假设未验证、豁免条件逻辑重叠 |

### R8整体评价

R8在R7质疑的响应上取得了显著进展:

1. **设计矛盾清零**: 胜利损耗50%→35%的修正消除了"胜利比失败更惨"的核心矛盾，数学推导链条(35%=20%x1.75)完整。跨轮文档一致性(情报值R6差异标注、声望衰减豁免修正)的处理比R7更加严谨。

2. **数值补全**: 声望获取速率的完整定义(+3/+1/+5/+10/+2五种来源)填补了R7的关键空白，6类玩家的30天/60天声望曲线模拟为平衡性验证提供了数据基础。

3. **系统闭环**: 冷却管理器从被动触发改为主动扫描，消除了UI驱动状态更新的设计缺陷。重试令牌5要素精确定义消除了开发歧义。

4. **方法论进步**: "设计意图验证"步骤的引入(R8第159-170行)是方法论的重要改进，使验证从"数据正确性"扩展到"设计合理性"。

**R8的核心弱点**:

1. **推导根基的主观性**: k=1.75的推导链条在数学上完整，但根基——"低兵力应有惩罚"、"2.0x为边界"——是设计哲学判断而非实证结论。R8将这些判断作为公理使用，未声明其假设性质。

2. **模拟的完整性**: 声望曲线模拟缺少"登录但不处理事件"这一关键边界类型，且模拟数值依赖多个未标注来源的频率假设。

3. **实现细节的鲁棒性**: 冷却管理器的并发竞态、内存管理、豁免条件的per-faction语义等实现层面的风险未被讨论。

**R7质疑解决率**: 8项R7质疑中，4项完全解决(R7-04/R7-05基本解决/R7-06/R7-07通过时序修正)，4项部分解决(R7-01推导根基不稳/R7-02溯源链断裂/R7-03模拟不完整/R7-05并发风险)。P0级的设计矛盾(胜利损耗)被有效修正，但修正的推导依据需要更坚实的论证基础。

**建议R9重点**: k值实证验证(或明确标注为初始假设) + 声望模拟补充边界玩家类型 + 冷却管理器并发安全方案 + 跨轮溯源链(区分"流程定义"与"质疑建议")准确性审查。
