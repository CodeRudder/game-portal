### MAP-F09-02-R9 胜利损耗k=1.75公理声明化 {#map-f09-02-r9}

> R9核心目标#1: 推导根基夯实
> CH-MAP-R8-01: k=1.75推导依赖未声明的游戏设计哲学假设，[1.0,2.0]区间边界缺乏实证支撑
> 修正原则: 将隐含的公理假设显式声明为"设计假设"，标注性质，提供A/B测试建议

#### 一、问题定位

R8的推导链条在数学上完整: `35% = 20% x 1.75`。但推导的关键环节——k值范围[1.0, 2.0]的确定——依赖三个未被声明、未被验证的设计哲学假设。R8将这些判断作为公理使用但未声明其假设性质，导致:

1. 开发者无法回答"为什么是1.75而不是1.6或1.9"
2. R8的"设计意图验证清单"5项全PASS是循环论证(验证R8数据是否符合R8自己的设计意图)
3. Challenger提出的三个反论(低兵力本身已是高风险、惩罚降低策略多样性、以少打多不划算)同样合理但未被回应

#### 二、设计公理声明(新增)

> R9方法论改进: 在"设计意图验证"之前增加"设计公理声明"步骤——列出推导依赖的假设，标注性质(实证/判断/惯例)

**公理1: "低兵力突袭应当有风险惩罚"** [设计假设]

```
声明:
  低兵力突袭(以少打多)在胜利损耗上应有额外惩罚，防止无脑以少打多策略。

假设性质: 设计哲学判断(非实证结论)

支撑论据:
  - 行业惯例: 多数策略游戏中低兵力出征的风险高于高兵力出征
  - 体验直觉: "以少打多"应有代价，否则玩家永远选择最低兵力出征
  - R8设计意图: 鼓励玩家合理配置兵力，而非"带100兵试探+重试令牌兜底"

已知反论(Challenger CH-MAP-R8-01):
  - 反论A: 低兵力突袭本身就是"高风险"(胜率7.9%)，不需要额外损耗惩罚
  - 反论B: 惩罚低兵力突袭降低策略多样性(试探、消耗战术)
  - 反论C: 如果以少打多不划算，玩家永远带满兵，决策空间被压缩

裁决状态: 设计假设，需玩家测试数据验证
```

**公理2: "胜利/失败损耗比不超过2.0x"** [设计假设]

```
声明:
  胜利时的兵力损耗不应超过失败时的2.0倍。超过此边界时，"胜利"在数值上劣于"失败"，
  违反"胜利应当有正向体验"的基本游戏设计原则。

假设性质: 设计哲学判断(非实证结论)

支撑论据:
  - 心理学直觉: 胜利代价超过失败代价的2倍时，玩家会产生"赢了反而亏"的负面体验
  - R7验证: 2.5x(R7的50%上限)已被确认为"过高"(胜利比失败更惨的设计矛盾)
  - 区间选择: [1.0, 2.0]区间内，1.0x=胜败持平(无战斗动机)，2.0x=胜利代价的合理上限

已知反论:
  - 2.0x作为边界是武断的——为何不是1.8x或2.2x？
  - 缺乏玩家体验测试数据或心理学研究支撑

裁决状态: 设计假设，需玩家测试数据验证。建议A/B测试1.8x和2.2x边界
```

**公理3: "k=1.75位于[1.5, 2.0]区间中值偏上"** [设计假设]

```
声明:
  在[1.5, 2.0]区间内选择k=1.75(中值偏上)，理由是:
  - 1.5x偏保守(低兵力惩罚偏弱，以少打多仍有明显优势)
  - 2.0x偏激进(接近边界，玩家感知"赢了多付100%")
  - 1.75x为平衡点(玩家感知"赢了多付75%，有点亏但可接受")

假设性质: 主观判断(区间内任意选择)

已知反论:
  - 1.6x(偏保守)或1.9x(偏激进)同样合理
  - 区间选择本身就是主观的

裁决状态: 初始设计值，需A/B测试验证。建议提供k=1.5/1.75/2.0三种方案对比
```

#### 三、R8设计意图验证清单修正

R8的"设计意图验证清单"存在循环论证问题——验证的是R8数据是否符合R8自己的设计意图，而非设计意图是否合理。R9增加"公理合理性"前置步骤:

```
R9设计意图验证清单(修正版):

  前置步骤: 公理合理性审查
    [声明] 公理1: 低兵力应有惩罚 → 设计假设，已知反论3条，需玩家测试
    [声明] 公理2: 胜/败比<=2.0x → 设计假设，需A/B测试1.8x/2.2x
    [声明] 公理3: k=1.75 → 初始设计值，需A/B测试1.5/1.75/2.0

  验证步骤:
    [PASS] 1. 胜利是否比失败损失更少? → 正常场景(兵力充足): 是(12.6% < 20%)
    [PASS] 2. 低兵力突袭是否有惩罚? → 极端场景(兵力不足): 是(35% > 20%)
    [PASS] 3. 惩罚是否过重(胜利代价>2x失败)? → 最高1.75x，不超过2.0x(公理2)
    [PASS] 4. 文档声称与数据是否一致? → 声称"1.75倍风险惩罚"，数据表一致
    [PASS] 5. 数值是否有推导依据? → 35% = 20% x 1.75，k=1.75选自[1.5,2.0]区间

  结论: 内部一致性通过。但公理1/2/3均为设计假设，k=1.75为初始设计值，
       需玩家测试数据验证后方可确认为最终值。
```

#### 三、修正方案: 离线事件上限参数正式定义(新增)

**offlineEventCap 参数定义**:

```
参数名: eventSystem.offlineEventCap
类型: integer
默认值: 5
定义: 玩家离线期间系统累计存储的最大事件数。超过此上限时，新事件替换最旧事件(FIFO)。
来源: 系统设计参数(R9首次正式定义)
绑定关系: intelPoints.dailyLimit = offlineEventCap(情报值每日获取上限等于离线事件上限)

设计理由:
  1. 防止长期离线后事件堆积: 玩家离线30天，若无限累计则登录时面对90+事件，体验灾难
  2. 离线事件以山贼为主: 山贼事件每日约触发1-3个(基于在线触发频率)，离线5天累计约5-15个
  3. 上限=5的合理性: 5个事件约等于离线1.7-5天的累计量，玩家登录后5-10分钟可处理完毕
  4. dailyLimit绑定: 每日情报值获取上限=离线事件上限，确保玩家不会因"事件多但收益上限低"产生割裂感

已知风险(Challenger CH-MAP-R8-07):
  - 若offlineEventCap=3: dailyLimit应为3，与R6质疑建议一致
  - 若offlineEventCap=8: dailyLimit=5限制了玩家的正常收益
  - 若玩家每日登录多次(早中晚各一次): 每次处理部分事件，实际不受offlineEventCap限制

裁决状态: 初始设计值=5，需玩家行为数据验证。建议监控"玩家登录时事件数量分布"指标
```

**数据变更**:

|------|-----------|-----------|------|
| eventSystem.offlineEventCap | 未正式定义(假设=5) | **正式定义=5(系统参数)** | CH-MAP-R8-07 |
| intelPoints.dailyLimit推导链 | "离线事件上限=5"(未验证) | "dailyLimit=offlineEventCap=5"(正式绑定) | CH-MAP-R8-07 |
| R6差异说明 | "R6选择3次"(错误) | "R6未定义，R6质疑建议=3" | CH-MAP-R8-05 |

### MAP-F12-01-R9 声望模拟完整性提升: 7类玩家 + 频率假设标注 + 周末玩家推算修正 {#map-f12-01-r9}

> R9核心目标#2: 模拟完整性提升
> CH-MAP-R8-02: "昨日活跃"引入登录但不处理事件的玩家可永久免疫衰减
> CH-MAP-R8-03: 声望曲线模拟缺少关键玩家类型，频率假设未标注来源，周末玩家推算不一致
> 修正原则: 补充第7类玩家类型 + 频率假设全部标注来源 + 修正周末玩家推算

#### 一、问题定位

**问题1: 缺少"登录但不处理事件"玩家类型(CH-MAP-R8-02)**

R8的6类玩家模拟未覆盖一个关键边界类型: 每日登录但不做任何事件的玩家。根据R8的豁免条件1(昨日活跃=豁免)，这类玩家声望永不衰减——如果他们有任何被动声望来源，声望只增不减。

**问题2: 频率假设未标注来源(CH-MAP-R8-03)**

R8的"典型活跃"玩家每日获取+8.5依赖多个频率假设:
```
手动处理: +3 x 1次/日 = +3    ← 假设来源?
快速处理: +1 x 3次/日 = +3    ← 假设来源?
阵营任务: +5 x 0.5次/日 = +2.5 ← 假设来源?
```
这些假设缺乏数据来源标注，降低了模拟的可信度。

**问题3: 周末玩家推算差异(CH-MAP-R8-03)**

R8第440行声称周末玩家"约+2/天"，但按R8数据推算:
```
每周3天登录，每日+8.5 → 每周获取+25.5
每周4天不登录 → 每周衰减约4(取决于豁免条件)
每周净变化: +25.5 - 4 = +21.5
每日平均: +21.5/7 ≈ +3.1/天
```
R8声称"+2/天"与推算"+3.1/天"不一致。

#### 二、修正方案: 7类玩家矩阵(新增第7类)

**玩家行为矩阵(R9)**:

| 象限 | 登录频率 | 事件参与度 | 典型玩家 |
|------|---------|-----------|---------|
| 高登录+高参与 | 每日 | 多事件 | 全勤玩家、活跃玩家 |
| 高登录+低参与 | 每日 | 少事件或无事件 | 轻度活跃、**登录不处理事件(新增)** |
| 低登录+高参与 | 每周1-3次 | 登录日多事件 | 周末玩家 |
| 低登录+低参与 | 每周1-2次 | 登录日少事件 | 休闲玩家 |
| 无登录 | 无 | 无 | 流失玩家 |

**第7类玩家定义: 登录但不处理事件** [新增]

```
玩家类型: 登录不处理事件
行为特征:
  - 每日登录一次(打开游戏→看一眼→关闭)
  - 不做任何阵营事件(不处理冲突、不做任务、不使用道具)
  - lastActiveDate = yesterday → 满足豁免条件1 → 声望永不衰减

声望行为:
  - 每日获取: +0(无事件)
  - 每日衰减: 0(豁免条件1: 昨日活跃)
```

### MAP-F06-07-R9 冷却管理器并发安全方案 {#map-f06-07-r9}

> R9核心目标#3: 实现鲁棒性保障
> CH-MAP-R8-04: 冷却管理器60s扫描的并发竞态、内存管理、UI跳变未被讨论
> 修正原则: 乐观锁去重 + destroy()生命周期 + 扫描间隔缩短至10s

#### 一、问题定位

R8的冷却管理器增加了60s定时扫描机制，但存在三个实现风险:

| 风险 | 场景 | 影响 |
|------|------|------|
| 并发竞态 | getCooldown()与_scanAllCooldowns()同时操作同一cooldown条目 | 事件重复触发 |
| 内存泄漏 | cooldownManager被销毁但未调用stopScan() | 定时器持续运行+TypeError |
| UI跳变 | 60s窗口内玩家看到"冷却中→已结束"跳变 | 体验不佳 |

#### 二、修正方案: 并发安全——乐观锁去重

**方案选择: 乐观锁(非互斥锁)**

```
选择理由:
  1. JavaScript单线程环境: 不存在真正的并行执行，"并发"实为交错执行
  2. 事件循环特性: getCooldown()和_scanAllCooldowns()不会真正同时执行，
     但在同一事件循环tick内可能被调度
  3. 乐观锁足够: 通过状态检查(已处理标记)避免重复操作，无需互斥锁的复杂度

方案: 在cooldown条目中增加_processed标记
  - getCooldown()检测到过期时: 检查_processed，若false则设置为true并触发事件
  - _scanAllCooldowns()检测到过期时: 检查_processed，若false则设置为true并触发事件
  - 效果: 先到达的触发事件，后到达的跳过(已_processed)
```

**修正后(R9)并发安全代码**:

```javascript
cooldownManager = {
  cooldowns: Map<cityId, CooldownState>,
  listeners: Set<callback>,
  _scanTimer: null,
  _scanInterval: 10000,  // R9: 10s扫描间隔(原60s)

  // R9修正: getCooldown()增加去重保护
  getCooldown(cityId):
    state = cooldowns.get(cityId)
    if !state: return null

    remaining = state.endAt - now()
    if remaining <= 0 && state.status === "active":
      // R9: 乐观锁——检查是否已被_scanAllCooldowns处理
      if !state._processed:
        state._processed = true  // 标记为已处理
        state.status = "ended"
        emit("cooldownStateChanged", { cityId, status: "ended" })
        cooldowns.delete(cityId)
      return null

    return { status: state.status, remaining }

  // R9修正: _scanAllCooldowns()增加去重保护
  _scanAllCooldowns():
    now = now()
    for [cityId, state] of cooldowns:
      if state.status === "active":
        remaining = state.endAt - now
        if remaining <= 0:
          // R9: 乐观锁——检查是否已被getCooldown()处理
          if !state._processed:
            state._processed = true  // 标记为已处理
            state.status = "ended"
            emit("cooldownStateChanged", { cityId, status: "ended" })
            cooldowns.delete(cityId)

    if cooldowns.size === 0:
      stopScan()

  // R9新增: destroy()生命周期方法
  destroy():
    stopScan()
    cooldowns.clear()
    listeners.clear()
    // 释放所有引用，允许GC回收
}
```

**并发竞态解决验证**:

```
修正前竞态场景:
  T+0.001 → getCooldown()检测到cityA过期，准备清理
  T+0.002 → _scanAllCooldowns()也检测到cityA过期
  T+0.003 → getCooldown()执行清理，emit事件
  T+0.004 → _scanAllCooldowns()执行清理，emit事件(重复!)

修正后(R9)竞态场景:
  T+0.001 → getCooldown()检测到cityA过期
  T+0.001 → 检查state._processed == false → 设置_processed = true
  T+0.001 → 执行清理，emit事件
  T+0.002 → _scanAllCooldowns()检测到cityA过期
  T+0.002 → 检查state._processed == true → 跳过(已处理)
  → 结果: 事件仅触发一次，无重复
```

#### 三、修正方案: 内存管理——destroy()生命周期

**问题**: R8未处理cooldownManager销毁场景。`setInterval`回调通过闭包持有`this`引用，销毁后定时器继续运行。

**R9修正: 增加destroy()方法**:

```
destroy()方法职责:
  1. 调用stopScan()清除定时器
  2. 清空cooldowns Map(释放所有条目引用)
  3. 清空listeners Set(释放所有回调引用)
  4. 后续调用getCooldown()/setCooldown()应安全返回(幂等)

调用时机:
  - 玩家退出天下Tab时
  - 页面卸载(beforeunload)时
  - 游戏实例销毁时

安全保护:
  - destroy()后调用getCooldown(): 返回null(不抛异常)
  - destroy()后调用setCooldown(): 重建cooldowns Map(惰性初始化)
  - destroy()后调用stopScan(): 幂等(无定时器时无操作)
```

#### 四、修正方案: 扫描间隔缩短至10s

**问题**: R8的60s扫描间隔导致最长60s的UI跳变窗口。

**R9修正: 扫描间隔从60s缩短至10s**:

```
性能影响分析:
  冷却时长: 通常24h(86400s)
  扫描间隔: 10s
  扫描次数/冷却周期: 86400/10 = 8640次
  单次扫描开销: 遍历cooldowns Map(通常1-5个条目) → O(n), n极小
  总开销: 可忽略(现代浏览器setInterval 10s开销约0.001ms/次)

UI跳变窗口:
  修正前(R8): 最长60s
  修正后(R9): 最长10s → 玩家几乎不可感知
```

#### 二、修正方案: per-faction语义明确化

**字段语义定义(R9)**:

```
字段: player.lastActiveDate

  语义: per-faction(每个阵营独立判定)
  含义: 玩家今日是否对指定阵营执行过声望相关操作

  类型: Map<faction, boolean>
  语义: per-faction(每个阵营独立判定)
  含义: 今日是否对指定阵营豁免衰减
  更新时机: 每日定时任务中根据条件1/2/3判定
```

**条件1与条件2的关系明确化(R9)**:

```
条件1(昨日活跃) — 全局判定:
  判定: player.lastActiveDate >= yesterday
  效果: 对所有阵营统一豁免
  场景: 玩家昨日登录过(即使只看了眼商城)→ 所有阵营不衰减

条件2(当日声望事件) — per-faction判定:
  判定: player.factionEventToday[faction] == true
  效果: 仅对有事件的阵营豁免
  场景: 玩家今日对"魏"有声望事件但对"蜀"没有 → 魏豁免，蜀可能衰减

条件2的可触达性(R9明确):
  条件2仅在条件1不满足时被检查(短路求值)
  条件1不满足 = 昨日未登录
  条件2满足 = 今日有声望事件(但昨日未登录)
  → 触发场景: 玩家昨日未登录，但今日通过其他途径(如推送通知+快速处理)产生了声望事件
  → 这是一个极小概率的边界场景，但逻辑上正确且符合R6"快速处理不触发声望衰减"声明
```

#### 三、修正方案: factionEventToday重置时机

**重置时机定义(R9)**:

```
重置时机: 每日00:00定时任务执行时(在衰减判定之后)

执行顺序:
  1. 00:00:00 → 定时任务开始
  2. 遍历所有玩家 → 对每个阵营执行衰减判定(使用当前factionEventToday值)
  3. 衰减判定完成后 → 重置所有玩家的factionEventToday为false
  4. 次日00:00:00 → 下一轮定时任务(使用新的factionEventToday值)

代码:
  function dailyReputationDecay():
    today = formatDate(now())
    yesterday = formatDate(now() - 24h)

    for each player in allPlayers:
      for each faction in [魏, 蜀, 吴]:
        currentRep = player.factionRep[faction]

        // 条件3: 已到下限
        if currentRep <= 0:
          continue

        // 条件1: 昨日活跃(全局判定)
        if player.lastActiveDate >= yesterday:
          player.decayExempt[faction] = true
          continue

        // 条件2: 当日声望事件(per-faction判定)
        if player.factionEventToday[faction]:  // R9: per-faction判定
          player.decayExempt[faction] = true
          continue

        // 无豁免，执行衰减
        player.factionRep[faction] = max(0, currentRep - 1)
        player.decayAmount[faction] = 1

    // R9新增: 衰减判定完成后重置factionEventToday
    for each player in allPlayers:
      player.factionEventToday = { 魏: false, 蜀: false, 吴: false }
```

**时序边界验证(R9)**:

| 场景 | 时间线 | factionEventToday | 衰减判定 | 重置 | 结果 |
|------|--------|------------------|---------|------|------|
| 23:59处理魏事件 | 23:59→00:00 | {魏:true, 蜀:false, 吴:false} | 00:00判定: 魏豁免 | 00:00后重置 | 魏不衰减，蜀/吴可能衰减 |
| 00:01处理蜀事件 | 00:01(定时任务已执行) | {魏:false, 蜀:false, 吴:false}(已重置) | 已执行 | N/A | 蜀事件记录在明日判定 |
| 次日00:00 | 00:00 | {魏:false, 蜀:true, 吴:false} | 判定: 蜀豁免 | 重置 | 蜀不衰减 |

**结论**: factionEventToday在衰减判定后重置，确保当日事件被正确计入判定。

### MAP-F12-02-R9b 战斗重试令牌期望值分析 {#map-f12-02-r9b}

> R9核心目标: 极端值验证完善
> CH-MAP-R8-06: 重试令牌在低胜率场景期望值为负，缺少期望值分析
> 修正原则: 极端值验证表增加期望值分析列

#### 一、问题定位

R8的极端值验证表(第689-695行)仅列出"重试成功"和"重试失败"两个结果，未计算期望收益。在低胜率场景(5%)下，重试令牌的期望收益为负。

#### 二、修正方案: 期望值分析表(新增)

**重试令牌期望值分析(不同胜率)**:

| 场景 | 胜率 | 首次失败损耗 | 重试失败损耗 | 重试胜利损耗 | E(重试收益) | 结论 |
|------|------|------------|------------|------------|-----------|------|
| Lv70带10000兵 | 95% | 200(20%x10000) | N/A | N/A | 不使用(首次大概率胜利) | 不适用 |
| Lv70带5000兵 | 50% | 1000(20%x5000) | 1000 | 1750(35%x5000) | 50%x(奖励-1750)+50%x(-1000) | **中等正期望**(若奖励>2750兵) |
| Lv70带3000兵 | 33% | 600(20%x3000) | 600 | 1050(35%x3000) | 33%x(奖励-1050)+67%x(-600) | **低正期望**(若奖励>2424兵) |
| Lv70带1000兵 | 5% | 200(20%x1000) | 200 | 350(35%x1000) | 5%x(奖励-350)+95%x(-200) | **负期望**(奖励需>4100兵才正) |
| Lv70带500兵 | 5% | 100(20%x500) | 100 | 175(35%x500) | 5%x(奖励-175)+95%x(-100) | **负期望**(奖励需>2175兵才正) |

**期望值公式**:

```
E(重试令牌价值) = P(重试胜利) x (奖励价值 - 首次失败损耗 - 重试胜利损耗)
                + P(重试失败) x (-首次失败损耗 - 重试失败损耗 - 令牌价值)

其中:
  P(重试胜利) = 战斗胜率(与首次相同)
  P(重试失败) = 1 - 战斗胜率
  首次失败损耗 = 出征兵力 x 20%
  重试胜利损耗 = 出征兵力 x min(35%, 20% x 山贼战力/出征兵力)
  重试失败损耗 = 出征兵力 x 20%
  令牌价值 = 情报值兑换成本(10情报值/枚)
```

**关键发现**:

```
1. 低胜率场景(5%): 重试令牌期望值为负
   - 玩家有95%概率再次失败，多损失200兵(出征兵力x20%)
   - 仅5%概率胜利，但胜利损耗350兵(出征兵力x35%)
   - 除非奖励价值极高(>4100兵等价)，否则期望值为负
   - 建议: 低胜率场景下UI应提示"胜率较低，使用令牌可能不划算"

2. 中等胜率场景(33-50%): 重试令牌期望值为正
   - 玩家有33-50%概率胜利，奖励收益可覆盖失败风险
   - 建议: 此场景下令牌使用是合理的策略选择

3. 高胜率场景(>80%): 重试令牌几乎无价值
   - 首次失败概率极低(<20%)，令牌大概率不被使用
   - 建议: 高胜率场景下令牌积压为已知问题，后续迭代可扩展消耗途径
```

**极端值验证表(R9修正版)**:

**问题分析**: 缺少"登录不处理事件"边界类型，频率假设置信度不明
**修订方案**: 补充第7类玩家，所有频率假设标注"初始假设值"，增加可信度声明
**优先级**: P1

### PRD-R029 冷却管理器并发安全+生命周期

**PRD原文**: R8定义60s主动扫描，无并发保护和destroy()
**问题分析**: 并发竞态导致事件重复，无destroy()导致内存泄漏
**修订方案**: 增加乐观锁去重、destroy()方法、扫描间隔缩短至10s
**优先级**: P1

### PRD-R030 R6差异说明溯源修正

**PRD原文**: R8声称"R6选择dailyLimit=3"
**问题分析**: R6流程文档从未定义此参数，数值来自R6质疑建议
**修订方案**: 修正为"R6未定义，R6质疑建议=3，R7新增=5"，增加跨轮引用溯源校验规则
**优先级**: P1

### PRD-R031 声望豁免per-faction语义+重置时机

**PRD原文**: R8伪代码中decayExempt[faction]但lastActiveDate和factionEventToday全局
**问题分析**: per-faction vs全局歧义，factionEventToday重置时机未定义
**修订方案**: factionEventToday改为Map<faction, boolean>，定义00:00衰减后重置
**优先级**: P2

### PRD-R032 重试令牌期望值分析

**PRD原文**: R8极端值验证表仅列出成功/失败结果
**问题分析**: 缺少期望值分析，低胜率场景令牌收益不明
**修订方案**: 极端值验证表增加期望值分析列，增加低胜率UI提示建议
**优先级**: P2
