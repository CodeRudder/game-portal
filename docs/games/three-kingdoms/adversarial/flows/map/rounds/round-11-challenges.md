### CH-MAP-R11-01 期望值极端值表10000兵场景R_critical表格值与公式推导值不一致 [P1]

**质疑流程**: MAP-F12-02-R11 期望值极端值表数值修正+三方交叉校验
**追问**: "这个数值对吗？" — 三方交叉校验是否对所有场景均通过？

**问题**:

R11在极端值表(第233-239行)中列出了5个场景的R_critical值。独立验算发现场景5(10000兵/95%胜率)存在表格值与公式推导值不一致:

```
R11表格(第235行):
  场景: Lv70带10000兵
  P(W): 95%
  L_win: 3500
  L_lose: 2000
  R_critical: ≈3605+1.05V_T
  推导公式: 3500+(1/19)x2000+V_Tx(20/19)

独立验算推导公式:
  R_critical = L_win + (P(L)/P(W)) x L_lose + V_T/P(W)
             = 3500 + (0.05/0.95) x 2000 + V_T/0.95
             = 3500 + 105.263... + 1.0526... x V_T
             = 3605.26... + 1.0526... x V_T

  V_T=3: R_critical = 3605.26 + 3.158 = 3608.42

  与R11表格声称的"≈3605+1.05V_T"一致(四舍五入差异可忽略)
```

**但R11在差异点说明表(第241-248行)中，R10对比表的R11精确阈值列与上述推导不一致**:

```
R11差异点说明表(第243行):
  5000兵/50%: R11精确阈值 >2756(含V_T=3)  ← 与公式一致: 2750+2x3=2756 ✓
  3000兵/33%: R11精确阈值 >2259(含V_T=3)  ← 与公式一致: 2250+3x3=2259 ✓
  1000兵/5%:  R11精确阈值 >4210(含V_T=3)  ← 与公式一致: 4150+20x3=4210 ✓
  500兵/5%:   R11精确阈值 >2135(含V_T=3)  ← 与公式一致: 2075+20x3=2135 ✓
```

4个实际决策场景的三方校验均通过。但极端值表中10000兵场景的表格值存在内部不一致:

```
极端值表(第235行): R_critical ≈ 3605+1.05V_T
  → V_T=3时: 3605 + 3.15 = 3608.15

R10对比表中未列出10000兵场景(合理，因标记为"不适用")

但三方交叉校验汇总表(第252-257行)也未包含10000兵场景
```

**核心问题**: R11声称"三方交叉校验扩展到期望值表"，但10000兵场景被排除在三方校验汇总表之外。虽然该场景标记为"不适用"，但三方校验方法论的核心原则是"所有数值产出均需三方一致"——排除任何场景都会削弱方法论的可信度。

**补充验证**: R11在10000兵场景的推导公式列为"3500+(1/19)x2000+V_Tx(20/19)"，验算:

```
3500 + (1/19) x 2000 + V_T x (20/19)
= 3500 + 105.26... + 1.0526... x V_T
= 3605.26... + 1.0526... x V_T

与表格声称的"≈3605+1.05V_T"一致 ✓
```

表格内的推导公式与R_critical值一致，但三方校验汇总表遗漏了该场景。

**影响**:

1. **方法论执行不完整**: R11的核心卖点是"三方交叉校验扩展到期望值表"，但三方校验汇总表仅覆盖4/5场景，10000兵场景被排除——这削弱了"三方校验全面覆盖"的声明
2. **实际决策无影响**: 10000兵/95%场景标记为"不适用(首次大概率胜利)"，其R_critical值不影响任何实际游戏决策
3. **方法论严格性**: 三方校验应覆盖所有数值产出(包括"不适用"场景)，以确保内部一致性——这是R10 Arbiter明确要求的("将三方校验扩展到所有数值产出")

**建议**:

1. 在三方校验汇总表中增加10000兵场景行: `| 10000兵/95% | 3605+1.05V_T | E(3608)≈0 ✓ | 3608 | ✓ 通过 |`
2. 或在汇总表下方注明: "10000兵场景因标记为'不适用'未纳入汇总表，但表格内推导公式与R_critical值已内部一致"

**质疑流程**: MAP-F12-02-R11a dailyLimit约束逻辑定义+多登录场景修正
**追问**: "如果失败呢？" — dailyReset未按时触发或并发执行时会怎样？

**问题**:

R11声称提供了"完整约束逻辑伪代码"(差异点说明表第14行):

```
R11差异点说明表:
  dailyLimit约束逻辑: 未定义 → 完整伪代码+约束时机+约束范围
```

R11实际提供的伪代码包含3个函数:

```
函数1: addIntelPoints(playerId, source, amount) — 第339-359行
  ✓ 完整: 包含remaining计算、截断逻辑、日志记录

函数2: processEvent(playerId, event) — 第362-372行
  ✓ 完整: 包含情报值获取、其他奖励处理

函数3: dailyReset(playerId) — 第375-377行
  ✗ 骨架: 仅3行(player获取+intelPointsToday=0)
```

**dailyReset的缺失细节**:

```
问题1: 触发机制未定义
  方案A: 服务器定时任务(每日0:00对所有玩家执行)
  方案B: 玩家登录时检查(如果日期已变则执行)
  方案C: 混合模式(服务器定时+登录时检查)
  R11未指定使用哪种方案。

问题2: 在线玩家并发风险
  若使用方案A: 服务器0:00执行dailyReset时，可能有玩家正在线处理事件
  竞态条件: 玩家在23:59:59开始处理事件 -> 0:00时dailyReset执行 -> intelPointsToday被重置
  可能结果: 事件处理中途intelPointsToday被重置为0，导致当日情报值计数不准确

问题3: R11跨日场景的隐含假设
  R11跨日场景(第440-448行): "23:50登录，处理3个事件(+3)，0:00重置，继续处理2个事件(+2)"
  该场景假设"0:00重置"自动发生(方案A)
  但如果使用方案B(登录时检查): 0:00不会触发重置(玩家已在线)
  → 玩家在0:00-0:10处理事件时，intelPointsToday仍=3(非0)
  → 处理2个事件: intelPointsToday=5(而非R11声称的2)
```

**与addIntelPoints的对比**:

```
addIntelPoints: 40行伪代码，包含:
  - remaining计算
  - 截断逻辑(actualAmount = min(amount, remaining))
  - 达上限返回0
  - 部分发放日志
  → 完整可实现

dailyReset: 3行伪代码，包含:
  - player获取
  - intelPointsToday=0
  → 缺少触发机制、并发控制、在线玩家处理
  → 不可直接实现
```

**影响**:

1. **"完整伪代码"声明不成立**: R11声称提供了"完整约束逻辑伪代码"，但dailyReset仅为骨架代码——addIntelPoints的完整度(40行)与dailyReset的完整度(3行)形成鲜明对比
2. **跨日场景依赖未定义行为**: R11展示的跨日场景(23:50-0:10)假设"0:00自动重置"，但该机制未在伪代码中定义——跨日场景的正确性依赖于实现者自行选择重置方案
3. **并发风险未评估**: 若使用服务器定时重置(方案A)，0:00时可能有玩家正在处理事件，dailyReset与事件处理的并发控制未定义

**建议**:

1. 补全dailyReset伪代码: 增加触发机制(推荐方案C: 服务器定时+登录时检查)
2. 增加并发控制说明: 使用数据库锁或原子操作确保intelPointsToday的读写一致性
3. 在跨日场景中明确标注所依赖的重置方案

### CH-MAP-R11-06 极端周期分析缺少N=1最简场景 [P2]

**质疑流程**: MAP-F12-03-R11 间歇性重度玩家极端周期分析
**追问**: "这个边界对吗？" — 周期矩阵是否覆盖最简场景？

**问题**:

R11的周期模式矩阵(MAP-F12-03-R11第733-738行)覆盖N=2/4/8/12，但缺少N=1:

```
N=1: 1周活跃 + 1周休息 = 2周周期(最简间歇模式)
  活跃周: +8.5 x 7 = +59.5, 声望50->100(3天即满)
  休息周: 第1天豁免, 第2-7天-1/天 = -6, 声望100->94
  周期净变化: +59.5 - 6 = +53.5
  周期平均: +53.5/14天 = +3.82/天
  声望波动范围: 94-100(波动6点)

  用公式验证: Delta = 60.5 - 7x1 = 53(非53.5)
  偏差原因: 公式Delta = 59.5 - 6 - 7x(N-1)中，N=1时为59.5-6-0=53.5
  公式应为: Delta = 59.5 - 6 - 7x(N-1) = 59.5 - 6 + 7 - 7N = 60.5 - 7N
  N=1: Delta = 60.5 - 7 = 53.5 ✓(与详细推导一致)
```

R11的公式Delta=60.5-7N在N=1时给出53.5(正确)，但周期模式矩阵中缺少该行。

**影响**: N=1是最频繁的间歇模式(每2周活跃1周)，也是声望最稳定的模式(波动仅6点)。它的缺失使矩阵不完整——读者无法直观看到"最佳间歇场景"的声望行为。

**建议**: 在周期模式矩阵中增加N=1行:

| 模式 | 活跃期 | 休息期 | 周期总长 | 活跃周获取 | 休息周衰减 | 周期净变化 | 平均/天 | 声望波动范围 |
|------|--------|--------|---------|-----------|-----------|----------|--------|------------|
| N=1 | 1周 | 1周 | 2周 | +59.5 | -6 | +53.5 | +3.82 | **94-100** |

### CH-MAP-R11-08 声望归零的精确N值未计算 [P2]

**质疑流程**: MAP-F12-03-R11 间歇性重度玩家极端周期分析
**追问**: "然后呢？" — N多大时声望会降至0？

**问题**:

R11给出了临界点N=8.64(周期净变化=0)和安全范围N<=12，并声称:

```
R11(第810行): "N>12: 声望可能降至0(不建议)"
```

但未给出声望归零的精确N值。独立推导:

```
假设起始声望=100(活跃周末)
休息期总衰减: 6(第1天豁免) + 7x(N-1)(第2-N天衰减) = 7N - 1
声望归零条件: 100 - (7N - 1) <= 0
  101 <= 7N
  N >= 101/7 = 14.43

结论: N>=15时，声望在休息期末降至0或以下

验证:
  N=14: 100 - (6 + 7x13) = 100 - 97 = 3(未归零)
  N=15: 100 - (6 + 7x14) = 100 - 104 = -4(归零)

R11声称"N>12: 声望可能降至0"偏保守:
  N=12: 声望最低17(安全)
  N=13: 声望最低10(安全)
  N=14: 声望最低3(安全但极低)
  N=15: 声望归零
```

**影响**: R11的安全范围定义(N<=12)偏保守——实际上N=13和N=14时声望仍>0。但"声望可能降至0"的表述过于模糊，应给出精确阈值。

**建议**: 将声望归零的精确阈值N>=15加入分析，并在平衡性结论中标注: "N<=14时声望不会降至0，N>=15时声望在休息期末可能归零"。

## R11总评

### R11核心成就

R11对R10全部9项质疑均有对应产出(覆盖率100%)，其中R10唯一的P1(dailyLimit约束逻辑)被实质性解决。具体改进:

1. **dailyLimit约束逻辑闭环(MAP-F12-02-R11a)**: addIntelPoints伪代码完整(min(amount, remaining)截断机制+日志记录)，约束时机(每次事件处理时实时检查)和范围(所有情报值获取来源)明确定义。多登录场景从R10的"总情报值8"修正为"总情报值5"。3个边界场景(截断/重置/跨日)覆盖了主要异常路径。这是R11最重要的改进——R10最后一个P1问题被实质性解决。

2. **期望值表精确化(MAP-F12-02-R11)**: 使用精确分数(P(W)=1/3而非0.67/0.33)求解，附完整推导过程。临界奖励公式显式提供(R_critical = L_win + (P(L)/P(W))xL_lose + V_T/P(W))。V_T敏感性分析揭示了低胜率场景对令牌价值的20倍放大效应。4个实际决策场景的三方校验均通过(独立验算确认)。

4. **间歇性重度玩家极端周期(MAP-F12-03-R11)**: 周期模式矩阵从N=2扩展为N=2/4/8/12四档，临界点N=8.64的计算正确(Delta=60.5-7N=0)，平衡性结论适用范围已标注(N<=12为安全)。周期净变化公式(Delta=60.5-7N)清晰可验证。

5. **术语统一+替换策略优化(MAP-F12-02-R11b)**: 术语标准已建立(事件队列/待处理事件/情报值)，FIFO替换策略升级为最低价值优先替换(Lowest Value First Replacement)，替换通知从"可选"改为"默认行为"。事件价值定义清晰(普通=1/遗迹/阵营=2/精英=3)。
