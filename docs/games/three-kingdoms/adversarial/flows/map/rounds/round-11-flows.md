### MAP-F12-02-R11 期望值极端值表数值修正 + 三方交叉校验 {#map-f12-02-r11}

> R11核心目标#1: 数值自洽性闭环(紧急)
> CH-MAP-R10-01: 期望值极端值验证表奖励阈值存在多处计算错误，Lv70带3000兵场景偏差+165(R11精确计算)
> 修正原则: 使用公式精确求解(非近似估算)，附完整推导过程，执行三方交叉校验(公式/推算/表格三方一致)

#### 一、问题定位

R10的期望值公式修正方向正确(移除沉没成本)，但极端值表的奖励阈值存在计算错误:

```
R10声称的阈值 vs R11精确阈值(使用精确分数求解):
  Lv70带3000兵/33%: R10声称>2424，R11精确值>2259(含V_T=3)，偏差+165
  Lv70带5000兵/50%: R10声称>2750，R11精确值>2756(含V_T=3)，偏差-6(可忽略)
  Lv70带1000兵/5%: R10声称>4200，R11精确值>4210(含V_T=3)，偏差-10(可忽略)
  Lv70带500兵/5%: R10声称>2150，R11精确值>2135(含V_T=3)，偏差+15(轻微)

根本原因: R10使用近似估算而非公式精确求解。3000兵/33%场景偏差最大(+165)，
          源于R10使用0.67/0.33≈2.03(近似)而非(2/3)/(1/3)=2(精确)
```

#### 二、公式精确定义

**期望值公式(R11精确定义)**:

```
E(重试 | 首次已失败) = P(W) x (R - L_win) + P(L) x (-L_lose) - V_T

其中:
  P(W) = 战斗胜率
  P(L) = 1 - P(W)
  R = 战斗奖励(兵力单位)
  L_win = 重试胜利损耗 = S x min(35%, 20% x F/S)
    S = 出征兵力
    F = 山贼战力
  L_lose = 重试失败损耗 = S x 20%
  V_T = 令牌价值(情报值兑换成本)

临界条件(正期望):
  E > 0
  P(W) x (R - L_win) + P(L) x (-L_lose) - V_T > 0
  P(W) x R > P(W) x L_win + P(L) x L_lose + V_T
  R > L_win + (P(L)/P(W)) x L_lose + V_T/P(W)

临界奖励阈值:
  R_critical = L_win + (P(L)/P(W)) x L_lose + V_T/P(W)
```

#### 三、极端值表重新计算(含完整推导过程)

**令牌价值说明**:

```
令牌价值(V_T)的定义:
  - 情报值兑换成本: 10情报值/枚令牌
  - 情报值与兵力的换算: 取决于游戏经济(非固定值)
  - R11处理: 以V_T作为符号保留，提供V_T=0(不含令牌)和V_T=3(含令牌)两个版本
  - V_T=3的来源: Challenger在R10验算中使用的近似值(情报值→兵力的等效换算)
```

  L_win = 5000 x 35% = 1750 (假设F/S > 1.75，取上限35%)
  L_lose = 5000 x 20% = 1000

推导:
  R_critical = L_win + (P(L)/P(W)) x L_lose + V_T/P(W)
             = 1750 + (0.50/0.50) x 1000 + V_T/0.50
             = 1750 + 1000 + 2 x V_T
             = 2750 + 2 x V_T

  V_T=0: R_critical = 2750

  E = 0.50 x (2756 - 1750) + 0.50 x (-1000) - 3

三方校验:

### MAP-F12-02-R11a dailyLimit约束逻辑定义 + 多登录场景修正 {#map-f12-02-r11a}

> R11核心目标#1: 数值自洽性闭环
> CH-MAP-R10-02: 多次登录场景总情报值8>dailyLimit=5，dailyLimit约束逻辑未定义
> 修正原则: 明确约束时机(每次事件处理时)和范围(所有情报值获取)，修正多登录场景示例，增加约束逻辑伪代码

#### 一、问题定位

R10解绑了loginEventQueueSize和dailyLimit，但未定义dailyLimit的约束逻辑:

```
R10多登录场景:
  早上登录: 处理5个离线事件 → +5情报值
  中午登录: 处理2个新事件 → +2情报值
  晚上登录: 处理1个事件 → +1情报值
  累计: 8情报值(超过dailyLimit=5)

R10结论: "两者独立，多次登录场景不受loginEventQueueSize限制"
问题: dailyLimit=5的目的是"限制每日情报值获取速率"，但场景中实际获取8
```

#### 二、dailyLimit约束逻辑定义

**参数定义(R11完整版)**:

```
参数名: intelPoints.dailyLimit
类型: integer
默认值: 5
目的: 限制每日情报值获取总量(经济平衡)
约束时机: 每次情报值获取时实时检查
约束范围: 所有情报值获取来源(离线事件+在线事件+任务奖励等)
重置时机: 每日0:00重置计数器
```

**约束逻辑伪代码(R11新增)**:

```
// 数据结构
player.intelPointsToday: integer  // 今日已获取情报值(每日0:00重置为0)
player.intelPoints: integer       // 情报值总量(永久累计)

// 情报值获取函数(所有来源统一入口)
function addIntelPoints(playerId, source, amount):
  player = getPlayer(playerId)

  // 步骤1: 计算实际可获取量(受dailyLimit约束)
  remaining = dailyLimit - player.intelPointsToday
  actualAmount = min(amount, remaining)

  // 步骤2: 如果已达上限，返回0(不发放)
  if actualAmount <= 0:
    logDailyLimitReached(playerId, source, amount)
    return 0

  // 步骤3: 发放情报值
  player.intelPointsToday += actualAmount
  player.intelPoints += actualAmount

  // 步骤4: 如果部分发放，记录截断
  if actualAmount < amount:
    logPartialGrant(playerId, source, amount, actualAmount)

  return actualAmount

// 事件处理函数(修正版)
function processEvent(playerId, event):
  reward = calculateReward(event)

  // 情报值奖励受dailyLimit约束
  intelReward = addIntelPoints(playerId, event.type, reward.intel)

  // 其他奖励(资源/声望等)不受dailyLimit约束
  addResources(playerId, reward.resources)
  addReputation(playerId, reward.reputation)

  return { intel: intelReward, resources: reward.resources, reputation: reward.reputation }

// 每日重置(0:00触发)
function dailyReset(playerId):
  player = getPlayer(playerId)
  player.intelPointsToday = 0
```

#### 三、修正后多登录场景

**场景: 玩家每日登录3次(早中晚各一次)(R11修正版)**:

```
前提: dailyLimit=5, loginEventQueueSize=5

早上登录:
  离线事件队列: 5个(假设离线1天)
  处理事件1: +1情报值 → intelPointsToday=1(≤5) → 发放
  处理事件2: +1情报值 → intelPointsToday=2(≤5) → 发放
  处理事件3: +1情报值 → intelPointsToday=3(≤5) → 发放
  处理事件4: +1情报值 → intelPointsToday=4(≤5) → 发放
  处理事件5: +1情报值 → intelPointsToday=5(=5) → 发放
  累计: 5情报值(dailyLimit已满)

中午登录:
  离线事件队列: 0个(早上已清空)
  在线触发事件: 2个

#### 一、主要指标重新评估

**R10指标问题**:

```
R10主要指标: 战斗参与率 = 每日参与战斗的玩家数 / DAU
判定标准: k=1.75组的战斗参与率显著高于k=1.0组(p<0.05)

逻辑方向疑问:
  k值越高 → 胜利损耗越大 → 理论上抑制战斗参与
  k=1.0(无额外惩罚): 战斗代价最低 → 参与率应最高
  k=1.75(有惩罚): 战斗代价较高 → 参与率应较低

  R10判定: k=1.75组参与率 > k=1.0组参与率
  这与直觉相反——惩罚更重的组参与率更高?
```

**指标重新评估(R11)**:

| 候选指标 | 定义 | 类型 | 优势 | 劣势 |
|---------|------|------|------|------|
| 战斗参与率(R10) | 参与战斗玩家数/DAU | 比例型 | 简单直观 | 无法区分有意义/无意义战斗 |
| **有意义战斗参与率** | 参与胜率>30%战斗的玩家数/DAU | 比例型 | 排除低胜率出征 | 需定义"有意义"阈值 |
| 玩家留存率 | 7日留存率 | 比例型 | 反映长期体验 | 反馈周期长 |
| 平均战斗次数 | 总战斗次数/DAU | 计数型 | 反映活跃度 | 不反映决策质量 |

**R11推荐主要指标: 有意义战斗参与率**

```
指标名称: 有意义战斗参与率
英文: Meaningful Battle Participation Rate (MBPR)
定义: 每日参与"胜率>30%"战斗的独立玩家数 / 当日DAU
类型: 比例型(0-100%)
方向说明:
  - 预期k=1.75组MBPR > k=1.0组MBPR
  - 理论依据: k=1.75通过提高低胜率战斗的代价，迫使玩家更谨慎选择目标
  - 效果: 低胜率(<30%)出征减少，高胜率(>30%)出征保持或增加
  - 净效果: MBPR可能上升(因为"无意义战斗"被抑制，"有意义战斗"不受影响)

判定标准:
  - H1成立: k=1.75组MBPR显著高于k=1.0组(p<0.05, Cohen's h >= 0.2)
  - H0成立: 两组MBPR无显著差异
  - H1被推翻: k=1.0组MBPR显著高于k=1.75组

与R10指标的对比:
  R10: 战斗参与率(不区分胜率) → 无法解释k=1.75组参与率更高的原因
  R11: 有意义战斗参与率(排除低胜率) → 直接反映k值对战斗决策质量的影响
```

**次要指标(R11更新)**:

| 指标 | 定义 | 方向说明 |
|------|------|---------|
| 低兵力出征频率 | 胜率<30%的出征次数/总出征次数 | 预期: k=1.75组 < k=1.0组(惩罚抑制低胜率出征) |

| 胜利代价满意度 | 问卷: "胜利后的损耗是否合理"(1-5分) | 预期: k=1.75组 >= 3.5分 |

参数定义:
  N = 休息周数(与1周活跃期的比值)
  周期 = 1周活跃 + N周休息 = (N+1)周

参数范围:
  N=1: 每2周活跃1周(最频繁) → 声望波动93-100
  N=2: 每3周活跃1周(默认) → 声望波动87-100
  N=4: 每5周活跃1周(中等) → 声望波动73-100
  N=8: 每9周活跃1周(低频) → 声望波动45-100
  N=12: 每13周活跃1周(临界) → 声望波动17-100
  N>12: 声望可能降至0(不建议)

周期净变化公式:
  Δ = 59.5 - 6 - 7x(N-1) = 59.5 - 6 - 7N + 7 = 60.5 - 7N

临界条件(Δ=0):
  60.5 - 7N = 0
  N = 60.5/7 ≈ 8.64

结论:
  N < 8.64: 周期净变化为正，声望长期稳定(波动范围>45)
  N ≈ 8.64: 周期净变化为0，声望在波动中维持
  N > 8.64: 周期净变化为负，声望长期下降(但仍不会降至0，除非N极大)
```

#### 三、平衡性结论适用范围(R11标注)

```
平衡性结论(R11标注版):

结论1: 间歇性重度玩家声望永远不会降至0
  适用范围: N <= 12(每13周活跃1周)
  依据: 即使N=12，声望最低仍有17(远高于0)
  边界: N极大时(如N=50)，活跃周获取(+59.5)可能不足以恢复声望

结论2: 间歇性重度玩家声望波动范围
  N=2: 87-100(波动13点)
  N=4: 73-100(波动27点)
  N=8: 45-100(波动55点)
  N=12: 17-100(波动83点)
  规律: N越大，波动范围越大，声望稳定性越差

结论3: 周期净变化的临界点
  N < 8.64: 周期净变化为正(声望长期上升趋势)
  N ≈ 8.64: 周期净变化为零(声望长期稳定)
  N > 8.64: 周期净变化为负(声望长期下降趋势)
```

#### 二、FIFO替换策略优化

**R10 FIFO策略问题**:

```
R10策略: 所有事件平等，不区分价值高低
问题: 高价值事件(精英山贼)可能被低价值事件(普通山贼)替换
R10评估: "这是FIFO策略的固有特性，非bug"
缺失: 未评估玩家体验影响，未考虑替代策略
```

**替换策略对比(R11)**:

| 策略 | 规则 | 优势 | 劣势 |
|------|------|------|------|
| FIFO(R10) | 替换最旧事件 | 简单公平，无价值判断 | 高价值事件可能丢失 |
| **最低价值优先(R11推荐)** | 替换价值最低的事件 | 保护高价值事件 | 需定义事件价值 |
| 混合策略 | 优先替换低价值，同价值FIFO | 兼顾公平和保护 | 实现复杂度中等 |

**最低价值优先替换策略(R11推荐)**:

```
策略名称: 最低价值优先替换(Lowest Value First Replacement)

事件价值定义:
  普通山贼: 价值=1(低)
  精英山贼: 价值=3(高)
  遗迹探索: 价值=2(中)
  阵营事件: 价值=2(中)

替换规则:
  1. 当事件队列已满且有新事件到达时:
     - 找出队列中价值最低的事件
     - 如果新事件价值 > 最低价值事件: 替换最低价值事件
     - 如果新事件价值 <= 最低价值事件: 替换最旧的最低价值事件(FIFO)
     - 如果多个事件同为最低价值: 替换其中最旧的(FIFO)

  2. 替换通知(默认行为):
     - 每次替换时记录被替换事件
     - 玩家登录时显示: "您有X个事件因队列已满被替换(含Y个高价值事件)"
     - 高价值事件被替换时额外提示: "其中包含精英山贼事件"

伪代码:
  function addEventToQueue(playerId, event):
    queue = player.eventQueue

    if queue.length >= loginEventQueueSize:
      // 最低价值优先替换
      lowestValueIdx = findLowestValueIndex(queue)
      lowestValueEvent = queue[lowestValueIdx]

      if event.value > lowestValueEvent.value:
        // 新事件价值更高，替换最低价值事件
        removedEvent = queue.splice(lowestValueIdx, 1)[0]
      else:
        // 新事件价值不高于队列中最低价值，替换最旧的最低价值
        oldestLowestIdx = findOldestLowestValueIndex(queue)
        removedEvent = queue.splice(oldestLowestIdx, 1)[0]

      // 记录替换(默认行为)
      logEventReplacement(playerId, removedEvent, event)
      addToReplacementSummary(playerId, removedEvent)

    queue.push(event)

  function findLowestValueIndex(queue):
    minIdx = 0
    for i in 1..queue.length-1:
      if queue[i].value < queue[minIdx].value:
        minIdx = i
    return minIdx
```

**替换通知(R11默认行为)**:

```
R10: 替换通知为"可选优化"
R11: 替换通知为"默认行为"

理由:
  1. 玩家有权知道哪些事件被替换
  2. 高价值事件被替换时，玩家可能采取补救措施(如缩短离线时间)
  3. 透明度提升玩家信任感

通知实现:
  - 替换时: 记录到player.replacementLog(最近10条)
  - 登录时: 在事件队列面板顶部显示替换摘要
  - 格式: "离线期间有X个事件被替换(普通x个, 精英x个)"
  - 详情: 点击可查看被替换事件的具体信息
```
