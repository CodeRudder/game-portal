### MAP-F09-02-R7a 遗迹探索Lv10行数据表修正 {#map-f09-02-r7a}

> R7核心目标#1: P0硬伤清零
> CH-MAP-R6-01: Lv10行概率数据表与公式完全矛盾(partial/success互换)
> 修正原则: 以公式为准，表为示意。修正后必须附带逐行推导过程。

#### 一、问题定位

**R6数据表(Lv10行)错误**:

| 攻城术等级 | 失败概率 | 部分成功概率 | 成功概率 | 状态 |
|-----------|---------|------------|---------|------|
| Lv10 (+20%) | 10% | **60%** | **30%** | R6数据表(错误) |

**R6公式计算结果**:

```
fail_threshold = max(5%, 30% - 20%) = 10%
partial_threshold = max(10%+5%, 50% - 20%×0.5) = max(15%, 40%) = 40%

失败概率 = roll < 10% = 10%         ← 数据表正确
部分成功概率 = 10% <= roll < 40% = 30%  ← 数据表写60%，错误!
成功概率 = roll >= 40% = 60%         ← 数据表写30%，错误!
```

**结论**: Lv10行的部分成功概率(30%)和成功概率(60%)被互换。这是人工抄写/计算错误。

#### 二、修正方案

**修正后(R7)完整科技加成效果表**:

| 攻城术等级 | 科技加成 | 失败阈值 | 部分成功阈值 | 失败概率 | 部分成功概率 | 成功概率 | 推导过程 |
|-----------|---------|---------|------------|---------|------------|---------|---------|
| Lv0 | +0% | 30% | 50% | 30% | 20% | 50% | fail=max(5%,30%-0%)=30%; partial=max(35%,50%-0%)=50% |
| Lv3 | +6% | 24% | 47% | 24% | 23% | 53% | fail=max(5%,30%-6%)=24%; partial=max(29%,50%-3%)=47% |
| Lv5 | +10% | 20% | 45% | 20% | 25% | 55% | fail=max(5%,30%-10%)=20%; partial=max(25%,50%-5%)=45% |
| **Lv10** | **+20%** | **10%** | **40%** | **10%** | **30%** | **60%** | fail=max(5%,30%-20%)=10%; partial=max(15%,50%-10%)=40% |

> **开发者注释**: 以公式为准，表为示意。所有概率值均由公式推导，实现时请使用公式而非查表。

- R6声称"优化前后在Lv10结果相同"——**错误**

### MAP-F09-02-R7b 山贼胜利损耗公式溢出截断 {#map-f09-02-r7b}

> R7核心目标#1: P0硬伤清零
> CH-MAP-R6-02: 胜利损耗公式在低兵力场景下可产生126%+损耗率
> 修正原则: 胜利损耗不超过出征兵力的50%

#### 一、问题定位

**R6胜利损耗公式**:
```
预计兵力损失(胜利时) = 出征兵力 × max(5%, 20% × (山贼战力 / 玩家总兵力))
```

**溢出场景验证**:

| 场景 | 山贼战力 | 出征兵力 | 比值 | 损耗率 | 损耗值 | 问题 |
|------|---------|---------|------|--------|-------|------|
| Lv70带1000兵打山贼王 | 6300 | 1000 | 6.3 | max(5%,126%)=**126%** | **1260** | 超过出征兵力! |
| Lv70带500兵打山贼王 | 6300 | 500 | 12.6 | max(5%,252%)=**252%** | **1260** | 2.5倍出征兵力! |
| Lv50带1000兵打精锐山贼 | 3500 | 1000 | 3.5 | max(5%,70%)=**70%** | 700 | 虽未溢出但损耗极高 |
| Lv70带3000兵打山贼王 | 6300 | 3000 | 2.1 | max(5%,42%)=**42%** | 1260 | 接近截断线 |

**核心问题**: `山贼战力/玩家总兵力`比值在兵力悬殊时可远超5.0，导致损耗率超过100%。

**更严重的体验问题**: 失败损耗固定为20%，但胜利损耗可达126%+——"故意输"比"侥幸赢"损失更少，这是数值设计的灾难。

#### 二、修正方案

**修正后(R7)胜利损耗公式**:
```
预计兵力损失(胜利时) = 出征兵力 × min(50%, max(5%, 20% × (山贼战力 / 玩家总兵力)))
```

**修正点**: 在原公式外层增加 `min(50%, ...)` 上限截断。

**修正前后公式对比**:

| 组成部分 | R6(修正前) | R7(修正后) | 变化说明 |
|---------|-----------|-----------|---------|
| 下限保护 | max(5%, ...) | max(5%, ...) | 不变: 胜利至少损失5% |
| 损耗计算 | 20% × (山贼战力/玩家兵力) | 20% × (山贼战力/玩家兵力) | 不变: 比值越高损耗越大 |
| 上限截断 | **无** | **min(50%, ...)** | **新增: 胜利损耗不超过50%** |
| 失败损耗 | 出征兵力 × 20% | 出征兵力 × 20% | 不变 |

**设计理由**:
- 50%上限确保胜利损耗永远不会超过出征兵力的一半
- 即使在最极端的兵力悬殊下，胜利也比失败(20%固定损耗)代价更高但有上限
- 玩家不会因为"侥幸赢了"而遭受比"故意输"更惨的损失(50% vs 20%，赢的代价更高但有合理上限)

#### 三、极端值验证

| 测试用例 | 山贼战力 | 出征兵力 | R6损耗率 | R6损耗值 | R7损耗率 | R7损耗值 | R7改进 |
|---------|---------|---------|---------|---------|---------|---------|--------|
| Lv10带1000兵打弱匪 | 300 | 1000 | max(5%,6%)=6% | 60 | 6% | 60 | 不变(未触发截断) |
| Lv50带5000兵打精锐 | 3500 | 5000 | max(5%,14%)=14% | 700 | 14% | 700 | 不变(未触发截断) |
| Lv70带10000兵打山贼王 | 6300 | 10000 | max(5%,12.6%)=12.6% | 1260 | 12.6% | 1260 | 不变(未触发截断) |
| Lv70带3000兵打山贼王 | 6300 | 3000 | max(5%,42%)=42% | 1260 | 42% | 1260 | 不变(未触发截断) |
| **Lv70带1000兵打山贼王** | 6300 | 1000 | max(5%,**126%**)=**126%** | **1260** | min(50%,126%)=**50%** | **500** | **截断生效** |
| **Lv70带500兵打山贼王** | 6300 | 500 | max(5%,**252%**)=**252%** | **1260** | min(50%,252%)=**50%** | **250** | **截断生效** |

### MAP-F12-02-R7 情报值系统最小化定义 {#map-f12-02-r7}

#### 一、情报值系统参数定义

**数据模型**:
```
player.intelPoints = {
  current: 0,          // 当前情报值
  maxCap: 100,         // 上限100
  dailyGained: 0,      // 今日已获取量
  dailyLimit: 5,       // 每日获取上限
  lastResetDate: null   // 上次重置日期(用于每日重算)
}
```

**参数说明**:

| 参数 | 值 | 说明 |
|------|---|------|
| 上限(maxCap) | 100 | 防止长期玩家无限积累。达到上限后不再获取 |
| 每日获取上限(dailyLimit) | 5 | 山贼快速处理每日最多触发5次(离线事件上限5个) |
| 单次获取量 | +1 | 每次山贼快速处理"绕行"分支获得1点 |
| 兑换门槛 | 10点 | 10点情报值可兑换1次"战斗重试机会" |
| 兑换内容 | 战斗重试令牌 | 可用于山贼战斗失败后免费重试1次(不消耗兵力) |

#### 二、获取流程

**山贼快速处理获取情报值(R7修订)**:
```
Step: 快速处理"绕行"分支
  1. 检查情报值上限:
     if player.intelPoints.current >= player.intelPoints.maxCap:
       → 提示: "情报值已达上限(100)，无法继续获取"
       → 仍然获得铜钱x100(保底)，但情报值不增加
       → 跳过情报值增加步骤

  2. 检查每日获取上限:
     if player.intelPoints.dailyGained >= player.intelPoints.dailyLimit:
       → 提示: "今日情报值获取已达上限(5点)"
       → 仍然获得铜钱x100(保底)，但情报值不增加
       → 跳过情报值增加步骤

  3. 增加情报值:
     player.intelPoints.current += 1
     player.intelPoints.dailyGained += 1
     → 提示: "情报值+1 (当前: X/100, 今日已获: Y/5)"
```

**每日重置逻辑**:
```
// 登录时检查并重置每日计数
function resetDailyIntelPoints():
  today = formatDate(now())
  if player.intelPoints.lastResetDate != today:
    player.intelPoints.dailyGained = 0
    player.intelPoints.lastResetDate = today
```

### MAP-F01-04-R7 一键收取超上限边界处理 {#map-f01-04-r7}

> R7核心目标#2: 边界条件补全
> CH-MAP-R6-05: 离线期间资源已超上限时actualCollect为负数的BUG
> 修正原则: 已超上限时跳过收取+提示，确认弹窗分资源标注状态

#### 一、问题定位

**R6溢出处理公式**:
```
if player.resources[type] + collectible[type] > storageCap[type]:
  actualCollect = storageCap[type] - player.resources[type]  // 若resources>cap则为负数!
  overflow = collectible[type] - actualCollect
  player.resources[type] = storageCap[type]
```

**问题场景**: 当`player.resources[type] > storageCap[type]`时(仓库降级/活动奖励注入等)，`actualCollect`为负数。

#### 二、修正方案

**修正后(R7)溢出处理逻辑**:
```
一键收取资源处理(每种资源类型独立处理):

Step 1: 已超上限检查
  if player.resources[type] >= storageCap[type]:
    → 收取状态: SKIP(跳过)
    → 该资源不收取
    → 提示: "{资源名}已达上限({storageCap})，无法收取。建议消费后重试"

Step 2: 正常溢出检查
  elif player.resources[type] + collectible[type] > storageCap[type]:
    → 收取状态: PARTIAL(部分收取)
    → actualCollect = storageCap[type] - player.resources[type]
    → overflow = collectible[type] - actualCollect
    → player.resources[type] = storageCap[type]
    → 提示: "{资源名}收取{actualCollect}，溢出{overflow}已丢弃"

Step 3: 正常收取
  else:
    → 收取状态: FULL(全额收取)
    → player.resources[type] += collectible[type]
    → 提示: "{资源名}收取{collectible[type]}"
```

#### 三、确认弹窗修订

**修正后(R7)一键收取确认弹窗**:
```
弹窗标题: "一键收取确认"

各资源收取状态(独立标注):

### MAP-F06-07-R7 内应冷却优先级规则独立化 {#map-f06-07-r7}

> R7核心目标#2: 边界条件补全
> CH-MAP-R6-03: 内应暴露冷却状态优先级规则仅作为附注，未独立定义
> 修正原则: 判定优先级提升为独立规则，冷却结束增加过渡处理

#### 一、问题定位

**R6状态定义**:
```
状态A: insiderLetter >= 1 && !insiderExposed → 可点击
状态B: insiderExposed = true → 置灰+冷却倒计时

**问题**:
1. "暴露冷却优先于信件不足"的规则仅在状态C的附注中出现，不是独立判定规则
2. 状态B的条件未检查`insiderLetter`数量，当`insiderExposed=true && insiderLetter=0`时存在歧义
3. 冷却结束瞬间(`remaining=0`)，`insiderExposed`自动变为`false`，UI从"暴露冷却"跳变到"需要内应信"

#### 二、修正方案

```
内应卡片状态判定(独立优先级规则):

  // 优先级规则(独立定义，非附注)
  // 判定顺序: 暴露 → 有信 → 其他
  // 原因: 暴露是临时状态(24h后解除)，应优先告知玩家

#### 二、修正方案: 冷却状态管理器

**cooldownManager设计(R7)**:
```
cooldownManager = {
  // 存储所有城池的冷却状态
  cooldowns: Map<cityId, CooldownState>,

  // 冷却状态结构
  CooldownState: {
    cityId: string,
    exposed: boolean,
    startAt: timestamp,
    duration: 24h(86400s),
    endAt: timestamp,
    remaining: number,  // 实时计算
    status: "active" | "ended" | "none"
  },

  // 方法
  getCooldown(cityId): CooldownState,
  setCooldown(cityId, startAt, duration): void,
  clearCooldown(cityId): void,
  refreshAll(): void,  // 刷新所有冷却状态
  onStateChanged(callback): void  // 注册状态变化监听
}
```

**核心方法实现**:
```
// 获取冷却状态(统一入口)
function getCooldown(cityId):
  state = cooldowns.get(cityId)
  if !state || !state.exposed:
    return { status: "none", remaining: 0 }

  remaining = state.endAt - now()

  if remaining <= 0:
    // 冷却已结束，自动清理
    state.exposed = false
    state.status = "ended"
    emit("cooldownStateChanged", { cityId, status: "ended" })
    return { status: "ended", remaining: 0 }

  state.remaining = remaining
  state.status = "active"
  return state

// 设置冷却
function setCooldown(cityId, startAt, duration):
  state = {
    cityId: cityId,
    exposed: true,
    startAt: startAt,
    duration: duration,
    endAt: startAt + duration,
    remaining: duration,
    status: "active"
  }
  cooldowns.set(cityId, state)
  emit("cooldownStateChanged", { cityId, status: "active", remaining: duration })

// 清除冷却
function clearCooldown(cityId):
  state = cooldowns.get(cityId)
  if state:
    state.exposed = false
    state.status = "ended"
    cooldowns.delete(cityId)
    emit("cooldownStateChanged", { cityId, status: "ended" })
```

// 服务器定时任务: 每日00:00执行
function dailyReputationDecay():

Step 1: 玩家点击"一键收取"按钮
  → 弹出收取确认面板

Step 2: 玩家操作
  情况A: 点击"确认收取"
    → 触发冷却: ui.overviewPanel.collectCooldownEnd = now() + 30s
    → 执行收取逻辑
    → "一键收取"按钮置灰，显示倒计时"30s"

  情况B: 点击"取消"
    → 不触发冷却
    → 关闭确认面板
    → "一键收取"按钮保持可点击状态

Step 3: 冷却期间
  → 按钮置灰，显示倒计时"XXs"
  → 每秒更新倒计时
  → 点击按钮弹出提示: "收取冷却中，请等待XX秒"
  → 页面刷新后冷却继续(数据持久化)

Step 4: 冷却结束
  → 按钮恢复可点击状态
  → 移除倒计时
```

### MAP-F01-04-R7c "正在被攻击"精确定义 {#map-f01-04-r7c}

> R7目标: P2优化项补充
> CH-MAP-R6-07: "正在被攻击的领土"排除条件定义不清

#### 一、精确定义

**修正后(R7)排除条件**:
```
一键收取排除条件:
  仅排除: territory.underSiege = true
    → 含义: 其他玩家正在对该领土发起攻城战(攻城动画/等待期)

  不排除:
    - territory.underEvent = true(正在遭受事件，如流寇入侵)
    - territory.inCooldown = true(占领冷却期内，产出50%但可收取)
    - territory.underNPCAttack = true(正在被NPC攻击，如阵营冲突)
```

**判定逻辑**:
```
function isCollectible(territory):
  // 必须是己方领土
  if territory.owner != player.id:
    return false

  // 排除正在被攻城的领土
  if territory.underSiege:
    return false

  // 其他状态不排除
  return true
```

**数据变更**:

| 数据 | 变更前(R6) | 变更后(R7) | 来源 |
|------|-----------|-----------|------|
| 一键收取排除条件 | "正在被攻击"(口语化) | `territory.underSiege = true`(精确定义) | CH-MAP-R6-07 |
