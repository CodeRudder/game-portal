### MAP-F06-07 内应信消费完整流程 {#map-f06-07}

> R5核心目标#1: 闭合内应信生命周期（CH-MAP-R4-01）
> 补充R4缺失的消费侧流程，使内应信从"只进不出"变为完整闭环

**玩家视角**: 背包中有内应信 → 攻城时选择"内应"策略 → 内应信消耗 → 成功率+20% → 结算

**前置条件**:
- 玩家背包中拥有内应信道具 ≥ 1封
- 目标城池与己方领土相邻（攻城基础条件）
- 当日攻城次数 < 3
- 内应信未过期（无保质期限制，永久有效）

**流程步骤**:

1. [前置] 玩家点击城池类领土，点击"攻城"按钮（MAP-F06 Step 1-3 通过）
2. [UI反馈] 攻城确认弹窗显示策略选择区域，4张策略卡片:
   - 强攻 | 围困 | 夜袭 | 内应
   - **内应卡片状态判定**:
     - 若背包内应信 ≥ 1: 卡片可点击，金色边框高亮，显示"持有×N"
     - 若背包内应信 = 0: 卡片置灰，显示"需要内应信×1"，点击后弹出获取途径提示
3. [玩家操作] 点击"内应"策略卡片
4. [系统校验] 检查内应信消耗条件:
   - a. 背包内应信数量 ≥ 1 → 通过
   - b. 背包内应信数量 = 0 → 校验失败，提示"内应信不足，请先获取"
5. [系统计算] 内应策略效果预估:
   ```
   内应策略成功率 = 基础成功率 + 20%(内应加成) + 科技加成(夜战等级×2%)
   基础成功率 = min(95%, max(5%, (我方战力/敌方战力)×50% + 地形修正))

   内应策略奖励倍率 = ×1.5
   内应策略时间倍率 = ×1.0（与基础攻城时间相同）
   内应策略消耗倍率 = ×1.0（与基础攻城消耗相同）
   ```
6. [UI反馈] 更新攻城确认弹窗:
   - 成功率预估: 显示"内应策略成功率: XX%（基础XX% + 内应+20%）"
   - 奖励预估: "奖励倍率: ×1.5（内应策略加成）"
   - 消耗明细: "内应信×1（从背包扣除）" + 兵力/粮草消耗
   - 成功率颜色编码(翠绿>80%/金色60-80%/琥珀橙40-60%/赤红<40%)
7. [玩家操作] 点击"确认攻城"
8. [系统执行-扣取] 从背包扣取内应信×1:
   ```
   player.bag.insiderLetter -= 1
   ```
9. [系统执行-攻城] 按内应策略参数执行攻城:
   - 攻城时间 = 基础时间 × 1.0
   - 消耗 = 基础消耗 × 1.0
   - 城防衰减速率按内应策略计算（内应策略下城防衰减速度 = 基础×1.2，因城内有人接应）
10. [系统结算] 攻城结果判定:

**成功分支**:
11a. [系统计算] 攻城成功:
    - 城防归零，城池归己方
    - 奖励 = 基础奖励 × 1.5
    - 内应特殊效果: 城防完整保留（不触发强攻的城防损坏debuff）
12a. [UI反馈] 攻城成功弹窗:
    - 胜利动画 + "内应成功，城池已归降！"
    - 奖励明细: 资源×1.5 + 道具掉落
    - "内应"标签高亮，区别于其他策略的胜利弹窗
13a. [数据变更] 奖励发放，城池归属变更

**失败分支**:
11b. [系统计算] 攻城失败:
    - 损失30%出征兵力（与标准攻城失败一致）
    - 内应信已消耗，不返还
    - 额外惩罚: 内应暴露，目标城池24h内对该策略免疫（再次使用内应策略成功率归零）
12b. [UI反馈] 攻城失败弹窗:
    - 失败动画 + "内应被识破，计划败露！"
    - 损失明细: 兵力-30% + 内应信×1（已消耗）
    - "内应暴露"debuff提示: "该城池24h内免疫内应策略"
13b. [数据变更] 兵力损失，内应信已扣，debuff生效

#### 一、山贼战斗机制

**触发条件**: 地图事件"山贼"，玩家选择"战斗"分支

**战斗结算公式**:
```
山贼战力 = 玩家总兵力 × 山贼难度系数
山贼难度系数:
  - 玩家等级1~20: ×0.3（弱匪）
  - 玩家等级21~40: ×0.5（悍匪）
  - 玩家等级41~60: ×0.7（精锐山贼）
  - 玩家等级61+: ×0.9（山贼王）

战斗胜率 = min(95%, max(5%,
  (玩家总兵力 / (玩家总兵力 × 山贼难度系数)) × 50%
  + 地形修正(森林+10%/山地-10%/其他0%)
  + 科技加成(攻城术等级×2%)
))

战斗结算 = 自动结算（无需玩家操作）
```

**战斗流程步骤**:

1. [前置] 玩家在山贼事件面板选择"战斗"分支
2. [系统计算] 计算山贼战力和战斗胜率
3. [UI反馈] 显示战斗预估面板:
   - "山贼规模:" + 难度等级（弱匪/悍匪/精锐/山贼王）
   - "战斗胜率:" + XX% + 颜色编码
   - "预计损失:" + XX%兵力（失败时）
   - "预计奖励:" + 资源 + 装备概率 + 内应信概率
4. [玩家操作] 点击"出战"确认 / 点击"放弃"返回事件面板
5. [系统执行] 自动战斗结算（动画持续3秒）
6. [系统判定] 根据胜率随机判定结果

**战斗胜利**:
7a. [UI反馈] 胜利面板:
   - 战斗动画（刀光+击退效果）
   - "山贼已被剿灭！"
   - 奖励明细:
     - 铜钱 × (500 + 玩家等级×50)
     - 粮草 × (200 + 玩家等级×20)
     - 装备掉落（概率见下表）
     - 内应信掉落: 15%概率
8a. [数据变更] 奖励发放

**战斗失败**:
7b. [UI反馈] 失败面板:
   - 战斗动画（败退效果）
   - "山贼过于强大，我军败退！"
   - 损失明细: 兵力损失 ×20%（出征兵力的20%）
   - "下次可提升兵力后再来剿匪"
8b. [数据变更] 兵力损失
9b. [系统规则] 山贼事件标记为"战斗失败"，不可重试（同一事件只能战斗一次）

**装备掉落品质表**:

| 装备品质 | 掉落概率 | 装备等级范围 | 说明 |
|---------|---------|------------|------|
| 白色(普通) | 60% | 玩家等级±5 | 基础属性 |
| 绿色(优秀) | 25% | 玩家等级±3 | 1条附加属性 |
| 蓝色(稀有) | 12% | 玩家等级±2 | 2条附加属性 |

#### 二、遗迹探索机制

**触发条件**: 地图事件"遗迹发现"，玩家选择"探索"分支

**探索结算公式**:
```
探索成功率 = 70%（基础）
  + 科技加成(攻城术等级×1%，因为探索也需要战术眼光)
  + 地形修正(山地+10%/森林+5%/其他0%)

探索结果:
  成功(70%+): 获得完整奖励
  部分成功(30%~70%): 获得50%奖励
  失败(<30%): 无奖励，损失少量兵力
```

**探索流程步骤**:

1. [前置] 玩家在遗迹事件面板选择"探索"分支
2. [系统计算] 计算探索成功率
3. [UI反馈] 显示探索预估面板:
   - "遗迹类型:" + 随机类型（古墓/废墟/密道/藏兵洞）
   - "探索成功率:" + XX% + 颜色编码
   - "预计发现:" + 内应信(25%) + 资源 + 特殊道具
   - "风险:" + 失败时损失XX%兵力
4. [玩家操作] 点击"探索"确认 / 点击"放弃"返回事件面板
5. [系统执行] 探索结算（动画持续5秒，显示探索进度条）

**探索成功**:
6a. [UI反馈] 探索成功面板:
   - 探索动画（发现宝箱/密函效果）
   - "在遗迹中发现了珍贵物品！"
   - 奖励明细:
     - 内应信: 25%概率 ×1
     - 铜钱 × (1000 + 玩家等级×100)
     - 特殊道具（随机1种）: 加速令/迁城令/高级装备图纸
7a. [数据变更] 奖励发放

**探索部分成功**:
6b. [UI反馈] 部成功面板:
   - "遗迹部分坍塌，仅获取部分物品"
   - 奖励明细: 上述奖励的50%（内应信概率降至12.5%）
7b. [数据变更] 奖励发放

**探索失败**:
6c. [UI反馈] 探索失败面板:
   - "遗迹结构不稳，探索失败！"
   - 损失明细: 兵力损失 ×10%（出征兵力的10%）
   - "可提升攻城术科技增加探索成功率"
7c. [数据变更] 兵力损失
8c. [系统规则] 遗迹事件标记为"探索失败"，不可重试

### MAP-F12-01-R5 离线事件回归体验重设计 {#map-f12-01-r5}

> R5核心目标#2: 修复离线回归体验（CH-MAP-R4-02）
> 核心原则: 离线回归应该是"奖励"而非"惩罚"

**玩家视角**: 离线期间事件积累 → 登录后看到"欢迎回来"面板 → 快速处理获得奖励 → 过期也有补偿

**与R4版本(MAP-F12-01)对比**:

| 机制 | R4(旧) | R5(新) | 变化理由 |
|------|--------|--------|---------|
| 过期时间 | 触发后24h | 登录后重新计算24h | 给回归玩家缓冲期 |
| 过期惩罚 | 每个事件独立判定50%流寇骚扰 | 仅最后一个过期事件触发一次 | 避免惩罚叠加 |
| 处理方式 | 逐个处理 / 一键忽略 | 逐个处理 / 一键快速处理 / 一键忽略 | 增加快速处理 |

### MAP-F12-02 离线/在线事件双计数器机制 {#map-f12-02}

> CH-MAP-R4-03: 离线事件累积与在线事件上限冲突

**玩家视角**: 在线时事件上限3个 → 下线后离线事件独立累积(上限5个) → 上线后先处理离线队列 → 恢复在线事件检测

**双计数器规则**:

| 计数器 | 上限 | 触发条件 | 说明 |
|--------|------|---------|------|
| 在线事件计数器 | 3个 | 玩家在线时TE-MAP-05触发 | 正常在线事件 |
| 离线事件计数器 | 5个 | 玩家离线时TE-MAP-08触发 | 离线期间累积 |

**流程步骤**:

1. [系统状态] 玩家在线:
   - 在线事件计数器: activeEvents.length ≤ 3
   - 离线事件计数器: 离线队列不参与在线事件检测
2. [系统状态] 玩家离线:
   - 在线事件计数器: 暂停检测（不消耗在线配额）
   - 离线事件计数器: pendingEvents.length ≤ 5
3. [玩家登录] 系统检测到离线事件队列:
   - 优先处理离线队列（MAP-F12-01-R5流程）
   - 在线事件计数器保持暂停状态
4. [处理完毕] 离线队列清空（全部处理/忽略/过期）:
   - 在线事件计数器恢复检测
   - 重新开始TE-MAP-05（在线事件触发）
5. [边界情况] 玩家处理部分离线事件后退出:
   - 未处理的离线事件保留在队列中
   - 下次登录时继续从上次中断处处理
   - 过期时间按MAP-F12-01-R5规则重新计算

**计数器交互矩阵**:

| 场景 | 在线计数器 | 离线计数器 | 说明 |
|------|-----------|-----------|------|
| 玩家在线，事件<3 | 活跃 | — | 正常触发在线事件 |
| 玩家在线，事件=3 | 暂停触发 | — | 达到上限，不再触发 |
| 玩家离线 | 暂停 | 活跃 | 离线队列累积 |
| 玩家上线，离线队列>0 | 暂停 | 处理中 | 先处理离线队列 |
| 玩家上线，离线队列=0 | 恢复 | — | 恢复在线事件检测 |

**数据变更**:

| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| eventCounter.online | 与离线共用(上限3) | 独立(上限3) | 双计数器 |
| eventCounter.offline | 与在线共用(上限3) | 独立(上限5) | 双计数器 |
| systemState.eventDetection | 统一 | 分离(在线/离线) | 状态管理 |

**后置流程**: → MAP-F09(在线事件触发恢复)

           ↓
内应信消耗完毕 → 玩家需要重新获取

| 参数 | 数值 | 说明 |
