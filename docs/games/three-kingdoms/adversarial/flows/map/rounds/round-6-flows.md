### MAP-F09-02-R6 事件战斗分支机制修正 {#map-f09-02-r6}

> R6核心目标#1: 数值公式修正
> CH-MAP-R5-03(山贼战斗兵力被约掉) + CH-MAP-R5-05(遗迹探索三档判定混乱)

#### 一、山贼战斗公式修正

**修正前(R5)公式**:
```
山贼战力 = 玩家总兵力 × 山贼难度系数
战斗胜率 = min(95%, max(5%,
  (玩家总兵力 / (玩家总兵力 × 山贼难度系数)) × 50%
  + 地形修正 + 科技加成
))
```

**R5公式存在的数学错误**:
- 玩家总兵力在分子分母中被完全约掉，结果 = `(1/难度系数) × 50%`
- 低等级玩家(难度x0.3)胜率 = 166.7% → 截断为95%，几乎必胜
- 高等级玩家(难度x0.9)胜率 = 55.6%，反而更低
- 玩家"带多少兵"的决策完全失去意义

**修正后(R6)公式**:
```
山贼战力 = 基础战力 × 难度系数
基础战力 = max(100, 玩家等级 × 100)
  // 独立数值，不再基于玩家兵力
  // Lv10基础战力=1000, Lv50基础战力=5000, Lv70基础战力=7000

山贼难度系数:
  - 玩家等级 1~20:  ×0.3（弱匪）
  - 玩家等级 21~40: ×0.5（悍匪）
  - 玩家等级 41~60: ×0.7（精锐山贼）
  - 玩家等级 61+:   ×0.9（山贼王）

战斗胜率 = min(95%, max(5%,
  (玩家总兵力 / 山贼战力) × 50%
  + 地形修正(森林+10% / 山地-10% / 其他0%)
  + 科技加成(攻城术等级×2%)
))

预计兵力损失(失败时) = 出征兵力 × 20%
预计兵力损失(胜利时) = 出征兵力 × max(5%, 20% × (山贼战力 / 玩家总兵力))
```

**修正前后数值对比**(假设无地形/科技修正):

| 玩家等级 | 难度系数 | 山贼战力 | 带兵1000胜率 | 带兵3000胜率 | 带兵5000胜率 | 修正后意义 |
|---------|---------|---------|-------------|-------------|-------------|-----------|
| Lv10 | x0.3 | 300 | 50% → 95%(截断) | 50% → 95%(截断) | 50% → 95%(截断) | 低等级弱匪，多数玩家可轻松应对 |
| Lv30 | x0.5 | 1500 | 33.3% | 50% → 95%(截断) | 50% → 95%(截断) | 中等级悍匪，需要一定兵力 |
| Lv50 | x0.7 | 3500 | 14.3% | 42.9% | 71.4% | 高等级精锐，兵力投入有明显回报 |
| Lv70 | x0.9 | 6300 | 7.9% | 23.8% | 39.7% | 顶级山贼王，需大量兵力才有胜算 |

**关键改进**:
1. 玩家总兵力在公式中不再被约掉，"带多少兵"直接影响胜率
2. 低等级玩家面对弱匪仍有优势(基础战力低)，但不是必胜
3. 高等级玩家面对山贼王需要投入更多兵力，"预计损失"才有参考价值
4. 胜率从"等级决定论"回归"策略决策"——玩家可以通过增加兵力提升胜率

**极端值验证**:

| 测试用例 | 公式代入 | 结果 | 符合直觉? |
|---------|---------|------|----------|
| Lv10带100兵打弱匪 | (100/300)×50% = 16.7% | 16.7% | Yes - 兵力太少，即使弱匪也有风险 |
| Lv10带1000兵打弱匪 | (1000/300)×50% = 166.7% → 95% | 95% | Yes - 兵力充足，弱匪几乎必胜 |
| Lv70带10000兵打山贼王 | (10000/6300)×50% = 79.4% | 79.4% | Yes - 大量兵力投入，较高胜率 |
| Lv70带1000兵打山贼王 | (1000/6300)×50% = 7.9% | 7.9%(截断为5%) | Yes - 兵力悬殊，几乎不可能 |

#### 二、遗迹探索三档判定修正

**修正前(R5)公式**:
```
探索成功率 = 70%(基础) + 科技加成(攻城术×1%) + 地形修正

探索结果:
  成功(70%+): 完整奖励
  部分成功(30%~70%): 50%奖励
  失败(<30%): 无奖励 + 损失10%兵力
```

**R5公式存在的逻辑混乱**:
- 基础成功率70%，若按`random < 成功率`判定则只有成功/失败两档，"部分成功"永远不触发
- 若按三档随机判定(随机数落在30%/70%/100%区间)，则基础成功率70%参数无意义
- 科技加成(+1%/级)在两种判定方式下都形同虚设

**修正后(R6)公式**:
```
三档随机判定模式:
  roll = random(0, 100)

  失败阈值 = max(5%, 30% - 科技加成 - 地形修正)
  部分成功阈值 = min(95%, 50% + 科技加成 + 地形修正)

  if roll < 失败阈值:
    → 失败: 无奖励, 损失出征兵力×10%
  elif roll < 部分成功阈值:
    → 部分成功: 50%奖励, 内应信概率降至12.5%
  else:
    → 成功: 完整奖励, 内应信概率25%

科技加成定义:
  攻城术等级×2% → 同时降低失败阈值和提升部分成功阈值

地形修正:
  山地+5% / 森林+3% / 其他0%
```

**修正前后概率分布对比**(假设无科技/地形修正):

| 结果档 | R5(修正前) | R6(修正后) | 说明 |
|-------|-----------|-----------|------|
| 失败 | 永远不触发(基础70%已超30%) | 5%~30% → **25%概率** | 有意义的失败风险 |
| 部分成功 | 永远不触发(基础70%已超70%) | 30%~50% → **20%概率** | 中间档回归 |
| 成功 | 100%(基础70%直接落入) | 50%~100% → **55%概率** | 基础成功率回归合理 |

**科技加成实际效果验证**:

| 攻城术等级 | 科技加成 | 失败阈值 | 部分成功阈值 | 失败概率 | 部分成功概率 | 成功概率 |
|-----------|---------|---------|------------|---------|------------|---------|
| Lv0 | +0% | 30% | 50% | 30% | 20% | 50% |
| Lv3 | +6% | 24% | 56% | 24% | 32% | 44% |
| Lv5 | +10% | 20% | 60% | 20% | 40% | 40% |
| Lv10 | +20% | 10% | 70% | 10% | 60% | 30% |

> 注意: 上表中"部分成功概率"在高等级科技下反而高于"成功概率"，这是因为科技加成同时收窄了失败区间和拓宽了中间区间。为进一步激励科技升级，将成功阈值调整公式优化如下:

**优化后的阈值公式**(使科技对成功概率有更直接的提升):
```
失败阈值 = max(5%, 30% - 科技加成 - 地形修正)
部分成功阈值 = max(失败阈值+5%, 50% - 科技加成×0.5 - 地形修正×0.5)
  // 注意: 部分成功阈值随科技提升而降低，使成功区间(>部分成功阈值)扩大

实际效果:
  成功区间 = 100% - 部分成功阈值
  部分成功区间 = 部分成功阈值 - 失败阈值
  失败区间 = 失败阈值
```

**优化后科技加成效果**:

| 攻城术等级 | 科技加成 | 失败阈值 | 部分成功阈值 | 失败概率 | 部分成功概率 | 成功概率 |
|-----------|---------|---------|------------|---------|------------|---------|
| Lv0 | +0% | 30% | 50% | 30% | 20% | 50% |
| Lv3 | +6% | 24% | 47% | 24% | 23% | 53% |
| Lv5 | +10% | 20% | 45% | 20% | 25% | 55% |
| Lv10 | +20% | 10% | 40% | 10% | 30% | 60% |

> 科技从Lv0到Lv10: 成功率从50%提升到60%，失败率从30%降低到10%。玩家能明确感知"升级科技 → 探索变好"。

**极端值验证**:

| 测试用例 | roll值 | 结果 | 符合直觉? |
|---------|-------|------|----------|
| Lv0科技, roll=3 | 3 < 30(失败阈值) | 失败 | Yes - 无科技时有合理失败风险 |
| Lv0科技, roll=45 | 30<45<50 | 部分成功 | Yes - 中间档可触发 |
| Lv0科技, roll=80 | 80 > 50 | 成功 | Yes - 基础成功率合理 |
| Lv10科技, roll=8 | 8 < 10(失败阈值) | 失败 | Yes - 高科技仍有小概率失败 |
| Lv10科技, roll=55 | 55 > 40 | 成功 | Yes - 高科技显著提升成功率 |

**数据变更**:

| 数据 | 变更前(R5) | 变更后(R6) | 来源 |
|------|-----------|-----------|------|
| explore.baseSuccessRate | 70%(基础) | 不再使用单参数 | 三档判定替代 |
| explore.failThreshold | 未定义 | max(5%, 30%-科技-地形) | R6新增 |
| explore.partialThreshold | 未定义 | max(失败阈值+5%, 50%-科技×0.5-地形×0.5) | R6新增 |

**冷却倒计时显示逻辑**:
```
function getCooldownDisplay(city):
  if !city.insiderExposed:
    return null  // 无冷却

  remaining = city.insiderCooldownEnd - now()
  if remaining <= 0:
    city.insiderExposed = false  // 冷却结束，自动解除
    return null

  hours = floor(remaining / 3600)
  minutes = floor((remaining % 3600) / 60)
  seconds = floor(remaining % 60)

  // 卡片角标(简短): "XX:XX"
  cardDisplay = pad(hours) + ":" + pad(minutes)

  // 弹窗(精确): "XX:XX:XX"
  popupDisplay = pad(hours) + ":" + pad(minutes) + ":" + pad(seconds)

  return {card: cardDisplay, popup: popupDisplay}
```

### MAP-F12-01-R6 离线事件过期时间防滥用修正 {#map-f12-01-r6}

#### 一、过期时间累计衰减机制

**修正前(R5)**:
```
event.expiresAt = loginTime + 24h
// 每次登录都重算为24h，理论上可无限重置
```

**修正后(R6)**:
```
过期时间累计衰减规则:
  第1次登录(首次): event.expiresAt = loginTime + 24h
  第2次登录(重算): event.expiresAt = loginTime + 12h
  第3次登录(重算): event.expiresAt = loginTime + 6h
  第4次登录: 不再重算，强制过期

  总存活时间上限: 72h (从事件触发时刻算起)
  即: event.maxLifespan = triggeredAt + 72h
  当 now > event.maxLifespan 时，无论登录次数，强制过期
```

**累计衰减计算表**:

| 登录次数 | 过期时间 | 最短存活 | 最长存活(72h上限) | 说明 |
|---------|---------|---------|-----------------|------|
| 第1次 | loginTime + 24h | 0h | 24h | 首次登录，标准缓冲 |
| 第2次 | loginTime + 12h | 24h | 36h | 再次登录，缓冲减半 |
| 第3次 | loginTime + 6h | 36h | 42h | 第三次，仅6h窗口 |
| 第4次 | 不重算 | 42h | 72h(上限) | 强制过期 |

**流程步骤(R6修订MAP-F12-01-R5 Step 4)**:

4. [系统重算-修订] **过期时间累计衰减重算**:
   ```
   对于每个 pendingEvent:
     // 读取该事件的登录重算次数
     recalcCount = event.recalcCount || 0

     // 检查总存活时间上限
     if (now - event.triggeredAt) >= 72h:
       // 超过总存活时间上限，强制过期
       event.status = "expired"
       event.forceExpired = true
       continue

     // 累计衰减重算
     switch(recalcCount):
       case 0: duration = 24h   // 第1次: 24h
       case 1: duration = 12h   // 第2次: 12h
       case 2: duration = 6h    // 第3次: 6h
       default:
         // 第4次+: 不重算，保持上次过期时间
         // 若已过期则强制过期
         if now > event.expiresAt:
           event.status = "expired"
         continue

     // 执行重算
     event.expiresAt = loginTime + duration
     event.recalcCount = recalcCount + 1

     // 确保不超过总存活上限
     event.expiresAt = min(event.expiresAt, event.triggeredAt + 72h)
   ```

## R6 PRD修订建议(PRD-R)

### PRD-R010 山贼战斗公式修正

**PRD原文**: MAP-5事件类型表中山贼"战斗→获装备"，但未定义战斗公式
**问题分析**: R5定义的公式(CH-MAP-R5-03)存在兵力被约掉的数学错误
**修订方案**: 在PRD MAP-5章节补充山贼战斗公式:
- 山贼战力 = max(100, 玩家等级x100) x 难度系数(0.3/0.5/0.7/0.9)
- 战斗胜率 = (玩家兵力/山贼战力) x 50% + 修正，截断至[5%, 95%]
- 兵力投入正向影响胜率
**优先级**: P0

## R6 方法论复盘

### "双轨闭环验证"方法论

R6在R5的"数据生命周期闭环"基础上，扩展为"数据+交互"双轨验证:

| 验证维度 | R5方法 | R6扩展 | 发现的盲区 |
|---------|--------|--------|-----------|
| 数据生命周期 | 产生→存储→查询→消耗→效果 | 不变 | R5已覆盖 |
| 交互生命周期 | 未系统化 | 用户感知→操作→反馈→状态更新 | 暴露冷却UI断裂(R5-01)、公式交互无意义(R5-03) |

**交互生命周期检查清单(R6新增)**:

对每个新增/修改流程，检查:
1. **感知**: 玩家是否能在操作前看到正确的状态信息？
2. **操作**: 操作是否在所有状态下都有合理的响应？
3. **反馈**: 操作结果是否有明确的UI告知？
4. **状态更新**: 结果是否正确更新到所有相关UI位置？
