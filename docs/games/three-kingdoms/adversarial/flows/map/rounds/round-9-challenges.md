## P0 质疑

### CH-MAP-R9-01 休闲玩家声望模拟净变化-3/天与行为描述严重不一致，模拟框架可信度受损 [P0]

**质疑流程**: MAP-F12-01-R9 声望模拟完整性提升: 7类玩家
**追问**: "这个数值对吗？" — 休闲玩家的-3/天是否与其行为描述一致？

**问题**:

R9的7类玩家模拟表(第413-421行)中，休闲玩家的定义为:

```
休闲玩家 | 每周1-2次 | 0次 | 2次/日 | 0个 | +2(登录日) | ~5/周 | 约-3
```

行为描述:
- 每周登录1-2次(平均1.5次)
- 登录日仅做快速处理2次，每次+1声望 → 每日获取+2
- 非登录日: 无获取，受衰减机制影响

**交叉校验: 行为描述 vs 净变化计算**

按R9的行为描述推算:

```
每周获取: +2 x 1.5次(登录日) = +3
每周衰减(非登录日): 取决于豁免条件判定
  - 登录日: lastActiveDate = today → 次日豁免(条件1)
  - 登录次日: 豁免(条件1: yesterday = 登录日)
  - 其余非登录日: 不豁免 → -1/天
  - 非豁免天数约: 7 - 1.5(登录日) - 1.5(登录次日豁免) = 4天
每周衰减: -1 x 4天 = -4
每周净变化: +3 - 4 = -1
每日平均: -1/7 ≈ -0.14/天
```

**R9声称-3/天 vs 推算-0.14/天**: 差异约21倍。

**30天模拟表一致性校验**:

R9的30天模拟表(第425-431行)显示:
```
休闲玩家: 50 → 29(第7天) → 8(第14天) → 0(第21天)
```

50 → 29: 变化-21，7天 → -3/天。模拟表与"-3/天"一致。

但模拟表与行为描述不一致。要达到-3/天，每周净变化需为-21，即:

```
所需: 每周衰减 = +3(获取) + 21(净损失) = +24
所需: 每周非豁免天数 = 24/1 = 24天(超过7天，不可能)
```

即使考虑per-faction衰减(3个阵营各-1/天):
```
每周衰减: -3 x 非豁免天数
若非豁免天数=4: -3 x 4 = -12
每周净: +3 - 12 = -9/周 = -1.29/天(仍不是-3)
```

**结论**: R9的休闲玩家-3/天无法从行为描述中推导出来。模拟表的数值与行为描述之间存在不可调和的矛盾。

**影响**:

1. **模拟框架可信度受损**: R9的核心改进之一是"7类玩家完整模拟"，但其中一类玩家的数值与行为描述不一致，质疑者无法信任其他6类玩家的数值准确性
2. **平衡性结论不可靠**: "休闲玩家30天声望降至0"的结论依赖-3/天的净变化率，如果实际为-0.14/天，30天后声望为50-0.14x30=45.8(而非0)，结论完全不同
3. **回流玩家恢复时间偏差**: R9声称休闲玩家回流后需"约71天恢复到50声望"，但如果实际净变化为-0.14/天且声望未降至0，回流恢复时间的计算前提不成立

**建议**:

1. 重新核算休闲玩家的每日净变化，确保与行为描述一致。如果-3/天是正确的，需要解释其推导过程(特别是衰减的计算方式)
2. 对7类玩家的每日净变化逐一执行"行为描述→推算→模拟表"三方交叉校验，确保一致性
3. 在模拟表中标注每类玩家的衰减计算方式(全局/per-faction/豁免条件判定)，使质疑者可独立验证

### CH-MAP-R9-03 offlineEventCap=5与dailyLimit=5的绑定是设计选择而非逻辑必然，每日多次登录场景仍未解决 [P1]

**质疑流程**: MAP-F12-02-R9 情报值R6溯源修正+离线事件上限定义
**追问**: "这个数值对吗？" — dailyLimit为什么必须等于offlineEventCap？

**问题**:

R9正式定义了offlineEventCap=5(第219-241行)，并建立绑定关系:

```
绑定关系: intelPoints.dailyLimit = offlineEventCap(情报值每日获取上限等于离线事件上限)
设计理由: "确保玩家不会因事件多但收益上限低产生割裂感"
```

但这个绑定存在逻辑问题:

**问题1: 绑定是设计选择，不是逻辑必然**

offlineEventCap控制"离线期间最多存储多少事件"，dailyLimit控制"每日最多获取多少情报值"。这两个参数服务于不同目的:

```
offlineEventCap的目的: 防止长期离线后事件堆积(体验保护)
dailyLimit的目的: 限制每日情报值获取速率(经济平衡)
```

将两者绑定意味着: "离线存储上限"决定了"每日收益上限"。但:
- 如果offlineEventCap调整为8(如R9"已知风险"中提到的场景)，dailyLimit也会变为8，情报值经济平衡被打破
- 如果offlineEventCap调整为3，dailyLimit变为3，与R6质疑建议一致，但玩家每日可处理的事件数被压缩

**问题2: 每日多次登录场景未解决**

R9在"已知风险"(第238行)中列出:

> "若玩家每日登录多次(早中晚各一次): 每次处理部分事件，实际不受offlineEventCap限制"

但未给出解决方案。如果一个玩家每日登录3次:
```
早上登录: 处理3个离线事件 → +3情报值
中午登录: 处理2个新事件 → +2情报值(已达dailyLimit=5)
晚上登录: 处理1个新事件 → +0情报值(已达上限)

总情报值: +5(dailyLimit生效)
总事件处理: 6个(超过offlineEventCap=5)
```

这种情况下dailyLimit=5有效限制了情报值获取，但事件处理数量超过offlineEventCap。这意味着offlineEventCap不是真正的"事件上限"——它只是"单次登录时的事件堆积上限"。

**问题3: offlineEventCap的FIFO替换策略未正式定义**

R9第225行提到"新事件替换最旧事件(FIFO)"，但未定义:
- 被替换的事件是否永久丢失？玩家是否知道有事件被替换？
- 替换时是否通知玩家？
- 如果被替换的事件是高价值事件(如精英山贼)，替换是否合理？

**影响**:

1. **参数耦合风险**: offlineEventCap与dailyLimit的绑定使两个独立参数无法独立调优
2. **命名误导**: "offlineEventCap"暗示"离线事件上限"，但实际是"单次登录事件堆积上限"
3. **FIFO策略未定义**: 开发者无法确定被替换事件的处理方式

**建议**:

1. 解除offlineEventCap与dailyLimit的绑定，将dailyLimit定义为独立参数(如dailyLimit=5，理由为"情报值经济平衡")
2. 将offlineEventCap重命名为loginEventQueueSize或类似名称，更准确反映其语义
3. 正式定义FIFO替换策略: 被替换事件是否丢失、是否通知玩家、替换优先级

### CH-MAP-R9-04 冷却管理器_scanAllCooldowns()遍历Map时调用delete()，存在迭代期间修改集合的隐患 [P1]

**质疑流程**: MAP-F06-07-R9 冷却管理器并发安全方案
**追问**: "如果失败呢？" — 乐观锁方案的代码实现是否可靠？

**问题**:

R9的乐观锁方案(第525-572行)中，_scanAllCooldowns()的代码:

```javascript
_scanAllCooldowns():
    now = now()
    for [cityId, state] of cooldowns:       // 遍历Map
      if state.status === "active":
        remaining = state.endAt - now
        if remaining <= 0:
          if !state._processed:
            state._processed = true
            state.status = "ended"
            emit("cooldownStateChanged", { cityId, status: "ended" })
            cooldowns.delete(cityId)          // 遍历期间删除!

    if cooldowns.size === 0:
      stopScan()
```

**问题1: 遍历期间调用delete()**

JavaScript的Map在遍历期间调用delete()的行为:

```
ECMAScript规范(ES2015+):
  - Map.prototype[Symbol.iterator]在遍历期间如果条目被删除，
    且该条目尚未被访问，则该条目不会被访问
  - 如果条目已被访问，则删除操作不影响遍历
  - 这是规范定义的行为，不是未定义行为
```

虽然ECMAScript规范保证了Map遍历期间delete()的安全性，但:
- 代码审查者可能不熟悉此规范细节，误认为是bug
- 如果未来将Map替换为其他数据结构(如WeakMap或自定义集合)，此行为可能不成立
- 在遍历期间修改集合是代码异味(code smell)，即使安全也不推荐

**问题2: 乐观锁的_processed标记与status检查之间的语义重叠**

代码中同时检查`state.status === "active"`和`!state._processed`:

```javascript
if state.status === "active":        // 检查1: 状态是否活跃
  if remaining <= 0:
    if !state._processed:             // 检查2: 是否已处理
```

两个检查的语义:
- `status === "active"`: 冷却是否正在进行
- `!_processed`: 是否已被清理逻辑标记

如果getCooldown()先执行: 设置`status = "ended"`并`delete(cityId)`。此时_scanAllCooldowns()遍历时，该条目已从Map中删除，不会被访问。_processed标记实际上不会被触达。

如果_scanAllCooldowns()先执行: 设置`_processed = true`、`status = "ended"`并`delete(cityId)`。此时getCooldown()调用`cooldowns.get(cityId)`返回null，不会进入过期处理逻辑。_processed标记同样不会被触达。

**结论**: 在JavaScript单线程模型中，getCooldown()和_scanAllCooldowns()不会真正并发执行。_processed标记在当前实现中永远不会被用于去重——真正的去重是由`cooldowns.delete()`和`cooldowns.get()返回null`实现的。_processed标记是冗余的。

**问题3: destroy()后setCooldown()的惰性初始化未在代码中实现**

R9第613行声称: "destroy()后调用setCooldown(): 重建cooldowns Map(惰性初始化)"

但代码(第526-572行)中未展示setCooldown()的实现，也未展示惰性初始化逻辑。如果destroy()执行了`cooldowns.clear()`，后续setCooldown()需要检查cooldowns是否存在并重建:

```javascript
// R9声称的惰性初始化(未在代码中展示):
setCooldown(cityId, duration):
  if !this.cooldowns:                    // 需要这个检查
    this.cooldowns = new Map()           // 惰性重建
    this.listeners = new Set()           // 惰性重建
  // ... 设置冷却
```

此逻辑未在代码中体现，开发者可能遗漏。

**影响**:

1. **代码审查误导**: 遍历期间delete()虽符合ECMAScript规范，但可能被代码审查者标记为bug
2. **_processed标记冗余**: 乐观锁的去重逻辑实际上由delete()/get()实现，_processed标记不产生实际效果，增加了代码复杂度但无收益
3. **destroy()后行为未实现**: 惰性初始化逻辑缺失，开发者可能遗漏

**建议**:

1. 将_scanAllCooldowns()改为先收集待删除的cityId列表，再统一删除，避免遍历期间修改集合:
   ```javascript
   const toDelete = []
   for [cityId, state] of cooldowns:
     if state.status === "active" && state.endAt - now <= 0:
       toDelete.push(cityId)
   for cityId of toDelete:
     cooldowns.delete(cityId)
     emit("cooldownStateChanged", { cityId, status: "ended" })
   ```
2. 评估_processed标记是否真正需要。如果delete()/get()已提供去重保护，移除_processed标记以简化代码
3. 在代码中展示setCooldown()的完整实现，包含destroy()后的惰性初始化逻辑

R9的期望值公式(第856-866行):

```
E(重试令牌价值) = P(重试胜利) x (奖励价值 - 首次失败损耗 - 重试胜利损耗)
                + P(重试失败) x (-首次失败损耗 - 重试失败损耗 - 令牌价值)
```

**问题1: 首次失败损耗是沉没成本**

在决策"是否使用重试令牌"时，首次战斗已经发生且已失败，首次失败损耗(出征兵力x20%)是已经发生的损失，不应纳入重试决策的期望值计算。

正确的重试决策期望值公式应为:

```
E(重试 | 首次已失败) = P(胜利) x (奖励价值 - 重试胜利损耗)
                      + P(失败) x (-重试失败损耗)
                      - 令牌价值

与不重试对比:
E(不重试) = 0

决策: 若 E(重试) > 0，则重试
```

R9的公式将首次失败损耗纳入两个分支，相当于:

```
R9公式 = 正确公式 + (-首次失败损耗)
```

即R9的期望值比正确值低了一个"首次失败损耗"的量。这导致:
- 所有场景的期望值被系统性低估
- 低胜率场景的"负期望"结论被夸大

**问题2: 数值验证**

以Lv70带1000兵(5%胜率)为例:

```
R9公式:
  E = 5% x (奖励 - 200 - 350) + 95% x (-200 - 200 - 令牌)
    = 5% x (奖励 - 550) + 95% x (-400 - 令牌)
    = 0.05x奖励 - 27.5 - 380 - 0.95x令牌
    = 0.05x奖励 - 407.5 - 0.95x令牌

正确公式:
  E = 5% x (奖励 - 350) + 95% x (-200) - 令牌
    = 0.05x奖励 - 17.5 - 190 - 令牌
    = 0.05x奖励 - 207.5 - 令牌
```

差异: R9公式比正确值低200(恰好=首次失败损耗)。

R9声称"奖励需>4100兵才正期望"。按正确公式:
```
0.05x奖励 - 207.5 > 0
奖励 > 4150兵
```

差异不大(4100 vs 4150)，因为首次失败损耗(200)相对于其他数值较小。但对于更低兵力(如带500兵)的场景:
```
R9: 0.05x(奖励-175) + 0.95x(-100-100-令牌) = 0.05x奖励 - 8.75 - 190 - 0.95x令牌
正确: 0.05x(奖励-175) + 0.95x(-100) - 令牌 = 0.05x奖励 - 8.75 - 95 - 令牌
```

差异: 95(首次失败损耗)。R9的"奖励需>2175兵才正" vs 正确的"奖励需>2075兵才正"。

**影响**:

1. **期望值被系统性低估**: 所有场景的期望值比正确值低一个"首次失败损耗"的量
2. **低胜率场景结论被夸大**: "负期望"的结论虽然方向正确，但幅度被夸大
3. **方法论错误**: 在决策分析中纳入沉没成本是经典的经济学错误，降低了文档的方法论可信度

**建议**:

1. 修正期望值公式，移除沉没成本(首次失败损耗):
   ```
   E(重试 | 首次已失败) = P(胜利) x (奖励 - 重试胜利损耗)
                         + P(失败) x (-重试失败损耗)
                         - 令牌价值
   ```
2. 重新计算极端值验证表中的期望值
3. 增加"与不重试对比"列: E(重试) vs E(不重试)=0，明确决策标准

### CH-MAP-R9-06 "间歇性重度玩家"类型在R8被标记为缺失，R9仍未补充 [P2]

**质疑流程**: MAP-F12-01-R9 声望模拟完整性提升: 7类玩家

R8质疑(CH-MAP-R8-03)第147-154行列出了4种缺失的玩家类型:

| 缺失类型 | R9是否补充 |

| 新玩家(声望0起步) | 未补充 |

**间歇性重度玩家的声望行为**:

```
行为: 某周每日活跃(获取+8.5/天)，下两周不登录(衰减-1/天)
第1周: +8.5 x 7 = +59.5，声望: 50 → 100(满)

第3周: -1 x 7 = -7，声望: 93 → 86
第4周: 活跃，+8.5/天，3天即满100

特征: 声望在86-100之间波动，不会降至0
```

这个玩家类型揭示了一个R9未讨论的问题: **声望波动性**。间歇性重度玩家的声望在86-100之间周期性波动，但永远不会降至0(因为100分的衰减缓冲足够大)。这与"流失玩家"和"休闲玩家"的单调递减形成对比。

**影响**:

**建议**:

1. 在7类玩家矩阵中增加第8类: "间歇性重度玩家"(周期性活跃+休息)
2. 在平衡性结论中增加"声望波动性"分析维度，评估不同玩家类型的声望变化模式

### CH-MAP-R9-07 factionEventToday从全局boolean改为per-faction Map，在阵营扩展场景下引入不必要的数据结构复杂度 [P2]

**质疑流程**: MAP-F12-01-R9a 声望豁免条件per-faction语义明确化
**追问**: "这个边界对吗？" — per-faction语义是否是最佳实现方案？

**问题**:

R9将factionEventToday从全局boolean改为`Map<faction, boolean>`:

```javascript
// R8(修正前):
player.factionEventToday: boolean  // 全局

// R9(修正后):
player.factionEventToday: Map<faction, boolean>  // per-faction
```

这个修改解决了CH-MAP-R8-08的per-faction语义歧义，但引入了新的问题:

**问题1: 条件1(全局)与条件2(per-faction)的不对称性**

R9的豁免条件:
- 条件1(昨日活跃): 全局判定 → 对所有阵营统一豁免
- 条件2(当日声望事件): per-faction判定 → 仅对有事件的阵营豁免

这意味着条件1和条件2使用不同的判定粒度。在实际场景中:

```
场景: 玩家昨日登录(条件1满足) → 所有阵营豁免
  → 条件2永远不会被触达(因为条件1已短路)

场景: 玩家昨日未登录(条件1不满足) → 检查条件2
  → 条件2仅对有事件的阵营豁免
  → 无事件的阵营被衰减
```

条件2的per-faction语义仅在"昨日未登录但今日有声望事件"的极小概率场景中有意义。为这个边界场景引入Map数据结构，增加了日常操作的复杂度(每次声望事件都需要更新Map)。

**问题2: 数据结构扩展性**

当前游戏有3个阵营(魏/蜀/吴)，Map<faction, boolean>只有3个条目，开销可忽略。但如果:
- 未来扩展到更多阵营(如群雄): Map条目线性增长
- 每个玩家每天需要存储和重置这个Map: 内存和CPU开销随阵营数线性增长

**问题3: 重置时机的批量操作开销**

R9第763-765行:
```javascript
// R9新增: 衰减判定完成后重置factionEventToday
for each player in allPlayers:
  player.factionEventToday = new Map([
    [魏, false], [蜀, false], [吴, false]
  ])
```

如果游戏有10万活跃玩家，每日00:00需要创建10万个Map对象。虽然单个Map很小，但批量创建的GC压力不可忽略。

**替代方案**: 使用位掩码(bitmask)代替Map:

```javascript
// 位掩码方案:
player.factionEventToday: number  // 0b000 = 无事件, 0b001 = 魏, 0b010 = 蜀, 0b100 = 吴
// 判定: player.factionEventToday & FACTION_MASK[faction]
// 设置: player.factionEventToday |= FACTION_MASK[faction]
// 重置: player.factionEventToday = 0
```

位掩码方案的优势:
- 单个number字段，无Map对象创建
- 重置为0，无批量对象创建
- 判定和设置为位操作，O(1)
- 扩展性好: 32位number支持32个阵营

**影响**:

1. **实现复杂度**: Map方案比简单的位掩码或数组方案更复杂
2. **性能**: 每日重置时创建大量Map对象，增加GC压力
3. **可读性**: Map<faction, boolean>的语义虽清晰，但位掩码在游戏中更常见且更高效

**建议**:

1. 考虑将factionEventToday改为位掩码(number类型)，降低实现复杂度和性能开销
2. 如果坚持Map方案，在重置时复用Map对象(调用.clear()而非创建新Map)以减少GC压力
3. 在文档中说明per-faction语义的必要性: 如果条件2的per-faction判定场景极小概率，是否值得为此增加实现复杂度

### R9整体评价

R9在R8质疑的响应上取得了实质性进展:

1. **公理声明化**: 将k=1.75的3条隐含假设显式声明为"设计假设"并标注性质，是方法论的重要改进。从"隐含使用"到"显式声明+已知反论+裁决状态"的转变，使推导根基可追溯、可质疑。

2. **溯源链修正**: R6差异说明从"R6选择3次"(错误)修正为"R6未定义，R6质疑建议=3"，跨轮引用溯源校验表(4条引用)建立了"流程定义"与"质疑建议"的区分标准。

3. **并发安全方案**: 乐观锁+_processed标记+destroy()生命周期+10s扫描间隔的组合方案，系统性地解决了R8提出的三个并发风险(竞态/内存/UI跳变)。

4. **per-faction语义明确化**: factionEventToday从全局boolean改为per-faction Map，factionEventToday重置时机定义为"00:00衰减判定完成后"，伪代码完整且时序边界验证通过。

**R9的核心弱点**:

1. **模拟数值一致性**: 休闲玩家-3/天的净变化无法从行为描述中推导出来，暴露了模拟框架中"行为描述→数值推算→模拟表"三方校验的缺失。这是R9的P0级问题，因为它质疑了R9核心改进(7类玩家模拟)的可信度。

2. **A/B测试方法论**: 公理声明化解决了"假设未声明"的问题，但A/B测试方案缺少方法论关键要素(样本量/持续时间/成功标准/统计方法)，无法作为"假设未验证"的解决机制。

3. **参数绑定逻辑**: offlineEventCap=5与dailyLimit=5的绑定是设计选择而非逻辑必然，两个参数服务于不同目的，绑定后无法独立调优。

**R8质疑解决率**: 8项R8质疑中，3项完全解决(R8-02登录免疫/R8-05溯源修正/R8-08 per-faction)，5项部分解决(R8-01公理声明化但A/B测试不完整/R8-03补充玩家类型但数值不一致/R8-04并发安全但代码有隐患/R8-06期望值分析但公式有误/R8-07参数定义但绑定逻辑)。

**建议R10重点**: 休闲玩家模拟数值修正(三方交叉校验) + A/B测试方法论补全 + offlineEventCap/dailyLimit解绑评估 + 期望值公式修正(移除沉没成本)。
