### CH-MAP-R10-01 期望值极端值验证表奖励阈值计算错误，多场景数值不自洽 [P1]

**质疑流程**: MAP-F12-02-R10 期望值公式修正(移除沉没成本)
**追问**: "这个数值对吗？" — 极端值验证表的奖励阈值是否正确？

**问题**:

R10的期望值公式(修正后)为:

```
E(重试 | 首次已失败) = P(胜利) x (奖励价值 - 重试胜利损耗)
                      + P(失败) x (-重试失败损耗)
                      - 令牌价值
```

R10声称"修正后公式正确"，并给出了极端值验证表。但逐场景验算后发现多处奖励阈值计算错误。

**场景逐一行验算**:

**场景1: Lv70带5000兵，50%胜率**

```
R10公式: E = 0.50 x (奖励 - 1750) + 0.50 x (-1000) - 令牌
设E=0(令牌忽略): 0.50x(奖励-1750) + 0.50x(-1000) = 0
  0.50x奖励 - 875 - 500 = 0
  0.50x奖励 = 1375
  奖励 = 2750

R10表格声称: "正期望(若奖励>2750兵)"
验算: E(奖励=2750) = 0.50x1000 + 0.50x(-1000) - 3 = -3 ≈ 0
结论: ✓ 正确(临界值约2753含令牌)
```

**场景2: Lv70带3000兵，33%胜率**

```
R10公式: E = 0.33 x (奖励 - 1050) + 0.67 x (-600) - 令牌
设E=0(令牌忽略): 0.33x(奖励-1050) + 0.67x(-600) = 0
  0.33x奖励 - 346.5 - 402 = 0
  0.33x奖励 = 748.5
  奖励 = 2268.2

R10表格声称: "正期望(若奖励>2424兵)"
验算: E(奖励=2424) = 0.33x(2424-1050) + 0.67x(-600) - 3
    = 0.33x1374 + (-402) - 3
    = 453.4 - 402 - 3 = +48.4 > 0 (确实正期望)

但正确临界值应为≈2268(不含令牌)或≈2277(含令牌)。
R10声称2424，比正确值高出约150。
```

**场景3: Lv70带1000兵，5%胜率**

```
R10公式: E = 0.05 x (奖励 - 350) + 0.95 x (-200) - 令牌
设E=0(令牌忽略): 0.05x(奖励-350) + 0.95x(-200) = 0
  0.05x奖励 - 17.5 - 190 = 0
  0.05x奖励 = 207.5
  奖励 = 4150

含令牌(令牌=3): 0.05x奖励 = 207.5 + 3 = 210.5
  奖励 = 4210

R10表格声称: "负期望(奖励需>4200兵才正)"
R10变更汇总声称: 低胜率奖励阈值从R9的">4100兵"修正为">4200兵"

验算: E(奖励=4200) = 0.05x3850 + 0.95x(-200) - 3
    = 192.5 - 190 - 3 = -0.5 < 0 (负期望)

验算: E(奖励=4210) = 0.05x3860 + 0.95x(-200) - 3
    = 193 - 190 - 3 = 0 (临界)

正确临界值: ≈4210(含令牌)或4150(不含令牌)
R10声称4200，低于正确值约10-60。
```

**场景4: Lv70带500兵，5%胜率**

```
R10公式: E = 0.05 x (奖励 - 175) + 0.95 x (-100) - 令牌
设E=0(令牌忽略): 0.05x(奖励-175) + 0.95x(-100) = 0
  0.05x奖励 - 8.75 - 95 = 0
  0.05x奖励 = 103.75
  奖励 = 2075

含令牌(令牌=3): 0.05x奖励 = 103.75 + 3 = 106.75
  奖励 = 2135

R10表格声称: "负期望(奖励需>2150兵才正)"
验算: E(奖励=2150) = 0.05x1975 + 0.95x(-100) - 3
    = 98.75 - 95 - 3 = +0.75 > 0 (正期望!)

正确临界值: ≈2135(含令牌)或2075(不含令牌)
R10声称2150，高于正确值约15-75。
```

**汇总表**:

| 场景 | R10声称阈值 | 正确阈值(含令牌) | 偏差 | 方向 |
|------|-----------|----------------|------|------|
| 5000兵/50% | >2750 | ≈2753 | -3 | 可忽略 |
| 3000兵/33% | >2424 | ≈2277 | **+147** | 高估 |
| 1000兵/5% | >4200 | ≈4210 | -10 | 可忽略 |
| 500兵/5% | >2150 | ≈2135 | +15 | 轻微 |

**影响**:

1. **数值不自洽**: R10声称通过"三方交叉校验"解决了R9的数值问题，但期望值表本身的数值存在计算错误——R10在修复一处数值错误(沉没成本)时引入了新的数值错误(奖励阈值)
2. **决策误导**: Lv70带3000兵场景，R10声称奖励需>2424才正期望，但实际>2277即可正期望——玩家可能因此放弃使用本应正期望的重试令牌

**建议**:

1. 重新计算所有场景的奖励阈值，使用公式精确求解(而非近似估算)
2. 在极端值表中增加"推导过程"列，展示每个阈值的计算步骤
3. 对期望值表执行与声望模拟相同的"三方交叉校验"——公式/推算/表格三方一致

### CH-MAP-R10-02 多次登录场景声称绕过dailyLimit但dailyLimit正是为此设计，逻辑自相矛盾 [P1]

**质疑流程**: MAP-F12-02-R10a offlineEventCap/dailyLimit解绑+FIFO策略
**追问**: "如果失败呢？" — 多次登录场景的dailyLimit约束是否被正确执行？

**问题**:

R10在MAP-F12-02-R10a的"每日多次登录场景处理"中给出了以下场景:

```
早上登录: 处理5个离线事件 → +5情报值
中午登录: 处理2个新事件 → +2情报值
晚上登录: 处理1个新事件 → +1情报值
累计: 8情报值
```

R10随后评论:

> "dailyLimit控制'每日情报值获取上限'，两者独立，多次登录场景不受loginEventQueueSize限制"

**逻辑矛盾**:

```
R10声称: 多次登录场景 → 总情报值8 > dailyLimit=5
R10结论: "两者独立，多次登录场景不受loginEventQueueSize限制"

问题: dailyLimit=5的目的是什么?
  R10定义: "限制每日情报值获取速率(经济平衡)"
  但R10的场景: 玩家实际获取8情报值，超过dailyLimit=5

如果dailyLimit=5不约束总情报值获取:
  → dailyLimit无意义(不生效的参数)
  → R9绑定offlineEventCap=dailyLimit的理由("确保玩家不会因事件多但收益上限低产生割裂感")实际上揭示了一个真实的设计需求

如果dailyLimit=5约束总情报值获取:
  → 早上+5已满，中午和晚上应+0(已达上限)
  → 累计=5，不是8
  → R10的场景计算错误
```

**核心问题**: R10声称"解绑"使两个参数独立运作，但在解绑后未说明dailyLimit如何在多次登录场景中生效。R10的场景示例实际上展示了dailyLimit不生效的情况——这要么是R10的场景计算错误，要么是dailyLimit参数设计不完整。

**影响**:

1. **参数设计不完整**: 解绑后，dailyLimit的约束逻辑未定义——何时检查？是每次事件处理时检查还是每日结算时检查？
2. **场景示例误导**: R10的多次登录场景示例展示了8情报值(超过dailyLimit=5)，但未标注这是否违反dailyLimit约束
3. **R9绑定理由未被回应**: R9的绑定理由是"避免割裂感"，R10解绑后应明确说明如何避免这种割裂感(如: dailyLimit仅约束离线事件情报值，在线事件不受限)

**建议**:

1. 明确dailyLimit的约束时机和范围: 是约束"所有情报值获取"还是仅约束"离线事件情报值"？
2. 修正多次登录场景示例: 如果dailyLimit生效，标注中午和晚上的+0；如果dailyLimit不生效，说明原因
3. 在参数定义中增加约束逻辑伪代码: `if (今日已获取情报值 >= dailyLimit) 则事件处理不产生情报值`

### CH-MAP-R10-05 A/B测试主要指标"战斗参与率"逻辑方向存疑 [P2]

**质疑流程**: MAP-F09-02-R10 A/B测试方法论补全
**追问**: "这个数值有意义吗？" — 主要指标是否能有效区分测试组？

**问题**:

R10定义的主要成功指标:

```
主要指标: 战斗参与率 = 每日参与战斗的玩家数 / DAU
判定标准: k=1.75组的战斗参与率显著高于k=1.0组(p<0.05)
```

**逻辑方向疑问**:

```
k值的作用: k越高 → 胜利损耗越大(惩罚越重) → 理论上抑制战斗参与

预期效果:
  k=1.0(无额外惩罚): 战斗代价最低 → 参与率应最高
  k=2.0(最重惩罚): 战斗代价最高 → 参与率应最低

R10的判定: k=1.75组参与率显著高于k=1.0组
这与直觉相反——惩罚更重的组参与率更高?
```

**可能的解释(但R10未说明)**:

1. k值影响的是"胜利后的损耗"而非"出征意愿"——如果低兵力玩家频繁出征但几乎必败(浪费兵力)，k=1.75通过提高代价迫使玩家更谨慎地选择目标，反而提高了"有意义战斗"的比例
2. 但"战斗参与率"指标无法区分"有意义战斗"和"无意义战斗"——它只统计参与人数/次数

**影响**:

1. **指标定义模糊**: "战斗参与率"可能被解读为"出征玩家比例"或"出征次数/DAU"，两种解读的统计性质不同
2. **判定标准可能反直觉**: 如果k=1.0组参与率反而更高(符合直觉)，R10的判定标准将永远无法满足
3. **次要指标更合理**: R10的次要指标"低兵力(<3000)出征频率"可能比主要指标更能反映k值的效果

**建议**:

1. 重新评估主要指标: 考虑使用"有意义战斗参与率"(排除低胜率出征)或"玩家留存率"作为主要指标
2. 明确指标定义: "战斗参与率"是比例型(参与人数/DAU)还是计数型(参与次数/DAU)？
3. 增加指标方向说明: 为什么预期k=1.75组参与率更高？给出理论依据

### CH-MAP-R10-06 离线事件队列术语混用: 参数名loginEventQueueSize与文档描述"离线事件队列"不一致 [P2]

**质疑流程**: MAP-F12-02-R10a offlineEventCap/dailyLimit解绑+FIFO策略
**追问**: "这个边界对吗？" — 命名修正是否彻底？

**问题**:

R10将offlineEventCap重命名为loginEventQueueSize，理由是:

> "更准确反映语义: '单次登录事件队列大小'"

但R10文档中多处仍使用"离线事件"术语:

```
第596行: "玩家单次登录时，离线事件队列的最大长度"
第600行: "防止长期离线后事件堆积"
第601行: "5个事件约等于离线1.7-5天的累计量"
第617行: "当事件队列已满(loginEventQueueSize=5)且有新事件到达时"
第631行: "替换通知...在玩家登录时显示'您有X个离线事件因队列已满被替换'"
第667行: "离线事件队列: 5个(假设离线1天)"
```

**术语不一致**:

```
参数名: loginEventQueueSize → 强调"登录时的事件队列"
文档描述: "离线事件队列" → 强调"离线期间产生的事件"

语义差异:
  - "离线事件"暗示: 事件在玩家离线期间产生
  - "登录事件队列"暗示: 队列在玩家登录时处理
  - 两者描述的是同一事物的不同方面，但混用会造成理解混乱
```

**影响**:

1. **开发者理解歧义**: 参数名暗示"登录时的事件队列"，但文档描述暗示"离线事件"——开发者可能不确定队列中是否包含在线触发的事件
2. **命名修正不彻底**: R10重命名了参数但未统一文档术语

**建议**:

1. 统一术语: 在文档中将"离线事件队列"统一改为"事件队列"或"登录事件队列"
2. 明确队列内容: 说明队列中是否包含在线触发的事件(如R10多登录场景中的"新事件")
3. 如果队列仅包含离线事件，考虑保留offline前缀但修正语义

**建议**:

### CH-MAP-R10-08 间歇性重度玩家声望波动性分析缺少"极端周期"场景 [P2]

**质疑流程**: MAP-F12-01-R10 间歇性重度玩家类型补充
**追问**: "如果中途退出呢？" — 极端周期模式下声望行为如何？

**问题**:

R10定义的间歇性重度玩家:

```
周期: 1周活跃 + 2周休息 = 3周周期
声望波动: 86-100之间
```

但R10未分析其他可能的周期模式:

```
极端周期1: 1周活跃 + 4周休息(5周周期)
  活跃周: +59.5，声望50→100
  休息周1-4: -7x4 = -28，声望100→72
  波动范围: 72-100

极端周期2: 1周活跃 + 8周休息(9周周期)
  活跃周: +59.5，声望50→100
  休息周1-8: -7x8 = -56，声望100→44(截断至44)
  波动范围: 44-100

极端周期3: 2天活跃 + 2周休息(16天周期)
  活跃天: +8.5x2 = +17，声望50→67
  休息周: -7x2 = -14，声望67→53
  波动范围: 53-67
```

**影响**:

1. **分析维度不完整**: R10的"1周+2周"只是间歇性重度玩家的一种模式，其他周期模式可能导致声望降至更低水平
2. **平衡性结论受限**: "声望永远不会降至0"的结论仅对"1周+2周"模式成立，对"1周+8周"模式不成立(声望可降至44)

**建议**:

1. 增加极端周期分析: 至少覆盖"1周+N周休息"的N=2/4/8场景
2. 定义"间歇性重度玩家"的周期参数范围: 如"活跃期1-2周，休息期1-4周"
3. 在平衡性结论中标注适用范围: "在活跃期/休息期比>=1/4的模式下，声望不会降至X以下"

### CH-MAP-R10-09 FIFO替换策略对高价值事件被替换的玩家体验风险未评估 [P2]

**质疑流程**: MAP-F12-02-R10a offlineEventCap/dailyLimit解绑+FIFO策略
**追问**: "这个数值让玩家有什么感受？" — 高价值事件被替换时玩家体验如何？

**问题**:

R10的FIFO替换策略:

```
规则3: 高价值事件替换
  - 所有事件平等，不区分价值高低
  - 高价值事件(如精英山贼)可能被低价值事件(如普通山贼)替换
  - 这是FIFO策略的固有特性，非bug
```

**玩家体验分析**:

```
场景: 玩家离线3天，期间触发了5个事件(含1个精英山贼高价值事件)
  事件队列: [普通山贼, 普通山贼, 精英山贼, 普通山贼, 普通山贼]
  队列已满(5个)

  玩家登录前，系统又触发了1个新事件(普通山贼)
  FIFO替换: 移除最早的"普通山贼"，添加新"普通山贼"
  队列变为: [普通山贼, 精英山贼, 普通山贼, 普通山贼, 普通山贼]

  此场景精英山贼未被替换(安全)

  但如果离线更长时间:
  事件队列: [精英山贼, 普通山贼, 普通山贼, 普通山贼, 普通山贼]
  新事件到达: [普通山贼]
  FIFO替换: 移除"精英山贼"(最早)，添加"普通山贼"
  → 玩家丢失了高价值事件!
```

**影响**:

1. **玩家感知不公平**: 玩家不知道精英山贼事件被替换，登录后只看到普通山贼——如果事后发现(如查看日志)，会感到不公平
2. **FIFO不是唯一选择**: 可以考虑"最低价值优先替换"策略，保护高价值事件
3. **R10标注为"非bug"但未评估体验影响**: FIFO的固有特性不等于好的设计

**建议**:

1. 评估"最低价值优先替换"策略: 优先替换低价值事件，保护高价值事件
2. 增加替换通知: R10已列出可选优化("您有X个离线事件因队列已满被替换")，建议作为默认行为而非可选
3. 在FIFO策略中增加"价值权重"选项: 如果事件有明确的价值标签(精英>普通)，优先替换低价值事件

## R10总评

### R10核心成就

R10对R9全部7项质疑均有对应产出(覆盖率100%)，其中R9的P0级问题(休闲玩家-3/天与行为描述矛盾)被彻底修复——这是R10最重要的成就。具体改进:

1. **P0修复(休闲玩家数值)**: 从-3/天修正为-0.14/天，三方交叉校验(行为描述=推算=模拟表)通过。平衡性结论从"30天降至0"修正为"几乎不下降(约350天降至0)"。这是R9核心弱点的彻底修复。

2. **A/B测试方法论**: 14要素框架完整(假设/分组/分流/样本量/持续时间/主指标/次指标/统计方法/效应量/决策规则/回退方案)，k=1.0对照组补齐了公理1验证盲区。从R9的"方案骨架"升级为"可执行方案"。

3. **期望值公式修正**: 正确移除沉没成本(首次失败损耗)，与R9公式对比清晰，决策分析方法论改进(新增"沉没成本检查"步骤)。

4. **参数解绑+命名修正**: offlineEventCap重命名为loginEventQueueSize，与dailyLimit解绑(独立定义)，FIFO替换策略正式定义。

5. **间歇性重度玩家**: 第8类玩家定义完整，声望波动性分析维度(5种变化模式)是平衡性分析的重要补充。

6. **冷却管理器代码优化**: 先收集后删除模式消除代码异味，移除冗余_processed标记，setCooldown()含destroy后惰性初始化。

R10在修复R9问题的同时，引入了新的问题:

### R10关键弱点

1. **期望值阈值计算错误(CH-MAP-R10-01)**: R10修复了沉没成本问题，但极端值表的奖励阈值存在多处数值错误(Lv70带3000兵场景偏差最大: R10声称>2424，正确值约为>2277)。这是R10在"数值自洽性"上的遗留问题——R9的P0是"模拟数值与行为描述矛盾"，R10修复了模拟数值，但期望值表出现了新的数值错误。

2. **多登录场景逻辑矛盾(CH-MAP-R10-03)**: R10声称解绑后多次登录场景"不受loginEventQueueSize限制"，但同时展示了总情报值8超过dailyLimit=5的场景——dailyLimit参数的约束逻辑未定义。这是参数设计的不完整，而非单纯的数值错误。

3. **流程ID重复(CH-MAP-R10-03)**: MAP-F12-01-R10被两个不同修改使用，违反方法论的编号唯一性规范。

### R10评分影响预判

R10修复了R9的P0(休闲玩家数值)，这是最重要的进展。但R10引入了3个P1(期望值阈值错误、多登录矛盾、ID重复)，其中期望值阈值错误与R9的P0性质类似——声称的数值无法从公式中推导出来。R10的数值自洽性从"一处P0硬伤"改善为"多处P1数值错误"，有进步但未完全解决。

R10在方法论层面的贡献(三方交叉校验、沉没成本检查、A/B测试模板)是实质性的改进，应获得PRD设计质量维度的加分。但流程完备性维度可能因P1数量(3个)和期望值阈值错误而受限。
