### MAP-F12-01-R10 休闲玩家数值修正 + 7类玩家三方交叉校验 {#map-f12-01-r10}

> R10核心目标#1: 数值自洽性修复(紧急)
> CH-MAP-R9-01: 休闲玩家声望模拟净变化-3/天与行为描述严重不一致，模拟框架可信度受损
> 修正原则: 重新核算休闲玩家每日净变化，对7类玩家逐一执行三方交叉校验，确保行为描述=推算=模拟表

#### 一、问题定位

R9的7类玩家模拟表中，休闲玩家的数值存在不可调和的矛盾:

```
R9声称: 休闲玩家净变化约-3/天
R9行为描述: 每周登录1-2次，每日+2获取

按行为描述推算:
  每周获取: +2 x 1.5次(登录日) = +3
  每周衰减(非登录日): 取决于豁免条件判定
    - 登录日: lastActiveDate = today → 次日豁免(条件1)
    - 登录次日: 豁免(条件1: yesterday = 登录日)
    - 其余非登录日: 不豁免 → -1/天
    - 非豁免天数约: 7 - 1.5(登录日) - 1.5(登录次日豁免) = 4天
  每周衰减: -1 x 4天 = -4
  每周净变化: +3 - 4 = -1
  每日平均: -1/7 ≈ -0.14/天

差异: R9声称-3/天 vs 推算-0.14/天，约21倍
```

**根本原因**: R9的模拟表数值无法从行为描述中推导出来，三方(行为描述/推算/模拟表)校验缺失。

**衰减计算方式详解**:

```
条件1豁免(每日登录玩家):
  - 判定: lastActiveDate >= yesterday
  - 效果: 对所有阵营统一豁免
  - 适用: #1全勤、#2活跃、#3轻度、#4登录不处理
  - 结果: 声望只增不减(或保持不变)

条件1+衰减混合(间歇登录玩家):
  - 登录日: 条件1豁免(lastActiveDate = today)
  - 登录次日: 条件1豁免(lastActiveDate = yesterday)
  - 其余日: 无豁免，执行衰减-1/天
  - 适用: #5周末、#6休闲
  - 计算: 非豁免天数 = 7 - 登录次数 - 登录次数(次日豁免)

无豁免(流失玩家):
  - 判定: lastActiveDate < yesterday
  - 效果: 每日衰减-1
  - 适用: #7流失
  - 结果: 声望单调递减至0
```

### MAP-F09-02-R10 A/B测试方法论补全 + 公理1验证方案 {#map-f09-02-r10}

> R10核心目标#2: 验证机制完善
> CH-MAP-R9-02: k=1.75 A/B测试方案缺少方法论关键要素，无法作为公理验证机制
> 修正原则: 补充A/B测试完整方法论，增加k=1.0对照组测试公理1，定义验证失败回退方案

#### 一、问题定位

R9的A/B测试方案存在两个核心缺陷:

**缺陷1: 方法论关键要素缺失**

| 缺失要素 | 为何必要 | R9是否提供 |

| 样本量计算 | 确保统计显著性 | 未提供 |

**缺陷2: 公理1验证盲区**

R9的A/B测试仅比较k=1.5/1.75/2.0，但公理1("低兵力突袭应当有风险惩罚")本身是否合理不在测试范围内。

```
测试盲区:
  A/B测试覆盖: 公理3(k值选择) → 1.5 vs 1.75 vs 2.0
  A/B测试未覆盖: 公理1(是否应有惩罚) → 有惩罚 vs 无惩罚
```

#### 二、修正方案: A/B测试完整方法论

**A/B测试方案(R10完整版)**:

```
测试名称: 胜利损耗k值优化测试

一、测试设计

  1. 测试假设:
     - H0(零假设): k=1.0(无额外惩罚)的玩家体验指标不劣于k=1.75
     - H1(备择假设): k=1.75的玩家体验指标优于k=1.0

  2. 测试组设计:
     - 对照组A: k=1.0(无额外惩罚) — 测试公理1是否成立
     - 实验组B: k=1.5(保守方案)
     - 实验组C: k=1.75(当前方案)
     - 实验组D: k=2.0(激进方案)

  3. 分流策略:
     - 分流方式: 玩家ID哈希取模，确保同一玩家始终在同一组
     - 分流比例: 4组均分(各25%)
     - 分流时机: 玩家首次进入天下Tab时分配

二、样本量计算

  4. 样本量估算:
     - 主要指标: 战斗参与率(假设基线=30%)
     - 最小可检测效应(MDE): 5个百分点(30%→35%)
     - 统计显著性水平(α): 0.05(双尾检验)
     - 统计功效(1-β): 0.80
     - 计算公式: n = (Z_α/2 + Z_β)² x [p1(1-p1) + p2(1-p2)] / (p1-p2)²
     - 计算结果: 每组约1,000名活跃玩家
     - 总样本量: 4组 x 1,000 = 4,000名活跃玩家

  5. 样本量验证:
     - 假设日活跃玩家(DAU) = 10,000
     - 测试期间可收集样本: 10,000 x 14天 = 140,000人次
     - 4组均分: 35,000人次/组
     - 远超所需1,000名/组 → 样本量充足

三、测试持续时间

  6. 测试时长:
     - 最短时长: 2周(14天) — 确保覆盖工作日和周末
     - 建议时长: 4周(28天) — 观察长期行为变化
     - 提前终止条件: 若某组出现严重负面指标(如流失率>20%)，立即终止

四、主要成功指标

  7. 主要指标(决策依据):
     - 指标1: 战斗参与率 = 每日参与战斗的玩家数 / DAU
     - 判定标准: k=1.75组的战斗参与率显著高于k=1.0组(p<0.05)

  8. 次要指标(辅助判断):
     - 指标2: 低兵力(<3000)出征频率 — 评估惩罚是否抑制低兵力策略
     - 指标3: 重试令牌使用频率 — 评估令牌价值感知
     - 指标4: 玩家反馈问卷(胜利代价满意度) — 主观体验评估

五、统计分析方法

  9. 显著性检验:
     - 方法: 卡方检验(Chi-squared test)用于比例比较
     - 显著性水平: α = 0.05(双尾)
     - 多重比较校正: Bonferroni校正(4组比较，α_adjusted = 0.05/6 = 0.0083)

  10. 效应量评估:
      - 方法: Cohen's h效应量
      - 小效应: h = 0.2
      - 中效应: h = 0.5
      - 大效应: h = 0.8
      - 判定: 仅当效应量 >= 0.2(小效应)且p < 0.0083时，认为有实际意义

六、结果解读与决策规则

  11. 决策规则:
      - 场景A: k=1.75组显著优于k=1.0组 → 公理1成立，保留惩罚机制
      - 场景B: k=1.75组与k=1.0组无显著差异 → 公理1存疑，需进一步测试
```

### MAP-F12-02-R10 期望值公式修正(移除沉没成本) {#map-f12-02-r10}

> R10核心目标#2: 验证机制完善
> CH-MAP-R9-05: 重试令牌期望值公式包含沉没成本(首次失败损耗)，夸大负面场景的负期望值
> 修正原则: 移除沉没成本，重新计算极端值验证表，增加"与不重试对比"列

#### 一、问题定位

R9的期望值公式包含沉没成本(首次失败损耗):

```
R9公式(错误):
E(重试令牌价值) = P(重试胜利) x (奖励价值 - 首次失败损耗 - 重试胜利损耗)
                + P(重试失败) x (-首次失败损耗 - 重试失败损耗 - 令牌价值)

问题: 首次战斗已失败，首次失败损耗是沉没成本，不应纳入重试决策
```

**经济学原理**:
- 沉没成本: 已经发生且无法收回的成本
- 决策原则: 沉没成本不应影响未来决策
- 应用: "是否使用重试令牌"的决策应基于重试的边际收益，而非已发生的首次失败损耗

#### 二、修正方案: 期望值公式修正

**正确的期望值公式(R10修正版)**:

```
E(重试 | 首次已失败) = P(胜利) x (奖励价值 - 重试胜利损耗)
                      + P(失败) x (-重试失败损耗)
                      - 令牌价值

与不重试对比:
E(不重试) = 0

决策规则: 若 E(重试) > 0，则重试; 否则不重试

其中:
  P(胜利) = 战斗胜率(与首次相同)
  P(失败) = 1 - 战斗胜率
  重试胜利损耗 = 出征兵力 x min(35%, 20% x 山贼战力/出征兵力)
  重试失败损耗 = 出征兵力 x 20%
  令牌价值 = 情报值兑换成本(10情报值/枚)
```

**与R9公式对比**:

```
R9公式 = 正确公式 + (-首次失败损耗)

即: R9的期望值比正确值低了一个"首次失败损耗"的量
```

### MAP-F12-02-R10a offlineEventCap/dailyLimit解绑评估 + FIFO策略定义 {#map-f12-02-r10a}

> R10核心目标#2: 验证机制完善
> CH-MAP-R9-03: offlineEventCap=5与dailyLimit=5的绑定是设计选择而非逻辑必然，每日多次登录场景仍未解决
> 修正原则: 评估解绑利弊，修正命名，正式定义FIFO替换策略

#### 一、问题定位

R9的offlineEventCap与dailyLimit绑定存在三个问题:

**问题1: 参数耦合风险**

```
offlineEventCap的目的: 防止长期离线后事件堆积(体验保护)
dailyLimit的目的: 限制每日情报值获取速率(经济平衡)

绑定后果:
  - 若offlineEventCap调整为8 → dailyLimit也会变为8 → 情报值经济平衡被打破
  - 若offlineEventCap调整为3 → dailyLimit变为3 → 玩家每日可处理事件数被压缩
```

**问题2: 命名误导**

```
"offlineEventCap"暗示"离线事件上限"，但实际是"单次登录事件堆积上限"
每日多次登录场景(早中晚各一次)可处理6个事件，超过offlineEventCap=5
```

**问题3: FIFO替换策略未定义**

```
R9提到"新事件替换最旧事件(FIFO)"，但未定义:
  - 被替换的事件是否永久丢失?
  - 替换时是否通知玩家?
  - 高价值事件替换是否合理?
```

#### 二、修正方案: 参数解绑评估

**解绑方案对比**:

| 方案 | offlineEventCap | dailyLimit | 优点 | 缺点 |
|------|----------------|------------|------|------|
| 方案A(保持绑定) | 5 | 5(=offlineEventCap) | 简单，无割裂感 | 参数耦合，无法独立调优 |
| **方案B(解绑)** | 5 | 5(独立定义) | 参数独立，灵活调优 | 需解释为何两者相等 |
| 方案C(解绑+不同值) | 8 | 5(独立定义) | 离线存储更大，收益限制更严 | 命名需调整 |

**推荐方案: 方案B(解绑)**

```
理由:
  1. offlineEventCap和dailyLimit服务于不同目的，解绑更合理
  2. 当前两者恰好相等(=5)是巧合，非逻辑必然
  3. 解绑后可独立调优，适应不同运营阶段

  eventSystem.offlineEventCap = 5

    目的: 限制每日情报值获取速率
    调整依据: 情报值经济平衡
    独立定义: 与offlineEventCap无关
```

#### 三、修正方案: 命名修正

**命名修正(R10)**:

| 原命名 | 新命名 | 修正理由 |
|--------|--------|---------|
| offlineEventCap | loginEventQueueSize | 更准确反映语义: "单次登录事件队列大小" |

**新命名定义**:

```

类型: integer
默认值: 5
定义: 玩家单次登录时，离线事件队列的最大长度。超过此上限时，新事件替换最旧事件(FIFO)。

  2. 队列大小=5的合理性: 5个事件约等于离线1.7-5天的累计量，玩家登录后5-10分钟可处理完毕

  - 两者独立，无绑定关系
  - 当前值恰好相等(=5)是巧合，非逻辑必然
  - 可独立调优，适应不同运营阶段
```

#### 四、修正方案: FIFO替换策略定义

**FIFO替换策略(R10正式定义)**:

```
策略名称: FIFO替换(First In, First Out)

规则:
  1. 当事件队列已满(loginEventQueueSize=5)且有新事件到达时:
     - 移除队列中最旧的事件(最早到达的)
     - 将新事件添加到队列末尾

  2. 被替换事件的处理:
     - 被替换的事件永久丢失
     - 玩家不会收到被替换事件的通知
     - 被替换事件的情报值奖励不会发放

  3. 高价值事件替换:
     - 所有事件平等，不区分价值高低
     - 高价值事件(如精英山贼)可能被低价值事件(如普通山贼)替换
     - 这是FIFO策略的固有特性，非bug

  4. 替换通知(可选优化):
     - 当前: 不通知
     - 可选: 在玩家登录时显示"您有X个离线事件因队列已满被替换"
     - 评估: 是否需要通知取决于玩家体验测试

伪代码:
  function addEventToQueue(playerId, event):
    queue = player.eventQueue

    if queue.length >= loginEventQueueSize:
      removedEvent = queue.shift()  // 移除最旧事件
      // 可选: 记录被替换事件用于统计
      logEventReplacement(playerId, removedEvent, event)

    queue.push(event)  // 添加新事件

  function processEventQueue(playerId):
    queue = player.eventQueue
    totalReward = 0

    while queue.length > 0:
      event = queue.shift()
      reward = calculateReward(event)
      totalReward += reward

    return totalReward
```

### MAP-F06-07-R10 冷却管理器代码优化(位掩码替代Map) {#map-f06-07-r10}

> R10核心目标#3: 模拟矩阵补全
> CH-MAP-R9-04: 冷却管理器_scanAllCooldowns()遍历Map时调用delete()，存在迭代期间修改集合的隐患
> CH-MAP-R9-07: factionEventToday用Map引入不必要的数据结构复杂度
> 修正原则: 先收集后删除+移除_processed标记+评估位掩码方案

#### 一、问题定位

**问题1: 遍历期间调用delete()(CH-MAP-R9-04)**

R9的_scanAllCooldowns()在遍历Map时调用delete():

```javascript
// R9代码(有问题):
for [cityId, state] of cooldowns:
  if state.status === "active" && state.endAt - now <= 0:
    if !state._processed:
      state._processed = true
      cooldowns.delete(cityId)  // 遍历期间删除!
```

虽然ECMAScript规范保证了Map遍历期间delete()的安全性，但:
- 代码审查者可能误认为是bug
- 遍历期间修改集合是代码异味

**问题2: _processed标记冗余(CH-MAP-R9-04)**

R9的_processed标记在JavaScript单线程模型中实际冗余:

```javascript
// R9代码:
if !state._processed:
  state._processed = true
  state.status = "ended"
  cooldowns.delete(cityId)

// 实际去重逻辑:
// - getCooldown()先执行: cooldowns.delete(cityId) → getCooldown()返回null
// - _scanAllCooldowns()先执行: cooldowns.delete(cityId) → getCooldown()返回null
// _processed标记永远不会被用于去重
```

**问题3: factionEventToday用Map(CH-MAP-R9-07)**

R9用Map<faction, boolean>存储factionEventToday，引入不必要的复杂度:

```javascript
// R9方案:
player.factionEventToday = new Map([
  [魏, false], [蜀, false], [吴, false]
])

// 问题:
// - 每日重置需创建10万个Map对象(10万活跃玩家)
// - 批量创建增加GC压力
```

#### 二、修正方案: 先收集后删除

**修正后(R10)_scanAllCooldowns()代码**:

```javascript
_scanAllCooldowns():
  now = now()
  toDelete = []  // 收集待删除的cityId

  // 阶段1: 遍历收集
  for [cityId, state] of cooldowns:
    if state.status === "active" && state.endAt - now <= 0:
      toDelete.push(cityId)

  // 阶段2: 统一删除
  for cityId of toDelete:
    state = cooldowns.get(cityId)
    state.status = "ended"
    emit("cooldownStateChanged", { cityId, status: "ended" })
    cooldowns.delete(cityId)

  if cooldowns.size === 0:
    stopScan()
```

**优势**:
- 遍历期间不修改集合，避免代码异味
- 逻辑清晰: 阶段1(收集) → 阶段2(删除)
- 代码审查者不会误认为是bug

#### 三、修正方案: 移除_processed标记

**修正后(R10)完整代码**:

```javascript
cooldownManager = {
  cooldowns: Map<cityId, CooldownState>,
  listeners: Set<callback>,
  _scanTimer: null,
  _scanInterval: 10000,  // 10s扫描间隔

  // R10修正: 移除_processed标记
  getCooldown(cityId):
    state = cooldowns.get(cityId)
    if !state: return null

    remaining = state.endAt - now()
    if remaining <= 0 && state.status === "active":
      // R10: 直接清理，无需_processed标记
      state.status = "ended"
      emit("cooldownStateChanged", { cityId, status: "ended" })
      cooldowns.delete(cityId)
      return null

    return { status: state.status, remaining }

  // R10修正: 先收集后删除，移除_processed标记
  _scanAllCooldowns():
    now = now()
    toDelete = []

    for [cityId, state] of cooldowns:
      if state.status === "active" && state.endAt - now <= 0:
        toDelete.push(cityId)

    for cityId of toDelete:
      state = cooldowns.get(cityId)
      state.status = "ended"
      emit("cooldownStateChanged", { cityId, status: "ended" })
      cooldowns.delete(cityId)

    if cooldowns.size === 0:
      stopScan()

  // R9保留: destroy()生命周期方法
  destroy():
    stopScan()
    cooldowns.clear()
    listeners.clear()

  // R10新增: setCooldown()完整实现(含destroy后惰性初始化)
  setCooldown(cityId, duration):
    if !this.cooldowns:                    // destroy后惰性初始化
      this.cooldowns = new Map()
      this.listeners = new Set()
      this.startScan()                     // 重新启动扫描

    this.cooldowns.set(cityId, {
      status: "active",
      endAt: now() + duration
    })

    if !this._scanTimer:
      this.startScan()
}
```

**优势**:
- 移除冗余的_processed标记，简化代码
- 去重逻辑由delete()/get()返回null实现，无需额外标记
- setCooldown()包含destroy后惰性初始化，避免遗漏

> R9方法论缺陷: 期望值分析纳入沉没成本

R10在决策分析流程中增加"沉没成本检查"步骤:

```
R10决策分析验证流程:
  1. 明确决策点: "是否使用重试令牌"
  2. 列出相关成本: 重试胜利损耗、重试失败损耗、令牌价值
  3. 沉没成本检查: 首次失败损耗是否已发生? → 是，则排除

```

**修订方案**: 修正为-0.14/天，增加三方交叉校验步骤，8类玩家逐类校验
**优先级**: P0

### PRD-R034 A/B测试方法论补全

**PRD原文**: R9的A/B测试方案缺关键要素
**问题分析**: 样本量/持续时间/主指标/阈值/分流/效应量/对照均未定义
**修订方案**: 补充完整方法论(14要素)，增加k=1.0对照组测试公理1
**优先级**: P1

### PRD-R035 期望值公式修正(移除沉没成本)

**PRD原文**: R9的期望值公式包含沉没成本
**问题分析**: 首次失败损耗是沉没成本，不应纳入重试决策
**修订方案**: 移除沉没成本，重新计算极端值验证表，增加"与不重试对比"列
**优先级**: P1

### PRD-R036 offlineEventCap/dailyLimit解绑+命名修正

**PRD原文**: R9的offlineEventCap与dailyLimit绑定
**问题分析**: 参数耦合，无法独立调优，命名误导
**修订方案**: 解绑，重命名为loginEventQueueSize，正式定义FIFO策略
**优先级**: P1

### PRD-R037 间歇性重度玩家类型+声望波动性

**PRD原文**: R9的7类玩家缺间歇性重度类型
**问题分析**: 模拟矩阵不完整，声望波动性未分析
**修订方案**: 增加第8类玩家，增加声望变化模式分析维度

### PRD-R038 冷却管理器代码优化

**PRD原文**: R9的_scanAllCooldowns()遍历期间delete()
**问题分析**: 代码异味，_processed标记冗余
**修订方案**: 先收集后删除，移除_processed标记，setCooldown()含惰性初始化
**优先级**: P2

### PRD-R039 factionEventToday位掩码优化

**PRD原文**: R9用Map<faction, boolean>
**问题分析**: 每日创建大量Map对象，GC压力大
**修订方案**: 改为位掩码(number)，重置为0，提供辅助函数
**优先级**: P2
