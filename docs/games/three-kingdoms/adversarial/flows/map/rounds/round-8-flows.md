### MAP-F09-02-R8 胜利损耗上限数值推导与修正 {#map-f09-02-r8}

> R8核心目标#1: 设计矛盾清零
> CH-MAP-R7-01: 50%上限导致"胜利比失败损失更大"的设计矛盾，缺乏数值推导依据
> 修正原则: 确定有推导依据的上限值，修正文档错误声称

#### 一、问题定位

**R7的设计矛盾**:

R7文档第153行声称: "玩家不会因为'侥幸赢了'而遭受比'故意输'更惨的损失"

但R7自身数据表(第170-176行)显示:

| 场景 | 胜利损耗 | 失败损耗 | 胜利/失败比 | 与声称矛盾? |
|------|---------|---------|-----------|-----------|
| Lv70带10000兵 | 12.6% | 20% | 0.63x | 不矛盾(胜利更省) |
| Lv70带3000兵 | 42% | 20% | 2.10x | 矛盾(胜利更贵) |
| Lv70带1000兵 | 50% | 20% | **2.50x** | **严重矛盾** |
| Lv70带500兵 | 50% | 20% | **2.50x** | **严重矛盾** |

**矛盾本质**: R7的50%上限解决了溢出BUG(技术正确)，但引入了"低兵力场景下胜利代价是失败的2.5倍"的设计矛盾。R7文档声称与数据表自相矛盾。

#### 二、上限值推导

**设计意图明确化**:

```
设计意图(修正后):
  1. 胜利应当比失败损失更少——这是"胜利奖励"的核心体验
  2. 但低兵力突袭(以少打多)应当有风险惩罚——防止无脑以少打多
  3. 因此: 胜利损耗可以高于失败损耗，但不应过高
  4. 合理范围: 胜利/失败比在[1.0, 2.0]之间
     - 1.0x = 胜败持平(无战斗动机，排除)
     - 1.5x = 胜利有代价但不极端
     - 2.0x = 胜利代价的合理上限
     - 2.5x = R7当前值(过高，排除)
```

**候选上限值对比分析**:

| 上限值 | Lv70/10000兵 | Lv70/3000兵 | Lv70/1000兵 | Lv70/500兵 | 胜/败比范围 | 评价 |
|--------|-------------|------------|------------|-----------|-----------|------|
| 20%(=失败) | 12.6% | 20% | 20% | 20% | 0.63~1.0x | 胜败持平，无战斗动机 |
| 25% | 12.6% | 25% | 25% | 25% | 0.63~1.25x | 胜利始终优于失败，但惩罚偏弱 |
| 30% | 12.6% | 30% | 30% | 30% | 0.63~1.50x | 胜利有代价，1.5x可接受 |
| **35%** | **12.6%** | **35%** | **35%** | **35%** | **0.63~1.75x** | **胜利代价合理，1.75x为上限** |
| 40% | 12.6% | 40% | 40% | 40% | 0.63~2.00x | 接近边界，2.0x偏高 |
| 50%(R7) | 12.6% | 42% | 50% | 50% | 0.63~2.50x | 代价过高，设计矛盾 |

**选择35%的推导过程**:

```
推导逻辑:
  1. 失败损耗 = 20%(固定)
  2. 设胜利/失败比上限 = k
  3. 则胜利损耗上限 = 20% × k
  4. k的合理范围 = [1.0, 2.0]
     - k < 1.0: 胜利代价低于失败，正常场景下胜利划算(正确)
     - k = 1.0~1.5: 低兵力突袭有轻微惩罚(保守设计)
     - k = 1.5~2.0: 低兵力突袭有明显惩罚(鼓励合理兵力配置)
     - k > 2.0: 低兵力突袭惩罚过重，胜利不如失败(设计矛盾)
  5. 选择 k = 1.75:
     - 理由: 1.75x位于[1.5, 2.0]区间中值偏上
     - 效果: 低兵力胜利代价为35%，比失败(20%)多75%兵力损耗
     - 体验: 玩家能感受到"以少打多的代价"，但不至于"赢了反而亏"
     - 数学: 35% = 20% × 1.75，推导链条完整
  6. 验证极端场景:
     - Lv70带1000兵打山贼王(战力6300): max(5%, 126%)=126% → min(35%, 126%)=35%
     - 胜利损耗35% vs 失败损耗20% → 1.75倍 → 在合理范围内
     - Lv70带10000兵: 12.6% < 35% → 不触发截断 → 胜利划算(正确)
```

#### 三、修正方案

**修正后(R8)胜利损耗公式**:
```
预计兵力损失(胜利时) = 出征兵力 × min(35%, max(5%, 20% × (山贼战力 / 玩家总兵力)))
```

**修正前后公式对比**:

| 组成部分 | R7(修正前) | R8(修正后) | 变化说明 |
|---------|-----------|-----------|---------|
| 下限保护 | max(5%, ...) | max(5%, ...) | 不变 |
| 损耗计算 | 20% x (山贼战力/玩家兵力) | 20% x (山贼战力/玩家兵力) | 不变 |
| 上限截断 | min(**50%**, ...) | min(**35%**, ...) | **下调: 50% → 35%** |
| 失败损耗 | 出征兵力 x 20% | 出征兵力 x 20% | 不变 |

**R7文档错误声称修正**:

| 位置 | R7原文(错误) | R8修正 |
|------|------------|--------|
| 第153行 | "玩家不会因为'侥幸赢了'而遭受比'故意输'更惨的损失" | "胜利损耗上限35%，在极端低兵力场景下胜利代价(35%)高于失败(20%)，比值1.75x，是对低兵力突袭的风险惩罚" |

### MAP-F12-02-R8 情报值跨轮一致性修正 {#map-f12-02-r8}

> R8核心目标#1: 设计矛盾清零
> CH-MAP-R7-02: R6情报值定义"每日3次"与R7"每日5次"矛盾
> 修正原则: 统一选择dailyLimit=5并说明R6选择3的理由差异

#### 一、问题定位

**R6 vs R7矛盾**:

| 参数 | R6定义 | R7定义 | 矛盾点 |
|------|--------|--------|--------|
| dailyLimit | 3 | 5 | 不一致 |
| maxCap | 20 | 100 | 不一致 |
| 理由 | "山贼每日最多触发3次" | "离线事件上限5个" | 触发频率假设不同 |

**R7未解释修改理由**: R7声称"落实R6全部9项质疑"，但修改了R6建议的数值而未说明——这是"修改"而非"落实"。

#### 二、统一选择: dailyLimit=5

**选择dailyLimit=5的理由**:

```
推导过程:
  1. 情报值来源: 山贼快速处理"绕行"分支
  2. 山贼事件触发机制: 离线期间累计，登录时批量展示
  3. 离线事件上限: 系统设计为5个(防止长期离线后事件堆积)
  4. 因此: 每日山贼快速处理上限 = 离线事件上限 = 5次
  5. R6选择3次的假设:
     - R6假设"山贼每日最多触发3次"(可能是基于在线触发频率)
     - 但山贼事件主要在离线期间累计，而非在线触发
     - R6的假设忽略了离线累计场景
  6. R7选择5次的假设:
     - R7基于"离线事件上限5个"
     - 更准确反映了玩家实际可处理的山贼事件数量
  7. 结论: dailyLimit=5更符合实际游戏场景

  补充验证:
    - 若选择dailyLimit=3: 玩家每日处理5个山贼事件，但只有3个给情报值 → 体验割裂
    - 若选择dailyLimit=5: 每个山贼事件都有情报值收益 → 体验一致
```

**R6差异标注(文档一致性修正)**:

| 参数 | R6原始值 | R8最终值 | 差异说明 |
|------|---------|---------|---------|
| dailyLimit | 3 | **5** | R6假设在线触发频率(3次/日)，R8基于离线事件上限(5个/日)，选择5以匹配实际游戏场景 |
| maxCap | 20 | **100** | R6为最小化测试值，R8扩展为正式参数(100=20天满仓，合理的长期积累上限) |

### MAP-F12-01-R8a 声望衰减时序边界修正 + 快速处理豁免矛盾修正 {#map-f12-01-r8a}

> R8核心目标#1: 设计矛盾清零
> CH-MAP-R7-04: R6"快速处理不触发声望衰减"与R7每日-1自然衰减矛盾
> CH-MAP-R7-07: 00:00定时任务时"当日活跃"判定边界问题
> 修正原则: "昨日活跃"判定 + 快速处理纳入豁免条件

#### 一、问题定位

**问题1: 时序边界(CH-MAP-R7-07)**

```
R7定义:
  衰减触发: 每日00:00(服务器时间)
  豁免条件1: player.lastActiveDate == today

时序问题:
  23:59 → 玩家最后操作，lastActiveDate = "2026-05-01"
  00:00 → 定时任务执行，today = "2026-05-02"
  → lastActiveDate("2026-05-01") != today("2026-05-02")
  → 条件1不满足 → 玩家被衰减-1(尽管23:59还在活跃)
```

**问题2: 快速处理豁免矛盾(CH-MAP-R7-04)**

```
R6声明: "快速处理的阵营冲突不触发声望衰减"
R7条件2: player.factionEventToday == true
R7排除: "不包括: 阵营冲突快速处理"

矛盾: 只做快速处理的玩家 → 不满足条件1/2/3 → 被衰减-1
      这与R6的"不触发声望衰减"声明直接冲突
```

#### 二、修正方案

**修正后(R8)豁免条件**:
```
豁免条件(满足任一则当日不衰减):

条件1: 昨日活跃(修正: "当日"改为"昨日")
  - 玩家昨日登录过(有过任意操作)
  - 判定: player.lastActiveDate >= yesterday 00:00
  - 修正理由: 00:00定时任务执行时，"当日"刚刚开始
     玩家昨日活跃应获得豁免，而非要求"今日"(今日才刚开始)
  - 效果: 23:59活跃的玩家不会被错误衰减

条件2: 当日声望事件(修正: 含快速处理)
  - 玩家当日处理过阵营相关事件
  - 判定: player.factionEventToday == true
  - 包括: 阵营冲突手动处理、阵营冲突快速处理、阵营任务完成、声望道具使用
  - 修正理由: R6声明"快速处理不触发声望衰减"，应将快速处理纳入豁免条件
  - 效果: 只做快速处理的玩家不被衰减，与R6声明一致

条件3: 声望为0
  - 不变: player.factionRep[faction] == 0
```

#### 三、修正前后对比

### MAP-F06-07-R8 冷却管理器主动检查机制 {#map-f06-07-r8}

> R8核心目标#3: 系统设计闭环
> CH-MAP-R7-05: 自然结束事件触发依赖被动调用getCooldown()，UI关闭时状态不一致
> 修正原则: 增加60s定时扫描，确保所有状态变化都有主动事件通知

#### 一、问题定位

**R7被动触发机制的问题**:

| 方法 | R7事件触发方式 | 依赖条件 | 问题 |
|------|------------|---------|------|
| setCooldown() | 主动触发 | 无 | 正常 |
| clearCooldown() | 主动触发 | 无 | 正常 |
| **自然结束** | **被动触发** | **需要getCooldown()被调用** | **UI关闭时事件不触发** |

**问题场景**:
```
  10:00 → 攻城失败，setCooldown(cityA, 24h) → 冷却开始
  次日09:59 → 玩家切换到其他页面(天下Tab关闭)
  次日10:00 → 冷却自然结束，但无UI调用getCooldown()
  → cooldownState停留在status: "active"
  → 直到玩家重新打开天下Tab才触发ended事件
  → 状态不一致时间窗口: 最长可达数小时
```

#### 二、修正方案: 主动60s定时扫描

**修正后(R8)cooldownManager自然结束机制**:
```
cooldownManager = {
  // R7原有字段
  cooldowns: Map<cityId, CooldownState>,
  listeners: Set<callback>,
  // R8新增
  _scanTimer: null,  // 定时扫描器

  // R8新增: 启动定时扫描
  startScan():
    if _scanTimer: return  // 防止重复启动
    _scanTimer = setInterval(60s, () => {
      this._scanAllCooldowns()
    })

  // R8新增: 扫描所有冷却状态
  _scanAllCooldowns():
    now = now()
    for [cityId, state] of cooldowns:
      if state.exposed && state.status === "active":
        remaining = state.endAt - now
        if remaining <= 0:
          // 冷却已结束，主动触发事件
          state.exposed = false
          state.status = "ended"
          emit("cooldownStateChanged", { cityId, status: "ended" })
          cooldowns.delete(cityId)

  // R8新增: 停止扫描(无冷却时节省资源)
  stopScan():
    if _scanTimer:
      clearInterval(_scanTimer)
      _scanTimer = null
}
```

**修正前后对比**:

| 行为 | R7(修正前) | R8(修正后) | 改进 |
|------|-----------|-----------|------|
| 自然结束检测 | getCooldown()被动检测 | **60s定时扫描主动检测** | 消除状态不一致窗口 |
| 事件触发时机 | 下次UI调用时 | **冷却结束后60s内** | 及时通知 |
| 状态不一致窗口 | 数小时(UI关闭期间) | **最长60s** | 缩短至可忽略 |
| 资源消耗 | 零(被动) | **每60s一次扫描** | 极低(仅遍历Map) |

**定时扫描启动/停止策略**:
```
策略:
  1. setCooldown()时: 若_scanTimer为null，启动定时扫描
  2. 每次_scanAllCooldowns()后: 若cooldowns为空，停止定时扫描
  3. 效果: 仅在有活跃冷却时运行定时器，无冷却时零开销

代码:
  function setCooldown(cityId, startAt, duration):
    // ...原有逻辑...
    cooldowns.set(cityId, state)
    emit("cooldownStateChanged", { cityId, status: "active", remaining: duration })
    startScan()  // R8新增: 确保定时扫描运行

  function _scanAllCooldowns():
    // ...扫描逻辑...
    if cooldowns.size === 0:
      stopScan()  // R8新增: 无冷却时停止扫描
```

#### 三、修正后完整事件触发覆盖

| 状态变化 | 触发方式 | 事件通知 | 状态一致性 |
|---------|---------|---------|----------|
| 冷却开始(setCooldown) | 主动触发 | 立即 | 立即一致 |
| 冷却被清除(clearCooldown) | 主动触发 | 立即 | 立即一致 |
| **冷却自然结束** | **主动扫描(60s)** | **60s内** | **60s内一致** |

> R8修正后，所有状态变化都有主动事件通知，不依赖UI调用。
> cooldownManager独立运行，符合"管理器独立管理状态"的设计原则。

**数据变更**:

| 数据 | 变更前(R7) | 变更后(R8) | 来源 |
|------|-----------|-----------|------|
| cooldownManager._scanTimer | 不存在 | **60s定时扫描器** | CH-MAP-R7-05 |
| cooldownManager.startScan() | 不存在 | **启动扫描** | CH-MAP-R7-05 |
| cooldownManager._scanAllCooldowns() | 不存在 | **扫描所有冷却** | CH-MAP-R7-05 |
| cooldownManager.stopScan() | 不存在 | **停止扫描** | CH-MAP-R7-05 |

## R8 PRD修订建议(PRD-R)

### PRD-R020 胜利损耗上限修正(50% → 35%)

**PRD原文**: R7定义 min(50%, max(5%, ...))
**问题分析**: 50%上限导致胜利代价2.5倍于失败，设计矛盾
**修订方案**: 修正为 min(35%, max(5%, ...))，附k=1.75推导过程
**优先级**: P0

### PRD-R021 声望衰减时序边界修正

**PRD原文**: R7定义条件1为"当日活跃(today)"
**问题分析**: 00:00定时任务执行时序边界导致活跃玩家被错误衰减
**修订方案**: 条件1改为"昨日活跃(yesterday)"，条件2纳入快速处理
**优先级**: P0

### PRD-R022 声望获取速率定义

**PRD原文**: 无(声望衰减每日-1但未定义获取速率)
**问题分析**: 衰减"温和"判断无数据支撑
**修订方案**: 在声望章节补充: 手动处理+3/快速处理+1/阵营任务+5/声望道具+10/阵营活动+2
**优先级**: P1

### PRD-R023 冷却管理器主动扫描

**PRD原文**: R7定义自然结束依赖getCooldown()被动检测
**问题分析**: UI关闭时状态不一致
**修订方案**: cooldownManager增加60s定时扫描机制，setCooldown()时启动，无冷却时停止
**优先级**: P1

### PRD-R024 战斗重试令牌精确定义

**PRD原文**: R7定义"不消耗额外兵力"(歧义)
**问题分析**: 3种解释导致开发歧义
**修订方案**: 精确定义5要素(触发时机+效果+消耗+失败处理+限制)
**优先级**: P2

### PRD-R025 情报值跨轮差异标注

**PRD原文**: R7定义dailyLimit=5但未说明与R6(=3)的差异
**问题分析**: 开发者同时参考R6/R7时无法确定实现哪个
**修订方案**: 在情报值章节增加"与R6差异说明"注释
**优先级**: P2
