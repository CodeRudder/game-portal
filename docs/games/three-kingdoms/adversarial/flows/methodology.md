# 游戏流程枚举对抗式测试方法论

> **版本**: v2.0 | **日期**: 2026-05-02
> **基于**: 3-Agent对抗式测试方法论 v2.0 扩展
> **目的**: 从游戏玩法/PRD/UI出发，系统性枚举游戏流程，**修正完善PRD设计**，发现流程断裂和缺失功能
> **核心理念**: 不仅是完整列举PRD的流程，更要**挑战PRD设计本身**，提升游戏可玩性

---

## 一、与API级对抗测试的区别

| 维度 | API级对抗测试 | 流程级对抗测试 |
|------|-------------|-------------|
| **测试对象** | 单个API的输入/输出/边界 | 玩家操作→系统响应→数据变更的完整链路 |
| **视角** | 开发者视角（代码逻辑） | 玩家视角（游戏体验） |
| **覆盖维度** | F-Normal/Boundary/Error/Cross/Lifecycle | 玩法完备性/流程连贯性/数据可追溯性/**PRD设计质量**/可玩性 |
| **输出** | 流程分支树（API级） | 游戏流程图（玩法级）+ **PRD修订建议** |
| **发现问题类型** | 代码缺陷（NaN/null/竞态） | 设计缺陷（流程断裂/**PRD不合理**/节奏失衡/缺少玩法） |

### 核心差异：流程级测试的双重目标

```
目标1: 枚举PRD已定义的流程 → 确保实现完整
目标2: 挑战PRD设计本身 → 发现设计缺陷并提出改进方案
       ↓
       PRD不是圣经，而是需要不断完善的草案
```

## 二、流程编号规范

### 2.1 编号格式

```
{子系统代码}-F{序号}-{子序号}

示例:
  BLD-F01     — 建筑系统 第1个主流程
  BLD-F01-01  — 建筑系统 第1个主流程 第1个子流程
  BLD-F01-01-A — 建筑系统 第1个主流程 第1个子流程 分支A
```

### 2.2 子系统代码

| 代码 | 子系统 | 代码 | 子系统 |
|------|--------|------|--------|
| BLD | 建筑系统 | EVT | 事件系统 |
| RES | 资源系统 | NPC | NPC系统 |
| HER | 武将系统 | SHP | 商店系统 |
| CBT | 战斗系统 | EQP | 装备系统 |
| CPN | 攻城系统 | ALC | 联盟系统 |
| TEC | 科技系统 | TRD | 贸易系统 |
| MAP | 地图系统 | OFR | 离线收益 |

### 2.3 流程分类

| 分类 | 代号 | 说明 |
|------|------|------|
| 主流程 | F | 玩家核心操作链路（必须覆盖） |
| 子流程 | F-XX | 主流程中的分支/变体 |
| 跨系统流程 | XI | 涉及多个子系统的流程 |
| 触发事件 | TE | 系统自动触发的流程（定时器/条件满足） |
| **PRD修订** | **PRD-R** | **对PRD设计的修订建议** |

## 三、3-Agent角色定义（流程级）

### 3.1 FlowBuilder（流程构建者）

| 属性 | 说明 |
|------|------|
| **职责** | 从PRD/UI/源码出发，枚举所有玩家可操作的游戏流程，**评估PRD设计质量** |
| **输入** | PRD文档 + UI布局 + 引擎源码 + 上一轮Arbiter改进建议 |
| **输出** | 子系统流程文档（flows.md）+ **PRD修订建议** |
| **关注点** | 大步骤为主，关键细节为辅；关注玩法完备性；**挑战PRD设计** |

**FlowBuilder工作准则**:
1. 从PRD的需求编号出发，枚举每个需求对应的玩家操作流程
2. 从UI布局的按钮/面板出发，枚举每个交互入口的完整流程
3. 关注大的步骤，不需要枚举补充具体细节（除非是关键细节）
4. 跨系统流程通过流程编号衔接，避免重复编写
5. 标记每个数据的产生来源（玩家操作/触发事件/系统计算）
6. 发现流程断裂/缺失功能时，必须标记断裂点和改进建议
7. **【新增】评估PRD中每个实体/数值/机制的设计合理性，标记不合理之处**
8. **【新增】提出PRD修订建议（新增实体/调整数值/补充机制）**

**FlowBuilder PRD评估维度**:

| 评估维度 | 检查内容 | 示例 |
|---------|---------|------|
| 实体数量 | 子系统中的实体是否足够支撑玩法深度 | 建筑只有8座，是否太少？ |
| 实体角色 | 每个实体是否有明确的、不可替代的角色 | 医馆和书院是否功能重叠？ |
| 数值合理性 | 数值曲线是否平衡，是否存在无意义的值 | 产出太低/费用太高？ |
| 机制完备性 | 是否缺少核心机制（如交互、反馈、目标） | 建筑之间无协同？ |
| 玩法深度 | 是否有足够多的有意义的决策点 | 升级策略是否单一？ |
| 长期目标 | 是否有清晰的终局目标和追求 | 满级后做什么？ |

### 3.2 FlowChallenger（流程挑战者）

| 属性 | 说明 |
|------|------|
| **职责** | 从可玩性/完备性/连贯性角度质疑流程的合理性，**重点挑战PRD设计缺陷** |
| **输入** | FlowBuilder输出的流程文档 + PRD + 引擎源码 |
| **输出** | 流程质疑清单（gaps and issues）+ **PRD设计缺陷清单** |
| **关注点** | 流程断裂、缺失功能、可玩性问题、**PRD设计不合理** |

**FlowChallenger工作准则**:
1. 检查每个流程是否有明确的起点和终点
2. 检查流程之间是否有断裂（前置条件未满足、数据缺失）
3. 检查是否有玩家可操作但未覆盖的流程
4. 检查数据流转是否完整（产出→消耗→反馈循环）
5. 评估流程的可玩性（节奏、反馈、策略深度）
6. 从玩家角度追问"然后呢？"来发现流程终点断裂
7. **【新增】挑战PRD中每个实体的设计：这个实体存在的意义是什么？去掉它游戏有区别吗？**
8. **【新增】挑战PRD中的数值：这个数值让玩家有什么感受？是否合理？**
9. **【新增】挑战PRD中的机制：这个机制是否有趣？是否有更好的替代方案？**

**FlowChallenger PRD质疑模板**:

```markdown
### PRD质疑: {实体/数值/机制名称}

**PRD原文**: 引用PRD中的具体描述
**质疑**: 这个设计存在什么问题？
**玩家视角**: 玩家会怎么看待/使用这个功能？
**缺失分析**: 如果去掉这个实体/修改这个数值，游戏会怎样？
**改进建议**: 具体的改进方案
**影响评估**: 改进后对游戏体验的影响
```

**FlowChallenger 追问清单（PRD设计专用）**:

| 追问 | 目的 | 示例（建筑系统） |
|------|------|----------------|
| "这个实体有存在感吗？" | 发现无意义的实体 | 医馆的恢复速率+3%有意义吗？ |
| "玩家会主动使用这个功能吗？" | 发现被动/无人问津的功能 | 书院产出科技点，玩家需要主动做什么？ |
| "这个数值让玩家有什么感受？" | 发现数值体验问题 | Lv10后升级需要5小时，玩家会等吗？ |
| "这个机制有策略深度吗？" | 发现浅层机制 | 升级建筑只是点一下等待，有决策吗？ |
| "缺少什么会让游戏更完整？" | 发现缺失机制 | 建筑之间没有互动，是否需要协同？ |
| "这个设计和其他游戏比如何？" | 参考行业最佳实践 | 其他SLG的建筑系统是怎么设计的？ |
| "终局玩家在这里做什么？" | 发现后期内容缺失 | 满级建筑后还能做什么？ |
| "新手在这里会困惑吗？" | 发现新手体验问题 | 新手知道先升级哪个建筑吗？ |

### 3.3 FlowArbiter（流程仲裁者）

| 属性 | 说明 |
|------|------|
| **职责** | 评估流程覆盖完备性，裁决可玩性问题优先级，**评判PRD修订建议的合理性** |
| **输入** | FlowBuilder的流程文档 + FlowChallenger的质疑清单 |
| **输出** | 评分 + 裁决 + **PRD修订决策** + 改进方向 |
| **关注点** | 流程覆盖度、可玩性评分、断裂点优先级、**PRD设计质量** |

**FlowArbiter工作准则**:
1. 从5个维度评分（流程完备性/数据可追溯性/可玩性/**PRD设计质量**/跨系统连贯性）
2. 总分 = 加权平均，封版线 9.0
3. 标记所有流程断裂点的优先级
4. 裁决FlowChallenger的质疑是否合理
5. **【新增】裁决PRD修订建议：接受/拒绝/延后**
6. **【新增】评估PRD修订对整体游戏设计的影响**
7. 指出下一轮改进方向

## 四、流程枚举维度

### 4.1 玩家操作流程（主流程）

从玩家视角出发的主动操作：

| 维度 | 说明 | 示例 |
|------|------|------|
| 核心循环 | 游戏的主要玩法循环 | 产出资源→升级建筑→提高产出 |
| 成长路径 | 玩家长线成长的里程碑 | 主城Lv1→5→10→20→30 |
| 决策点 | 玩家需要做出选择的时刻 | 先升农田还是先升兵营 |
| 反馈回路 | 操作后的即时反馈 | 升级完成→外观变化→产出提升 |

### 4.2 系统触发流程（TE）

系统自动触发的流程：

| 维度 | 说明 | 示例 |
|------|------|------|
| 定时触发 | 每隔N秒/分钟触发 | 建筑产出tick、升级计时 |
| 条件触发 | 满足条件时自动触发 | 主城升级→解锁新建筑 |
| 事件触发 | 响应玩家操作触发 | 点击升级→加入队列→开始计时 |
| 离线触发 | 重新登录时触发 | 离线收益结算、离线升级完成 |

### 4.3 跨系统流程（XI）

涉及多个子系统的流程：

| 维度 | 说明 | 示例 |
|------|------|------|
| 数据流转 | 资源从一个系统流向另一个系统 | 建筑产出→资源系统→科技消耗 |
| 加成注入 | 一个系统对另一个系统的效果 | 武将政治属性→建筑产出加成 |
| 前置条件 | 一个系统的状态作为另一个系统的条件 | 主城等级→建筑解锁 |
| 联动效果 | 一个系统的操作影响另一个系统 | 城墙等级→攻城防御值 |

### 4.4 【新增】PRD设计评估维度（PRD-R）

对PRD本身的设计进行评估和修订：

| 维度 | 检查内容 | 输出 |
|------|---------|------|
| 实体完备性 | 实体数量是否足够、角色是否清晰 | PRD-R: 新增/合并/删除实体建议 |
| 数值平衡性 | 数值曲线是否合理、是否有无意义的值 | PRD-R: 数值调整建议 |
| 机制有趣度 | 机制是否有策略深度、是否有更好的替代 | PRD-R: 机制改进/新增建议 |
| 玩法深度 | 是否有足够的决策点和长期目标 | PRD-R: 玩法扩展建议 |
| 体验流畅度 | 节奏是否合理、反馈是否及时 | PRD-R: 体验优化建议 |

## 五、迭代规则

### 5.1 迭代流程

```
每轮迭代（每个子系统独立执行）:

  Step 1: FlowBuilder
    - 枚举PRD已定义的流程
    - 【新增】评估PRD设计质量，提出修订建议
    → 输出: flows.md + PRD修订建议

  Step 2: FlowChallenger
    - 质疑流程完备性
    - 【新增】挑战PRD设计缺陷（实体/数值/机制）
    → 输出: challenges.md + PRD设计缺陷清单

  Step 3: FlowArbiter
    - 评分+裁决
    - 【新增】评判PRD修订建议（接受/拒绝/延后）
    → 输出: verdict.md + PRD修订决策

  Step 4: 判定:
    - 评分 ≥ 9.0 且无流程断裂且PRD修订完成 → 封版
    - 评分 < 9.0 或PRD有重大修订 → 进入下一轮
```

### 5.2 子任务执行规则

1. **每个子系统每轮迭代的每个独立角色使用独立子任务执行**，避免思维干扰
2. **每个子系统迭代至少10次**，且覆盖所有发现的问题
3. 不要在一个任务做多个模块的事情，单一任务，提升完成质量
4. 每次迭代生成几个独立任务：对抗builder、对抗挑战者、对抗裁决
5. **【新增】每个角色的任务必须包含PRD评估部分，不能只枚举流程**

### 5.3 封版标准

| # | 检查项 | 门槛 | 说明 |
|---|--------|------|------|
| 1 | 流程完备性 | ≥ 90% | PRD中每个需求至少1个主流程 |
| 2 | 数据可追溯性 | 100% | 每个数据可追溯到玩家操作/触发事件 |
| 3 | 流程断裂点 | 0个P0 | 无阻断性流程断裂 |
| 4 | 可玩性评分 | ≥ 8.0 | 玩法有趣、节奏合理、反馈及时 |
| 5 | 跨系统连贯性 | ≥ 85% | 跨系统流程衔接完整 |
| 6 | 最终轮新问题 | 0 | 收敛信号 |
| **7** | **PRD修订完成** | **所有P0修订已采纳** | **PRD设计质量达标** |
| **8** | **实体角色清晰** | **每个实体有不可替代的角色** | **无"僵尸实体"** |

## 七、优化迭代方法

### 7.1 追问补全法（Challenger核心方法）

通过不断追问来发现流程缺失：

1. **"然后呢？"** — 每个流程终点后追问下一步
2. **"如果失败呢？"** — 每个成功路径追问失败路径
3. **"如果中途退出呢？"** — 每个长流程追问中断恢复
4. **"新玩家怎么做？"** — 每个功能追问新手引导
5. **"老玩家还做什么？"** — 每个功能追问长期目标
6. **"和XX系统有什么关系？"** — 每个功能追问跨系统联动

### 7.2 【新增】PRD设计挑战法（Challenger核心方法）

通过挑战PRD设计来发现设计缺陷：

1. **"这个实体有存在感吗？"** — 检查每个实体是否被玩家感知到
   - 医馆的+3%恢复速率，玩家能感受到吗？
   - 如果去掉医馆，游戏有任何区别吗？

2. **"这个数值有意义吗？"** — 检查每个数值是否产生有意义的体验
   - Lv1农田产出0.8/秒，玩家需要等多久才能升级？
   - 升级费用和产出的ROI是否合理？

3. **"这个机制有趣吗？"** — 检查每个机制是否提供策略深度
   - 升级建筑只是"点击→等待→完成"，有决策吗？
   - 建筑之间有互动吗？有组合效果吗？

4. **"缺少什么？"** — 检查PRD中缺失的设计
   - 建筑数量够吗？8座能支撑多久的游戏？
   - 有地图系统吗？有建筑布局吗？
   - 有终局内容吗？

5. **"这个设计和其他游戏比如何？"** — 参考行业最佳实践
   - Clash of Clans 的建筑系统是怎么设计的？
   - 其他SLG的建筑升级有什么特色机制？

### 7.3 思维链模板（Builder构建流程时使用）

```
[需求/功能点] → [玩家操作入口] → [系统处理逻辑] → [数据变更] → [UI反馈] → [后续流程/循环点]
```

**【新增】PRD评估思维链**:
```
[PRD设计] → [玩家感知/操作] → [体验评估] → [问题识别] → [修订建议] → [影响评估]
```

### 7.4 覆盖率计算

```
流程覆盖率 = 已枚举流程数 / (PRD需求数 × 每需求平均流程数) × 100%

其中:
- PRD需求数 = PRD文档中的需求编号总数
- 每需求平均流程数 = 3~5（正常/异常/边界/跨系统/离线）

PRD设计质量分 = Σ(实体角色清晰度 + 数值平衡度 + 机制有趣度) / 维度数
```

## 八、【新增】PRD设计质量评估框架

### 8.1 实体角色评估

每个实体（建筑/武将/装备等）必须通过以下检查：

| 检查项 | 通过标准 | 不通过表现 |
|--------|---------|-----------|
| 存在感 | 玩家能明确感知到这个实体的存在 | 医馆+3%恢复速率，玩家感受不到 |
| 不可替代性 | 去掉这个实体游戏会有明显区别 | 去掉医馆，游戏照常运行 |
| 交互性 | 玩家需要主动与这个实体交互 | 书院只是被动产出科技点 |
| 差异化 | 与其他实体有明显区别 | 农田和市集都是"产出资源"，区别不大 |
| 成长感 | 升级这个实体有明显的体验提升 | 医馆从+3%到+6%，玩家无感 |

### 8.2 数值平衡评估

| 检查项 | 通过标准 | 不通过表现 |
|--------|---------|-----------|
| ROI合理性 | 升级投资回报在合理范围内 | Lv10后回本时间>5小时 |
| 节奏感 | 升级间隔不会太长也不会太短 | Lv29→30需要2.64年 |
| 感知阈值 | 数值变化在玩家可感知范围内 | 产出从0.8到1.0，玩家感受不到 |
| 曲线平滑 | 数值增长曲线不会突然跳跃 | Lv14→15费用突然翻倍 |

### 8.3 机制有趣度评估

| 检查项 | 通过标准 | 不通过表现 |
|--------|---------|-----------|
| 决策深度 | 玩家需要做出有意义的选择 | 只是按顺序升级，无需决策 |
| 策略多样性 | 有多种可行的策略路径 | 只有一种最优升级顺序 |
| 互动性 | 机制之间有有趣的互动 | 建筑之间完全独立 |
| 反馈及时 | 操作后有即时的正面反馈 | 升级后看不到明显变化 |
| 长期目标 | 有清晰的追求目标 | 满级后无事可做 |

## 九、【新增】方法论迭代改进规则

> 每轮迭代不仅是对游戏设计的对抗测试，也是对测试方法论本身的复盘和改进。

### 9.1 三重目标体系

| # | 目标 | 说明 | 产出 |
|---|------|------|------|
| 1 | 流程完备性 | 根据PRD和UI，补充完善测试流程，保证PRD可正确完整实现 | flows.md 流程文档 |
| 2 | 设计改进 | 改进游戏设计，提升丰富性/可玩性/合理性/体验 | PRD-R 修订建议 |
| 3 | 方法论改进 | 完善对抗性测试体系，优化测试方法和思维链 | 方法论改进记录 |

### 9.2 每轮复盘机制（Arbiter 职责扩展）

Arbiter 在每轮评分后，必须输出**方法论复盘**：

```markdown
### 方法论复盘 — Round {N}

**本轮发现的测试方法问题**:
- 问题描述: 什么测试方法/规则导致了遗漏或低效？
- 影响评估: 这个问题导致了什么后果？
- 改进建议: 如何修改方法论/规则来避免这个问题？

**本轮测试效率评估**:
| 维度 | 评分 | 说明 |
|------|------|------|
| 流程发现效率 | N/10 | Builder 枚举流程的速度和质量 |
| 问题发现深度 | N/10 | Challenger 发现问题的深度和准确性 |
| 裁决合理性 | N/10 | Arbiter 评分和裁决的公正性 |
| PRD评估有效性 | N/10 | PRD修订建议的质量和可操作性 |

**下一轮方法论调整**:
- 调整项1: 具体描述
- 调整项2: 具体描述
```

### 9.3 方法论改进追踪

每轮的方法论改进应记录在 verdict 文档中，并在下一轮实际应用：

| 来源 | 改进项 | 应用轮次 | 效果 |
|------|--------|---------|------|
| R1 复盘 | — | — | — |

### 9.4 方法论版本管理

方法论随迭代演进，每轮可产生小版本更新：

```
methodology.md v2.0 → R1复盘后 → v2.1 → R5复盘后 → v2.2 → ...
```

版本变更需记录在文档头部，标注修改内容和原因。
