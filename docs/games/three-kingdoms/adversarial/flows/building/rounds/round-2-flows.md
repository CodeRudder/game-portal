## 一、新增主流程

### BLD-F10 一键收取流程 {#bld-f10}

**PRD来源**: BLD-5-3 "右下角悬浮按钮，一键收取所有建筑累积产出"

**玩家视角**: 点击右下角悬浮按钮 → 所有建筑累积产出飞入资源栏 → 资源数字增加

**流程步骤**:
1. [UI入口] 右下角显示悬浮收取按钮（圆形，带资源图标和累积数量角标）
2. [玩家操作] 点击收取按钮
3. [系统触发-TE06] 触发一键收取事件
4. [系统计算] 遍历所有已解锁且 level > 0 的建筑:
   - 读取各建筑当前产出速率: `calculateTotalProduction()` 返回各 resourceType 累加值
   - 计算自上次收取以来的累积量（实时累加模型下，资源已实时累加，此处仅计算视觉展示数值）
5. [视觉效果] 收取动画:
   - 各建筑上方飘出产出数字（如"+12.5粮草"、"+8.3铜钱"）
   - 资源数字沿弧线飞向顶部资源栏对应位置
   - 顶部资源栏数字跳动增加（如有资源上限则显示"已满"提示）
6. [UI反馈] 收取按钮角标清零、Toast "收取成功！粮草+125 铜钱+83 兵力+45"
7. [后置] 按钮进入冷却（视觉冷却动画，1秒）

**产出模型说明**（PRD-R006）:
- 采用**实时累加模型**: 资源在每帧 `update(dt)` 时已实时累加到资源系统
- 一键收取本质是**UI快捷操作+视觉效果**，不是真正的"收取"逻辑
- 玩家不点收取按钮，资源也会正常累加（只是没有飞行动画）
- 收取按钮的价值在于: 提供明确的"收获"仪式感 + 积累数量角标的满足感

**数据变更**:
| 数据 | 变更前 | 变更后 | 说明 |
|------|--------|--------|------|
| 资源数值 | 无变化 | 无变化 | 实时累加模型下，资源已在每帧累加 |
| UI角标 | 显示累积量 | 清零 | 仅视觉层面 |

**前置条件**: 至少一座建筑已解锁且 level > 0
**后置流程**: 无（纯UI操作）

#### BLD-F10-01 VIP自动收取

**PRD来源**: BLD-5-3 "自动收取：VIP3+ 解锁离线自动收取（无需手动点击）"

**玩家视角**: VIP3+玩家无需手动点击，资源自动持续累加

**流程步骤**:
1. [系统检查] 玩家VIP等级 >= 3
2. [自动行为] 资源持续实时累加（与手动收取效果完全一致）
3. [UI差异] 收取按钮显示"自动"标识，角标不显示（无需手动操作）
4. [离线场景] VIP3+离线时资源照常累加（非VIP离线时资源暂停累加）

**VIP等级与收取规则**:
| VIP等级 | 收取方式 | 离线产出 |
|:-------:|---------|:-------:|
| VIP0~2 | 手动点击收取 | 暂停 |
| VIP3+ | 自动收取 | 持续累加 |

**数据变更**:
| 数据 | 变更前 | 变更后 | 说明 |
|------|--------|--------|------|
| resources（离线） | 不累加 | 持续累加 | VIP3+专属 |

**前置条件**: VIP等级 >= 3
**后置流程**: 无

### BLD-F11 升级加速流程 {#bld-f11}

**PRD来源**: BLD-4-3 "铜钱加速 / 天命加速 / 元宝秒完成"

**玩家视角**: 建筑正在升级中 → 点击建筑 → 看到加速选项 → 选择加速方式 → 支付费用 → 升级时间减少/立即完成

**流程步骤**:
1. [玩家操作] 点击正在升级中的建筑卡片
2. [UI响应] 详情面板显示:
   - 升级进度条（百分比 + 剩余时间倒计时）
   - 三种加速按钮: 铜钱加速 / 天命加速 / 元宝秒完成
   - 每个按钮显示对应消耗和效果
3. [玩家操作] 选择一种加速方式
4. [系统校验] 检查加速条件:
   - 建筑确实在升级中: `status === 'upgrading'`
   - 对应加速资源充足
   - 加速次数/限制未超限
5. [玩家确认] 弹出确认面板（显示消耗和效果）
6. [系统执行] 根据选择的加速方式执行 → 见 BLD-F11-01/02/03
7. [UI反馈] 进度条跳动、倒计时更新、Toast提示

**三种加速方式总览**:
| 加速方式 | 消耗 | 效果 | 限制 | 适用场景 |
|---------|------|------|:----:|---------|
| 铜钱加速 | 升级费用x0.3的铜钱 | 减少50%剩余时间 | 每次升级最多1次 | 铜钱充裕时 |
| 天命加速 | 天命道具x1 | 减少30%剩余时间 | 无限制 | 有道具时 |
| 元宝秒完成 | 元宝（按剩余时间定价） | 立即完成 | 无限制 | 付费玩家 |

**前置条件**: 建筑正在升级中
**后置流程**: → BLD-F02（升级完成）、→ BLD-F01（产出提升）

#### BLD-F11-01 铜钱加速

**流程步骤**:
1. [系统计算] 铜钱消耗 = `upgradeCost.gold x 0.3`（当前等级的升级铜钱费用的30%）
2. [系统校验] `resources.gold >= 铜钱消耗`，否则提示"铜钱不足"
3. [系统校验] 本次升级未使用过铜钱加速，否则提示"本次升级已使用过铜钱加速"
4. [执行] 扣除铜钱 → 计算新结束时间:
   - `剩余时间 = upgradeEndTime - Date.now()`
   - `新剩余时间 = 剩余时间 x 0.5`
   - `upgradeEndTime = Date.now() + 新剩余时间`
5. [UI反馈] Toast "铜钱加速成功！升级时间减少50%" + 进度条跳动

**数据变更**:
| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| resources.gold | X | X - 消耗 | 铜钱加速 |
| buildings[type].upgradeEndTime | T | T - 50%剩余 | 铜钱加速 |

**示例**:
- 农田 Lv10→11 升级时间 629秒，铜钱消耗 = 9448 x 0.3 = 2835铜钱
- 使用后剩余时间从 400秒 → 200秒

#### BLD-F11-03 元宝秒完成

**流程步骤**:
1. [系统计算] 元宝消耗（按剩余时间分档定价）:
   - 剩余时间 <= 1小时: 10元宝
   - 1小时 < 剩余时间 <= 1天: 50元宝
   - 1天 < 剩余时间 <= 7天: 200元宝
   - 剩余时间 > 7天: 500元宝
2. [系统校验] `resources.jade >= 元宝消耗`，否则提示"元宝不足"并引导充值
3. [玩家确认] 弹出确认面板: "花费50元宝立即完成升级？"
4. [执行] 扣除元宝 → 升级立即完成:
   - `buildings[type].level += 1`
   - `buildings[type].status = 'idle'`
   - 清空 upgradeStartTime/EndTime
   - 从 upgradeQueue 移除
5. [后置触发] 若为主城升级 → BLD-F05（解锁链检查）
6. [UI反馈] 升级完成特效（金色光效）+ Toast "元宝秒完成！农田升级至Lv.13"

**元宝定价表**:
| 剩余时间 | 元宝消耗 | 折算"元宝/小时" |
|----------|:-------:|:-------------:|
| <= 1小时 | 10 | 10 |
| 1小时~1天 | 50 | ~2.1 |
| 1天~7天 | 200 | ~1.2 |
| > 7天 | 500 | <0.3 |

**设计意图**: 剩余时间越长，元宝性价比越高，鼓励玩家在升级早期使用加速（付费转化）

**数据变更**:
| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| resources.jade | X | X - 消耗 | 元宝秒完成 |
| buildings[type].level | N | N+1 | 元宝秒完成 |
| buildings[type].status | 'upgrading' | 'idle' | 元宝秒完成 |
| upgradeQueue | [..slot..] | [..] | 元宝秒完成 |

**前置条件**: 建筑正在升级中、元宝充足
**后置流程**: → BLD-F05（解锁检查）、→ BLD-F01（产出重新计算）

### BLD-F12 自动升级流程 {#bld-f12}

**PRD来源**: BLD-4-3 "可开启开关，系统自动将可升级建筑加入队列"

**玩家视角**: 开启自动升级开关 → 系统自动将可升级的建筑加入队列 → 无需手动操作

**流程步骤**:
1. [玩家操作] 在建筑界面找到"自动升级"开关，点击开启
2. [系统注册] 设置 `autoUpgradeEnabled = true`
3. [系统触发-TE07] 每次升级队列有空位时（tick()中检测），触发自动升级检查:
   - a. 检查 `autoUpgradeEnabled === true`
   - b. 检查队列是否有空位: `!isQueueFull()`
   - c. 遍历所有建筑，筛选可升级建筑（见 BLD-F12-01）
   - d. 按优先级排序，选择优先级最高的建筑
   - e. 检查资源是否充足（含资源保护，见 BLD-F12-02）
   - f. 资源充足 → 自动调用 `startUpgrade(type, resources)`
   - g. 资源不足 → 跳过，等待资源累积
4. [UI反馈] 自动升级的建筑卡片显示"自动"标识 + 进度条
5. [玩家操作] 可随时关闭自动升级开关 → `autoUpgradeEnabled = false`

**与手动操作的冲突处理**:
| 场景 | 处理方式 |
|------|---------|
| 手动升级时自动升级触发 | 手动优先，自动升级跳过本轮 |
| 自动升级的建筑手动取消 | 取消正常执行（BLD-F03），自动升级下轮重新排队 |
| 队列已满时 | 自动升级不触发，等待队列空位 |

**数据变更**:
| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| autoUpgradeEnabled | false | true | 玩家操作 |
| upgradeQueue | [N slots] | [N+1 slots] | 自动升级 |
| resources | R | R - cost | 自动升级 |

**前置条件**: 玩家手动开启开关
**后置流程**: → BLD-F02（升级流程）、→ BLD-F05（解锁检查）

#### BLD-F12-01 自动升级优先级算法

**流程步骤**:
1. 遍历所有 BUILDING_TYPES，筛选候选建筑:
   - `status === 'idle'`（未在升级中）
   - `level < maxLevel`（未满级）
   - 非主城: `level <= castle.level`（满足等级限制）
   - 主城: 检查特殊前置条件
2. 对每个候选建筑计算优先级分数:
   - **主城**: priority = 100（永远最高优先）
   - **有产出的建筑**: priority = 50 + (产出提升量 / 升级费用) x 1000
   - **无产出的建筑**（医馆/城墙）: priority = 30
   - **资源不足**: priority -= 20（降低但不剔除，等待资源累积）
3. 按 priority 降序排列
4. 选择 priority 最高的建筑作为升级目标

**优先级示例**（假设 Lv10 阶段）:
| 建筑 | 产出提升 | 升级费用 | ROI | 优先级 |
|------|:-------:|:-------:|:---:|:------:|
| 主城 | +2%全资源 | 高 | — | 100 |
| 农田 | +1.6/秒 | 30,234 | 53 | 53 |
| 市集 | +1.2/秒 | 24,187 | 50 | 50 |
| 兵营 | +1.15/秒 | 36,280 | 32 | 48 |
| 书院 | +0.5/秒+研究 | 16,142 | 31 | 45 |
| 医馆 | +3%恢复 | 8,708 | — | 30 |
| 城墙 | +2.5%防御 | 36,202 | — | 30 |

#### BLD-F12-02 自动升级资源保护

**流程步骤**:
1. 自动升级前检查资源是否满足"最低保护线":
   - 最低保护 = 当前产出速率 x 300秒（5分钟产出量）
   - 目的: 防止自动升级耗尽资源导致玩家无法进行其他操作
2. 可用资源 = 当前资源 - 最低保护
3. 若可用资源 >= 升级费用 → 执行自动升级
4. 若可用资源 < 升级费用 → 跳过，等待资源累积

**资源保护示例**:
```
当前粮草: 50,000
农田产出: 8.0/秒
最低保护: 8.0 x 300 = 2,400
可用粮草: 50,000 - 2,400 = 47,600

主城升级费用: 40,000 → 47,600 >= 40,000 → 执行
```

**设计意图**: 避免自动升级把资源花光，导致玩家想手动升级其他建筑时资源不足

- **频率**: 玩家主动触发

## 三、新增跨系统流程

### XI-007 武将→建筑系统（派驻流程）

**衔接点**: HER-F03 → BLD-F01
**PRD来源**: BLD-3 "武将（统率型）→兵营 兵力产出+统率%; 武将（政治型）→农田/市集 产出+政治%"

**派驻流程步骤**:
1. [玩家操作] 在建筑详情面板中点击"派驻武将"按钮
2. [UI响应] 显示可派驻武将列表（筛选条件: 统率型→兵营，政治型→农田/市集）
3. [玩家操作] 选择武将 → 确认派驻
4. [系统处理] 记录建筑与武将的关联关系: `buildingHeroMap[buildingType] = heroId`
5. [加成生效] 产出计算时注入武将加成:
   - 兵营: `产出 x (1 + hero.command / 100)`
   - 农田/市集: `产出 x (1 + hero.politics / 100)`
6. [UI反馈] 建筑卡片显示武将头像角标，详情面板显示加成百分比

**卸任流程**:
1. [玩家操作] 在建筑详情面板中点击"卸任"按钮
2. [系统处理] 移除关联关系: `buildingHeroMap[buildingType] = null`
3. [加成移除] 产出计算中移除武将加成
4. [UI反馈] 建筑卡片武将角标消失

**加成叠加规则**（回应CH-F009）:
```
最终产出 = 基础产出 x 主城加成 x 科技加成 x 武将加成
```
- 各环节之间采用**乘法叠加**

### PRD-R006 产出模型统一 {#prd-r006}

**问题**: PRD暗示"累积制"（需一键收取），但引擎实现实时累加（CH-F005）

**采纳方案**: 统一为**实时累加模型**，一键收取为UI快捷操作

#### 4.2.1 统一后的产出模型

```
资源产出模型（统一）:
├── 在线期间: 每帧 update(dt) → calculateTotalProduction() x dt → 资源实时累加
├── 一键收取: 仅UI操作（飞行动画+角标清零），不涉及资源逻辑
├── 离线期间:
│   ├── VIP3+: 资源持续累加（产出速率 x 离线秒数 x 离线效率系数1.2）
│   └── 非VIP: 资源暂停累加（登录后从上次存档时间起算）
└── 资源上限: 可选机制（base + 建筑等级 x 系数），超过上限停止累加
```

#### 4.2.2 离线效率系数实现

PRD BLD-4-4 规定: "离线效率 = 资源效率 x 1.2（建造额外 20% 效率）"

实际含义拆解:
- **资源离线效率**: 离线产出 = 在线产出速率 x 离线时间 x 1.0（不打折）
- **建造离线效率**: 离线期间升级进度正常推进（受离线效率系数影响）
- **1.2系数**: 离线期间建造速度额外+20%（鼓励玩家下线，减少服务器压力）

实现位置: `BuildingSystem.deserialize()` 中处理离线升级完成时，应计算实际经过时间 x 1.2

#### 4.2.3 一键收取的UI价值

虽然资源已实时累加，一键收取仍有UI价值:
1. **仪式感**: 点击按钮 → 飞行动画 → 数字跳动，提供"收获"的心理满足
2. **信息聚合**: 角标显示各建筑累积产出总量，让玩家一目了然
3. **VIP差异**: VIP3+自动收取 vs 非VIP手动收取，体现付费价值
4. **资源上限提醒**: 接近上限时收取按钮变红提醒，引导玩家消费资源
