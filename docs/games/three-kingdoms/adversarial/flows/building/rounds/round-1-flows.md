## 一、主流程

### BLD-F01 资源产出循环 {#bld-f01}

**玩家视角**: 建筑持续产出资源 → 资源累积 → 用于升级/消费

**流程步骤**:
1. [系统触发-TE01] 每帧 `update(dt)` 调用 → 内部调用 `tick()`
2. [系统计算] `calculateTotalProduction()` 遍历所有非 castle 建筑
3. [系统计算] 对每个 level > 0 的建筑，读取 `BUILDING_DEFS[type].levelTable[level-1].production`
4. [系统计算] 按 resourceType 累加：grain(农田)、gold(市集)、troops(兵营)、techPoint(书院)
5. [系统计算] 产出乘以加成链: base × `getCastleBonusMultiplier()` × 科技加成 × 武将加成
6. [数据变更] 资源系统: grain/gold/troops/techPoints += 产出值 × dt
7. [UI反馈] 顶部资源栏数字递增、各建筑上方飘出产出数字

**数据变更**:
| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| resources.grain | X | X + farmland产出×dt | calculateTotalProduction |
| resources.gold | X | X + market产出×dt | calculateTotalProduction |
| resources.troops | X | X + barracks产出×dt | calculateTotalProduction |
| resources.techPoints | X | X + academy产出×dt | calculateTotalProduction |

**源码追溯**:
- `BuildingSystem.ts` L344-L356: `calculateTotalProduction()` 遍历 BUILDING_TYPES，跳过 castle，累加各 resourceType
- `BuildingSystem.ts` L193-L198: `getProduction(type, level?)` 从 levelTable 读取 production
- `BuildingSystem.ts` L201-L209: `getCastleBonusPercent()` / `getCastleBonusMultiplier()` 含 NaN 防护

**前置条件**: 建筑已解锁且 level > 0
**后置流程**: → RES-F01（资源累积）、→ BLD-F01-02（产出上限检查）

---

#### BLD-F01-01 主城全资源加成计算

**流程步骤**:
1. 调用 `getCastleBonusPercent()` → `getProduction('castle')`
2. 返回 `castle.levelTable[level-1].production`（如 Lv5 → 8%）
3. 转换为乘数: `1 + percent/100`（如 1.08）
4. NaN 防护: `!Number.isFinite(pct)` → 返回 1.0

**数据来源**: castle 等级 → levelTable 配置
**源码追溯**: `BuildingSystem.ts` L206-L209

---

#### BLD-F01-02 产出上限检查

**流程步骤**:
1. 资源系统检查当前资源是否达到上限
2. 达到上限 → 产出停止累加（溢出丢弃或等待收取）

**断裂点**: 当前引擎未实现资源上限机制 → 见 GAP-004

---

#### BLD-F01-03 特殊建筑产出（非资源类）

**流程步骤**:
1. **城墙**: `getWallDefense()` → 读取 `BUILDING_DEFS.wall.levelTable[level-1].specialValue`（城防值）; `getWallDefenseBonus()` → 读取 production（防御加成%）
2. **医馆**: `getClinicRecoveryRate()` → 读取 `BUILDING_DEFS.clinic.levelTable[level-1].production`（恢复速率%）
3. **铁匠铺**: 产出材料/小时，resourceType 未在 calculateTotalProduction 中注册

**断裂点**: 医馆恢复速率是被动百分比加成，玩家无法直接感知 → 见 PRD-R001
**断裂点**: 铁匠铺产出单位为"材料/小时"，与其他建筑"每秒"不统一 → 见 GAP-006

### BLD-F02 建筑升级流程 {#bld-f02}

**玩家视角**: 点击建筑 → 查看详情 → 点击升级 → 等待 → 升级完成

**流程步骤**:
1. [玩家操作] 点击建筑网格中的建筑卡片
2. [UI响应] 右侧滑入详情面板(420×696px)，显示:
   - 建筑头部(图标+名称+分类+等级+状态)
   - 产出信息(当前产出+加成后+离线效率)
   - 升级预览(等级变化+产出对比+消耗+时间)
   - 解锁条件(前置条件列表+满足状态)
   - 联动效果(科技加成+主城加成)
   - 操作按钮(升级 / 一键满级)
3. [玩家操作] 点击"升级"按钮（无需确认弹窗，直接执行+Toast反馈）
4. [系统校验] `checkUpgrade(type, resources)` 检查 7 项条件:
   - a. 建筑未锁定: `status !== 'locked'`
   - b. 未在升级中: `status !== 'upgrading'`
   - c. 未达等级上限: `level < BUILDING_MAX_LEVELS[type]`
   - d. 非主城等级限制: `level > castle.level` → 拒绝（允许子建筑领先主城1级）
   - e. 主城特殊前置: Lv4→5 需任一建筑 Lv4; Lv9→10 需任一建筑 Lv9
   - f. 队列未满: `!isQueueFull()`（queue.length >= maxSlots）
   - g. 资源充足: grain >= cost.grain, gold >= cost.gold, troops >= cost.troops
   - h. NaN 防护: `!Number.isFinite(resources.grain/gold/troops)` → 拒绝
5. [校验失败] 返回 `{ canUpgrade: false, reasons: [...] }`，UI 显示 Toast 错误提示
6. [校验通过] `startUpgrade(type, resources)`:
   - a. 获取费用 `getUpgradeCost(type)` → `levelTable[level].upgradeCost` 深拷贝
   - b. 保存快照用于回滚(prevStatus, prevStartTime, prevEndTime)
   - c. 设置 `status = 'upgrading'`, `upgradeStartTime = Date.now()`, `upgradeEndTime = now + timeSeconds * 1000`
   - d. 添加到 `upgradeQueue`: `{ buildingType, startTime, endTime }`
   - e. 返回费用（由调用方扣除资源）
7. [UI反馈] Toast "农田升级至Lv.13，预计8分钟" + 建筑卡片显示进度条+蓝色脉冲边框
8. [系统触发-TE02] 每帧 `tick()` 检查 `now >= slot.endTime`
9. [升级完成] `level += 1`, `status = 'idle'`, 清空 upgradeStartTime/EndTime
10. [UI反馈] Toast "农田升级完成！粮草产出 +2.5/秒" + 建筑外观升级动画(scale弹跳)
11. [系统触发] 若为主城升级 → BLD-F05（解锁链）

**数据变更**:
| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| buildings[type].level | N | N+1 | tick() |
| buildings[type].status | 'upgrading' | 'idle' | tick() |
| buildings[type].upgradeStartTime | timestamp | null | tick() |
| buildings[type].upgradeEndTime | timestamp | null | tick() |
| resources.grain | X | X - cost.grain | startUpgrade |
| resources.gold | X | X - cost.gold | startUpgrade |
| resources.troops | X | X - cost.troops | startUpgrade |
| upgradeQueue | [...] | [...+slot] | startUpgrade |

**源码追溯**:
- `BuildingSystem.ts` L132-L181: `checkUpgrade()` 完整校验链
- `BuildingSystem.ts` L215-L252: `startUpgrade()` 含事务回滚
- `BuildingSystem.ts` L280-L306: `tick()` 升级完成检测+主城解锁触发

**前置条件**: 建筑已解锁、资源充足、队列未满
**后置流程**: → BLD-F05（解锁检查）、→ BLD-F01（产出重新计算）

#### BLD-F02-02 升级前置条件 — 主城等级限制

**流程步骤**:

2. 主城 Lv4→5: 需要 `BUILDING_TYPES.some(t !== castle && level >= 4)`
3. 主城 Lv9→10: 需要 `BUILDING_TYPES.some(t !== castle && level >= 9)`
4. Lv10→15→20→25→30: **无额外前置条件** → 见 GAP-001

#### BLD-F02-03 升级前置条件 — 队列管理

**流程步骤**:
1. `getMaxQueueSlots()` 根据 castle.level 返回槽位数:
   - Lv1-5 → 1 slot
   - Lv6-10 → 2 slots
   - Lv11-20 → 3 slots
   - Lv21-30 → 4 slots
2. `isQueueFull()` 检查 `queue.length >= maxSlots`
3. 队列满 → 拒绝升级，返回"升级队列已满"

**源码追溯**: `BuildingSystem.ts` L329-L339, `building-config.ts` L442-L447

#### BLD-F02-04 升级前置条件 — 资源校验（含NaN防护）

**流程步骤**:

2. `resources.grain < cost.grain` → 返回"粮草不足：需要 X"
3. `resources.gold < cost.gold` → 返回"铜钱不足：需要 X"

### BLD-F03 取消升级流程 {#bld-f03}

**玩家视角**: 在升级中点击建筑 → 点击"取消升级" → 返还80%资源

**流程步骤**:
1. [玩家操作] 点击正在升级中的建筑
2. [UI响应] 详情面板显示升级进度(进度条+倒计时)和"取消升级"按钮
3. [玩家操作] 点击"取消升级"
4. [系统处理] `cancelUpgrade(type)`:
   - a. 校验 `status === 'upgrading'`，否则 return null
   - b. 获取原始费用 `getUpgradeCost(type)`
   - c. 计算返还: `Math.floor(cost.grain * 0.8)`, `Math.floor(cost.gold * 0.8)`, `Math.floor(cost.troops * 0.8)`
   - d. 状态重置: `status = 'idle'`, `upgradeStartTime = null`, `upgradeEndTime = null`
   - e. 从 upgradeQueue 移除: `this.upgradeQueue.filter(s => s.buildingType !== type)`
5. [数据变更] 资源系统: grain/gold/troops += refund
6. [UI反馈] Toast "升级已取消，返还80%资源" + 资源数字增加 + 进度条消失

**数据变更**:
| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| buildings[type].status | 'upgrading' | 'idle' | cancelUpgrade |
| buildings[type].upgradeStartTime | timestamp | null | cancelUpgrade |
| buildings[type].upgradeEndTime | timestamp | null | cancelUpgrade |
| resources.grain | X | X + refund.grain | cancelUpgrade |
| resources.gold | X | X + refund.gold | cancelUpgrade |
| resources.troops | X | X + refund.troops | cancelUpgrade |
| upgradeQueue | [..slot..] | [..] | cancelUpgrade |

**源码追溯**: `BuildingSystem.ts` L255-L275, `building-config.ts` L454 (`CANCEL_REFUND_RATIO = 0.8`)

**前置条件**: 建筑正在升级中
**后置流程**: → BLD-F01（产出重新计算）

### BLD-F04 批量升级流程 {#bld-f04}

**玩家视角**: 选择多个建筑 → 一键批量升级 → 依次执行

**流程步骤**:
1. [玩家操作] 通过推荐系统或手动选择多个建筑类型
2. [玩家操作] 点击"批量升级"
3. [系统处理] `batchUpgrade(types, resources)`:
   - a. 初始化剩余资源: remainingGrain/Gold/Troops = resources
   - b. 遍历 types 列表:
     - 用剩余资源构建 currentResources
     - `checkUpgrade(t, currentResources)` → 失败则记录 reason，跳过
     - `startUpgrade(t, currentResources)` → 成功则累加 totalCost，递减剩余资源
   - c. 异常处理: startUpgrade 抛错时记录失败但继续后续建筑
   - d. 返回 `{ succeeded, failed, totalCost }`
4. [UI反馈] Toast 显示成功/失败列表，资源变化

**数据变更**:
| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| resources | R | R - totalCost | batchUpgrade |
| upgradeQueue | [...] | [...+new slots] | batchUpgrade |
| buildings[types] | 各自level | 各自level+1(成功部分) | tick()后续 |

**源码追溯**: `BuildingBatchOps.ts` L57-L101 (FIX-404: 事务性保证)

**前置条件**: 至少一个建筑可升级
**后置流程**: → BLD-F02（每个单独升级流程）

### BLD-F05 建筑解锁链 {#bld-f05}

**玩家视角**: 主城升级到指定等级 → 新建筑自动解锁 → 出现在建筑网格中

**流程步骤**:
1. [触发] 主城升级完成 (`tick()` 中 completed 包含 'castle')
2. [系统处理] `checkAndUnlockBuildings()`:
   - a. 遍历所有 BUILDING_TYPES
   - b. 对 status === 'locked' 的建筑检查 `checkUnlock(type)`:
     - `BUILDING_UNLOCK_LEVELS[type]` === 0 → 初始解锁
     - `castle.level >= BUILDING_UNLOCK_LEVELS[type]` → 满足
   - c. 符合条件 → `status = 'idle'`, `level = 1`
   - d. 返回新解锁建筑列表
3. [UI反馈] 新建筑出现在网格中（从灰色蒙版+锁变为正常卡片）、解锁特效

**解锁时间线**:
| 主城等级 | 解锁建筑 | 累计建筑数 | 流程编号 |
|:--------:|---------|:---------:|---------|
| Lv1(初始) | castle(主城) Lv1, farmland(农田) Lv1 | 2 | BLD-F05-初始 |
| Lv2 | market(市集), barracks(兵营) | 4 | BLD-F05-Lv2 |
| Lv3 | smithy(铁匠铺), academy(书院) | 6 | BLD-F05-Lv3 |
| Lv4 | clinic(医馆) | 7 | BLD-F05-Lv4 |
| Lv5 | wall(城墙) | 8 | BLD-F05-Lv5 |
| Lv6~30 | **无新建筑** | 8 | **断裂: GAP-001** |

**数据变更**:
| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| buildings[type].status | 'locked' | 'idle' | checkAndUnlockBuildings |
| buildings[type].level | 0 | 1 | checkAndUnlockBuildings |

**源码追溯**: `BuildingSystem.ts` L111-L128, `building-config.ts` L32-L41

**前置条件**: 主城升级完成
**后置流程**: → BLD-F01（新建筑开始产出）

### BLD-F07 离线建造流程 {#bld-f07}

**玩家视角**: 关闭游戏 → 重新登录 → 查看离线期间完成的升级和产出

**流程步骤**:
1. [触发] 玩家重新登录，调用 `deserialize(saveData)`
2. [校验] 存档版本检查: `data.version !== BUILDING_SAVE_VERSION` → 警告
3. [校验] 存档有效性: `!data || !data.buildings` → 使用默认状态 `reset()`
4. [校验修复] 对每个建筑:
   - `status === 'upgrading' && (level < 0 || !upgradeEndTime)` → 修正为 idle
   - `status === 'locked' && level > 0` → 修正为 idle
   - `status === 'idle' && level <= 0 && level !== 0` → 修正为 locked
5. [离线完成] 遍历所有 upgrading 建筑:
   - `now >= upgradeEndTime` → `level += 1`, `status = 'idle'`（离线完成）
   - `now < upgradeEndTime` → 重建 upgradeQueue（继续等待）
6. [系统处理] `checkAndUnlockBuildings()` — 检查是否有新建筑可解锁
7. [UI反馈] 离线收益面板显示完成的升级和资源产出

**数据变更**:
| 数据 | 变更前 | 变更后 | 来源 |
|------|--------|--------|------|
| buildings[completed].level | N | N+1 | deserialize |
| buildings[completed].status | 'upgrading' | 'idle' | deserialize |
| upgradeQueue | [] | [remaining slots] | deserialize |

**源码追溯**: `BuildingSystem.ts` L389-L445

**前置条件**: 有存档数据、有正在升级的建筑
**后置流程**: → BLD-F05（解锁检查）、→ OFR-F01（离线收益计算）

---

#### BLD-F07-01 离线资源产出计算

**流程步骤**:
1. 离线时长 = now - lastSaveTime
2. 离线产出 = calculateTotalProduction() × 离线时长 × 离线效率系数
3. PRD 规定: 离线效率 = 资源效率 × 1.2（建造额外 20% 效率）

**断裂点**: 引擎未实现离线效率系数，产出按 100% 计算 → 见 GAP-006

**流程步骤**:
1. [玩家操作] 点击"升级推荐"按钮
2. [系统计算] `getUpgradeRouteRecommendation(resources)`:
   - a. 遍历 BUILDING_TYPES
   - b. 跳过 locked / upgrading / 满级建筑
   - c. 非主城: 跳过 level >= castle.level 的
   - d. 主城: priority = 100
   - e. 有 production 的建筑: priority = 50 + prodGain * 10
   - f. 无 production 的建筑: priority = 30
   - g. 资源不足: priority -= 20
   - h. 按 priority 降序排列
3. [UI反馈] 显示推荐列表，包含升级原因和预估收益

**替代方案**: `recommendUpgradePath(context)` 按游戏阶段推荐:
- newbie: castle → farmland → market → barracks → smithy → academy → clinic → wall
- development: castle → smithy → academy → barracks → farmland → market → wall → clinic
- late: castle → wall → clinic → barracks → smithy → academy → farmland → market

**源码追溯**: `BuildingRecommender.ts` L42-L152

**流程断裂**: 推荐仅基于当前资源和 priority 算法，不考虑长期规划 → 见 GAP-005

## 二、触发事件（TE）

### TE-01 建筑产出 Tick

- **触发条件**: 每帧 `update(dt)` 调用
- **触发频率**: 60fps（每秒60次）
- **处理**: `calculateTotalProduction()` → 资源累加
- **关联流程**: BLD-F01

### TE-02 升级计时 Tick

- **触发条件**: 每帧 `update(dt)` 调用 → `tick()`
- **触发频率**: 60fps
- **处理**: 遍历 upgradeQueue，`now >= slot.endTime` → 完成升级
- **关联流程**: BLD-F02

### TE-03 主城升级解锁

- **触发条件**: `tick()` 中 completed 包含 'castle'
- **处理**: `checkAndUnlockBuildings()` → 新建筑解锁
- **关联流程**: BLD-F05

### TE-04 离线升级完成

- **触发条件**: `deserialize()` 时 `now >= upgradeEndTime`
- **处理**: `level += 1`, `status = 'idle'`
- **关联流程**: BLD-F07

### TE-05 存档校验修复

- **触发条件**: `deserialize()` 时数据异常
- **处理**: upgrading 状态异常→修正为 idle; locked 但 level>0→修正为 idle
- **关联流程**: BLD-F07

## 三、跨系统流程（XI）

### XI-001 建筑→资源系统（产出注入）

**衔接点**: BLD-F01 → RES-F01
**数据流**: `calculateTotalProduction()` 返回值 → 资源系统累加
**加成链**: `base × castleBonus × techBonus × heroBonus × prestigeBonus`
**注意**: 加成链中各环节的叠加方式（乘法/加法）未在建筑域内定义

### XI-002 资源→建筑系统（升级扣费）

**衔接点**: RES-F02 → BLD-F02
**数据流**: `startUpgrade()` 扣除 resources.grain/gold/troops
**回滚**: `cancelUpgrade()` 返还 80%（`CANCEL_REFUND_RATIO = 0.8`）

### XI-003 建筑→科技系统（科技点产出）

**衔接点**: BLD-F01 → TEC-F01
**数据流**: academy 产出 techPoints/秒 → 科技系统消费
**注意**: 书院 production.resourceType = 'techPoint'

### XI-004 建筑→攻城系统（城防值）

**衔接点**: BLD-F06 → CPN-F01
**数据流**: `getWallDefense()` 返回城防值(specialValue) → 攻城防御计算
**数据流**: `getWallDefenseBonus()` 返回防御加成%(production) → 攻城防御乘数

### XI-005 武将→建筑系统（加成注入）

**衔接点**: HER-F02 → BLD-F01
**数据流**: 武将政治型→农田/市集产出 +政治%; 统率型→兵营产出 +统率%
**断裂点**: 武将派驻建筑的具体流程未设计 → 见 CH-F008

### XI-006 科技→建筑系统（加成注入）

**衔接点**: TEC-F02 → BLD-F01
**数据流**: 科技效果（如"农耕改良"粮草+10%、"商贸繁荣"铜钱+15%）→ 建筑产出加成
**断裂点**: 多个科技加成的叠加规则未明确 → 见 CH-F009

## 五、流程断裂点与缺失功能

### GAP-001 主城Lv5后无新建筑/新机制 [严重度: 高]

**位置**: BLD-F05（解锁链）
**现象**: 主城 Lv5 解锁城墙后，Lv6~30 无新建筑解锁，后续仅数值堆叠
**数据支撑**: `BUILDING_UNLOCK_LEVELS` 中 wall=5 是最高解锁等级
**影响**:
- 游戏中期（第5天后）失去新鲜感
- 6×5 网格有 30 格但只有 8 座建筑，22 格空地无意义
- 建筑网格 UI 设计的大量空间被浪费
**建议**: 见 PRD-R002

### GAP-002 世界地图缺失 [严重度: 高]

**位置**: 跨系统（BLD ↔ MAP）
**现象**: 建筑仅在固定网格中，无地理维度；攻城无实际地图
**影响**: 策略深度不足，建筑布局无意义
**建议**: 引入地图系统，建筑位置影响产出/防御

### GAP-003 资源产出与升级费用失衡 [严重度: 高]

**位置**: BLD-F01（产出）vs BLD-F02（费用）
**现象**:
- Lv5→6 农田: 产出提升 0.8/秒, 费用 5000 粮草, 回本 104 分钟
- Lv10→11 农田: 产出提升 0.8/秒, 费用 14400 粮草, 回本 5 小时
- Lv20→21 农田: 产出提升 2/秒, 费用 2954880 粮草, 回本 17 天
- Lv29→30 主城: 费用 80,808,755 粮草, 升级时间 83,216,586 秒 = 2.64 年
**影响**: Lv10 后 ROI 急剧恶化，玩家体验差
**建议**: 见 PRD-R003

### GAP-004 建筑间无协同/对抗机制 [严重度: 中]

**位置**: BLD-F01（产出计算）
**现象**: 8 座建筑完全独立运作，无互相加成/限制
**影响**: 升级策略单一（均衡升级即可），缺乏决策深度
**建议**: 见 PRD-R004

### GAP-005 升级路线推荐缺乏长期规划 [严重度: 中]

**位置**: BLD-F09（推荐流程）
**现象**: 推荐仅基于当前资源和 priority 算法，不考虑:
- 玩家当前游戏阶段（新手/发展/后期）
- 下一个解锁里程碑（如主城 Lv5 解锁城墙）
- 资源产出-消耗平衡
**影响**: 推荐不够智能，玩家可能按推荐升级后陷入资源困境
**建议**: 引入阶段性推荐策略，考虑资源平衡

### GAP-006 PRD功能缺失（一键收取/加速/自动升级/离线效率） [严重度: 高]

**位置**: BLD-F02/F07
**现象**: PRD 提到但引擎未实现:
- BLD-5-3: 一键收取（右下角悬浮按钮）
- BLD-4-3: 加速选项（铜钱/天命/元宝）
- BLD-4-3: 自动升级（可开启开关）
- BLD-4-4: 离线效率系数（×1.2）
**影响**: PRD 覆盖率仅 76.9%
**建议**: 见 PRD-R005/R006

## 九、PRD 设计质量评估

### 9.1 实体角色评估

对 PRD 定义的 8 座建筑逐一评估其作为游戏实体的存在感、不可替代性、交互性、差异化和成长感。

#### 9.1.1 主城 (castle)

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 存在感 | ★★★★★ | 全资源加成、解锁前置、队列控制，处处可见 |
| 不可替代性 | ★★★★★ | 移除主城则整个系统崩溃 |
| 交互性 | ★★★★☆ | 升级触发解锁链，但无主动技能 |
| 差异化 | ★★★★★ | 核心建筑，独一无二 |
| 成长感 | ★★★★★ | 每级 +2% 全资源加成，30级外观变化 |

**结论**: 主城设计优秀，是系统的绝对核心。

#### 9.1.2 农田 (farmland)

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 存在感 | ★★★★☆ | 粮草是主要资源，产出数字持续可见 |
| 不可替代性 | ★★★★☆ | 粮草是升级费用的主要组成部分 |
| 交互性 | ★★★☆☆ | 只有升级操作，无主动交互 |
| 差异化 | ★★★☆☆ | 与市集/兵营模式完全相同，仅资源类型不同 |
| 成长感 | ★★★☆☆ | 产出线性增长，后期提升感知弱 |

**结论**: 存在感尚可，但与市集/兵营高度同质化。

#### 9.1.3 市集 (market)

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 存在感 | ★★★★☆ | 铜钱产出，升级费用需要 |
| 不可替代性 | ★★★★☆ | 铜钱是第二大资源 |
| 交互性 | ★★★☆☆ | 只有升级操作 |
| 差异化 | ★★☆☆☆ | 与农田完全同质，仅产出"铜钱"而非"粮草" |
| 成长感 | ★★★☆☆ | 同农田 |

**结论**: 与农田高度同质化，缺乏独特身份。

#### 9.1.4 兵营 (barracks)

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 存在感 | ★★★★☆ | 兵力产出，出征需要 |
| 不可替代性 | ★★★★☆ | 兵力是军事行动基础 |
| 交互性 | ★★★☆☆ | 只有升级操作 |
| 差异化 | ★★☆☆☆ | 与农田/市集同质 |
| 成长感 | ★★★☆☆ | 同农田 |

**结论**: 产出建筑三兄弟（农田/市集/兵营）完全同质化。

#### 9.1.5 铁匠铺 (smithy)

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 存在感 | ★★★☆☆ | 产出材料/小时，单位与其他建筑不同 |
| 不可替代性 | ★★☆☆☆ | 材料用途（装备强化）在当前系统中不明确 |
| 交互性 | ★★☆☆☆ | 只有升级操作，产出单位为"材料/小时"而非"每秒" |
| 差异化 | ★★★☆☆ | 产出单位不同（材料/小时 vs 资源/秒） |
| 成长感 | ★★☆☆☆ | 材料用途不明确，升级动力不足 |

### 9.2 数值平衡评估

#### 9.2.1 ROI 合理性分析

**问题**: 产出线性增长 vs 费用指数增长，后期 ROI 急剧恶化

| 等级段 | 农田产出增长 | 费用倍率 | 时间倍率 | ROI趋势 |
|--------|:----------:|:-------:|:-------:|:-------:|
| Lv1→5 | 0.8→3.0 (+275%) | 基准 | 基准 | 优秀 |
| Lv5→10 | 3.0→8.0 (+167%) | ×1.8/级 | ×1.6/级 | 良好 |
| Lv10→15 | 8.0→16.0 (+100%) | ×1.6/级 | ×1.5/级 | 一般 |
| Lv15→20 | 16.0→28.0 (+75%) | ×1.5/级 | ×1.4/级 | 差 |
| Lv20→25 | 28.0→45.0 (+61%) | ×1.4/级 | ×1.3/级 | 极差 |

**具体数据**:
- Lv10→11 农田: 回本 5.25 小时 — 勉强可接受
- Lv15→16 农田: 回本 2.16 天 — 开始失去动力
- Lv20→21 农田: 回本 11.1 天 — 完全不划算
- Lv24→25 农田: 回本 25.4天 — 荒谬

**核心矛盾**: 产出每级增加固定量(~2.2/级)，但费用每级乘以 1.4~1.8。这意味着回本时间呈指数增长。

#### 9.2.2 节奏感分析

**新手期（Lv1→5）**: 节奏优秀
- 升级时间 5s→60s，快速获得升级快感
- 每次升级产出提升明显（+0.2~+0.8/秒）
- 5天内可解锁全部8座建筑

**发展期（Lv5→10）**: 节奏尚可
- 升级时间 1.5m→2h，有等待但可接受
- 产出提升开始放缓
- 新建筑解锁完毕，新鲜感下降

**中期（Lv10→15）**: 节奏急剧恶化
- 升级时间 4h→64h，等待感强烈
- 产出提升微弱，ROI恶化
- 无新建筑、无新机制，纯数值堆叠

**后期（Lv15→30）**: 节奏崩溃
- 升级时间 50天→2.64年，完全不可接受
- 产出提升几乎无感知
- 玩家必然流失

#### 9.2.3 感知阈值分析

**问题**: 后期产出增量低于玩家感知阈值

| 等级 | 农田产出 | 单级增量 | 增量占比 | 感知 |
|:----:|:-------:|:-------:|:-------:|:----:|
| Lv5 | 3.0 | +0.8 | +36% | 明显 |
| Lv10 | 8.0 | +1.05 | +15% | 可感知 |
| Lv15 | 16.0 | +1.6 | +11% | 勉强 |
| Lv20 | 28.0 | +2.6 | +10% | 勉强 |
| Lv25 | 45.0 | +3.7 | +9% | 弱 |

**分析**: 虽然绝对增量在增加（0.8→3.7），但相对增量在下降（36%→9%）。加上主城加成和科技加成后，相对增量更低。Lv15 后每次升级的感觉是"升了等于没升"。

#### 9.2.4 曲线平滑度分析

**费用曲线**: 分 4 段，每段倍率递减（×1.8→×1.6→×1.5→×1.4），设计意图是缓和后期增长，但效果不足。

**时间曲线**: 同样分 4 段（×2→×1.8→×1.5→×1.3），后期时间增长仍然过快。

**产出曲线**: 线性增长（每级 +1~4），与指数增长的费用形成剪刀差。

**建议**: 产出应采用超线性增长（如每级增量递增），或费用采用亚指数增长（如对数增长）。

### 9.3 机制有趣度评估

#### 9.3.1 决策深度

**当前**: 几乎无决策
- 8 座建筑独立运作，无互相影响
- 升级策略: 均衡升级即可（PRD 推荐顺序固定）
- 无资源分配权衡（产出建筑产出不同类型资源，不存在竞争）
- 无机会成本（升级 A 不影响升级 B，只要有队列槽位）

**问题**: 缺乏"选择 A 就不能选择 B"的决策张力。

#### 9.3.2 策略多样性

**当前**: 策略单一
- PRD 给出固定推荐顺序: 主城→农田→市集→兵营→...
- 所有玩家的最优路径相同
- 无"军事流"vs"经济流"vs"科技流"的分化
- 建筑协同缺失导致无组合策略

**问题**: 缺乏"不同玩家有不同玩法"的多样性。

#### 9.3.3 互动性

**当前**: 互动性极低
- 建筑之间无互动（无协同/对抗）
- 建筑与玩家之间只有"升级"一个操作
- 无主动技能（如"加速产出"、"临时buff"）
- 无建筑事件（如"丰收"、"瘟疫"）

**问题**: 建筑是静态的数值容器，不是活的游戏实体。

#### 9.3.4 反馈及时性

**当前**: 反馈存在但不够丰富
- 升级开始: Toast 提示 + 进度条 ✅
- 升级完成: Toast + 外观变化 ✅
- 产出增加: 数字飘出 ✅
- **缺失**: 升级后的即时奖励、里程碑庆祝、满级特效

#### 9.3.5 长期目标

**当前**: 目标模糊
- 短期目标: 解锁全部 8 座建筑（~5天）✅
- 中期目标: ??? （无明确目标）
- 长期目标: 全建筑满级（~2年，不现实）❌

**问题**: Lv5 之后玩家不知道"下一步做什么"，缺乏阶段性里程碑。

## 十、PRD 修订建议

### PRD-R001 医馆角色重新设计 [优先级: P0]

**PRD 原文**: 医馆产出伤兵恢复速率，Lv1→2 +5%，Lv20 +60%

**问题分析**:
1. **无感知**: +3%~+60% 恢复速率是被动百分比加成，玩家无法直接感知效果
2. **无战斗依赖**: 当前无战斗系统，恢复速率完全无意义
3. **无资源产出**: 医馆是唯一不产出任何资源的建筑（城墙至少有城防值）
4. **无主动交互**: 只有升级操作，无任何主动技能
5. **存在感最低**: 在 8 座建筑评估中，5 个维度全部 1~2 星

**玩家视角**: "医馆升到 Lv10 有什么用？我感受不到任何区别。为什么要花资源升级它？"

**数值佐证**:
- 医馆 Lv1: 恢复速率 +3% — 完全无感
- 医馆 Lv10: 恢复速率 +30% — 仍无感（无战斗场景）
- 医馆 Lv20: 恢复速率 +60% — 仍无感
- 升级费用: Lv1→2 仅需 100 粮草 + 150 铜钱，但无升级动力

**修订方案**:

**方案 A: 改为产出"伤兵"资源**
- 医馆每秒恢复 N 个伤兵→可用兵力
- 产出类型: resourceType = 'troops'（与兵营相同但来源不同）
- Lv1: 0.3 伤兵/秒, Lv20: 12 伤兵/秒
- 优点: 有直接产出感知，与兵营形成互补
- 缺点: 与兵营产出类型相同，可能混淆

**方案 B: 增加主动技能"治疗"**
- 消耗粮草立即恢复兵力（CD 30分钟）
- 恢复量 = 医馆等级 × 100
- Lv1: 恢复 100 兵力, Lv20: 恢复 2000 兵力
- 优点: 增加交互性，玩家主动使用
- 缺点: 需要设计 CD 和消耗平衡

**方案 C: 增加 buff 系统**
- 主动技能"治疗": 消耗粮草，获得 30 分钟攻击/防御加成
- Lv1: +5% 攻击, Lv20: +30% 攻击
- 优点: 提供策略选择（何时使用 buff）
- 缺点: 需要战斗系统支持

**推荐**: 方案 B（主动技能），兼顾感知和交互性

**影响评估**: 使医馆从"无感被动"变为"有感主动"，提升 8 座建筑的整体玩法深度
**优先级**: P0

### PRD-R002 建筑数量扩充评估 [优先级: P0]

**PRD 原文**: 8 座建筑，主城 Lv5 后无新建筑

**问题分析**:
1. **解锁节奏过快**: 8 座建筑在主城 Lv5 前全部解锁完毕（约 5 天内）
2. **中期真空**: Lv5→Lv30 的 25 级升级过程中无新建筑、无新机制
3. **网格浪费**: 6×5 网格有 30 格但只有 8 座建筑，22 格空地无意义
4. **行业对比**: SLG 游戏通常有 20+ 种建筑（如《率土之滨》30+种、《三国志战略版》25+种）
5. **玩家预期**: 玩家在 Lv5 后期待新内容，但系统无法提供

**玩家视角**: "Lv5 之后就是无限等待升级，没有任何新鲜感。6×5 的网格只放了 8 个建筑，剩下的空地有什么用？"

**修订方案**:

**短期方案（优先实现）: 建筑特化系统**
- Lv10 后每个建筑可选择 2 种特化方向
- 例: 农田→"精耕细作"(产出+30%) 或 "屯田制"(容量+50%)
- 例: 兵营→"精锐训练"(兵力质量+20%) 或 "大规模征兵"(兵力数量+30%)
- 优点: 不增加建筑数量，但增加决策深度
- 实现成本: 低（仅增加配置和 UI）

**中期方案: 新增高级建筑**
- Lv10 解锁: "工坊"(装备打造) 或 "酒馆"(武将招募)
- Lv15 解锁: "藏宝阁"(稀有资源) 或 "演武场"(武将经验)
- Lv20 解锁: "祭坛"(特殊 buff) 或 "商会"(贸易)
- 优点: 持续提供新鲜感
- 实现成本: 中（需要新建筑系统）

**长期方案: 建筑组合/协同系统**
- 特定建筑组合产生额外效果
- 例: 农田+市集 Lv10 = "商贸繁荣"(产出+15%)
- 例: 兵营+铁匠铺 Lv10 = "精锐之师"(兵力质量+20%)
- 优点: 增加策略深度和重玩价值
- 实现成本: 中（需要协同效果系统）

**影响评估**: 大幅延长游戏新鲜感，增加决策深度
**优先级**: P0

### PRD-R003 后期升级数值优化 [优先级: P0]

**PRD 原文**: Lv25→30 费用 ×1.4/级, 时间 ×1.3/级

**问题分析**:
1. **时间荒谬**: Lv29→30 升级时间 83,216,586 秒 = 2.64 年
2. **ROI 崩溃**: Lv20→21 农田回本时间 17 天，升级完全不划算
3. **产出-费用剪刀差**: 产出线性增长 vs 费用指数增长
4. **全满级不可达**: 全建筑满级预期 ~2 年，完全不现实
5. **玩家流失**: Lv10 后等待时间过长，玩家必然流失

**玩家视角**: "升级一个建筑要等 2 年？这游戏是认真的吗？Lv10 之后每次升级都要等好几天，产出却只增加一点点。"

**数值佐证**:
| 等级 | 农田产出 | 升级费用 | 升级时间 | 回本时间 |
|:----:|:-------:|:-------:|:-------:|:-------:|
| Lv10→11 | +1.6/秒 | 30,234 | 604s | 5.25小时 |
| Lv15→16 | +1.6/秒 | 298,419 | 4,273s | 2.16天 |
| Lv20→21 | +2.2/秒 | 2,111,549 | 21,789s | 11.1天 |
| Lv24→25 | +3.7/秒 | 8,111,725 | 62,232s | 25.4天 |
| Lv29→30 | +2/秒 | 80,808,755 | 83,216,586s | 2.64年 |

**修订方案**:

**方案 1: 降低后期升级时间**
- Lv25→30 升级时间从 ×1.3/级 降低到 ×1.15/级
- Lv29→30 时间从 2.64 年降至 ~120 天
- 全满级预期时间调整为 ~150 天

**方案 2: 提高后期产出速率**
- Lv20+ 产出增长从线性改为超线性
- Lv20→25: 产出每级 +4~6（而非当前 +2~3）
- Lv25→30: 产出每级 +8~12
- 效果: 回本时间从 25 天降至 ~8 天

**方案 3: 引入加速机制**
- 铜钱加速: 花费升级费用 ×0.3 的铜钱，减少 50% 时间
- 天命加速: 使用天命道具，减少 30% 时间
- 元宝秒完成: 花费元宝立即完成（付费转化）

**推荐**: 方案 1 + 方案 3 组合

**影响评估**: 保持长期追求感的同时不至于荒谬，引入加速机制增加付费转化
**优先级**: P0

**影响评估**: 增加升级决策深度，鼓励差异化发展，提升重玩价值
**优先级**: P1

### PRD-R006 产出模型统一 [优先级: P1]

**PRD 原文**: 未明确产出模型

**问题分析**:
1. **模型矛盾**: PRD 提到"一键收取"暗示"先累积后收取"，但源码使用实时累加模型
2. **离线收益**: 两种模型的离线收益计算不同
   - 实时累加: 离线期间资源持续累加，登录时直接获得
   - 累积制: 离线期间产出累积在建筑中，登录后需要手动收取
3. **资源上限**: 累积制需要资源上限机制，实时累加则不需要

**修订方案**:

**统一为实时累加模型**:
- 资源实时累加到资源系统，无需手动收取
- 一键收取仅为 UI 快捷操作 + 收取动画（视觉效果）
- 离线期间资源持续累加，登录时直接获得
- 离线效率系数: PRD 规定 ×1.2（建造额外 20% 效率），引擎需实现

**资源上限机制**（可选）:
- 引入资源上限 = base + farmland/market/barracks 等级 × 系数
- 超过上限后产出停止累加（溢出丢弃）
- 一键收取在资源接近上限时有实际意义

**影响评估**: 消除 PRD 与引擎的矛盾，简化系统设计
**优先级**: P1
