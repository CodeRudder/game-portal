## 二、Challenger 质疑裁决

### 2.1 P0 质疑裁决

| 质疑ID | 描述 | Challenger评级 | Arbiter裁决 | 理由 |
|--------|------|:-------------:|:----------:|------|
| CH-F032 | BLD-F10 资源溢出处理缺失 | P0 | **确认P0** | BLD-F10步骤5提到"如有资源上限则显示'已满'提示"，但PRD-R009(资源上限机制)虽被R1-Arbiter接受，R2-Builder并未设计具体方案。一键收取依赖资源上限(R009)但R009未落地，是**未解决的前置依赖**。离线场景更严重: VIP3+离线持续累加，无上限则登录后资源可升10级，破坏升级节奏。 |
| CH-F034 | BLD-F11 三种加速方式严重失衡 | P0 | **确认P0** | 数据验证: 铜钱加速(-50%时间，费用×0.3)碾压天命加速(-30%时间，消耗道具)。天命加速在任何场景下都不优于铜钱加速，是**严格劣选策略**。元宝秒完成定价阶梯跳跃过大(59分钟=10元宝，61分钟=50元宝)，边界效应严重。三种加速方式应各有适用场景，当前设计导致天命加速沦为鸡肋。 |
| CH-F011 | 医馆僵尸实体(遗留) | P0 | **接受延迟至R3，附加条件** | R2核心目标是"补齐PRD游戏步骤"(用户反馈)，医馆重设计不在R2范围内。但医馆在自动升级优先级中固定为30(BLD-F12-01)，说明系统已"放弃"医馆。**附加条件**: R3必须将医馆重设计作为第一优先级(PRD-R001方案B)，不能再延迟。 |
| CH-F021 | 建筑间无协同机制(遗留) | P0 | **接受延迟至R3，附加条件** | R2自动升级优先级算法(BLD-F12-01)基于独立建筑ROI，未考虑协同效应。若R3引入协同系统，优先级算法需完全重写。**附加条件**: R3引入协同系统时必须同步更新优先级算法，将协同效应纳入ROI计算。 |

**P0裁决汇总**: 2个确认(CH-F032/CH-F034)，2个接受延迟但附加条件(CH-F011/CH-F021)

## 五、R3 改进方向

### 5.1 用户反馈: "先补充游戏步骤"

R2已完成PRD要求的3个游戏步骤(一键收取/加速/自动升级)，PRD覆盖率达100%。R3应聚焦**修复新增流程的设计缺陷**+**补齐仍缺失的游戏步骤**:

### 5.2 R3 必须完成的任务

| # | 任务 | 优先级 | 来源 | 说明 |
|---|------|:------:|------|------|
| 1 | **医馆重设计** | P0 | PRD-R001 + CH-F011 | 方案B(主动技能"治疗"): 消耗粮草立即恢复兵力，CD 30分钟。使医馆从僵尸实体变为有感实体。同步更新自动升级优先级(BLD-F12-01)。 |

### R2→R3 改进建议

**问题1: 新增流程的一致性检查不足**
- 现象: BLD-F10(一键收取)与PRD-R006(产出模型)存在根本矛盾；BLD-F11(加速)三种方式严重失衡
- 根因: Builder补齐流程时，未充分检查新流程与已有流程/PRD修订的一致性
- 改进: R3 Builder新增流程时必须进行**一致性自检**: 新流程与已有流程是否矛盾？子步骤是否自洽？数值是否平衡？

**问题2: PRD修订的执行跟踪不足**
- 现象: PRD-R009(资源上限)被R1-Arbiter接受，但R2-Builder未执行，导致BLD-F10依赖断裂
- 根因: 缺乏PRD修订的执行跟踪机制，"接受"不等于"已执行"
- 改进: R3 Arbiter必须建立**PRD修订执行清单**，每轮检查已接受PRD-R的执行状态

**问题3: 遗留P0问题的交付时间线缺失**
- 现象: CH-F011(医馆)、CH-F021(协同)被延迟，但无明确交付时间线和验收标准
- 改进: 延迟的P0必须有**目标轮次+交付标准+验收条件**，避免无限延迟

**问题4: 离线场景覆盖不足**
- 现象: BLD-F11(加速)、BLD-F10-01(VIP离线)、PRD-R006(离线效率)三者之间存在多个交叉盲区
- 改进: R3必须统一设计**离线场景完整流程**: 离线资源产出规则、离线建造进度、离线加速规则、VIP离线特权——作为一个整体设计，而非分散在各流程中

## 七、总结

### R2 核心成果

1. **PRD覆盖率100%**: 3个缺失游戏步骤(一键收取/加速/自动升级)全部补齐
2. **数值优化采纳**: Lv29→30从2.64年降至118天，全满级预期~150天
3. **产出模型统一**: 确定实时累加模型，消除PRD与引擎矛盾
4. **武将派驻流程**: XI-007补齐跨系统加成链

### R2 核心问题

1. **一键收取是"假按钮"**: 实时累加模型下收取无实际效果，需采纳PRD-R012混合模型
2. **加速方式严重失衡**: 天命加速被铜钱加速碾压，需采纳PRD-R013重平衡
3. **资源上限未设计**: PRD-R009被接受但未执行，是BLD-F10的断裂依赖
4. **离线场景多处盲区**: VIP离线规则矛盾、离线加速规则缺失、1.2系数适用范围不明

### R2 在整体收敛中的位置

```
R1(6.6) → R2(7.2) → R3(目标8.0~8.5) → R4(目标9.0+封版)
  │           │              │                  │
  │           │              │                  └── 收敛验证，封版
  │           │              └── 修复设计缺陷 + 协同/特化 + 医馆重设计
  │           └── PRD流程补齐 + 数值优化采纳 + 武将派驻
  └── 核心循环覆盖 + 问题发现
```

R2完成了"补齐游戏步骤"的核心目标(用户反馈)，但新增流程的设计质量未达预期。R3的重心应从"补步骤"转向"修质量"——修复一键收取/加速/离线的设计缺陷，同时推进医馆重设计和建筑特化/协同系统。
