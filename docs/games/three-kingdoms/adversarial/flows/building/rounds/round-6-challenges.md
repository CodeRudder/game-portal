# 建筑系统 Round 6 — Challenger 质疑报告

> Challenger: FlowChallenger v4.0 | 时间: 2026-05-02
> 基于: R6-Builder(round-6-flows.md) + R5-Challenger(round-5-challenges.md)
> 方法论: 游戏步骤推演——逐步推演玩家操作和内心OS，找出"看起来完整但实际玩起来有问题"的环节

---

## 总览

R6-Builder提交了4个重大新增系统(医馆损失框架、事件重设计、主动决策、建筑进化)，PRD 25/25全覆盖，GAP保持0。指标层面非常漂亮。

但Challenger的核心质疑是: **这4个系统在实际游玩中是"好玩的游戏机制"还是"又一堆会被忽略的信息面板"？**

**质疑统计**: P0 x 2, P1 x 7, P2 x 6, 共15项

| 严重度 | 数量 | 核心主题 |
|:------:|:----:|---------|
| P0 | 2 | 进化重置代价致命、事件气泡必然被忽略 |
| P1 | 7 | 损失报告骚扰、焦点无决策、挑战变强迫症、巡查例行公事、进化数值倒退、9因子复杂度、交叉影响 |
| P2 | 6 | 损失框架认知负担、事件超时浪费、挑战跳过付费墙、巡查概率不透明、进化2星瓶颈、产出公式不可读 |

## 一、医馆损失框架质疑

### CH-F092 [P1] 损失框架治愈报告——每日弹窗会不会变成骚扰？

**问题**: BLD-F19-04设计"每日首次登录弹出损失框架报告"，但R5的治愈报告已经被设计为每日弹窗，R6只是换了内容。每日弹窗的宿命是: 第1天新鲜，第3天习惯，第7天直接点"知道了"关闭。

**游戏步骤推演**:

```
步骤1: 玩家第1天登录，弹出"医馆Lv10每日价值报告"
步骤2: 玩家看到"今日医馆为您节省 12,345 粮草"
步骤3: 玩家内心OS: "哦，还不错" → 点击"知道了"
步骤4: 玩家第2天登录，又弹出同样的报告，数字变成"13,102"
步骤5: 玩家内心OS: "和昨天差不多" → 快速点击"知道了"
步骤6: 玩家第7天登录，弹出报告
步骤7: 玩家内心OS: "又是这个……" → 条件反射点击"知道了"，完全不看内容
步骤8: 玩家第30天登录，弹出报告
步骤9: 玩家内心OS: "这东西能不能关掉？" → 去设置里关闭
```

**核心问题**: 损失框架报告的"信息价值"随时间递减，但"打断感"恒定不变。

- **第1天**: 新鲜感，玩家认真看——有效
- **第7天**: 习惯化，玩家快速跳过——失效
- **第30天**: 厌烦，玩家关闭——变成僵尸功能

**心理学依据**: 这是"习惯化效应"(habituation)——重复暴露于同一刺激，反应强度持续下降。损失框架报告必须有"变化"才能维持注意力。

**建议方向**:
- 报告内容应随时间变化: 第1天显示"节省量"，第7天显示"节省量排名(和其他玩家比)"，第14天显示"医馆升级ROI预测"
- 或: 报告不弹窗，而是整合进主界面的"每日收益汇总"中，与其他信息一起展示
- 或: 报告只在"医馆可升级"时弹出，结合升级引导而非纯信息展示

### CH-F093 [P2] 未建造红色负值——信息过载还是有效引导？

**问题**: BLD-F19-03设计"医馆未建造时，在所有建筑产出详情中插入红色行"。但玩家在前期还没有医馆时，看到每个建筑都有一行红色"医馆未建造: -40%产出"，会产生什么感受？

**游戏步骤推演**:

```
步骤1: 新手玩家建造了农田(Lv3)、市集(Lv2)、兵营(Lv1)
步骤2: 玩家查看农田产出详情，看到:
   基础产出: 1.5 粮草/秒
   × 主城加成: +6%
   × 科技加成: +0%
   × 武将加成: +0%
   × 医馆加成: [未建造] -6% ← 红色闪烁
   × 协同+特化: +0%
   × 治疗Buff: +0%
   = 最终产出: 1.59 粮草/秒 (损失了 0.1 粮草/秒!)
步骤3: 玩家内心OS: "损失了0.1？这也叫损失？"
步骤4: 玩家去查看市集、兵营，每个都有同样的红色行
步骤5: 玩家内心OS: "每个建筑都说我损失了0.1~0.2，这点损失谁在乎？"
步骤6: 玩家习惯了红色行的存在，开始忽略它
步骤7: 当玩家真正需要建医馆时，已经对红色提示免疫了
```

**核心问题**: 红色负值的"损失量"在玩家前期微不足道(0.1~0.2粮草/秒)，但红色闪烁的"紧迫感"却很强。这种"信号强度"与"实际损失"的不匹配会导致:

- **前期**: 红色行制造焦虑但实际损失微小——玩家觉得系统在"虚张声势"
- **后期**: 玩家已经习惯忽略红色行——当损失真正变大时反而看不见

**数值验证**:
- 医馆Lv0，主城Lv5时: 医馆加成本应为0%(未建造)，而非"-6%"
- R6设计显示"假设医馆=主城等级"的损失，但这引入了一个假设: 玩家应该有医馆吗？如果医馆是可选建筑，为什么用红色提示"你没选它"？

**建议方向**:
- 红色负值只在"医馆已解锁但未建造"时显示，而非所有阶段
- 损失量应使用绝对值("损失 0.1 粮草/秒"而非"-6%")，让玩家判断是否值得在意
- 每个建筑都显示同一行红色信息是信息冗余——应只在主界面的"全局加成总览"中显示一次

### CH-F094 [P2] 损失框架的认知负担——玩家需要同时理解"节省"和"损失"两套数字吗？

**问题**: BLD-F19同时展示了三套数字: 今日节省量(绿色正向)、降级损失量(红色警告)、无医馆损失量(红色加粗)。玩家需要同时理解这三个数字的关系。

**游戏步骤推演**:

```
步骤1: 玩家打开医馆损失报告，看到:
   今日医馆为您节省: 粮草 12,345
   若医馆降为Lv9，明日您将损失: 粮草 -3,840/天
   若无医馆，您每日损失: 粮草 -18,500/天
步骤2: 玩家内心OS: "12,345是今天省的，3,840是降级会少的，18,500是没有医馆会少的……等等，这三个数字是什么关系？"
步骤3: 玩家尝试计算: "12,345 + 3,840 = 16,185？不等于18,500啊"
步骤4: 玩家内心OS: "这些数字怎么来的？我算不清楚"
步骤5: 玩家放弃理解，只记住"医馆很重要"
```

**核心问题**: 三套数字的数学关系不直观:

- "今日节省 12,345" = 医馆Lv10在24小时内的贡献
- "降级损失 3,840/天" = Lv10 vs Lv9的差异 × 24小时
- "无医馆损失 18,500/天" = Lv10的全部贡献 × 24小时

关系: 12,345 ≈ 18,500 × (今日已过时间/24小时)。但玩家不会做这个计算。

**建议方向**:
- 只展示1~2个核心数字，而非3个。建议: "今日节省" + "无医馆损失"(去掉"降级损失"，因为降级场景罕见)
- 或: 用可视化图表(柱状图)替代数字，让比例关系一目了然

## 二、事件系统质疑

### CH-F095 [P0] 非强制气泡=玩家必然忽略——事件系统等于不存在

**问题**: BLD-F20-01设计"事件气泡为非强制通知，玩家可忽略继续游戏"。但"非强制"在实际游戏中的宿命是: 大部分玩家会忽略它。

**游戏步骤推演(休闲玩家)**:

```
步骤1: 玩家早上登录游戏，目标是"收取资源+升级建筑"
步骤2: 看到某座建筑上方有金色气泡在浮动
步骤3: 玩家内心OS: "什么东西？先收资源再说"
步骤4: 玩家点击收取按钮，开始升级建筑
步骤5: 气泡还在那里浮动
步骤6: 玩家内心OS: "等我忙完再看"
步骤7: 玩家忙完(升级开始倒计时)，准备退出
步骤8: 玩家内心OS: "那个气泡……算了，下次再看"
步骤9: 次日0点，气泡消失
步骤10: 重复10天，玩家从未点击过气泡
```

**游戏步骤推演(硬核玩家)**:

```
步骤1: 玩家早上登录，看到气泡
步骤2: 玩家内心OS: "又是事件？上次奖励才500粮草，不值得打断我的升级节奏"
步骤3: 玩家继续执行原有计划(升级/收取/挑战)
步骤4: 气泡被忽略
```

**核心问题**: 非强制=低优先级=被忽略。这是游戏设计的基本规律。

- **SLG玩家的登录目标**: 收取资源、升级建筑、参加活动。事件气泡不在核心目标中
- **气泡的视觉权重**: 一个小小的金色气泡，相比主界面的资源数字、升级按钮、挑战面板，视觉优先级太低
- **奖励预期**: 如果事件奖励不够有吸引力，玩家没有动力点击; 如果奖励太强，不点击就变成"隐性损失"
- **"非强制"的悖论**: 如果大部分玩家忽略气泡，事件系统21种事件(7类建筑×3种)等于不存在，R6-Builder精心设计的"双向收益选择"全部白费

**数据推演**:
- 假设玩家平均每日登录3次，每次停留5分钟
- 事件气泡在建筑卡片上方，玩家需要"看到+决定点击+查看+选择"约30秒
- 如果玩家的核心操作(收取/升级/挑战)需要4分钟，只剩1分钟浏览
- 气泡被点击的概率估计: 30%~50%(乐观)

**建议方向**:
- 气泡应该有"渐进式紧迫感": 第1小时金色闪烁→第2小时红色闪烁→第3小时气泡变大+显示奖励预览
- 或: 事件不依赖气泡，而是整合进"每日任务"系统——"完成1个建筑事件"作为每日任务之一
- 或: 事件奖励必须足够吸引人(至少相当于5分钟产出)，让玩家有动力点击
- 或: 气泡点击后直接在当前页面展开(而非弹出半屏面板)，减少操作打断

### CH-F096 [P2] 事件超时=免费放弃——每日1次的机会被浪费

**问题**: BLD-F20-01设计"若玩家当日未点击查看，次日0点气泡消失，不触发冷却，不计入今日已触发"。这意味着玩家可以"免费跳过"事件而不受惩罚。

**游戏步骤推演**:

```
步骤1: 玩家某天很忙，登录只做了收取就退出
步骤2: 气泡未被点击查看
步骤3: 次日，气泡消失，todayEventTriggered重置
步骤4: 玩家再次登录，触发新事件
步骤5: 玩家内心OS: "昨天那个事件错过了也没关系嘛"
步骤6: 从此玩家把事件气泡视为"可选的额外内容"，而非"每日必做的任务"
```

**核心问题**: 无惩罚=无动力。当跳过事件没有任何代价时，事件系统沦为"可有可无的装饰"。

**对比**: 如果每日任务不完成会怎样？大部分SLG会扣除活跃度或错过奖励。但事件气泡跳过什么都不会发生。

**建议方向**:
- 跳过事件应该有"累积机制": 连续3天未点击事件，第4天事件奖励翻倍(补偿+激励)
- 或: 事件不消失，而是累积(最多3个)，玩家想看时一起看
- 或: 跳过事件计入"活跃度"系统，影响每日活跃度奖励

### CH-F097 [P1] 事件选择"双向收益"——玩家如何判断哪个更优？

**问题**: BLD-F20-02设计"两个选择都是收益，方向不同(即时vs持续)"。但"即时收益"和"持续收益"的比较需要玩家做数学计算。

**游戏步骤推演**:

```
步骤1: 玩家看到农田事件"丰收之年":
   选择A: 立即获得2000粮草
   选择B: 粮草产出+10%持续4小时
步骤2: 玩家内心OS: "2000粮草直接拿，还是+10%持续4小时？哪个更值？"
步骤3: 玩家尝试计算:
   "我的粮草产出是13.7/秒，+10%就是+1.37/秒"
   "4小时 = 14,400秒"
   "1.37 × 14,400 = 19,728 粮草"
步骤4: 玩家内心OS: "选择B相当于19,728粮草，比选择A的2000多很多！选B！"
步骤5: 但实际上，玩家在这4小时内可能不会全程在线
步骤6: 如果玩家只在线1小时，实际收益 = 1.37 × 3,600 = 4,932 粮草
步骤7: 玩家内心OS: "等等，4小时是游戏时间还是在线时间？"
```

**核心问题**: "持续N小时"的收益取决于玩家的在线时长，但系统没有说明这一点。

- **全时在线玩家**: 持续收益 >> 即时收益，选B永远最优
- **碎片化玩家**: 持续收益可能 << 即时收益，选A更稳
- **系统没有提示**: 玩家不知道"4小时"是游戏时间还是在线时间

**建议方向**:
- 事件选择面板应显示"预期收益"(基于玩家的平均在线时长)
- 或: 持续效果改为"离线也生效"(降低决策门槛)
- 或: 两个选择的期望收益设计为接近相等(如A=2000即时，B=2500持续4小时)，让选择变成"风格偏好"而非"数学题"

## 三、主动决策质疑

### CH-F098 [P1] 建筑焦点——"标记2个建筑"然后忘记，算决策吗？

**问题**: BLD-F21-01设计"标记1-2座建筑为焦点，获得+5%产出加成"。但这个操作的决策深度极浅。

**游戏步骤推演**:

```
步骤1: 玩家第1天登录，看到"建筑焦点"功能解锁
步骤2: 玩家内心OS: "标记2个建筑获得+5%？那我标记产出最高的2个"
步骤3: 玩家标记农田(产出最高)和市集(产出第二高)
步骤4: 焦点标记完成，+5%加成生效
步骤5: 玩家内心OS: "好了，以后不用管了"
步骤6: 30天后，玩家从未修改过焦点标记
步骤7: 焦点系统从"决策"变成了"一次性设置"
```

**核心问题**: 焦点的最优策略是"标记产出最高的2个建筑"，这个决策在游戏初期就能做出，且几乎不需要修改。

- **决策频率**: 一次性(标记后忘记)
- **决策深度**: 极浅(标记产出最高的=最优)
- **决策后果**: +5%产出，感知微弱(R5已经论证过5%以下的变化玩家感受不到)

**数值验证**:
- 假设农田产出13.7粮草/秒，焦点+5% = +0.685粮草/秒
- 每日额外产出 = 0.685 × 86,400 = 59,184粮草
- 59,184粮草看起来不少，但对比每日总产出(约118万)，只占5%
- 玩家能感知到"今天比昨天多了5%"吗？大概率不能

**建议方向**:
- 焦点应该有"动态效果": 每次切换焦点有冷却(如24小时)，让切换本身成为决策
- 或: 焦点加成应该更大(+15%~20%)，让"选择哪个建筑"变成有意义的决策
- 或: 焦点应该有"联动效果"——焦点建筑的升级速度+20%，让焦点影响升级策略
- 或: 焦点数量从2个改为1个，强迫玩家在"产出"和"其他功能"之间做取舍

### CH-F099 [P1] 每日挑战——"限时升级"是决策还是强迫症？

**问题**: BLD-F21-02设计"每日挑战: 在N小时内升级某建筑X次"。但这个机制的本质是"系统告诉你该升级什么"，而非"玩家自己决定升级什么"。

**游戏步骤推演**:

```
步骤1: 玩家登录，看到今日挑战: "在6小时内将农田升级2次"
步骤2: 玩家内心OS: "农田？我本来想升兵营的……但挑战奖励是5000粮草+10元宝"
步骤3: 玩家检查农田升级费用: Lv12→13需要8,000粮草，Lv13→14需要12,000粮草
步骤4: 玩家内心OS: "花20,000粮草升级农田，换5000粮草+10元宝？亏了啊"
步骤5: 玩家内心OS: "但不完成挑战就浪费了免费元宝……"
步骤6: 玩家被迫升级农田(而非自己想升的兵营)
步骤7: 玩家内心OS: "我被系统逼着升了不想升的建筑"
```

**核心问题**: 每日挑战的"目标建筑"是随机的，可能与玩家当前的升级计划冲突。

- **当挑战目标=玩家想升级的建筑**: 挑战是"白送奖励"——没有决策
- **当挑战目标≠玩家想升级的建筑**: 挑战是"强迫升级"——负面体验
- **50元宝跳过机制**: 变成"付费跳过不想要的任务"——Pay-to-Skip

**数值验证**:
- 挑战奖励: 粮草×5000 + 元宝×10
- 挑战成本: 2次升级的粮草(假设20,000~50,000) + 2次升级的等待时间(假设2~4小时)
- 净收益: 5000-50000=-45000粮草(亏) + 10元宝(赚)
- 玩家的核心矛盾: 挑战奖励中的元宝是稀缺资源，但粮草奖励对比升级成本是亏损的

**建议方向**:
- 挑战目标应该是"升级任意建筑X次"，而非"升级特定建筑"——让玩家自己选择升级哪个
- 或: 挑战奖励应该远大于升级成本(如奖励=升级费用的150%)，让挑战变成"净赚"
- 或: 挑战改为"升级你的焦点建筑"——与焦点系统联动，而非随机选择
- 或: 提供3个挑战选项让玩家选择1个完成——增加决策感

### CH-F100 [P2] 建筑巡查——"每4小时点一下"是互动还是例行公事？

**问题**: BLD-F21-03设计"访问建筑详情时，若冷却已过，触发巡查奖励"。但这个机制的操作模式是"每4小时点一下建筑→看到奖励→关闭"，决策深度为零。

**游戏步骤推演**:

```
步骤1: 玩家登录，进入建筑列表
步骤2: 点击农田查看详情 → "巡查发现! 获得粮草×1,200"
步骤3: 玩家内心OS: "哦" → 关闭详情
步骤4: 点击市集查看详情 → "巡查发现! 获得铜钱×800"
步骤5: 玩家内心OS: "哦" → 关闭详情
步骤6: 重复点击所有建筑
步骤7: 玩家内心OS: "每4小时把所有建筑点一遍，这就是巡查？"
步骤8: 3天后，玩家把"点所有建筑"变成例行操作，毫无乐趣
```

**核心问题**: 巡查没有"巡查感"。真正的巡查应该是"发现问题→解决问题→获得奖励"，而非"点一下→领奖→走人"。

- **随机奖励**: 60%资源/20%加速/15%材料/5%特殊——奖励类型是随机的，玩家无法预期
- **概率不透明**: 玩家不知道5%的特殊奖励有多稀有，可能连续巡查30次都拿不到元宝
- **操作无意义**: "点击建筑详情"本身不是有趣的操作，只是触发奖励的手段

**建议方向**:
- 巡查应该有"发现问题"环节: 建筑上出现小问题(如"农田有杂草")，玩家点击后选择处理方式(手动清理/花费资源/忽略)
- 或: 巡查奖励应该与建筑状态相关(如"农田产出即将溢出，巡查可提前收取")
- 或: 巡查改为"建筑互动"——点击建筑后出现小游戏(如快速点击收资源)而非直接领奖

## 四、进化系统质疑

### CH-F101 [P0] 进化重置Lv1——玩家愿意让满级建筑"归零"吗？

**问题**: BLD-F22-02设计"进化后等级重置为Lv1"。这意味着玩家花了数周升到Lv25的建筑，一瞬间回到Lv1。产出从满级直接跌到谷底。

**游戏步骤推演**:

```
步骤1: 玩家的农田终于升到Lv25(满级)，产出45.2粮草/秒
步骤2: 玩家看到"进化"按钮: "进化至1星，等级重置为Lv1，永久获得+10%产出"
步骤3: 玩家内心OS: "重置到Lv1？那我的产出不是直接从45降到1.0？"
步骤4: 玩家计算:
   - Lv25产出: 45.2 粮草/秒
   - Lv1产出: 0.8 粮草/秒
   - 1星加成: 0.8 × 10% = 0.08 粮草/秒
   - 进化后产出: 0.88 粮草/秒
   - 产出暴跌: 45.2 → 0.88，下降98%！
步骤5: 玩家内心OS: "我花了3周升到满级，现在要我从头再来？而且产出只有0.88？"
步骤6: 玩家内心OS: "就算1星加成是永久的，我也要再花3周才能回到Lv25的产出"
步骤7: 玩家内心OS: "不进化的话，45.2粮草/秒一直有。进化的话，要忍受3周的产出低谷"
步骤8: 玩家决定不进化
```

**核心问题**: 进化的"短期代价"(产出暴跌98%)远远大于"长期收益"(+10%永久加成)。

**数值验证(1星进化ROI)**:

```
进化前: Lv25产出 = 45.2 粮草/秒
进化后: Lv1产出 = 0.88 粮草/秒 (0.8 × 1.1)

回到Lv25需要: 从Lv1升到Lv25
  - 假设平均升级时间 = 2小时/级，25级 = 50小时(约2天)
  - 假设平均升级费用 = 5000粮草/级，25级 = 125,000粮草

回到Lv25后的产出: 45.2 × 1.1 = 49.72 粮草/秒
产出增益: 49.72 - 45.2 = 4.52 粮草/秒
每日额外产出: 4.52 × 86,400 = 390,528 粮草

进化消耗: 50,000粮草 + 30,000铜钱 + 100元宝
升级回Lv25消耗: 125,000粮草

总消耗: 175,000粮草 + 30,000铜钱 + 100元宝
回本时间: 175,000 / 390,528 = 0.45天(仅粮草部分)

看起来0.45天就能回本？但问题在于:
  - 升级回Lv25的50小时内，产出从45.2暴跌到逐步恢复
  - 这50小时内的产出损失 = (45.2 - 平均产出) × 50小时
  - 假设线性恢复，平均产出 ≈ (0.88 + 49.72) / 2 = 25.3
  - 产出损失 = (45.2 - 25.3) × 50 × 3600 = 3,582,000粮草

真实回本时间: (175,000 + 3,582,000) / 390,528 ≈ 9.6天
```

**结论**: 1星进化的真实回本时间约10天。这还不算100元宝的机会成本。玩家需要忍受10天的产出低谷，换来之后每天多39万粮草。

**更严重的问题**: 进化到2星需要再次从Lv1升到Lv30(进化后上限提升)。这意味着玩家要经历两次"产出归零"。3星则要经历三次。

**建议方向**:
- 进化不应重置等级，而是"等级锁定"(如进化后等级变为Lv25→Lv20，而非Lv1)
- 或: 进化后等级重置但产出不完全归零(如保留50%的基础产出)
- 或: 进化提供"保护期"(进化后72小时内产出不变，让玩家有时间升级)
- 或: 进化消耗改为"满级建筑暂停产出N小时"，而非"重置等级"

### CH-F102 [P1] 星级加成的递增设计——1星+10% vs 2星+25%，中间的15%去哪了？

**问题**: BLD-F22-03设计"1星+10%，2星+25%(替代1星的10%)，3星+50%(替代2星的25%)"。注意是"替代"而非"叠加"。

**游戏步骤推演**:

```
步骤1: 玩家进化到1星，获得+10%永久加成
步骤2: 玩家再次升到满级(Lv30)，准备进化到2星
步骤3: 玩家看到: "2星进化: 永久获得+25%产出(替代1星的10%)"
步骤4: 玩家内心OS: "替代？那我1星的+10%没了？2星只比1星多了15%？"
步骤5: 玩家计算:
   - 1星→2星的净增益: 25% - 10% = 15%
   - 1星进化消耗: 5万粮草+3万铜钱+100元宝
   - 2星进化消耗: 15万粮草+10万铜钱+300元宝(3倍!)
   - 但净增益只有15%(而非25%)
步骤6: 玩家内心OS: "花3倍的代价，只多15%？性价比暴跌"
```

**核心问题**: "替代"设计让2星的实际增益被高估。

- **1星**: 0% → +10%，净增益10%，消耗5万粮草
- **2星**: +10% → +25%，净增益15%，消耗15万粮草(3倍)
- **3星**: +25% → +50%，净增益25%，消耗50万粮草(3.3倍)

每星的"性价比"(净增益/消耗)递减:
- 1星: 10% / 5万 = 0.0002%/粮草
- 2星: 15% / 15万 = 0.0001%/粮草 (下降50%)
- 3星: 25% / 50万 = 0.00005%/粮草 (再下降50%)

**建议方向**:
- 星级加成应改为"叠加"而非"替代": 1星+10%，2星+10%(累计+20%)，3星+10%(累计+30%)
- 或: 保持替代设计但降低高级进化消耗(2星消耗应为1星的2倍而非3倍)
- 或: 明确告知玩家"净增益"(面板显示"1星→2星: 净增+15%")

**问题**: BLD-F22-02设计"3星进化后等级上限提升至Lv40"。但从Lv1升到Lv40需要多少资源？

步骤5: R6-Builder没有定义Lv25以上的升级费用曲线

## 五、复杂度和系统交互质疑

### CH-F104 [P1] 9个乘法因子——新手能理解产出公式吗？

**问题**: R6的产出公式有9个乘法因子: 基础产出 × 主城 × 科技 × 武将 × 医馆 × (协同+特化) × 治疗 × 进化 × 焦点。

**游戏步骤推演**:

```
步骤1: 新手玩家查看农田产出详情，看到:
   基础产出: 3.0 粮草/秒
   × 主城加成: +10% (主城Lv5)
   × 科技加成: +0% (无科技)
   × 武将加成: +0% (无武将)
   × 医馆加成: +10% (医馆Lv5)
   × 协同+特化: +0% (无)
   × 治疗Buff: +0% (无)
   × 进化加成: +0% (无进化)
   × 焦点加成: +0% (非焦点)
   = 最终产出: 3.63 粮草/秒
步骤2: 玩家内心OS: "8行加成，大部分都是0%……这些有什么用？"
步骤3: 玩家尝试理解: "主城+10%和医馆+10%，为什么产出不是+20%而是+21%？"
步骤4: 玩家内心OS: "乘法？什么乘法？我只想知道我的建筑产多少粮草"
步骤5: 玩家放弃理解，只看最终数字
```

**核心问题**: 9个乘法因子对新手来说是"信息噪音"。

- **大部分因子在前期为0**: 新手看到8行加成中6行是+0%，会认为这些功能"没用"
- **乘法关系不直观**: "主城+10% × 医馆+10% = +21%"而非"+20%"，违反直觉
- **信息量 vs 理解力**: 系统展示了完整公式，但玩家无法从中获得有用信息

**建议方向**:
- 产出详情应分层展示: "基础层"(基础产出+主城+科技)和"加成层"(医馆+协同+进化+焦点)
- 或: 隐藏+0%的因子，只显示生效的加成
- 或: 提供"简化视图"(只显示最终产出)和"详细视图"(显示全部因子)两个模式

### CH-F105 [P1] 焦点(+5%)、进化(10%/25%/50%)、医馆(2%/级)的交叉影响

**问题**: R6新增了3个加成源(焦点、进化、医馆损失框架优化)，它们与原有加成的交互是否有问题？

**游戏步骤推演(老手玩家，全部加成)**:

```
步骤1: 老手玩家的农田产出计算:
   基础产出(Lv40): 80 粮草/秒
   × 主城加成(Lv30): 1.60
   × 科技加成: 1.15
   × 武将加成: 1.05
   × 医馆加成(Lv20): 1.40
   × (协同+特化): 1.55 (上限80%内)
   × 治疗Buff: 1.10
   × 进化加成(3星): 1.50
   × 焦点加成: 1.05
   = 80 × 1.60 × 1.15 × 1.05 × 1.40 × 1.55 × 1.10 × 1.50 × 1.05
   = 80 × 8.96
   = 716.8 粮草/秒

步骤2: 对比新手玩家(Lv5，无加成):
   基础产出: 3.0 粮草/秒
   × 所有加成: ~1.10 (只有主城)
   = 3.3 粮草/秒

步骤3: 差距 = 716.8 / 3.3 = 217倍!
```

**核心问题**: R5时新手vs老手差距是36倍，R6增加进化和焦点后，差距扩大到217倍。

- **R5差距**: 36倍(R5-CH-F088已标记为P1)
- **R6差距**: 217倍(增加了6倍!)
- **原因**: 进化加成(+50%)和焦点(+5%)是独立乘法层，直接放大差距

**数值验证**:
- 进化加成贡献: 1.50倍
- 焦点加成贡献: 1.05倍
- 新增总乘数: 1.50 × 1.05 = 1.575倍
- 但进化需要从Lv1重新升级，Lv40的基础产出远高于Lv25
- Lv40基础产出(推测) ≈ Lv25 × 2^6 = 45.2 × 64 = 2892.8? (需要R7定义)

**建议方向**:
- R7必须评估217倍差距是否可接受(对比SLG行业标准20~60倍)
- 考虑引入"加成递减机制": 当总加成超过某个阈值时，超出部分效果减半
- 或: 进化加成改为"加法叠加"(与其他加成同一层)而非"独立乘法层"

### CH-F106 [P2] 挑战跳过机制——50元宝是"便利"还是"付费墙"？

**问题**: BLD-F21-02设计"可用50元宝跳过挑战，获得全额奖励"。这引入了Pay-to-Skip机制。

**游戏步骤推演**:

```
步骤1: 玩家看到今日挑战: "在4小时内升级城墙1次"
步骤2: 玩家内心OS: "城墙？我不想升城墙，我缺粮草"
步骤3: 玩家看到跳过按钮: "50元宝跳过"
步骤4: 玩家检查元宝余额: 120元宝
步骤5: 玩家内心OS: "50元宝换粮草×2000+元宝×10？花50赚10，亏了"
步骤6: 玩家内心OS: "但不完成挑战就浪费了今天的免费元宝机会……"
步骤7: 玩家被迫选择: 花20,000粮草升级城墙，或花50元宝跳过
步骤8: 两个选择都是"亏损"，玩家感到被系统"卡脖子"
```

**核心问题**: 跳过机制的定价(50元宝)对比挑战奖励(10元宝)是"倒挂"的——跳过比完成更贵。

- **完成挑战**: 花20,000粮草 → 获得5,000粮草+10元宝(净亏15,000粮草)
- **跳过挑战**: 花50元宝 → 获得5,000粮草+10元宝(净亏40元宝)
- **不做挑战**: 获得0(错过奖励)

**三个选择都是"亏损"**，玩家没有"赚到"的选项。

**建议方向**:
- 挑战奖励应大于挑战成本(否则挑战是"惩罚"而非"奖励")
- 跳过费用应低于奖励元宝(如跳过=5元宝，奖励=10元宝)
- 或: 取消跳过机制，改为"提供3个挑战让玩家选1个"——总有1个符合玩家当前需求

| CH-F082 锻造30%概率 | BLD-F16-01: 每5次保底 | **有效** — 保留随机性但增加确定性 |
| CH-F083 兵营屯田铜钱 | BLD-F16-01: 训练速度替代铜钱 | **有效** — 功能一致性恢复 |
| CH-F087 事件假选择 | BLD-F20-02: 双向收益选择 | **部分有效** — 设计方向正确，但"即时vs持续"的比较需要玩家做数学(CH-F097) |
| CH-F090 主动操作偏少 | BLD-F21: 焦点+挑战+巡查 | **部分有效** — 增加了3个操作，但决策深度不足(CH-F098/099/100) |
| CH-F091 建筑里程碑缺失 | BLD-F22: 进化系统 | **部分有效** — 提供了长期目标，但进化代价太大(CH-F101) |

**总结**: R6-Builder对R5的9个质疑都给出了响应，但其中5个是"部分有效"——方向正确但执行有问题。核心原因是: **Builder在"增加功能"时没有充分考虑"玩家实际会怎么用"**。

### 7.2 R6 Challenger 方法论改进

**本轮创新**:

1. **"非强制=被忽略"定律**: 本轮发现了一个重要设计原则——非强制通知在玩家有明确目标时必然被忽略。这不是事件系统的问题，而是所有"非强制"通知的宿命。R7应将此定律纳入设计检查清单。

2. **"收益vs成本"倒挂检测**: 每日挑战和进化系统都存在"玩家付出的成本大于获得的收益"的问题。R7应引入"玩家净收益"计算——每个机制都必须通过"玩家操作后的资源变化是正还是负"的检查。

3. **"217倍差距"预警**: 9个乘法因子的叠加让新手vs老手差距从36倍扩大到217倍。R7必须评估这个差距是否会导致"新手永远追不上老手"的问题。

**R5→R6方法论改进效果**:

| R5复盘提出的改进 | R6执行情况 | 效果 |
|-----------------|-----------|:----:|
| "游戏步骤推演"替代数值分析 | R6全面使用 | **有效** — 推演发现了多个"看起来合理但玩起来有问题"的设计 |
| "假选择"深层挖掘 | R6在事件选择中应用 | **有效** — 发现了"即时vs持续"的数学比较问题 |
| "被动vs主动"框架 | R6通过焦点/挑战/巡查响应 | **部分有效** — 增加了主动操作，但决策深度仍不足 |

### 7.3 R6→R7 改进建议

1. **引入"玩家净收益"检查**: 每个新机制必须计算"玩家操作后的净资源变化"，如果净收益为负，机制设计有问题。

2. **引入"非强制机制存活率"评估**: 非强制通知/功能必须评估"在没有强制的情况下，多少比例的玩家会使用"。如果存活率<50%，需要增加引导机制。

3. **引入"乘法因子膨胀"监控**: 每新增一个独立乘法因子，必须计算"新手vs老手差距扩大倍数"。如果差距>100倍，需要引入递减机制。

4. **引入"操作-决策比"评估**: 每个主动操作必须评估"决策深度"。如果操作有但决策无(如巡查=点一下领奖)，机制需要重设计。
