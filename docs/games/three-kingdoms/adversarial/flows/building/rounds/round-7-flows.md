## 一、进化机制重设计 [P0] — R7任务#1

> **来源**: PRD-R031 + CH-F101
> **核心问题**: 进化重置Lv1导致产出暴跌98%，玩家不会使用进化系统
> **方案**: 等级保留至Lv15(非Lv1)，产出仅下降~44%(非98%)

### 1.1 BLD-F22-02(修订) 进化执行子流程

**R6设计(问题)**: `building.level = 1` — 重置为Lv1，产出暴跌98%

**R7修订**: `building.level = 15` — 保留至Lv15，产出仅下降~44%

**修订后的流程步骤**:
1. [玩家操作] 在进化面板点击"进化"按钮
2. [二次确认] 弹出确认面板:
   - "确定将[建筑名]进化至{N+1}星?"
   - "进化后等级将保留至Lv15 (当前Lv{X})"
   - "预计产出变化: {当前产出} → {Lv15产出} (约-{降幅}%)"
   - "永久获得: [加成列表]"
   - "确认" / "再想想"
3. [材料扣除] 扣除进化所需材料
4. **[等级保留(修订)]** 建筑等级保留至Lv15:
   - `building.level = 15` (R6是`= 1`)
   - 建筑产出降低但幅度可控
   - **关键变化**: 玩家保留了大部分升级成果
5. [星级提升] 建筑星级+1，永久加成立即生效
6. [进化动画] 播放进化动画(3秒)
7. [UI反馈] Toast "进化成功! [建筑名]已进化至{N}星，等级保留至Lv15"
8. [解锁新上限] 进化后建筑等级上限提升(不变):
   - 1星: 上限Lv30
   - 2星: 上限Lv35
   - 3星: 上限Lv40

**数据变更**:
| 数据 | R6 | R7 | 说明 |
|------|----|----|------|
| building.level(进化后) | 1 | 15 | 等级保留至Lv15 |

### 1.2 进化产出变化对比(R6 vs R7)

以农田为例(Lv25满级，基础产出45.2粮草/秒):

| 进化阶段 | R6(重置Lv1) | R7(保留Lv15) | 改善幅度 |
|---------|:----------:|:----------:|:-------:|
| 进化后基础产出 | 0.8(98%下降) | 25.2(44%下降) | 损失缩小54% |
| 1星进化后产出 | 0.88(0.8×1.10) | 27.7(25.2×1.10) | +30倍 |
| 2星进化后产出 | 1.0(0.8×1.25) | 31.5(25.2×1.25) | +31倍 |
| 3星进化后产出 | 1.2(0.8×1.50) | 37.8(25.2×1.50) | +31倍 |

**玩家体验变化**:
- R6: 进化后产出从45.2暴跌到0.88(下降98%)→ 玩家: "我花了3周的成果归零了!" → 不进化
- R7: 进化后产出从45.2降到27.7(下降39%)→ 玩家: "降了一些但还能接受，而且有永久加成" → 愿意进化

### 1.3 进化ROI计算(等级保留至Lv15)

**1星进化ROI**:

```
进化前: Lv25, 产出45.2粮草/秒
进化后: Lv15(1星), 产出25.2×1.10 = 27.7粮草/秒

恢复路径: Lv15 → Lv25
  - 升级费用合计: 347,000粮草(见Lv25-40费用表)
  - 升级时间合计: 40.0小时
  - 恢复期间平均产出: ~37.5粮草/秒(从27.7线性恢复到49.7)
  - 恢复期间产出损失: (45.2 - 37.5) × 40 × 3600 = 1,109,000粮草

总成本: 347,000(升级费) + 50,000(进化费) + 1,109,000(产出损失) = 1,506,000粮草

恢复后产出: 45.2 × 1.10 = 49.72粮草/秒
每日额外产出: 4.52 × 86,400 = 390,528粮草

回本时间: 1,506,000 / 390,528 = 3.9天
```

**R6对比**: Lv1重置的回本时间约10天。R7缩短至3.9天，改善61%。

**2星进化ROI**:

```
进化前: Lv30(1星), 产出约53.5粮草/秒
进化后: Lv15(2星), 产出25.2×1.25 = 31.5粮草/秒

恢复路径: Lv15 → Lv30
  - 升级费用合计: 747,000粮草
  - 升级时间合计: 85.5小时(~3.6天)
  - 恢复期间平均产出: ~42.5粮草/秒
  - 恢复期间产出损失: (53.5 - 42.5) × 85.5 × 3600 = 3,385,000粮草

总成本: 747,000 + 150,000 + 3,385,000 = 4,282,000粮草

恢复后产出: Lv30基础 × 1.25 ≈ 66.9粮草/秒
每日额外产出(对比1星Lv30): ~1,175,000粮草

回本时间: 4,282,000 / 1,175,000 = 3.6天
```

**3星进化ROI**:

```
进化前: Lv35(2星), 产出约73.5粮草/秒
进化后: Lv15(3星), 产出25.2×1.50 = 37.8粮草/秒

恢复路径: Lv15 → Lv40
  - 升级费用合计: 2,572,000粮草
  - 升级时间合计: 249.5小时(~10.4天)
  - 恢复期间平均产出: ~55.8粮草/秒
  - 恢复期间产出损失: (73.5 - 55.8) × 249.5 × 3600 = 15,900,000粮草

总成本: 2,572,000 + 500,000 + 15,900,000 = 18,972,000粮草

恢复后产出: Lv40基础 × 1.50 ≈ 120粮草/秒
每日额外产出(对比2星Lv35): ~4,061,000粮草

回本时间: 18,972,000 / 4,061,000 = 4.7天
```

**进化ROI汇总**:

| 星级 | R6回本时间 | R7回本时间 | 改善 | 玩家感受 |
|:----:|:--------:|:--------:|:----:|---------|
| 1星 | ~10天 | ~3.9天 | -61% | "3天回本，值得进化" |
| 2星 | ~15天(推测) | ~3.6天 | -76% | "回本更快，因为基础高了" |
| 3星 | ~20天(推测) | ~4.7天 | -77% | "虽然投入大但回报也大" |

### 1.5 星级加成重新设计 [P2] — R7任务#9

> **来源**: CH-F102
> **R6问题**: "替代"设计让2星实际净增益仅15%(非25%)，性价比递减
> **R7方案**: 改为"叠加"——每星+10%，累计递增

**R6设计(问题)**:
- 1星: +10%(净增10%)
- 2星: +25%替代1星(净增15%，消耗×3)
- 3星: +50%替代2星(净增25%，消耗×3.3)

**R7设计(修订)**:
- 1星: +10%(净增10%)
- 2星: +10%，累计+20%(净增10%，消耗×3)
- 3星: +10%，累计+30%(净增10%，消耗×3.3)

**为什么选择叠加而非替代**:
1. **认知简单**: "每颗星+10%"比"2星替代1星，净增15%"更直观
2. **无隐藏损失**: 玩家不会觉得"1星的10%被吃了"
3. **总加成可控**: 3星+30%(叠加)vs +50%(替代)，叠加方案总加成更低，对数值平衡更安全

**注意**: 叠加方案下3星总加成为+30%(非+50%)，需配合Lv40等级上限和更高基础产出来保证后期玩家的追求感。

**进化加成数值表(修订)**:

## 二、事件系统修正 [P1] — R7任务#2, #8

### 2.1 BLD-F20-01(修订) 事件气泡渐进式紧迫感 [P1] — R7任务#2

> **来源**: PRD-R030 + CH-F095
> **R6问题**: 非强制气泡=玩家必然忽略，事件系统等于不存在
> **R7方案**: 渐进式紧迫感——视觉权重随时间递增，引导玩家点击

**R6设计(问题)**:
- 金色气泡，轻微上下浮动
- 气泡持续存在直到玩家点击查看或当日结束
- 无紧迫感，玩家可以永远忽略

**R7设计(修订) — 渐进式紧迫感**:

| 时间段 | 气泡状态 | 视觉效果 | 心理效果 |
|:------:|---------|---------|---------|
| 0~1小时 | 金色气泡 | 轻微浮动，"!"图标 | "有新事件，不急" |
| 1~2小时 | **红色气泡** | 中等浮动，红色脉冲，气泡放大至120% | "这个事件等了蛮久了" |
| 2~3小时 | **红色大气泡** | 强烈浮动，红色脉冲加速，气泡放大至150%，**显示奖励预览文字** | "奖励快没了，赶紧点" |
| 3小时+ | **红色大气泡+奖励倒计时** | 最强效果，气泡上方显示"仅剩X:XX" | "再不点就没了!" |

**修订后的流程步骤**:
1. [触发条件-TE23] 每日首次登录时检查(不变)
2. [建筑选择] 从Lv5+且无冷却的建筑中随机选择(不变)
3. [事件选择] 从该建筑类型的事件池中随机选择(不变)
4. **[UI展示-渐进式气泡(修订)]** 在目标建筑卡片上方显示:
   - **第1阶段(0~1h)**: 金色气泡，轻微浮动，"!"图标
   - **第2阶段(1~2h)**: 红色气泡，中等浮动，红色脉冲，气泡放大120%
   - **第3阶段(2~3h)**: 红色大气泡，强烈浮动，放大150%，气泡内显示奖励预览(如"+2000粮草")
   - **第4阶段(3h+)**: 最强视觉效果，气泡上方显示"仅剩X:XX:XX"倒计时
5. [非强制性(修订)] 气泡仍然不强制，但视觉权重足够强:
   - 不弹出全屏遮罩(保持)
   - 不打断玩家当前操作(保持)
   - **但红色大气泡+奖励预览的视觉权重足以引起玩家注意**
6. [超时处理] 若玩家当日未点击查看:
   - 次日0点气泡消失
   - 不触发冷却(未查看=未参与)
   - 不计入"今日已触发"

**气泡点击率预估(R7 vs R6)**:
- R6: 30%~50%(静态金色气泡)
- R7: 70%~85%(渐进式紧迫感+奖励预览)
- **提升原因**: (1)红色比金色更醒目；(2)气泡变大吸引注意力；(3)奖励预览给玩家"点击理由"；(4)倒计时制造紧迫感

### 2.2 BLD-F20-02(修订) 事件选择预期收益 [P1] — R7任务#8

> **来源**: CH-F097
> **R6问题**: "即时vs持续"选择需要玩家做数学计算，且持续效果是否算在线时间不明确
> **R7方案**: 持续效果改为"离线也生效"+显示预期收益

**R6设计(问题)**:
- 选择A: 立即获得2000粮草
- 选择B: 粮草产出+10%持续4小时
- 玩家不知道4小时是"游戏时间"还是"在线时间"

**R7设计(修订)**:

**修订1: 持续效果改为"离线也生效"**
- 持续N小时的效果，在玩家离线期间也继续计算收益
- 消除"在线时长影响选择正确性"的问题
- 两个选择的期望收益对所有玩家一致

**修订2: 显示预期收益**
- 选择面板上显示两个选项的预期收益:
  ```
  选择A: 立即获得 2,000 粮草
         预期收益: 2,000 粮草

  选择B: 粮草产出+10% 持续4小时(离线也生效)
         预期收益: ~2,500 粮草 (基于您当前产出)
  ```
- 预期收益计算: `持续收益 = 当前产出速率 × 加成比例 × 持续秒数`

**修订3: 两个选项期望收益接近**
- 通过数值调优，让两个选项的期望收益差距<20%
- 选择变成"风格偏好"(即时满足vs延迟回报)而非"数学题"
- 示例调整:
  - 选择A: 立即获得2,000粮草
  - 选择B: 粮草产出+8%持续4小时(预期收益~2,100粮草)
  - 差距仅5%，玩家可凭偏好选择

**修订后的事件选择面板**:
```

│       丰收之年              │
│  今年风调雨顺，农田大丰收    │

│    选择A         │  选择B    │
│  [立即收获]      │ [持续丰收] │
│                  │          │
│  立即获得        │  产出+8%  │
│  2,000粮草       │  持续4小时 │
│                  │ (离线生效) │
│  预期收益:       │ 预期收益:  │
│  2,000粮草       │ ~2,100粮草│
│                  │          │
│  [选择此选项]    │ [选择此选项]│
```

### 4.2 217倍差距评估与修正 [P1] — R7任务#6

> **来源**: CH-F105
> **R6问题**: 9个乘法因子让新手vs老手差距达217倍
> **R7方案**: 进化加成改为加法层+重新计算差距

**R6产出公式(问题)**:
```
最终产出 = 基础产出
         × 主城加成    (1 + castle.level × 2%)
         × 科技加成    (1 + techBonus%)
         × 武将加成    (1 + heroBonus%)
         × 医馆加成    (1 + clinic.level × 2%)
         × (协同+特化)  (1 + synergy% + spec%)    上限80%
         × 治疗Buff    (1.1 if active)
         × 进化加成    (1 + prestigeBonus%)       ← 独立乘法层
         × 焦点加成    (1 + focusBonus%)           ← 独立乘法层
```

**R7产出公式(修订)**:
```
最终产出 = 基础产出
         × 主城加成        (1 + castle.level × 2%)
         × 科技加成        (1 + techBonus%)
         × 武将加成        (1 + heroBonus%)
         × 医馆加成        (1 + clinic.level × 2%)
         × (协同+特化)     (1 + synergy% + spec%)    上限80%
         × 治疗Buff        (1.1 if active)
         × (1 + 进化加成% + 焦点加成%)               ← 合并为加法层
```

**关键变化**:
1. **进化加成从乘法层改为加法层**: 与焦点加成合并为同一加法层
2. **公式简化**: 9个乘法因子→8个乘法因子+1个加法层
3. **差距压缩**: 进化和焦点不再独立放大差距

**差距重新计算**:

**R6计算(217倍)**:
```
老手: 80 × 1.60 × 1.15 × 1.05 × 1.40 × 1.55 × 1.10 × 1.50 × 1.05 = 716.8
新手: 3.0 × 1.10 = 3.3
差距: 716.8 / 3.3 = 217倍
```

**R7计算(进化改加法层)**:
```
老手: 80 × 1.60 × 1.15 × 1.05 × 1.40 × 1.55 × 1.10 × (1 + 30% + 5%)
    = 80 × 1.60 × 1.15 × 1.05 × 1.40 × 1.55 × 1.10 × 1.35
    = 80 × 5.44
    = 435.5

新手(Lv5): 3.0 × 1.10 = 3.3
差距: 435.5 / 3.3 = 132倍
```

**差距压缩效果**:
| 方案 | 老手产出 | 新手产出 | 差距 |
|------|:-------:|:-------:|:----:|
| R6(进化乘法) | 716.8 | 3.3 | 217倍 |
| R7(进化加法) | 435.5 | 3.3 | 132倍 |
| 改善幅度 | -39% | — | -39% |

**但132倍仍然很高。进一步分析**:

差距的主要驱动因素不是进化/焦点，而是**等级差异**:
- Lv40基础产出80 vs Lv5基础产出3.0 = 26.7倍(仅等级差异)
- 加上所有加成后，即使没有进化/焦点，差距仍有:
  - 80 × 1.60 × 1.15 × 1.05 × 1.40 × 1.55 × 1.10 / (3.0 × 1.10) = 80 × 4.03 / 3.3 = 97.7倍

**结论**: 217倍差距的核心原因是**等级跨度太大**(Lv5→Lv40)和**加成层数太多**，而非进化系统单独造成。

**R7进一步建议(跨系统协同)**:
1. **Lv5→Lv10的基础产出增幅降低**: 从3.0→5.0(降低新手阶段的产出梯度)
2. **高等级加成递减**: 当总加成超过+100%时，超出部分效果减半
3. **这些修改涉及资源/科技/武将等系统，建议在跨系统统一数值时处理**

**当前R7可做的修正**: 进化加成改为加法层(已执行)，差距从217→132倍。剩余差距需要跨系统协同修正(延后至数值统一阶段)。

4. [标记确认] 弹出确认面板:
   - "将[建筑名]设为焦点建筑?"
   - "焦点建筑获得 +15% 产出加成"
   - "切换焦点后需等待24小时才能再次切换"
5. [系统执行] 标记建筑为焦点:
   - `buildings[type].focus = true`
   - `focusBonus = 15%` (R6是5%)
   - `focusSwitchCooldown = now + 24h`
6. [UI显示] 焦点建筑特殊标识:
   - 建筑卡片左上角显示金色星标(更大更醒目)
   - 产出详情中显示"焦点加成: +15%"
   - 焦点切换冷却倒计时显示在焦点按钮旁

**焦点与其他系统的联动**:
- 焦点建筑可作为每日挑战的目标("升级焦点建筑1次")
- 焦点建筑的巡查奖励+50%
- 焦点建筑升级速度+10%

### 6.2 BLD-F21-03(修订) 巡查机制增强 [P2] — R7任务#12

> **来源**: CH-F100
> **R6问题**: "每4小时点一下→领奖→关闭"，决策深度为零
> **R7方案**: 增加"发现问题"环节+与建筑状态关联

**R6设计(问题)**:
- 点击建筑详情→自动获得巡查奖励→关闭
- 无决策，无巡查感
- 奖励随机，玩家无法预期

**R7设计(修订) — "发现问题"环节**:

**修订后的巡查流程**:
1. [触发条件-TE27] 玩家点击建筑卡片进入详情时检查:
   - 条件: `building.lastInspectionTime + 4小时 <= now`
2. **[发现问题(新增)]** 系统随机生成"建筑问题":
   - **杂草丛生**(农田): "农田出现杂草，需要清理"
   - **货架积灰**(市集): "市集货架积灰，需要擦拭"
   - **训练场破损**(兵营): "训练场地面破损，需要修补"
   - **炉火不旺**(铁匠铺): "铁匠铺炉火不旺，需要添柴"
   - **藏书发霉**(书院): "书院藏书发霉，需要晾晒"
   - **药材过期**(医馆): "部分药材即将过期，需要处理"
   - **城墙裂缝**(城墙): "城墙出现裂缝，需要加固"
3. **[玩家选择(新增)]** 3种处理方式:
   - **手动处理**: 立即获得巡查奖励(标准奖励)
   - **花费资源**: 消耗少量资源，获得1.5倍巡查奖励
   - **忽略问题**: 不获得奖励，问题累积(连续3次忽略则建筑产出-5%持续1小时)
4. [UI展示] 建筑详情面板顶部显示问题横幅:
   - 问题图标 + 问题描述
   - 三个处理方式按钮
   - 问题持续24小时后自动消失(不处理=忽略)
5. [巡查奖励(修订)] 奖励与建筑状态关联:
   - 建筑产出即将溢出(资源池>80%上限): "巡查发现产出即将溢出! 立即收取可获得额外20%"
   - 廻筑刚完成升级: "巡查发现新设施! 获得升级贺礼"
   - 建筑处于焦点状态: "焦点建筑巡查奖励+50%"

**巡查奖励数值(修订)**:

| 处理方式 | 奖励倍率 | 条件 |
|:-------:|:-------:|------|
| 手动处理 | 1.0x | 免费，立即获得 |
| 花费资源 | 1.5x | 消耗该建筑1分钟产出的资源 |
| 忽略问题 | 0x | 无奖励，累积可能导致产出惩罚 |

**设计合理性自检**:
- **玩家会主动使用吗?** 是——有问题需要解决(而非单纯领奖)，且忽略有惩罚
- **净收益正吗?** 是——手动处理免费获得奖励，花费资源的1.5倍回报也划算
- **不使用后果?** 有——连续忽略3次导致产出-5%，鼓励玩家定期巡查

## 七、设计合理性自检

> **来源**: R6-Arbiter方法论改进建议
> **要求**: 每个修改后的机制必须通过3项检查

### 7.1 自检清单

| 机制 | 检查1: 会主动使用? | 检查2: 净收益正? | 检查3: 不使用后果? | 结果 |
|------|:--:|:--:|:--:|:--:|
| 进化(等级保留至Lv15) | 是——产出仅降44%，3~4天回本 | 是——ROI 3.9天(1星) | 有——错过永久加成和新等级上限 | **通过** |
| 事件气泡(渐进式紧迫感) | 是——红色+变大+奖励预览，点击率>70% | 是——两个选择都是正收益 | 有——错过每日事件奖励 | **通过** |
| 挑战(3选1+奖励>成本) | 是——奖励>成本，玩家有选择权 | 是——所有挑战净收益为正 | 轻微——错过每日元宝(8~10) | **通过** |
| 事件选择(离线生效+预期收益) | 是——显示预期收益，降低决策门槛 | 是——两个选项收益接近(差距<20%) | 无——选择哪个都是收益 | **通过(轻微)** |
| 损失报告(动态化) | 是——内容随时间变化，保持新鲜感 | 无成本——纯信息展示 | 轻微——错过升级引导信息 | **通过** |
| 星级(叠加+10%/次) | 是——每颗星都有明确的+10%收益 | 是——1星ROI与Lv15保留联动 | 有——错过永久加成 | **通过** |
| 焦点(1座+15%+冷却) | 是——+15%超过感知阈值 | 是——+15%产出，免费 | 有——错过15%产出加成 | **通过** |
| 巡查(发现问题+选择) | 是——有问题需要解决，忽略有惩罚 | 是——免费处理可获得奖励 | 有——累积忽略导致产出-5% | **通过** |
| 未建造提示(收窄条件) | 被动——系统自动显示 | 无成本——纯信息 | 轻微——不建医馆有损失但很小 | **通过(轻微)** |
| 损失报告(精简数字) | 被动——系统自动显示 | 无成本——纯信息 | 轻微——信息更清晰 | **通过** |

### 7.2 自检发现的薄弱环节

**1. 事件选择"不使用后果"偏弱**:
- 两个选项都是收益，不选择(超时)也无惩罚
- R7保留了"超时无惩罚"设计(事件气泡的渐进式紧迫感已足够引导点击)
- 如果R7气泡改进后点击率仍<60%，可考虑增加超时惩罚(延后评估)

**2. 未建造提示"主动使用"偏弱**:
- 被动信息展示，玩家不需要主动操作
- 这是可接受的——该机制的目的是"信息传递"而非"主动操作"

### 8.4 产出公式最终版本(R7修订)

```
最终产出 = 基础产出(Lv表)
         × 主城加成(1 + castle.level × 2%)
         × 科技加成(1 + techBonus%)
         × 武将加成(1 + heroBonus%)
         × 医馆加成(1 + clinic.level × 2%)
         × (协同加成 + 特化加成)(1 + synergy% + spec%)   上限80%
         × 治疗Buff(1.1 if active, else 1.0)
         × (1 + 进化加成% + 焦点加成%)                   ← R7: 合并为加法层
```

**因子变化**: 9个乘法因子 → 8个乘法因子 + 1个加法层(进化+焦点)
