## 一、流程完备性质疑

### CH-F001 | 一键收取流程完全缺失

- **严重度**: 🔴 P0
- **质疑**: PRD BLD-5-3 明确要求"右下角悬浮按钮，一键收取所有建筑累积产出"，但 R1-Builder 枚举的 9 个主流程中完全没有一键收取流程。这是 PRD 覆盖率 76.9% 的直接原因之一（13 个需求中 3 个未覆盖）。
- **玩家视角**: "我看到右下角有个收取按钮，点击后会发生什么？资源从哪里来？动画怎么播放？"
- **缺失分析**:
  - 一键收取的触发入口（右下角悬浮按钮）
  - 收取动画（资源数字从各建筑飞向顶部资源栏）
  - 收取逻辑（实时累加模型下，收取仅为视觉效果？还是累积制下有实际意义？）
  - VIP3+ 自动收取的解锁和触发逻辑
  - 与 CH-F005（产出模型矛盾）直接关联：不明确产出模型就无法设计收取流程
- **改进建议**: 补充完整的一键收取流程，明确产出模型（实时累加 vs 累积制），设计收取动画和 VIP 自动收取机制。

---

### CH-F002 | 加速系统流程完全缺失

- **严重度**: 🔴 P0
- **质疑**: PRD BLD-4-3 提到"铜钱加速 / 天命加速 / 元宝秒完成"三种加速选项，但 R1-Builder 完全没有枚举加速流程。这是付费转化的核心入口，缺失严重。结合 CH-F017（Lv29→30 升级时间 2.64 年），没有加速机制意味着后期升级完全不可行。
- **玩家视角**: "升级要等 2 小时，我想花点钱加速，但找不到加速按钮。"
- **缺失分析**:
  - 铜钱加速的触发、消耗计算、时间减少逻辑
  - 天命加速的道具消耗和效果
  - 元宝秒完成的定价公式和立即完成逻辑
  - 加速后的 UI 反馈（进度条跳动、倒计时更新）
  - 加速限制（每次升级最多 1 次铜钱加速？）
  - 加速与取消升级的交互（加速后取消，返还多少资源？）
- **改进建议**: 补充完整的加速系统流程，设计合理的定价公式，确保付费转化路径畅通。

### CH-F003 | 自动升级流程完全缺失

- **严重度**: 🟡 P1
- **质疑**: PRD BLD-4-3 提到"可开启开关，系统自动将可升级建筑加入队列"，但 R1-Builder 没有枚举自动升级流程。源码 `BuildingRecommender.ts` L42-L85 实现了 `recommendUpgradePath()` 函数，但没有自动升级的调用逻辑。
- **玩家视角**: "我开启了自动升级，但不知道系统会按什么顺序升级，会不会浪费资源？"
- **缺失分析**:
  - 自动升级的开启/关闭开关
  - 自动升级的触发时机（资源足够时？队列空闲时？）
  - 自动升级的优先级算法（主城优先？均衡升级？使用现有 recommendUpgradePath？）
  - 自动升级的资源保护（避免把资源全部用完）
  - 自动升级的 UI 提示（正在自动升级哪个建筑）
  - 自动升级与手动操作的冲突处理（手动优先？）
- **改进建议**: 补充自动升级流程，设计智能的优先级算法，增加资源保护机制。

### CH-F004 | 离线效率系数未实现

- **严重度**: 🟡 P1
- **质疑**: PRD BLD-4-4 明确规定"离线效率 = 资源效率 × 1.2（建造额外 20% 效率）"，但源码 `BuildingSystem.ts` L389-L445 的 `deserialize()` 方法中完全没有实现离线效率系数。离线期间的升级进度按 100% 计算，没有额外 20% 效率。R1-Builder 在 BLD-F07-01 中也标记了这个断裂点（GAP-006）。
- **玩家视角**: "PRD 说离线有 20% 额外效率，但我离线 8 小时和在线 8 小时的升级进度完全一样。"
- **缺失分析**:
  - 离线期间升级进度的效率系数计算
  - 离线期间资源产出的效率系数计算
  - 离线效率系数的配置化（方便调整）
  - 离线效率的 UI 展示（离线收益面板）
- **改进建议**: 在 `deserialize()` 中实现离线效率系数，或者明确 PRD 中的"离线效率"定义。

### CH-F005 | 资源上限机制缺失 — 产出模型矛盾

- **严重度**: 🟡 P1
- **质疑**: R1-Builder 在 BLD-F01-02 中提到"当前引擎未实现资源上限机制"，这意味着资源可以无限累加。更深层的问题是 PRD 与引擎的产出模型矛盾：
  - **PRD 描述**: "累积产出"需要"收取"，暗示累积制
  - **引擎实现**: `calculateTotalProduction()` 直接累加到资源系统，是实时累加制
  - **矛盾点**: 如果是实时累加制，一键收取没有实际意义；如果是累积制，需要资源上限和收取逻辑
- **玩家视角**: "我离线了 3 天，登录后资源直接够升 10 级，这正常吗？"
- **缺失分析**:
  - 产出模型的明确定义（实时累加 vs 累积制）
  - 资源上限的计算公式
  - 资源达到上限后的处理（停止产出？溢出丢弃？）
  - 资源上限与一键收取的关联
- **改进建议**: 明确产出模型，引入资源上限机制（如果采用累积制）。

### CH-F006 | 建筑主动技能流程缺失

- **严重度**: 🟡 P1
- **质疑**: R1-Builder 枚举的所有流程中，建筑只有"升级"一个主动操作。8 座建筑没有任何主动技能，玩家与建筑的交互极度单一。源码 `BuildingSystem.ts` 只有 `startUpgrade()`、`cancelUpgrade()`、`batchUpgrade()` 三个玩家操作入口，没有任何主动技能相关的方法。
- **玩家视角**: "建筑就是放着等产出，点一下升级，然后就没别的事了。太无聊了。"
- **缺失分析**:
  - 建筑主动技能的设计（如农田"丰收"、兵营"征兵令"、医馆"治疗"）
  - 主动技能的触发、冷却、消耗逻辑
  - 主动技能的 UI 入口和反馈
  - 主动技能与建筑等级的关联
- **改进建议**: 为每座建筑设计 1-2 个主动技能，增加玩家与建筑的交互频率。

### CH-F007 | 建筑事件系统缺失

- **严重度**: 🟢 P2
- **质疑**: 8 座建筑完全没有随机事件，建筑是"静态的数值容器"而非"活的游戏实体"。其他 SLG 游戏通常有丰富的建筑事件（丰收、瘟疫、暴乱等）。
- **玩家视角**: "建筑永远在安静地产出，没有任何惊喜或意外。"
- **缺失分析**:
  - 正面事件（丰收：产出翻倍 1 小时；工匠来访：升级费用减半）
  - 负面事件（瘟疫：产出减半 30 分钟；暴乱：需要安抚）
  - 事件的触发概率和条件
  - 事件的 UI 展示和处理流程
- **改进建议**: 引入建筑事件系统，增加游戏的随机性和趣味性。

### CH-F009 | 科技加成叠加规则未明确

- **严重度**: 🟡 P1

  - 不同类加成的叠加方式
  - 加成链的 UI 展示（建筑详情面板中的加成明细）

### CH-F010 | 批量升级的队列槽位溢出问题

- **严重度**: 🟢 P2
- **质疑**: R1-Builder 在 BLD-F04 中枚举了批量升级流程，源码 `BuildingBatchOps.ts` L57-L101 实现了批量升级逻辑。但批量升级没有考虑队列槽位限制：如果玩家选择 5 个建筑批量升级，但队列只有 2 个槽位，`checkUpgrade()` 会返回"队列已满"，后续建筑全部失败。
- **玩家视角**: "我选了 5 个建筑批量升级，但只有 2 个成功了，其他 3 个失败了，我不知道为什么。"

  - 批量升级的 UI 反馈（哪些成功、哪些失败、失败原因）
  - 批量升级与单个升级的队列竞争
- **改进建议**: 在批量升级前检查队列槽位，队列满时跳过后续建筑并提示玩家。

## 二、PRD实体角色质疑

### CH-F011 | 医馆存在感为零 — "僵尸实体"

- **严重度**: 🔴 P0
- **质疑**: 医馆是 8 座建筑中最没有存在感的。R1-Builder 的实体评估中，医馆在存在感、不可替代性、交互性、差异化、成长感 5 个维度全部只有 1-2 星。源码 `building-config.ts` L305-L331 确认医馆的 `production` 字段为恢复速率百分比（3%~60%），没有 `resourceType`，意味着不产出任何资源。
- **玩家视角**: "医馆升到 Lv10 有什么用？我感受不到任何区别。为什么要花资源升级它？"
- **深度分析**:
  - **无感知**: +3%~+60% 恢复速率是被动百分比加成，玩家无法直接感知效果
  - **无战斗依赖**: 当前无战斗系统（CBT），恢复速率完全无意义
  - **无资源产出**: 医馆是唯一不产出任何资源的建筑（城墙至少有城防值 specialValue）
  - **无主动交互**: 只有升级操作，源码中 `getClinicRecoveryRate()` 只是读取 production 值
  - **数值佐证**: 医馆 Lv1 +3%、Lv10 +30%、Lv20 +60%，但无使用场景
  - **升级费用过低**: Lv1→2 仅需 100 粮草 + 150 铜钱，但无升级动力
  - **存在感测试**: 如果移除医馆，当前游戏没有任何变化
- **改进建议**: 重新设计医馆角色：
  - **方案 A**: 改为产出"伤兵"资源（每秒恢复 N 个伤兵→可用兵力）
  - **方案 B**: 增加主动技能"治疗"（消耗粮草立即恢复兵力，CD 30 分钟）
  - **方案 C**: 增加 buff 系统（主动技能提供攻击/防御加成）

### CH-F012 | 农田/市集/兵营三兄弟同质化 — "换皮实体"

- **严重度**: 🟡 P1
- **质疑**: 农田、市集、兵营三座建筑的交互模式完全相同：升级 → 产出资源 → 等待 → 再升级。源码 `building-config.ts` 确认三者的 `levelTable` 结构完全相同（production + upgradeCost），唯一的区别是 `resourceType` 不同（grain/gold/troops）。
- **玩家视角**: "农田、市集、兵营就是换了个名字和图标，操作完全一样。"
- **深度分析**:
  - **交互模式相同**: 都是"升级→产出→等待"
  - **策略无差异**: 升级哪个都一样，没有"选择 A 就不能选择 B"的决策张力
  - **成长感相同**: 都是线性产出增长，后期提升感知弱
  - **费用曲线相同**: 都是"每级 ×1.8→×1.6→×1.5→×1.4"
  - **行业对比**: 优秀 SLG 游戏中，不同资源建筑有不同的交互机制（如农田有季节加成、市集有贸易路线）
- **改进建议**: 为每座产出建筑设计独特的交互机制：
  - 农田: 季节系统（春夏秋冬产出不同）
  - 市集: 贸易路线（选择贸易伙伴影响产出类型）
  - 兵营: 兵种选择（步兵/骑兵/弓兵，不同兵种消耗不同资源）

### CH-F013 | 铁匠铺材料用途不明确 — "悬空实体"

- **严重度**: 🟡 P1
- **质疑**: 铁匠铺产出"材料/小时"，但材料的用途（装备强化）在当前系统中不明确。源码 `building-config.ts` L235-L262 确认铁匠铺的 `levelTable` 只有 `production`（材料/小时），没有 `resourceType`。这意味着铁匠铺的产出不会进入 `calculateTotalProduction()` 的累加逻辑（因为没有 `production.resourceType`）。
- **玩家视角**: "铁匠铺产出材料，但我不知道材料能干什么。为什么要升级铁匠铺？"
- **深度分析**:
  - 材料的消费场景（装备强化？装备打造？）在当前系统中不明确
  - 材料与其他资源的兑换关系不存在
  - 材料产出单位的不一致性（材料/小时 vs 资源/秒）
  - 铁匠铺升级的动力不足
  - 铁匠铺的产出不参与 `calculateTotalProduction()` 累加
- **改进建议**: 明确材料的用途，或者将铁匠铺改为直接产出装备/强化石。

### CH-F014 | 城墙依赖未实现的攻城系统 — "空壳实体"

- **严重度**: 🟡 P1
- **质疑**: 城墙产出"城防值"（specialValue）和"防御加成%"（production），但攻城系统（CPN）未实现。源码 `BuildingSystem.ts` L369-L377 确认 `getWallDefense()` 和 `getWallDefenseBonus()` 只是读取配置值，没有任何消费逻辑。
- **玩家视角**: "城墙升到 Lv10 有什么用？城防值在哪里用？防御加成加在哪里？"
- **深度分析**:
  - 城防值的消费场景（攻城防御计算）不存在
  - 防御加成的应用场景（被攻击时的伤害减免？）不存在
  - 城墙在无攻城内容时的价值为零
  - 升级城墙的机会成本（资源用于城墙 vs 用于产出建筑）
  - 城墙是唯一需要三种资源（粮草+铜钱+兵力）升级的建筑，增加资源压力
- **改进建议**: 在攻城系统实现前，为城墙设计临时用途（如增加资源上限、提供全局防御 buff）。

### CH-F015 | 书院研究速度加成感知度低

- **严重度**: 🟢 P2
- **质疑**: 书院是唯一有双产出的建筑（科技点/秒 + 研究速度加成），源码 `building-config.ts` L270-L297 确认书院的 `levelTable` 有 `production`（科技点/秒），PRD 提到研究速度加成（+2%~+38%），但研究速度加成的感知度很低。
- **玩家视角**: "书院升到 Lv10 研究速度 +18%，但我感觉不到科技研究变快了。"
- **深度分析**:
  - 研究速度加成的 UI 展示缺失（科技面板中应显示当前研究速度）
  - 研究速度加成的感知阈值（+18% 是否足够明显？）
  - 研究速度加成与科技点产出的关系不明确
  - 书院升级的 ROI（科技点产出 + 研究速度，哪个更重要？）
- **改进建议**: 在科技面板中明确展示研究速度加成，或者将研究速度加成改为更直观的效果（如减少科技研究时间）。

## 三、PRD数值平衡质疑

### CH-F016 | Lv10 后 ROI 急剧恶化 — "升级陷阱"

- **严重度**: 🔴 P0
- **质疑**: R1-Builder 的 ROI 分析（round-1-flows.md 第 619-631 行）显示：
  - Lv5→6 农田: 回本 104 分钟
  - Lv10→11 农田: 回本 5.25 小时
  - Lv15→16 农田: 回本 2.16 天
  - Lv20→21 农田: 回本 11.1 天
  - Lv24→25 农田: 回本 25.4 天

  这意味着 Lv10 后升级完全不划算，玩家会陷入"升级陷阱"——花大量资源升级，但产出提升微乎其微。
- **玩家视角**: "我花了一天的资源升级农田，结果每秒只多了 1.6 粮草。这值得吗？"
- **深度分析**:
  - **产出线性增长**: 农田每级产出增加约 1.6~3.7，增长缓慢
  - **费用指数增长**: 升级费用每级乘以 1.4~1.8，增长极快
  - **剪刀差**: 产出增长 vs 费用增长的剪刀差越来越大
  - **回本时间指数增长**: 从分钟级 → 小时级 → 天级
  - **感知阈值**: Lv15 后每次升级的感觉是"升了等于没升"（相对增量从 36% 降至 9%）
- **改进建议**:
  - 方案 1: 提高后期产出增长率（Lv20+ 每级 +4~6 而非 +2~3）
  - 方案 2: 降低后期升级费用倍率（×1.4 → ×1.2）
  - 方案 3: 引入加速机制（铜钱/元宝加速）

### CH-F017 | Lv29→30 升级时间 2.64 年 — "荒谬数值"

- **严重度**: 🔴 P0
- **质疑**: R1-Builder 的数值追溯（round-1-flows.md 第 606 行）显示，主城 Lv29→30 升级时间 83,216,586 秒 = 2.64 年。源码 `building-config.ts` L97 确认 `timeSeconds: 83216586`。这是一个完全不可接受的数值。即使引入加速机制，2.64 年的升级时间也意味着游戏的生命周期被设定为至少 3 年。

- **深度分析**:

  - **玩家留存**: Lv10 后等待时间过长，玩家必然流失

- **改进建议**:
  - 方案 1: 降低后期升级时间倍率（×1.3 → ×1.15）
  - 方案 2: 引入加速机制（铜钱/天命/元宝）
  - 方案 3: 重新设计等级上限（主城 30 级 → 20 级）

### CH-F018 | 产出-费用剪刀差 — "数学陷阱"

- **玩家视角**: "我算了算，升级农田 Lv24→25 需要 25 天才能回本。这还不如不升级。"
- **深度分析**:
  - **产出曲线**: 线性增长（每级 +1~4）
  - **费用曲线**: 指数增长（每级 ×1.4~1.8）
  - **时间曲线**: 指数增长（每级 ×1.3~2.0）
  - **ROI 曲线**: 指数恶化
  - **核心矛盾**: 产出线性增长 vs 费用指数增长 = 剪刀差
- **改进建议**: 重新设计产出-费用曲线：
  - 产出采用超线性增长（如每级增量递增）
  - 费用采用亚指数增长（如对数增长）
  - 时间采用亚指数增长

## 四、PRD机制设计质疑

### CH-F021 | 建筑间无协同机制 — "孤岛系统"

- **严重度**: 🔴 P0
- **质疑**: R1-Builder 在 GAP-004 中指出"8 座建筑完全独立运作，无互相加成/限制"。源码 `BuildingSystem.ts` 确认每座建筑的状态完全独立，没有任何交互逻辑。这意味着升级策略极其单一：均衡升级即可，没有"选择 A 就不能选择 B"的决策张力。
- **玩家视角**: "建筑之间没有任何关系，随便升就行了。没有什么'我的建筑布局'的概念。"
- **深度分析**:
  - **无协同效果**: 农田 Lv10 + 市集 Lv10 没有额外加成
  - **无对抗效果**: 升级兵营不会影响其他建筑
  - **无组合策略**: 没有"经济流"、"军事流"、"科技流"的分化
  - **无布局概念**: 建筑位置不影响任何效果
  - **行业对比**: 优秀 SLG 游戏通常有丰富的建筑协同系统
- **改进建议**: 引入建筑协同系统：
  - 农田 + 市集 Lv10 = "商贸繁荣"（产出各 +15%）
  - 兵营 + 铁匠铺 Lv10 = "精锐之师"（兵力质量 +20%）
  - 书院 + 医馆 Lv10 = "文武双全"（科技研究速度 +25%）

### CH-F022 | 升级策略单一 — "无决策游戏"

- **严重度**: 🟡 P1
- **质疑**: R1-Builder 在 9.3.1 中指出"几乎无决策"。源码 `BuildingRecommender.ts` L42-L85 确认推荐算法是固定顺序（newbie/development/late），所有玩家的最优路径相同。这意味着建筑系统是一个"点点点"的等待游戏，没有任何策略深度。
- **玩家视角**: "我按照推荐顺序升级就行了，不需要动脑。"
- **深度分析**:
  - **无资源分配权衡**: 产出建筑产出不同类型资源，不存在竞争
  - **无机会成本**: 升级 A 不影响升级 B，只要有队列槽位
  - **无策略多样性**: 所有玩家的最优路径相同
  - **无"军事流"vs"经济流"vs"科技流"的分化**
- **改进建议**: 引入决策点：
  - 建筑协同系统（选择不同的协同组合）
  - 建筑特化系统（每个建筑可选择 2 种特化方向）
  - 资源分配权衡（升级 A 会消耗 B 需要的资源）

### CH-F023 | 队列系统设计问题 — "新手瓶颈"

- **严重度**: 🟡 P1
- **质疑**: 源码 `building-config.ts` L442-L447 确认 `QUEUE_CONFIGS` 中 Lv1-5 只有 1 个队列槽位。这意味着新手期只能同时升级 1 座建筑。新手期是玩家体验的关键时期，1 个槽位会导致：
  1. 等待时间过长（升级一个建筑后才能升级下一个）
  2. 资源囤积（等待期间资源持续产出，但无法使用）
  3. 新手流失（等待无聊）
- **玩家视角**: "我有足够资源升级 3 个建筑，但只能一个一个来。太慢了。"
- **深度分析**:
  - 1 个槽位对新手体验的影响
  - 资源囤积与升级节奏的矛盾
  - 队列槽位解锁的节奏（Lv6 才解锁第 2 个槽位）
  - 付费解锁队列槽位的可能性
- **改进建议**:
  - 方案 1: 新手期提供 2 个槽位（Lv1-5 = 2 slots）
  - 方案 2: 引入付费解锁队列槽位（元宝购买永久槽位）
  - 方案 3: 引入"建筑工人"系统（类似 Clash of Clans）

### CH-F024 | 取消升级 80% 返还不公平 — "惩罚机制"

- **严重度**: 🟢 P2
- **质疑**: 源码 `building-config.ts` L454 确认 `CANCEL_REFUND_RATIO = 0.8`，`BuildingSystem.ts` L255-L275 确认取消升级返还 80% 资源。但 80% 返还意味着玩家取消升级会损失 20% 资源，这是一个惩罚机制。
- **玩家视角**: "我误点了升级，想取消，但要损失 20% 资源。太不公平了。"
- **深度分析**:
  - 80% 返还的设计意图（防止滥用取消？）
  - 误操作的处理（是否需要确认弹窗？）
  - 取消升级的场景（资源不足？策略改变？）
  - 100% 返还 vs 80% 返还的利弊
- **改进建议**:
  - 方案 1: 升级前增加确认弹窗，取消升级 100% 返还
  - 方案 2: 保持 80% 返还，但增加"免费取消"次数（每日 3 次）
  - 方案 3: 根据取消时间返还不同比例（刚开始取消 100%，快完成取消 50%）

### CH-F025 | 推荐系统缺乏长期规划 — "短视推荐"

- **严重度**: 🟡 P1
- **质疑**: R1-Builder 在 GAP-005 中指出推荐系统"仅基于当前资源和 priority 算法，不考虑长期规划"。源码 `BuildingRecommender.ts` L90-L152 确认推荐算法只考虑当前资源和产出增益，不考虑：
  - 下一个解锁里程碑（如主城 Lv5 解锁城墙）
  - 资源产出-消耗平衡
  - 玩家当前游戏阶段
- **玩家视角**: "推荐系统让我升级农田，但我马上要解锁城墙了，应该先升主城。"
- **深度分析**:
  - 推荐算法的局限性（只看当前，不看未来）
  - 长期规划的缺失（不考虑解锁里程碑）
  - 阶段性推荐策略的缺失（newbie/development/late 是固定的，不根据实际状态调整）
  - 推荐理由的单一性（只说"产出提升"，不说"为什么现在升级这个"）
- **改进建议**: 改进推荐算法：
  - 考虑下一个解锁里程碑
  - 考虑资源产出-消耗平衡
  - 根据游戏阶段调整推荐策略
  - 提供多种推荐选项（而非单一推荐）

### CH-F026 | 无终局内容 — "满级后无事可做"

- **质疑**: R1-Builder 在 9.3.5 中指出"长期目标: 全建筑满级（~2 年，不现实）"。即使满级可以实现，满级后玩家能做什么？建筑系统没有任何终局内容。

  - 满级后的社交缺失（联盟建筑？城市展示？）
  - 满级后的经济缺失（资源溢出处理？）

- **改进建议**: 设计终局内容：
  - 建筑转生系统（满级后重置，获得永久加成）

  - 排行榜系统（建筑等级排名）
  - 联盟建筑系统（联盟共同升级的超级建筑）

## 六、新增PRD修订建议

### PRD-R007 | 建筑主动技能系统 [优先级: P1]

**PRD 原文**: 无（PRD 未涉及建筑主动技能）

**问题分析**:
1. 8 座建筑只有"升级"一个主动操作，交互极度单一
2. 建筑是"静态的数值容器"，不是"活的游戏实体"
3. 玩家与建筑的交互频率极低（升级一次后就只能等待）

**玩家视角**: "建筑就是放着等产出，点一下升级，然后就没别的事了。太无聊了。"

**修订方案**:

为每座建筑设计 1-2 个主动技能：

| 建筑 | 技能名称 | 效果 | 冷却 | 消耗 |
|------|---------|------|------|------|
| 农田 | 丰收 | 产出翻倍 1 小时 | 4 小时 | 100 粮草 |
| 市集 | 商贸繁荣 | 铜钱产出 +50% 2 小时 | 6 小时 | 200 铜钱 |
| 兵营 | 征兵令 | 立即获得 1 小时兵力产出 | 3 小时 | 50 粮草 |
| 铁匠铺 | 精工打造 | 下一次装备强化必定成功 | 12 小时 | 500 铜钱 |
| 书院 | 学术交流 | 当前科技研究进度 +10% | 8 小时 | 100 科技点 |
| 医馆 | 治疗 | 立即恢复 1 小时伤兵 | 2 小时 | 200 粮草 |
| 城墙 | 坚壁清野 | 防御加成 +100% 30 分钟 | 6 小时 | 100 兵力 |
| 主城 | 诏令 | 所有建筑产出 +20% 1 小时 | 12 小时 | 500 铜钱 |

**影响评估**: 大幅增加玩家与建筑的交互频率，提升游戏趣味性
**优先级**: P1

### PRD-R008 | 建筑事件系统 [优先级: P2]

**PRD 原文**: 无（PRD 未涉及建筑事件）

**问题分析**:
1. 建筑完全静态，没有任何随机事件
2. 游戏缺乏惊喜和意外
3. 玩家登录后没有"今天有什么新鲜事"的感觉

**玩家视角**: "建筑永远在安静地产出，没有任何惊喜或意外。"

**修订方案**:

引入建筑事件系统：

| 事件类型 | 事件名称 | 触发概率 | 效果 | 持续时间 |
|---------|---------|---------|------|---------|
| 正面 | 丰收之年 | 5%/天 | 农田产出翻倍 | 2 小时 |
| 正面 | 商队来访 | 5%/天 | 市集产出 +50% | 2 小时 |
| 正面 | 名将投奔 | 2%/天 | 兵营产出 +100% | 1 小时 |
| 负面 | 瘟疫 | 3%/天 | 农田产出 -50% | 1 小时 |
| 负面 | 盗贼横行 | 3%/天 | 市集产出 -50% | 1 小时 |
| 负面 | 士气低落 | 3%/天 | 兵营产出 -50% | 1 小时 |

**影响评估**: 增加游戏的随机性和趣味性，鼓励玩家频繁登录
**优先级**: P2

### PRD-R009 | 资源上限机制 [优先级: P1]

**PRD 原文**: 无（PRD 未涉及资源上限）

**问题分析**:
1. 资源可以无限累加，破坏升级节奏
2. 离线收益没有上限，长时间离线后资源溢出
3. 一键收取失去意义（资源已经实时累加）

**玩家视角**: "我离线了 3 天，登录后资源直接够升 10 级，这正常吗？"

**修订方案**:

引入资源上限机制：

| 资源类型 | 上限计算公式 | 说明 |
|---------|------------|------|
| 粮草 | 1000 + 农田等级 × 500 | 农田等级越高，上限越高 |
| 铜钱 | 800 + 市集等级 × 400 | 市集等级越高，上限越高 |
| 兵力 | 500 + 兵营等级 × 300 | 兵营等级越高，上限越高 |

**资源达到上限后的处理**:
- 产出停止累加（溢出丢弃）
- UI 显示"资源已满"警告
- 一键收取按钮闪烁提醒

**影响评估**: 控制资源膨胀，维护升级节奏，增加一键收取的意义
**优先级**: P1

### PRD-R010 | 武将派驻系统 [优先级: P1]

**PRD 原文**: BLD-3 提到"武将（政治型）→ 农田/市集产出 + 政治%"和"武将（统率型）→ 兵营产出 + 统率%"

**问题分析**:
1. PRD 提到武将加成，但没有具体的派驻流程
2. 玩家不知道如何将武将派驻到建筑
3. 武将加成无法生效

**玩家视角**: "我有政治型武将，但不知道怎么把他派驻到农田去加成产出。"

**修订方案**:

完整的武将派驻系统：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 点击建筑详情面板中的"派驻武将"按钮 | 打开武将选择界面 |
| 2 | 选择可派驻的武将 | 显示武将列表，标注属性加成 |
| 3 | 确认派驻 | 武将派驻到建筑，开始提供加成 |
| 4 | 查看加成效果 | 建筑详情面板显示当前武将加成 |
| 5 | 卸任武将 | 武将离开建筑，加成消失 |

**派驻限制**:
- 一个武将只能同时派驻 1 座建筑
- 每座建筑最多派驻 1 名武将
- 武将派驻后 24 小时内不可卸任（防止滥用）

**影响评估**: 增加武将的用途，提升建筑产出，增加策略深度
**优先级**: P1

### PRD-R011 | 终局内容设计 [优先级: P0]

**PRD 原文**: 无（PRD 未涉及终局内容）

**问题分析**:
1. 全建筑满级后无事可做
2. 长期目标不明确
3. 玩家缺乏持续追求

**玩家视角**: "我花了一年把所有建筑升到满级，然后呢？没东西玩了。"

**修订方案**:

终局内容系统：

| 系统 | 说明 | 目标 |
|------|------|------|
| 建筑转生 | 满级后重置为 Lv1，获得永久加成 | 追求更高的永久加成 |
| 建筑进化 | 满级后解锁新形态（如农田→皇家园林） | 追求更华丽的外观 |
| 排行榜 | 建筑等级排名、总产出排名 | 追求排名和荣誉 |
| 联盟建筑 | 联盟共同升级的超级建筑 | 追求联盟荣誉 |
| 城市展示 | 展示自己的城市布局 | 追求美观和炫耀 |

**影响评估**: 提供长期追求目标，延长游戏生命周期
**优先级**: P0

## 七、方法论复盘

### 本轮测试方法评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 流程发现效率 | 8/10 | Builder 枚举了 9 个主流程、32 个子流程、6 个跨系统流程，覆盖率 76.9%，效率较高。但缺失了一键收取、加速系统、自动升级等 PRD 明确提到的流程。 |
| 问题发现深度 | 9/10 | Challenger 发现了 30 个问题，其中 8 个 P0、12 个 P1，覆盖了流程完备性、实体角色、数值平衡、机制设计、跨系统流程 5 个维度。深度较好。 |
| 裁决合理性 | N/A | R1 无 Arbiter 裁决 |
| PRD评估有效性 | 8/10 | PRD 修订建议覆盖了医馆重设计、建筑扩充、数值优化、协同系统、一键收取、产出模型统一 6 个方面，质量较高。但缺少建筑主动技能、事件系统、资源上限、武将派驻、终局内容等重要修订。 |

### 本轮测试发现的方法论问题

1. **PRD 流程枚举不完整**: Builder 枚举了源码中实现的流程，但遗漏了 PRD 中提到但未实现的流程（一键收取、加速、自动升级）。这说明 Builder 过于依赖源码，而忽视了 PRD。
   - **影响评估**: PRD 覆盖率从 100% 降至 76.9%，3 个需求未覆盖
   - **改进建议**: 在 Builder 工作准则中增加"必须同时参考 PRD 和源码，不能遗漏 PRD 中提到但未实现的流程"

2. **跨系统流程枚举不足**: Builder 只枚举了 6 个跨系统流程，但建筑系统与武将系统、科技系统、攻城系统都有密切关联。跨系统流程的枚举需要更深入。
   - **影响评估**: 武将派驻、科技加成叠加、攻城防御消费等流程缺失
   - **改进建议**: 在 Builder 工作准则中增加"必须枚举所有跨系统的数据流和加成注入流程"

3. **数值分析深度不够**: Builder 提供了 ROI 分析和数值追溯，但没有深入分析"为什么这样设计"以及"是否有更好的方案"。数值分析应该更注重设计意图和替代方案。
   - **影响评估**: 只发现了问题，没有提供解决方案
   - **改进建议**: 在 Builder 工作准则中增加"数值分析必须包含设计意图分析和替代方案"

4. **实体角色评估模板不够细化**: Builder 的实体评估使用了 5 个维度（存在感、不可替代性、交互性、差异化、成长感），但缺少"玩家感知度"和"策略深度"两个重要维度。
   - **影响评估**: 医馆的"无感知"问题被低估
   - **改进建议**: 在实体角色评估模板中增加"玩家感知度"和"策略深度"两个维度

### 下一轮改进建议

1. **Builder 改进**:
   - 枚举流程时，同时参考 PRD 和源码，不要遗漏 PRD 中提到但未实现的流程
   - 增加跨系统流程的枚举深度，特别是武将派驻、科技加成、攻城防御等流程
   - 数值分析应更注重设计意图和替代方案

2. **Challenger 改进**:
   - 增加"玩家感知度"和"策略深度"两个评估维度
   - 更深入地挑战数值设计，分析"为什么这样设计"以及"是否有更好的方案"
   - 更关注跨系统流程的完整性

3. **方法论改进**:
   - 在 Builder 工作准则中增加"必须枚举 PRD 中提到但未实现的流程"
   - 在实体角色评估模板中增加"玩家感知度"和"策略深度"两个维度
   - 在数值平衡评估中增加"设计意图分析"和"替代方案分析"
