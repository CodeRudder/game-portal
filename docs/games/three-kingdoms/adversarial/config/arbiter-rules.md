# Arbiter Agent Rules — 三国霸业

> 版本: v1.6 | 初始化: 2026-05-01
> 每轮复盘后更新此文件

## 评分规则

### 5维度评分（各0-10分）
| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 完备性 | 25% | 所有API和场景是否覆盖 |
| 准确性 | 25% | covered标注是否真实，P0声称是否可靠 |
| 优先级 | 15% | P0/P1/P2分配是否合理 |
| 可测试性 | 15% | 每个节点能否转化为测试用例 |
| 挑战应对 | 20% | Builder是否充分回应了Challenger的质疑 |

### 封版线: 9.0分

### R1特殊考量
- 首轮评分应考虑系统性问题的影响（一个系统性问题可能影响多个节点）
- 虚报率>5%应扣准确性分
- 算法正确性问题应提高优先级

### 穿透质量评估
- FIX穿透率纳入准确性评分（穿透率>20%扣1分）
- 集成缺失问题提升为最高优先级P0
- 可玩性阻断项独立追踪，不计入常规P0计数

### 保存/加载架构评审 (AR-008)
- 每个子系统必须验证"六处同步"：GameSaveData类型 + SaveContext接口 + buildSaveData序列化 + toIGameState传递 + fromIGameState提取 + applySaveData反序列化
- 遗漏任何一处即视为架构级P0
- 典型案例：6个子系统有serialize/deserialize但未接入engine-save，存档后数据全部丢失

### 收敛加速规则 (AR-009)
- R3起引入"架构级缺陷一票否决"：保存/加载流程缺失子系统直接判定为CONTINUE
- 收敛判断需额外验证：所有子系统的serialize/deserialize是否被engine-save覆盖
- 如果R3仍发现架构级P0，下一轮必须专项审查而非常规迭代

### 可玩性阻断项封版权重 (AR-010)
- 可玩性阻断项（如存档丢失）应提升权重至常规P0的1.5倍
- 评分时单独计算可玩性阻断项的影响
- 典型案例：6个子系统存档丢失导致玩家进度归零，直接影响可玩性评分

### covered标注验证 (AR-011)
- covered标注必须经源码验证，虚报率>5%扣准确性分
- NaN防护标注为covered时必须验证所有入口
- 配置一致性标注必须交叉验证

## 封版条件（全部满足）

1. 评分 >= 9.0
2. API覆盖率 >= 90%
3. F-Cross覆盖率 >= 75%
4. F-Lifecycle覆盖率 >= 70%
5. P0节点覆盖 = 100%
6. 虚报数 = 0
7. 最终轮新P0 = 0
8. 所有子系统覆盖 = 是

## 收敛判断规则

1. 连续2轮评分差 < 0.5 → 收敛信号
2. 最终轮新P0 = 0 → 强收敛信号
3. 节点增量 < 30 → 饱和信号
4. Challenger 5分钟内无新P0 → 封版信号

## 三Agent复盘规则

每轮必须对三个Agent进行复盘：
1. Builder表现评分 + 改进建议
2. Challenger表现评分 + 改进建议
3. Arbiter独立发现（约10%补充）
4. 规则进化建议（具体到规则文件）

## 可玩性评估规则

对游戏模块额外评估：
| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 趣味性 | 25% | 玩法是否有趣 |
| 进度平衡 | 25% | 成长曲线是否合理 |
| 经济平衡 | 20% | 资源获取/消耗是否平衡 |
| 玩家体验 | 15% | 整体体验是否流畅 |
| 系统一致性 | 15% | 各系统是否协调 |

### 有条件封版规则 (AR-012)
- 对称函数对（如attack/defense、add/remove、serialize/deserialize）中，如果一侧有修复记录，必须验证另一侧是否需要相同修复
- 未验证完毕前，该模块不得进入封版状态
- 适用于所有P0修复的封版审查

### 关卡系统状态锁验证 (AR-013)
- completeChallenge/completeStage类API必须验证前置状态锁（如preLock/preCheck）
- 未满足前置条件时拒绝执行并返回空结果
- 典型案例：completeChallenge未验证preLockedResources，跳过preLockResources直接调用可免费获得奖励
- 经济系统"预锁→消费→发奖"链路完整性纳入准确性评分
