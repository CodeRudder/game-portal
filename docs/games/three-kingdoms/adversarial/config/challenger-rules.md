# Challenger Agent Rules — 三国霸业

> 版本: v1.4 | 初始化: 2026-05-01
> 每轮复盘后更新此文件

## 通用规则

1. 从5个维度系统性质疑（F-Normal/F-Boundary/F-Error/F-Cross/F-Lifecycle）
2. 每个质疑必须引用源码（文件+行号）
3. 每个P0声称必须包含复现场景
4. 虚报率必须测量并报告
5. 封版轮仅关注P0级遗漏

## 高频P0检查清单

### null/undefined防护
- 检查每个公开API的参数是否有null guard
- 检查可选参数解构是否有fallback
- 检查deserialize(null)是否安全

### NaN传播
- 检查数值计算链是否有NaN检查
- 检查Math.max/Math.min是否处理NaN输入
- 检查除法是否有除零保护

### 负值漏洞
- 检查伤害/消耗/扣除是否有<=0检查
- 检查负值是否会变成"治疗"或"奖励"

### 浅拷贝副作用
- 检查数组展开后是否修改嵌套属性
- 检查Object.assign后是否修改深层对象

### 竞态条件
- 检查资源预锁后异常是否回滚
- 检查循环体内异常是否导致永久卡死

### 经济漏洞
- 检查购买/兑换是否有日限购/总限购
- 检查是否有无限刷资源路径

### NaN绕过 <= 0 检查
- 搜索所有 `if (x <= 0)` 模式，验证是否被NaN绕过
- 搜索所有 `if (x > 0)` 模式，验证NaN是否通过
- 修复建议：使用 `!Number.isFinite(x) || x <= 0`

### 配置交叉验证
- 不同配置文件中同名ID是否一致
- 配置值范围是否合理
- 新增配置项是否有对应的获取路径

### FIX穿透验证
- 每个FIX必须验证底层函数是否也被修复
- 修复调用方≠修复根因，需追溯调用链
- 穿透率 = 遗漏修复数 / 总修复数，目标<10%

### 修复方案要求
- P0修复必须附带验证测试
- 修复方案必须说明根因而非仅处理调用方

### 保存/加载完整性挑战
- 每个子系统必须验证serialize()输出是否被engine-save的buildSaveData()引用
- 每个子系统必须验证deserialize()是否被applySaveData()调用
- 新增子系统时必须挑战"六处同步"完整性（GameSaveData/SaveContext/buildSaveData/toIGameState/fromIGameState/applySaveData）
- 典型案例：6个子系统有serialize/deserialize但未被engine-save调用，存档后数据丢失

### 新维度探索
- 不满足于已有维度，主动探索新P0维度
- R3新发现维度：保存/加载流程完整性（独立于F-Lifecycle）
- 建议持续探索：配置热更新、跨版本迁移、离线数据同步

### 遗留P0重新评估
- 每轮应重新评估前轮降级为P1/P2的缺陷
- 系统性修复可能改变遗留缺陷的影响范围
- 典型案例：R1的null guard问题在R3编队系统中仍有残留

### 战斗数值安全
- 所有伤害计算函数必须验证NaN/负数/Infinity输入
- buff.value必须验证NaN
- skillMultiplier必须验证范围(0, maxMultiplier)

### 配置-枚举同步验证
- 枚举值必须与配置数组完全对应
- 新增枚举值时必须同步更新配置

## 三国霸业特定挑战策略

1. 武将系统：重点检查双路径一致性（HeroSystem vs HeroLevelSystem）
2. 战斗系统：重点检查NaN/负值传播链
3. 攻城系统：重点检查子系统数据完整性
4. 经济系统：重点检查无限购买/负值漏洞

## 可玩性评估维度

1. 进度曲线是否平滑（不会太快也不会太慢）
2. 奖励反馈是否及时（玩家行为是否有明确反馈）
3. 策略深度是否足够（是否有多种有效策略）
4. 经济平衡是否合理（通胀/通缩风险）
5. 新手体验是否友好（前10分钟体验）
