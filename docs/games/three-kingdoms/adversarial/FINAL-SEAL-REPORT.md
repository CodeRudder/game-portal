## 一、项目概述

## 四、方法论总结 — 3-Agent 对抗式测试流程

### 4.1 核心架构

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Builder   │────▶│ Challenger  │────▶│   Arbiter   │
│  (构建者)    │     │  (挑战者)    │     │  (仲裁者)    │
│             │     │             │     │             │
│ · 枚举测试树 │     │ · 5维度挑战  │     │ · 逐项裁决   │
│ · 标注优先级 │     │ · 发现P0缺陷 │     │ · 5维度评分  │
│ · 修复缺陷   │     │ · 验证FIX    │     │ · 封版判定   │
└─────────────┘     └─────────────┘     └─────────────┘
       │                    │                    │
       └────────────────────┼────────────────────┘
                            ▼
                   ┌─────────────────┐
                   │   规则进化引擎    │
                   │                 │
                   │ · 每轮复盘更新    │
                   │ · P0模式积累     │
                   │ · 跨模块知识复用  │
                   └─────────────────┘
```

### 4.2 三 Agent 职责

| Agent | 角色 | 核心职责 |
|-------|------|----------|
| **Builder** | 构建者 | 枚举测试分支树（5 DAG 类型）、标注优先级、修复缺陷、验证 FIX 穿透 |
| **Challenger** | 挑战者 | 5 维度系统性挑战（Normal/Boundary/Error/Cross/Lifecycle）、发现 P0 缺陷、验证修复完整性 |
| **Arbiter** | 仲裁者 | 逐项裁决 P0/P1/P2、5 维度加权评分、封版条件判定、规则进化建议 |

### 4.3 封版条件（8 项）

| # | 条件 | 阈值 |
|---|------|------|
| 1 | 综合评分 | ≥ 9.0/10 |
| 2 | 新 P0 缺陷 | = 0 |
| 3 | P0 修复率 | = 100% |
| 4 | FIX 穿透率 | = 0%（无遗漏） |
| 5 | 虚报率 | ≤ 5% |
| 6 | API 覆盖率 | ≥ 85% |
| 7 | 可玩性评分 | 无阻断项 |
| 8 | 跨系统链路 | 全部 covered |

### 4.4 迭代收敛模式

```
R1: 发现大量 P0 → 修复 → 评分 7.0~8.5
R2: 验证 FIX 穿透 → 少量新 P0 → 修复 → 评分 9.0+ → SEALED
R3/R4: 深度验证 → 极少新 P0 → 确认封版（仅 battle/hero/pvp）
```
