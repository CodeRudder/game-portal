## 1. 概述

### 1.1 目标

建立 **自我进化的 AI 对抗测试反馈机制**，使三国霸业项目在 30 轮迭代内达到封版质量。核心思路：对抗驱动 → 闭环反馈 → 持续进化。

### 1.2 适用范围

| 维度 | 覆盖 |
|------|------|
| 模块 | 武将、战斗、资源、城建、战役、联盟 |
| 类型 | 单元 / 集成 / E2E / 对抗 / 性能 |
| 层级 | Engine Core → Subsystem → API → UI |

### 1.3 术语表

| 术语 | 定义 |
|------|------|
| **Round** | 一次完整的 对抗测试→修复→验证 循环 |
| **Phase** | 5 轮为一组，聚焦特定测试维度 |
| **P0-P3** | 阻塞 / 严重 / 一般 / 轻微 |
| **规则库** | 对抗测试积累的缺陷模式扫描规则集合 |

## 2. 反馈闭环模型

### 2.1 六阶段闭环

```
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ ① 对抗   │───▶│ ② 缺陷   │───▶│ ③ 修复   │
  │   测试   │    │   发现   │    │   执行   │
  └──────────┘    └──────────┘    └──────────┘
       ▲                               │
       │                               ▼
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ ⑥ 下一轮 │◀───│ ⑤ 方法论 │◀───│ ④ 架构   │
  │  对抗测试 │    │   优化   │    │   审查   │
  └──────────┘    └──────────┘    └──────────┘
```

### 2.2 各阶段说明

| 阶段 | 执行者 | 输入→输出 | 关键规则 |
|------|--------|----------|---------|
| **① 对抗测试** | Red Team Agent | 代码+规则库 → 原始缺陷报告 | 每轮 ≤ 2h |
| **② 缺陷发现** | QA Agent | 原始报告 → 结构化清单 | 去重、P0-P3分级、模式标记 |
| **③ 修复执行** | Blue Team Agent | 缺陷清单 → 补丁+回归用例 | P0: 24h / P1: 48h |
| **④ 架构审查** | Arch Agent | 补丁+模式分析 → 健康度评分 | P0修复后额外专项审查 |
| **⑤ 方法论优化** | Strategy Agent | 审查结果 → 更新规则库+策略 | 见 §4 进化规则 |
| **⑥ 回归对抗** | Red Team Agent | 下一轮使用更新后规则库 | 闭环完成 |

### 2.3 数据流

```
对抗测试 ─┬─ 缺陷原始数据 ──▶ 缺陷发现 ─┬─ 新P0模式 ──▶ 规则库入库
          ├─ 覆盖率数据 ────▶ 度量仪表盘  ├─ 重复模式 ──▶ 规则强化
          └─ 性能基线 ──────▶ 性能追踪    └─ 虚报分析 ──▶ 验证策略调优
修复执行 ─┬─ 修复补丁 ──────▶ 架构审查 ──┬─ 架构评分 ──▶ 封版判定
          └─ 回归用例 ──────▶ 用例库扩充  └─ 改进建议 ──▶ 方法论优化
```

## 3. 度量指标体系

### 3.1 核心指标 (KPI)

| 指标 | 缩写 | 公式 | 目标 | 频率 |
|------|------|------|------|------|
| **缺陷发现率** | DDR | 新缺陷数/执行用例数×100% | Phase内递减 | 每轮 |
| **修复质量率** | FQR | 一次通过修复数/总修复数×100% | ≥90% | 每轮 |
| **覆盖率增长率** | CRG | (本轮-上轮)/上轮×100% | 前期≥5%,后期≥1% | 每轮 |
| **方法论进化率** | MER | 新增更新规则数/规则库总数×100% | ≥10%(前期) | 每轮 |

### 3.2 辅助指标

| 指标 | 缩写 | 公式 | 用途 |
|------|------|------|------|
| 缺陷密度 | DD | 缺陷总数/千行代码 | 质量基准 |
| 虚报率 | FPR | 虚报数/报告总数×100% | 验证策略有效性 |
| 回归失败率 | RFR | 失败用例/回归总数×100% | 修复稳定性 |
| 架构健康度 | AHS | 加权评分(10分制) | 架构质量追踪 |
| P0趋势 | P0T | 近5轮P0移动平均 | 稳定性趋势 |
| 规则命中率 | RHR | 命中缺陷数/规则数×100% | 规则有效性 |

### 3.3 阈值与告警

| 指标 | 🟢 健康 | 🟡 警告 | 🔴 危险 | 动作 |
|------|---------|---------|---------|------|
| DDR | <5% | 5-10% | >10% | 策略升级 |
| FQR | >90% | 80-90% | <80% | 修复流程审查 |
| CRG | >2% | 1-2% | <1% | 测试策略调整 |
| FPR | <5% | 5-15% | >15% | 验证策略重构 |
| AHS | >8.5 | 7.0-8.5 | <7.0 | 架构专项治理 |

## 4. 进化规则引擎

### 4.1 规则映射

```
缺陷模式 ──────▶ 进化动作
  ├── 新P0模式 ────────▶ ER-1: 扫描规则库扩充
  ├── 虚报模式 ────────▶ ER-2: 验证策略优化
  ├── 架构级缺陷 ──────▶ ER-3: 预防检查注入
  ├── 重复出现缺陷 ────▶ ER-4: 规则精度提升
  └── 跨模块连锁缺陷 ──▶ ER-5: 集成策略升级
```

### 4.2 进化规则详解

**ER-1: 新P0模式 → 扫描规则库扩充**
- 触发: 发现规则库未覆盖的新P0缺陷模式
- 动作: 提取缺陷特征 → 生成扫描规则 → 回溯历史轮次 → 入库标记Critical → 下轮强制包含变体
- 验证: 新规则下轮至少命中1次有效缺陷，虚报率<10%

**ER-2: 虚报分析 → 验证策略优化**
- 触发: 单轮FPR>10% 或 连续2轮FPR>5%
- 动作: 虚报聚类 → 区分类型(环境抖动/时序竞争/断言过严) → 针对性调整验证模板
- 验证: 调整后下轮FPR<5%

**ER-3: 架构问题 → 预防检查注入**
- 触发: 缺陷根因标记为「架构级」
- 动作: 记录模式(循环依赖/状态泄漏/单点故障) → 生成预防规则(静态分析+设计约束+运行时监控) → 注入CI流水线
- 验证: 同类架构缺陷后续5轮零复发

**ER-4: 重复缺陷 → 规则精度提升**
- 触发: 同模式3轮内重复≥2次
- 动作: 分析原因 → 扩展覆盖(参数组合+状态空间+边界变体) → 标记「已强化」
- 验证: 强化后同类缺陷覆盖率≥95%

**ER-5: 跨模块连锁 → 集成策略升级**
- 触发: 单缺陷影响≥2个子系统
- 动作: 绘制影响链路图 → 生成跨模块场景(正向/反向/并发链路) → 更新集成策略
- 验证: 下轮链路覆盖率≥80%

### 4.3 规则生命周期

```
创建 ──────▶ 活跃 ──────▶ 强化 ──────▶ 退休
提案审核     正式生效     精度提升     连续5轮零命中归档
```

### 4.4 规则结构

```yaml
rule:
  id: "RULE-ADV-001"
  name: "武将属性溢出扫描"
  category: "武将系统"
  priority: "Critical"
  status: "Active"           # Active / Strengthened / Retired
  version: 3
  created_round: "R03"
  last_hit_round: "R12"
  pattern:
    input_type: "hero_attribute_overflow"
    target_fields: ["attack", "defense", "hp", "speed"]
    boundary_strategy: "beyond_max_int + negative + zero + float_injection"
  validation:
    assertion_type: "state_invariant"
    expected: "属性值在 [0, MAX_STAT] 范围内"
  evolution_log:
    - { round: "R03", action: "创建", trigger: "P0攻击力溢出导致崩溃" }
    - { round: "R07", action: "强化", trigger: "ER-4扩展浮点数注入" }
```

### 5.8 每轮执行模板 (≤4h)

```
Step 1: 环境准备 (15min)     → 拉代码、部署、加载规则库
Step 2: 对抗测试 (≤2h)       → AI生成用例、自动执行、收集数据
Step 3: 缺陷分析 (30min)     → 去重分级、根因分类、模式识别
Step 4: 修复验证 (≤1h)       → 按优先级修复、回归验证、架构审查
Step 5: 方法论进化 (15min)   → 规则库更新、策略调整、生成报告
```

## 6. 封版标准

### 6.1 硬性标准（必须全部满足）

| # | 维度 | 标准 | 验证方式 |
|---|------|------|---------|
| 1 | **API覆盖率** | **≥95%** | 覆盖率工具 |
| 2 | **P0缺陷** | **= 0** | 缺陷追踪 |
| 3 | **P1修复率** | **≥90%** | 修复记录 |
| 4 | **架构健康度** | **≥9.0** | Arch Agent评分 |
| 5 | **规则库** | **≥50条有效规则** | 规则库统计 |

### 6.2 推荐标准（应满足≥3项）

| # | 维度 | 标准 |
|---|------|------|
| 6 | 分支覆盖率 | ≥80% |
| 7 | 虚报率 | ≤3% |
| 8 | 修复质量率 | ≥95% |
| 9 | P99响应 | <200ms |
| 10 | 72h稳定性 | 零崩溃、零泄漏 |

### 6.3 封版决策流程

```
R30完成 → 生成检查报告 → 硬性标准全通过？
  │ No  → 延期修复，追加R31+
  │ Yes → 推荐标准≥3项通过？
           │ No  → 评估风险，有条件封版或延期
           │ Yes → QA+架构师+PM联合签字 → ✅ 正式封版
```

## 7. 风险管理

| ID | 风险 | 概率 | 影响 | 应对 |
|----|------|------|------|------|
| R-01 | AI对抗用例质量不足 | 中 | 高 | 人工审核+规则模板 |
| R-02 | 修复引入新缺陷 | 中 | 高 | 强制回归+架构审查门禁 |
| R-03 | 30轮内未达封版标准 | 低 | 极高 | Phase评估点可追加轮次 |
| R-04 | 规则库膨胀执行过慢 | 中 | 中 | 规则退休机制+优先级调度 |
| R-05 | 虚报率过高 | 中 | 中 | ER-2验证策略优化 |

**应急预案**:

| 场景 | 触发 | 动作 |
|------|------|------|
| P0爆发 | 单轮≥3个P0 | 暂停对抗，专项修复+根因分析 |
| 进度落后 | Phase退出条件未满足 | 追加2-3轮，调整后续计划 |
| 架构退化 | AHS连续2轮下降 | 架构专项治理Sprint |
| 规则失效 | RHR<30%持续3轮 | 全面规则库审查+重建 |
