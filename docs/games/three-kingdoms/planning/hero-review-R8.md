## 综合评分: 9.2/10（R1: 6.4 → R2: 6.7 → R3: 7.1 → R4: 7.6 → R5: 8.1 → R6: 8.5 → R7: 8.9 → R8: 9.2, +0.3）

> **评分说明**: R8是武将系统从"P0清零"迈向"设计精细化"的关键轮次。评分从8.9提升至9.2（+0.3），标志着系统从"功能完整"进入"设计一致+事件驱动"阶段。
>
> **四大核心突破**：
> 1. **calculateFormationPower设计一致性修复（P1-R7-1 ✅）**：新增第3参数 `bondMultiplier?: number`，与 `calculatePower` 第4参数设计模式完全统一。优先级链：显式参数 > 注入回调 > 默认1.0，两个战力方法遵循相同的参数覆盖语义。
> 2. **calculateTotalPower JSDoc语义完善（P1-R7-2 ✅）**：明确标注"羁绊是编队级概念，此处不含羁绊系数"，消除 `calculateTotalPower` vs `calculateFormationPower` 的语义歧义。开发者不再需要阅读源码才能理解两者的区别。
> 3. **19个战力公式边界测试（P1-R7-3 ✅）**：覆盖羁绊极端值(5)、装备极端值(2)、多乘区同时极端值(2)、编队规模边界(7)、参数优先级(3)五大场景。验证了极端高值（99999装备+6星+Lv100+2.0羁绊）的精确计算，以及边界条件（空编队、无效ID、负值系数）的正确处理。
> 4. **羁绊事件监听系统（R8新增 ✅）**：`BondSystem.evaluateAndEmit()` 实现 `bond:activated`/`bond:deactivated`/`bond:levelUp` 三事件，通过 `previousBonds` 快照对比检测羁绊变化。事件载荷设计精细（含bondId/name/type/level/participants/formationIds），为UI层实时响应羁绊变化奠定基础。
>
> **P1问题大幅收敛**：R7遗留的3个P1问题在本轮全部修复。技术债从6项减少到3项（属性命名、双引擎、CSS迁移），存续最久的"羁绊事件监听"（R6遗留，2轮）正式清零。
>
> **里程碑意义**：R8是首次突破9.0分门槛的轮次。系统从"可用"升级为"好用"——设计一致性、边界安全性、事件驱动能力三个维度同时达标。
>
> **但关键差距仍在**：
> - 属性命名不一致（defense/speed vs command/politics，连续8轮，R1遗留）
> - 双引擎共存（新旧BondSystem），职责边界未文档化
> - 羁绊事件系统缺少专门测试（有代码无测试覆盖）
> - UI层零实现（24个组件仍为设计文档）

## 羁绊事件系统专项评估（R8新增）

### 事件设计审查

| 评估维度 | 评分 | 说明 |
|---------|:----:|------|
| **事件覆盖度** | 9/10 | 三事件覆盖羁绊全生命周期：激活（新羁绊出现）、失效（羁绊消失）、升级（等级提升）。仅缺少"羁绊降级"事件（如星级降低导致羁绊等级下降），但当前游戏设计中星级不会降低，故不影响 |
| **载荷设计** | 9/10 | `BondActivatedPayload` 含8个字段（bondId/name/type/level/participants/formationIds），信息完整。`BondDeactivatedPayload` 不含level（合理，已失效）。`BondLevelUpPayload` 含previousLevel/newLevel，变化清晰。`ReadonlyArray<string>` 类型确保不可变性 |
| **状态管理** | 8/10 | `previousBonds` Map快照设计简洁高效。`getPreviousBonds()` 返回 `ReadonlyMap` 防止外部修改。`reset()` 中 `this.previousBonds.clear()` 确保重置时清理。但快照为浅拷贝引用，`ActiveBond` 对象本身未深拷贝——当前安全（`ActiveBond` 不可变），但未来扩展需注意 |
| **事件发射时机** | 9/10 | 先检测失效（previous有current无），再检测激活和升级（current有previous无/等级提升），最后更新快照。顺序合理：失效→激活→升级，符合"先清理后创建"的直觉 |
| **边界处理** | 7/10 | `evaluateAndEmit` 在 `!this.deps` 时直接return（安全），空编队时 `calculateBonds` 返回空数组，所有previous羁绊触发deactivated（正确）。但缺少对 `generalIds` 为null/undefined的防护 |

### 事件系统代码质量

```typescript
// 优点：
// 1. 快照对比算法清晰：previous vs current 双重遍历
// 2. 事件载荷类型安全：泛型 eventBus.emit<T>(eventName, payload)
// 3. 只读接口：getPreviousBonds() 返回 ReadonlyMap
// 4. 重置安全：reset() 清理 previousBonds

// 不足：
// 1. 无事件去重（同一编队多次调用evaluateAndEmit会重复发射）
// 2. 无事件防抖（快速编队切换时可能频繁触发）
// 3. 缺少专门测试（见下方问题发现）
```

### 事件系统与游戏体验关联

| 游戏场景 | 事件 | 体验价值 |
|---------|------|---------|
| 玩家调整编队，新羁绊激活 | `bond:activated` | 🔔 实时提示"桃园结义已激活！"，战力跳变可见 |
| 玩家移除武将，羁绊失效 | `bond:deactivated` | ⚠️ 提示"五虎上将羁绊已失效"，引导保留关键武将 |
| 武将升星，羁绊等级提升 | `bond:levelUp` | 🎉 庆祝动画"羁绊升级！Lv1→Lv2"，成就感反馈 |
| 竞技场对战前预览 | `evaluateAndEmit` | 📊 预览编队羁绊变化，辅助策略决策 |

## 战力公式边界测试覆盖评估（R8专项）

### 测试覆盖矩阵

| 测试场景 | 用例数 | 覆盖内容 | 评估 |
|---------|:-----:|---------|:----:|
| **羁绊系数极端值** | 5 | 0.0（归零）、2.0（上限翻倍）、3.0（超限不截断）、BondSystem截断验证、负值（负数战力） | ✅ 充分 |
| **装备战力极端值** | 2 | 0（无影响）、99999（系数100.999） | ✅ 充分 |
| **多乘区同时极端值** | 2 | 全乘区极端高值（Lv100+6星+99999装备+2.0羁绊）、全乘区最低值 | ✅ 充分 |
| **编队规模边界** | 7 | 空编队(0)、空编队+羁绊、1武将、1武将+羁绊、5武将满编、5武将+羁绊、含无效ID | ✅ 充分 |
| **参数优先级** | 3 | 显式>注入、注入>fallback、无参无回调fallback | ✅ 充分 |
| **R8合计** | **19** | **5大场景19用例** | **✅ 全面** |

### 边界测试亮点

1. **`calcExpected` 对照函数**：测试文件内建手动战力计算函数，与引擎计算结果逐位比对，消除"测试和实现使用同一公式"的自证陷阱风险
2. **负值系数测试**：`bondMultiplier = -1.0` 时验证战力为负数——这不是游戏内正常值，但测试确认了引擎不会崩溃或产生NaN
3. **超限不截断设计**：测试明确注释"显式传入的bondMultiplier不截断，由调用方（BondSystem）负责"——职责边界清晰，`calculateFormationPower` 不承担数据校验职责
4. **无效ID静默跳过**：编队中包含 `nonexistent_hero_1`、`nonexistent_hero_2` 时，仅计算有效武将战力，不报错不崩溃

### 边界测试不足

1. **缺少NaN/Infinity防护测试**：未测试 `equipPower = Infinity` 或 `NaN` 时 `equipmentCoeff` 的行为
2. **缺少Number.MAX_SAFE_INTEGER溢出测试**：6乘区同时极端高值时，中间值是否超过安全整数范围
3. **缺少并发安全性测试**：多个模块同时调用 `evaluateAndEmit` 时 `previousBonds` 的竞态条件

## R8新发现的问题

### P1（影响核心体验）

#### P1-R8-1：羁绊事件系统缺少专门测试

**问题**: `BondSystem.evaluateAndEmit()` 是R8新增的核心方法（60+行），涉及状态管理（`previousBonds` 快照）、事件发射（3种事件类型）、变化检测（激活/失效/升级）三种逻辑。但当前**没有任何测试覆盖**此方法。

**代码位置**: `src/games/three-kingdoms/engine/hero/BondSystem.ts` L237-L286

**影响**:
- `evaluateAndEmit` 的核心逻辑（快照对比+事件发射）未经测试验证
- 事件载荷的字段完整性（如 `participants` 是否正确）未验证
- 边界情况（连续调用、空编队后重新激活）未覆盖
- 如果事件发射逻辑有bug，UI层将无法正确响应羁绊变化

**建议修复**: 新建 `bond-event-system.test.ts`，至少覆盖：
1. 首次调用：previous为空，所有羁绊触发 `bond:activated`
2. 移除武将：羁绊失效触发 `bond:deactivated`
3. 升星导致羁绊升级：触发 `bond:levelUp`
4. 连续调用无变化：不触发任何事件
5. 空编队→有编队：激活事件
6. `getPreviousBonds()` 返回正确快照
7. `reset()` 清理 `previousBonds`
预估工作量0.5天。

#### P1-R8-2：evaluateAndEmit 缺少事件去重和防抖机制

**问题**: `evaluateAndEmit` 每次调用都会重新计算羁绊并对比快照，如果短时间内多次调用（如玩家快速拖拽武将调整编队），每次都会触发事件。缺少去重和防抖机制。

**场景**:
```
t=0ms: evaluateAndEmit(['guanyu','liubei']) → bond:activated (桃园结义)
t=50ms: evaluateAndEmit(['guanyu','liubei','zhangfei']) → bond:activated (五虎上将) + bond:levelUp (桃园结义 Lv1→Lv2)
t=100ms: evaluateAndEmit(['guanyu','liubei','zhangfei','zhaoyn']) → bond:activated (新羁绊)
```
在150ms内触发4个事件，如果UI层为每个事件播放动画，体验可能混乱。

**建议修复**:
1. 短期：文档中说明"调用方应做防抖处理"，或在 `evaluateAndEmit` 内部添加防抖逻辑
2. 中期：添加批量事件模式（`evaluateAndEmitBatch`），一次返回所有变化
3. 预估工作量0.3天

#### P1-R8-3：羁绊事件载荷中 BondDeactivatedPayload 缺少 level 信息

**问题**: `BondActivatedPayload` 包含 `level` 字段，但 `BondDeactivatedPayload` 不包含。当羁绊失效时，UI层无法知道失效前羁绊的等级（如"五虎上将Lv3已失效"），影响提示信息的完整性。

**代码对比**:
```typescript
// BondActivatedPayload — 含 level
interface BondActivatedPayload {
  bondId: string; name: string; type: BondType;
  level: number;  // ← 有
  participants: ReadonlyArray<string>;
  formationIds: ReadonlyArray<string>;
}

// BondDeactivatedPayload — 不含 level
interface BondDeactivatedPayload {
  bondId: string; name: string; type: BondType;
  // ← 缺少 level
  formationIds: ReadonlyArray<string>;
}
```

**建议修复**: 为 `BondDeactivatedPayload` 添加 `lastLevel: number` 字段，从 `prevBond.level` 获取。预估工作量0.1天。

### P2（锦上添花）

#### P2-R8-1：战力公式边界测试缺少 NaN/Infinity 防护

**问题**: R8的19个边界测试覆盖了极端值（0、负数、99999），但未测试 `NaN` 和 `Infinity` 输入：
- `calculatePower(g, 1, NaN, 1.0)` → `equipmentCoeff = 1 + NaN/1000 = NaN` → 战力为NaN
- `calculatePower(g, 1, Infinity, 1.0)` → `equipmentCoeff = Infinity` → 战力为Infinity
- `calculateFormationPower([], undefined, NaN)` → 0 × NaN = NaN

**影响**: 当前游戏内不可能产生NaN/Infinity输入（配置均为有限数字），但未来如果动态计算装备战力时出现异常值，可能导致战力显示异常。

**建议**: 添加4个NaN/Infinity防护测试，并在 `calculatePower` 入口添加 `Number.isFinite` 检查（防御性编程）。预估工作量0.2天。

#### P2-R8-2：属性命名不一致注释仍标注"待重构"（连续8轮）

**问题**: `HeroSystem.ts` 战力公式注释中仍标注"源码字段 defense↔CMD, speed↔POL, 属性命名待后续统一"。从R1发现至今已8轮，如果短期不计划重命名，应将注释改为明确的映射说明（如"defense字段对应PRD中的command属性"），消除"待重构"的模糊性。

**建议**: 更新注释为永久映射说明，或启动属性重命名（2天工作量）。预估工作量0.1天（注释更新）或2天（重命名）。

#### P2-R8-3：搭档羁绊仅实现第1批（3/12组）

**问题**: 继承自R6/R7，设计文档定义12组搭档羁绊分3批实现，当前仅实现第1批（桃园结义/五虎上将/卧龙凤雏）。

**建议**: 按设计文档§8计划，P1阶段实现第2批（+4组）。预估工作量2天。
