## 1. 武将升级流程

### 1.1 流程图

```
获取经验道具/战斗通关
        │
        ▼
  ┌─────────────┐
  │ 经验值累加   │ ← 经验书(小100/中500/大2000)、战斗(关卡等级×10)、离线挂机
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐     否
  │ 经验 ≥ 升级需求？├──────────→ 等待更多经验
  └──────┬──────┘
         │ 是
         ▼
  ┌─────────────┐     否
  │ 铜钱足够？   ├──────────→ 提示铜钱不足
  └──────┬──────┘
         │ 是
         ▼
  ┌─────────────┐     否
  │ 等级 < 等级上限？├───────→ 触发突破流程（见第3节）
  └──────┬──────┘
         │ 是
         ▼
  ┌─────────────┐
  │ 扣除经验+铜钱 │
  │ 等级 +1      │
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ 属性变化     │ → 四维属性 × (1 + 等级 × 0.03) 基础成长率
  │ 战力变化     │ → (ATK×2.0 + CMD×1.5 + INT×2.0 + POL×1.0) × (1+等级×0.05) × 星级系数
  │ 派驻加成更新  │ → 派驻建筑产出加成随等级提升
  └──────┬──────┘
         │
         ▼
    继续检查经验是否满足下一级
```

|:-------:|:----------:|:------:|:------------------:|:------------------:|
| 1→10 | 50 | 20 | 2,750（Σ₁¹⁰ n×50） | 1,100（Σ₁¹⁰ n×20） |
| 11→20 | 120 | 50 | 18,600（Σ₁¹²⁰ n×120） | 7,750（Σ₁¹²⁰ n×50） |
| 21→30 | 250 | 100 | 63,750（Σ²¹³⁰ n×250） | 25,500（Σ²¹³⁰ n×100） |
| 31→40 | 500 | 200 | 155,500（Σ³¹⁴⁰ n×500） | 62,200（Σ³¹⁴⁰ n×200） |
| 41→50 | 1,000 | 400 | 455,000（Σ⁴¹⁵⁰ n×1000） | 182,000（Σ⁴¹⁵⁰ n×400） |
| **1→50 合计** | — | — | **695,600** | **278,550** |

### 1.3 属性成长规则

每升一级，四维属性按基础值 × 3% 成长：

```
当前属性 = 基础属性 × (1 + (等级 - 1) × 0.03)
```

**示例（关羽，LEGENDARY品质）**：

| 等级 | 攻击力(ATK) | 统率(CMD) | 智力(INT) | 政治(POL) | 战力 |
|:---:|:---------:|:--------:|:--------:|:--------:|:---:|
| 1 | 115 | 90 | 65 | 78 | 699 |
| 10 | 146 | 114 | 83 | 99 | 1,158 |
| 20 | 181 | 141 | 103 | 122 | 1,849 |
| 30 | 215 | 168 | 122 | 145 | 2,688 |
| 40 | 250 | 195 | 142 | 168 | 3,674 |
| 50 | 284 | 222 | 162 | 191 | 4,808 |

> **战力公式**: (ATK×2.0 + CMD×1.5 + INT×2.0 + POL×1.0) × (1+等级×0.05) × 星级系数(1星=1.0)

### 3.5 突破石获取验证

> **关联PRD**: HER-8 突破石获取途径、HER-10.4 突破石经济

单武将全突破需 **75个突破石**，按HER-10.4修正后的日产出~17.5个计算：

| 目标 | 需求突破石 | 所需天数 |
|------|:---------:|:-------:|
| 一阶突破 | 5 | ~0.3天 |
| 二阶突破（累计15） | 15 | ~0.9天 |
| 三阶突破（累计35） | 35 | ~2.0天 |
| 四阶突破（累计75） | 75 | ~4.3天 |
| 5武将并行全突破 | 375 | ~21.5天 |

**结论**: 突破石经济设计合理。单武将突破节奏快（~4天），多武将并行时产生资源竞争（5武将~3周），符合"快速反馈+策略分配"的设计目标。

### 3.6 等级上限联动机制（v1.1 新增）

> ✅ **已修复**：连续3轮P0问题（R1→R2→R3），v1.1 彻底解决。

**核心规则**：每次突破成功 → 等级上限 +10（30→40→50→60→70）

**联动数据流**：

```
┌─────────────────────────────────────────────────────────┐
│                    突破 → 等级上限联动                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  玩家触发突破                                             │
│       │                                                 │
│       ▼                                                 │
│  HeroStarSystem.breakthrough(heroId)                    │
│       │                                                 │
│       ├─ 1. 校验条件：等级 ≥ 当前上限 && 资源充足           │
│       │                                                 │
│       ├─ 2. 扣除资源（碎片 + 铜钱 + 突破石）               │
│       │                                                 │
│       ├─ 3. breakthroughStages[heroId] += 1              │
│       │       │                                         │
│       │       ▼                                         │
│       │   getLevelCap(heroId) 自动返回新上限              │
│       │   ┌──────────────────────────────────┐          │
│       │   │ stage 0 → 30 (INITIAL_LEVEL_CAP) │          │
│       │   │ stage 1 → 40                      │          │
│       │   │ stage 2 → 50                      │          │
│       │   │ stage 3 → 60                      │          │
│       │   │ stage 4 → 70 (FINAL_LEVEL_CAP)    │          │
│       │   └──────────────────────────────────┘          │
│       │                                                 │
│       ├─ 4. 返回 BreakthroughResult                     │
│       │       { newLevelCap: 新上限, ... }               │
│       │                                                 │
│       ▼                                                 │
│  HeroLevelSystem 感知新上限                               │
│       │                                                 │
│       ├─ getMaxLevel(heroId)                            │
│       │   → 调用注入的 getLevelCap 回调                   │
│       │   → 返回 HeroStarSystem.getLevelCap(heroId)      │
│       │   → 自动获取最新突破后的等级上限                    │
│       │                                                 │
│       ├─ 升级操作：level < 新上限 → 允许继续升级            │
│       │                                                 │
│       └─ 一键强化：目标等级 cap = 新上限（非硬编码50）       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**回调注入链路**（`engine-hero-deps.ts`）：

| 注入目标 | 注入方法 | 回调来源 | 代码位置 |
|---------|---------|---------|:-------:|
| `HeroLevelSystem` | `setLevelDeps({ getLevelCap })` | `heroStar.getLevelCap()` | 第93行 |
| `HeroSystem` | `setLevelCapGetter(callback)` | `heroStar.getLevelCap()` | 第97行 |

**等级上限变化验证表**：

| 操作 | 突破阶段 | getLevelCap() 返回值 | 可升级范围 | 预期行为 |
|------|:-------:|:-------------------:|:---------:|---------|
| 初始状态 | 0 | 30 | 1~30 | 新武将上限30级 |
| 一阶突破后 | 1 | 40 | 1~40 | 可继续升级至40 |
| 二阶突破后 | 2 | 50 | 1~50 | 可继续升级至50 |
| 三阶突破后 | 3 | 60 | 1~60 | 可继续升级至60 |
| 四阶突破后 | 4 | 70 | 1~70 | 达到最终上限 |
| 未注入回调 | — | 50（fallback） | 1~50 | HERO_MAX_LEVEL 兜底 |

**边界条件**：
- 满突破后再调用 `getLevelCap()` → 返回 `FINAL_LEVEL_CAP = 70`，不再增加
- 突破失败（资源不足/等级未达上限）→ `breakthroughStages` 不变，等级上限不变
- `HERO_MAX_LEVEL = 50` 仅在未注入回调时生效（测试场景），生产环境始终使用动态获取

### 4.2 招贤令经济模型计算（4h在线基准）

#### 日产出模型

| 来源 | 计算公式 | 日产出 | 占比 |
|------|---------|:-----:|:---:|
| 被动产出（在线4h） | 0.002 × 14,400秒 | 29 | 14.5% |
| 离线收益（20h×50%，上限12h） | 0.002 × 43,200秒 × 50% | 43 | 21.5% |

| 关卡首通（日均5关） | 7.5 × 5 | 38 | 19.0% |
| **合计** | — | **~200** | — |

> **注**: 离线收益按 min(离线时长, 12h) × 50% 计算。20h离线实际按12h上限计算 = 0.002 × 43,200 × 50% = 43.2 ≈ 43。
