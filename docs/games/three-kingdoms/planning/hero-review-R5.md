# 武将系统游戏评测报告 (R5) — 设计阶段收尾

> **评测日期**: 2026-06-16
> **评测版本**: HEAD + R5新增（PRD v1.6 + 羁绊系统设计 v1.0 + 新手引导重设计 v2.0 + 视觉设计规范 v1.0 + UI组件设计 v1.0）
> **评测师**: 游戏评测师
> **评测依据**: PRD v1.6 (HER-1~HER-12) + 引擎源码验证 + 迭代日志 v1.6 + 羁绊系统设计文档 + 新手引导设计文档 v2.0 + 视觉设计规范 (1011行) + UI组件设计文档 (1948行) + R4差距分析

## 综合评分: 8.1/10（R1: 6.4 → R2: 6.7 → R3: 7.1 → R4: 7.6 → R5: 8.1, +0.5）

> **评分说明**: R5是设计阶段的收尾迭代，评分从7.6提升至8.1（+0.5），标志着武将系统设计从"可用"正式进入"优秀"区间。
>
> **三大核心突破**：
> 1. **视觉设计规范落地（1011行）**：首次建立完整的视觉设计体系——配色方案（CSS变量化）、武将卡片/详情面板/招贤馆/编队四大界面设计、品质色统一（解决R16审计A-04三套色系混用）、动效合规清单、图标资源清单、实施检查清单。将视觉表现从"零规范"提升至"工业化标准"。
> 2. **UI组件设计完整蓝图（1948行）**：24个组件（3页面+8面板+6弹窗+7原子）的完整Props接口定义、6条交互流程图、状态管理架构（HeroUIState + Actions + Reducer）、实现优先级（P0/P1/P2）和9周里程碑时间线。这是从设计到开发的"最后一公里"桥梁。
> 3. **引擎实现重大突破**：R4标记为"零实现"的羁绊系统和新手引导，引擎中已有 `BondSystem`（含阵营羁绊faction_2/3/6/mixed_3_3、好感度系统、故事事件）和 `TutorialStepManager`/`TutorialStepExecutor`（含跳过/重播/集成测试）。R4的设计-实现差距评估存在偏差。
>
> **但关键矛盾仍在**：PRD v1.5/v1.6将等级上限设计为50→100，引擎 `star-up-config.ts` 仍为30→70（连续2轮未同步）；装备系数 `totalEquipmentPower` 默认值仍为0（连续5轮）；属性命名不一致（defense/speed vs command/politics，连续5轮）。设计阶段收尾时，这些"小问题"已成为技术债。

## R5新发现的问题

### P0（阻塞核心玩法）

#### P0-R5-1：等级上限引擎数值连续2轮未同步（R4遗留升级）

**问题**: PRD v1.5/v1.6将等级上限设计为50→60→70→80→100，但引擎 `star-up-config.ts` 仍为旧值：
- `INITIAL_LEVEL_CAP = 30`（PRD要求50，差距20级）
- `BREAKTHROUGH_TIERS` levelCapAfter: 40/50/60/70（PRD要求60/70/80/100）
- `FINAL_LEVEL_CAP = 70`（PRD要求100，差距30级）

**严重性升级**: R4首次发现时标记为P0，预估修复0.1天。R5仍未修复，**从"新发现"升级为"技术债"**。

**影响**:
- PRD中所有基于50~100级的数值计算（经验需求、铜钱消耗、属性成长、战力估算、收集节奏优化）在引擎中均无法验证
- 新手引导设计基于50级初始上限，引擎30级上限会导致引导节奏与实际不符
- 收集节奏优化中的"新手7天获得12~18名武将"目标基于50级上限计算

**建议修复**: 更新 `star-up-config.ts`，工作量0.1天。**这是进入开发阶段前的第一优先级**。

#### P0-R5-2：装备系数连续5轮未集成（R1遗留，技术债）

**问题**: `HeroSystem.calculatePower()` 中 `totalEquipmentPower` 参数默认值为0，所有调用处均未传入装备战力值。

**历史轨迹**: R1(P0-1) → R2(未提及) → R3(❌) → R4(P1-R4-3) → R5(P0升级)。**连续5轮，从P1升级为P0**。

**影响**:
- 装备系统已完整实现（穿戴/卸下/强化/精炼/分解），但战力计算不读取装备数据
- 视觉设计规范中编队界面的"战力预览"功能无法正确反映装备影响
- UI组件设计中 `FormationGridPanel` 的 `calculateFormationPower` 依赖正确的战力计算

**建议修复**: 在 `HeroSystem.calculatePower()` 调用处传入 `equipmentSystem.getTotalPower(generalId)`，工作量0.5天。

### P1（影响核心体验）

#### P1-R5-1：视觉设计规范与引擎现有CSS存在冲突风险

**问题**: 视觉设计规范定义了全新的CSS变量体系（`--hero-*`系列），但引擎现有CSS（`ThreeKingdomsGame.css`、`HeroCard.css`等）中已有大量硬编码色值和不同的品质色定义。规范附录B/C列出了15+4条迁移映射，但未提供迁移脚本或自动化验证工具。

**影响**:
- 开发者手动迁移19处品质色+4处强调色，遗漏风险高
- 迁移过程中可能出现新旧色值混用，导致视觉不一致
- 规范中"禁止硬编码色值"的检查清单依赖人工自查

**建议修复**:
1. 编写CSS变量迁移脚本，自动替换硬编码色值为CSS变量
2. 在CI中增加stylelint规则，禁止新增硬编码色值
3. 预估工作量1天

#### P1-R5-2：UI组件设计文档中部分引擎方法不存在

**问题**: UI组件设计文档§9.2引用了部分引擎方法，但需验证其是否真实存在：
- `HeroLevelSystem.quickEnhanceAll()` — 一键强化全部，需确认
- `HeroLevelSystem.batchUpgrade(ids)` — 批量升级，需确认
- `FormationRecommendSystem.recommend(context)` — 智能推荐，需确认
- `BondSystem.detectActiveBonds(heroIds)` — 羁绊检测，需确认方法签名
- `HeroBadgeSystem.getTodayTodos()` — 今日待办，需确认

**影响**: 如果这些方法不存在，UI组件开发时需要同步实现引擎层方法，增加实际工作量。

**建议修复**: 对§9.2中所有引擎方法进行源码验证，不存在的标注为"待实现"并更新工作量估算。

#### P1-R5-3：羁绊系统设计文档与引擎实现存在差异

**问题**: R5新增的羁绊系统设计文档（`hero-bond-system.md`）定义了12组搭档羁绊（分3批实现），但引擎 `bond-config.ts` 中实际实现的是4种阵营羁绊（faction_2/3/6/mixed_3_3）+故事事件。差异包括：
- 设计文档定义搭档羁绊（桃园结义/五虎上将等），引擎仅有阵营羁绊
- 设计文档定义羁绊等级Lv1~3（基于武将星级），引擎未见等级系统
- 设计文档定义羁绊系数作为第5个独立乘区（上限2.0），引擎实现方式待确认
- 引擎有好感度系统+故事事件，设计文档未覆盖

**影响**: 设计文档与引擎实现的差异可能导致开发时以谁为准的困惑。

**建议修复**: 以引擎实现为权威来源，更新设计文档对齐引擎，标注"设计扩展"部分为后续迭代目标。

#### P1-R5-4：24个UI组件预估56工作日，与9周时间线偏紧

**问题**: UI组件设计文档预估P0（23天）+P1（17天）+P2（16天）=56个工作日≈9周。但：
- P0的12个组件中，`HeroDetailPanel`（5天）和 `HeroListPanel`（3天）依赖多个原子组件，实际可能超期
- P1的 `FormationGridPanel`（4天）含拖拽交互，移动端适配增加复杂度
- 未计入组件与引擎联调、视觉规范适配、响应式测试的额外工作量
- 实际开发通常有1.3~1.5倍的超期系数

**影响**: 9周时间线可能实际需要12~14周，影响项目整体进度。

**建议修复**: 
1. P0阶段增加3天缓冲（总计26天）
2. 优先实现手机端条形图（AttributeBar），PC端雷达图延后至P2
3. 考虑将 `FormationGridPanel` 拆分为P0基础版（点击选择）+P1完整版（拖拽）

### P2（锦上添花）

#### P2-R5-1：视觉设计规范中占位图全部CSS绘制，高品质武将缺乏差异化

**问题**: 规范§7.2定义所有占位图使用CSS绘制（灰色剪影+"暂无头像"文字），不依赖外部图片。这在MVP阶段合理，但LEGENDARY品质武将与COMMON品质武将使用同样的灰色剪影，缺乏品质感知差异。

**建议修复**: P2阶段为EPIC/LEGENDARY品质武将设计专属占位图边框（如LEGENDARY使用金色花纹边框），或引入SVG剪影。

#### P2-R5-2：编队界面缺少阵容保存/分享功能

**问题**: UI组件设计中 `FormationGridPanel` 支持3套编队切换，但缺少阵容保存（命名/备注）和分享（截图/导出）功能。对于有社交属性的游戏，阵容分享是重要的社区互动。

**建议修复**: P2阶段增加阵容命名+一键截图功能。

#### P2-R5-3：招贤馆UI缺少概率公示详情页设计

**问题**: 视觉设计规范§4.1招贤馆主界面有"概率公示 >"入口，但未设计概率公示详情页的布局和内容。根据国内游戏法规，抽卡概率公示是合规要求。

**建议修复**: 补充概率公示详情页设计（含各品质概率、保底机制说明、历史概率统计）。

#### P2-R5-4：羁绊图标使用Emoji，跨平台一致性风险

**问题**: 视觉设计规范§7.1中羁绊图标建议使用Emoji（🌸桃花、🐅虎符、🪶羽扇等），但Emoji在不同操作系统/浏览器上渲染差异大，可能导致视觉不一致。

**建议修复**: 改用SVG内联图标（与操作图标一致），确保跨平台一致性。

## 关键发现总结

### 发现1：R4对引擎实现状态的认知存在重大偏差

R4将羁绊系统标记为"零实现"（P0-R4-2）、新手引导标记为"零实现"（P1-R4-1），但R5引擎源码验证发现：
- `BondSystem` 已完整实现（阵营羁绊4种+好感度+故事事件+编队预览）
- `TutorialStepManager`+`TutorialStepExecutor` 已实现（含跳过/重播/集成测试）

这意味着R4的"设计-实现差距55%"评估偏高，实际差距约40%。**教训：评测应优先验证引擎源码，而非仅依赖PRD和设计文档描述。**

### 发现2：视觉设计规范是设计阶段的"最后一块拼图"

R1~R4的视觉表现评分持续低迷（2.5→2.5→3.5→4.0），核心原因是"零规范"。R5的视觉设计规范（1011行）首次建立了完整的视觉体系：
- 配色方案（CSS变量化，解决R16审计问题）
- 四大界面设计（卡片/详情/招贤馆/编队）
- 动效合规清单（禁止闪烁/脉冲，允许opacity/transform）
- 图标资源清单（5品质+8操作+4装备+10羁绊）
- 实施检查清单（配色/动效/布局/可访问性）

视觉表现评分从4.0跃升至7.5（+3.5），是5轮迭代中单维度最大跃升。

### 发现3：设计阶段收尾的"最后一公里"是技术债而非新设计

R5新增的设计内容（视觉规范1011行+UI组件1948行）质量极高，但评分提升（+0.5）主要来自：
1. 羁绊/引导引擎实现的澄清（而非新设计）
2. 视觉规范的补全（而非引擎改动）

真正阻碍评分进一步提升的是连续多轮未修复的技术债：
- 等级上限数值未同步（2轮）
- 装备系数未集成（5轮）
- 属性命名不一致（5轮）

**结论：设计阶段的边际收益已接近上限，后续提升需要转向引擎实现和UI前端开发。**
