## 五、详细分析

### 5.1 DDD分层违规 (P1-1)

`core/pvp/pvp.types.ts` 反向依赖 engine 层：

```typescript
// core/pvp/pvp.types.ts L10-11
import type { Faction } from '../../engine/hero/hero.types';           // ❌ core → engine
import type { BattleAction, BattleResult, BattleState } from '../../engine/battle/battle.types'; // ❌ core → engine
```

**正确做法**: 将 `Faction` 提升到 `core/types` 或在 pvp.types.ts 中定义 `ArenaFaction` 别名类型。将 `BattleAction/BattleResult/BattleState` 移至 `core/battle/` 或在 pvp.types.ts 中用更松耦合的类型替代。

### 5.2 存档缺失 (P1-2)

`engine-save.ts` 的 `SaveContext` 接口和序列化逻辑中完全缺失 v11 子系统：
- 无 `ArenaSystem` 存档
- 无 `FriendSystem` 存档
- 无 `ChatSystem` 存档
- 无 `ArenaShopSystem` 存档
- 无 `LeaderboardSystem` 存档

这意味着玩家关闭页面后，所有竞技场积分、段位、好友列表、聊天记录、竞技币等数据将丢失。

### 5.3 Tick循环缺失 (P1-3)

`engine-tick.ts` 中未集成 v11 子系统的定时逻辑：
- Arena 每日挑战次数/刷新次数重置（0:00）
- Friend 每日互动次数/友情点获取重置（0:00）
- Chat 过期消息清理（按频道保留时间）
- Arena 赛季天数推进和结算检查

### 5.4 排行榜系统重叠 (P1-5)

| 维度 | RankingSystem (pvp) | LeaderboardSystem (social) |
|------|---------------------|---------------------------|
| 数据源 | ArenaOpponent[] | 自管理 entries |
| 维度 | SCORE/POWER/SEASON (3种) | POWER/EXPEDITION/ARENA/WEALTH (4种) |
| 分页 | 无 | 有 (每页20) |
| 赛季 | 无 | 有 (30天周期) |
| 奖励 | 无 | 有 (前100名) |
| 持久化 | 有 serialize/deserialize | 有内部 state |

两个系统功能高度重叠但接口不兼容，应考虑合并或明确职责边界。
