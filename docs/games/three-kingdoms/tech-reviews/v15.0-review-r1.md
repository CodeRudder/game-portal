## 6. 架构问题分析

### 6.1 🔴 P0 — ChainEventSystem vs ChainEventEngine 职责重叠

**问题描述**: `ChainEventSystem`（v7.0）和 `ChainEventEngine`（v15.0）存在高度相似的 API：

| 方法 | ChainEventSystem | ChainEventEngine |
|------|-----------------|-----------------|
| `registerChain()` | ✅ | ✅ |
| `registerChains()` | ✅ | ✅ |
| `getChain()` | ✅ | ✅ |
| `startChain()` | ✅ | ✅ |
| `advanceChain()` | ✅ | ✅ |

两者都实现 `ISubsystem`，都管理连锁事件链，但使用不同的类型定义（`EventChainDef` vs `ChainEventDefV15`）。

**影响**: 
- 调用者需要知道用哪个系统
- 类型不兼容导致无法互换
- 未来维护成本翻倍

**建议**: 
- **ADR-15-001**: 将 `ChainEventSystem` 标记为 `@deprecated`，v15.0+ 统一使用 `ChainEventEngine`
- 在 `ChainEventEngine` 中增加兼容层，支持旧 `EventChainDef` 格式自动转换

### 6.2 🔴 P0 — EventTriggerSystem vs EventTriggerEngine 职责重叠

**问题描述**: 与链式事件类似，`EventTriggerSystem`（v7.0）和 `EventTriggerEngine`（v15.0）功能重叠。

`EventTriggerEngine` 是 v15.0 新增的增强版，支持概率公式 `P = clamp(base + Σ(additive) * Π(multiplicative), 0, 1)` 和 6 级通知优先级。

**建议**:
- **ADR-15-002**: 将 `EventTriggerSystem` 标记为 `@deprecated`
- 在 `index.ts` 中添加迁移指引注释

### 6.3 🟡 P1 — EventEngine 未编排子系统

**问题描述**: `EventEngine` 实现了 `ISubsystem`，但它是一个独立子系统而非编排器。它不引用 `ChainEventSystem`、`ChainEventEngine`、`EventTriggerSystem` 等子系统。

这意味着：
- 各子系统需要外部手动协调
- 事件生命周期分散在多个独立子系统中
- 缺少统一的"事件管理器"来编排所有事件相关子系统

**建议**: 
- 考虑让 `EventEngine` 作为 **Facade/Coordinator**，内部持有其他子系统的引用
- 或引入 `EventOrchestrator` 作为顶层编排器

### 6.4 🟡 P1 — 类型定义分散在 core 和 engine 两层

**问题描述**: 连锁事件类型存在三套：
1. `core/event/event-v15-chain.types.ts` — `ChainNodeDefV15`, `EventChainDefV15`
2. `engine/event/event-chain.types.ts` — `EventChain`, `EventChainNode`  
3. `engine/event/ChainEventSystem.ts` 内联类型 — `ChainId`, `ChainNodeDef`
4. `engine/event/ChainEventEngine.ts` 内联类型 — `ChainEngineId`, `ChainNodeDefV15`

**影响**: 类型名相似但不同，增加理解和维护成本。

**建议**: 
- 统一类型定义到 `core/event/` 层
- 引擎层通过 `import type` 引用，不再内联定义

### 6.5 🟢 P2 — OfflineEventHandler vs OfflineEventSystem 分层合理

`OfflineEventHandler`（284行，纯函数式处理器）和 `OfflineEventSystem`（451行，完整 ISubsystem）的分层是合理的：
- Handler 负责无状态的事件堆积模拟
- System 负持有状态、规则管理、序列化

**结论**: ✅ 设计合理，无需调整。
