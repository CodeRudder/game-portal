## 问题清单

### P0 — 阻塞性问题（必须修复）

#### P0-01: 离线收益系统未集成到主引擎
- **文件**: `src/games/three-kingdoms/engine/ThreeKingdomsEngine.ts`
- **描述**: `OfflineRewardSystem`、`OfflineSnapshotSystem`、`OfflineEstimateSystem` 三个核心子系统虽然已实现，但完全没有集成到 `ThreeKingdomsEngine` 主引擎中。主引擎没有引用这些类，没有在构造函数中创建实例，没有在 `registerSubsystems()` 中注册，没有在 `init/reset/save/load` 生命周期中调用。
- **影响**: 玩家下线/上线时，离线收益系统不会被触发，所有离线收益逻辑形同虚设。
- **对比**: `MailSystem` 已通过 `engine-r11-deps.ts` 正确集成（构造、注册、初始化、重置），离线收益系统应参照此模式集成。
- **修复建议**: 在 `engine-r11-deps.ts` 或新建 `engine-offline-deps.ts` 中创建离线收益系统集合，集成到主引擎的构造、注册、初始化、存档、重置流程中。

#### P0-02: UI组件全部缺失
- **文件**: 无（`src/games/three-kingdoms/ui/` 目录为空）
- **描述**: 以下5个UI组件均未创建：
  1. `OfflineRewardModalComponent` — 离线收益弹窗
  2. `OfflineSummaryComponent` — 回归综合面板
  3. `OfflineEstimateComponent` — 离线预估面板
  4. `MailPanelComponent` — 邮件面板
  5. `MailDetailComponent` — 邮件详情
- **影响**: 功能点5/7/11的UI层无法工作，邮件系统无可视界面。
- **修复建议**: 按DDD架构在UI层创建对应组件，使用 `data-testid` 属性，通过门面(engine/index.ts)引用引擎层。

#### P0-03: 邮件系统存在双重类型定义冲突
- **文件**: `src/games/three-kingdoms/engine/mail/MailSystem.ts` (行31~68) vs `src/games/three-kingdoms/engine/mail/mail.types.ts` (行30~67)
- **描述**: 两个文件各自定义了同名但结构不同的类型：
  - `MailCategory`: MailSystem.ts = `'system'|'battle'|'social'|'reward'`，mail.types.ts = `'system'|'reward'|'combat'|'trade'|'social'|'alliance'`
  - `MailStatus`: 两者值相同但来自不同文件
  - `MailAttachment`: MailSystem.ts = `{id, resourceType, amount, claimed}`，mail.types.ts = `{id, type:'resource'|'item'|'hero_fragment', content:Resources|Record<string,number>, claimed}`
  - `MailData`: MailSystem.ts 用 `content` 字段，mail.types.ts 用 `body` 字段；MailSystem.ts 无 `priority/starred`，mail.types.ts 有
  - `MailSaveData`: MailSystem.ts = `{mails, nextId, version}`，mail.types.ts = `{mails, lastCleanupTime, version}`
- **影响**: `mail/index.ts` 不得不使用 `MailCategoryV2/MailStatusV2/MailAttachmentV2/MailDataV2/MailSaveDataV2` 别名导出来缓解冲突。`MailTemplateSystem.createFromTemplate()` 返回的 `MailData`（来自 mail.types）与 `MailSystem` 使用的 `MailData`（本地定义）结构不兼容，无法直接传递。
- **修复建议**: 统一类型定义到一个源头（推荐 mail.types.ts），MailSystem.ts 导入使用，删除重复定义。

### P1 — 重要问题（建议修复）

#### P1-01: 衰减系数表档位数与PRD不一致
- **文件**: `src/games/three-kingdoms/engine/offline/offline-config.ts` (行24~38)
- **描述**: PRD要求6档衰减 `100%→80%→60%→40%→25%→15%` 覆盖0~72h+，实际实现只有5档 `100%→80%→60%→40%→25%` 覆盖0~72h，超过72h为0%而非15%。配置注释明确写 `>72h: 0% 无收益`。
- **影响**: 超过72h回归的玩家收益比PRD预期低（PRD第6档15% vs 实现0%）。
- **修复建议**: 如确认PRD中 `>72h: 15%` 是正确的，需新增第6档 `{id:'tier6', startHours:72, endHours:Infinity, efficiency:0.15}`；如确认封顶72h后0%是设计意图，则更新PRD文档。

#### P1-02: OfflineRewardSystem与OfflineRewardEngine职责重叠
- **文件**: `src/games/three-kingdoms/engine/offline/OfflineRewardSystem.ts` + `OfflineRewardEngine.ts`
- **描述**: 两个文件都实现了衰减快照计算、翻倍机制、系统修正、溢出处理等核心逻辑，代码高度重复。例如：
  - `OfflineRewardSystem.calculateSnapshot()` ≈ `OfflineRewardEngine.calculateOfflineSnapshot()`
  - `OfflineRewardSystem.applyDouble()` ≈ `OfflineRewardEngine.applyDouble()`
  - `OfflineRewardSystem.applyCapAndOverflow()` ≈ `OfflineRewardEngine.applyOverflowRules()`
  - `OfflineRewardSystem.applySystemModifier()` ≈ `OfflineRewardEngine.applySystemModifier()`
- **影响**: 维护成本高，修改一处容易遗漏另一处，导致行为不一致。
- **修复建议**: 明确职责划分——`OfflineRewardEngine` 作为纯计算层（无状态函数），`OfflineRewardSystem` 作为有状态聚合根（调用 Engine 函数 + 管理状态）。OfflineRewardSystem 应委托调用 OfflineRewardEngine 的函数，而非重复实现。

#### P1-03: OfflineSnapshotSystem未在offline/index.ts中导出
- **文件**: `src/games/three-kingdoms/engine/offline/index.ts`
- **描述**: `OfflineSnapshotSystem` 是一个完整的类（407行），但在 `offline/index.ts` 门面中没有导出。外部无法通过门面使用它。
- **影响**: 外部模块无法通过标准门面路径导入 `OfflineSnapshotSystem`。
- **修复建议**: 在 `offline/index.ts` 中添加 `export { OfflineSnapshotSystem } from './OfflineSnapshotSystem'`。

#### P1-04: OfflineRewardEngine和OfflineTradeAndBoost未在engine/index.ts中导出
- **文件**: `src/games/three-kingdoms/engine/index.ts`
- **描述**: `engine/index.ts` 只导出了 `OfflineRewardSystem` 和 `OfflineEstimateSystem` 两个类，以及纯函数（通过 offline/index.ts 的重导出）。但 `OfflineRewardEngine` 类中的 `calculateFullOfflineReward`、`OfflineRewardContext` 等虽然通过 offline/index.ts 导出了，engine/index.ts 中缺少部分导出（如 `OfflineRewardEngine` 类本身、`OfflinePanelHelper`、`OfflineTradeAndBoost`）。
- **影响**: UI层如需直接使用 `calculateFullOfflineReward` 等函数，需要绕过门面直接引用子模块。
- **修复建议**: 在 engine/index.ts 中补全导出，或确认 offline/index.ts 的重导出已足够（当前已导出纯函数，类本身不需要导出则可忽略）。

#### P1-05: engine/index.ts 超过500行限制
- **文件**: `src/games/three-kingdoms/engine/index.ts` (616行)
- **描述**: 门面导出文件超过500行限制（616行）。
- **影响**: 文件过长，维护困难。
- **修复建议**: 将 v9.0 的 offline/mail 导出拆到 `exports-v9.ts`，在 index.ts 中通过 `export * from './exports-v9'` 引入（与 v6/v10/v11/v13/v17 模式一致）。

#### P1-06: MailSystem未实现ISubsystem接口
- **文件**: `src/games/three-kingdoms/engine/mail/MailSystem.ts`
- **描述**: `MailSystem` 类没有实现 `ISubsystem` 接口（无 `init/reset` 标准方法签名）。虽然 `engine-r11-deps.ts` 中手动调用了 `mailSystem.reset()`，但不符合 ISubsystem 规范。
- **影响**: 与其他子系统（如 ResourceSystem、BuildingSystem）的标准化接口不一致。
- **修复建议**: 让 `MailSystem` 实现 `ISubsystem` 接口，提供标准的 `init(deps)` 和 `reset()` 方法。

### P2 — 建议改进

#### P2-01: OfflineRewardSystem也未实现ISubsystem接口
- **文件**: `src/games/three-kingdoms/engine/offline/OfflineRewardSystem.ts`
- **描述**: 同 P1-06，`OfflineRewardSystem` 没有 `ISubsystem` 接口。虽然有 `reset()` 方法，但缺少 `init(deps)` 标准方法。
- **修复建议**: 实现 `ISubsystem` 接口。

#### P2-02: MailTemplateSystem 中 combat 模板使用 'combat' 类别
- **文件**: `src/games/three-kingdoms/engine/mail/MailTemplateSystem.ts` (行66)
- **描述**: 战报模板使用 `category: 'combat'`，但 `MailSystem.ts` 中 `MailCategory` 类型定义为 `'battle'`（不是 `'combat'`）。虽然 `mail.types.ts` 中有 `'combat'`，但由于双重类型问题（P0-03），如果 MailSystem 使用自己的类型定义，则 `'combat'` 不被接受。
- **修复建议**: 统一邮件类别枚举值，消除 `'battle'` vs `'combat'` 的不一致。

#### P2-03: OfflinePanelHelper.estimateOfflineReward 有未使用参数
- **文件**: `src/games/three-kingdoms/engine/offline/OfflinePanelHelper.ts` (行145)
- **描述**: `estimateOfflineReward` 函数的第4个参数 `_calculateSnapshot` 标记为未使用，注释写"向后兼容参数位"。这个参数在 `OfflineRewardEngine.estimateOfflineReward` 中被传入但从未调用。
- **修复建议**: 移除未使用参数，或实际使用它来避免重复计算逻辑。

#### P2-04: 邮件持久化缺少 IndexedDB 备份
- **文件**: `src/games/three-kingdoms/engine/mail/MailPersistence.ts`
- **描述**: 邮件系统只使用 localStorage 持久化。PRD风险点提到"浏览器清理localStorage导致数据丢失"，建议"多重存储：localStorage + IndexedDB"。
- **修复建议**: 评估是否需要增加 IndexedDB 备份机制（可作为后续迭代）。

#### P2-05: 离线收益计算缺少分段优化
- **文件**: `src/games/three-kingdoms/engine/offline/OfflineRewardEngine.ts`
- **描述**: PRD风险点提到"长时间离线后计算耗时"，建议"分段计算，预缓存中间结果"。当前实现是一次性遍历所有衰减档位，对于72h离线计算量不大，但如果未来扩展可能需要优化。
- **修复建议**: 当前实现可接受，标记为未来优化项。
