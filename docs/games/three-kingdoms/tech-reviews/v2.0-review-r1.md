### 引擎层架构评价

**DDD 四层架构合规性：⭐⭐⭐⭐ 良好**
- UI层 → 引擎层 → 子系统领域层 → 配置层 的依赖方向严格单向
- 6个子系统中有5个实现了 `ISubsystem` 接口（HeroFormation 例外）
- 子系统之间通过依赖注入（`RecruitDeps` / `LevelDeps` / `StarSystemDeps`）解耦，不直接互相引用
- 配置层文件（`hero-config.ts` / `hero-recruit-config.ts` / `star-up-config.ts`）严格零逻辑，仅含常量和数据结构

**单一职责原则：⭐⭐⭐⭐ 良好**
- `HeroSystem`：武将状态 + 碎片 + 战力（聚合根）
- `HeroRecruitSystem`：招募流程 + 保底 + 历史（聚合根）
- `HeroLevelSystem`：升级 + 一键强化 + 批量强化（聚合根）
- `HeroStarSystem`：升星 + 突破 + 碎片进度（聚合根）
- `HeroFormation`：编队管理（独立 class）
- `BondSystem`：羁绊检测 + 加成计算（聚合根，但有两个版本）
- `HeroSerializer`：序列化（辅助模块）

**资源解耦策略：⭐⭐⭐⭐⭐ 优秀**
- 所有子系统通过回调函数（`ResourceSpendFn` / `ResourceCheckFn`）解耦资源系统
- 不直接依赖 `ResourceSystem`，符合 DDD 域间解耦原则
- `engine-hero-deps.ts` 提供安全资源操作封装（类型守卫 + try/catch）

   - 影响范围：仅CSS文件，无逻辑变更

2. **BondSystem 重复实现整合**
   - 当前存在 `engine/hero/BondSystem.ts`（基于 `core/heritage`）和 `engine/bond/BondSystem.ts`（基于 `core/bond`）两个版本
   - 建议统一为一个版本，推荐保留 `engine/bond/BondSystem.ts`（功能更完整，含好感度+故事事件）
   - 删除 `engine/hero/BondSystem.ts` 或将其改为对 `engine/bond/BondSystem.ts` 的重导出
   - 在 `ThreeKingdomsEngine` 中注册 BondSystem 并通过门面暴露 API

3. **HeroFormation 实现 ISubsystem**
   - 添加 `name`、`init()`、`update()`、`getState()`、`reset()` 方法
   - 使其可注册到引擎子系统注册表
   - 当前已有 `reset()` 方法，仅需补充其余4个

4. **FormationPanel 集成羁绊展示**
   - 在编队面板中添加羁绊激活状态展示区域
   - 调用 `BondSystem.detectBonds()` 或 `BondSystem.getFormationPreview()` 获取羁绊数据
   - 显示激活羁绊名称 + 属性加成 + 潜在羁绊提示

5. **门面补全导出**
   - 在 `engine/index.ts` 中补导出 `HeroLevelSystem`、`HeroStarSystem`、`BondSystem` 三个类
   - 补导出 `GENERAL_DEFS`、`QUALITY_MULTIPLIERS`、`POWER_WEIGHTS` 等常用常量

6. **HeroLevelSystem 拆分（预防性）**
   - 将批量强化逻辑（`batchUpgrade` / `quickEnhanceAll` / `getBatchEnhancePreview`）拆分到 `HeroBatchEnhance.ts`
   - 当前 477行尚在限制内，但余量仅23行
