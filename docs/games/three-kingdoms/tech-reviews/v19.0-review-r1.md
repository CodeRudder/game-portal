## 五、问题清单

### P1-1: `as any` 用于 navigator.deviceMemory
- **文件**: `GraphicsQualityManager.ts:222`
- **代码**: `this.deviceMemoryGB = (navigator as any)?.deviceMemory ?? 4;`
- **风险**: 绕过类型检查，若 navigator 上不存在 deviceMemory 属性不会报错
- **建议**: 定义 `INavigatorDeviceMemory` 接口，或使用类型守卫：
  ```typescript
  const nav = navigator as unknown as { deviceMemory?: number };
  this.deviceMemoryGB = nav?.deviceMemory ?? 4;
  ```

### P1-2: `calcRebirthMultiplier` 函数签名冲突
- **文件**: `engine/unification/BalanceCalculator.ts` vs `engine/prestige/RebirthSystem.ts`
- **详情**: 两个模块均导出同名函数 `calcRebirthMultiplier`，但签名不同：
  - unification 版: `calcRebirthMultiplier(count: number, config: RebirthBalanceConfig): number`
  - prestige 版: `calcRebirthMultiplier(count: number): number`（使用硬编码配置）
- **风险**: 消费者可能导入错误版本；unification/index.ts 重导出了 unification 版本，prestige/index.ts 重导出了 prestige 版本，但两个同名导出在同时使用时会产生冲突
- **建议**: 重命名其中一个（如 prestige 版改为 `calcRebirthMultiplierPrestige`），或让 prestige 版委托 unification 版

### P2-1: 6个文件超过400行阈值
- **文件**: PerformanceMonitor(471), BalanceValidator(442), IntegrationValidator(427), InteractionAuditor(422), BalanceReport(393), GraphicsQualityManager(359)
- **风险**: 单文件职责过多，维护成本上升
- **建议**: PerformanceMonitor 已拆分 ObjectPool/DirtyRectManager，可进一步拆分 FPS/Memory/Loading 为独立策略类；BalanceValidator 可按维度拆分

### P2-2: index.ts 重导出 settings 模块内容
- **文件**: `unification/index.ts`
- **详情**: 从 `../settings/` 重导出 SettingsManager、AnimationController、CloudSaveSystem、AccountSystem
- **风险**: 增加模块间耦合，消费者可能不清楚实际来源
- **建议**: 在重导出注释中已说明原因（向后兼容），可接受。未来版本应标记 `@deprecated` 并引导消费者直接从 settings 导入

### P2-3: BalanceReport.ts 未实现 ISubsystem
- **文件**: `BalanceReport.ts`（393行）
- **详情**: 作为纯函数模块导出 `validateSingleResource`/`validateSingleHero` 等，不实现 ISubsystem
- **风险**: 393行的纯函数模块可能包含过多职责
- **建议**: 可拆分为 ResourceReporter/HeroReporter/BattleReporter 等
