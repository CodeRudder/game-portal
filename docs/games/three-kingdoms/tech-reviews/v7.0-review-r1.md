## 四、问题清单

### P0 — 必须修复

#### P0-1: PvP 存档未集成到引擎存档系统

**严重程度**: 🔴 P0 — 数据丢失风险
**位置**: `engine-save.ts`, `shared/types.ts` (GameSaveData)

**问题描述**:
- `GameSaveData` 接口中没有 PvP 相关字段（arena/pvp/ranking 等）
- `buildSaveData()` 不序列化任何 PvP 子系统
- `applySaveData()` 不反序列化任何 PvP 子系统
- 玩家关闭游戏后，所有竞技场积分、段位、排名、商店购买记录将丢失

**影响范围**: ArenaSystem, PvPBattleSystem, ArenaSeasonSystem, RankingSystem, ArenaShopSystem 的所有持久化数据

**修复建议**:
1. 在 `GameSaveData` 中添加 PvP 字段：
   ```typescript
   pvp?: {
     arena: ArenaSaveData;
     season: { season: SeasonData; highestRankId: string };
     ranking: RankingSaveData;
     shop: ArenaShopSaveData;
   };
   ```
2. 在 `buildSaveData()` 中序列化 PvP 数据
3. 在 `applySaveData()` 中反序列化 PvP 数据
4. 在 `SaveContext` 中添加 PvP 子系统引用

---

### P1 — 应当修复

#### P1-1: RankingSystem.name 命名不一致

**位置**: `engine/pvp/RankingSystem.ts:87`

**问题**: `readonly name = 'PvpRankingSystem'` 使用了 `Pvp` 前缀，但其他 PvP 子系统均使用无前缀命名（如 `ArenaSystem`、`ArenaShopSystem`）。在注册表中查找时可能导致混淆。

**修复建议**: 统一为 `'RankingSystem'` 或使用完整前缀 `'PvpRankingSystem'`，与其他系统保持一致。

#### P1-2: ArenaSeasonSystem 包含商店购买逻辑

**位置**: `engine/pvp/ArenaSeasonSystem.ts:196-203`

**问题**: `buyArenaShopItem()` 方法放在 ArenaSeasonSystem 中，但竞技商店的购买逻辑应该由 ArenaShopSystem 统一管理。当前设计导致货币扣除分散在两个系统中。

**修复建议**: 将 `buyArenaShopItem()` 移至 ArenaShopSystem，或改为通过 ArenaShopSystem 的 `buyItem()` 方法处理竞技币消耗。

---

### P2 — 建议优化

#### P2-1: ArenaSystem.calculatePower() 战力估算过于简化

**位置**: `engine/pvp/ArenaSystem.ts:310-313`

**问题**: 战力计算使用 `score * 10 + heroCount * 1000 + 5000` 硬编码公式，与实际武将战力无关。注释中标注"简化版"，但匹配算法依赖此值。

**建议**: 后续版本接入 HeroSystem 获取真实战力。

#### P2-2: PvPBattleSystem.executeBattle() 使用 Math.random()

**位置**: `engine/pvp/PvPBattleSystem.ts:193-209`

**问题**: 战斗结果直接使用 `Math.random()`，不可预测也不可回放。对于竞技系统，建议使用确定性随机（seeded RNG），以便回放验证。

**建议**: 引入 seeded PRNG，将 seed 记录在 BattleReplay 中。

#### P2-3: DefenseFormationSystem 无内部状态但实现 ISubsystem

**位置**: `engine/pvp/DefenseFormationSystem.ts`

**问题**: DefenseFormationSystem 是纯工具类，`getState()` 返回 `{}`，`update()` 和 `reset()` 均为空操作。作为 ISubsystem 注册到引擎中增加了不必要的开销。

**建议**: 可保留当前设计（统一接口），但在注册时标记为无状态工具类，避免 update() 空调用。

#### P2-4: PvP 子系统间存在功能重叠

**问题**:
- ArenaSystem 和 DefenseFormationSystem 都有 `addDefenseLog()` / `getDefenseStats()` 方法
- ArenaSystem 和 PvPBattleSystem 都有 `calculatePower()` / `estimatePower()` 方法（逻辑相同）

**建议**: 防守日志统一由 DefenseFormationSystem 管理，战力估算统一由 PvPBattleSystem 提供。
