## 四、问题清单

### P1 — 应当修复

#### P1-1: trade/index.ts 门面包含无关的 core/map 重新导出

**严重程度**: 🟡 P1 — 门面职责污染
**位置**: `engine/trade/index.ts:71-82`

**问题描述**:
门面文件末尾从 `../../core/map` 重新导出了大量地图相关内容：
```typescript
export {
  MAP_SIZE, GRID_CONFIG, VIEWPORT_CONFIG, MAP_PIXEL_SIZE,
  REGION_IDS, REGION_DEFS, REGION_LABELS, REGION_COLORS,
  TERRAIN_TYPES, TERRAIN_DEFS, TERRAIN_LABELS, TERRAIN_COLORS,
  DEFAULT_LANDMARKS, LANDMARK_POSITIONS, MAP_SAVE_VERSION,
  getRegionAtPosition, getTerrainAtPosition, generateAllTiles,
} from '../../core/map';
```

这违反了门面单一职责原则：
- 贸易门面不应导出地图相关内容
- 消费方可能误以为这些是贸易相关的类型
- 增加了门面文件的耦合度

**修复建议**: 删除 `trade/index.ts` 中的 core/map 重新导出。地图相关内容应由 `core/map/index.ts` 或专门的地图门面导出。

#### P1-2: 子系统 name 属性风格不一致

**位置**:
- `TradeSystem.ts:54` → `readonly name = 'trade' as const;`
- `CaravanSystem.ts:84` → `readonly name = 'caravan' as const;`

**问题**: 
1. 使用小写名称（`'trade'`、`'caravan'`），而其他所有子系统使用 PascalCase（如 `'ArenaSystem'`、`'BuildingSystem'`）
2. 使用 `as const` 断言，而其他系统不使用

**修复建议**: 统一为 PascalCase 命名：
- `'trade'` → `'TradeSystem'`
- `'caravan'` → `'CaravanSystem'`
- 移除 `as const` 断言

---

### P2 — 建议优化

#### P2-1: TradeSystem 构造函数中调用 Date.now()

**位置**: `engine/trade/TradeSystem.ts:68`

**问题**: 构造函数中 `this.initData(Date.now())` 直接调用 `Date.now()`，导致：
- 单元测试中时间不可控
- 与引擎的 `init(deps)` 生命周期不一致

**建议**: 将数据初始化延迟到 `init()` 方法中，使用 `deps` 提供的时间。

#### P2-2: CaravanSystem.update() 中使用 Date.now()

**位置**: `engine/trade/CaravanSystem.ts:119`

**问题**: `update()` 方法中直接使用 `Date.now()` 而非传入的 `dt` 参数。这导致：
- 无法在测试中精确控制时间
- 与引擎帧驱动的 update(dt) 模式不一致

**建议**: 维护内部累计时间 `elapsedTime += dt`，或在 `init()` 时记录基准时间。

#### P2-3: TradeSystem + CaravanSystem 序列化版本使用 trade-config 常量

**位置**: 
- `TradeSystem.ts:263` → `version: TRADE_SAVE_VERSION`
- `CaravanSystem.ts:366` → `version: TRADE_SAVE_VERSION`

**问题**: 两个子系统共用同一个 `TRADE_SAVE_VERSION`，但它们的序列化格式可能独立变化。如果只修改其中一个的格式，版本号会同时影响两者。

**建议**: 为 TradeSystem 和 CaravanSystem 分别定义独立的存档版本号。

## 七、架构亮点

1. **聚合根模式**: TradeSystem 作为贸易域聚合根，统一管理商路/价格/繁荣度/事件/NPC商人
2. **回调解耦**: `TradeCurrencyOps` 和 `RouteInfoProvider` 两个回调接口，完美避免循环依赖
3. **纯函数抽取**: `trade-helpers.ts` 将可测试的纯函数独立出来，职责清晰
4. **配置分离**: `core/trade/trade-config.ts` 集中管理常量，便于调整数值平衡
5. **护卫互斥**: `guardAssignments` Map 实现武将互斥检查，简洁高效
6. **状态机设计**: CaravanSystem 的 traveling→trading→returning→idle 状态转换清晰

---

## 八、结论

v8.0 贸易系统引擎实现完整，架构设计优秀。聚合根模式 + 回调解耦 + 纯函数抽取的三层架构值得推广。**主要问题是门面文件包含无关的 core/map 重新导出**（P1），以及子系统 name 属性风格不一致（P1）。修复后即可通过审查。
