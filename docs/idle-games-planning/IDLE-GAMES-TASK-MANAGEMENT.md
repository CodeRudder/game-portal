## 二、单款游戏开发流水线

每款游戏按 **策划 → 美术 → 开发 → 测试** 四步拆分，使用对应专业 Agent：

### 2.1 流水线阶段

| 阶段 | Agent | 职责 | 产出文件 |
|------|-------|------|----------|
| 📋 策划 | `game-designer` | 游戏概念设计、玩法机制、数值平衡、关卡设计 | `constants.ts` (数值/资源/升级配置) |
| 🎨 美术 | `game-artist` | 视觉风格、配色方案、UI/UX、动画参数 | 渲染方案 (Canvas绘制逻辑+视觉规范) |
| 💻 开发 | `game-developer` | HTML5游戏引擎实现、渲染、物理、状态管理 | `XxxEngine.ts` + `constants.ts` (最终版) |
| 🧪 测试 | `tester` | 测试用例设计、质量保障、自动化测试 | `XxxEngine.test.ts` (120-190用例) |

### 2.2 流水线执行顺序

```
game-designer (策划)  ──→  game-artist (美术)  ──→  game-developer (开发)  ──→  tester (测试)
      │                       │                        │                        │
      ▼                       ▼                        ▼                        ▼
  数值配置                  视觉规范                 游戏引擎                  测试用例
  constants.ts             渲染参数                 XxxEngine.ts             test文件
  (初版)                   (融入引擎)               (含最终constants)         (120-190个)
```

### 2.3 子任务合并策略

由于存在 **3个子任务并发上限**，单款游戏的4个阶段可灵活合并：

```
方案A — 策划+美术合并，开发+测试合并（推荐）:
  子任务1: game-designer (策划+美术，输出设计文档+视觉规范)
  子任务2: game-developer (开发，基于策划文档实现引擎)
  子任务3: tester (测试，编写并运行测试用例)

方案B — 策划独立，开发合并美术+开发，测试独立:
  子任务1: game-designer (策划，输出详细设计文档)
  子任务2: game-developer (美术+开发，含渲染逻辑)
  子任务3: tester (测试)

方案C — 每次开发2款游戏时（并发3个子任务）:
  子任务1: 游戏A — game-developer (策划+美术+开发，一体化)
  子任务2: 游戏B — game-developer (策划+美术+开发，一体化)
  子任务3: tester (游戏A+游戏B 联合测试)
```

## 三、Agent 角色定义

### 3.1 game-designer（游戏策划）

**职责**: 游戏概念设计、玩法机制、数值平衡、关卡设计

**输入**: 游戏名称、题材来源、核心玩法描述（来自 IDLE-GAMES-ANALYSIS.md）

**输出**:
- 资源类型定义（名称、基础产出、上限）
- 升级树定义（名称、费用公式、效果、前置条件）
- Prestige 系统设计（转生条件、永久加成）
- 成就列表（条件、奖励）
- 数值平衡表（前期/中期/后期数值曲线）
- 离线收益参数（效率、最大时间）

### 3.2 game-artist（游戏主美）

**职责**: 视觉风格、配色方案、UI/UX 和动画设计

**输入**: 游戏题材、画风要求（来自 IDLE-GAMES-ANALYSIS.md 的画风列）

**输出**:
- 配色方案（主色/辅色/强调色/背景色，HEX值）
- UI布局规范（资源栏/升级面板/游戏区/设置面板位置）
- 动画参数（帧率/粒子密度/动画时长）
- Canvas渲染方案（绘制顺序/层级/特效）
- 程序化美术生成参数

### 3.3 game-developer（游戏主程序）

**职责**: HTML5 游戏引擎实现、渲染、物理和状态管理

**输入**: 策划文档 + 美术规范 + IdleGameEngine 基类

**输出**:
- `XxxEngine.ts` — 游戏引擎（继承 IdleGameEngine）
- `constants.ts` — 常量定义（资源/升级/成就配置）
- 游戏页面组件（如需要）

### 3.4 tester（QA 测试工程师）

**职责**: 测试用例设计、质量保障、自动化测试

**输入**: 游戏引擎代码

**输出**:
- `XxxEngine.test.ts` — 120-190 个测试用例
- 测试报告（通过率、覆盖率）

## 四、任务计划管理规范

### 4.1 核心原则

1. **每次开发 1~2 款游戏**，按 Phase 0 → Batch 1 → Batch 2 → ... 顺序推进
2. **更新 todos 前必须先读取**，确保不丢失任务计划
3. **每款游戏按策划→美术→开发→测试流水线**执行
4. **每批次完成后**运行全量测试 + 构建验证 + git commit

### 4.2 Todos 更新规范

```python
# ✅ 正确的更新流程（先读取再更新）
todos = read_current_todos()        # 1. 先读取当前 todos
write_todos(updates=[                # 2. 只更新状态，不替换整个列表
    {"index": 0, "status": "completed"},
    {"index": 1, "status": "in_progress"}
])

# ❌ 错误的做法
write_todos(todos=[...])             # 用 todos= 会替换整个列表，可能丢失任务
```

### 4.3 每次会话的工作流

```
1. 读取当前 todos → 确认进度
2. 选择下一项 pending 任务 → 标记为 in_progress
3. 启动子任务（策划→美术→开发→测试）
4. 等待子任务完成 → 检查产出
5. 标记当前任务 completed → 下一个任务标记 in_progress
6. 每批次5款完成后 → 全量测试 + 构建 + git commit + push
```

### 4.4 子任务 Prompt 模板

#### 策划子任务模板

```
你是游戏策划专家。请为放置游戏「{游戏名}」设计完整的游戏系统。

## 题材
- 来源: {IP来源}
- 核心玩法: {核心玩法描述}
- 画风: {画风}

## 要求
1. 定义所有资源类型（名称/基础产出/上限）
2. 设计升级树（至少20个升级项，含费用公式和效果）
3. 设计Prestige系统（转生条件/永久加成/多层Prestige）
4. 设计成就系统（至少15个成就）
5. 数值平衡表（前期0-1h/中期1-24h/后期1-7d/大后期7d+）
6. 离线收益参数

## 输出
将完整设计写入 src/engines/{game-id}/constants.ts
```

4. 运行测试确保全部通过
