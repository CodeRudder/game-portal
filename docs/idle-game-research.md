## 1. 经典放置游戏深度分析

### 1.1 Cookie Clicker — 增量游戏的鼻祖

**核心循环：**  
点击大饼干 → 获得饼干 → 购买建筑（光标、奶奶、农场、矿场等）→ 自动产出饼干 → 购买更多/更高级建筑 → 循环加速

**升级系统层次：**
| 层次 | 机制 | 说明 |
|------|------|------|
| 基础建筑 | 15种自动产出建筑 | 每种建筑价格指数增长（×1.15），产出线性增长 |
| 升级商店 | 200+种升级 | 解锁条件绑定建筑数量（如10/25/50/100个同种建筑） |
| 声望系统 | 天堂碎片（Prestige） | 重置所有进度，按总饼干数获得永久加成 |
| 糖果系统 | 糖果块 | 长期积累的稀有资源，用于全局升级 |
| 迷你游戏 | 花园、万神殿、法术书 | 通过特定建筑解锁的子系统，增加策略深度 |
| 皱皮怪 | Wrinklers | 吞食饼干但被击杀时返还更多，形成"延迟收益"策略 |

**设计亮点：**
- **指数增长的价格体系**：建筑价格 = 基础价格 × 1.15^数量，确保玩家始终有追求
- **多层次的声望货币**：天堂碎片 + 糖果块，形成双层长期进度
- **隐藏内容和彩蛋**：黄金饼干随机事件、圣诞老人事件等，增加惊喜感
- **"数字膨胀"的满足感**：从个位数到10^100+，数字增长本身就是奖励

### 1.2 Adventure Capitalist — 经理人自动化

**核心循环：**  
投资企业 → 等待收益 → 再投资 → 雇佣经理人实现自动化 → 积累天使投资人 → 声望重置

**经理人机制（核心创新）：**
| 经理人类型 | 功能 | 设计意义 |
|------------|------|----------|
| 普通经理 | 自动点击企业 | 解放玩家双手，实现"放置"核心体验 |
| 会计经理 | 提升企业效率x3 | 加速产出，形成升级追求 |
| 折扣经理 | 降低升级成本 | 优化购买策略，增加决策深度 |

**声望系统 — 天使投资人：**
- 声望货币 = 150 × √(总收益 / 10^15)
- 天使投资人提供全局收益加成（每个+2%）
- 玩家需权衡：何时重置？积累更多天使 vs 继续当前进度
- 多个星球（地球/月球/火星）= 多条独立进度线

**设计启示：**
- **自动化是放置游戏的灵魂**：经理人让游戏从"点击器"变成"放置器"
- **声望决策点**：重置时机选择是核心策略维度
- **多区域并行**：不同星球提供不同节奏和策略

### 1.3 Realm Grinder — 阵营选择系统

**阵营系统（核心差异化机制）：**
| 阵营类型 | 代表阵营 | 特色 | 解锁条件 |
|----------|----------|------|----------|
| 善良阵营 | 精灵、天使 | 高频低额收益，适合活跃玩家 | 初始可选 |
| 邪恶阵营 | 哥布林、恶魔 | 低频高额收益，适合放置玩家 | 初始可选 |
| 中立阵营 | 矮人、亡灵 | 平衡型，需要特殊条件解锁 | R3+ |
| 精英阵营 | 神圣秩序、腐化等 | 高阶策略，需要大量声望 | 高R数 |
| 雇佣兵 | 自定义搭配 | 终极灵活性 | 特殊解锁 |

**多层声望系统：**
- **Abdication（退位）**：软重置，获得阵营硬币
- **Reincarnation（转世）**：硬重置，获得转世点数，解锁新机制
- **Ascension（飞升）**：更高层重置，解锁精英阵营

**设计启示：**
- **阵营选择 = 玩法风格选择**：善良/邪恶对应活跃/放置，尊重不同玩家偏好
- **多层声望**：每层重置解锁新内容，形成"洋葱式"深度
- **Build多样性**：不同阵营+血契组合形成数百种有效Build

### 1.4 Kittens Game — 资源链与科技树

**核心特色：** 没有指数增长，所有增长都是线性的，通过科技解锁新资源链来推进

**资源链设计：**
```
猫薄荷 → 木材 → 矿物 → 铁 → 钢板 → 合金
                                    ↓
                              钛 → 钛合金
                                    ↓
                              硅 → 电子元件
```

**科技树结构：**
- 农业 → 建筑 → 工业 → 空间 → 核能 → 量子
- 每个科技解锁新建筑/新资源/新机制
- 科技之间有前置依赖关系，形成分支选择

**Tick系统：**
- 每秒5个Tick，每个Tick = 200ms
- 每天由10个Tick组成（1天 = 2秒）
- 所有事件基于Tick队列处理

**设计启示：**
- **线性增长也可以很深**：通过资源链转换和科技解锁代替数字膨胀
- **资源瓶颈设计**：不同阶段有不同的稀缺资源，驱动玩家策略调整
- **"科技解锁"比"数值增长"更有满足感**

### 1.5 Trimps — 自动化与战斗系统

**核心循环：**  
分配小怪物（Trimps）工作 → 采集资源 → 建造 → 训练军队 → 探索区域 → 战斗 → 解锁新区域

**自动化系统：**
| 自动化类型 | 功能 | 解锁时机 |
|------------|------|----------|
| AutoJobs | 自动分配工作 | 中期 |
| AutoStructure | 自动建造 | 中期 |
| AutoBattle | 自动战斗 | 中后期 |
| AutoUpgrade | 自动升级 | 后期 |
| Geneticists | 自动管理Trimps人口 | 后期 |

**战斗系统：**
- 区域制：每个区域有敌人，击败后前进
- BOSS区域：每10/20个区域一个BOSS，需要策略调整
- 挑战系统：不同挑战限制条件（如速度挑战、生命挑战等）
- 多层声望：Portal（传送门）重置获得氦元素，用于永久升级

**设计启示：**
- **渐进式自动化**：玩家先手动理解机制，再逐步自动化
- **挑战系统**：自我限制 = 新玩法，极大增加内容量
- **"管理"比"操作"更有深度**：Trimps分配是核心策略

### 1.6 NGU Idle — 多进度条系统

**核心创新：** 将所有进度可视化为进度条（Bars），每个系统都有独立的进度条

**多进度条系统：**
| 进度条类型 | 说明 | 能量分配 |
|------------|------|----------|
| NGU（Numbers Go Up） | 基础属性提升 | 攻击/魔法能量 |
| Advanced Training | 高级训练 | 攻击能量 |
| Augmentation | 增强装备 | 魔法能量 |
| Time Machine | 时间机器 | 魔法能量 |
| Beards | 胡子（被动增益） | 自动运行 |
| Res3（Wandoos） | 操作系统模拟 | 攻击+魔法 |

**核心机制 — 能量分配：**
- 玩家有攻击能量和魔法能量两种有限资源
- 所有进度条竞争同一能量池
- 玩家需决定：集中力量加速一个，还是分散推进多个
- 形成深层的"资源分配"策略

**设计启示：**
- **进度条可视化**：让玩家直观看到"正在进步"
- **资源竞争**：有限能量 vs 无限需求 = 策略决策
- **多系统并行**：不同系统互相独立但互相增益

### 1.7 Melvor Idle — RuneScape简化版

**核心定位：** 将RuneScape的核心体验提炼为纯放置形式

**技能系统（20+技能）：**
| 类别 | 技能 | 互相依赖 |
|------|------|----------|
| 战斗技能 | 攻击、力量、防御、生命、魔法、远程 | 装备需求 |
| 采集技能 | 伐木、采矿、钓鱼、种田 | 原料供给 |
| 加工技能 | 锻造、烹饪、工艺、符文制作、草药 | 原料消耗 |
| 辅助技能 | 敏捷、偷窃、占星、魔法 | 全局增益 |

**战斗系统：**
- 自动战斗，基于回合制
- 3种战斗风格：近战/远程/魔法
- 地下城（Dungeon）系统：连续战斗多个敌人
- BOSS战：特殊机制和稀有掉落

**离线收益：**
- 离线时间上限24小时
- 战斗和偷窃默认不支持离线（需手动开启）
- 离线收益 = 在线效率 × 离线时间

**设计启示：**
- **技能互联**：采集→加工→装备→战斗，形成完整经济循环
- **Mastery（精通）系统**：每个技能的每个动作都有独立精通等级
- **完成度追踪**：Completion Log激励收集和全完成

## 2. 放置游戏核心设计模式

### 2.1 资源增长模型

根据Kongregate《The Math of Idle Games》系列研究，放置游戏主要使用以下增长模型：

#### 指数增长模型（最常见）
```
产出速率 = 基础产出 × 数量 × 乘数
建筑价格 = 基础价格 × 增长系数^数量
```
- **增长系数**通常在 1.07 ~ 1.15 之间
- Cookie Clicker使用1.15，Adventure Capitalist使用1.07-1.11
- **设计意义**：指数价格 + 指数产出 = 对数时间进度，确保游戏有持续挑战

#### 多项式增长模型
```
产出 = 基础值 × (1 + 等级 × 系数)
```
- 用于声望加成、升级效果等
- 与指数增长形成平衡

#### 对数增长模型
```
实际效果 = log(基础值 × 乘数)
```
- 用于限制极端增长
- 防止某个乘数过于强大

#### 导数增长模型（高级）
```
产出 = d(资源)/dt，即资源的导数
```
- Antimatter Dimensions、Derivative Clicker等游戏使用
- 形成"元增长"：不仅资源在增长，增长率本身也在增长

### 2.2 声望/转生系统设计

声望系统是放置游戏的核心"重复可玩性"来源：

#### 声望货币计算公式
| 公式类型 | 表达式 | 适用场景 | 代表游戏 |
|----------|--------|----------|----------|
| 平方根 | √(总资源 / 阈值) | 通用 | Cookie Clicker |
| 对数 | log(总资源 / 阈值) | 长期游戏 | Trimps |
| 阶梯 | floor(总资源 / 阈值) | 简单直观 | Adventure Capitalist |
| 多项式 | (总资源 / 阈值)^指数 | 灵活调控 | Realm Grinder |

#### 多层声望结构
```
游戏进度 → 声望1（软重置）→ 声望2（硬重置）→ 声望3（飞升）→ ...
```
- **每层声望**应有独特的货币和升级树
- **解锁新机制**而非仅提供数值加成
- **典型比例**：每层声望的循环时间约为上层的10倍

#### Revolution Idle的多层声望实例：
- **Prestige**：基础重置，获得Prestige货币
- **Promote**：中级重置，获得Promote货币
- **Ascension**：高级重置，获得Ascension货币
- **Infinity**：终极重置，解锁全新内容

### 2.3 离线收益计算方式

#### 常见离线收益模型：
| 模型 | 公式 | 优点 | 缺点 |
|------|------|------|------|
| 线性 | 收益 = 速率 × 时间 | 简单直观 | 长时间离线收益过高 |
| 衰减 | 收益 = 速率 × log(1+时间) | 防止过度积累 | 计算复杂 |
| 上限 | 收益 = min(速率×时间, 上限值) | 控制平衡 | 需要合理设上限 |
| 阶梯 | 不同时间段不同效率 | 灵活 | 实现复杂 |

#### 主流游戏的离线设计：
- **Melvor Idle**：上限24小时，战斗需手动开启
- **Cookie Clicker**：无上限，但Wrinklers机制鼓励定期回归
- **AFK Arena**：上限12-24小时，VIP可延长
- **Tap Titans 2**：部分离线奖励，活跃玩家收益更高

### 2.4 解锁和进度门槛设计

#### 解锁设计原则：
1. **可预测性**：玩家应能看到"下一个目标"
2. **节奏感**：解锁间隔应形成"紧张→释放"的节奏
3. **意义感**：每次解锁都应带来明显的玩法变化

#### 典型解锁节奏：
```
0-5分钟：基础教程（点击、购买第一个建筑）
5-30分钟：第一个自动化（经理人/自动点击）
1-4小时：第一个子系统（迷你游戏/新资源）
1-3天：声望系统解锁
1-2周：第二个声望层
1月+：终局内容
```

### 2.5 瓶颈和节奏控制

#### 瓶颈设计类型：
| 瓶颈类型 | 机制 | 效果 |
|----------|------|------|
| 时间瓶颈 | 建筑/升级需要等待时间 | 强制休息，防止速通 |
| 资源瓶颈 | 某种资源产出极慢 | 引导玩家优化策略 |
| 声望瓶颈 | 必须重置才能推进 | 鼓励声望循环 |
| 随机瓶颈 | 稀有掉落/随机事件 | 增加不确定性和惊喜 |
| 技能瓶颈 | 需要特定操作技巧 | 区分活跃/放置玩家 |

## 3. 放置RPG战斗系统设计

### 3.1 即时战斗 vs 回合制战斗

#### 即时战斗（Real-Time）
**代表游戏：** AFK Arena、Firestone Idle RPG、Tap Titans 2

| 特性 | 说明 |
|------|------|
| 战斗节奏 | 持续进行，角色自动攻击 |
| 玩家参与 | 可手动释放技能/终极技 |
| 视觉表现 | 角色在战场上移动、攻击 |
| 策略深度 | 队伍站位、技能时机 |
| 适合场景 | 想要视觉冲击力和操作感 |

**设计要点：**
- 自动普攻 + 手动大招是最受欢迎的模式
- 战斗速度控制：1x/2x/4x/8x加速
- 自动战斗AI配置：可设置技能释放优先级

#### 回合制战斗（Turn-Based）
**代表游戏：** Melvor Idle、Gacha Idle、Idle Heroes

| 特性 | 说明 |
|------|------|
| 战斗节奏 | 按回合/间隔进行 |
| 玩家参与 | 配置装备和技能，战斗自动 |
| 视觉表现 | 文字/简单动画 |
| 策略深度 | 装备搭配、技能组合 |
| 适合场景 | 想要深度策略和配置 |

### 3.2 自动战斗 + 手动技能释放（推荐模式）

这是目前最主流的放置RPG战斗模式：

```
┌─────────────────────────────────┐
│         战斗画面                │
│  ┌───┐  ┌───┐  ┌───┐          │
│  │角色1│  │角色2│  │角色3│ ← 我方 │
│  └───┘  └───┘  └───┘          │
│                                 │
│  [自动攻击进行中...]            │
│                                 │
│  ┌───┐  ┌───┐  ┌───┐          │
│  │敌人1│  │敌人2│  │BOSS │ ← 敌方│
│  └───┘  └───┘  └───┘          │
│                                 │
│  [技能1] [技能2] [技能3] [大招] │ ← 手动释放
│  ═══════════════════ HP条      │
└─────────────────────────────────┘
```

**设计要点：**
- 普攻自动进行，玩家只需关注关键技能
- 技能冷却时间（CD）控制释放节奏
- 终极技能需要充能条满才能释放
- 可设置"自动释放"选项，满足纯放置玩家

### 3.3 队伍编成和角色搭配

#### AFK Arena式5人队伍：
```
┌──────────────────────┐
│  前排：坦克 + 战士    │
│  ──────────────────  │
│  后排：法师 + 射手 + 辅助│
└──────────────────────┘
```

**阵营加成系统（AFK Arena）：**
- 同阵营3人：基础属性加成
- 同阵营5人：更高加成
- 混搭也有特定加成组合

**角色定位：**
| 定位 | 职责 | 关键属性 |
|------|------|----------|
| 坦克 | 吸收伤害 | HP、防御、闪避 |
| 战士 | 近战输出 | 攻击、暴击 |
| 法师 | AOE伤害 | 魔攻、穿透 |
| 射手 | 单体输出 | 攻速、暴击 |
| 辅助 | 治疗/增益 | 治疗量、效果持续时间 |
| 控制 | 减益/控制 | 效果命中、持续时间 |

### 3.4 BOSS战机制

#### 常见BOSS战设计：
| 机制 | 说明 | 代表游戏 |
|------|------|----------|
| 限时击杀 | 30秒内必须击败 | Tap Titans 2 |
| 阶段转换 | BOSS血量低于X%进入新阶段 | AFK Arena |
| 护盾机制 | 需要先破盾再造成伤害 | Idle Heroes |
| 狂暴计时 | 超过时间BOSS伤害暴增 | Firestone |
| 弱点机制 | 特定属性/攻击方式克制 | Melvor Idle |
| 召唤小怪 | BOSS定期召唤小怪 | AFK Arena |

#### BOSS战节奏设计：
```
普通关卡 1-9 → BOSS关卡 10 → 普通关卡 11-19 → BOSS关卡 20 → ...
```
- BOSS关卡是进度瓶颈
- 失败后可以重复挑战
- 击败BOSS解锁新区域/新内容

### 3.5 PVP系统

#### PVP设计模式：
| 模式 | 说明 | 适合场景 |
|------|------|----------|
| 异步竞技场 | 挑战其他玩家的防御阵容 | 最常见，不要求同时在线 |
| 排位赛 | 匹配相近战力的对手 | 竞技性强的游戏 |
| 公会战 | 公会间对抗 | 社交驱动 |
| 锦标赛 | 限时排名赛 | 活跃玩家 |
| 跨服战 | 服务器间对抗 | 长期运营 |

## 4. 放置游戏的丰富玩法

### 4.1 抽卡/召唤系统设计

#### 稀有度系统：
| 稀有度 | 概率 | 说明 |
|--------|------|------|
| N（普通）| 40-50% | 基础角色，容易获得 |
| R（稀有）| 30-35% | 有一定价值 |
| SR（超稀有）| 10-15% | 核心角色 |
| SSR（超超稀有）| 1-3% | 顶级角色，追求目标 |
| UR/Legendary | 0.1-0.5% | 极稀有，通常有保底 |

#### 保底系统（Pity System）：
- **硬保底**：X次抽取必定获得SSR（通常50-100次）
- **软保底**：超过一定次数后概率逐步提升
- **50/50机制**：保底可能是随机SSR或限定SSR（各50%）
- **保底继承**：未触发的保底可以累积到下期卡池

#### 抽卡货币设计：
- 免费货币：通过游戏内获得（每日任务、活动等）
- 付费货币：充值获得
- 混合使用：部分抽卡可用免费货币，限定卡池需付费货币

### 4.2 塔防融合模式

#### The Tower - Idle Tower Defense 的成功模式：
- **核心循环**：敌人波次 → 塔自动攻击 → 获得金币 → 升级塔 → 更强波次
- **塔升级树**：基础塔 → 分支升级（火焰/冰冻/毒素等）
- **技能系统**：主动技能（空袭/地雷等）+ 被动增益
- **实验室系统**：研究永久升级
- **事件系统**：限时事件，独特奖励

#### 塔防+放置的设计要点：
- 塔的放置是初始策略，之后自动战斗
- 升级决策是核心玩法（资源有限，选择升级哪个塔）
- 波次难度指数增长
- 失败后保留部分收益，鼓励重试

### 4.3 随机事件和突发事件

#### 事件类型设计：
| 事件类型 | 频率 | 效果 | 代表游戏 |
|----------|------|------|----------|
| 金色饼干 | 随机出现 | 临时大幅增益 | Cookie Clicker |
| 商人来访 | 定期出现 | 特殊交易 | Kittens Game |
| 怪物入侵 | 随机触发 | 需要应对战斗 | Firestone |
| 宝箱发现 | 探索触发 | 随机奖励 | 多数RPG |
| 市场波动 | 定期变化 | 资源价格变化 | 经济类 |
| 天气效果 | 周期性 | 影响产出效率 | 种田类 |

#### 设计原则：
- **正面事件 > 负面事件**：7:3比例
- **事件应有明确提示**：不让玩家错过
- **可选择性**：玩家可以选择参与或忽略
- **与核心循环挂钩**：事件应增强而非打断核心循环

### 4.4 成就系统和收集要素

#### 成就系统设计层次：
```
第一层：基础成就（达到X级、收集Y个）
第二层：挑战成就（在Z条件下完成W）
第三层：隐藏成就（不提示条件，探索发现）
第四层：里程碑成就（达到游戏终局）
```

#### 收集要素类型：
- **图鉴/百科**：收集所有角色/怪物/物品的信息
- **皮肤/外观**：视觉定制，不影响数值
- **称号系统**：完成特定成就解锁
- **收藏品**：稀有物品，展示用
- **星图/地图**：探索进度可视化

### 4.5 多条科技发展路线

#### 科技树设计模式：
```
              ┌→ 火焰路线（高伤害）
基础科技 → ──┼→ 冰霜路线（控制型）
              ├→ 雷电路线（速攻型）
              └→ 自然路线（持续型）
```

**设计要点：**
- 每条路线应有独特的玩法体验
- 路线间应有交叉点，鼓励混合搭配
- 后期应能解锁多路线并行
- 重置科技应消耗声望货币

### 4.6 限时活动和赛季内容

#### 活动类型：
| 活动类型 | 周期 | 内容 | 奖励 |
|----------|------|------|------|
| 每日任务 | 每天 | 简单任务 | 基础资源 |
| 每周挑战 | 每周 | 中等难度 | 中级资源 |
| 月度活动 | 每月 | 特殊玩法 | 限定角色/皮肤 |
| 赛季排名 | 2-3月 | 竞技排名 | 赛季独占奖励 |
| 节日活动 | 节日 | 主题内容 | 节日限定 |
| 周年庆 | 年度 | 大型庆祝 | 超稀有奖励 |

#### Firestone Idle RPG的活动设计参考：
- 世界探索：不同区域有独特怪物、环境和故事
- 连续更新和赛季活动保持新鲜感
- 声望系统确保长期进度
- 公会系统增加社交维度

## 5. 放置游戏的叙事与沉浸感

### 5.1 如何在放置游戏中融入故事

#### 叙事融入方式：
| 方式 | 实现难度 | 效果 | 代表游戏 |
|------|----------|------|----------|
| 区域解锁叙事 | 低 | 中 | AFK Arena |
| NPC对话碎片 | 中 | 高 | Firestone |
| 日志/笔记系统 | 低 | 中 | Kittens Game |
| 剧情关卡 | 中 | 高 | AFK Journey |
| 环境叙事 | 高 | 极高 | Long Gone |
| 事件驱动叙事 | 中 | 高 | Crusaders of the Lost Idols |

#### 叙事节奏与游戏节奏的配合：
- **放置游戏的叙事应是"碎片化"的**：短文本，不打断放置体验
- **剧情作为"奖励"**：通过关卡后解锁剧情片段
- **背景故事补充**：物品描述、角色介绍中融入世界观
- **里程碑叙事**：每次重大进度解锁新章节

### 5.2 关卡/地图探索系统

#### AFK Journey式地图探索：
- **节点式地图**：不是自由移动，而是按节点推进
- **每节点有独特事件**：战斗、宝箱、NPC、谜题
- **支线探索**：主线路径外有隐藏区域
- **地图收集**：每张地图有完成度追踪

#### 地图设计层次：
```
世界地图 → 大区域 → 小区域 → 关卡节点
  │           │          │          │
  └─ 区域解锁  └─ 区域BOSS └─ 探索度  └─ 关卡战斗
```

### 5.3 NPC对话和任务系统

#### 任务系统设计：
| 任务类型 | 说明 | 奖励 |
|----------|------|------|
| 主线任务 | 推进剧情 | 解锁新内容 |
| 支线任务 | 世界观补充 | 额外资源 |
| 日常任务 | 每日重复 | 基础资源 |
| 成就任务 | 长期目标 | 稀有奖励 |
| 连锁任务 | 多步骤任务链 | 独特奖励 |

#### NPC设计原则：
- **功能性NPC**：商人、训练师、任务发布者
- **叙事性NPC**：提供世界观、故事碎片
- **互动性NPC**：可对话、有好感度系统
- **放置友好**：NPC对话不应阻断自动进度

### 5.4 剧情驱动的进度解锁

#### "剧情门"设计：
```
玩家推进 → 达到条件 → 剧情事件触发 → 新内容解锁
    ↑                                    │
    └────── 继续推进 ←───────────────────┘
```

- **剧情作为"解锁仪式"**：让解锁有仪式感
- **剧情不应成为瓶颈**：跳过功能是必要的
- **回顾功能**：允许玩家回看已解锁的剧情

### 5.5 世界观构建

#### 世界观设计框架：
1. **核心设定**：世界的基本规则（魔法/科技/混合）
2. **势力/阵营**：不同利益集团
3. **历史背景**：过去发生了什么
4. **当前冲突**：玩家为什么要冒险
5. **未来走向**：玩家的终极目标

#### 沉浸感增强技巧：
- **环境音效**：不同区域不同背景音
- **视觉主题**：每个区域独特的色彩和风格
- **物品描述**：每件装备都有故事背景
- **怪物图鉴**：描述怪物的生态和习性

### 6.5 Idle Slayer — 平台跳跃融合

**创新点：**

- **放置 + 操作混合**：满足不同类型玩家
- **操作带来额外收益**：但不强制
- **视觉反馈丰富**：跳跃和战斗动画增加趣味

## 8. 关键设计模式总结

### 8.1 放置游戏的核心设计模式

#### 模式1：核心循环（Core Loop）
```
        ┌──────────────┐
        ↓              │
   获得资源 → 消费资源 → 提升产出 → 获得更多资源
        ↑                              │
        └──── 声望重置 ←───────────────┘
```
**所有成功的放置游戏都遵循这个基本循环。**

#### 模式2：自动化渐进（Automation Progression）
```
手动操作 → 半自动（购买自动）→ 全自动（管理策略）→ 元管理（优化自动化）
```
**玩家从"操作者"逐步变成"管理者"。**

#### 模式3：多层进度（Multi-Layer Progression）
```
微观进度：每次点击/每秒产出
中观进度：升级、解锁新内容
宏观进度：声望系统、转生
元进度：成就、收集、完成度
```
**四个层次同时运行，确保玩家始终有短期和长期目标。**

#### 模式4：瓶颈-突破循环（Bottleneck-Breakthrough）
```
流畅进度 → 遇到瓶颈 → 努力积累 → 突破瓶颈 → 新的流畅进度 → 新瓶颈
```
**瓶颈创造"需要"，突破创造"满足"。**

#### 模式5：并行系统（Parallel Systems）
```
战斗系统 ←→ 经济系统 ←→ 探索系统
    ↓           ↓           ↓
 英雄培养    资源管理     地图解锁
    ↓           ↓           ↓
    └────── 共同驱动 ──────┘
```
**多个系统互相独立但互相增益，增加策略深度。**

### 8.2 不同类型放置游戏的设计差异

| 维度 | 纯增量/点击器 | 放置RPG | 放置经营 | 放置策略 |
|------|-------------|---------|---------|---------|
| 核心驱动 | 数字增长 | 角色培养 | 资源管理 | 策略决策 |
| 战斗系统 | 无/简单 | 核心 | 无 | 可选 |
| 叙事深度 | 低 | 中-高 | 低-中 | 中 |
| 社交需求 | 低 | 高 | 中 | 高 |
| 美术需求 | 低 | 高 | 中 | 中 |
| 数值复杂度 | 高 | 中 | 高 | 高 |
| 代表游戏 | Cookie Clicker | AFK Arena | Adventure Capitalist | Realm Grinder |

## 9. 对Game Portal项目的具体建议

### 9.1 核心架构建议

#### 推荐的游戏类型：放置RPG + 增量元素
理由：
- RPG元素提供角色培养的深度和叙事空间
- 增量元素确保放置体验的核心满足感
- 战斗系统提供视觉反馈和策略深度
- 最适合HTML5 Canvas实现

#### 状态机设计：
```
┌─────────┐     点击开始     ┌─────────┐
│  开始画面 │ ─────────────→ │  主游戏   │
│ (Title)  │               │  (Game)  │
└─────────┘               └────┬─────┘
     ↑                         │
     │                    ┌────┴─────┐
     │                    ↓          ↓
     │               ┌────────┐ ┌────────┐
     │               │  战斗   │ │  管理   │
     │               │(Battle)│ │(Manage)│
     │               └────────┘ └────────┘
     │                    │          │
     │                    └────┬─────┘
     │                         │
     │    声望重置              │
     └─────────────────────────┘
```

### 9.2 核心系统优先级

#### Phase 1 — MVP（最小可行产品）：
1. **基础点击/自动产出系统**
   - 点击获得金币
   - 建筑自动产出
   - 升级系统（至少5种建筑）
   
2. **简单战斗系统**
   - 自动战斗 + 手动技能
   - 3-5个角色
   - 区域推进（至少10个区域）

3. **基础UI**
   - 资源显示
   - 升级面板
   - 战斗画面

#### Phase 2 — 核心深度：
1. **声望系统**
   - 第一层声望（软重置）
   - 声望升级树
   
2. **角色系统扩展**
   - 10+角色
   - 装备系统
   - 角色升级/觉醒

3. **更多战斗内容**
   - BOSS战
   - 地下城
   - 竞技场（异步PVP）

#### Phase 3 — 丰富内容：
1. **抽卡系统**
   - 召唤池
   - 保底机制
   
2. **成就和收集**
   - 成就系统
   - 图鉴
   - 称号

3. **叙事内容**
   - NPC对话
   - 任务系统
   - 世界观

#### Phase 4 — 长期运营：
1. **多层声望**
2. **公会系统**
3. **限时活动**
4. **赛季内容**

### 9.3 数值设计建议

#### 推荐的数值框架：
```javascript
// 建筑价格增长
buildingCost(baseCost, owned) = baseCost * 1.15^owned

// 英雄升级价格
heroUpgradeCost(baseCost, level) = baseCost * 1.07^level

// 声望货币
prestigeCurrency(totalEarnings) = floor(sqrt(totalEarnings / 1e12))

// 离线收益（带衰减）
offlineEarnings(rate, hours) = rate * hours * (1 / (1 + hours/24))

// 战斗伤害
damage = baseAtk * (1 + atkBonus) * critMultiplier * skillMultiplier
```

#### 推荐的进度节奏：
| 时间节点 | 应达到的进度 |
|----------|-------------|
| 1分钟 | 理解核心操作，购买第一个升级 |
| 5分钟 | 第一个自动化，开始"放置" |
| 30分钟 | 第一个BOSS战 |
| 2小时 | 第一个声望重置 |
| 1天 | 解锁第二个区域/系统 |
| 1周 | 解锁声望第二层 |
| 1月 | 接近终局内容 |

### 9.4 UI/UX设计建议

#### 响应式布局策略：
- **Mobile First设计**：先设计竖屏布局，再适配宽屏
- **断点设计**：
  - 移动端：< 768px（竖屏单栏）
  - 平板：768px - 1024px（双栏）
  - 桌面：> 1024px（三栏或自由布局）

#### 推荐的底部导航栏（移动端）：
```
[🏠 主页] [⚔ 战斗] [👥 英雄] [📦 背包] [⚙ 设置]
```

#### 推荐的三栏布局（桌面端）：
```
[左：导航+信息] [中：主画面] [右：操作面板]
```

### 9.5 技术实现建议

#### 核心架构：
```
GameEngine (主循环，requestAnimationFrame)
  ├── GameState (状态管理)
  │    ├── Resources (资源系统)
  │    ├── Buildings (建筑系统)
  │    ├── Heroes (英雄系统)
  │    ├── Combat (战斗系统)
  │    └── Prestige (声望系统)
  ├── Renderer (Canvas渲染)
  │    ├── BattleScene (战斗场景)
  │    ├── UIOverlay (UI叠加层)
  │    └── Effects (特效系统)
  ├── SaveManager (存档管理)
  │    ├── AutoSave (自动保存)
  │    ├── OfflineCalc (离线计算)
  │    └── Export/Import (导入导出)
  └── EventSystem (事件系统)
       ├── GameEvents (游戏事件)
       ├── Achievements (成就)
       └── Quests (任务)
```

#### 大数处理：
- 使用科学计数法显示（1.23e15）
- 或自定义后缀（1.23M, 1.23B, 1.23T, 1.23Qa...）
- 推荐使用 break_infinity.js 库处理超大数

#### 存档设计：
- 每30秒自动保存
- 使用 localStorage + 导出功能
- 离线时间记录上次保存时间戳
