## 第一章：设计哲学与核心原则

### 1.1 放置游戏的本质

放置游戏（Idle/Incremental Game）的本质是**"时间投资回报"**——玩家投入时间（在线或离线），获得进度增长。根据学术研究 *"Playing to Wait: A Taxonomy of Idle Games"* (NSF, 2018)，放置游戏的核心特征是**自动化进度循环**，而非简单的"点击游戏"。

放置游戏的体验可以分为三个递进层次：

| 层次 | 体验 | 设计目标 | 示例 |
|------|------|----------|------|
| **被动增长** | 基础层 | 玩家离线也能获得进度 | 离线收益、自动采集 |
| **主动决策** | 策略层 | 玩家做有意义的选择 | 科技路线、声望时机、阵容搭配 |
| **情感投入** | 沉浸层 | 玩家产生归属感和成就感 | 故事推进、伙伴互动、收集完成 |

**关键洞察**：当前 Game Portal 已完成的放置游戏（Batch 1-2）主要停留在第一层（被动增长），缺乏第二层的策略深度和第三层的情感投入。本指南的目标是将所有游戏提升到三个层次兼备。

### 1.2 六大核心设计原则

#### 原则一：自然增长为主，决策驱动进度

资源应主要通过自动化系统自然产出，而非依赖玩家反复点击。玩家的核心体验是"做决策"而非"做苦力"——升级什么建筑、走哪条科技路线、何时声望、如何搭配阵容。

- ✅ 资源自动产出，玩家规划产出方向
- ✅ 点击作为辅助手段（触发事件、释放技能、探索地图）
- ❌ 核心进度依赖玩家反复点击

**设计参考**：AFK Arena 的成功在于"挂机收益 + 策略编队"，玩家不需要一直操作，但需要思考阵容搭配。根据 Deconstructor of Fun 的分析，AFK Arena 将进度和商业化循环绑定到 idle 机制上，创造了新的会话化玩家旅程 [citation:AFK Arena Analysis](https://www.deconstructoroffun.com/blog/2019/6/6/afk-arena-puts-lilith-into-the-billionaire-club)。

#### 原则二：每款游戏独一无二

禁止"换皮"设计——相同引擎 + 不同数值 ≠ 新游戏。每款游戏必须有：

1. **独特的核心玩法机制**（7大模式中选一种为主 + 1-2种辅助）
2. **专属的故事线和世界观**
3. **差异化的UI/视觉风格**（8种视觉风格中选一种）
4. **独特的虚拟伙伴角色**

100款游戏应覆盖至少15种不同的核心玩法组合。

#### 原则三：丰富玩法超越数值堆砌

单纯的数值增长（金币→建筑→更多金币）是最低级的放置设计。必须融合：

- **塔防元素**：波次防御、塔的搭配策略
- **多科技路线**：至少2条互斥或协同的发展路线
- **随机/突发/定期事件**：每款游戏至少10种不同事件
- **抽卡系统**：角色/装备收集，保底机制
- **成就系统**：里程碑 + 隐藏 + 系列成就

玩家应面临**有意义的选择**（choice that matters）——不同选择导致显著不同的游戏体验。

#### 原则四：故事驱动沉浸感

每款游戏有独立的世界观、角色、剧情线。关卡/地图探索系统让玩家感受"在冒险"而非"在等数字增长"。

- **世界观**：每个系列有统一的背景设定
- **角色**：有名字、有性格、有成长弧线的NPC
- **剧情线**：随游戏进度推进的主线故事
- **支线任务**：NPC委托、隐藏任务、收集任务

**设计参考**：《一念逍遥》主策划复盘指出，他们的设计基于两个原则：**基于代入感去设计游戏体验**，以及**基于放置聚焦框架的开发**。核心流程是去野外挂机秘境→击败怪物→获取材料→定制丹药/炼制装备→提升境界修为 [citation:一念逍遥复盘](https://zhuanlan.zhihu.com/p/419121185)。

#### 原则五：虚拟伙伴激励系统（重点新增）

每款游戏至少有一个虚拟伙伴角色（导师/助手/宠物/战友），贯穿游戏全程。

**伙伴五大功能**：

| 功能 | 描述 | 示例 |
|------|------|------|
| **称赞鼓励** | 达成里程碑时给予个性化赞美 | "太厉害了！你的修为已经突破金丹期！" |
| **指引帮助** | 卡关时给出策略提示 | "试试升级丹炉，可以提升炼丹效率哦" |
| **贴心服务** | 离线回来时汇报进展 | "你离开的8小时里，我帮你采集了2340灵药~" |
| **情感陪伴** | 根据时长/节日/成就触发特殊对话 | "今天是你来到修仙界的第100天呢！" |
| **危机提醒** | 资源满仓、建筑可升级、事件即将结束 | "仓库快满了！该花一些了~" |

**伙伴性格设定**：按游戏主题匹配（修仙类是师姐/师弟、三国类是军师、宠物类是萌宠管家等）。对话文本丰富（每场景≥5条候选），避免重复，按游戏进度解锁新台词。

#### 原则六：PC/移动端自适应 + 多分辨率支持

#### 多分辨率适配（720p / 1080p / 2K）

游戏 Canvas 逻辑分辨率必须支持以下三档，运行时根据设备自动选择：

| 画质档位 | 逻辑分辨率 | devicePixelRatio | 粒子数量 | 阴影效果 | 适用设备 |
|----------|------------|------------------|----------|----------|----------|
| **720p 标清** | 1280×720 | 1x | 0~少量 | 无 | 低端手机/平板 |
| **1080p 高清** | 1920×1080 | 1.5x | 中等 | 简单 | 主流 PC/高端手机 |
| **2K 超清** | 2560×1440 | 2x | 丰富 | 精细 | 4K 显示器/旗舰设备 |

**分辨率适配规则**：
- Canvas 内部逻辑坐标统一以 **1920×1080** 为基准设计（16:9 横屏）
- 720p 模式下按 0.667 倍缩放渲染，2K 模式下按 1.333 倍缩放渲染
- 所有 UI 元素坐标和尺寸使用**相对比例**（百分比或基准倍率），不硬编码像素值
- 字体大小按分辨率档位分级：720p→14px / 1080p→18px / 2K→24px
- 玩家可在设置中手动切换画质档位，也可选择"自动"（根据设备性能检测）
- **坚决不使用 480×640 等低分辨率**

**画质设置 API 设计**：
```typescript
type QualityLevel = '720p' | '1080p' | '2k' | 'auto';

interface RenderConfig {
  quality: QualityLevel;
  width: number;     // 实际渲染宽度
  height: number;    // 实际渲染高度
  pixelRatio: number;
  particles: boolean;
  shadows: boolean;
  fontScale: number; // 字体缩放系数
}
```

#### 设备布局适配

| 设备 | 布局 | Canvas | 交互 |
|------|------|--------|------|
| PC宽屏（≥1024px） | 左右分栏/三栏 | 横屏 16:9，按画质档渲染 | **鼠标点击+悬停+拖拽** + **键盘快捷键** |
| 手机竖屏（<768px） | 上下布局 | 全宽×自适应，自动降档至 720p | 触控（按钮≥44px） |
| 手机横屏 | 紧凑双栏 | 横屏 16:9，自动降档至 720p | 触控+手势 |

#### 鼠标+按键操作（PC端核心交互）

PC 端必须同时支持**鼠标操作**和**键盘快捷键**，两者功能完全对等：

**鼠标操作**：
- 左键单击：选中/激活（建筑、角色、按钮）
- 左键双击：快速升级/确认
- 右键单击：上下文菜单（详情/升级/拆除）
- 左键拖拽：滚动地图、移动建筑、拖放装备
- 滚轮：缩放地图、滚动列表
- 悬停（Hover）：Tooltip 详情提示

**键盘快捷键**：
- `Space`：核心操作（点击/采集/攻击）
- `1-9`：快捷选中建筑/角色槽位
- `Enter`：确认 / `Esc`：取消
- `↑↓←→` / `W/A/S/D`：地图滚动、列表选择
- `B` 建筑 / `H` 角色 / `M` 地图 / `P` 暂停
- `R`：声望/转生（需二次确认）
- `+/-`：游戏速度调节（1x/2x/3x）

详细操作规范见第七章 7.4 节。

### 1.3 反模式（禁止使用）

- ❌ 所有游戏共享相同的"点击→升级→声望"循环
- ❌ 纯数值升级，没有视觉/玩法变化
- ❌ 明暗变化、闪烁、neon/glow 等干扰视觉的效果
- ❌ 强制广告或付费墙阻断游戏体验
- ❌ 虚假的"选择"（所有路线最终结果相同）
- ❌ 虚拟伙伴对话重复单调、缺乏个性

## 第二章：放置游戏类型分类与玩法模式

### 2.1 七大核心玩法模式

每款游戏以一种模式为主，融合1-2种辅助模式，形成独特的玩法组合。

#### 模式A：资源链经营型
**参考**: Kittens Game、Melvor Idle、Adventure Capitalist

核心是多条资源生产线形成供应链（木材→木板→家具→贸易品）。玩家优化生产链效率，决定资源分配优先级。特色是资源之间的转化关系和产业链优化。

适合：模拟经营、文明建设、模拟城市类

关键设计要素：
- 至少3级资源转化链
- 每级转化有建筑/角色负责
- 转化效率可通过升级提升
- 高级资源需要多种低级资源组合

#### 模式B：角色养成RPG型
**参考**: AFK Arena、Idle Heroes、剑与远征

核心是收集/培养角色，组建队伍，自动战斗。角色有稀有度、技能、属性差异，玩家需要策略搭配阵容。

适合：武侠、仙侠、动漫、神话、三国类

关键设计要素：
- 角色稀有度体系（N/R/SR/SSR）
- 角色技能和属性差异
- 队伍编成策略（前排/后排/辅助/输出）
- 角色羁绊/组合效果
- 角色培养路线（升级/突破/觉醒）

#### 模式C：地图探索冒险型
**参考**: Trimps、IdleOn、Legends of IdleOn

核心是逐步解锁地图区域，每个区域有不同怪物/资源/事件。玩家有RPG小地图，探索未知区域。

适合：冒险、地下城、生存、RPG类

关键设计要素：
- 区域解锁系统（相邻区域通关后解锁）
- 迷雾系统（未探索区域被覆盖）
- 区域Boss和隐藏宝箱
- 探索完成度（0%-100%）
- 快速传送（已解锁区域）

#### 模式D：塔防策略型
**参考**: The Tower、Idle Hero TD、Arknights

核心是建造/升级防御塔，抵御波次敌人进攻。塔有类型差异和克制关系。

适合：军事、科幻、植物大战僵尸类

关键设计要素：
- 防御塔类型（近战/远程/范围/减速）
- 敌人波次设计（普通→精英→Boss）
- 塔的升级和进化
- 路径和地形策略
- 特殊能力（技能释放）

#### 模式E：卡牌收集型
**参考**: 放置奇兵、剑与远征、Genshin Impact

核心是抽卡获取角色/装备，卡牌升星/进化。有保底机制和限定卡池。

适合：三国、封神、动漫IP、RPG类

关键设计要素：
- 抽卡概率体系（N/R/SR/SSR）
- 保底机制（50抽保SR，100抽保SSR）
- 卡池轮换（普通池+限定池）
- 卡牌升星/进化系统
- 收集图鉴和完成度

#### 模式F：技能树科技型
**参考**: Realm Grinder、Kittens Game、Civilization

核心是多条科技/技能发展路线，玩家选择不同分支，路线之间有互斥或协同关系。

适合：修仙、文明、科幻、策略类

关键设计要素：
- 至少3条科技/技能路线
- 路线之间的互斥或协同关系
- 每条路线的独特加成和玩法
- 路线选择不可轻易更改（增加决策分量）
- 组合效果（同时走两条路线的额外加成）

#### 模式G：事件驱动型（原创模式）
**参考**: 原创设计

核心是游戏通过随机事件、突发事件、定期事件推动发展。每个事件是一个"故事片段"，玩家选择影响后续发展。

适合：所有类型，作为辅助系统

关键设计要素：
- 随机事件池（每款游戏≥10种）
- 事件分级（普通/稀有/史诗/传说）
- 选择分支（每个事件2-3个选项）
- 后果系统（短期+长期影响）
- 事件链（相关事件形成故事线）

## 第三章：核心系统设计规范

### 3.1 资源增长系统

#### 基础产出
- 建筑/角色自动产出资源（每秒/每分钟计算）
- 产出公式：`实际产出 = 基础产出 × 等级加成 × 科技加成 × 声望加成 × 事件加成`
- 不同资源的产出速率差异设计（主资源快速，稀有资源缓慢）

#### 离线收益
- 基于离线时长 × 产出速率 × 离线效率系数
- 离线效率：通常50%-100%，可通过升级提升
- 离线上限：24小时（可配置）
- 回归时虚拟伙伴汇报："你离开了X小时，我帮你获得了Y资源~"

#### 资源转化链
- 至少3级转化：原材料→加工品→高级品
- 每级转化需要对应的建筑/角色
- 转化有损耗率（可通过升级降低）
- 高级资源需要多种低级资源组合

#### 资源类型
每款游戏至少3-5种资源，形成经济生态：
- **主资源**：最基础，产出最快（如金币、灵气）
- **中级资源**：由主资源转化或中速产出（如铁锭、丹药）
- **稀有资源**：产出缓慢，用于高级升级（如宝石、仙丹）
- **声望货币**：声望后获得，永久性加成（如传承点、天道点）
- **高级货币**：用于抽卡等特殊功能（如仙缘、招募令）

### 3.2 声望/转生系统

#### 触发条件
- 达到特定里程碑触发（非固定数值，而是游戏进度节点）
- 示例：通关第5章、达到金丹期、收集30个武将

#### 声望奖励
- **永久倍率加成**：所有资源产出 ×1.5（每次声望叠加）
- **新功能解锁**：声望后解锁新建筑/新区域/新角色
- **专属角色/皮肤**：声望次数达到阈值获得
- **虚拟伙伴进化**：伙伴外观和对话升级

#### 声望层次
- 可多次声望，每次奖励递增但幅度递减
- 声望层数标记（如"三转修士"）
- 设计原则：声望是"加速器"而非"必需品"——不声望也能玩，声望后更快

#### 声望数学模型
根据 *"The Math of Idle Games, Part III"* (Game Developer, 2017) 的分析：
- 最佳声望时机：当声望加成能使下一轮速度显著超过当前轮时
- 声望加成通常采用对数或指数增长：`bonus = base × log(prestige_count + 1)` 或 `bonus = base × 1.5^count`（递减）
- 每次声望后前期速度应提升5-10倍 [citation:Math of Idle Games](https://www.gamedeveloper.com/design/the-math-of-idle-games-part-iii)

### 3.3 抽卡/召唤系统

#### 卡池设计

| 卡池类型 | 货币 | 内容 | 刷新 |
|----------|------|------|------|
| 普通池 | 游戏币 | N/R为主，极低概率SR | 永久 |
| 高级池 | 高级货币 | R/SR/SSR | 每月轮换 |
| 限定池 | 高级货币 | 限定SSR + 常规 | 限时活动 |
| 友情池 | 友情点 | N/R为主 | 永久 |

#### 概率体系
```
N  (普通) : 70%
R  (稀有) : 20%
SR (超稀有): 8%
SSR(传说) : 2%
```

#### 保底机制
- 每50抽保底至少1张SR
- 每100抽保底至少1张SSR
- 保底计数器独立于卡池（切换卡池不重置）
- 显示保底进度条（"距离保底SR还有X抽"）

#### 抽卡视觉表现
- 卡牌翻转动画（3D翻转效果）
- 稀有度对应不同特效（N无特效，R蓝光，SR金光，SSR全屏特效）
- 连抽10张的快速展示动画
- 虚拟伙伴对稀有结果的反应对话

### 3.4 事件系统

#### 事件分类

**随机事件**（随时触发，概率基于 deltaTime）
- 商人来访：以优惠价格购买稀有物品
- 宝箱发现：获得随机资源/道具
- NPC求助：帮助后获得奖励，拒绝可能降低声望
- 灵感闪现：临时加成（如2小时产出翻倍）

**突发事件**（条件触发）
- 敌人入侵：需要防御或损失资源（三国：敌方势力进攻）
- 自然灾害：建筑损坏、资源损失（生存系列：暴风雪）
- 武将叛变：忠诚度过低的武将叛逃（三国系列专属）
- 突发机遇：限时高收益活动

**定期事件**（定时触发）
- 每周Boss：限时挑战，排名奖励
- 每月大赛：PVP模拟，AI对手
- 季节活动：春/夏/秋/冬不同加成
- 节日活动：与现实节日同步

**剧情事件**（进度触发）
- 通关解锁：新区域/新角色/新功能
- NPC加入：新伙伴/新武将加入队伍
- 世界变化：游戏世界因玩家选择而改变

#### 事件选择与后果
- 每个事件提供2-3个选项
- 不同选择影响：资源、声望、NPC关系、后续事件
- 选择有短期和长期后果
- 某些选择会开启/关闭后续事件链

### 3.7 科技发展树系统设计规范

科技发展树（Tech Tree）是放置游戏的核心策略系统，为玩家提供**长期规划感**和**有意义的选择**。科技树通过分支、层级、前置依赖和互斥关系，构建出丰富的策略空间，使每次游戏体验都有所不同。

**设计参考**：
- **Stellaris（群星）**：3大研究域（物理/社会/工程），每域12个子类，400+科技节点，使用"卡牌抽取"机制，有前置依赖和稀有度权重
- **Factorio（异星工厂）**：线性+分支混合，部分科技无限升级，严格的依赖链
- **Civilization（文明系列）**：时代制+分支制，每个时代解锁新科技层，有"尤里卡"加速机制

#### 3.7.1 科技树结构规范

**最低结构要求**（每款游戏必须满足）：

| 指标 | 最低要求 | 推荐值 | 说明 |
|------|----------|--------|------|
| 主分支数量 | ≥4 条 | 4-6 条 | 每条分支代表一个发展领域 |
| 层级深度 | ≥5 层 | 5-8 层 | 从基础到终极的递进深度 |
| 可研究节点总数 | ≥30 个 | 40-80 个 | 不含自动解锁的起始节点 |
| 开叉节点比例 | ≥15% | 20-30% | 一个前置→多个后续的节点 |
| 汇合节点比例 | ≥10% | 15-25% | 需要≥2个前置的节点 |
| 互斥节点组数 | ≥3 组 | 3-6 组 | 选择A则不能选B，增加策略深度 |

**节点关系类型**：

```
┌──────────────┐     ┌──────────────┐
│   前置依赖    │     │    开叉      │
│  A ──→ B     │     │  A ─┬─→ B   │
│  (线性推进)   │     │     └─→ C   │
│              │     │  (一前多后)   │
└──────────────┘     └──────────────┘

┌──────────────┐     ┌──────────────┐
│   汇合依赖    │     │    互斥      │
│  A ──┐       │     │  A ⊗ B      │
│      ├─→ C   │     │ (选A不可B)   │
│  B ──┘       │     │              │
│  (多前一后)   │     │              │
└──────────────┘     └──────────────┘
```

**节点状态流转**：

```
  未解锁(locked) ──前置满足──→ 可研究(unlockable) ──开始研究──→ 研究中(researching)
       ↑                                                          │
       │                                                    研究完成
       │                                                          ↓
       │←──────────── 声望不重置 ──────────────────────  已完成(completed)
```

**科技树层级设计原则**：

| 层级 | 定位 | 研究时间 | 策略作用 | 典型内容 |
|------|------|----------|----------|----------|
| Tier 1 | 基础科技 | 30秒-1分钟 | 建立方向感 | 基础工具、初级技能 |
| Tier 2 | 进阶科技 | 1-3分钟 | 开始分化路线 | 专精方向选择 |
| Tier 3 | 核心科技 | 3-10分钟 | 引入跨分支依赖 | 需要其他分支前置 |
| Tier 4 | 高级科技 | 10-30分钟 | 引入互斥选择 | 二选一/三选一 |
| Tier 5+ | 终极科技 | 30分钟-2小时 | 长期目标 | 变革性效果，声望后保留 |

#### 3.7.2 科技树数据结构

```typescript
/** 科技分支定义 */
interface TechBranch {
  id: string;                       // 分支唯一标识，如 "military"
  name: string;                     // 分支名称，如 "军事"
  description: string;              // 分支描述
  color: string;                    // 分支主色（十六进制），如 "#E74C3C"
  icon: string;                     // 分支图标 sprite key
  tierCount: number;                // 该分支的总层数
}

/** 科技节点 */
interface TechNode {
  id: string;                       // 节点唯一标识，如 "mil_t3_siege"
  name: string;                     // 科技名称，如 "攻城术"
  description: string;              // 科技描述，含效果说明
  branch: string;                   // 所属主分支 ID
  tier: number;                     // 层级 (1-based, 1=最底层)
  position: { x: number; y: number }; // 在科技树画布中的坐标

  // ── 研究消耗 ──
  cost: Record<string, number>;     // 研究消耗，如 { "gold": 5000, "iron": 200 }
  researchTime: number;             // 基础研究时间（毫秒）

  // ── 前置与互斥关系 ──
  prerequisites: string[];          // 前置科技 ID 列表（全部满足才可研究）
  excludes?: string[];              // 互斥科技 ID 列表（研究了本节点则不可研究互斥节点）

  // ── 科技效果 ──
  effects: TechEffect[];            // 科技生效后的效果列表

  // ── 视觉资源 ──
  icon: string;                     // 节点图标 sprite key
  rarity?: 'common' | 'rare' | 'epic' | 'legendary'; // 稀有度（影响视觉表现）
}

/** 科技效果 */
interface TechEffect {
  type: 'unlock' | 'boost' | 'transform' | 'enable';
  // unlock:  解锁新内容（建筑/角色/区域/事件类型）
  // boost:   数值加成（资源产出/战斗力/研究速度等）
  // transform: 转化已有内容（升级建筑形态/改变角色外观等）
  // enable:  启用新机制/新系统（如自动战斗/离线研究等）
  target: string;                   // 目标对象 ID（建筑/资源/角色/系统名）
  value: number;                    // 效果数值（加成百分比/解锁数量等）
  description: string;              // 效果的玩家可见描述
}

/** 科技树整体结构 */
interface TechTree {
  id: string;                       // 科技树 ID（通常等于 gameId）
  branches: TechBranch[];           // 所有分支
  nodes: TechNode[];                // 所有节点
}

/** 科技树运行时状态（存档数据） */
interface TechTreeState {
  completed: Set<string>;           // 已完成研究的节点 ID 集合
  researching: {
    nodeId: string;
    startTime: number;              // 开始研究的时间戳
    progress: number;               // 当前进度 (0-1)
  } | null;
  excluded: Set<string>;            // 因互斥而永久排除的节点 ID 集合
  totalResearched: number;          // 历史已研究总数（含声望前）
}

/** 科技树模块接口 */
interface TechTreeModule {
  tree: TechTree;                   // 静态科技树定义
  state: TechTreeState;             // 运行时状态

  // ── 查询方法 ──
  getNodeState(nodeId: string): 'locked' | 'unlockable' | 'researching' | 'completed' | 'excluded';
  getAvailableNodes(): TechNode[];                      // 当前可研究的节点
  getBranchProgress(branchId: string): number;          // 分支完成度 0-1
  getOverallProgress(): number;                         // 总完成度 0-1
  getActiveEffects(): TechEffect[];                     // 所有已激活效果

  // ── 操作方法 ──
  startResearch(nodeId: string): boolean;               // 开始研究（扣资源、设状态）
  cancelResearch(): boolean;                            // 取消当前研究（退回部分资源）
  completeResearch(nodeId: string): TechEffect[];       // 完成研究（返回生效的效果）
  tick(deltaTime: number): TechNode | null;             // 每帧更新，研究完成时返回节点

  // ── 声望相关 ──
  getPrestigeBonus(): Record<string, number>;           // 声望加成（基于已完成节点数）
  onPrestige(): void;                                   // 声望处理：保留已完成，重置研究中
}
```

**科技效果类型详解**：

| 效果类型 | 说明 | 示例 |
|----------|------|------|
| `unlock` | 解锁新内容 | 解锁"铁匠铺"建筑、解锁"火攻"技能、解锁"北方荒原"地图区域 |
| `boost` | 数值加成 | 粮食产出+20%、研究速度+15%、所有武将攻击+10% |
| `transform` | 转化已有内容 | 农田升级为"灌溉农田"（外观+产出变化）、木剑→铁剑（装备升级） |
| `enable` | 启用新机制 | 启用自动战斗、启用离线研究、启用贸易系统、启用PVP模拟 |

#### 3.7.4 科技树平衡原则

**研究时间与资源消耗曲线**：

| 层级 | 研究时间 | 资源消耗 | 占游戏总进度 | 设计意图 |
|------|----------|----------|-------------|----------|
| Tier 1 | 30秒 - 1分钟 | 低（基础资源） | 快速解锁 | 建立方向感，鼓励尽早选择分支 |
| Tier 2 | 1 - 3分钟 | 中低 | 稳步推进 | 开始分化，引入少量跨分支需求 |
| Tier 3 | 3 - 10分钟 | 中等 | 核心阶段 | 跨分支依赖出现，鼓励多线发展 |
| Tier 4 | 10 - 30分钟 | 中高 | 深度策略 | 互斥选择出现，路线分化明显 |
| Tier 5+ | 30分钟 - 2小时 | 高（稀有资源） | 长期目标 | 变革性效果，声望后保留 |

**平衡设计核心原则**：

1. **前期低成本快速解锁**：Tier 1-2 科技成本低、时间短，让玩家在前 10 分钟内完成 3-5 个科技研究，建立投入感
2. **中期跨分支依赖**：Tier 3 起引入需要其他分支前置的汇合节点，鼓励玩家多线发展而非单线冒进
3. **后期互斥增加策略深度**：Tier 4+ 引入互斥选择，同一局游戏中无法获得所有科技，增加重玩价值
4. **声望保留已研究科技**：声望/转生后**不重置科技树**，已完成的科技永久保留，研究中的科技退还资源。声望奖励可包含"研究速度加成"
5. **不存在"必须全点"的科技**：每条分支都有独立价值，单线发展也能通关，多线发展效率更高但不强制
6. **互斥选择不可逆但影响有限**：互斥科技之间效果强度相当，只是方向不同（如"攻击+30%" vs "防御+30%"），不会因选错而卡关

**研究速度调节机制**：

```typescript
// 研究速度计算公式
actualResearchTime = baseResearchTime
  / (1 + researchSpeedBonus)          // 科技加成
  / (1 + prestigeBonus * 0.1)         // 声望加成（每次声望 +10% 速度）
  / (1 + buildingBonus)               // 建筑加成（如书院等级）
  * (1 - eventDiscount);              // 事件折扣（如"灵感闪现"事件临时 -50% 时间）
```

**科技树与声望系统的协同**：

```typescript
// 声望时科技树的处理逻辑
function onPrestige(techState: TechTreeState): TechTreeState {
  return {
    // ✅ 保留：已完成的科技永久保留
    completed: techState.completed,

    // ❌ 重置：研究中的科技取消，退还部分资源
    researching: null,

    // ✅ 保留：互斥选择永久生效（不可反悔）
    excluded: techState.excluded,

    // ✅ 保留：统计计数
    totalResearched: techState.totalResearched,
  };
}

// 声望加成：基于已完成科技数量
function getPrestigeTechBonus(completedCount: number): Record<string, number> {
  return {
    researchSpeed: Math.floor(completedCount / 5) * 0.05,  // 每5个已完成科技+5%研究速度
    resourceBonus: Math.floor(completedCount / 10) * 0.03,  // 每10个已完成科技+3%全资源产出
  };
}
```

**科技树与虚拟伙伴的交互**：

- **研究开始时**：伙伴给予鼓励（"开始研究火药研制了，这可是改变战局的关键！"）
- **研究完成时**：伙伴称赞反馈（"太好了！火药研制成功，我们的攻城能力大大提升了！"）
- **遇到互斥选择时**：伙伴给出建议（"主公，重税强国和轻徭薄赋各有利弊，需三思而后行..."）
- **卡关提示**：伙伴推荐研究方向（"主公，也许该研究一下攻城术了，下一个关卡城防很高。"）
- **跨分支依赖提示**：伙伴提醒（"要研究火攻赤壁，还需要先完成火药研制哦~"）

#### 叛变系统
- 武将叛变后加入敌方势力

- 叛变武将携带部分资源和兵力
- 可通过特定事件或战役重新捕获/招降
- 叛变武将再次招降后初始忠诚度较低（需重点维护）

- 玩家可以尝试挖角其他势力的武将
- 成功率公式：`成功率 = 基础概率 × (1 - 目标忠诚度/100) × 己方魅力加成 × 关系加成`
- 挖角失败可能降低外交关系
- 被挖角的势力可能发起报复

## 第五章：战斗系统设计

### 5.1 战斗模式分类

#### 即时战斗模式（适合武侠/动作类）
- 角色自动攻击，玩家手动释放技能
- 技能有冷却时间，需要策略性选择释放时机
- BOSS有特殊机制（阶段变化、弱点暴露、狂暴模式）
- 视觉表现：角色动画 + 技能特效 + 伤害数字飘字

**角色动画要求**：
- 待机（idle）：轻微呼吸/摇摆动画
- 攻击（attack）：挥砍/施法动画
- 受伤（hurt）：后仰/闪烁
- 死亡（death）：倒下/消散
- 技能（skill）：每个技能独立动画

#### 回合制战斗模式（适合策略/三国类）
- 自动执行回合，玩家在战前编队 + 战中选择技能
- 速度属性决定行动顺序
- 连击/反击/合击等高级机制
- 视觉表现：左右对阵 + 技能动画 + 状态图标

**回合制特殊机制**：
- 合击：特定武将组合在同一回合触发额外攻击
- 反击：被攻击时有概率反击（武力差越大概率越高）
- 连击：速度高的角色可能一回合行动两次
- 暴击：暴击率由武力/智力决定，暴击伤害×1.5

#### 自动战斗 + 手动介入（通用模式）
- 默认自动战斗，玩家可随时介入操作
- 介入操作：释放终极技能、使用道具、切换阵型
- 自动战斗AI可设置策略：保守（优先防御）/ 平衡 / 激进（优先攻击）

#### 塔防战斗模式（适合军事/植物类）
- 放置防御单位，敌人沿路径进攻
- 波次设计：普通波 → 精英波 → Boss波
- 防御单位可升级、合成
- 路径设计：直线/弯道/多路径

### 5.2 战斗视觉表现规范

| 元素 | 规范 | 备注 |
|------|------|------|
| 角色动画 | 至少4组（待机/攻击/受伤/死亡） | 帧动画，8-16帧 |
| 技能特效 | 粒子系统实现 | 品质越高特效越华丽 |
| 伤害数字 | 飘字效果 | 暴击放大+特殊颜色 |
| 血条 | HP+护甲条 | 颜色区分（绿→黄→红） |
| 战斗场景 | 背景匹配主题 | 至少3层视差滚动 |
| 战斗日志 | 右侧滚动文字 | 可折叠 |
| 胜利/失败 | 特殊动画 | 胜利：金色特效；失败：灰色 |

## 第六章：地图与探索系统

### 6.1 小地图系统

- Canvas实现RPG风格小地图（160×160px缩略图）
- 区域解锁：相邻区域通关后解锁
- 地图标记：当前任务点（!）、NPC位置（头像）、宝箱（箱子图标）、Boss（骷髅图标）
- 迷雾系统：未探索区域被半透明迷雾覆盖
- 快速传送：已解锁区域可快速传送（消耗少量资源）
- 玩家位置：闪烁的亮点标记

### 6.2 区域设计规范

每个区域包含：
- **主题**：独特视觉主题（森林/沙漠/雪山/火山/海底/天空/地下...）
- **怪物**：3-5种普通怪 + 1种精英怪 + 1个区域Boss
- **资源点**：2-3种可采集资源
- **NPC**：1-2个NPC，提供任务或交易
- **隐藏宝箱**：1-2个隐藏宝箱（需要探索发现）
- **事件触发点**：特定位置触发剧情事件

区域难度递增设计：
```
区域1-3：入门级，快速通关，建立信心
区域4-6：需要策略，引入克制机制
区域7-9：需要培养，引入声望系统
区域10+：高难度，需要多次声望+完美培养
```

### 6.3 探索事件

移动时随机触发探索事件：
- **宝箱**：随机资源/道具（概率30%）
- **陷阱**：损失少量HP或资源（概率15%）
- **NPC遭遇**：获得任务或交易机会（概率20%）
- **隐藏Boss**：强力敌人，击败获稀有奖励（概率5%）
- **秘密通道**：通往隐藏区域（概率3%）
- **无事发生**：继续探索（概率27%）

探索完成度：
- 每个区域0%-100%
- 影响因素：击败怪物（+10%）、发现宝箱（+15%）、完成NPC任务（+20%）、发现隐藏区域（+25%）
- 100%完成度获得区域专属奖励

- 无效操作：轻微抖动动画 + 红色闪烁提示

#### 7.4.2 键盘快捷键规范

| 按键 | 功能 | 说明 |
|------|------|------|
| `Space` | 核心操作（点击/采集/攻击） | 最常用的操作键 |
| `1-9` | 快捷选中建筑/角色槽位 | 对应底部快捷栏 |
| `Enter` | 确认/对话推进 | 弹窗确认、NPC对话 |
| `Esc` | 取消/关闭面板/返回 | 通用取消操作 |
| `Tab` | 切换面板（建筑↔角色↔地图↔背包） | 循环切换 |
| `↑↓←→` | 地图滚动 / 列表选择 | 方向键导航 |
| `W/A/S/D` | 地图滚动（游戏模式） | WASD 移动 |
| `B` | 打开建筑面板 | Build |
| `H` | 打开英雄/角色面板 | Hero |
| `M` | 打开大地图 | Map |
| `P` | 暂停/继续 | Pause |
| `R` | 声望/转生确认（需二次确认） | Prestige |
| `S` | 手动存档 | Save |
| `Ctrl+S` | 快速存档 | 强制存档 |
| `F1` | 帮助/操作指南 | 新手教程 |
| `+/-` | 游戏速度调节（1x/2x/3x） | 加速模式 |
| `Q/E` | 切换选中建筑/角色 | 快速切换 |

**键盘操作原则**：
- 所有核心操作必须支持键盘完成，不依赖鼠标
- 快捷键在游戏中全程可用，不因面板状态失效
- 弹窗/模态框中 Enter 确认、Esc 取消
- 危险操作（转生、拆除）需二次确认
- 按键响应延迟 < 50ms

#### 7.4.3 触控操作规范（移动端）

| 操作 | 行为 | 适用场景 |
|------|------|----------|
| **单指点击** | 选中/激活 | 等同鼠标左键 |
| **双指缩放** | 地图缩放 | 地图浏览 |
| **单指滑动** | 地图/列表滚动 | 等同鼠标拖拽 |
| **长按（500ms）** | 上下文菜单 | 等同鼠标右键 |
| **双击** | 快速升级/确认 | 等同鼠标双击 |

#### 7.4.4 游戏内操作模式

不同游戏状态下，操作模式应自动切换：

| 状态 | 鼠标模式 | 键盘模式 | 说明 |
|------|----------|----------|------|
| **主界面** | 点击建筑/角色操作 | 快捷键操作 | 默认状态 |
| **地图浏览** | 拖拽滚动+滚轮缩放 | WASD/方向键移动 | 自由浏览 |
| **战斗中** | 点击选择目标+技能 | 数字键选技能+Enter确认 | 战斗模式 |
| **建造模式** | 拖放建筑到地图格子 | 方向键选位置+Enter放置 | 布局规划 |
| **对话/事件** | 点击选项 | 方向键选择+Enter确认 | 故事推进 |
| **弹窗/模态** | 点击按钮 | Enter/Esc | 确认/取消 |

#### 7.4.5 操作引导与提示

- **首次进入游戏**：显示操作指南覆盖层（可跳过），包含鼠标+键盘操作说明
- **新功能解锁**：首次解锁时显示 3 秒操作提示（如"按 B 打开建筑面板"）
- **快捷键提示**：按钮旁显示对应快捷键标签（灰色小字）
- **操作反馈**：所有操作都有视觉反馈（动画+音效），无效操作有错误提示
- **无障碍**：支持自定义快捷键绑定，支持色盲模式

// 响应式适配工具
class ResponsiveAdapter {
  private canvas: HTMLCanvasElement;
  private baseWidth: number = 640;
  private baseHeight: number = 480;
  
  adapt() {
    const isMobile = window.innerWidth < 768;
    const isLandscape = window.innerWidth > window.innerHeight;
    
    if (!isMobile) {
      // PC宽屏：固定尺寸，CSS缩放
      this.canvas.style.width = '640px';
      this.canvas.style.height = '480px';
    } else if (isLandscape) {
      // 手机横屏

      this.canvas.style.maxWidth = '640px';
    } else {
      // 手机竖屏：全宽，高度自适应
      this.canvas.style.width = '100%';
      this.canvas.style.height = 'auto';
      this.canvas.style.aspectRatio = '4/3';
    }
  }
}

## 第十章：开发流程与质量标准

### 10.1 游戏开发流水线

每款游戏按以下四阶段流水线开发，使用专业 Agent 执行：

#### Phase 1：策划设计（game-designer Agent）

**输入**：游戏ID、主题系列、设计指南
**输出**：游戏设计文档（GDD）

GDD 必须包含：
1. **游戏概述**：名称、主题、一句话描述、独特卖点（USP）
2. **核心玩法**：主模式+辅助模式、核心循环图
3. **资源体系**：资源类型、产出方式、转化链
4. **系统设计**：声望系统、抽卡系统、事件系统（≥10种事件）
5. **角色设计**：角色/武将/英雄列表、属性体系、技能设计
6. **虚拟伙伴**：伙伴设定、性格、对话库（每场景≥3条）
7. **关卡/地图**：区域设计、Boss设计、探索要素
8. **数值规划**：各阶段资源产出曲线、升级费用、声望阈值
9. **视觉风格**：选择的风格类型、配色方案

**审核点**：GDD 必须包含独特卖点（USP），不能与已有游戏雷同

#### Phase 2：美术设计（game-artist Agent）

**输入**：GDD
**输出**：配色方案、UI布局图、角色设定

- 配色方案（主色、辅色、强调色、背景色）
- UI布局草图（PC+手机双版本）
- 角色外观描述（用于Canvas绘制）
- 场景主题描述

**审核点**：视觉风格匹配游戏主题，不使用干扰视觉的效果

#### Phase 3：引擎开发（game-developer Agent）

**输入**：GDD + 美术方案
**输出**：引擎代码（TypeScript）+ 常量文件

**技术规范**：
- 继承 `IdleGameEngine` 基类
- 实现所有核心玩法系统
- 代码量 800-1500 行
- 编译无错误

**关键约束**：

### 10.3 数值平衡原则

#### 阶段性节奏设计

| 阶段 | 时长 | 设计目标 | 节奏 |
|------|------|----------|------|
| 新手期 | 0-30分钟 | 快速上手，建立信心 | 频繁解锁，每1-2分钟一个新功能 |
| 成长期 | 30分钟-1天 | 深入系统，培养习惯 | 每10-30分钟一个升级/解锁 |
| 稳定期 | 1-7天 | 策略思考，长期投入 | 引入声望系统，多路线选择 |
| 成熟期 | 7天+ | 长线目标，收集完成 | 限时活动，稀有收集，极限挑战 |

#### 资源产出曲线
- 前期线性增长（容易理解）
- 中期指数增长（需要策略优化）
- 后期需要声望突破（引入重置循环）

#### 声望数学模型
```
声望奖励 = 基础倍率 × (1 + 0.5 × 声望次数)
每次声望后前期速度提升: 5x ~ 10x
声望间隔建议: 1-3天（首次）→ 3-7天（后续）
```

#### 抽卡经济平衡
```
免费抽卡: 每天1-3次（普通池）
付费抽卡: 高级货币获取速率 → 约每2天可抽1次
保底周期: 50抽保SR ≈ 2-4周；100抽保SSR ≈ 1-2月
```

### 11.3 对话系统技术实现

#### 对话存储结构

```typescript
// 对话库 JSON 结构
interface DialogDatabase {
  companionId: string;
  dialogs: {
    [trigger in DialogTrigger]: DialogEntry[];
  };
}

interface DialogEntry {
  id: string;
  text: string;
  expression: 'normal' | 'happy' | 'sad' | 'excited' | 'worried' | 'thinking';
  priority: number;         // 优先级（越高越优先）
  condition?: (state: any) => boolean;  // 显示条件
  cooldown: number;          // 冷却时间（ms）
  oneTime?: boolean;         // 是否只显示一次
  unlockLevel?: number;      // 伙伴等级要求
}
```

#### 对话调度算法

```typescript
class DialogScheduler {
  private lastShown: Map<string, number> = new Map();
  private shownOnce: Set<string> = new Set();
  
  selectDialog(trigger: DialogTrigger, entries: DialogEntry[]): DialogEntry | null {
    const now = Date.now();
    
    // 过滤：条件满足 + 冷却完成 + 等级达标 + 非一次性已展示
    const available = entries.filter(e => {
      if (e.oneTime && this.shownOnce.has(e.id)) return false;
      if (e.unlockLevel && companion.level < e.unlockLevel) return false;
      if (e.condition && !e.condition(gameState)) return false;
      const lastTime = this.lastShown.get(e.id) || 0;
      if (now - lastTime < e.cooldown) return false;
      return true;
    });
    
    if (available.length === 0) return null;
    
    // 按优先级排序，同优先级随机选择
    available.sort((a, b) => b.priority - a.priority);
    const maxPriority = available[0].priority;
    const topEntries = available.filter(e => e.priority === maxPriority);
    const selected = topEntries[Math.floor(Math.random() * topEntries.length)];
    
    // 标记已展示
    this.lastShown.set(selected.id, now);
    if (selected.oneTime) this.shownOnce.add(selected.id);
    
    return selected;
  }
}
```

#### 对话显示UI

- 以气泡形式显示在游戏界面底部或右下角
- 气泡渐入动画（300ms ease-out）
- 伙伴头像 + 表情变化 + 对话文字
- 玩家可点击关闭，或5秒后自动淡出
