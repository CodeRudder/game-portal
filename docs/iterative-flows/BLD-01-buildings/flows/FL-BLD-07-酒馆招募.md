## 关键要求

- 3种招募类型：普通招募(铜钱) / 高级招募(招贤令+元宝) / 友情招募(友情点)
- 招募概率受酒馆等级、科技、武将智力加成
- 高级招募(Lv6+)每日免费1次；友情招募(Lv11+)解锁
- Lv16+解锁保底机制：每50次高级招募必出稀有武将

### R2 新增约束

- **概率计算公式（乘法叠加）**:
  ```
  final_rate = base_rate × (1 + tavern_bonus%) × (1 + tech_bonus%) × (1 + hero_int_bonus%)
  ```
  示例: 紫色武将 base=16%, 酒馆+20%, 科技+5%, 智力+0% → 16%×1.20×1.05×1.00 = 20.16%
- **保底系统规则**: (1) 保底计数器仅对高级招募生效 (2) 获得紫+品质武将时重置计数 (3) 计数器永久保存，不随酒馆升级/刷新重置 (4) "稀有"定义为紫+橙品质 (5) 每50次未出稀有，第50次保底出紫+品质
- **双击防护**: 招募按钮点击后立即禁用（加载态），动画+结果展示完成后恢复
- **网络失败回滚**: (1) 服务端事务机制: 扣除资源+随机抽取+创建武将实例为原子操作 (2) 网络中断后客户端重连查询招募结果 (3) 服务端确认失败时已扣资源自动返还

### R6 新增约束

- **酒馆等级解锁表**:

  | 等级段 | 解锁功能 | 招募概率加成 | 附加效果 |
  |:------:|---------|:-----------:|---------|
  | Lv1~5 | 普通招募 | +2%~10% | 仅铜钱招募，白色/绿色为主 |
  | Lv6~10 | 高级招募 | +12%~20% | 招贤令+元宝招募，每日免费1次 |
  | Lv11~15 | 友情招募 | +22%~30% | 友情点招募，高级概率提升 |
  | Lv16~20 | 保底机制 | +32%~40% | 每50次高级招募必出紫+品质 |

- **三种招募类型费用表**:

  | 类型 | 费用 | 品质池 | 保底适用 | 解锁条件 |
  |------|------|--------|:--------:|---------|
  | 普通招募 | 💰5,000 铜钱 | 白80% / 绿15% / 蓝4% / 紫1% | 否 | 酒馆 Lv1 |
  | 高级招募 | 🎫×1 招贤令 + 💰50,000 元宝(非免费时) | 蓝60% / 紫30% / 橙10% | 是 | 酒馆 Lv6 |
  | 友情招募 | ❤️×100 友情点 | 白70% / 绿25% / 蓝5% | 否 | 酒馆 Lv11 |

- **品质动画时长表**:

  | 品质 | 动画时长 | 动画效果 | 屏幕范围 |
  |:----:|:-------:|---------|---------|
  | 白色(普通) | 即时(0ms) | 无特效，直接显示 | 卡片区域 |
  | 绿色(良品) | 1.0s | 微光闪烁+卡牌轻微抖动 | 卡片区域 |
  | 蓝色(上品) | 2.0s | 光芒扩散+卡牌翻转 | 内容区域 |
  | 紫色(精品) | 3.0s | 全屏紫色光芒+粒子特效+卡牌升起 | 全屏 |
  | 橙色(传说) | 5.0s | 全屏金光+龙凤特效+震屏+卡牌升起+音效 | 全屏+震动 |

- **免费高级招募规则**: (1) 每日00:00服务端重置 (2) 免费次数=1次/天 (3) 使用免费次数时不消耗招贤令和元宝 (4) 免费次数不累积，当日未使用不保留 (5) 面板显示"今日免费: X/1 次"

### FL-BLD-07-01 进入招募Tab

> **来源**: [UI BLD-2-1 招贤馆详情面板 状态B](../../games/three-kingdoms/ui-design/ui-layout/BLD-buildings.md#bld-2-1)

**用户感知**: 在酒馆详情面板点击[招募]Tab → 内容区淡入淡出切换(150ms) → 看到招募面板（三种招募类型卡片+底部操作按钮）
**系统行为**:
1. 发起数据请求 `GET /api/tavern/recruit/state`（含卡池、概率、保底计数、免费次数、历史记录摘要）
2. 若酒馆等级 < 11：隐藏友情招募区域（或显示为锁定态）

5. 计算并渲染每种类型的实际概率分布（调用乘法叠加公式）

**关键要求**: Tab切换时立即显示上次缓存的招募状态（乐观UI），同时发起新请求刷新。若数据变化（如免费次数已重置），用300ms过渡动画更新

**验收标准**: Tab切换流畅（< 200ms可交互），按钮联动正确，概率数据准确

**📱 空间要点**:

- 底部按钮区: 3个按钮各 120x44px，间距12px

**⏱ 时间要点**: 数据请求 200-500ms（首次）/ < 100ms（缓存命中）。Tab切换动画150ms。总耗时 < 300ms可交互

### FL-BLD-07-02 查看招募类型和概率

> **来源**: [PRD BLD-6 酒馆核心功能](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-6) / [UI 招募内容](../../games/three-kingdoms/ui-design/ui-layout/BLD-buildings.md#bld-2-1)

#### 07-02a 三种招募类型概览

**用户感知**: 看到1~3种招募类型卡片（视酒馆等级解锁情况）→ 每种卡片显示：招募名称、费用、品质概率分布条（白/绿/蓝/紫/橙色段占比）、免费次数标签（高级招募）、保底进度条（高级招募Lv16+）

**系统行为**:
1. 根据酒馆等级决定显示哪些招募类型
2. 调用概率计算公式为每种类型计算实际概率:
   ```
   final_rate = base_rate × (1 + tavern_bonus%) × (1 + tech_bonus%) × (1 + hero_int_bonus%)
   ```
3. 渲染概率分布条（水平堆叠条形图，各品质颜色段占比 = final_rate%）
4. 高级招募卡片额外显示：
   - "今日免费: X/1 次"标签（绿色=可用，灰色=已用）
   - 保底进度: "距保底: X/50 次"进度条（Lv16+时显示）
5. 卡片下方小字显示加成来源（点击可展开详细明细 → 07-02b）

**关键要求**:
- 概率分布条的总和必须=100%（白色段 = 100% - 其他品质之和）
- 高级招募的概率分布条不含白色（只有蓝/紫/橙三段）
- 友情招募的概率分布条不含紫色和橙色（只有白/绿/蓝三段）
- 未解锁的类型显示为锁定态：半透明灰色蒙版 + "酒馆LvXX解锁" + 🔒图标

**验收标准**: 概率数值准确（含加成后），总和100%，未解锁类型正确显示锁定态

**📱 空间要点**:
- 概率分布条: 360x24px，圆角4px，各品质段无间隙
- 品质色值: 白=#ccc / 绿=#27ae60 / 蓝=#3498db / 紫=#9b59b6 / 橙=#e67e22
- 保底进度条: 360x8px，金色进度条 (#c8a96e)

#### 07-02b 概率详情Tooltip

**用户感知**: 点击任意招募类型卡片上的概率分布条 → 弹出Tooltip浮层 → 显示完整概率明细

**系统行为**:
1. 渲染Tooltip浮层，内容包含:
   - 标题: "XX招募概率明细"
   - 每品质行: 品质图标 + 品质名 + 基础概率 + 加成后概率 + 加成来源分解
   - 加成来源分解:
     ```
     紫色武将: 基础16%
       × 酒馆加成(1+20%) = 19.20%
       × 科技加成(1+5%)  = 20.16%
       × 智力加成(1+0%)  = 20.16%
       → 实际概率: 20.16%
     ```
   - 底部: "总计: 100.00%"
2. Tooltip 定位: 卡片上方居中，最大宽度 380px
3. 点击Tooltip外部区域或再次点击概率条关闭

**关键要求**: 所有概率显示保留2位小数。加成计算必须展示乘法叠加过程（不是简单加法）

**验收标准**: Tooltip显示正确的加成分解，关闭操作流畅

**📱 空间要点**:
- Tooltip: 380x auto px，最大高度280px（超出可滚动）
- 背景: rgba(20,25,35,0.95)，边框1px solid #c8a96e
- 每品质行高: 32px
- 手机端: Tooltip改为底部Bottom Sheet（40%高度），方便阅读

**⏱ 时间要点**: Tooltip弹出动画 120ms（scale 0.95→1.0 + opacity 0→1）

### FL-BLD-07-04 校验招募条件

> **来源**: [PRD BLD-6 招募类型](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-6)

#### 07-04a 前端资源预校验

**用户感知**: 无感知（后台即时校验，< 50ms）

**系统行为**:
1. 前端即时校验（按优先级顺序）:
   - (1) 酒馆等级是否满足该招募类型要求
   - (2) 资源是否充足（铜钱≥5000 / 招贤令≥1 或免费次数 / 友情点≥100）
   - (3) 高级招募免费次数是否可用（检查本地缓存的免费计数）
   - (4) 卡池是否有可招募武将（本地缓存的卡池数量>0）
2. 前端校验通过 → 进入07-04b服务端校验
3. 前端校验失败 → 直接显示错误反馈，不发起服务端请求

**关键要求**: 高级招募每日免费1次时，校验优先使用免费次数。免费次数仅消耗免费计数，不扣除招贤令和元宝

**验收标准**: 前端校验覆盖所有条件，无遗漏

**⏱ 时间要点**: 即时校验 < 50ms

#### 07-04b 服务端完整校验

**用户感知**: 按钮显示加载态（旋转图标），等待服务端响应

**系统行为**:
1. 发起 `POST /api/tavern/recruit/validate`，请求体:
   ```json
   { "tavernId": "xxx", "recruitType": "normal|advanced|friend" }
   ```
2. 服务端校验（按优先级）:
   - (1) 玩家版本号（optimistic lock）
   - (2) 酒馆等级匹配招募类型
   - (3) 资源余额充足
   - (4) 免费次数状态（高级招募）
   - (5) 卡池非空
   - (6) 每日招募次数上限（防刷限制：普通100次/天，高级50次/天，友情100次/天）
3. 服务端返回:
   - 成功: `{ "valid": true, "cost": { ... }, "freeUsed": false }`
   - 失败: `{ "valid": false, "reason": "INSUFFICIENT_RESOURCE", "detail": { ... } }`

**关键要求**: 服务端校验为最终判定。前端缓存状态可能与服务端不一致时，以服务端为准

**验收标准**: 服务端校验覆盖所有维度，返回详细错误信息

**⏱ 时间要点**: 服务端校验响应 100-300ms。超时5s按网络失败处理（→ 07-05b）

**异常分支**:

| 条件 | 处理方式 | 转至步骤 |
|------|---------|:--------:|
| 铜钱不足(普通) | Toast"铜钱不足，还需要💰XX" + 显示差额和[前往市集]按钮 | 07-02 |
| 招贤令不足(高级) | Toast"招贤令不足" + [前往商店]按钮 | 07-02 |
| 元宝不足(高级-非免费) | Toast"元宝不足" + [充值]按钮 | 07-02 |
| 友情点不足(友情) | Toast"友情点不足，还需要❤️XX" + 显示差额 | 07-02 |
| 酒馆等级不足 | 按钮置灰+Tooltip"酒馆LvXX解锁" | 07-02 |
| 卡池为空 | Toast"当前无可招募武将，请等待卡池刷新" | 07-02 |
| 每日招募次数耗尽 | Toast"今日XX招募次数已用完，明日00:00刷新" | 07-02 |
| 版本冲突(409) | 强制刷新玩家状态，重新校验 | 07-04b |

### FL-BLD-07-05 执行招募

> **来源**: [PRD BLD-6 招募概率](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-6)

#### 07-05a 服务端招募事务

**用户感知**: 按钮立即进入禁用态（加载旋转图标）→ 等待招募结果

**系统行为**:
1. 客户端发起 `POST /api/tavern/recruit/execute`:
   ```json
   {
     "tavernId": "xxx",
     "recruitType": "normal|advanced|friend",
     "useFree": false,
     "clientVersion": "abc123"
   }
   ```
2. 服务端执行原子事务（6步，任一失败全部回滚）:
   - Step 1: 乐观锁检查（player_version匹配）
   - Step 2: 扣除资源（铜钱/招贤令+元宝/友情点，免费时不扣资源）
   - Step 3: 判断是否触发保底（高级招募：计数器+1，若>=50则强制紫+品质）
   - Step 4: 根据概率随机抽取武将品质（乘法叠加公式计算实际概率）
   - Step 5: 在对应品质池中随机选取具体武将（已拥有武将可重复获得，转为碎片/觉醒材料）
   - Step 6: 创建武将实例（或碎片）→ 写入英雄系统 → 更新保底计数器 → 更新免费次数
3. 服务端返回:
   ```json
   {
     "success": true,
     "hero": { "id": "xxx", "name": "关羽", "rarity": "orange", "attrs": {...} },
     "isDuplicate": false,
     "pityCounter": 3,
     "freeUsed": true,
     "cost": { "copper": 0, "scroll": 0, "gem": 0, "friendship": 0 },
     "newVersion": "def456"
   }
   ```
4. 客户端收到成功响应 → 更新本地缓存（资源余额、保底计数、免费次数）→ 进入07-06招募动画

**关键要求**:
- 抽取过程不可中断，按钮在动画+结果展示完成前保持禁用
- 已拥有武将重复获得: 转化为"武将碎片×30"（觉醒材料），结果展示中标注"[已拥有] → 转化为碎片×30"
- 保底触发时: 保底计数器重置为0，结果标记 `"pityTriggered": true`
- 免费次数使用后: 服务端标记今日已用，客户端更新免费标签为灰色

**验收标准**: 招募结果符合概率分布，保底机制正确触发，资源正确扣除，重复武将正确处理

**⏱ 时间要点**:
- 服务端事务: 200-500ms
- 客户端收到响应后立即播放动画（不等额外处理）
- 总计: 用户点击到动画开始 < 600ms

#### 07-05b 网络失败3级处理

**用户感知**: 点击招募后无响应/加载超时 → 根据失败级别不同反馈

**系统行为**（3级递进）:

**L1 - 自动重试（网络波动）**:
- 触发: 请求超时(>5s) 或 网络错误(无响应)
- 行为: 自动重试3次，间隔 1s / 2s / 3s（指数退避）
- 用户感知: 按钮保持加载态，Toast"网络波动，正在重试..."
- 成功恢复: 正常进入07-06动画

**L2 - 结果查询（连接恢复）**:
- 触发: L1重试3次全部失败，但服务端事务可能已成功执行
- 行为: 客户端发 `GET /api/tavern/recruit/last-result` 查询最近一次招募结果
- 服务端: 返回最后一次招募结果（成功/失败/无记录）
- 用户感知:
  - 有成功结果 → 直接跳转到07-06结果展示（跳过动画）
  - 确认失败 → Toast"招募失败，资源已返还" → 恢复按钮
  - 无记录 → 进入L3

**L3 - 强制刷新（数据不一致）**:
- 触发: L2查询返回"无记录"或L2查询也失败
- 行为: 强制刷新玩家完整状态 `GET /api/player/state`
- 用户感知: 全屏加载遮罩"正在同步数据..."（最大3s）→ 刷新完成后恢复招募面板
- 数据恢复: 对比前后资源余额，判断招募是否已执行（资源已扣 = 招募成功，展示结果；资源未扣 = 招募未执行，恢复按钮）

**关键要求**: L1/L2/L3 任一级成功恢复后，按钮状态必须正确（禁用→恢复）。资源不可丢失也不可双重扣除

**验收标准**: 网络中断不丢失资源，不漏展示结果，不双重扣除

### FL-BLD-07-06 招募动画与结果展示

> **来源**: [PRD BLD-6 武将招募](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-6)

#### 07-06a 品质动画

**用户感知**: 根据抽中武将品质，播放不同级别的招募动画:

| 品质 | 时长 | 用户感知 |
|:----:|:----:|---------|
| 白色 | 0ms | 无动画，直接显示结果（07-06b） |
| 绿色 | 1.0s | 卡片区域微光闪烁(白色脉冲300ms) → 卡牌轻微抖动(200ms) → 翻转揭示(500ms) |
| 蓝色 | 2.0s | 内容区光芒从中心扩散(600ms) → 卡牌从背面翻转到正面(800ms) → 品质光晕稳定(600ms) |
| 紫色 | 3.0s | 全屏紫色遮罩淡入(300ms) → 紫色粒子从四周汇聚到中心(1200ms) → 卡牌从底部升起+翻转(1000ms) → 紫色光环扩散(500ms) |
| 橙色 | 5.0s | 全屏金光爆炸(500ms) → 龙凤图案从两侧飞入交汇(1000ms) → 屏幕震动(200ms, 幅度4px) → 卡牌从底部升起+金色光柱(2000ms) → 金色粒子散落+音效(1300ms) |

**系统行为**:
1. 根据服务端返回的品质决定动画类型
2. 播放期间: 所有UI交互屏蔽（点击无效），按钮保持禁用
3. 动画资源预加载: 进入招募Tab时预加载白色/绿色/蓝色动画资源；紫色/橙色动画按需加载（文件较大，~200KB/个）
4. 动画资源加载失败: 降级为白色品质的即时显示（0ms），Toast"特效加载失败，跳过动画"
5. 手机端: 动画时长压缩20%（绿色0.8s / 蓝色1.6s / 紫色2.4s / 橙色4.0s），震屏关闭（改为闪烁）

**关键要求**: 动画不可跳过。动画期间不可进行其他操作。动画完成后自动进入07-06b

**验收标准**: 每种品质动画效果和时长正确，手机端压缩正确，资源加载失败有降级处理

**📱 空间要点**:
- 白色/绿色/蓝色: 限内容区（420x520px）
- 紫色/橙色: 全屏覆盖（使用固定定位，z-index: 50）
- 橙色震屏: transform translate 震动，不影响其他DOM

**⏱ 时间要点**: 参见品质动画时长表。动画完成后0ms延迟自动进入07-06b

#### 07-06b 武将卡片揭晓

**验收标准**: 卡片信息完整准确，重复武将正确标记，跳转正确

## 酒馆等级效果详解

### 等级解锁对招募的影响

| 酒馆等级 | 可用招募类型 | 保底机制 | 免费高级招募 | UI变化 |
|:--------:|:----------:|:-------:|:-----------:|-------|
| Lv1~5 | 普通招募 | 无 | 无 | 仅显示普通招募卡片，高级/友情区域锁定 |
| Lv6~10 | 普通+高级 | 无 | 每日1次 | 高级招募卡片解锁，显示免费标签，保底条不显示 |
| Lv11~15 | 普通+高级+友情 | 无 | 每日1次 | 友情招募卡片解锁，3种全部可见 |
| Lv16~20 | 全部 | 每50次必出紫+ | 每日1次 | 高级招募卡片显示保底进度条 |

### 升级触发招募面板刷新

- 酒馆升级完成时，若当前在招募Tab:
  - 新解锁的招募类型从锁定态变为可用态（解锁动画: 锁图标碎裂200ms + 卡片淡入300ms）
  - 新解锁的保底机制显示保底进度条（从0开始）
- 酒馆升级完成时，若当前在概览Tab:
  - 概览Tab显示红点提示"新功能已解锁"
  - 切换到招募Tab时自动展示解锁动画
