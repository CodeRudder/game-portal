# FL-BLD-06 升级队列管理

> **版本**: v4 | **最后更新**: 2026-05-05
> **归属功能**: 建筑系统 — 建筑操作 (Phase B)
> **触发条件**: 玩家点击右上角升级队列悬浮窗（PC）/ 顶部队列条（手机）
> **结束条件**: 队列操作完成（查看/加速/取消/配置自动升级），回到网格浏览
> **前置状态**: 至少有1个升级任务进行中或排队中（查看空队列也可触发）
> **后置状态**: 队列状态已更新（加速/取消/配置已执行）
> **来源文档**: [PRD BLD-4 建筑队列管理](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-4)
> **关联流程**: [FL-BLD-03 建筑升级](./FL-BLD-03-建筑升级.md)（升级创建）/ [FL-BLD-02 建筑网格浏览](./FL-BLD-02-建筑网格浏览.md)（网格状态）

## 关键要求

- 队列槽位数随主城等级增长：Lv1-5: 1个, Lv6-10: 2个, Lv11-20: 3个, Lv21-30: 4个
- 当前升级完成后，队列中下一项自动开始
- 取消升级返还80%已消耗资源
- 加速选项：铜钱加速 / 天命加速 / 元宝秒完成

### R2 新增约束

- **加速公式**:
  - 铜钱加速: 费用 = 剩余时间(秒) x 10铜钱, 减少30%剩余时间
  - 天命加速: 消耗天命点(每日任务/成就获取), 减少50%剩余时间。天命点获取: 每日任务+10, 成就+50
  - 元宝秒完成: 费用 = 剩余时间(秒) x 1元宝, 立即完成升级
- **自动升级策略**: 当队列出现空槽时触发检查（非定时轮询）。优先级: 等级最低的建筑优先 -> 资源消耗最少的优先。离线期间不触发自动升级。玩家可配置排除列表（不自动升级的建筑）
- **离线队列结算**: 服务端模拟队列按序执行，离线效率系数1.2x应用于每个升级任务，依次结算到离线时间耗尽

### R3 新增约束

- **队列项状态机**: waiting(等待中) -> active(升级中) -> completing(完成中) -> completed(已完成)
- **多任务顺序执行**: 多个队列任务按添加顺序依次执行。同一时刻每个活跃槽位只有1个active任务
- **离线期间队列不新增**: 离线期间不触发自动升级，但已有队列任务继续执行

### R4 新增约束

- **队列槽位扩展流程**: 主城升级完成时立即检查是否跨越槽位阈值。新槽位立即可用，无需额外操作。若有等待中的队列任务，自动分配到新槽位开始执行。扩展时播放"新槽位解锁"动画（蓝色光圈扩散300ms）
- **取消升级详细流程**: 区分"升级中(active)"和"等待中(waiting)"两种状态的取消行为。active取消：立即停止升级进度，返还80%资源，建筑恢复原状态。waiting取消：从队列移除，返还100%资源（尚未开始消耗）
- **加速升级详细流程**: 三种加速方式各自的校验、扣费、进度更新完整流程。加速后进度条平滑过渡（从当前位置线性动画到新位置，300ms）。多次加速叠加计算（每次基于当前剩余时间重新计算费用）
- **自动升级详细流程**: 触发条件（队列空槽+开关开启+有可升级建筑+非离线）。建筑优先级排序算法。排除列表配置UI。资源不足时的跳过逻辑。自动升级操作的Toast反馈
- **离线队列结算详细流程**: 服务端按序模拟每个升级任务。1.2x效率系数应用方式。结算结果推送。多任务级联完成。资源不足时的处理

## 验收标准

- 队列正确显示当前升级和排队中的任务
- 加速后剩余时间正确减少
- 取消后资源正确返还（80% for active / 100% for waiting）
- 自动升级开关正确运作
- 队列槽位随主城等级正确扩展
- 离线队列结算正确应用1.2x效率
- 队列项状态正确流转

### FL-BLD-06-01 查看队列悬浮窗

> **来源**: [UI BLD-1 升级队列](../../games/three-kingdoms/ui-design/ui-layout/BLD-buildings.md#bld-1)

**用户感知（see->act->feedback）**:
- **See**: 右上角悬浮窗显示当前升级任务（建筑名+目标等级+剩余时间）。无升级任务时悬浮窗自动隐藏
- **Act**: 可查看倒计时；点击悬浮窗展开队列详情(06-02)；直接点击[加速]或[取消]按钮
- **Feedback**: 倒计时每秒更新；完成后自动切换到下一项任务；新任务加入时悬浮窗内容更新（fade过渡150ms）

**系统行为**:
1. 实时更新队列倒计时（每秒刷新）
2. 当前任务完成时：
   - 播放完成动画（任务名淡出，100ms）
   - 自动切换到下一项任务（新任务淡入，150ms）
   - 若无下一项任务，悬浮窗自动隐藏（向上滑出，200ms）
3. 新任务加入队列时：悬浮窗内容更新（若显示的不是新任务则不影响，若队列从空变为有任务则悬浮窗出现——向下滑入200ms）
4. 多槽位同时运行时：悬浮窗显示"当前N项升级中"，点击展开查看全部

**队列悬浮窗状态**:

| 状态 | 显示内容 | 行为 |
|------|---------|------|
| 0个任务 | 悬浮窗隐藏 | 不可见 |
| 1个active任务 | 建筑名+目标等级+倒计时+[加速][取消] | 每秒更新 |
| 2+个任务（多槽位） | "N项升级中 (点击查看)" | 点击展开详情 |
| 任务刚完成 | 完成->下一项过渡动画（250ms） | 自动切换 |

**关键要求**: 无升级任务时悬浮窗自动隐藏。倒计时精确到秒。多任务时显示数量+引导展开

**时间要点**:
- 倒计时刷新: 每秒1次
- 悬浮窗出现/隐藏动画: 200ms（向下滑入/向上滑出）
- 任务切换过渡: 250ms（旧任务淡出100ms + 新任务淡入150ms）

**空间要点**:
- PC悬浮窗: 右上角280x80px, z-index:30，内容：建筑图标(24x24)+名称+目标等级+倒计时+[加速][取消]按钮
- 手机固定条: 顶部32px高，横向紧凑显示：农田->Lv.6 02:30 [加速]
- 按钮尺寸: [加速]40x28px蓝色, [取消]40x28px灰色

**异常分支**:

| 条件 | 处理方式 | 转至步骤 |
|------|---------|:--------:|
| 悬浮窗数据与服务端不一致 | 以服务端为准，刷新悬浮窗内容 | 06-01 |
| 客户端崩溃重启 | 重新加载时从`GET /api/queue`获取队列状态，重建悬浮窗 | 06-01 |
| 悬浮窗遮挡重要UI元素 | 悬浮窗可拖拽移动（PC端），记住位置（会话内） | 06-01 |

### FL-BLD-06-03 加速升级

> **来源**: [PRD BLD-4 加速选项](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-4)

**用户感知（see->act->feedback）**:
- **See**: 点击[加速]按钮 -> 出现加速选项面板（3种方式），每种显示费用和效果
- **Act**: 选择一种加速方式 -> 点击对应按钮
- **Feedback**: 扣除加速费用 -> 进度条平滑跳转到新位置（300ms线性动画）-> 倒计时更新 -> Toast"加速成功，剩余时间减少XXX"

**加速方式详细规格**:

#### 方式1: 铜钱加速

| 参数 | 值 |
|------|------|
| 费用公式 | 剩余时间(秒) x 10铜钱 |
| 效果 | 减少30%剩余时间 |
| 上限 | 无上限（可多次使用） |
| 适用 | 所有定时升级任务 |

**计算示例**: 剩余300秒 -> 费用=300x10=3,000铜钱 -> 新剩余=300x0.7=210秒 -> 减少90秒

**系统行为**:
1. 客户端计算费用并预显示
2. 发送`POST /api/queue/{task_id}/speedup`，body: `{ method: "copper", amount: 3000 }`
3. 服务端校验铜钱余额 -> 扣除铜钱 -> 减少剩余时间（更新duration或remaining_time）
4. 返回新的剩余时间
5. 客户端更新进度条（平滑过渡300ms）+ 倒计时

#### 方式2: 天命加速

| 参数 | 值 |
|------|------|
| 费用 | 1天命点/次 |
| 效果 | 减少50%剩余时间 |
| 上限 | 每日最多3次（可配置） |
| 适用 | 所有定时升级任务 |

**天命点获取来源**:

| 来源 | 获取量 | 频率 |
|------|--------|------|
| 每日任务完成 | +10 | 每日 |
| 成就达成 | +50 | 一次性/里程碑 |
| 活动奖励 | +5~20 | 活动期间 |

**系统行为**:
1. 校验天命点余额 >= 1 且 今日已用次数 < 3
2. 发送`POST /api/queue/{task_id}/speedup`，body: `{ method: "destiny", points: 1 }`
3. 服务端扣除天命点 + 记录今日使用次数 + 减少50%剩余时间
4. 返回新的剩余时间 + 剩余天命点 + 今日剩余次数

#### 方式3: 元宝秒完成

| 参数 | 值 |
|------|------|
| 费用公式 | 剩余时间(秒) x 1元宝 |
| 效果 | 立即完成升级 |
| 上限 | 无上限 |
| 适用 | 所有定时升级任务 |

**计算示例**: 剩余300秒 -> 费用=300元宝 -> 立即完成

**系统行为**:
1. 校验元宝余额 >= 剩余秒数
2. 发送`POST /api/queue/{task_id}/speedup`，body: `{ method: "gem", amount: 300 }`
3. 服务端扣除元宝 -> 立即完成升级 -> 更新建筑等级+产出 -> 释放队列槽位
4. 返回完成结果（新等级、新产出数据）
5. 客户端播放升级完成动画（同FL-BLD-03-05）

**多次加速叠加规则**:

- 加速选项面板手机: Bottom Sheet 50%
- 每个加速选项行: 100px高度

| 天命点不足 | 天命行标红"天命点不足"+[天命加速]按钮置灰 | 06-03 |
| 天命加速今日已用3次 | 天命行标灰"今日次数已用完"+按钮置灰 | 06-03 |
| 加速请求失败（网络） | Toast"加速失败，请重试"（3秒），进度条不变 | 06-03 |
| 加速期间升级刚好完成 | 加速请求返回"任务已完成"，客户端播放完成动画，退还加速费用 | 06-02 |
| 多次快速点击加速 | 300ms冷却间隔防重复点击，期间按钮禁用 | 06-03 |
| 即时升级（<=5秒）不显示加速按钮 | 按钮不显示（即时升级无升级中阶段） | -- |

### FL-BLD-06-04 取消升级

> **来源**: [PRD BLD-4 取消升级](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-4)

**用户感知（see->act->feedback）**:
- **See**: 点击[取消]按钮 -> 弹出确认提示"取消升级将返还N%资源"
- **Act**: 点击[确认取消] -> 执行取消操作
- **Feedback**: 升级取消 -> Toast"升级已取消，返还XX资源" -> 建筑恢复原状态 -> 队列更新

**取消行为按状态区分**:

#### 取消 active 状态任务（升级中）

**系统行为**:
1. 弹出确认弹窗：
   - 标题："取消XX升级？"
   - 内容："已消耗资源将返还80%（损失20%）：粮草-N 铜钱-N ..."
   - 按钮：[继续升级]（蓝色，默认）+ [确认取消]（红色）
2. 点击[确认取消]后：
   - 发送`POST /api/queue/{task_id}/cancel`
   - 服务端计算返还资源 = 消耗资源 x 80%（取整：向下取整）
   - 服务端：取消任务 + 返还资源 + 恢复建筑状态（等级不变，仍为升级前等级）+ 释放队列槽位
   - 返回：`{ refunded_resources, building_state }`
3. 客户端处理：
   - 资源栏数字增加动画（+200ms）
   - 建筑卡片从"升级中"状态恢复为"正常"状态（边框颜色渐变300ms）
   - 进度条消失（200ms收起动画）
   - 队列悬浮窗更新（移除该任务）
   - 若有等待任务，500ms后自动开始下一个

**确认弹窗布局**:


### FL-BLD-06-05 自动升级配置与管理

> **来源**: [PRD BLD-4 自动升级](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-4)

- **See**: 队列面板底部显示[自动升级]开关（当前状态）
- **Act**: 切换开关开启/关闭 -> 点击[配置排除列表]管理不自动升级的建筑
- **Feedback**: 开关切换后Toast"自动升级已开启/关闭"；自动升级执行时Toast"自动升级: XX->Lv.YY"

**系统行为**:

#### 自动升级开关

1. 开关切换：
   - 发送`POST /api/settings/auto-upgrade`，body: `{ enabled: true/false }`
   - 服务端持久化开关状态
   - 开启时立即触发一次队列空槽检查
   - 关闭时不影响已排队的自动升级任务

2. 触发条件检查（队列空槽时触发）：
   ```
   触发条件（全部满足才执行）:
   1. 自动升级开关 = 开启
   2. 队列有空闲槽位
   3. 玩家在线（非离线状态）
   4. 存在满足升级条件的建筑
   5. 玩家资源足够升级该建筑
   ```

#### 自动升级建筑选择算法

```

  - 所有已建造且不在升级中的建筑
  - 排除排除列表中的建筑
  - 排除等级已达上限的建筑
  - 排除等级已达主城等级上限的建筑

步骤2: 按优先级排序
  排序规则（按优先级从高到低）:
  1. 等级最低的建筑优先（升基础产出）
  2. 同等级时，资源消耗最少的优先
  3. 同消耗时，升级时间最短的优先
  4. 同时间时，按建筑固定优先级: 农田>市集>矿场>伐木场>兵营>书院>工坊>医馆>城墙>酒馆

步骤3: 从排序结果中选择第一个资源足够的建筑
  - 检查粮草+铜钱+特殊资源是否满足
  - 若不够，跳过，检查下一个
  - 若所有建筑资源都不够，不执行自动升级（不提示）

步骤4: 执行自动升级
  - 自动扣除资源
  - 创建升级任务（占用空槽位）
  - Toast"自动升级: XX -> Lv.YY"
  - 重新检查是否还有空槽位（循环，直到无空槽或无符合条件的建筑）
```

### FL-BLD-06-06 队列槽位扩展与多任务管理

> **来源**: [PRD BLD-4 建筑队列管理](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-4)

**用户感知（see->act->feedback）**:
- **See**: 主城升级完成时（跨越阈值等级），队列面板显示新槽位解锁动画
- **Act**: 无需操作（自动扩展）
- **Feedback**: 新槽位出现动画（蓝色光圈扩散300ms）+ Toast"新队列槽位已解锁！" + 若有等待任务自动开始

**系统行为**:

#### 队列槽位扩展规则

| 主城等级变化 | 槽位数变化 | 扩展时机 |
|:----------:|:---------:|---------|
| ->Lv6 | 1->2 | 主城升至Lv6时立即生效 |
| ->Lv11 | 2->3 | 主城升至Lv11时立即生效 |
| ->Lv21 | 3->4 | 主城升至Lv21时立即生效 |

**扩展流程**:
1. 主城升级完成（在FL-BLD-03-05中触发）
2. 服务端检查主城新等级是否跨越槽位阈值
3. 若跨越：
   - 服务端扩展槽位数
   - 返回新的槽位数
4. 客户端处理：
   - 队列悬浮窗/面板显示新槽位（蓝色光圈扩散动画300ms）
   - Toast"新队列槽位已解锁！当前N个槽位"（3秒）
   - 若有等待中的队列任务，自动分配到新槽位开始执行
   - 若自动升级开关开启且有空槽，触发自动升级检查

#### 多任务管理

**多任务执行规则**:
1. 每个活跃槽位独立运行一个active任务
2. 等待中的任务按添加时间排序（先进先出FIFO）
3. 任一active任务完成后：
   - 释放该槽位
   - 从等待队列中取出第一个任务分配到该槽位
   - 开始执行（延迟500ms）
4. 添加新升级任务时：
   - 优先分配到空闲槽位（即时开始）
   - 无空闲槽位则加入等待队列尾部
   - 一键满级产生的多个任务按级顺序排队

**队列项状态机详细流转**:

```
                    创建任务
                       |
                       v
                  +---------+
          +------>| waiting |<--------+
          |       +---------+         |
          |          |                |
          |   槽位空闲时自动开始       |
          |          |                |
          |          v                |
          |     +---------+           |
          |     | active  |           |
          |     +---------+           |
          |       /    \              |
          |   加速完成  取消           |
          |      /       \            |
          |     v         v           |
          | +----------+ +------+    |
          | |completing| | 取消 |    |
          | +----------+ +------+    |
          |      |                    |
          |  完成动画结束              |
          |      |                    |
          |      v                    |
          | +-----------+             |
          | | completed |             |
          | +-----------+             |
          |      |                    |
          |   移出队列                 |
          +------+                    |
```

**状态持续时间**:

| 状态 | 持续时间 | 说明 |
|------|---------|------|
| waiting | 不定（直到有空闲槽位） | 等待中 |
| active | 升级时间（5秒~数小时） | 升级中 |
| completing | 500ms | 完成动画 |
| completed | 0ms（立即移出） | 已完成 |

**关键要求**: 槽位扩展立即可用。多任务并行独立运行。等待队列FIFO。新任务优先使用空槽

**时间要点**:
- 槽位扩展动画: 300ms（蓝色光圈扩散）
- 任务开始延迟: 500ms（从waiting->active）
- completing动画: 500ms

**空间要点**:
- 新槽位动画: 蓝色光圈从中心扩散，直径0->100px，300ms
- 队列面板显示: 每个槽位80px高度（active）或48px（空闲）

**异常分支**:

| 条件 | 处理方式 | 转至步骤 |
|------|---------|:--------:|
| 扩展后所有等待任务资源不足 | 跳过资源不足的任务，依次检查 | 06-05 |
| 扩展时多个任务同时可开始 | 按等待队列顺序依次开始，间隔500ms | 06-06 |
| 降级（如通过特殊道具降主城等级） | 队列槽位不减少（已获得的槽位保留），但不新增 | -- |

### FL-BLD-06-07 离线队列结算

> **来源**: [PRD BLD-4 离线建造进度](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-4)

**用户感知（see->act->feedback）**:
- **See**: 玩家上线后，弹出离线结算弹窗，显示离线期间完成的升级+进行中的升级
- **Act**: 点击[确认]关闭弹窗
- **Feedback**: 弹窗显示离线结算详情 -> 确认后数据生效 -> 建筑网格刷新

**系统行为**:

#### 离线结算流程

```
步骤1: 玩家上线
  - 客户端发送`POST /api/offline/settle`
  - 服务端计算离线时长 = now() - last_online_time

步骤2: 服务端模拟队列执行
  - 获取离线开始时的队列状态（active任务+waiting任务）
  - 按顺序模拟每个任务的执行:
    a. 当前active任务的剩余时间 / 1.2（离线效率系数）
      = 实际完成所需离线时间
    b. 若剩余离线时间 >= 完成所需时间:
      - 标记该任务为completed
      - 更新建筑等级+产出
      - 从waiting队列取下一个任务（若有）
      - 继续模拟
    c. 若剩余离线时间 < 完成所需时间:
      - 任务仍在active状态
      - 更新进度 = (已用离线时间 * 1.2) / duration
      - 停止模拟

步骤3: 1.2x效率系数应用方式
  - 效率系数仅应用于升级时间（不应用于资源消耗）
  - 实际升级时间 = 原始升级时间 / 1.2
  - 即：原本需要100分钟的升级，离线期间只需100/1.2 = 83.3分钟
  - 效率系数不影响资源产出（资源产出另有离线公式）

步骤4: 生成结算报告
  - 已完成的升级列表（建筑名+原等级->新等级）
  - 仍在进行中的升级（进度百分比+剩余时间）
  - 等待中未开始的升级（排队顺序）
```

**离线结算弹窗布局**:

```

| 离线时长: 8小时23分钟      |

| | 农田   Lv.5 -> Lv.7     ||
| | 市集   Lv.4 -> Lv.6     ||
| | 主城   Lv.5 -> Lv.6     ||
| |  (队列扩展: 1->2槽)      ||
| +--------------------------+|
|                            |
| +-- 进行中升级 -----------+|
| | 兵营   Lv.5 -> Lv.6     ||
| | 进度: 65% 剩余: 5分12秒  ||
| +--------------------------+|
|                            |
| +-- 等待中 ---------------+|
| | 书院   Lv.3 -> Lv.4     ||
| +--------------------------+|
|                            |
| 离线效率: 1.2x (升级加速)  |
|                            |
|          [确认]             |
+----------------------------+
  PC: 400x450px 居中弹窗
  手机: Bottom Sheet 70%
```

#### 离线结算特殊情况处理

**主城升级在离线期间完成**:
1. 主城升级完成后，触发槽位扩展（同在线逻辑）
2. 扩展后若有waiting任务且离线时间仍有剩余，继续模拟执行
3. 级联效果：主城升级 -> 槽位扩展 -> 等待任务开始 -> 可能继续完成

**资源不足导致等待任务无法开始**:
1. 离线模拟中，等待任务变为active前不检查资源（资源已在创建任务时扣除）
2. 因此离线期间不会因"资源不足"而跳过等待任务
3. 但离线期间不会触发自动升级（资源检查在自动升级触发时进行）

**离线时间不足以完成任何升级**:
1. 结算弹窗中"已完成升级"为空
2. "进行中升级"显示当前进度
3. 效率系数已应用于进度计算

**离线期间队列完全为空**:
1. 不弹出结算弹窗（无升级任务可结算）
2. 仅弹出资源离线收益弹窗（属于FL-BLD-05资源收取范围）

**关键要求**: 离线效率系数1.2x仅应用于升级时间。级联效果（主城升级->槽位扩展->后续任务）正确处理。离线期间不触发自动升级。结算弹窗清晰展示完成/进行中/等待状态

**时间要点**:
- 服务端结算计算: <=1秒
- 结算弹窗出现: 200ms
- 建筑网格刷新: <=500ms（批量更新所有建筑状态）

**空间要点**:
- 结算弹窗PC: 400x450px居中，遮罩rgba(0,0,0,0.6)
- 结算弹窗手机: Bottom Sheet 70%高度
- 每个升级项: 32px高度
