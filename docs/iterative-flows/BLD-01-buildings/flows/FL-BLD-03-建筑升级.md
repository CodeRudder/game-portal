# FL-BLD-03 建筑升级

> **版本**: v4 | **最后更新**: 2026-05-05
> **归属功能**: 建筑系统 — 建筑操作 (Phase B)
> **触发条件**: 玩家在建筑详情面板点击[升级]按钮
> **结束条件**: 升级开始（进入队列）或升级完成（即时升级），Toast反馈已显示
> **前置状态**: 建筑详情面板已打开，建筑未在升级中
> **后置状态**: 建筑进入升级队列（或立即完成），资源已扣除
> **来源文档**: [PRD BLD-2 建筑升级](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-2) / [UI BLD-4 升级确认](../../games/three-kingdoms/ui-design/ui-layout/BLD-buildings.md#bld-4)

## 关键要求

- 升级不需要确认弹窗，点击[升级]直接执行（一键满级除外，需独立确认弹窗）
- 建筑等级 <= 主城等级
- 升级消耗资源：粮草+铜钱（部分建筑还需矿石/木材/兵力）
- 即时升级（<=5秒，可配置阈值）无队列占用，直接完成
- 升级动画序列：按钮加载态 → Toast升级中 → 卡片进度条 → 完成Toast+外观弹跳

### R2 新增约束

- **一键满级确认弹窗**: 一键满级弹出独立确认弹窗，显示总消耗明细（逐级资源汇总）+ 目标等级 + [确认][取消]按钮
- **一键满级资源快照**: 资源计算基于操作开始时的快照余额，不追踪升级期间的产出变化。原子性保证：全部成功或全部回滚
- **双击防护**: [升级]/[一键满级]按钮点击后立即禁用（加载态），服务端响应后恢复。动画期间不可重复触发
- **网络失败回滚**: 服务端使用事务机制扣除资源+创建升级任务；若任一步失败，已扣资源自动返还
- **校验优先级**: 资源 -> 主城等级 -> 队列槽位 -> 前置条件，显示第一个失败项
- **进度更新机制**: 客户端本地倒计时（基于服务端返回的start_time + duration），服务端确认完成。重连后客户端重新计算进度
- **按钮布局**: [升级]与[一键满级]之间增加间距（>=16px），一键满级按钮使用警告色/图标标识防止误触

### R3 新增约束

- **即时升级路径**: 升级时间<=5秒（可配置阈值）的建筑走即时升级路径——不占队列槽位，点击后直接显示完成Toast，无进度条动画。服务端原子操作：扣除资源+更新等级一步完成
- **定时升级路径**: 升级时间>5秒的建筑走定时升级路径——占用队列槽位，显示进度条+倒计时。创建升级任务后服务端返回task_id+start_time+duration
- **一键满级确认弹窗细节**: 弹窗内分级展示资源消耗（每级一行：Lv.X->Lv.X+1 消耗XXX），底部汇总总消耗。每行标绿（够）或标红（不够）。弹窗尺寸500x480px（PC），支持内滚动（超过6级时）
- **升级中面板行为**: 建筑升级中时打开详情面板——面板内升级预览区变为"升级进行中"状态区，显示进度条+倒计时+加速按钮，[升级]和[一键满级]按钮隐藏，队列信息可见
- **升级完成面板刷新**: 若升级期间详情面板处于打开状态，升级完成时面板内容自动刷新（产出数据、等级、外观），播放弹跳动画（scale 1.0->1.05->1.0, 300ms）

### R4 新增约束

- **主城升级特殊流**: 主城升级有特殊前置条件（如"任一建筑Lv4"用于4->5级，"任一建筑Lv9"用于9->10级），需在升级预览中明确展示。主城升级完成后立即触发队列槽位扩展检查（Lv5->6新增1槽，Lv10->11新增1槽，Lv20->21新增1槽），新槽位立即可用。主城升级完成后触发全建筑产出加成刷新（+2%/级）
- **升级完成通知**: 升级完成后，除Toast外还需发送全局通知（通知中心可查）；建筑Tab页签显示红点Badge（数字1），点击后Badge消失。若玩家在建筑网格页，卡片弹跳动画自动播放
- **升级失败分级处理**: 网络超时/服务端错误分为三级：L1瞬时错误（重试3次，间隔1s）→ L2服务端错误（显示错误码+错误文案，提供[重试]按钮）→ L3数据不一致（强制刷新，提示"数据已更新"）
- **资源耗尽中途升级**: 升级过程中资源产出不会中断，但若加速操作需要额外资源而资源已耗尽，加速按钮置灰+提示"资源不足"
- **并发修改保护**: 服务端对建筑数据使用乐观锁（version字段），若版本不匹配返回409 Conflict，客户端自动刷新数据后重新展示

## 验收标准

- 资源足够时点击升级立即执行，Toast显示升级信息
- 资源不足时Toast明确提示缺少的资源及数量
- 升级过程中卡片显示蓝色脉冲边框+进度条
- 升级完成后建筑外观更新，产出数值立即变化
- 主城升级完成后，队列槽位和全建筑加成立即生效
- 升级完成通知出现在全局通知中心+Tab Badge
- 即时升级（<=5秒）无进度条，直接完成
- 一键满级确认弹窗逐级展示，资源快照模式

## 步骤详情

### FL-BLD-03-01 查看升级预览

> **来源**: [UI BLD-2 建筑详情面板](../../games/three-kingdoms/ui-design/ui-layout/BLD-buildings.md#bld-2) / [PRD BLD-2 建筑升级](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-2)

**用户感知（see->act->feedback）**:
- **See**: 在详情面板"升级预览"区块看到当前等级->下一等级、产出变化对比、资源消耗逐项（标绿满足/标红不足+缺少数量）、升级预计时间、前置条件
- **Act**: 无需操作（信息展示），可点击"加成来源"展开详细加成明细
- **Feedback**: 资源变化时消耗项颜色实时更新（绿<->红），点击展开时加成明细fade-in（100ms）

**系统行为**:
1. 计算升级所需资源（粮草+铜钱+特殊资源），查PRD费用表
2. 计算升级时间（根据当前等级查表）
3. 校验前置条件（主城等级+特殊前置如"任一建筑Lv4"）
4. 逐项与玩家当前资源对比，标记满足/不足
5. 计算产出变化：当前产出 vs 升级后产出，显示差值和百分比
6. 实时监听资源变化（产出/收取/消耗），刷新标记颜色

**主城升级预览特殊行为**:
1. 检测目标建筑是否为主城——若是，额外显示"特殊前置条件"区块
2. 查询主城特殊前置表：
   - 主城 4->5: 需"任一建筑Lv4"
   - 主城 9->10: 需"任一建筑Lv9"
   - 主城 10~30: 对应等级段有"对应前置"
3. 前置条件区块显示：条件描述 + 当前状态（满足=绿色勾 / 不满足=红色叉+缺少说明）
4. 主城升级预览额外显示"升级效果"：全资源加成变化（如 +8%->+10%）+ 队列槽位变化（如 1->2，仅限等级跨越阈值时）
5. 主城升级后队列扩展预览：若当前等级在阈值边界（5->6, 10->11, 20->21），预览区显示"升级后队列槽位: N->N+1"提示

**关键要求**: 资源不足的项标红显示缺少数量。升级时间<=5秒的标注"即时完成"标签（绿色）。主城升级必须展示特殊前置条件和全资源加成变化

**时间要点**:
- 预览计算: <50ms（纯前端查表）
- 资源标记更新: 实时（资源变化时立即刷新）
- 产出变化展示: 当前->目标，动画过渡200ms
- 主城前置校验: <100ms（需遍历所有建筑等级）

**空间要点**:
- PC: 详情面板"升级预览"区块约200px高度，等级变化行24px，产出对比行40px（含进度条），资源消耗行每项24px（粮草/铜钱/矿石/木材/兵力），时间行24px，前置条件行24px
- 手机: 同样区块，宽度=面板宽度-24px内边距
- 即时完成标签: 绿色圆角小标签24x16px，文字"即时"10px
- 主城升级预览: 额外增加"特殊前置"区块（40px）+"升级效果"区块（56px: 加成变化+队列变化）

**异常分支**:

| 条件 | 处理方式 | 转至步骤 |
|------|---------|:--------:|
| 建筑已达最高等级 | 升级预览区显示"已达最高等级Lv.XX"灰色文字，隐藏[升级]/[一键满级]按钮 | -- |
| 主城等级=建筑等级 | 升级预览区显示"需主城升至Lv.XX"黄色提示，[升级]按钮置灰 | -- |
| 主城升级前置不满足（如需任一建筑Lv4） | 前置条件行标红，显示"需将任一建筑升至Lv.4"，[升级]按钮置灰 | -- |
| 主城在升级中时其他建筑尝试升级 | 检查是否因主城升级中导致加成变化——不影响（升级完成才生效），正常显示预览 | 03-02 |

### FL-BLD-03-02 校验升级条件

> **来源**: [PRD BLD-2 建筑升级](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-2) / [PRD BLD-4 建筑联动](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-4)

**用户感知（see->act->feedback）**:
- **See**: 点击[升级]按钮后的校验反馈（Toast或按钮状态变化）
- **Act**: 点击[升级]按钮（触发校验）
- **Feedback**: 校验通过->进入执行流程(03-03)；校验失败->Toast显示第一个失败项（3秒），升级预览区对应项闪烁红色（200ms x2）

**系统行为**:
1. 前端快速校验（预检）：资源是否足够、主城等级、队列槽位
2. 发送`POST /api/buildings/{id}/upgrade`请求（服务端完整校验），请求头携带`If-Match: {version}`（乐观锁）
3. 服务端校验（按优先级顺序），**返回第一个失败项**：
   1. 资源是否足够（服务端实时余额校验，防止并发扣除）
   2. 主城等级 >= 目标等级
   3. 升级队列是否有空槽（定时升级路径才需要）
   4. 特殊前置条件（主城升级的"任一建筑LvX"条件）
   5. 乐观锁版本校验（数据是否被其他操作修改）
4. 全部通过->执行升级；任一失败->返回错误码+错误消息
5. 主城升级额外校验：遍历所有已建建筑，检查是否有任一建筑等级满足前置要求

**校验优先级详细规则**:

| 优先级 | 校验项 | 校验方式 | 失败返回码 |
|:------:|--------|---------|:----------:|
| 1 | 资源足够 | 服务端实时余额 vs 费用表 | RES_INSUFFICIENT |
| 2 | 主城等级 >= 目标等级 | 服务端查主城等级 | LEVEL_CAP |
| 3 | 队列有空槽（定时升级） | 服务端查活跃任务数 < 槽位数 | QUEUE_FULL |
| 4 | 特殊前置（主城升级） | 服务端遍历建筑等级 | PREREQ_FAIL |
| 5 | 乐观锁版本 | version字段匹配 | VERSION_CONFLICT |
| 6 | 建筑未在升级中 | 服务端查该建筑任务状态 | ALREADY_UPGRADING |

**关键要求**: 校验优先级 资源->主城等级->队列槽位->前置条件->版本校验，显示第一个失败项。即时升级（<=5秒）跳过队列槽位校验

**时间要点**:
- 前端预检: <50ms
- 服务端校验: <100ms（主城升级前置校验可能<200ms，需遍历建筑）
- Toast显示时长: 3秒

**空间要点**:
- Toast位置: 顶部居中（资源栏下方），宽度自适应（最大320px），高度40px
- 失败项闪烁: 升级预览区对应行background-color闪烁rgba(231,76,60,0.15)，200ms x2

**异常分支**:

| 条件 | 处理方式 | 转至步骤 |
|------|---------|:--------:|
| 资源不足 | Toast"XX不足，还需要YYYY" + 预览区该资源项闪烁红色 | 03-01 |
| 主城等级不足 | Toast"主城等级不足，需达到LvXX" | 03-01 |
| 队列已满（定时升级） | Toast"升级队列已满" | 03-01 |
| 建筑等级已达上限 | 隐藏[升级]按钮，显示"已达上限"灰色标签 | -- |
| 前置条件未满足（主城特殊前置） | Toast"前置条件未满足: 需任一建筑达到Lv.XX" | 03-01 |
| 建筑已在升级中 | Toast"该建筑正在升级中" | 03-01 |
| 服务端校验与前端不一致 | 以服务端结果为准，前端刷新数据后更新预览区 | 03-01 |
| 版本冲突（并发修改） | 服务端返回409，客户端自动刷新数据，Toast"数据已更新，请重试" | 03-01 |
| 服务端不可用（5xx） | Toast"服务暂时不可用，请稍后重试"（3秒），不自动重试 | 03-01 |

### FL-BLD-03-03 执行升级（含主城升级特殊流）

> **来源**: [UI BLD-4 升级确认](../../games/three-kingdoms/ui-design/ui-layout/BLD-buildings.md#bld-4) / [PRD BLD-2 建筑升级](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-2)

**用户感知（see->act->feedback）**:
- **See**: 升级预览区显示资源消耗和等级变化信息，[升级]按钮高亮
- **Act**: 点击[升级]按钮（双击防护：按钮立即禁用+旋转齿轮图标）
- **Feedback** (分路径，详见下方03-03a和03-03b)

**系统行为（通用）**:
1. 按钮立即禁用（双击防护），发送升级请求（携带version字段）
2. 服务端原子操作（事务）：校验->扣除资源->创建升级任务（定时）或直接升级（即时）
3. 客户端更新：扣除资源动画（资源栏数字递减200ms）-> 更新建筑状态 -> 刷新卡片+面板

**主城升级特殊行为**:
1. 服务端扣除主城升级费用（粮草+铜钱+兵力，主城升级不消耗矿石/木材）
2. 若走即时路径（主城1->2: 10秒，按配置阈值>5秒走定时路径）：
   - 主城升级均为定时路径（最短10秒 > 5秒阈值），所以主城升级始终走定时路径
   - 特殊：主城升级期间，其他建筑不可升级（若队列仅有1个槽位），但已有升级任务不受影响
3. 主城升级完成后（在03-05中处理）：
   - 触发队列槽位扩展（Lv5->6: 1->2槽，Lv10->11: 2->3槽，Lv20->21: 3->4槽）
   - 触发全建筑产出加成刷新（每级+2%全资源加成）
   - 触发建筑等级上限刷新（所有建筑上限=新主城等级）
   - 触发解锁检查（新解锁的建筑自动变为可建造状态）

**关键要求**: 即时升级（<=5秒，可配置阈值）不占队列槽，直接完成；定时升级（>5秒）占队列槽。按钮在请求期间保持禁用。主城升级走定时路径，完成后触发全局状态刷新

**时间要点**:
- 按钮禁用响应: 立即（0ms）
- 请求响应: <=500ms（正常网络）
- 按钮恢复: 服务端确认后100ms
- 资源扣除动画: 200ms（数字递减）

#### FL-BLD-03-03a 即时升级路径（<=5秒）

**触发条件**: 升级时间 <= 5秒（可配置阈值 `INSTANT_UPGRADE_THRESHOLD`）

**用户感知（see->act->feedback）**:
- **See**: 升级预览区显示绿色"即时完成"标签
- **Act**: 点击[升级]
- **Feedback**: 按钮禁用100ms -> Toast"XX升级至Lv.YY！"出现 + 建筑卡片弹跳动画（scale 1.0->1.05->1.0, 300ms）-> 等级数字更新 -> 产出数字更新 -> 按钮恢复

**系统行为**:
1. 服务端收到请求后，原子操作：扣除资源 + 更新建筑等级 + 更新产出数据，一步完成
2. 返回数据：`{ new_level, new_production, version }`
3. 客户端收到响应后：
   - 资源栏数字递减动画（200ms）
   - 建筑卡片播放弹跳动画（300ms）
   - 等级数字更新（无动画，直接替换）
   - 产出数字更新（旧值->新值，数字滚动200ms）
   - 详情面板刷新（若打开）
4. 不创建升级任务，不占用队列槽位
5. 不发送WebSocket升级开始事件（因为没有"升级中"阶段）
6. 发送升级完成通知（全局通知中心 + Tab Badge）

**时间要点**:
- 即时完成动画总计: 约500ms（弹跳300ms + 等级更新 + 产出更新）

### FL-BLD-03-06 一键满级

> **来源**: [UI BLD-2 操作按钮区](../../games/three-kingdoms/ui-design/ui-layout/BLD-buildings.md#bld-2) / [PRD BLD-2 建筑升级](../../games/three-kingdoms/ui-design/prd/BLD-buildings-prd.md#bld-2)

**用户感知（see->act->feedback）**:
- **See**: 详情面板底部显示[升级]和[一键满级]两个按钮，间距>=16px，一键满级按钮使用警告色（橙色#e67e22）+皇冠图标标识
- **Act**: 点击[一键满级]按钮 -> 弹出独立确认弹窗（500x480px PC / Bottom Sheet 70% 手机）
- **Feedback**: 弹窗显示逐级资源消耗明细+总消耗汇总 -> 点击[确认]->弹窗关闭+批量执行升级 -> Toast"XX将升级至Lv.YY（共N级）"

**系统行为**:
1. 点击[一键满级]：基于当前资源快照（开始时锁定余额），循环计算逐级升级所需资源，直到资源不足为止
2. 渲染确认弹窗：逐级展示（Lv.X->Lv.X+1 消耗XXX），每行标绿（够）或标红（不够），底部汇总总消耗
3. 弹窗内显示关键计算参数：
   - 当前资源余额（快照时刻）
   - 主城等级限制（最高升级到什么等级）
   - 队列剩余空间（定时升级占槽，即时升级不占）
4. 点击[确认]：发送`POST /api/buildings/{id}/upgrade-batch`请求，携带目标等级列表+资源快照版本号
5. 服务端原子性保证：全部成功创建升级任务或全部回滚
6. 多级升级任务依次排队（定时升级），即时升级直接完成

**一键满级计算详细步骤**:

```
步骤1: 获取资源快照
  - 读取当前所有资源余额（粮草/铜钱/矿石/木材/兵力）
  - 锁定快照：snapshot_id = timestamp
  - 所有后续计算基于此快照

步骤2: 确定上限
  - 最高等级 = min(建筑等级上限, 主城等级, 资源允许的最高等级)
  - 主城等级限制: 其他建筑最高=主城等级

步骤3: 逐级计算（从当前等级+1开始）
  - 对每一级:
    a. 查费用表得到该级消耗
    b. 检查快照余额是否足够
    c. 若足够: 扣除消耗（快照内虚拟扣除），记录该级
    d. 若不足: 标记该行为红色，停止计算
  - 区分即时升级（<=5秒）和定时升级（>5秒）

步骤4: 汇总
  - 计算总消耗（各资源）
  - 计算队列占用（定时升级级数）
  - 计算最终目标等级
  - 计算预计总时间（定时升级级数的升级时间之和）
```

**确认弹窗内容详细布局**:

```
| 目标: 农田 Lv.5 -> Lv.9      |  标题区40px

| +-- 逐级消耗（可滚动）-------+|
| | Lv.5->6  粮草1K  铜钱500   ||  每行32px，标绿
| | Lv.6->7  粮草1.8K 铜钱900  ||  标绿
| | Lv.7->8  粮草3.2K 铜钱1.6K ||  标绿
| | Lv.8->9  粮草5.8K 铜钱2.9K ||  标红(不够)
| +----------------------------+|
|                               |
| +-- 总消耗 -------------------+|
| | 粮草 11,800 / 15,000       ||  总消耗区48px
| | 铜钱 5,900 / 8,000         ||
| | 即时升级: 1级  定时升级: 2级 ||
| +----------------------------+|
|                               |
| [取消]        [确认升级(3级)]  |  按钮区56px

  500x480px PC / Bottom Sheet 70% 手机
```

**关键要求**: 一键满级**需要独立确认弹窗**（区别于单次升级的无确认设计）。按钮使用警告色/图标标识。资源计算使用快照模式。确认弹窗内滚动（超过6级时每级行高32px，总计超过384px时滚动）

**时间要点**:
- 资源计算: <100ms（纯前端查表+累加）
- 弹窗出现动画: 200ms
- 批量请求响应: <=1秒（多级升级）
- 确认弹窗关闭动画: 150ms

**空间要点**:
- 确认弹窗PC: 500x480px居中，遮罩rgba(0,0,0,0.6)，内部分区：标题区40px+逐级消耗列表（可滚动，最大320px）+总消耗区48px+按钮区56px
- 确认弹窗手机: Bottom Sheet 70%高度=355px，布局同PC但更紧凑
- 逐级消耗行: 高32px，左侧"Lv.X->Lv.X+1"，右侧消耗项（标绿/标红）
- 总消耗区: 高48px，粗体显示总粮草+总铜钱等
- 按钮区: [取消]（灰色）+ [确认升级]（橙色警告色），高度40px

**异常分支**:
