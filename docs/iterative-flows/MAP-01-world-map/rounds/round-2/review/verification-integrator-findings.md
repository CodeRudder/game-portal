## P0 评审: Dev Proxy 的 2 项 P0 优先级重评估

### P0-M-01: 攻城撤退按钮位置 — **降级为 P1**

**Dev Proxy 原始判定**: P0（阻塞可实施性）

**Integrator 评审**:

1. **是否阻塞编码?** 否。攻城撤退按钮的位置是 UI 布局问题，开发者在编码攻城覆盖层时可以先按常规布局（底部/右下角）放置按钮，后续仅需调整位置和样式即可。这与 R1 的 P0（架构假设缺失、公式未定义、接口未建立）有本质区别——R1 的 P0 是"完全无法动手写代码"，而此问题是"写了之后可能需要调整位置"。

2. **是否影响核心逻辑?** 否。撤退的业务逻辑（P0-V07）已完整定义：退款比例、状态转移、编队清理。按钮位置不影响逻辑正确性。

3. **可替代方案**: 开发者可参考已有的"行军取消按钮"定义（V-005: 迷你面板+任务面板），采用相同的设计模式。这是一个合理的工程决策，不需要文档明确定义才能动手。

**重评估结果**: **P1**（UI 细节补充，不阻塞编码框架搭建）

**建议**: 在攻城覆盖层文档中补充按钮位置定义，可在 R3 或实施阶段边编码边补充。

---

### P0-M-02: 胜利信息叠加时序 — **降级为 P1**

**Dev Proxy 原始判定**: P0（严重影响首次体验）

**Integrator 评审**:

1. **是否阻塞编码?** 否。胜利弹窗的 UI 组件可以正常编码，Toast 的延迟展示逻辑是典型的前端 UI 调度问题，不需要特殊文档定义。开发者完全可以自行决定"弹窗先展示、Toast 延迟 1-2s"的时序方案。

2. **是否影响核心逻辑?** 否。涉及的三个信息（胜利弹窗、自动驻防 Toast、兵力不足警告）的数据来源和内容均已定义完整。时序是纯展示层问题。

3. **严重程度评估**: "多个 Toast 互相遮挡"是用户体验问题而非功能正确性问题。即使在最坏情况下（两个 Toast 同时出现），用户仍然可以看到所有信息，只是体验不够优雅。

4. **与 R1 P0 的对比**: R1 的 P0 是"编队总战力公式未定义，无法计算战斗结果"。这个 P0 是"两个 Toast 可能同时出现"。两者在阻塞程度上的差距是数量级的。

**重评估结果**: **P1**（用户体验优化，可在实施阶段处理）

**建议**: 在攻城结算文档中补充信息展示时序建议，但不必阻塞 R2 结束。

### P0 评审总结

Dev Proxy 将上述 2 项标为 P0 出发点是"可实施性完美主义"——希望文档覆盖到每一个 UI 细节。但按照 R 系列迭代的核心原则：

- **P0 = 阻塞编码，完全无法动手**
- **R1 的 3 个 P0** 是架构假设缺失、公式未定义、TypeScript 接口未建立——这些都是"不知道该写什么代码"的级别
- **R2 这 2 项** 是"知道该写什么，但位置/时序可以自行决定"的级别

**结论: R2 零 P0。所有问题均为 P1 及以下。**

## User Proxy P1 深度评审

### P1-R2-U01 / P1-INT-01: 编队战力公式跨文档不一致

**问题定性**: 这是**公式定义的问题**，而非流程描述的问题。

- FL-MAP-09 P3 使用的是 PLAN.md 附录 C 的简化公式（`进攻武将战力系数 = (武力x3+统率x2+智力)/300`），这是一个**比例系数**设计
- FL-MAP-16 S2 [P0-02-R2] 使用的是**绝对值**设计（`攻击x10 + 防御x8 + 智力x5`），这是 R2 新增的完整公式

**根本原因**: R2 新增完整公式时，未同步更新 FL-MAP-09 P3 中的旧引用。

**解决方案**: 以 FL-MAP-16 S2 [P0-02-R2] 为权威定义，FL-MAP-09 P3 改为引用 `编队总战力计算见 FL-MAP-16 S2 [P0-02-R2]`。PLAN.md 附录 C 标注为"简化预览版，实际计算以 FL-MAP-16 为准"。

**优先级确认**: P1 正确。公式选错会导致游戏平衡性问题，但不阻塞代码框架搭建。

### P1-R2-U02 / P1-INT-02: 城防初始值公式跨文档冲突

**问题定性**: 这同样是**公式定义的问题**。

- V-029 和 types 中的 `基础值(500) + 城墙等级x50` 是一个简化静态公式
- FL-MAP-09 P8 内联的 `城市等级x100 + 驻防兵力x0.5` 是一个动态公式，依赖运行时数据

**根本原因**: 两个公式的适用场景可能不同（静态初始值 vs 动态加成），但文档未明确说明两者关系。也可能是其中一个是旧版本未清理。

**解决方案**: 需要产品确认哪个公式为权威版本。确认后统一所有引用点。

**优先级确认**: P1 正确。城防值直接影响攻城胜负判定，需要统一。

## R2 结束判定

### 判定标准回顾

- P0 = 阻塞编码，完全无法动手
- R1 结束条件: 所有 P0 修复且验证通过
- R2 结束条件: R2 新增修复验证通过 + 无新增 P0

### 逐项检查

| 检查项 | 状态 | 说明 |
|--------|:----:|------|
| R1 的 3 个 P0 修复验证 | **全部通过** | 架构假设(P0-01)、编队战力公式(P0-02)、TS 接口(P0-03) 三项 Dev Proxy 均验证 PASS |
| R1 遗留 14 个 P1 验证 | **全部通过** | Dev Proxy 逐项验证 14/14 PASS |
| R2 新增 7 项修复验证 | **全部通过** | User Proxy 验证 7/7 PASS，Dev Proxy 验证 7/7 PASS |
| R2 新增 P0 | **零项** | Dev Proxy 的 2 项经评审降级为 P1 |
| 跨文档一致性 | **3 处公式不一致** | 均为 P1 级别，不阻塞编码 |
| TypeScript 接口覆盖 | **94%** | 核心结构全覆盖，缺 3 个辅助接口（P2 级别） |

### 判定结论

**R2 可以结束。**

理由:

1. **零 P0**: 经过严格评审，R2 没有阻塞编码的问题。Dev Proxy 标记的 2 项 P0 经分析为 UI 细节问题，不满足"完全无法动手"的 P0 标准。

2. **所有 R2 修复验证通过**: 6 项焦点修复（骨架屏百分比、查找算法、奖励文案、cancelled 清理、编队战力公式、伤亡公式备注）+ 1 项策略修正系数区分，User Proxy 和 Dev Proxy 双重验证全部通过。

3. **剩余 P1 不阻塞编码框架**: 15 项待修复 P1 均为"编码时可能实现错误"的级别，开发者可以在编码过程中逐项确认和修复。特别是:
   - 编队战力公式统一: 开发者先按 FL-MAP-16 完整版编码，后续仅需删除旧引用
   - 城防公式统一: 产品确认后修改一个常量文件即可
   - UI 细节（按钮位置、信息时序）: 边编码边调整

4. **可实施性评分**: Dev Proxy 评估综合 88%，修复 P1 后预估 96%。即使是当前 88% 的完整度，也已远超"可开始编码"的阈值。

5. **分阶段实施已规划**: Dev Proxy 明确区分了"立即可开始"（地图渲染、领土交互、伤亡系统等）和"待修复后开始"（攻城引擎、行军系统），不存在"什么都做不了"的情况。
