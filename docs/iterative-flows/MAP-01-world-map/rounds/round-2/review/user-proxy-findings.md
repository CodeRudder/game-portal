## 评审维度总评

### 1. 视觉感知
整体优秀。每个步骤均有"用户感知"描述，颜色值(#hex)、动画时长(ms)、过渡效果均已内联。特别好的实践：
- FL-MAP-03的TERRAIN_PALETTE完整内联定义
- FL-MAP-17的结果弹窗ASCII线框图（三种结果各有不同布局）
- FL-MAP-18的移动端布局ASCII图和PC/移动端差异对照表

需改进：
- 攻城覆盖层透明度值未定义(P2-R2-04)
- 领土放弃视觉过渡动画未描述(P2-R2-03)

### 2. 操作引导
新手引导(FL-MAP-01-SP01)设计合理，4步递进有重试和跳过机制。移动端Step Wizard分步引导清晰。但存在引导盲区：
- 移动端长按功能无发现机制(P1-08遗留，P1-R2-03新增定义)
- 攻城条件多条不满足时缺少修正引导(P1-R2-03)
- PC端编队确认后回退机制未定义(P1-R2-04)

### 3. 反馈完整性
反馈体系完善，Toast/弹窗/动画覆盖大部分操作。亮点包括攻城中断退款政策(P0-V07)分阶段定义清晰、将领空状态面板(P0-V10)避免死胡同、应用切后台恢复遮罩(P0-03)视觉状态明确。

需改进：
- 征服胜利后多信息叠加可能压垮玩家(P0-R2-02)
- 攻城撤退按钮位置缺失(P0-R2-01)
- 一键收取Toast文案不一致(P2-R2-05)

### 4. 异常场景
异常流程处理非常全面。每个流程都有独立的"异常流程汇总"表，覆盖了数据异常、边界条件、并发冲突等场景。FL-MAP-09的中断退款政策(P0-V07)按marching/sieging阶段分别定义退款比例、每日次数和冷却惩罚，逻辑清晰无矛盾。

需改进：
- 攻城撤退按钮位置缺失(P0-R2-01)
- 离线回归面板10s超时可能不足(P2-R2-07)
- 奖励领取关闭弹窗后通知未定义(P1-R2-06)

### 5. 流程连贯性
流程间引用关系清晰（INDEX.md子流程索引完整）。R2 Step0的P0修复提升了架构一致性：
- 架构假设声明(INDEX.md)消除了"是否联网"的困惑
- types/map-interfaces.md提供了统一的数据契约
- FL-MAP-04补充了敌方行军实体来源(P0-01-R2)
- FL-MAP-16内联了编队总战力公式(P0-02-R2)
- FL-MAP-09 P9结算后执行序列(P1-45)统一了跨流程执行顺序

连贯性不足之处：
- 编队战力公式在FL-MAP-09和FL-MAP-16间简要表述差异可能导致理解困难(P2-R2-08)
- FL-MAP-07胜利→FL-MAP-08驻防过渡存在信息过载(P0-R2-02)
- FL-MAP-09 P3→P4 PC端回退机制未定义(P1-R2-04)

## 数据结构评审

### types/map-interfaces.md
**优点**：
- 30个TypeScript接口/类型覆盖全部核心数据结构
- 每个接口标注来源流程，便于追溯
- 公式索引表集中汇总10个关键公式
- 坐标系约定清晰（色块/像素/视口三种坐标）+ 转换函数内联
- 存储策略表明确localStorage vs IndexedDB的使用场景

**发现的问题**：

1. **PendingReward接口两个版本不一致**
   - map-interfaces.md定义：`{ type, amount, itemId?, claimed, expireAt }` (简洁版)
   - FL-MAP-11-03定义：`{ id, siegeTaskId, items[], strategyType, strategyMultiplier, expiresAt, status }` (详细版)
   - 两处定义结构完全不同，实施时可能造成混淆。建议以FL-MAP-11-03为准更新map-interfaces.md

2. **ForceStatus缺少cancelled映射**
   - SiegeTaskStatus有`cancelled`终态但ForceStatus没有对应映射
   - 当SiegeTask从marching→cancelled时，关联编队的ForceStatus如何变更？
   - FL-MAP-16 P1-39映射表中cancelled无对应行，编队可能卡在marching状态

3. **缺少TerritoryType枚举**
