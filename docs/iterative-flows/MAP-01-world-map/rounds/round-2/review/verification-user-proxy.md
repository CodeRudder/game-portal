## 新发现

### P1 问题 (2个)

#### P1-R2-U01: 编队战力公式跨文档定义不一致

**位置**: FL-MAP-09 Stage P3 vs FL-MAP-16 S2 [P0-02-R2]

**问题描述**:
- FL-MAP-09 Stage P3 定义编队总战力公式为:
  `编队总战力 = SUM(每个武将分配的兵力 x 该武将的进攻武将战力系数)`
  其中进攻武将战力系数 = (武力x3 + 统率x2 + 智力) / 300
- FL-MAP-16 S2 [P0-02-R2] 定义编队总战力公式为:
  `编队总战力 = 兵力战力 + 将领属性加成 + 科技加成总和`
  其中兵力战力 = troops x 1.0, 将领属性加成 = attack x 10 + defense x 8 + intel x 5

这两个公式计算结果不同:
- FL-MAP-09 公式: 3000兵 + 张飞(武力95, 统率70, 智力30) → 单武将系数=(95x3+70x2+30)/300 = 1.383 → 3000x1.383 = 4150
- FL-MAP-16 公式: 3000x1.0 + (95x10 + 70x8 + 30x5) + 科技 = 3000+1660+科技 = 4660+科技

**用户影响**: 开发者按不同文档实施将得到不同的战力值，直接影响游戏平衡性。

**建议**: 选择一个公式作为权威定义，另一个文档改为"详见权威文档"引用。同时更新 types/map-interfaces.md 中的攻方总战力公式引用。

#### P1-R2-U02: 城防初始值公式跨文档冲突

**位置**: FL-MAP-09 Stage P8 vs FL-MAP-09 V-029 vs types/map-interfaces.md

**问题描述**:
- FL-MAP-09 Stage P8 城防初始值计算区块写:
  `基础城防值 = 目标城市等级 x 100 + 守军驻防兵力 x 0.5`
  `城防初始值 = 基础城防值 x 城防系数`
- FL-MAP-09 关键公式索引 V-029 写:
  `城防初始值=基础值(500)+城墙等级x50`
- types/map-interfaces.md 城防初始值公式写:
  `城防初始值 = 基础值(500) + 城墙等级 * 50`

V-029 和 types 文件中的公式为简单公式（仅依赖城墙等级），而 Stage P8 内联公式更复杂（依赖城市等级和驻防兵力）。两个公式的输入参数完全不同，结果也必然不同。

**用户影响**: 开发者不知道应按哪个公式实施城防值。

**建议**: 选择一个公式作为权威定义。如果 Stage P8 内联的复杂公式为正确版本，需同步更新 V-029 和 types/map-interfaces.md；反之亦然。

### P2 问题 (6个)

#### P2-R2-U01: FL-MAP-07 胜率公式与 PLAN.md 附录C 公式表达不一致

**位置**: FL-MAP-07 FL-MAP-07-02 vs PLAN.md 附录 C.3

**问题描述**: FL-MAP-07-02 写 `胜率 = clamp(5%, 95%, (我方战力/敌方战力 x 0.5) + 地形修正)`，并明确说明括号内优先级为"先除法→再乘以0.5→再加地形修正→最后截断"。但 PLAN.md 附录 C.3 的公式表达可能不同（用户无法直接验证PLAN.md，但文档声明"完整公式链定义见PLAN.md"）。

**用户影响**: 若开发者直接看 FL-MAP-07 文档，公式可独立理解。但引用PLAN.md时可能产生歧义。

**建议**: 在 FL-MAP-07-02 中增加注释"本公式与PLAN.md附录C.3完全一致"，消除疑虑。

#### P2-R2-U02: FL-MAP-01 新手引导步骤3"进入攻城确认弹窗"的操作细节不够直观

**位置**: FL-MAP-01 SP01 步骤3

**问题描述**: 步骤3完成条件为"进入攻城确认弹窗（点击'攻占'按钮）"，引导文案为"这是一座敌方城池，需要攻城才能占领"。但用户在教程中需要先找到敌方城池（通过查找算法自动定位），然后点击它，再点击"攻占"按钮。中间的"点击领土→弹出详情面板→在面板中找到攻占按钮"这个操作链在引导文案中没有体现。

**用户影响**: 新手可能在找到敌方城池后，不知道需要先点击领土弹出详情面板，再在面板中找"攻占"按钮。步骤3的引导箭头指向"敌方城池"而非"攻占按钮"。

**建议**: 将步骤3引导文案改为"点击这座敌方城池查看详情"，完成条件拆为两步: (1) 点击敌方城池弹出详情面板 (2) 在面板中点击"攻占"按钮。或至少在引导文案中补充"点击城池后，在弹出的面板中找到'攻占'按钮"。

#### P2-R2-U03: FL-MAP-09 粮草消耗公式在两处表达不同

**位置**: FL-MAP-09 Stage P2 vs FL-MAP-09 关键公式索引 V-033

**问题描述**:
- Stage P2 内联的粮草消耗公式:
  `粮草消耗 = 基础消耗(50) x 兵力系数(兵力/100) x 距离系数(格数x0.3) x 策略修正`
  示例: 50 x 8 x 6 x 1.2 = 2880
- V-033 索引中的公式:
  `粮草消耗 = 基础消耗(100) + 出征兵力 x 0.1 + 行军距离(格数) x 5 + 策略修正`

基础消耗不同（50 vs 100），计算方式不同（乘法链 vs 加法），结果差异巨大。Stage P2 示例结果2880，而V-033公式对于同样800兵力、20格、强攻: 100+80+100x1.5 = 330，差距近9倍。

**用户影响**: 实施时不知道按哪个公式计算。

**建议**: 选定一个公式，另一处标注"已废弃，以XXX为准"。

#### P2-R2-U04: FL-MAP-09-01 攻城中断处理标注"旧版描述"但仍保留

**位置**: FL-MAP-09-01

**问题描述**: 文档顶部写"注意: 此为旧版描述。取消/退款规则以 Stage P6 为准。"但整个子流程仍然存在。开发者可能困惑是否需要阅读此节。

**用户影响**: 信息冗余，可能误读旧版规则。

**建议**: 将旧版描述替换为"取消/退款规则见 Stage P6 [P0-V07] 统一取消规则声明和 FL-MAP-09-08 状态机表"，移除旧版内容或折叠。

#### P2-R2-U05: FL-MAP-14 自动升级触发条件中"占领时间"的计时起点不明确

**位置**: FL-MAP-14 步骤1

**问题描述**: 15级逐级提升规则表中"占领 6 小时"/"占领 12 小时"等条件使用"占领时间"，[V-046]说明"占领时间使用挂钟时间(wall-clock time)"。但Lv5后条件写"Lv5 后 12 小时"——这是从Lv5确认时刻开始计时，还是从占领时刻开始？Lv5是"手动确认"的，确认时间可能远晚于占领24小时。

**用户影响**: 开发者可能误解为从占领时刻开始计算所有升级时间，而非从上一级确认时刻。

**建议**: 在规则表下方增加注释"自动升级的时间条件以完成上一级的时间点为起点（Lv1~4以占领时刻为起点，Lv5以后以Lv5确认时刻为起点，Lv10以后以Lv10确认时刻为起点）"。

#### P2-R2-U06: FL-MAP-12 山贼战斗兵力来源描述有歧义

**位置**: FL-MAP-12-02 S1

**问题描述**: [V-035] 定义出征兵力 = `min(玩家最近己方领土驻防兵力 x 80%, 全局未分配兵力 x 50%)`。"最近己方领土驻防兵力"可能被理解为离山贼事件位置最近的己方领土，也可能被理解为离玩家当前视口最近的己方领土。且"驻防兵力"和"未分配兵力"两个概念混合使用，可能导致实施困惑。

**用户影响**: 两个开发者可能对"最近"理解不同。

**建议**: 明确"最近己方领土"为"距离事件触发位置（色块坐标）欧几里得距离最近的己方领土"。

#### P3-R2-U01: FL-MAP-01 产出气泡状态表与产出概览面板中"已超上限"边界处理重复

**位置**: FL-MAP-01 SP02

**问题描述**: 产出气泡状态表描述了"已满(100%)"行为，产出概览面板中又重复描述了"一键收取边界处理(P1-17): 已超上限领土跳过收取+提示"。两处描述的边界条件略有不同: 气泡说"资源>=上限→停止累加"，一键收取说"已超上限领土跳过收取"。实际上"停止累加"后资源不会超过上限，除非在停止后因其他原因（如科技加成导致上限降低）超过。

#### P3-R2-U04: FL-MAP-10 热力图颜色方案与产出高低语义不一致

**位置**: FL-MAP-10-02 / FL-MAP-10-05

**问题描述**: 热力图颜色: Top 20%=红色(#FF4444), 80~100%=绿色(#44AA44)。红色在用户直觉中代表"危险/不好"，但此处红色代表"高产/好"。与产出气泡的三色（绿色=正常、黄色=预警、红色=已满）形成冲突——同一屏幕上红色可能同时代表"最高产"和"资源已满"。

**建议**: 考虑将热力图高产色改为暖色系（金色/橙色），与气泡红色区分；或在热力图图例中增加明确说明"红色=最高收益"。

#### P3-R2-U05: FL-MAP-03 数据格式错误异常处理链路中Toast和"重置地图数据"按钮的用户感知描述不够

**位置**: FL-MAP-03-00

**问题描述**: 异常处理为"显示Toast'地图数据加载异常，部分地形可能不准确' + '重置地图数据'按钮"。但未说明"重置地图数据"按钮的具体行为: 是清除IndexedDB重新加载? 会丢失哪些数据? 是否影响领土归属等游戏进度?

**建议**: 明确"重置地图数据"按钮的作用范围（仅重置地图配置，不影响游戏进度数据），避免用户误操作。
