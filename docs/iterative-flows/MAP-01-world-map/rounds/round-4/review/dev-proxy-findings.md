## P0 级发现（无法实施）

### F-P0-01: 城防初始值公式自相矛盾

- **流程/数据编号**: Stage P8 / 关键公式索引
- **实施困惑**: 同一文档内存在两套城防初始值公式，计算结果完全不同。

  **版本 A** (Stage P8 内联公式):
  ```
  城防初始值 = 基础城防值 x 城防系数
  基础城防值 = 目标城市等级 x 100 + 守军驻防兵力 x 0.5
  城防系数 = 1.0 + 城墙等级 x 0.05
  ```
  例: 城市等级5, 守军1000, 城墙等级3 -> (5x100 + 1000x0.5) x (1.0 + 3x0.05) = 1000 x 1.15 = 1150

  **版本 B** (关键公式索引 [V-029]):
  ```
  城防初始值 = 基础值(500) + 城墙等级 x 50
  ```
  例: 城墙等级3 -> 500 + 3x50 = 650

  两个公式计算结果差距极大（1150 vs 650），且版本 A 考虑了城市等级和守军兵力，版本 B 完全不考虑。版本 B 还标注"完整参数详见PRD MAP-4"，暗示自己可能不完整。
- **可能的理解方式**: 以 Stage P8 内联公式为准（更详细，包含更多变量）；或以 [V-029] 索引为准（标注了版本号）。
- **需补充内容**: 明确唯一权威公式，废弃另一个。建议标注 `[已废弃]` 并说明替换版本。
- **优先级**: **P0**

### F-P0-02: 策略修正系数存在三套不同数值

- **流程/数据编号**: Stage P8 / PLAN.md C.2 / AC-09-01
- **实施困惑**: 攻城城防衰减公式中的"策略修正系数"存在三套不一致的值：

  | 策略 | AC-09-01 / Stage P8 公式索引 | PLAN.md C.2 编队战力修正 | PLAN.md C.4 城防衰减修正 |
  |------|:---:|:---:|:---:|
  | 强攻 | 1.2 | x1.5 | x1.5 |
  | 围困 | 0.8 | x0.8 | x0.8 |
  | 夜袭 | 1.5 | x1.2 | x1.2 |
  | 内应 | 1.3 | x2.0 | x2.0 |

  - AC-09-01 明确标注为城防衰减公式的策略修正，且文档声明"以 AC-09-01 为权威来源"
  - PLAN.md C.2 标注为编队战力修正，C.4 标注为城防衰减修正（C.4 有 [P1-M-09] 注释说明以 AC-09-01 为准）
  - 但强攻(1.2 vs 1.5)、夜袭(1.5 vs 1.2)、内应(1.3 vs 2.0) 差异巨大

  核心问题: AC-09-01 和 PLAN.md C.4 都是"城防衰减公式的策略修正"，但数值完全不同。P1-M-09 注释虽然说"以 AC-09-01 为准"，但两套系数差距过大，如果选错了直接影响游戏平衡。
- **可能的理解方式**: AC-09-01 的系数是城防衰减专用（与 C.2 战力修正分开计算）；或者 AC-09-01 系数是笔误，应与 PLAN.md C.4 一致。
- **需补充内容**: 明确 C.2（编队战力修正）与城防衰减策略修正是否为同一组参数。如果是不同参数，需命名区分（如 `attack_strategy_modifier` vs `decay_strategy_modifier`）并明确各自使用场景。如果 C.2 根本不参与城防衰减计算，需在 AC-09-01 中明确声明。
- **优先级**: **P0**

### F-P0-03: 编队总战力公式在 FL-MAP-09 与 FL-MAP-16 之间结构矛盾

- **流程/数据编号**: Stage P3 (FL-MAP-09) / S2 (FL-MAP-16)
- **实施困惑**: FL-MAP-09 Stage P3 定义的编队总战力公式与 FL-MAP-16 S2 的权威公式结构完全不同。

  **FL-MAP-09 P3 版本** (简化):
  ```
  编队总战力 = Σ(每个武将分配的兵力 x 该武将的进攻武将战力系数)
  进攻武将战力系数 = (武力x3 + 统率x2 + 智力) / 300
  ```
  特征: 乘法结构，兵力与将领系数相乘后求和。

  **FL-MAP-16 S2 版本** (权威 [P0-02-R2]):
  ```
  编队总战力 = 兵力战力 + 将领属性加成 + 科技加成总和
  兵力战力 = troops x 1.0
  将领属性加成 = attack x 10 + defense x 8 + intel x 5
  科技加成总和 = Σ(科技.level x 科技.power_per_level)
  ```
  特征: 加法结构，兵力、将领属性、科技三项独立相加。包含科技加成。

  差异点:
  1. FL-MAP-09 是乘法公式（兵力x系数），FL-MAP-16 是加法公式（兵力+属性+科技）
  2. FL-MAP-16 有科技加成项，FL-MAP-09 没有
  3. 属性权重不同: FL-MAP-09 (3:2:1), FL-MAP-16 (10:8:5 即 2:1.6:1)
  4. FL-MAP-09 文档注释"完整公式见FL-MAP-16"，但实际两套公式的数学结果不同

  计算示例对比（张飞: 攻击95, 防御70, 智力30, 兵力3000, 无科技）:
  - FL-MAP-09: 3000 x ((95x3+70x2+30)/300) = 3000 x 1.383 = 4150
  - FL-MAP-16: 3000x1.0 + 95x10 + 70x8 + 30x5 = 3000 + 950 + 560 + 150 = 4660
  - 差异: 12%
- **可能的理解方式**: FL-MAP-09 P3 是"简化的战力预览"（用于攻城确认弹窗的快速估算），FL-MAP-16 S2 是实际战斗计算用的完整公式。需要确认城防衰减公式中的"攻方总战力"到底用哪个。
- **需补充内容**: 明确城防衰减公式 `每回合城防减少 = 攻方总战力 x ...` 中的"攻方总战力"用 FL-MAP-09 P3 公式还是 FL-MAP-16 S2 公式计算。建议统一为一个函数，在所有引用点标注公式版本号。
- **优先级**: **P0**

### F-P0-04: 城防衰减公式的分母在不同位置定义不同

- **流程/数据编号**: AC-09-01 vs PLAN.md C.4
- **实施困惑**: 两个版本的城防衰减公式分母结构不同:

  **AC-09-01 版本**:
  ```
  每回合城防减少 = 攻方总战力 x 策略修正 / (城防系数 x 回合数上限)
  城防系数 = 1.0 + 城墙等级 x 0.05
  ```
  分母 = `(1.0 + 城墙等级 x 0.05) x 20`（假设回合数上限=20）

  **PLAN.md C.4 版本**:
  ```
  defense_decay_per_tick = attacker_power x strategy_modifier / (wall_level x 5 + 100) x tick_interval
  ```
  分母 = `wall_level x 5 + 100`（无回合数上限参与）

  计算对比（城墙等级3, 回合数上限20）:
  - AC-09-01 分母: (1.0 + 0.15) x 20 = 23
  - PLAN.md C.4 分母: 3 x 5 + 100 = 115
  - 差异: 约 5 倍

  虽然有 [P1-M-09] 注释说"以 AC-09-01 为准"，但两者数学结构完全不同（一个有 `回合数上限` 参与分母，一个没有），这意味着:
  - AC-09-01: 总伤害被 `回合数上限` 控制，20 回合内几乎必然攻下（分母小，每回合伤害大）
  - C.4: 分母包含 `wall_level x 5 + 100`，城墙等级高时很难攻下

  这不是参数微调的问题，是游戏平衡的根本性差异。
- **可能的理解方式**: AC-09-01 是权威版本（文档多次声明）。PLAN.md C.4 是旧版但未标注废弃。
- **需补充内容**: 在 PLAN.md C.4 处明确标注 `[已废弃，以 FL-MAP-09 AC-09-01 为准]`。同时在 AC-09-01 中验证: 当前分母 `(1.0 + 0.05x等级) x 20` 在城墙等级10时 = 30，这意味着每回合衰减 = 战力/30，20 回合总衰减 = 战力 x 20/30 = 0.67 x 战力。需确认这是否为设计意图（城防值需要大约 0.67 倍战力才能扛住）。
- **优先级**: **P0**

### F-P1-01: SiegeTask 状态机 `cancelled` 状态缺少从 `preparing` 跳转的定义

- **流程/数据编号**: FL-MAP-09-08 状态机表
- **实施困惑**: 状态机表格中:
  - `marching -> cancelled` 有定义（玩家取消/超时）

  - 但 [INT-R2-08] cancelled 终态 UI 行为中提到 "preparing 阶段取消: Toast '行军已取消' + 将 marchStartTime 设为 null"
  - 异常跳转表中 Phase 1 有 "玩家取消/返回 → 关闭弹窗，回到地图空闲状态"

  问题: preparing 阶段用户按返回/取消时，SiegeTask 是否已经创建？如果 P4 原子事务已完成（SiegeTask 已创建为 preparing），用户在 P5 之前取消，状态机没有 `preparing -> cancelled` 转换。如果 SiegeTask 尚未创建（取消发生在 P4 确认之前），则不存在 cancelled 转换问题，但 INT-R2-08 的 preparing 阶段取消描述是多余的且会产生歧义。
- **可能的理解方式**: SiegeTask 在 Stage P4 点击"发动攻城"后才创建（P5），所以 preparing 阶段的取消不会经过状态机，直接清理 UI 即可。INT-R2-08 的描述是为边缘情况准备的。
- **需补充内容**: 明确 SiegeTask 的创建时机（P4 确认时还是 P5 开始时）。在状态机表格中补充 `preparing -> (无SiegeTask) / cancelled` 的说明，或移除 INT-R2-08 中 preparing 的描述。

### F-P1-02: `settling` 阶段的持续时间未明确界定

- **流程/数据编号**: Stage P9 / FL-MAP-16 编队映射表
- **实施困惑**: FL-MAP-16 编队映射表声明 `settling` 阶段 "预计持续时间 < 5s"，但实际 settling 阶段包含:
  1. 结果动画 (2~3s)
  2. 伤亡计算 (<200ms)
  3. 奖励计算 (<200ms)
  4. 显示结果弹窗 -- **等待用户点击**
  5. 玩家点击"领取奖励" -- **等待用户操作**
  6. 奖励写入事务 (<0.5s)

  步骤 4 和 5 的等待时间完全取决于用户操作，可能是几秒也可能一直不点。如果 settling 阶段包含这些等待，则 "< 5s" 完全不准确。如果不包含，那么状态转换 `settling -> returning` 是在哪个时间点触发？
- **可能的理解方式**: settling 仅指"系统结算操作"（步骤 1-3），不包含用户交互（步骤 4-5）。状态在结算完成后转为 returning，用户在弹窗中的操作异步进行。
- **需补充内容**: 明确 `settling -> returning` 的触发条件。如果弹窗等待不影响状态转换，需在 P9 结算后执行序列图中标注状态转换的精确触发点。
- **优先级**: **P1**

### F-P1-03: 攻城失败时是否走回城流程未明确

- **流程/数据编号**: Stage P9 / Stage P10
- **实施困惑**: L1 主流程 Phase 4 的进入条件写的是"攻城战斗结束（Phase 3 完成），战斗结果已确定"，退出条件包含"编队回到出发城市"。但:

  - 胜利: 编队回城，路径为去程逆序，目的地为"最近己方城市"（可能是刚攻下的城）
  - 失败: 编队是否也需要回城？文档说"编队状态更新 → 自动回城"，但失败情况下编队可能严重减员（惨败损失50%~80%）
  - 撤退: 走中断结算 → 回城 → 销毁

  Stage P10 的系统行为第 6 条 "编队对象销毁，将领释放为可用状态"，以及编队生命周期终点统一声明"攻城完成后（无论胜败），编队对象执行以下操作"。这暗示失败也要回城。但城防初始值公式（版本 A）中的 "守军驻防兵力" 在失败后是否有变化？如果城池未被攻下，守军状态是否改变？
- **可能的理解方式**: 所有结果（胜/败/撤退）都走回城流程，只是结算内容不同。失败时没有奖励、没有领土变更，但仍然需要回城和编队销毁。
- **需补充内容**: 在 Stage P9 的战斗结束分支表中，失败分支应明确标注 "转至 Stage P10（编队回城+销毁）"，目前只写了 "Stage P9（失败结算）"。
- **优先级**: **P1**

### F-P1-05: 5档判定条件中"战斗时间 T"的精确含义未定义

- **流程/数据编号**: FL-MAP-09-12 5档伤亡判定表
- **实施困惑**: 5档判定基于"战斗时间 T"，条件如 `T < 20s`、`20s <= T < 30s` 等。但:
  - T 是从什么时间点开始计算的？Stage P8 的系统行为第 2 条说"进入回合制战斗"，是从第 1 回合开始？还是从集结动画结束后开始？

  - 如果 T 从第 1 回合开始，每回合 1s，最长 20 回合，那 T 的范围是 1~20s，但 5 档表中出现 `30s <= T <= 60s`。这与"最长 20 回合（每回合 1s）= 20s"矛盾。
  - CR-04 写"最长20回合，每回合1s"，但 L2 文档 Stage P8 写"最长60s/20回合"。20 回合 x 1s = 20s，不是 60s。CR-04 的 60s 从何而来？

- **需补充内容**: 明确 T 的精确定义（起始时间点、是否含集结动画、是秒数还是回合数）。统一 CR-04、Stage P8、5档表中的时间描述。
- **优先级**: **P1**

### F-P1-06: 恢复超时在 Phase 2（行军阶段）的行为与 Phase 3 描述存在冲突

- **流程/数据编号**: Stage P6 [V-006] vs Stage P8

- **需补充内容**: 在异常汇总表 ERR-S09 中明确说明 Phase 2 恢复超时走 sieging 撤退规则的原因。在 P0-V07 统一取消规则中补充 Phase 2 恢复超时的特殊条目。
