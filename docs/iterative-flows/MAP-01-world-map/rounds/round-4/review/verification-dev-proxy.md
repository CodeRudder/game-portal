## 新发现

### NEW-001 [P0]: SiegeTask 状态机表缺少 marching 阶段恢复超时转换

- **发现维度**: 状态机完整性
- **位置**: `FL-MAP-09-siege-warfare.md` FL-MAP-09-08 状态机表 (line 646)
- **问题**: 状态机表中 `marching` 状态行写的是 `marching | 玩家取消/超时 | cancelled | 100%退还`，将"主动取消"和"恢复超时"捆绑在一起指向 `cancelled` 终态。但 ISS-010 修复明确规定 marching 阶段恢复超时应走 `settling(撤退) -> returning` 路径（50%退还+扣次数+5分钟冷却），与主动取消（100%退还、cancelled 终态）是两条完全不同的路径。L1 主流程 ERR-S09 和取消退款汇总表已正确反映此区别，但 L2 状态机表未更新。
- **影响**: 工程师实现状态机时可能按照表格实现为 marching 恢复超时 -> cancelled（100%退还），与文档其他位置的 50%退还规则矛盾。
- **建议修正**: 将状态机表中 `marching` 行拆分为两行：
  - `marching | 主动取消 | cancelled | 100%退还，不扣次数，不冷却`
  - `marching | 恢复超时(30s移动/10sPC) | settling(撤退简化结算) | 50%退还，扣次数，5分钟冷却`

### NEW-002 [P0]: SiegeTask 状态机表 sieging 恢复超时时间未同步 ISS-008 修改

- **发现维度**: 状态机完整性
- **位置**: `FL-MAP-09-siege-warfare.md` FL-MAP-09-08 状态机表 (line 651)
- **问题**: 状态机表 `sieging | 恢复超时（10s）` 仍使用旧的超时时间 10s。ISS-008 修复已将恢复超时改为移动端 30s/PC端 10s。AC-09-02、CR-10、ERR-S14 等多处已同步更新，但状态机表中此行未更新。
- **影响**: 工程师可能按 10s 实现恢复超时，与 AC-09-02 的 30s/10s 矛盾。
- **建议修正**: 将 `恢复超时（10s）` 改为 `恢复超时（30s移动/10sPC）`。

### NEW-003 [P1]: 5档判定表与5->3档映射表中 victory 损失范围不一致

- **发现维度**: 公式完整性
- **位置**: `FL-MAP-09-siege-warfare.md` FL-MAP-09-12
- **问题**: 5档判定表(line 713-715)中险胜(victory)的兵力损失为"中(15%~20%)"，但5->3档映射表(line 466)中险胜映射到 victory 的损失比例为 5%~15%。险胜在5档表中可达 20%，但映射到3档后上限变为 15%，差距 5%。PLAN.md C.6 定义 victory 的 casualty_rate 为 5%~15%，FL-MAP-17 按3档计算。5档表中的险胜损失范围(15%~20%)超出了3档 victory 的上限(15%)。
- **分析**: 可能的解读是：(1) 5档表为UI展示用途（文字描述而非精确范围），(2) 实际伤亡计算统一走 FL-MAP-17 的3档系统(victory=5%~15%)。但这种解读对工程师而言不直观。
- **影响**: 险胜时5档UI显示损失"15%~20%"但实际计算为"5%~15%"，用户看到的数字范围与实际不一致。
- **建议修正**: 将5档表中"险胜"的损失范围修正为与3档 victory 一致的"中（5%~15%）"，或在映射表处添加明确注释"5档损失范围为UI展示描述，实际计算统一使用3档范围"。

### NEW-004 [P1]: SiegeTask 状态机 ASCII 图缺少 marching->settling 路径

- **发现维度**: 状态机完整性
- **位置**: `FL-MAP-09-siege-warfare.md` FL-MAP-09-08 (line 632-638)
- **问题**: ISS-010 新增了 marching 阶段恢复超时 -> settling(撤退) 的转换路径，但 ASCII 状态机图未更新。当前图中 `marching` 状态仅有 `-> cancelled` 一条异常路径，缺少 `-> settling(恢复超时)` 路径。L1 主流程的异常汇总表和退款政策汇总表已正确反映此路径。
- **影响**: 状态机图与表格描述不一致，工程师可能遗漏此路径。
- **建议修正**: 更新 ASCII 状态机图，添加 `marching -> settling(恢复超时)` 路径。

### NEW-005 [P2]: 5档判定表中"大胜""胜利""险胜"的损失范围为文字描述非精确数值

- **发现维度**: 公式完整性
- **位置**: `FL-MAP-09-siege-warfare.md` FL-MAP-09-12 (line 711-717)
- **问题**: 5档判定表中损失列使用文字描述"低(5%~10%)""中低(10%~15%)""中(15%~20%)"等，这些范围在括号内有精确数值。但 5->3 映射表和 PLAN.md C.6 中 victory 统一为 5%~15%。文档未明确说明5档表的括号内数值是"推荐展示范围"还是"实际计算范围"。
- **影响**: 轻微歧义，不阻塞实施（因有明确的"5档仅UI展示，实际用3档"声明）。
- **建议修正**: 在5档表上方添加注释："括号内损失范围为UI展示参考值，实际伤亡计算统一使用5->3映射表中的3档范围，由 FL-MAP-17 执行。"

### NEW-006 [P1]: Stage P8 城防衰减公式中"攻方总战力"引用 FL-MAP-16 S2 但未说明是否含策略修正

- **发现维度**: 公式完整性
- **位置**: `FL-MAP-09-siege-warfare.md` Stage P8 (line 403), 关键公式索引 (line 788)
- **问题**: 城防衰减公式 `每回合城防减少 = 攻方总战力 x 策略修正 / (城防系数 x 回合数上限)` 中，"攻方总战力"标注使用 FL-MAP-16 S2 完整公式。但 FL-MAP-16 S2 的完整公式 `兵力x1.0 + 攻击x10 + 防御x8 + 智力x5 + 科技加成` 不含策略修正（策略修正在 PLAN.md C.2 中是独立叠加的 `force_combat_power = base x strategy_modifier x terrain_modifier`）。城防衰减公式中的 `decay_strategy_modifier` 是额外乘的。这意味着实际计算中攻方总战力 = FL-MAP-16 S2 base（不含策略）x decay_strategy_modifier，而非 FL-MAP-16 S2 result（已含 power_strategy_modifier）x decay_strategy_modifier。文档未明确"攻方总战力"是指 base 值还是含 power_strategy_modifier 的值。
- **影响**: 如果攻方总战力已含 `power_strategy_modifier`(强攻1.5)，再乘以 `decay_strategy_modifier`(强攻1.2)，则强攻总修正为 1.5x1.2=1.8，可能非设计意图。
- **建议修正**: 明确"攻方总战力"在城防衰减公式中使用 FL-MAP-16 S2 的 base 值（不含 power_strategy_modifier），策略效果通过 decay_strategy_modifier 独立计算。或明确说明叠加关系。
