## P0-V04: Siege Battle Recovery Timeout Undefined

### Problem
FL-MAP-09 P8 application recovery says "timeout -> treat as retreat" but never specifies timeout duration. User sees spinner with no countdown, creating a user-facing dead end.

### Fix Applied
- **Recovery timeout defined = 10 seconds** (fixed value)
- **Countdown display**: "恢复战斗进度... ({N}s)" where N counts down from 10 to 0
- **Timeout behavior**: Treated as siege-phase retreat (50% resource refund, daily attempt consumed, 5min cooldown)
- **Timeout toast**: "恢复超时，按撤退处理（已退还50%资源）"

## P0-V05: Battle Logic Error Dead End

### Problem
FL-MAP-05-02 handles "battle logic error (FL-MAP-09-P8 returns error)" by showing "battle error" but provides no recovery path. Were resources consumed? Was daily attempt used? Can they retry?

### Fix Applied
Complete recovery flow defined after battle logic error:
1. **Resources**: 100% refunded (food/grass + strategy items + troops)
2. **Daily attempt**: NOT consumed
3. **Cooldown**: NOT triggered
4. **User can retry immediately**
5. **UI**: "战斗异常，资源已退还，请重新尝试" panel with:
   - [重试] button -> restarts siege flow at FL-MAP-09 Stage P2
   - [返回地图] button -> closes all siege panels, returns to map idle

## P0-V07: Cancel Refund Policy Contradiction

### Problem
FL-MAP-09 P6 says "marching cancel: 100% refund, no daily count, no cooldown" but FL-MAP-09-01 says "player cancels siege: no resource refund". These contradict each other. The boundary between phases was unclear.

### Fix Applied
Clear two-phase cancel/retreat policy defined:

| Phase | SiegeTask State | Resource Refund | Daily Count | Cooldown |
|:------|:---------------|:----------------|:-----------|:---------|
| Marching-phase cancel | preparing/marching | 100% (food + items + troops) | NOT consumed | NOT triggered |
| Siege-phase retreat | sieging | 50% (food + items, troops fully kept) | Consumed | 5 minutes |

### Files Changed

**FL-MAP-09** (`/docs/iterative-flows/MAP-01-world-map/flows/FL-MAP-09-siege-warfare.md`):

## P0-V10: No General Available = Dead End

### Problem
If all generals are injured or deployed, siege cannot proceed and the flow provides no guidance. User has no idea how long to wait. Tutorial mode has no protection against this state.

### Fix Applied
When general list is empty (all unavailable), show "暂无可用将领" panel with:
- **Injured generals**: Each shown with name + injury level (轻伤/中伤/重伤) + recovery countdown
- **Deployed generals**: Each shown with name + return countdown
- **Estimated wait time**: min(all recovery/return times), displayed as "最早可用: {name} {time}"
- **[返回] button**: Closes siege confirmation panel, returns to map
- **Tutorial protection**: Tutorial mode ensures tutorial general is always available; if data is corrupted, force-reset tutorial general to available state
