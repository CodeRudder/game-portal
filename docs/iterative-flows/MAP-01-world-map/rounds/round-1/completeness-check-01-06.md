## 架构问题（本地游戏 vs 服务端描述）

三国霸业是**本地放置游戏**，所有战斗逻辑、地图数据、行军计算等均在客户端完成。以下流程描述中存在不合理的"服务端/WebSocket/网络请求"描述，需标记为架构问题。

| 编号 | 流程 | 位置 | 问题描述 | 建议修正 |
|------|------|------|---------|---------|
| ARCH-01 | FL-MAP-01 | FL-MAP-01-02 地图数据加载 | 描述为"网络请求耗时"、"加载失败（网络错误）"、"超时15s" | 本地游戏地图数据应从本地JSON/IndexedDB加载，不应依赖网络。应改为"本地数据加载"、"数据解析失败"等 |
| ARCH-02 | FL-MAP-01 | FL-MAP-01-02 异常流程汇总 | "网络加载失败"应改为本地数据加载异常 | 若确有云存档同步场景，应明确区分"本地数据加载"和"云端同步"两个不同阶段 |
| ARCH-03 | FL-MAP-01 | 子流程FL-MAP-01-01 关键行为 | "完成标记: 服务端写入 player.tutorialCompleted: true" | 本地游戏应使用 localStorage/IndexedDB 存储，不应依赖服务端写入 |
| ARCH-04 | FL-MAP-01 | 状态持久化 | "产出上限完成标记 — 存储方式: 服务端 — 永久" | 本地游戏应使用本地持久化（localStorage/IndexedDB），不应标记为"服务端"存储 |
| ARCH-05 | FL-MAP-04 | FL-MAP-04-02 拦截状态机 | "拦截触发: 服务端推送 intercept 事件" | 本地游戏的拦截战斗应由客户端本地逻辑判定触发，不存在"服务端推送" |
| ARCH-06 | FL-MAP-05 | FR-05-05 | "支持断线重连后恢复攻城动画状态" | 本地游戏无断线概念。若指App切后台恢复，应改为"应用恢复后/切回前台后恢复攻城状态" |
| ARCH-07 | FL-MAP-05 | FL-MAP-05-01 异常分支 | "玩家断线重连"（出现两次） | 本地游戏应改为"应用切回前台恢复"或"页面可见性恢复" |
| ARCH-08 | FL-MAP-05 | FL-MAP-05-02 步骤详情 | "战斗总时长10~60s（由服务端战斗逻辑决定）" | 本地游戏战斗逻辑应在客户端执行，应改为"由客户端战斗引擎决定" |
| ARCH-09 | FL-MAP-06 | NFR-06-02 | "领土详情数据请求（GET /api/map/territory/{id}/detail）响应时间 <500ms" | 本地游戏应从本地数据源读取，不应有REST API调用。应改为"本地数据查询响应时间" |
| ARCH-10 | FL-MAP-06 | S5 展示详情信息 | "前端通过 territoryId 调用 GET /api/map/territory/{id}/detail 获取聚合数据" | 本地游戏应从本地数据模型/Store中获取，不应调用HTTP API |
