## 全局一致性检查

### 跨流程一致性

#### ICC-01: FL-MAP-04 与 FL-MAP-09 使用旧流程编号体系引用
- **问题**: FL-MAP-04 的触发条件写 "MAP-F06-P6"，后置条件写 "MAP-F06-P7/P9"，FL-MAP-05 写 "MAP-F06-P8/P9"。但当前流程编号体系为 FL-MAP-XX，MAP-F06 是旧体系。其他流程（FL-MAP-01/06/07/08）均使用新编号。
- **影响**: 开发者在 FL-MAP-04/05 中需要额外心理映射，容易混淆。
- **建议**: FL-MAP-04 和 FL-MAP-05 中所有 `MAP-F06-P*` 引用替换为 `FL-MAP-09 Stage P*`。

#### ICC-02: FL-MAP-05-02 攻城动画时长与 FL-MAP-09 Stage P8 战斗时长矛盾
- **问题**: FL-MAP-05-02 流程结构标注 "5s~15s"，步骤详情又说 "5~15秒...与 MAP-F06-P8 战斗逻辑对齐（最短10秒，最长60秒）"。FL-MAP-09 Stage P8 战斗为 20 回合（最长 60s）。攻城动画是循环播放直到战斗结束？还是播放一次？
- **影响**: 动画组和战斗逻辑组对时长的理解可能不同步。
- **建议**: 明确攻城动画是"循环播放直到收到战斗结束信号"模式，统一描述为"动画单次循环 5~15s，战斗总时长 10~60s，动画循环直至战斗结束"。

#### ICC-03: FL-MAP-06 引用旧编号 MAP-F03，FL-MAP-07 引用 MAP-F04
- **问题**: FL-MAP-06 头部写"旧流程对应: MAP-F03"，FL-MAP-07 写"MAP-F04"。虽然作为"旧流程对应"可以接受，但与 FL-MAP-04/05 中大量使用旧编号作为当前引用混在一起，降低了可读性。
- **建议**: 统一旧流程引用格式，仅在元数据区域标注旧编号对应关系，正文中统一使用 FL-MAP-XX。

#### ICC-04: FL-MAP-09 Stage P3 编队选择引用 FL-MAP-09-09，但编号跳跃
- **问题**: FL-MAP-09 的子流程编号为 01~15，但跳过了 04 和 08（无 FL-MAP-09-04/FL-MAP-09-08）。编号不连续无说明。
- **建议**: 补充说明跳过原因，或重新编号使子流程连续。

#### ICC-05: FL-MAP-18 手机端色块尺寸与全局 8x8 定义冲突
- **问题**: FL-MAP-18 性能优化中提到手机端 "16x16px 色块"（异常E18-05），但全局所有文档（FL-MAP-01~17）的色块基础尺寸为 8x8px。如果 16x16 是逻辑尺寸，色块数量变化影响视口裁剪、寻路、坐标映射等全部下游逻辑。
- **影响**: 如果实施团队不明确这是"逻辑替换"还是"渲染缩放"，会导致坐标系统不统一。
- **建议**: 明确手机端色块变化是逻辑层替换还是渲染层缩放，并在 FL-MAP-03/FL-MAP-18 中分别说明。

#### ICC-06: FL-MAP-09 与 FL-MAP-18 攻城流程步骤顺序不一致
- **问题**: FL-MAP-09 Stage 顺序为 P1(选目标) → P2(确认) → P3(编队) → P4(策略) → P5~P10。FL-MAP-18 Step Wizard 顺序为 1/4(目标情报) → 2/4(攻城策略) → 3/4(编队配置) → 4/4(确认出征)。Stage P3(编队) 和 P4(策略) 的顺序在 PC 和移动端不同。
- **影响**: 移动端策略选择在编队之前，但 PC 端编队在策略之前。策略倍率影响编队战力计算（FL-MAP-09-02 中策略有"胜率加成"），如果策略在编队之后选，战力预览可能不准确。
- **建议**: 统一 PC/移动端的步骤顺序，或说明移动端"策略先选"时战力预览如何计算。

### 术语一致性

#### ICT-01: "领土" vs "领地" 混用
- **问题**: FL-MAP-10 快捷按钮使用"高产领地"（FL-MAP-10-06），其余所有流程统一使用"领土"。
- **建议**: 统一为"领土"。

#### ICT-02: "攻占" vs "攻城" vs "征服" 边界模糊
- **问题**: FL-MAP-06 S6 定义清晰："征服"(FL-MAP-07) 用于非城池，"攻占"(FL-MAP-09) 用于城池。但 FL-MAP-09 自身标题是"攻城战"，FL-MAP-11 标题是"攻城奖励领取"。FL-MAP-09 内部混用"攻城"、"攻占"、"攻城战"。
- **建议**: 统一术语定义："攻城战"指整个流程，"攻占"指对城池的攻击行为，"征服"指对非城池的快速占领。

#### ICT-03: "兵力池" vs "全局兵力池" vs "TotalArmyPool"
- **问题**: FL-MAP-08 使用"全局兵力池"和"TotalArmyPool"数据模型名。FL-MAP-16 使用"全局兵力池"。FL-MAP-17 使用"unallocated"（字段名）。术语层级混乱。
- **建议**: 统一命名：模型名 TotalArmyPool，中文名"全局兵力池"，字段名 unallocated/cap/replenishRate，在 INDEX.md 中定义一次。

#### ICT-04: 区域命名 "C区/E区/D区" 不一致
- **问题**: 大部分流程使用 C区（地图）和 E区（弹窗）。FL-MAP-04 使用 E区（行军预览面板）。但 FL-MAP-13 使用"D区 420x696px"（D区未在任何其他流程中定义）。
- **建议**: 在 INDEX.md 中统一区域命名表（A区/B区/C区/D区/E区 分别是什么），各流程引用统一。

#### ICT-05: "产出上限" 公式缺失
- **问题**: FL-MAP-01 子流程 FL-MAP-01-02 定义产出上限机制，但"资源上限"的具体计算公式仅引用 PRD。FL-MAP-06 产出详情（FL-MAP-06-01）和 FL-MAP-13 统计面板均依赖此上限值。3 个流程都引用同一公式但无内联定义。
- **建议**: 至少在 INDEX.md 公共数据字典中定义一次产出上限公式。

### 数据一致性

#### ICD-01: 伤亡规则三处定义不一致
- **问题**:
  - FL-MAP-05-03: 3 档（胜利 5%~15%、失败 20%~40%、惨败 50%~80%）
  - FL-MAP-09-14: 5 档（大胜/胜利/险胜/失败/惨败，按战斗时间 T 判定）
  - FL-MAP-17: 3 档（胜利 5%~15%、失败 20%~40%、惨败 50%~80%）
  - FL-MAP-09 Stage P9 与 FL-MAP-17 的映射关系未定义。5 档中"险胜"映射到哪个？
- **影响**: 这是最严重的数据冲突之一。前后端如果各取一套，伤亡计算会不一致。
- **建议**: 以 FL-MAP-09（5 档）为攻城战权威来源，FL-MAP-17（3 档）为征服战权威来源，FL-MAP-05 仅展示不计算。明确 5 档→3 档映射表。

#### ICD-02: 战力计算公式两处不同
- **问题**:
  - FL-MAP-09 Stage P3: 编队战力 = `SUM(每将领(武x3+统x2+智x1) x 等级系数) + 兵力 x 单兵战力`
  - FL-MAP-17 S4: 战力重算 = `基础战力 x (1 - 伤亡比例) x (1 - 受伤减益)`
  - 两套公式用途不同但未说明关系。
- **建议**: 明确 FL-MAP-09 是"初始战力计算"，FL-MAP-17 是"战后战力修正"，并定义完整的战力计算链路。

#### ICD-03: 城防衰减两套公式矛盾
- **问题**: FL-MAP-09 Stage P8 给出两套城防衰减公式（回合制和每秒实时UI），策略修正值不同（强攻: 回合x1.0 vs 每秒1.5；内应: 回合x1.5 vs 每秒1.3），城防系数定义不同（回合: 城墙等级x0.05; 每秒: 常数100）。
- **影响**: 实时UI显示的衰减率可能与服务端真实计算不匹配，导致用户看到的进度条与实际结果不一致。
- **建议**: 明确回合制为服务端权威计算，每秒公式为UI近似显示（平滑进度条），标注差异。

#### ICD-04: 行军时间核心常量"游戏时间系数"全局缺失
- **问题**: FL-MAP-04 定义行军时间双轨计算（战略时间 vs 动画时长），FL-MAP-09 Stage P6 引用行军动画。但"游戏时间系数"的值从未在任何 18 个文档中给出。没有此常量，无法计算行军战略时间。
- **影响**: 行军系统无法实施。
- **建议**: 在 INDEX.md 公共常量区域定义游戏时间系数。

#### ICD-05: 阵营色值三处不同
- **问题**:
  - FL-MAP-03 城市渲染: 魏 #2E5090, 蜀 #8B2500, 吴 #2E6B3E
  - FL-MAP-04 精灵调色板: 魏 #3E7AB8, 蜀 #C83E3E, 吴 #3E8B3E
  - 同一阵营在不同渲染层使用不同色值。
- **建议**: 统一为同一套阵营色，或在 INDEX.md 中定义"城市色/精灵色/旗帜色"分别使用的色值及原因。

#### ICD-06: 胜率公式关键系数缺失
- **问题**: FL-MAP-07-02 胜率公式中 "武将战力系数" 和 "防御系数" 的值未给出。FL-MAP-06 TerritoryDetailViewModel 中 "战力系数" 同样未定义。
- **影响**: 无法实施胜率预估功能。
- **建议**: 给出默认值（如无武将时 = 1.0）和计算公式。

#### ICD-07: 兵力池 cap 公式变量定义不清
- **问题**: FL-MAP-08 定义 TotalArmyPool.cap = "基础兵力(500) + 领土数 x 200 + 建筑加成(兵营等级 x 50)"。"领土数"是己方领土总数？"兵营等级"是所有兵营等级之和？
- **建议**: 明确变量含义。

### 流程衔接

#### ICF-01: FL-MAP-09 Stage P9 → FL-MAP-11 过渡不明确
- **问题**: FL-MAP-09 P9 说"计算奖励暂存为 pendingReward"，FL-MAP-11 说"触发条件: FL-MAP-09 阶段9"。但 FL-MAP-09 Stage P7→P8→P9 的流程中，结果动画(FL-MAP-05-03)、结果弹窗(FL-MAP-11-01)、伤亡系统(FL-MAP-17)的执行顺序不清晰。
- **建议**: 定义 P9 结算后的精确执行序列：结果动画 → 伤亡计算(FL-MAP-17) → 奖励暂存(FL-MAP-11) → 结果弹窗显示。

#### ICF-02: FL-MAP-07 征服胜利 → FL-MAP-08 驻防管理的过渡方式未描述
- **问题**: FL-MAP-07 S8 说胜利后"自动驻防 50% 兵力上限"，后置状态写"进入驻防管理(FL-MAP-08)"。但用户看到的是什么？面板自动弹出？还是仅后台执行自动驻防？
- **建议**: 明确征服胜利后是"自动驻防（后台）+ Toast 通知"还是"弹出驻防管理面板让玩家确认"。

#### ICF-03: FL-MAP-06 → FL-MAP-07/09 的入口条件校验不一致
- **问题**: FL-MAP-06 S6 按钮渲染规则基于 owner 和 type，但 FL-MAP-07 S2 还有额外校验（相邻+兵力）。如果用户在 FL-MAP-06 看到了"征服"按钮，点击后才被告知不相邻，体验不好。
- **建议**: FL-MAP-06 S6 按钮渲染时就应该考虑基本的可操作性（如是否相邻），不满足条件时按钮置灰。

#### ICF-04: FL-MAP-15 离线事件 → FL-MAP-12 在线事件的累积规则缺失
- **问题**: FL-MAP-12 定义在线事件最多 3 个同时存在。FL-MAP-15 定义离线累积事件。但离线累积事件是否受 3 个上限约束？FL-MAP-15-01 说"累积上限详见PRD"，但未在流程文档中内联。
- **建议**: 明确离线累积事件的上限和与在线 3 个上限的关系。

#### ICF-05: FL-MAP-17 编队兵力为 0 与 FL-MAP-09 Stage P10 回城逻辑冲突
- **问题**: FL-MAP-17 S4 说"编队兵力降为 0 → 编队自动解散"。FL-MAP-09 P10 说"生成回城行军精灵 → 剩余兵力归入兵力池"。兵力为 0 时是否还有回城精灵？将领如何回归？
- **建议**: 定义兵力为 0 的边界处理：无回城精灵，将领以"受伤状态"直接传送回出发城市。
