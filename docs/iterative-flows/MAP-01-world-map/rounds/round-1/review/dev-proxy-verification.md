### P0-01 | FL-MAP-09 | 本地战斗引擎公式集不完整

**流程/数据**: FL-MAP-09 Stage P8, FL-MAP-09-11

**实施困惑**: 文档反复提及"本地战斗引擎"执行回合制计算，给出了每回合城防衰减公式 `攻方总战力 x 策略修正 / (城防系数 x 回合数上限)`，但完整的战斗引擎公式集缺失:
1. 攻方兵力是否在每回合递减？还是仅在战斗结束时一次性计算伤亡？
2. 守方是否有反击？城防衰减是唯一战斗计算吗？
3. 战斗时长10~60s与20回合的关系是固定的(每回合1s)还是可变？
4. "攻方总战力"的完整公式仅引用PRD未内联(见P0-02)

**可能的理解方式**: 城防衰减为唯一计算(攻方不损失兵力)，战斗结束后由FL-MAP-17一次性计算伤亡。每回合1s，最长20回合=60s。

**需补充内容**: 战斗引擎的完整公式清单 -- 每回合计算步骤、攻方是否有损失、回合与时间的精确映射。

**优先级**: P0

## 跨流程问题

### X-01 | AI敌方系统

18个流程中多处提及"AI敌方"行为(AI攻城FL-MAP-06、AI行军FL-MAP-04、AI占领领土FL-MAP-15)，但从未有任何流程定义AI的行为规则。如果AI是核心游戏机制，缺少AI行为定义将导致:
- 离线变化(FL-MAP-15)无法计算
- 行军拦截(FL-MAP-04)无法触发
- 领土丧失后无法重新获取
- "已被AI敌方占领"(FL-MAP-06 E06-04)无法发生

**建议**: 如果AI系统在当前实施范围外，应在FL-MAP-01或PLAN.md中明确声明AI行为将在后续迭代中定义，当前阶段离线变化仅模拟产出/事件，不模拟AI领土变化。

### X-02 | 核心数据模型定义分散

以下数据结构在多个流程中被引用但无独立完整定义:

| 数据结构 | 使用流程 | 定义状态 |
|---------|---------|---------|
| TerritoryData | FL-MAP-03/06/07/08/14 | 仅在PRD中，未内联 |
| SiegeTask | FL-MAP-09 | 状态机已定义，完整schema缺失 |
| TotalArmyPool | FL-MAP-07/08/09/16 | 三要素已定义，replenishRate值缺失 |
| MarchToSiegeTransition | FL-MAP-04/05 | 字段列表已给出，完整类型缺失 |
| PendingReward | FL-MAP-11 | **已完整内联(P1-33)** -- 范例 |
| PathNode | FL-MAP-04 | 提及x/y/px/py，完整类型缺失 |
| ForceData(编队) | FL-MAP-16 | 状态枚举已定义，完整schema缺失 |
| MapEvent | FL-MAP-12 | 引用但完整schema缺失 |

**建议**: 参照PendingReward(FL-MAP-11 P1-33)的内联方式，为所有核心数据模型补充完整TypeScript接口定义。

### X-03 | PRD引用密度

18个流程中"详见PRD"类引用统计(按参数类型):

| 参数类别 | 引用次数 | 对编码的影响 |
|---------|:-------:|------------|
| 公式/计算规则 | ~25处 | 直接影响核心逻辑实现 |
| 数值参数(阈值/倍率/成本) | ~30处 | 直接影响游戏平衡和数据校验 |
| UI规格(尺寸/布局/动画) | ~15处 | 影响UI组件实现 |
| 数据结构定义 | ~10处 | 影响TypeScript类型定义 |

约55处公式/数值类引用是实施编码的直接输入，如果PRD不可立即获取或不够精确，将直接阻塞编码。

## 领土生命周期状态机缺失

18个流程中缺少一个关键的跨流程状态机 -- **领土所有权生命周期**:

```
中立 → (征服/攻城) → 己方 → (被AI夺) → 敌方 → (再攻城) → 己方
                  → (放弃) → 中立(冷却期)
```

此状态机涉及FL-MAP-06(查看)、FL-MAP-07(征服)、FL-MAP-09(攻城)、FL-MAP-08-01(放弃)、FL-MAP-14(升级)、FL-MAP-15(离线变化)共6个流程，建议补充统一的领土状态机定义。
