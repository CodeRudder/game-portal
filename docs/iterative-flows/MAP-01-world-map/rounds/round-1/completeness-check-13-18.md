## 架构问题（本地游戏 vs 服务端描述）

三国霸业是**本地放置游戏**，所有核心逻辑在客户端本地运行，不依赖服务端实时通讯（除云存档等在线功能）。以下流程描述中存在不合理的"服务端/WebSocket/网络请求"描述，标记为架构问题：

| 编号 | 流程 | 问题描述 |
|------|------|---------|
| ARCH-001 | FL-MAP-13 | NFR-13-03提到"网络中断时使用本地缓存数据，标注数据可能不是最新"。作为本地游戏，统计数据完全在本地，不应有"网络中断"和"缓存数据"的概念。本地数据即为最新数据，无需区分在线/离线数据来源。 |
| ARCH-002 | FL-MAP-13 | 异常流程"网络中断"条目描述"使用本地缓存数据"。本地游戏的统计面板数据来自本地存储，不存在"网络中断"导致数据不是最新的场景。应删除此异常项或改为"数据读取异常"。 |
| ARCH-003 | FL-MAP-14 | 步骤5提到"同一数据库事务"。本地游戏使用本地存储（如IndexedDB/localStorage），不应描述为"数据库事务"。应改为"本地存储原子操作"或"客户端事务"。 |
| ARCH-004 | FL-MAP-15 | ISS-046定义"离线=WebSocket断开连接超过60秒"和"服务端最后收到心跳的时间"。本地游戏不存在WebSocket心跳和服务器连接。离线定义应为"应用进入后台/切出地图Tab超过60秒"。 |
| ARCH-005 | FL-MAP-15 | S1"请求离线变化数据"暗示从服务端请求数据。本地游戏的离线变化应通过本地计算得出（对比本地快照），无需网络请求。 |
| ARCH-006 | FL-MAP-15 | AC-15-05"网络异常时最多重试3次获取离线数据"。本地游戏不应有"网络异常重试获取离线数据"的验收标准。应改为"本地数据读取异常时重试3次"。 |
| ARCH-007 | FL-MAP-15 | E15-02"网络异常导致离线变化数据获取失败"和E15-03"快速处理时服务端部分失败"。本地游戏不存在服务端处理和请求数据的网络依赖，应改为本地计算错误处理。 |
| ARCH-008 | FL-MAP-16 | AC-16-04"服务端二次校验防止并发冲突，乐观锁失败时提示重试"。本地游戏是单客户端单线程运行，不存在服务端并发冲突和乐观锁。应改为"客户端状态校验"或直接删除此条。 |
| ARCH-009 | FL-MAP-16 | S5步骤描述"向服务端发送编队创建请求"和"服务端二次校验将领状态和兵力池余量（防止并发冲突）"。本地游戏编队创建应在本地完成，无需向服务端发送请求。 |
| ARCH-010 | FL-MAP-16 | S5异常分支"服务端校验失败"和"网络失败"。本地游戏无服务端校验环节。E16-05"服务端乐观锁失败"、E16-06"确认时网络失败"、E16-08"并发边界"、E16-09"服务端判定"均涉及不存在的服务端交互。 |

## 总结

### 整体评价

6个流程文件在结构完整性上表现良好，10维度达标率为**91.7%**（55/60项达标）。主要不足集中在：

1. **时间描述不够精细** (1个未达标): FL-MAP-13部分步骤缺少⏱耗时

4. **空间描述遗漏** (1个未达标): FL-MAP-17缺少📱手机端适配描述

### 关键风险
