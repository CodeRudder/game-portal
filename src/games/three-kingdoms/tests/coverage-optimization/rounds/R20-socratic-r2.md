### 关键改进分析

**✅ 条件断言大幅修复（R17问题#1）**
- v19文件中，大部分 `if (stageIds.length > 0) { expect... }` 模式已改为前置断言 `expect(stageIds.length).toBeGreaterThan(0)`
- §1.2~§1.3 的关卡测试全部使用前置断言模式，消除了"空测试"风险
- §3 跨系统联动中仍有4处条件断言残留（第445/451/465/470行），但已有前置断言保护

**✅ 强/弱断言比从6.3:1提升到9.0:1**
- 新增测试文件大量使用 `toBe`、`toEqual`、`toBeGreaterThan` 等强断言
- campaign-chapters-1to3.test.ts 是标杆：零弱断言，全部精确验证
- pvp-full-chain 使用 `toBeCloseTo`、`toThrow` 精确验证业务规则

**⚠️ as any 未改善（449处不变）**
- 对抗性测试 adversarial.test.ts 贡献36处 `as any`（合理 — 对抗测试需要绕过类型系统）
- SiegeSystem.test.ts 零 `as any`（优秀）
- 但总体存量未清理

**❌ toStrictEqual 仍为0**
- R17建议未采纳，序列化/反序列化测试仍使用 `toEqual`
- 存在 `undefined` 属性差异被忽略的风险

## 三、仍存在的问题

### 🟡 问题1: 条件断言残留（严重程度降低）

**位置**: `v19-unification-upper-flow.integration.test.ts` §3 第445/451/465/470行
**模式**: `if (stageIds.length > 0) { ... expect ... }`
**影响**: 跨系统联动测试中，当数据为空时断言被跳过
**对比R17**: §1.2/§1.3已全部修复，仅§3残留4处，改善率约80%

### 🟡 问题2: as any存量未清理

**数据**: 449处，与R17持平
**分析**:
- 对抗性测试36处（合理）
- v19集成测试1处（合理）
- PvP集成测试2处（合理）
- **但大量存量文件中的 `as any` 未清理**
- 建议：新增文件已基本杜绝，存量可逐步清理

### 🟡 问题3: toStrictEqual零使用

**影响**: 序列化/反序列化测试中 `toEqual` 忽略 `undefined` 属性差异
**风险等级**: 中 — 如果序列化逻辑引入多余字段，`toEqual` 不会发现
**建议**: 至少在 SiegeSystem.serialize/deserialize 和 save/migration 测试中使用

### 🟢 问题4: tech-research-full-flow 条件断言集中

**位置**: `tech/__tests__/integration/tech-research-full-flow.integration.test.ts`
**数量**: 8处 `if (firstTier.length > 0)`
**模式**: 与R17中v19问题相同，但影响较小（科技树测试中前置条件确实可能为空）

### 🟢 问题5: eventBus.emit mock为空函数

**位置**: SiegeSystem.test.ts 等使用 `createMockDeps` 的文件
**影响**: 事件驱动的副作用完全不可测
**对比R17**: R17中此问题被评为🔴高，但新增的集成测试（v19/pvp）已使用真实引擎规避了此问题

## 四、改进建议（优先级排序）

### 建议1: 修复v19 §3残留条件断言（优先级: 🟡中）

```typescript
// 当前（第445行）
if (stageIds.length > 0) {
  campaign.completeStage(stageIds[0], 3);
}

// 修复为
expect(stageIds.length).toBeGreaterThan(0);
campaign.completeStage(stageIds[0], 3);
```

### 建议2: 序列化测试引入toStrictEqual（优先级: 🟡中）

在 SiegeSystem、CampaignSystem、TerritorySystem 的序列化/反序列化测试中：
```typescript
// 替换
expect(deserialized).toEqual(original);
// 为
expect(deserialized).toStrictEqual(original);
```

### 建议3: 创建共享test-utils（优先级: 🟢低）

 SiegeSystem.test.ts 中的 `createMockDeps` 可提取到 `test-utils/mock-deps.ts`，与v19/pvp的 `createSim` 并列，统一管理测试基础设施。

### 建议4: 对抗性测试扩展并发场景（优先级: 🟢低）

当前场景12"并发修改"实际是快速连续调用，并非真正并发。可考虑：
```typescript
// 使用 Promise.all 模拟真正并发
await Promise.all([
  engine.resource.addResource('grain', 100),
  engine.resource.addResource('grain', 200),
]);
```

## 六、总体评分: 8.1/10

### 评分明细

| 维度 | R17 | R20 | 变化 | 理由 |
|------|-----|-----|------|------|
| **覆盖率广度** | 9/10 | 9/10 | → | 651文件、18678用例，与R17持平 |
| **断言质量** | 8/10 | **9/10** | ↑1 | 强/弱比从6.3:1→9.0:1，新增文件断言质量极高 |
| **测试有效性** | 7/10 | **8.5/10** | ↑1.5 | v19条件断言核心问题已修复，新增文件有效性均≥8 |
| **Mock合理性** | 6/10 | **7.5/10** | ↑1.5 | 新增集成测试(v19/pvp)全部使用真实引擎，零mock |
| **可维护性** | 6/10 | **7.5/10** | ↑1.5 | 新增文件组织清晰，辅助函数可复用，但存量mockDeps未清理 |
| **边界覆盖** | 7/10 | **8/10** | ↑1 | 对抗性测试覆盖了极端边界，每日重置跨天模拟完整 |

### 评分理由

**提升0.9分的主要驱动力**:
1. **条件断言修复**（+0.4）: v19从"空测试"变为有效测试，这是R17最严重的问题，已基本解决
2. **强/弱断言比提升42.9%**（+0.3）: 从6.3:1→9.0:1，新增测试断言质量显著高于存量
3. **新增文件质量高于平均**（+0.2）: campaign-chapters(10/10)、pvp-full-chain(9/10)等树立了新标杆

**未能突破8.5的原因**:
1. as any存量449处未清理（-0.2）
2. toStrictEqual零使用，序列化深度验证缺失（-0.1）
3. v19 §3仍有4处条件断言残留（-0.1）
4. 存量mockDeps模式未统一（-0.1）

### 一句话总结

> **R17识别的两个核心问题（条件断言、mockDeps重复）中，条件断言已基本修复，mockDeps通过新增集成测试"绕过"而非"解决"。新增测试文件质量显著高于存量，形成了"campaign-chapters"和"pvp-full-chain"两个新标杆。下一步应聚焦于存量清理和toStrictEqual引入，可冲击8.5+。**
