# AI自驱动测试覆盖率突破方法论

> **版本**: v1.0  
> **适用项目**: 三国霸业 (Three Kingdoms) 游戏引擎  
> **核心理念**: 用对抗性思维打破AI测试的"自我确认循环"

---

## 1. 问题定义：AI测试盲区的五个维度

AI生成测试时，存在系统性的认知盲区。这些盲区不是随机错误，而是由AI的工作模式——"基于已有代码推导测试"——所决定的结构性缺陷。

### 1.1 功能盲区 (Functional Blind Spot)

**定义**: AI只测试了函数的"主路径"(happy path)，遗漏了次要功能分支和异常处理。

**典型表现**:
- 只测试 `buildBuilding()` 的成功路径，不测试资源不足、等级已满、前置未解锁等失败路径
- 只测试 `getReward()` 正常领取，不测试重复领取、过期领取、背包已满等边界
- 对 `void` 返回值函数的副作用验证不足（状态变更、事件触发）

**根因**: AI倾向于从函数签名推断行为，而非从业务规则推导场景。

### 1.2 边界盲区 (Boundary Blind Spot)

**定义**: 对数值边界、状态转换边界、集合边界缺乏感知。

**典型表现**:
- 资源为 0 时的运算（除零、负数溢出）
- 数组/列表为空或仅有一个元素时的算法行为
- 等级从 N 到 N+1 的临界点（升级消耗跳变、属性突变）
- 时间戳边界（跨天、跨赛季、离线时长极值）
- 整数溢出：`Number.MAX_SAFE_INTEGER`、`Infinity`、`NaN`

**根因**: AI缺乏对游戏数值系统的"数值直觉"，不会主动构造极端输入。

### 1.3 集成盲区 (Integration Blind Spot)

**定义**: 单个模块测试通过，但模块间交互产生的组合行为未被覆盖。

**典型表现**:
- 建筑升级消耗资源 → 资源系统扣减 → 事件通知 → 任务进度更新，这条链路断裂
- NPC好感度变化 → 解锁对话 → 触发任务 → 奖励发放，跨4个模块的流程
- 科技研究完成 → 效果应用 → 战力计算 → 排行榜更新，异步时序问题

**根因**: AI以模块为单位生成测试，缺乏跨模块的全局流程视角。

### 1.4 时序盲区 (Temporal Blind Spot)

**定义**: 对状态随时间变化的场景覆盖不足。

**典型表现**:
- 离线收益计算的边界：1分钟 vs 24小时 vs 7天 vs 365天
- 赛季切换时的数据重置与保留
- 并发操作：同一帧内多次 `tick()` 对状态的影响
- 异步回调顺序：Promise resolve 的不确定性
- 定时器精度：`setTimeout` 在不同环境下的行为差异

**根因**: AI倾向于测试"静态快照"，而非"状态变迁序列"。

### 1.5 数据盲区 (Data Blind Spot)

**定义**: 对不同数据规模、数据分布、数据异常的覆盖不足。

**典型表现**:
- 空存档加载、损坏存档加载、跨版本存档迁移
- 大量NPC（1000+）时的性能与正确性
- 配置数据缺失或非法时的降级行为
- 序列化/反序列化的往返一致性（round-trip）

**根因**: AI默认使用"标准数据"测试，不会主动构造异常数据集。

## 2. 突破策略：6层测试金字塔

传统测试金字塔是3层（单元→集成→E2E）。针对AI测试盲区，我们扩展为6层，每层针对特定盲区维度。

```
                    ┌─────────┐
                    │  L6:混沌 │  ← 数据盲区
                   ┌┴─────────┴┐
                   │ L5:对抗测试 │  ← 边界盲区
                  ┌┴───────────┴┐
                  │ L4:场景测试  │  ← 时序盲区
                 ┌┴─────────────┴┐
                 │  L3:集成测试   │  ← 集成盲区
                ┌┴───────────────┴┐
                │  L2:模块测试    │  ← 功能盲区
               ┌┴─────────────────┴┐
               │  L1:单元测试      │  ← 基础覆盖
               └───────────────────┘
```

### L1: 单元测试 (Unit Tests)

**目标**: 每个函数/方法的基本正确性  
**覆盖维度**: 功能盲区  
**要求**:
- 所有 `export` 的函数/类必须有至少1个测试
- 每个公共方法的 happy path 必须覆盖
- 纯函数需要覆盖典型输入输出

**命名规范**: `{SourceFile}.test.ts`

### L2: 模块测试 (Module Tests)

**目标**: 模块内部多个函数协作的正确性  
**覆盖维度**: 功能盲区 + 边界盲区  
**要求**:
- 模块内核心业务流程的完整路径覆盖
- 错误处理分支覆盖
- 状态转换覆盖（初始化→运行→销毁）

**命名规范**: `{SourceFile}-p{N}.test.ts`（分片测试）

### L3: 集成测试 (Integration Tests)

**目标**: 跨模块交互的正确性  
**覆盖维度**: 集成盲区  
**要求**:
- 覆盖所有已知的跨模块业务流程
- 验证模块间接口契约
- 验证事件传递和数据流转

**命名规范**: `integration/{flow-name}.integration.test.ts`

### L4: 场景测试 (Scenario Tests)

**目标**: 完整玩家行为路径的正确性  
**覆盖维度**: 时序盲区 + 集成盲区  
**要求**:
- 模拟真实玩家操作序列
- 覆盖关键游戏循环（新手引导→日常→赛季→终局）
- 验证状态在时间维度上的一致性

**命名规范**: `scenarios/{scenario-name}.scenario.test.ts`

### L5: 对抗测试 (Adversarial Tests)

**目标**: 用"攻击者思维"主动寻找漏洞  
**覆盖维度**: 边界盲区 + 功能盲区  
**要求**:
- 极端输入测试（零值、负值、极大值、特殊字符）
- 非法状态测试（未初始化就使用、重复销毁）
- 资源竞争测试（并发操作、重复操作）
- 负面测试（"不应该发生什么"）

**命名规范**: `adversarial/{module}-adversarial.test.ts`

### L6: 混沌测试 (Chaos Tests)

**目标**: 验证系统在异常条件下的韧性  
**覆盖维度**: 数据盲区 + 时序盲区  
**要求**:
- 随机数据注入（fuzzing）
- 存档损坏恢复
- 模拟网络延迟/超时
- 内存压力测试

**命名规范**: `chaos/{module}-chaos.test.ts`

## 3. 评估机制

### 3.1 覆盖率指标 (Coverage Metrics)

| 指标 | 公式 | 含义 | 目标 |
|------|------|------|------|
| 源文件覆盖率 (SFC) | `已测试源文件数 / 总源文件数 × 100%` | 有测试的源文件比例 | ≥ 95% |
| 函数覆盖率 (FC) | `已调用函数数 / 总函数数 × 100%` | 被测试调用过的函数比例 | ≥ 90% |
| 分支覆盖率 (BC) | `已执行分支数 / 总分支数 × 100%` | if/else/switch 分支覆盖 | ≥ 85% |
| 路径覆盖率 (PC) | `已执行路径数 / 总可行路径数 × 100%` | 执行路径组合覆盖 | ≥ 70% |

**优先级**: SFC > FC > BC > PC（先广度后深度）

### 3.2 质量指标 (Quality Metrics)

| 指标 | 公式 | 含义 | 目标 |
|------|------|------|------|
| 缺陷发现率 (DD) | `测试发现的缺陷数 / 总测试用例数 × 100%` | 测试用例发现问题的效率 | ≥ 5% |
| 测试有效性 (TE) | `能检测到代码变更的测试数 / 总测试数 × 100%` | 测试对回归的敏感度 | ≥ 30% |
| 变异杀死率 (MK) | `被杀死的变异体数 / 总变异体数 × 100%` | 测试发现代码缺陷的真实能力 | ≥ 80% |
| 断言密度 (AD) | `总断言数 / 总测试用例数` | 每个测试的平均断言数 | ≥ 3 |

### 3.3 对抗指标 (Adversarial Metrics)

| 指标 | 公式 | 含义 | 目标 |
|------|------|------|------|
| 对抗测试通过率 (ATP) | `通过的对抗测试数 / 总对抗测试数 × 100%` | 系统对异常输入的防御能力 | 100% |
| 边界击穿率 (BBR) | `边界测试发现的缺陷数 / 总边界测试数 × 100%` | 边界条件的脆弱程度 | ≤ 2% |
| 混沌存活率 (CSR) | `混沌测试中系统正常运行的次数 / 总混沌测试次数 × 100%` | 系统韧性 | ≥ 95% |

## 4. 度量体系

### 4.1 盲区指数 (Blind Spot Index, BSI)

```
BSI = 无测试源文件数 / 总非平凡源文件数 × 100%
```

**非平凡文件**: 排除 `index.ts`、`*Types.ts`、`*types.ts`、`*Config.ts`、`*-config.ts`、`*.d.ts`

**分级标准**:
- 🔴 危险: BSI > 40%
- 🟡 警告: 20% < BSI ≤ 40%
- 🟢 健康: BSI ≤ 20%
- 🏆 优秀: BSI ≤ 5%

### 4.2 测试深度 (Test Depth, TD)

```
TD = 总断言数 / 总导出函数数
```

**含义**: 平均每个被测函数有多少个断言来验证其行为。

**分级标准**:
- 🔴 浅层: TD < 2
- 🟡 中等: 2 ≤ TD < 5
- 🟢 深入: 5 ≤ TD < 10
- 🏆 穷举: TD ≥ 10

### 4.3 交叉验证率 (Cross-Validation Rate, CVR)

```
CVR = 被独立测试验证的模块数 / 总模块数 × 100%
```

**独立测试**: 非同源AI生成的测试，或由不同策略生成的测试。

**含义**: 避免同一AI的思维定式，确保测试视角的多样性。

**实现方式**:
- 单元测试与集成测试由不同轮次生成
- 对抗测试使用"反向思维"策略生成
- 定期用变异测试交叉验证

### 4.4 综合健康度 (Composite Health Score, CHS)

```
CHS = w1×(1-BSI) + w2×min(TD/10, 1) + w3×CVR + w4×BC
```

**权重**: w1=0.3, w2=0.2, w3=0.25, w4=0.25

**分级**:
- 🔴 危险: CHS < 0.4
- 🟡 警告: 0.4 ≤ CHS < 0.6
- 🟢 健康: 0.6 ≤ CHS < 0.8
- 🏆 优秀: CHS ≥ 0.8

## 5. 迭代流程

### 5.1 每轮迭代 R(n) 的标准流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  扫描    │───→│  评估    │───→│  补全    │───→│  验证    │───→│  记录    │
│  Scan    │    │ Evaluate │    │  Patch   │    │  Verify  │    │  Record  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
     │               │               │               │               │
     ▼               ▼               ▼               ▼               ▼
  发现未覆盖     计算BSI/TD      生成测试代码    运行测试套件    更新报告
  的源文件       识别盲区类型     优先级排序      确认覆盖率     沉淀经验
```

### 5.2 各阶段详细说明

#### Stage 1: 扫描 (Scan)

**输入**: 项目源码目录  
**操作**:
1. 扫描所有 `engine/` 下的 `.ts` 源文件
2. 排除类型定义文件和索引文件
3. 检查每个源文件是否有对应的测试文件
4. 输出未覆盖文件清单

**输出**: `uncovered-files.md`

#### Stage 2: 评估 (Evaluate)

**输入**: 未覆盖文件清单  
**操作**:
1. 分析每个未覆盖文件的复杂度（行数、函数数、分支数）
2. 按五维盲区模型分类风险
3. 计算当前 BSI、TD、CVR
4. 确定本轮优先级

**输出**: `evaluation-report.md`

#### Stage 3: 补全 (Patch)

**输入**: 评估报告 + 优先级排序  
**操作**:
1. 按优先级从高到低，为未覆盖文件生成测试
2. 遵循6层金字塔策略：
   - 奇数轮侧重 L1-L2（广度覆盖）
   - 偶数轮侧重 L3-L4（深度覆盖）
   - 每5轮插入 L5-L6（对抗/混沌）
3. 每个测试文件必须包含：
   - Happy path 测试
   - 至少2个边界/异常测试
   - 状态一致性断言

**输出**: 新增测试文件

#### Stage 4: 验证 (Verify)

**输入**: 新增测试文件  
**操作**:
1. 运行全量测试套件，确认无回归
2. 运行覆盖率工具，收集最新指标
3. 对新增测试执行变异测试抽样
4. 验证新测试确实能检测到代码变更

**输出**: `coverage-report.json`

#### Stage 5: 记录 (Record)

**输入**: 全部本轮数据  
**操作**:
1. 填写 `rounds/round-N.md` 迭代记录
2. 更新 `baseline-report.md` 的当前指标
3. 记录本轮发现的新问题和经验教训
4. 规划下一轮重点

**输出**: `rounds/round-N.md`

### 5.3 轮次规划策略

| 轮次类型 | 频率 | 重点 | 盲区维度 |
|----------|------|------|----------|
| 广度轮 (Breadth) | 奇数轮 | 覆盖更多源文件 | 功能盲区 |
| 深度轮 (Depth) | 偶数轮 | 加深已有覆盖 | 边界盲区 |
| 集成轮 (Integration) | 每3轮 | 跨模块流程 | 集成盲区 |
| 对抗轮 (Adversarial) | 每5轮 | 极端/异常场景 | 边界+功能 |
| 混沌轮 (Chaos) | 每10轮 | 随机/压力/恢复 | 数据+时序 |

## 6. 反自洽陷阱机制

### 6.1 问题：AI既写代码又写测试

当AI同时负责实现和测试时，会产生"自我确认循环"：
- AI实现的bug → AI认为这是正确行为 → AI写的测试也验证了这个错误行为
- AI遗漏的边界 → AI不会想到测试这个边界 → 覆盖率看起来很高但实际有盲区

### 6.2 对策：四重校验机制

#### 校验1: 变异测试 (Mutation Testing)

```
对源码注入变异 → 运行测试 → 检查是否被检测到
```

变异类型：
- 算术运算替换：`+` → `-`, `*` → `/`
- 条件边界偏移：`<` → `<=`, `>` → `>=`
- 返回值替换：返回 `0`、`null`、`undefined`
- 逻辑运算替换：`&&` → `||`, `!` → ``
- 常量替换：配置值偏移 ±10%

#### 校验2: 契约测试 (Contract Testing)

```
从接口定义推导 → 生成断言 → 验证实现是否符合契约
```

- 从 TypeScript 类型定义自动推导属性断言
- 从函数签名推导参数组合
- 从 JSDoc 推导行为预期

#### 校验3: 逆向测试 (Inverse Testing)

```
从"不应该发生什么" → 生成负面断言
```

- 资源扣减后余额不应为负
- 升级后等级不应回退
- 事件不应重复触发
- 并发操作不应导致数据不一致

#### 校验4: 交叉审查 (Cross Review)

```
轮次A的测试 → 轮次B审查 → 发现遗漏 → 补充测试
```

- 每轮结束后，回顾前2轮的测试质量
- 用"如果我是攻击者，怎么绕过这个测试？"的思维审查
- 记录发现的遗漏作为下一轮的输入

## 7. 实施路线图

### Phase 1: 基线建立 (Round 0)

- [x] 扫描全部源文件和测试文件
- [x] 计算初始 BSI、TD
- [x] 识别未覆盖模块清单
- [x] 建立迭代框架和模板

### Phase 2: 广度覆盖 (Round 1-5)

- 目标：BSI 从当前值降至 < 15%
- 策略：L1 单元测试 + L2 模块测试
- 重点：未覆盖的源文件

### Phase 3: 深度覆盖 (Round 6-10)

- 目标：BC > 80%, TD > 5
- 策略：L3 集成测试 + L4 场景测试
- 重点：关键业务流程的路径覆盖

### Phase 4: 对抗验证 (Round 11-15)

- 目标：MK > 75%, BBR < 3%
- 策略：L5 对抗测试 + 变异测试
- 重点：边界条件和异常处理

### Phase 5: 持续优化 (Round 16+)

- 目标：CHS > 0.8 并持续维持
- 策略：L6 混沌测试 + 持续监控
- 重点：回归防护和新功能同步测试

  ! -name "index.ts" ! -name "*Types.ts" ! -name "*types.ts" \
  ! -name "*.types.ts" ! -name "*-types.ts" ! -name "*Config.ts" \
  ! -name "*-config.ts" ! -name "*.config.ts" ! -name "*.d.ts"); do
  base=$(basename "$f" .ts)
  dir=$(dirname "$f")
  has_test=false
  for t in "$dir/__tests__/${base}"*.test.ts; do
    [ -f "$t" ] && has_test=true && break
  done
  $has_test || UNCOVERED=$((UNCOVERED + 1))
done

BSI=$(echo "scale=2; $UNCOVERED * 100 / $TOTAL_NON_TRIVIAL" | bc)
echo "BSI: ${BSI}% (${UNCOVERED}/${TOTAL_NON_TRIVIAL})"
